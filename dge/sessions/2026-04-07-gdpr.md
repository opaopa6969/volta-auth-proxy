## Scene 1: 適用範囲と個人データインベントリ

GDPR は EU 居住者のデータを扱う全サービスに適用。
日本の APPI も同様の要件あり → GDPR ベースで設計すれば APPI もカバー。

個人データを持つテーブル:
  users (email, display_name, google_sub)
  sessions (ip_address, user_agent)
  trusted_devices (user_agent, ip_address)
  auth_flows (context JSONB — PII含む)
  auth_flow_transitions (context_snapshot — @Sensitive redacted)
  audit_logs (actor_ip, detail JSONB)
  user_mfa (secret — 暗号化済み)
  passkeys (credential_id, public_key)
  memberships (user_id ↔ tenant_id)
  invitations (email)

→ Gap GDPR-1: 個人データインベントリ [Critical]
→ Gap GDPR-2: APPI/GDPR 両対応確認 [Medium]

## Scene 2: データエクスポート

API: POST /api/v1/users/me/data-export → 非同期 → download URL
形式: JSON (GDPR Article 20 要件: 構造化・機械可読)
含める: users, memberships, sessions, trusted_devices, passkeys(メタのみ), mfa(状態のみ), audit_log
除外: TOTP secret, passkey 秘密鍵, recovery code hash (漏洩時のリスク)
完了通知: メールで通知
ファイル有効期限: 24時間後に自動削除

→ Gap GDPR-3: エクスポート API [High]
→ Gap GDPR-4: 含む/含まないデータ [High]
→ Gap GDPR-5: 完了通知 [Medium]
→ Gap GDPR-6: ファイル保管・有効期限 [Medium]

## Scene 3: データ削除 (Right to be Forgotten)

削除対象の分類:
  即時削除: users, sessions, trusted_devices, user_mfa, passkeys, auth_flows PII
  匿名化: audit_logs (actor_id→"DELETED_USER", actor_ip→null)
  保持: audit_logs のイベント構造 (匿名化後), billing関連

猶予期間: 30日間は論理削除 (deleted_at)。ログインで撤回可能。30日後に物理削除。
App 連携: Webhook (user.deletion_requested / user.deletion_completed)
FK 制約: audit_logs は ON DELETE SET NULL に変更

Phase 1: 同期エクスポート + 論理削除 + GCジョブ + audit匿名化
Phase 2: 非同期エクスポート + Webhook + 管理画面
Phase 3: DPA テンプレート + テナント別保持ポリシー + 削除レポート

→ Gap GDPR-7: 削除対象の分類 [Critical]
→ Gap GDPR-8: 30日猶予設計 [High]
→ Gap GDPR-9: audit_logs匿名化 [High]
→ Gap GDPR-10: App削除連携 Webhook [High]
→ Gap GDPR-11: FK制約見直し [High]
→ Gap GDPR-12: 削除UI + 確認フロー [Medium]

## Scene 4: エッジケース

再登録: 同一メールで再登録しても新規ユーザー扱い。以前のデータは復元しない。以前のアカウントの存在を暗示しない。
プライバシーポリシー: 削除猶予期間、エクスポート手段、保持期間を明記。
DB: users に deleted_at カラム追加。auth_logs の FK を SET NULL に変更。

→ Gap GDPR-13: 再登録時の扱い [Medium]
→ Gap GDPR-14: プライバシーポリシー更新 [Medium]
→ Gap GDPR-15: DB migration [High]
