# Decision 001: フォーム入力状態の復元を実装しない

**Status:** Rejected
**Date:** 2026-04-01

## 何を検討したか

セッション切れ → ログインリダイレクト → 戻ってきたときに、
ユーザーがフォームに入力していた値を復元する仕組みをvoltaに持たせるか。

実装候補として以下を検討した：

- `sessionStorage` を使ったクライアントサイド保存・復元
- `return_to` URLにフォーム値を含める
- `__volta_flash` の拡張版としてサーバーサイドform state cookie（Max-Age ≒ 60秒）

## なぜやらないか

**責務が違う。**

フォームの入力値を守るのはアプリケーション層の責務であり、認証基盤の責務ではない。
voltaがこれを吸収し始めると、次の芋づるが見えている：

1. フォームデータのcookie暗号化
2. 大きなペイロードのための一時ストレージ
3. アプリごとに異なるフォーム構造への対応
4. …

これはKeycloakが辿った「設定地獄への道」の入り口と同じ形をしている。

**今の技術前提では問題自体が発生しにくい。**

voltaの設計（ForwardAuth + JWT 5分 + silent refresh）と
現代のSPA / AJAX前提のアプリ構成では、
セッション切れは silent refresh によってユーザーに見えないまま処理される。
フォーム送信中に401が返っても、リトライでリカバリできる。

「ブログを書いていたら切れた」問題は、
認証基盤ではなく **autosave**（`localStorage` or 定期的なAPI保存）で解くべき問題。

## 正しい解決場所

| 問題 | 解決場所 |
|------|----------|
| セッション切れに気づかずフォーム送信失敗 | volta-sdk-js の silent refresh |
| 長文入力中のデータ消失 | アプリ側の autosave（localStorage / draft API） |
| ログイン後に元のページに戻る | volta の `return_to` パラメータ（実装済み） |

## 教訓

「やればできる」と「やるべき」は別。
Convenience / sugar を足したくなる衝動は、過去に痛い目にあった経験から来ることが多い。
その衝動に気づいたときこそ、**入れない理由を言語化してrejectする**のが設計の筋肉。

voltaの強みは「認証に集中していること」。その範囲を守る。
