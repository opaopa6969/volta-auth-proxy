# Decision 004: テナント切り替え時に MFA 再検証を要求する

**Status:** Accepted
**Date:** 2026-04-14

## 何を決めたか

テナント切り替え (`/auth/switch-tenant`) 時に、旧セッションの `mfaVerifiedAt` を新セッションへ
コピーする処理を削除した。MFA はテナントスコープとし、テナント切り替えはセキュリティ境界の
越境として扱う。

**変更前**: 旧セッションで MFA 検証済みなら、新テナントのセッションにも `mfaVerifiedAt` をコピー → MFA スキップ
**変更後**: 新セッションは `mfaVerifiedAt = null` で発行 → MFA 有効ユーザーは新テナントで再検証が必要

## 検討した選択肢

| 選択肢 | 内容 | 採否 |
|--------|------|------|
| **(a)** carry-over を維持 | 「MFA は人を証明する、テナントではない」として現状維持。ADR で明文化 | 却下 |
| **(b)** 設定可能にする | `tenancy.mfa.scope: session \| tenant` で切り替え | 却下 — 複雑さに見合わない |
| **(c)** carry-over を削除 | テナント切り替え = セキュリティ境界越え → MFA 再検証 | **採用** |

## なぜ (c) を採用したか

1. **セキュリティ原則**: マルチテナント環境ではテナントがセキュリティ境界。テナント A の MFA 検証がテナント B で有効であるべきではない
2. **Rust ポートとの整合**: `volta-gateway/auth-server` は既にこの挙動（carry-over なし）で実装済み。Java 側を合わせることで両実装の一貫性を確保
3. **コードの正しさ**: `issueSession()` は MFA 有効ユーザーに対して `mfaVerifiedAt = null` を正しく設定している。carry-over はその正しいデフォルトを上書きしていた
4. **元 issue との整合**: セキュリティ issue #12 が指摘していたのはまさにこの session-level MFA state の越境

## なぜ (a) を却下したか

「MFA は人を証明する」という論理は一見もっともだが、テナントごとに異なるセキュリティポリシーを
持つ可能性がある。テナント B の管理者が MFA を要求しているのに、テナント A からの切り替えで
スキップされるのはポリシー違反になりうる。

## なぜ (b) を却下したか

現時点でテナントスコープ MFA を不要とするユースケースがない。設定項目の追加は複雑さのコストに
見合わない。将来必要になった時点で追加すれば十分。

## 実装

- `AuthRouter.java` — `/auth/switch-tenant` から MFA carry-over ブロック（旧 L308-318）を削除
- 旧セッションの `revokeSession()` は維持
