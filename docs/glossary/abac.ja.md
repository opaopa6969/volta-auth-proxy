# ABAC（属性ベースアクセス制御）

[English version](abac.md)

***

## これは何？

ABAC（Attribute-Based Access Control）は、属性に基づいてアクセス判断を行う認可モデルです。ユーザー、リソース、アクション、環境のプロパティを評価します。単に「このユーザーはADMINロールを持っているか？」とチェックする代わりに、ABACは「このユーザーはエンジニアリング部門に所属しているか、リソースはそのテナント内か、業務時間内か、企業IPからアクセスしているか」といった複雑なルールを評価できます。

非常に詳細なチェックをするナイトクラブの門番のようなものです。シンプルな門番はゲストリストに名前があるかだけチェックします（ロールベース）。ABACの門番は年齢、服装、時間帯、クラブの収容状況、VIPリストバンドの有無、今夜のイベントがチケットタイプを許可するかをチェックします。各属性がアクセス判断に次元を追加します。

ABACは単純な[ロール](role.md)ベースアクセス制御（RBAC）より柔軟です。RBACが粗い制御（「管理者はすべてできる」）を提供するのに対し、ABACはきめ細かい、コンテキストを考慮したポリシーを提供します。

***

## なぜ重要なのか？

RBACは単純なシナリオでは機能しますが、現実のアクセス制御は単純であることが稀です：

```
  RBAC（シンプルだが限定的）：
  ┌─────────────────────────────────────┐
  │  ユーザーはADMINロールを持つ？        │
  │    YES → すべて許可                  │
  │    NO  → 拒否                       │
  └─────────────────────────────────────┘

  ABAC（柔軟でコンテキスト考慮）：
  ┌─────────────────────────────────────────────────────┐
  │  ユーザー属性：                                       │
  │    department = "engineering"                         │
  │    clearance  = "confidential"                       │
  │                                                      │
  │  リソース属性：                                       │
  │    tenant_id  = "acme-uuid"                          │
  │    sensitivity = "confidential"                      │
  │                                                      │
  │  コンテキスト属性：                                    │
  │    time       = 14:30（業務時間）                      │
  │    ip_range   = "10.0.0.0/8"（企業）                  │
  │                                                      │
  │  ポリシー：以下の場合ALLOW                              │
  │    user.department == "engineering" AND               │
  │    user.clearance >= resource.sensitivity AND         │
  │    context.time IN business_hours AND                 │
  │    context.ip IN corporate_range                      │
  └─────────────────────────────────────────────────────┘
```

主な利点：

- **きめ細かい**：ロールレベルではなく属性レベルでアクセスを制御
- **動的**：ポリシーがコンテキスト（時間、場所、リスクスコア）に適応
- **スケーラブル**：ロール階層を再構築せずに新しい属性を追加可能
- **コンプライアンス対応**：規制要件をポリシーとして表現可能（例：「PHIは勤務時間中の医療スタッフのみアクセス可能」）

## どう動くのか？

### ABACコンポーネント

```
  ┌──────────────────────────────────────────────────────┐
  │                   ABAC判断                            │
  │                                                       │
  │  主体            アクション      リソース     コンテキスト│
  │  （誰？）         （何を？）      （何に？）    （いつ？）  │
  │  ┌──────────┐   ┌──────────┐   ┌──────────┐ ┌──────┐│
  │  │ user_id  │   │ read     │   │ tenant_id│ │ time ││
  │  │ roles    │   │ write    │   │ owner_id │ │ ip   ││
  │  │ dept     │   │ delete   │   │ type     │ │ device│
  │  │ clearance│   │ approve  │   │ sensitiv.│ │ risk ││
  │  └──────────┘   └──────────┘   └──────────┘ └──────┘│
  │         │              │              │          │    │
  │         ▼              ▼              ▼          ▼    │
  │  ┌──────────────────────────────────────────────────┐│
  │  │              ポリシーエンジン                       ││
  │  │                                                   ││
  │  │  IF subject.dept == "engineering"                  ││
  │  │  AND action == "read"                             ││
  │  │  AND resource.tenant_id == subject.tenant_id      ││
  │  │  AND context.time IN "09:00-18:00"                ││
  │  │  THEN → ALLOW                                     ││
  │  │  ELSE → DENY                                      ││
  │  └──────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────┘
```

### ポリシー構造

典型的なABACポリシーには4つの部分があります：

1. **ターゲット**：このポリシーはどのリクエストに適用されるか？
2. **条件**：どの属性が満たされる必要があるか？
3. **効果**：ALLOWまたはDENY
4. **義務**：副作用（例：「このアクセスをログに記録」）

voltaのフォーマットでのポリシー例：

```json
{
  "id": "policy-001",
  "name": "エンジニアリングは自テナントデータを読める",
  "target": {
    "action": ["read"],
    "resource_type": ["tenant_data"]
  },
  "condition": {
    "all": [
      {"subject.department": {"eq": "engineering"}},
      {"resource.tenant_id": {"eq": "${subject.tenant_id}"}},
      {"context.hour": {"between": [9, 18]}}
    ]
  },
  "effect": "ALLOW",
  "obligations": [
    {"type": "audit_log", "level": "info"}
  ]
}
```

### ABAC vs RBAC vs ACL

| 機能 | ACL | RBAC | ABAC |
|------|-----|------|------|
| 粒度 | リソースごと | ロールごと | 属性ごと |
| コンテキスト対応 | なし | なし | あり |
| スケーラビリティ | 低（N ユーザー x M リソース） | 良好 | 優秀 |
| ポリシー表現力 | 低 | 中 | 高 |
| 複雑さ | 低 | 中 | 高 |
| 例 | 「Aliceはファイル X を読める」 | 「管理者はすべてのファイルを読める」 | 「エンジニアは業務時間中に自テナントのファイルを読める」 |

### RBACとABACの組み合わせ

実際にはほとんどのシステムが両方を組み合わせます。voltaはロールを多くの属性の1つとして使います：

```
  ┌─────────────────────────────────────────────┐
  │  voltaのハイブリッドアプローチ：               │
  │                                              │
  │  1. RBACレイヤー（高速、粗い）：               │
  │     ユーザーはADMINロール？ → ほぼ全操作許可   │
  │     ユーザーはMEMBERロール？ → ABACチェック    │
  │                                              │
  │  2. ABACレイヤー（柔軟、きめ細かい）：         │
  │     テナントメンバーシップチェック              │
  │     リソース所有権チェック                     │
  │     時間/IP/リスクコンテキストチェック          │
  │     カスタム属性チェック                       │
  │                                              │
  │  結果：一般的なケースは高速、                   │
  │        エッジケースは精密                      │
  └─────────────────────────────────────────────┘
```

## volta-auth-proxyではどう使われている？

### ABACポリシーエンジン（Phase 4）

voltaのPhase 4では、リクエストが[アップストリーム](upstream.md)アプリに到達する前にプロキシレイヤーでポリシーを評価するABACポリシーエンジンを追加します：

```
  ABACありのリクエストフロー：
  ┌──────────┐     ┌──────────────────────────────┐     ┌──────────┐
  │ クライアント│────►│  volta-auth-proxy             │────►│アップ     │
  │          │     │                               │     │ストリーム │
  │          │     │  1. 認証（JWT/セッション）       │     │  アプリ   │
  │          │     │  2. 属性を抽出                  │     │          │
  │          │     │  3. ABACポリシーを評価           │     │          │
  │          │     │  4. ALLOW → 転送               │     │          │
  │          │     │     DENY  → 403 Forbidden      │     │          │
  └──────────┘     └──────────────────────────────────┘     └──────────┘
```

### voltaの属性ソース

| 属性カテゴリ | ソース | 例 |
|------------|--------|-----|
| 主体（ユーザー） | [JWT](jwt.md)クレーム、ユーザープロファイル | `volta_roles`、`department`、`tenant_id` |
| リソース | リクエストパス、[ヘッダー](header.md) | `/api/v1/tenants/{tid}/...`、[content-type](content-type.md) |
| アクション | HTTPメソッド | GET=read、POST=create、DELETE=delete |
| コンテキスト | リクエストメタデータ | クライアントIP、時間、user-agent |

### ポリシー管理API

```
  POST /api/v1/tenants/{tid}/policies
  {
    "name": "メンバーは自テナントのみ読み取り可",
    "target": {"action": ["read"]},
    "condition": {
      "all": [
        {"subject.volta_roles": {"contains": "MEMBER"}},
        {"resource.tenant_id": {"eq": "${subject.volta_tid}"}}
      ]
    },
    "effect": "ALLOW"
  }
```

### ABAC + 既存のvolta機能

ABACポリシーは他のvolta機能のデータを参照できます：

- **[ロール](role.md)**：`subject.volta_roles contains "ADMIN"`
- **[テナント](tenant.md)**：`resource.tenant_id == subject.volta_tid`
- **[M2M](m2m.md)**：`subject.volta_client == true`（マシンに異なるルール）
- **[停止](suspension.md)**：`subject.active == true`（停止ユーザーを自動拒否）

## よくある間違いと攻撃

### 間違い1：過度に複雑なポリシー

ABACのパワーは同時に危険でもあります。10以上の条件を持つポリシーは監査とデバッグが困難です。ポリシーをシンプルに保ち、組み合わせてください。

### 間違い2：ポリシーの相互作用をテストしない

2つのポリシーが矛盾する可能性：一方が許可、他方が拒否。明確な競合解決戦略を定義してください（例：拒否が許可を上書き）。

### 間違い3：クライアント側の属性だけに依存する

属性がクライアントから来る場合（例：`department=admin`と主張するカスタムヘッダー）、偽造される可能性があります。属性は信頼できるソースから取得すべきです：[JWT](jwt.md)、データベース、またはvolta自身のコンテキスト。

### 間違い4：デフォルト拒否がない

リクエストにどのポリシーも一致しない場合、何が起きるか？デフォルト拒否がないと、一致しないリクエストが通過する可能性があります。常にデフォルト拒否ポリシーを設定してください。

### 攻撃：属性インジェクション

攻撃者が偽の属性（例：偽装されたIPヘッダー）を含むリクエストを作成してポリシーを迂回する。防御：voltaが信頼できるソースからのみ属性を除去して再導出。

### 攻撃：ポリシー改ざん

ポリシーストアが保護されていない場合、攻撃者がポリシーを変更して自分にアクセスを付与できる。防御：ポリシー変更にADMINロールが必要、すべての変更が監査ログに記録、ポリシーはバージョン管理。
