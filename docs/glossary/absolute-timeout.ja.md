# 絶対タイムアウト

[English version](absolute-timeout.md)

---

## これは何？

絶対タイムアウトとは、ユーザーがどれだけ操作していても、セッション作成から一定時間が経過したら強制的に切れるタイムアウト方式です。映画のチケットのようなものです。19時の上映回のチケットは、18:30 に着いても 18:59 に着いても有効ですが、上映が終われば無効になります。

**スライディングウィンドウ有効期限**（[sliding-window-expiry.md](sliding-window-expiry.md) 参照）とは逆のアプローチです。

---

## なぜ重要？

絶対タイムアウトはセキュリティの硬い境界線を提供します。攻撃者がセッションを乗っ取って操作を続けても、いずれ必ずセッションが切れます。被害の時間枠を限定できるのです。

銀行、医療、政府機関などの高セキュリティ環境では特に重要で、規制により一定時間でセッションを切ることが求められることがあります（例：銀行は15分、企業は8時間）。

### どちらを使うべき？

| 方式 | 向いている場面 | 例 |
|------|--------------|-----|
| スライディングのみ | 低リスクアプリ、利便性重視 | ブログ、Wiki |
| 絶対タイムアウトのみ | 高セキュリティ、厳格なコンプライアンス | 銀行、医療ポータル |
| 両方組み合わせ | ほとんどの SaaS でベストプラクティス | スライディング=30分、絶対=8時間 |

組み合わせ方式がベストです。アクティブな作業中は中断されず（スライディング）、でもセッションが永遠に生き続けることもない（絶対タイムアウト）。

---

## 簡単な例

```
09:00  ユーザーがログイン     絶対期限 = 17:00（最大8時間）
11:30  ページをクリック       絶対期限 = 17:00（変わらず）
14:00  ページをクリック       絶対期限 = 17:00（変わらず）
16:59  ページをクリック       絶対期限 = 17:00（変わらず）
17:00  セッション終了。再ログインが必要。
```

どれだけ操作しても、17:00 の期限は動きません。

---

## volta-auth-proxy では

volta は現在、**スライディングウィンドウのみ**（8時間）を使っています。絶対タイムアウトの上限は設けていません。`sessions` テーブルには `created_at` カラムがあるため、将来的に絶対タイムアウトを追加できますが、現時点ではチェックしていません。

ほとんどの SaaS 製品では、volta のアプローチで十分です。8時間のスライディングウィンドウにより、操作しない日は実質的に業務終了後にセッションが切れます。厳格なセッション制御が求められる場合（例：SOC 2 準拠）は、以下のようなチェックを追加します：

```java
if (session.createdAt().plusSeconds(absoluteMaxSeconds).isBefore(Instant.now())) {
    // 再認証を強制
}
```

これは volta が企業コンプライアンス認証を目指す際の将来の拡張候補です。
