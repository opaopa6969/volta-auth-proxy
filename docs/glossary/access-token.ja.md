# Access Token（アクセストークン / OAuth 2.0）

## これは何？

アクセストークンとは、認可（許可）を表す認証情報です。保有者に対して、特定のリソースへのアクセスや特定のアクションの実行を許可します。サービスにログインして、そのサービスの API で使うトークンをもらったら、それがアクセストークンです。

カンファレンスの名札のようなものです。受付で身分を証明します（身分証明書を見せる）。名札をもらいます。以降は、セッション入場、展示会場へのアクセス、昼食の受け取りで名札を見せるだけです。名札がアクセストークンです。ドアごとに身分証明書を見せる必要はなく、名札を見せればいいのです。

アクセストークンには 2 つの形式があります:

**不透明（Opaque）トークン**はランダムな文字列のように見えます（例: `ya29.a0AfH6SMA...`）。読める情報は含まれていません。意味を知るには、発行者のイントロスペクション(内省)エンドポイントに送る必要があります。受付に電話して「名札 #4523 は有効ですか？」と聞くようなものです。

**JWT ベースのトークン**は構造化されていて、自己完結しています。情報（claims）が内部に含まれ、発行者が署名しています。発行者の公開鍵があれば、ネットワーク通信なしに読み取り・検証できます。名前、役割、権限が印刷されていて、ホログラムシールで封印された名札のようなものです。

```
  不透明トークン:            JWT アクセストークン:
  +-----------------+     +---------------------------+
  | ya29.a0AfH6SMA  |     | Header: {alg, kid}        |
  | (読み手には      |     | Payload: {sub, iss, aud,  |
  |  意味がない)     |     |   exp, roles, tenant}     |
  |                 |     | Signature: (RSA-SHA256)   |
  +-----------------+     +---------------------------+
  検証には発行者への       公開鍵 (JWKS) を使って
  問い合わせが必要         ローカルで検証可能
```

## なぜ重要？

アクセストークンは API セキュリティの骨格です。これがなかったら、毎回の API コールにユーザー名とパスワードを含める必要があり、以下の問題が生じます:

- **安全でない** -- パスワードが常に通信され、より多くの場所に保存される。
- **柔軟でない** -- アプリケーションができることを制限（スコーピング）できない。
- **取り消せない** -- パスワードを変更せずにアクセスを遮断できない。

アクセストークンはこれらの問題を解決します:

1. **限られた有効期間。** トークンは設定された期間（分、時間、日）で期限切れになる。盗まれても被害が限定される。
2. **権限の範囲指定。** トークンを特定のアクション（例: 「メールの読み取り」はできるが「メールの削除」はできない）に制限できる。
3. **取り消し可能。** ユーザーのパスワードを変更せずにトークンを無効化できる。
4. **委任。** サードパーティのアプリケーションが、ユーザーのパスワードを見ることなく、ユーザーに代わってトークンを取得できる。

## どう動くの？

典型的な OAuth 2.0 のフロー:

```
  ユーザー     アプリ         認証サーバー        API
  |           |              |                  |
  | ログイン   |              |                  |
  |---------->|              |                  |
  |           | 認証リクエスト  |                  |
  |           |------------->|                  |
  |           |              |                  |
  | 認証情報   |              |                  |
  | 入力       |              |                  |
  |---------->|------------->|                  |
  |           |              |                  |
  |           | access_token |                  |
  |           |<-------------|                  |
  |           |              |                  |
  |           | API コール                       |
  |           | Authorization: Bearer <token>   |
  |           |-------------------------------->|
  |           |              |                  |
  |           |              |   トークン検証     |
  |           |              |   スコープ確認     |
  |           |              |                  |
  |           | レスポンス     |                  |
  |           |<--------------------------------|
```

API サーバーは 2 つの方法のいずれかでトークンを検証します:
- **不透明トークン:** 認証サーバーのイントロスペクションエンドポイントを呼び出す。
- **JWT トークン:** 認証サーバーの公開鍵（JWKS）を使ってローカルで署名を検証する。

### 有効期間の考慮事項

アクセストークンの有効期間はトレードオフです:

| 短い有効期間（分単位） | 長い有効期間（時間/日単位） |
|---|---|
| 盗まれた場合の被害が少ない | 認証のやり取りが少ない |
| 頻繁な更新が必要 | ユーザー体験が良い |
| より安全 | 盗まれた場合のリスクが高い |

よくあるパターンは、短命のアクセストークン（5-15分）と長命のリフレッシュトークンの組み合わせです。API コールにはアクセストークンを使い、期限切れになったらリフレッシュトークンで新しいものを取得します。ユーザーに再ログインを求める必要はありません。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は内部エコシステム向けに JWT ベースのアクセストークンを使用しています。

**volta は独自の JWT を発行します。** ユーザーが認証すると（Google OIDC 経由）、volta はセッションを作成し、JWT アクセストークンを発行できます。この JWT は volta の RSA 秘密鍵で RS256 を使って署名され、下流のアプリケーションは volta の JWKS エンドポイント（`/.well-known/jwks.json`）で検証します。

volta の JWT アクセストークンの内容:

```json
{
  "iss": "volta-auth",
  "aud": ["volta-apps"],
  "sub": "user-uuid",
  "exp": 1711896700,
  "iat": 1711896400,
  "jti": "unique-token-id",
  "volta_v": 1,
  "volta_tid": "tenant-uuid",
  "volta_roles": ["ADMIN"],
  "volta_display": "Alice Smith",
  "volta_tname": "Acme Corp",
  "volta_tslug": "acme"
}
```

主な特徴:

- **短命。** デフォルトの TTL は 300 秒（5 分）で、`JWT_TTL_SECONDS` で設定可能。
- **認可データを含む。** テナント ID、ロール、表示名が埋め込まれているので、下流のアプリが volta に追加のコールをする必要がない。
- **発行ごとにユニーク。** 各トークンは監査証跡のためにユニークな `jti`（JWT ID）を持つ。
- **スキーマバージョン付き。** `volta_v` claim がトークンのスキーマバージョンを追跡する。

**トークン配信の 2 つの経路:**

1. **ForwardAuth 経路。** Traefik が volta の `/auth/verify` エンドポイントを呼び出すと、volta はユーザーのセッション Cookie を検証し、新しい JWT を発行します。この JWT は `X-Volta-JWT` レスポンスヘッダで下流のアプリケーションに渡されます。ブラウザは JWT を見ません。サーバー間で移動します。

2. **直接 API 経路。** アプリケーションは有効なセッション Cookie を持って `POST /auth/refresh` を呼び出し、JSON レスポンスで JWT を受け取れます。volta-sdk-js（ブラウザ SDK）はこれを使って直接 API コール用のトークンを取得します。

```
  ForwardAuth 経路:                    直接 API 経路:

  ブラウザ -> Traefik -> volta         ブラウザ -> volta
                         |                         |
                   /auth/verify             /auth/refresh
                         |                         |
                   セッション Cookie         セッション Cookie
                   を検証                    を検証
                         |                         |
                   X-Volta-JWT              JWT を JSON
                   ヘッダで JWT              ボディで返す
                   を発行                          |
                         |                  ブラウザが JWT を
                   下流アプリが              API コールに使用
                   JWT を受信
```

**マシン間トークン。** volta はサーバー間通信用の M2M（Machine-to-Machine）アクセストークンも発行します。これらのトークンには `volta_client: true` と `volta_client_id` の claims が含まれ、ユーザートークンと区別されます。

## よくある間違い

**1. 長命のアクセストークンを使う。**
数日や数週間有効なトークンは、盗まれた場合の脆弱性の窓口が大きくなります。アクセストークンは短命（分単位）にして、更新にはリフレッシュトークンやセッション Cookie を使いましょう。

**2. アクセストークンを localStorage に保存する。**
localStorage はページ上のどんな JavaScript からもアクセスできます。XSS の脆弱性があると、トークンが盗まれます。volta はブラウザでは HttpOnly セッション Cookie を使い、JWT はサーバー間でのみ発行することでこれを回避しています。

**3. API 側でトークンを検証しない。**
トークンを盲目的に信頼する開発者がいます。常に署名を検証し、発行者、対象者、有効期限を確認しましょう。volta の `JwtService.verify()` はこれら全てをチェックします。

**4. JWT の claims に機密データを入れる。**
JWT は署名されていますが、暗号化されていません。トークンを持っている人なら誰でもペイロードをデコードできます。パスワード、クレジットカード番号、その他の秘密を JWT の claims に入れてはいけません。volta の JWT には身元と認可の claims のみが含まれます。

**5. アクセストークンと id_token を混同する。**
目的が異なります。id_token は身元を証明します（アプリ向け）。アクセストークンはアクセスを許可します（API 向け）。詳しくは id_token の記事を参照してください。

**6. 対象者（audience）claim を含めない。**
`aud` claim がないと、トークンがどの API でも使えてしまいます。同じ発行者を信頼する複数のサービスがある場合は危険です。volta は常に `volta-apps` という対象者を含め、検証時にチェックします。
