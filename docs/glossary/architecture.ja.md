# アーキテクチャ

[English version](architecture.md)

---

## これは何？

アーキテクチャとは、ソフトウェアシステムの高レベルな構造のことです -- どんな部品があり、各部品が何をし、互いにどう接続するか。詳細なコードを書く前に、アプリケーションの構成を決める設計図です。

建物の間取り図で考えてみましょう。工事の前に、建築家が決めます：何階建てか、入口をどこにするか、配管をどこに通すか、どの部屋がどの廊下につながるか。あとで家具の配置換えはできます（コードのリファクタリング）。しかし、建物ができた後で耐力壁を動かすのは非常に高くつきます。ソフトウェアアーキテクチャも同じです -- 早期に下す大きな決定が、その後のすべてを形作ります。

アーキテクチャはコードの全行を記述するものではありません。箱と矢印を記述します：「TraefikがVolta-auth-proxyと通信し、それがPostgreSQLと通信する。」各箱の中身は実装であり、箱と矢印がアーキテクチャです。

---

## なぜ重要なのか？

- **悪いアーキテクチャは後から直すのが高くつく。** データベースの変更やモノリスのマイクロサービス化は日単位ではなく月単位。
- **アーキテクチャがスケーラビリティを決める。** シングルプロセスアーキテクチャは、スケール計画なしに10倍のトラフィックを処理できない。
- **アーキテクチャがセキュリティ境界を決める。** 認証がどこで行われるか、どのコンポーネントが互いを信頼するか、どんなデータがどこに流れるかは、すべてアーキテクチャ上の決定。
- **新メンバーのオンボーディングにアーキテクチャが必要。** なければ、接続方法を理解するためにすべてのファイルを読む必要がある。
- **アーキテクチャが機能を制約し、可能にする。** マルチテナントアーキテクチャはテナント分離を可能にするが、シングルテナントではできない。

## どう動くのか？

### 一般的なアーキテクチャパターン

| パターン | 説明 | 例 |
|----------|------|-----|
| **モノリス** | 単一デプロイ単位、すべてのコードが一体 | volta-auth-proxy（Phase 1） |
| **マイクロサービス** | 多数の小サービス、各自DBを持つ | Netflix、Uber |
| **モジュラーモノリス** | 単一デプロイだが内部でモジュール分離 | Shopify |
| **サーバーレス** | イベントで発火する関数、サーバー管理不要 | AWS Lambda |
| **イベント駆動** | コンポーネントがイベント/メッセージで通信 | Kafkaベースのシステム |

### アプリケーション内のレイヤー

ほとんどのWebアプリケーションは内部的にレイヤードアーキテクチャに従います：

```
  ┌─────────────────────────────────────┐
  │          HTTPレイヤー                │  ← ルート、リクエスト解析
  │     (Javalinハンドラー)              │
  ├─────────────────────────────────────┤
  │        サービスレイヤー              │  ← ビジネスロジック
  │   (AuthService, TenantService)      │
  ├─────────────────────────────────────┤
  │       リポジトリレイヤー             │  ← データベースアクセス
  │   (SQLクエリ、HikariCP)             │
  ├─────────────────────────────────────┤
  │        データベース                  │  ← PostgreSQL
  └─────────────────────────────────────┘
```

各レイヤーは直下のレイヤーとのみ通信します。HTTPレイヤーはデータベースに直接触れません -- サービスレイヤーを呼び、サービスレイヤーがリポジトリレイヤーを呼びます。

### アーキテクチャの決定はトレードオフ

すべてのアーキテクチャ選択にはトレードオフがあります。「最良の」アーキテクチャはありません -- あるのは制約に対して最良のアーキテクチャだけです：

```
  モノリス                          マイクロサービス
  ┌────────────────┐               ┌────┐ ┌────┐ ┌────┐
  │  すべてが       │               │認証│ │ユーザ│ │課金│
  │  1プロセスに    │               │    │ │    │ │    │
  └────────────────┘               └────┘ └────┘ └────┘

  + デプロイが簡単                  + 独立してスケール
  + デバッグが簡単                  + チームの自律性
  + ネットワークホップなし           + 障害の分離
  - 一体としてスケール              - ネットワークの複雑さ
  - 1つのバグで全体がクラッシュ     - 分散デバッグ
  - 大規模でチームのボトルネック     - 運用オーバーヘッド
```

## volta-auth-proxy ではどう使われている？

### voltaのアーキテクチャ（Phase 1）

volta-auth-proxyは**モジュラーモノリス**パターンに従います：明確な内部分離を持つ単一のデプロイ可能なJARです。

```
  ┌──────────────────────────────────────────────────────┐
  │                   インターネット                      │
  └───────────────────────┬──────────────────────────────┘
                          │
                          ▼
  ┌──────────────────────────────────────────────────────┐
  │                    Traefik                           │
  │            (リバースプロキシ + TLS)                   │
  │                                                      │
  │   ForwardAuth ──┐        ルーティング ──┐            │
  └─────────────────┼───────────────────────┼───────────┘
                    │                       │
                    ▼                       ▼
  ┌──────────────────────┐   ┌─────────────────────┐
  │  volta-auth-proxy     │   │  あなたのアプリ       │
  │                        │   │  (X-Volta-*ヘッダー  │
  │  ┌──────────────────┐ │   │   を読む)            │
  │  │ OIDCログインフロー │ │   └─────────────────────┘
  │  │ セッション管理    │ │
  │  │ JWT発行           │ │
  │  │ テナント解決      │ │
  │  │ RBAC強制          │ │
  │  │ 招待フロー        │ │
  │  │ Internal API      │ │
  │  └──────────────────┘ │
  │           │            │
  │           ▼            │
  │  ┌──────────────────┐ │
  │  │   PostgreSQL      │ │
  │  │   (9テーブル)      │ │
  │  └──────────────────┘ │
  │                        │
  │  ┌──────────────────┐ │
  │  │ Caffeineキャッシュ │ │
  │  │ (インメモリ)      │ │
  │  └──────────────────┘ │
  └──────────────────────┘
```

### voltaの主要なアーキテクチャ決定

| 決定 | 選択 | 根拠 |
|------|------|------|
| フレームワーク | Javalin（軽量） | フルコントロール、魔法なし、最小依存 |
| 認証アプローチ | 自前構築、ライブラリ不使用 | すべての行を理解；[設定地獄](config-hell.ja.md)を回避 |
| テンプレートエンジン | [jte](jte.ja.md) | 型安全、高速、シンプル |
| データベース | PostgreSQL | 信頼性、広いサポート |
| キャッシュ | [Caffeine](caffeine-cache.ja.md) | プロセス内、外部依存なし |
| トークン形式 | [JWT RS256](rs256.ja.md) | 非対称 -- アプリはシークレットなしで検証可能 |
| デプロイ | [シングルプロセス](single-process.ja.md) | Phase 1ではシンプルな運用 |
| マルチテナンシー | サブドメインによる[テナント解決](tenant-resolution.ja.md) | URLレベルでの明確な分離 |

```
  Phase 1 (現在):                 Phase 2 (計画中):
  ┌───────────┐                   ┌───────────────┐
  │ volta (1) │                   │ ロードバランサ  │

```

## よくある間違いと攻撃

### 間違い1：時期尚早なマイクロサービス化

必要になる前にマイクロサービスに分割すると、膨大な複雑さが追加されます -- ネットワーク呼び出し、サービスディスカバリ、分散トレーシング、結果整合性。モノリスから始め、痛みが現実になったら分割しましょう。voltaは意図的にシングルプロセスから始めています。

### 間違い2：レイヤー間の明確な境界がない

HTTPハンドラーが直接SQLクエリを実行すると、ビジネスロジックの独立テスト、データベースの交換、キャッシュレイヤーのクリーンな追加ができなくなります。

### 間違い3：データフローを無視

箱は描くがデータフローを描かないアーキテクチャ図は、最も重要なセキュリティ上の質問を隠します：「ユーザーのパスワードはどこを流れる？どのコンポーネントが秘密鍵を持つ？あるアプリが別のテナントのデータを読めるか？」

### 間違い4：アーキテクチャ宇宙飛行

まだ存在しない問題のためにアーキテクチャを過度に設計すること。[フェーズベースのアプローチ](phase-based-development.ja.md) -- シンプルに始めて必要に応じて進化する -- がより実践的です。
