# Auth0

[English version](auth0.md)

---

## これは何？

Auth0は、ログイン、サインアップ、ユーザー管理をゼロから構築せずにアプリに追加できる、クラウドホスト型の認証・認可サービス（Identity-as-a-Service）です。

オフィスビルに警備会社を雇うようなものです。ガードマン、IDバッジシステム、カメラ、アクセス制御を提供してくれます。自分で構築する必要はありません。ただし月額料金がかかり、ガードマンはあなたのルールではなく警備会社のルールに従い、別の警備会社に変えたい場合はシステム全体を引き剥がして交換しなければなりません。

---

## なぜ重要なのか？

Auth0は最も人気のある[IDaaS](idaas.ja.md)プロバイダの一つです。2013年に設立され、2021年にOktaに65億ドルで買収されました。スタートアップから大企業まで何千もの企業が使っています。Webアプリを作るなら、誰かが「Auth0を使えばいいじゃん」と言うでしょう。

Auth0を理解すると、クラウドホスト型認証と[セルフホスティング](self-hosting.ja.md)のトレードオフが分かり、なぜvolta-auth-proxyが存在するのかが理解できます。

---

## Auth0がうまくやっていること

公平に言いましょう -- Auth0が人気になったのには正当な理由があります：

| 強み | 詳細 |
|------|------|
| **素早い立ち上げ** | 1時間以内に新しいアプリにログインを追加できる。クイックスタートガイドが優秀。 |
| **あらゆるSDK** | JavaScript、React、Next.js、iOS、Android、Flutter、Python、Go、Java、.NET -- ほとんどのプラットフォーム向けのSDKがある。 |
| **ソーシャルログイン** | Google、GitHub、Facebook、Apple -- ダッシュボードでトグルするだけ。 |
| **エンタープライズ機能** | SAML、Active Directory、LDAP接続で企業顧客に対応。 |
| **Universal Login** | すべての認証フローを処理するホスト型ログインページ。ユーザーをそこにリダイレクトし、ログインさせて戻す。 |
| **ダッシュボード** | Webベースの管理UI。アプリケーション作成、ルール設定、ログ閲覧。コマンドライン不要。 |

2-3人のチームがMVPを構築する場合、Auth0は数週間の作業を節約できます。これは本当に価値があります。

## 問題点

### 1. スケール時のコスト

Auth0は[MAU](mau.ja.md)（月間アクティブユーザー）で課金します。無料枠は約7,500MAUまで。その先：

```
  Auth0のコスト成長：

  1年目:  5,000 MAU   →  無料        ← 「Auth0最高！」
  2年目:  25,000 MAU  →  ~$575/月    ← 「うーん、コストが...」
  3年目:  100,000 MAU →  ~$2,400/月  ← 「え、ログインだけで年$28,800!?」
  4年目:  500,000 MAU →  個別見積り  ← 「$$$$、しかも逃げられない」
```

スケール時には、Auth0がインフラコストのトップ3に入ることがあります -- 「ただの」ログインシステムなのに。

### 2. ベンダーロックイン

全体像は[vendor-lock-in.ja.md](vendor-lock-in.ja.md)を参照。Auth0に特有のポイント：

- **ユーザーデータベース：** パスワードハッシュをエクスポートできない。Auth0を離れると全ユーザーがパスワードリセット必須。
- **Auth0 Actions：** ログインフロー中に動くサーバーサイドJavaScript。Auth0内でしか動かない。移植性ゼロ。
- **Auth0 Organizations：** マルチテナンシー機能。彼らのAPIに深く紐づく。他に同等のものがない。
- **Auth0 SDK：** フロントエンドが`@auth0/auth0-spa-js`に依存。除去するにはすべてのページで認証を書き直す必要がある。

### 3. 限定的なカスタマイズ

Auth0のUniversal Loginはカスタマイズできますが、限界があります。彼らのUIフレームワーク内での作業です。ログインページのピクセル単位のコントロールが欲しい場合、またはAuth0のメンタルモデルに合わないログインフローが必要な場合、壁にぶつかります。

### 4. Okta買収後の懸念

Okta買収以降、一部の開発者から以下が報告されています：
- 値上げ
- 機能の廃止
- サポート品質の変化
- 製品ロードマップの不確実性

## voltaがAuth0を使わないことを選んだ理由

volta-auth-proxyはAuth0（および類似サービス）が特定の要件を満たさないために存在します：

| 要件 | Auth0 | volta |
|------|-------|-------|
| セルフホスト可能 | 不可 | 可（専用） |
| 100k MAUで$0 | 不可（~$2,400/月） | 可 |
| 完全なUIコントロール | 限定的 | 完全（[jte](jte.ja.md)テンプレート） |
| SDK依存なし | Auth0 SDK必須 | 標準HTTPヘッダー |
| データ所有権 | Auth0のクラウド | あなたのPostgreSQL |
| 設計上のマルチテナント | Organizations（有料機能） | コアアーキテクチャ |
| 起動時間 | N/A（クラウド） | ~200ms |
| メモリ使用量 | N/A（クラウド） | ~30MB |

voltaは「アンチAuth0」ではありません。Auth0は運用負荷ゼロで予算があるチームには良い製品です。voltaは完全なコントロール、予測可能なコスト、ロックインなしを求めるチーム向けです。

---

## Auth0が適している場合

トレードオフに正直になりましょう。以下の場合はAuth0が正しい選択かもしれません：

- MVPを構築中の小さなチームで、1日で認証が必要
- インフラを管理したい人がいない
- MAUベースの料金を長期的に吸収できる予算がある
- エンタープライズSSO接続（SAML/LDAP）がすぐに必要で、自分で構築したくない
- ロックインのトレードオフを受け入れられる

---

## voltaがより適している場合

以下の場合はvoltaのほうが適しています：

- マルチテナントSaaSを構築しており、認証をコアアーキテクチャの一部にしたい
- 10,000+ MAUを超える成長を見込んでおり、予測可能なコストが欲しい
- ログイン体験の完全なコントロールが必要
- データ所在地やコンプライアンスの要件でセルフホスティングが必要
- 初日からベンダーロックインを避けたい
