# 認証 vs. 認可（Authentication vs. Authorization）

[English version](authentication-vs-authorization.md)

---

## 一言で言うと？

認証は**あなたが誰か**を証明すること（IDカードを見せるように）、認可は**あなたが何をしてよいか**を決めること（どのドアをキーカードで開けられるか）です。

---

## テック業界で最も紛らわしいペア

この2つの言葉はほとんど同じに聞こえ、よく「authn」（認証）と「authz」（認可）と略されるので、さらにややこしくなります。でもこれは全く違う2つの質問です：

| | 認証（AuthN） | 認可（AuthZ） |
|---|---|---|
| **質問** | 「あなたは誰？」 | 「何ができる？」 |
| **現実世界** | 空港でパスポートを見せる | 搭乗券に「101便に搭乗可」と書いてある |
| **結果** | 「あなたは山田太郎です」 | 「太郎はプロジェクトを編集できるがユーザーは削除できない」 |
| **失敗時** | 401 Unauthorized（「あなたが誰か分かりません」） | 403 Forbidden（「あなたが誰かは分かるけど、それはできません」） |

---

## IDカード vs キーカードのたとえ

セキュリティの厳しいオフィスビルがある大企業で働いていると想像してください。

**認証 = 写真入りIDバッジ**
ドアをくぐるとき、警備員が写真入りIDを確認します。あなたは本当に山田太郎ですか？顔と写真は一致していますか？警備員はあなたが何をしてよいかは確認しません。あなたが名乗る通りの人物かどうかだけです。IDがなければ、そもそも入れません。

**認可 = キーカードのアクセスレベル**
中に入ると、ドアにキーカードをかざして開けます。キーカードにはこう書いてあります：
- 3階（あなたのチームのフロア）：許可
- 5階（役員フロア）：拒否
- サーバールーム：拒否
- 休憩室：許可

ビルはあなたが誰かを知っています（入り口で認証済み）。そして今、あなたが何にアクセスできるかを確認します（各ドアで認可が行われる）。

**重要なポイント：** 認証は常に最初に来ます。誰かが何をしてよいかは、その人が誰か分かるまで確認できません。

## 日常生活の例

| 場面 | 認証 | 認可 |
|---|---|---|
| Netflix | メールとパスワードを入力 | プランによって4Kで見られるか決まる |
| ホテル | フロントでIDを見せてチェックイン | ルームキーは自分の部屋だけ開く |
| コンサート | ゲートでチケットを見せる | VIPチケットならバックステージに行けるが、一般チケットは不可 |
| ATM | カードを入れて暗証番号を入力 | 口座によって引き出し限度額が決まる |

## voltaでの認証の仕組み

voltaは、あなたが誰かを証明する方法をいくつかサポートしています：

1. **Googleログイン（OIDC）** -- 「Googleでサインイン」をクリック。Googleがあなたの身元を確認。voltaはGoogleの回答を信頼する。
2. **SAML（企業向け）** -- 会社のIDプロバイダー（OktaやAzure ADなど）があなたが誰かを確認する。
3. **セッションCookie** -- 一度ログインすると、voltaがブラウザにCookieを渡すので、毎回ログインする必要がない。

認証が成功すると、voltaは「この人はtaro@acme.comだ」と分かります。

認証が失敗すると、voltaは**401 Unauthorized**を返します。「あなたが誰か分かりません。ログインしてください。」

---

## voltaでの認可の仕組み

あなたが誰か分かったら、voltaは何をしてよいかを確認します：

1. **テナントメンバーシップ** -- アクセスしようとしているテナント（ワークスペース）のメンバーか？メンバーでなければ、そのテナント内のものには何もアクセスできない。
2. **ロール確認** -- このテナントでの役割は？voltaには4つのレベルがある：OWNER > ADMIN > MEMBER > VIEWER。
3. **アプリアクセス** -- 各アプリがアクセス可能なロールを定義。例えば、管理パネルはADMINまたはOWNERを要求するかもしれない。

```
  ユーザー：taro@acme.comがadmin.example.comにアクセスしようとする

  ステップ1 - 認証：
  volta：「セッションCookie発見。あなたはtaro@acme.comです」 ✓

  ステップ2 - 認可：
  volta：「ACME Corpでのあなたのロールを確認中...」
  volta：「あなたはMEMBERです」
  volta：「admin.example.comにはADMINまたはOWNERが必要です」
  volta：「403 Forbidden。認証はされていますが、認可されていません」 ✗
```

認可が失敗すると、voltaは**403 Forbidden**を返します。「あなたが誰かは分かっていますが、この操作の権限がありません。」

## 簡単な例

```
  太郎（ACME CorpのMEMBER）が2つのアプリにアクセスしようとする：

  ┌─────────────────────────────────────────────────────┐
  │ wiki.example.com                                     │
  │ allowed_roles: [MEMBER, ADMIN, OWNER]                │
  │                                                      │
  │ 認証：✓（太郎はログイン済み）                          │
  │ 認可：✓（MEMBERは許可リストに含まれる）                │
  │ 結果：200 OK -- 太郎はwikiを使える                    │
  └─────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────┐
  │ admin.example.com                                    │
  │ allowed_roles: [ADMIN, OWNER]                        │
  │                                                      │
  │ 認証：✓（太郎はログイン済み）                          │
  │ 認可：✗（MEMBERはリストに含まれない）                  │
  │ 結果：403 Forbidden -- 太郎は管理画面にアクセスできない│
  └─────────────────────────────────────────────────────┘
```

## よくある間違い

初心者が最もよくやる間違いは、401と403を混同することです：

- **401 Unauthorized**は実際には「認証されていない」という意味。サーバーはあなたが誰か分からない。解決策：ログインする。
- **403 Forbidden**は「認可されていない」という意味。サーバーはあなたが誰か知っているが、権限がない。解決策：管理者に適切なロールを付与してもらう。

はい、「401 Unauthorized」という名前は「認可されていない」と聞こえるので紛らわしいです。これはHTTP仕様の歴史的な命名ミスとして広く認識されています。覚え方：401 = 「あなたは誰？」、403 = 「ダメです」。
