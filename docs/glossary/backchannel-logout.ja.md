# Backchannel Logout（バックチャネルログアウト）

## これは何？

バックチャネルログアウトとは、ユーザーがログアウトしたとき、認証プロバイダ（IdP）が全てのアプリケーション（リライングパーティ）にサーバー間で通知を送る仕組みです。「バックチャネル」とは、通知がユーザーのブラウザを経由せず、サーバー間で直接送られることを意味します。

これと対照的な「フロントチャネル」ログアウトでは、ユーザーのブラウザが各アプリケーションのログアウト URL に順番にリダイレクトされます。バックチャネルの方が信頼性が高いです。ブラウザに依存しないため、ユーザーがブラウザを閉じたりネットワーク接続を失っても、ログアウト通知はアプリケーションに届きます。

```
  フロントチャネルログアウト:         バックチャネルログアウト:

  ユーザーが IdP で "ログアウト"     ユーザーが IdP で "ログアウト"
  をクリック                        をクリック

  ブラウザ:                         IdP (サーバーサイド):
  1. App A /logout にリダイレクト    1. App A /backchannel-logout に POST
  2. App B /logout にリダイレクト    2. App B /backchannel-logout に POST
  3. App C /logout にリダイレクト    3. App C /backchannel-logout に POST
  4. "ログアウトしました" 表示       4. "ログアウトしました" 表示

  問題: ステップ 2 でブラウザが       ブラウザが何をしても
  閉じると、App C は知らない         全アプリに通知される
```

バックチャネルログアウトの通知は通常、「Logout Token」と呼ばれる署名済み JWT で、IdP が各アプリケーションの事前登録された URL に HTTP POST で送信します。

## なぜ重要？

SSO（シングルサインオン）環境では、ユーザーは複数のアプリケーションにセッションを持っています。1 つのアプリケーション（または IdP）からログアウトしたとき、全てのアプリケーションからログアウトされることが期待されます。ログアウトを伝搬する仕組みがないと、他のアプリケーションでのユーザーセッションはアクティブなままです。

これはセキュリティリスクを生みます:

- **共有コンピュータ。** ユーザーが IdP からログアウトしても、App B のセッションはまだアクティブ。次にコンピュータを使う人が前のユーザーとして App B にアクセスできてしまう。
- **侵害されたアカウント。** 管理者が IdP でユーザーのアカウントを無効にしても、各アプリでの既存セッションは期限切れになるまで機能し続ける。
- **コンプライアンス。** 一部の規制では、ユーザーのアクセスが取り消されたとき、全てのシステムで即座に効力を発することを要求している。

## どう動くの？

OIDC バックチャネルログアウト仕様（OpenID Connect Back-Channel Logout 1.0）がプロトコルを定義しています:

```
  ユーザー       IdP              App A            App B
  |             |                |                |
  | ログアウト   |                |                |
  |------------>|                |                |
  |             |                |                |
  |             | POST /backchannel-logout        |
  |             | Body: logout_token (JWT)        |
  |             |--------------->|                |
  |             |                |                |
  |             | POST /backchannel-logout        |
  |             | Body: logout_token (JWT)        |
  |             |------------------------------->|
  |             |                |                |
  |             |                | ユーザーの      |
  |             |                | セッションを    |
  |             |                | 無効化          |
  |             |                |                |
  |             |                |    ユーザーの   |
  |             |                |    セッションを |
  |             |                |    無効化       |
  |             |                |                |
  | "ログアウト  |                |                |
  |  しました"   |                |                |
  |<------------|                |                |
```

Logout Token は以下を含む JWT です:

```json
{
  "iss": "https://idp.example.com",
  "sub": "user-123",
  "aud": "app-a-client-id",
  "iat": 1711900000,
  "jti": "unique-logout-event-id",
  "events": {
    "http://schemas.openid.net/event/backchannel-logout": {}
  },
  "sid": "session-id-at-idp"
}
```

アプリケーションがすべきこと:
1. IdP の JWKS を使って JWT 署名を検証する。
2. 発行者と対象者が有効か確認する。
3. `sub`（ユーザー ID）または `sid`（セッション ID）に関連付けられたローカルセッションを検索する。
4. そのセッションを無効化する。
5. ログアウトの確認として HTTP 200 を返す。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は現在バックチャネルログアウトを実装して**いません**。その理由と、なぜ（まだ）不要なのかを説明します:

**短命の JWT。** volta は TTL 5 分の JWT を発行します（デフォルト `JWT_TTL_SECONDS=300`）。ユーザーのアクセスを取り消すべき場合でも、既存の JWT は 5 分以内に無効になります。アクセスを付与し続ける長命のトークンは存在しません。

**即時失効可能なサーバーサイドセッション。** volta のセッションは PostgreSQL に保存されています。ユーザーがログアウトしたり、管理者がアクセスを取り消すと:

```
  ユーザーがログアウト              管理者がアクセスを取り消す
  |                               |
  v                               v
  store.revokeSession(sessionId)  store.deactivateMembership(userId, tenantId)
  |                               |
  v                               v
  Session.invalidatedAt = now     membership.active = false
  |                               |
  v                               v
  次にこの Cookie でリクエスト:     次にこのユーザーでリクエスト:
  セッションが拒否される            メンバーシップチェックが失敗
  再認証が必要                     再認証が必要
```

全てのリクエストが volta の `/auth/verify` エンドポイント（ForwardAuth）を通るため、毎回のリクエストでセッションがデータベースと照合されます。失効はまさに次のリクエストで即座に効力を発します。

**シングルゲートウェイアーキテクチャ。** volta は全ての下流アプリの唯一の認証ゲートウェイです。各アプリが IdP と独自のセッションを持つ従来の SSO セットアップと異なり、volta のアーキテクチャでは:

```
  従来の SSO:                       volta のアーキテクチャ:

  IdP セッション --> App A セッション  volta セッション --> 全アプリ
                --> App B セッション  （リクエストごとにチェック）
                --> App C セッション
                                     アプリごとに無効化する
  App A, B, C のセッションを          セッションがない
  無効化するためにバックチャネル        volta のセッションを失効すれば
  ログアウトが必要                    即座に全アプリに影響
```

下流のアプリは volta との独自のセッションを維持しないため（ForwardAuth 経由でリクエストごとに ID ヘッダを受け取る）、下流アプリレベルで「ログアウト」するものがありません。

### volta にバックチャネルログアウトが必要になるのはいつ？

volta がバックチャネルログアウトの実装を必要とするシナリオ:

**1. 下流アプリが JWT をキャッシュする場合。**
下流のアプリが `X-Volta-JWT` で受け取った JWT をキャッシュし（volta に再確認せずに複数のリクエストに使う）、そのキャッシュ済み JWT は期限切れまで有効です。バックチャネルログアウトで、キャッシュ済み JWT を破棄するようアプリに通知できます。

**2. 上流 IdP のログアウトを伴うマルチ IdP 環境。**
フェーズ 2-3 で volta が複数の IdP（Google、Azure AD、Okta）をサポートするとき、それらの IdP が volta にバックチャネルログアウト通知を送る可能性があります。ユーザーが Azure AD からログアウトすると、Azure AD が volta にログアウトトークンを POST し、volta がユーザーのセッションを無効化すべきです。volta は送信者ではなく、バックチャネルログアウトの**受信者**になります。

**3. 分散デプロイメント。**
volta が独立したセッションストアを持つ複数のインスタンスとしてデプロイされた場合（現在の設計とは異なる）、あるインスタンスがセッションを失効させても他のインスタンスには影響しません。バックチャネルログアウト（または Redis のような共有セッションストア）が必要になります。

## よくある間違い

**1. IdP でのログアウトが全てをログアウトすると思い込む。**
バックチャネルログアウト（または同等のメカニズム）がないと、IdP からのログアウトは IdP セッションを終了するだけです。アプリケーションのセッションは存続する可能性があります。volta はリクエストごとにセッションをチェックすることでこれを回避しています。

**2. ログアウトトークンを検証しない。**
ログアウトトークンは他のトークンと同様に検証すべき JWT です: 署名、発行者、対象者、`events` claim にバックチャネルログアウトイベントが含まれることを確認します。未検証のログアウトトークンは、攻撃者が任意のユーザーをログアウトさせるために使われる可能性があります。

**3. フロントチャネルログアウトだけに頼る。**
フロントチャネルログアウトは、ユーザーのブラウザが各アプリケーションの隠し iframe を読み込むことに依存します。ブラウザが閉じられた場合、JavaScript がブロックされた場合、アプリケーションが多すぎる場合に失敗します。バックチャネルの方が信頼性が高いです。

**4. バックチャネル呼び出しにタイムアウトを設定しない。**
あるアプリケーションがログアウト通知への応答が遅い場合、他の全てのアプリケーションのログアウトフローを遅延させるべきではありません。短いタイムアウトとファイアアンドフォーゲットの方式を使いましょう。

**5. バックチャネルログアウトが常に必要だと思い込む。**
アーキテクチャがリクエストごとにセッションをチェックしている場合（volta の ForwardAuth パターンのように）、セッション失効は即座に反映され、バックチャネルログアウトは不必要な複雑さを追加するだけです。アーキテクチャが既に必要な保証を提供しているか評価しましょう。

**6. 部分的な失敗を処理しない。**
バックチャネルログアウトフローでは、一部のアプリケーションがダウンしているか到達できない場合があります。IdP は到達できるアプリケーションについてはログアウトを完了し、失敗をリトライや調査のためにログに記録すべきです。
