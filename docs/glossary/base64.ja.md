# Base64

[English version](base64.md)

---

## これは何？

Base64は、バイナリデータ（生のバイト列）を64種類の安全な文字（A-Z、a-z、0-9、`+`、`/`、パディング用の`=`）のみを使ったテキスト文字列に変換するエンコーディング方式です。暗号化ではありません。バイナリデータをURL、HTTP[ヘッダー](header.md)、JSON、その他のテキストベースプロトコルで安全に送信できるテキスト形式で表現するだけです。

モールス信号のようなものです。モールス信号は文字を点と線に変換して電信線で送れるようにします。メッセージは秘密ではありません。モールス信号を知っている人なら誰でも読めます。同様に、Base64はバイナリデータをテキスト文字に変換して、テキストしか理解しないシステムを通過できるようにします。転送用のエンコーディングであり、セキュリティ手段ではありません。

volta-auth-proxyでは、Base64（特にURLセーフなバリアントであるbase64url）が[JWT](jwt.md)の各部分、[招待コード](invitation-code.md)、URLやHTTPヘッダーに含める必要がある他のバイナリ値のエンコーディングに使われます。

---

## なぜ重要なのか？

多くのプロトコルやデータ形式はテキストのみをサポートします：

```
  問題：テキスト専用コンテキストでのバイナリデータ

  生のバイナリ：[0x7A, 0x3F, 0xB2, 0x00, 0xFF, 0x91]
                                    ↑
                                  NULLバイト！
                          （C文字列、HTTPヘッダー、
                           JSON値、URLを壊す...）

  Base64：     "ej+yAP+R"
              （安全なASCIIテキスト、どこでも動く）
```

Base64がないと：

- JWTトークンをHTTPヘッダーに入れられない（バイナリデータがヘッダー形式を壊す）
- 招待コードをURLに入れられない（特殊文字が誤解される）
- 暗号ハッシュをJSONに保存できない（NULLバイトが文字列を切り詰める）

## どう動くのか？

### エンコーディングプロセス

Base64は3バイト（24ビット）を4文字（各6ビット）に変換します：

```
  入力：3バイト = 24ビット
  ┌────────────────────────────────────────┐
  │ バイト1: 01001101  (M = 77)            │
  │ バイト2: 01100001  (a = 97)            │
  │ バイト3: 01101110  (n = 110)           │
  └────────────────────────────────────────┘

  6ビットチャンクに再グループ化：
  ┌────────┬────────┬────────┬────────┐
  │ 010011 │ 010110 │ 000101 │ 101110 │
  │  = 19  │  = 22  │  = 5   │  = 46  │
  └────────┴────────┴────────┴────────┘

  Base64アルファベットで検索：
  ┌─────┬─────┬─────┬─────┐
  │  T  │  W  │  F  │  u  │
  └─────┴─────┴─────┴─────┘

  "Man" → "TWFu"
```

### Base64アルファベット

```
  インデックス：文字
  0-25:  A-Z
  26-51: a-z
  52-61: 0-9
  62:    +
  63:    /
  パディング：=
```

### パディング

入力の長さが3の倍数でない場合、パディング（`=`）が追加されます：

```
  "M"   (1バイト)  → "TQ=="    (パディング2つ)
  "Ma"  (2バイト)  → "TWE="    (パディング1つ)
  "Man" (3バイト)  → "TWFu"    (パディングなし)
```

### Base64 vs base64url

標準のBase64はURLで特別な意味を持つ`+`と`/`を使います。Base64urlはこれらを置換します：

```
  標準Base64:  +  /  =
  Base64url:   -  _  （パディングなし）

  例：
  標準：    "ej+yAP+R/w=="
  URLセーフ："ej-yAP-R_w"
```

voltaはURLやHTTPコンテキストに現れるすべてのものにbase64url（RFC 4648 Section 5）を使用します。

### サイズ増加

Base64エンコーディングはサイズを約33%増加させます：

```
  元のサイズ：32バイト
  Base64：    ceil(32 / 3) * 4 = 44文字
  Base64url:  43文字（パディングなし）
```

## volta-auth-proxyではどう使われている？

### JWT構造

[JWT](jwt.md)はドットで区切られた3つのbase64urlエンコードされた部分で構成されます：

```
  JWT = base64url(header) + "." + base64url(payload) + "." + base64url(signature)

  例：
  eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2VyLXV1aWQiLCJ2b2x0YV90aWQiOiJhY21lIn0.signature...
  ├─── header ────────┤ ├──────────── payload ────────────────────────────────┤ ├── sig ──┤

  デコードされたヘッダー：{"alg":"RS256"}
  デコードされたペイロード：{"sub":"user-uuid","volta_tid":"acme"}
```

### 招待コード

[招待コード](invitation-code.md)は32バイトのランダムデータをbase64urlでエンコードしたものです：

```java
byte[] random = new byte[32];
SecureRandom.getInstanceStrong().nextBytes(random);
String code = Base64.getUrlEncoder().withoutPadding().encodeToString(random);
// → "ej-ykYH3mQ_7vKxN2bFqLpR8s0T4uWdZ9aIcBvExHJM"
```

コードが招待URLに現れるためbase64urlが使われます：

```
https://app.acme.com/invite?code=ej-ykYH3mQ_7vKxN2bFqLpR8s0T4uWdZ9aIcBvExHJM
```

### Cookie値

[Cookie](cookie.md)に保存される[セッション](session.md)トークンは、HTTP `Set-Cookie`ヘッダーで安全になるようbase64urlエンコードされます。

### HMAC署名

[Webhook](webhook.md)のHMAC署名はBase64ではなく16進エンコーディングが多いですが、Webhookの[ペイロード](payload.md)自体にBase64エンコードされたバイナリフィールドが含まれることがあります。

## よくある間違いと攻撃

### 間違い1：Base64が暗号化だと思う

Base64はエンコーディングであり、暗号化ではありません。誰でも即座にデコードできます：

```
  "Secret"のBase64："U2VjcmV0"
  デコード：echo "U2VjcmV0" | base64 -d → "Secret"

  Base64で機密データを「隠す」のは絶対にやめてください。
  秘密にするには実際の暗号化（AES、RSA）を使ってください。
```

### 間違い2：URLで標準Base64を使う

標準Base64はURLで特別な意味を持つ`+`、`/`、`=`を使います。URLコンテキストでは常にbase64urlを使ってください。voltaはこれを徹底しています。

### 間違い3：二重エンコーディング

データを2回エンコードすると膨らんで混乱する値になります。境界（テキストコンテキストに入る時）で1回エンコードし、もう一方の境界で1回デコードしてください。

### 間違い4：パディングを正しく処理しない

`=`パディングを追加するシステムとしないシステムがあります。voltaはbase64urlに`withoutPadding()`を使用します。デコード時はパディングありなし両方を受け入れてください。

### 攻撃：JWT[ペイロード](payload.md)改ざん

攻撃者がJWTのbase64urlペイロードをデコードし、変更し（例：ロールを"ADMIN"に変更）、再エンコードして、署名チェックがスキップされることを期待する。防御：voltaは常にRS256署名を検証。[ホワイトリスト](whitelist.md)がRS256のみの受け入れを保証。

### 攻撃：エンコーディング混乱

攻撃者が混合エンコーディング（例：Base64の中にURLエンコーディング、その中にBase64）のデータを送って入力検証を迂回する。防御：正確に1回デコードし、デコード結果を検証。
