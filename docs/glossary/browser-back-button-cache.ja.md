# ブラウザの「戻る」ボタンとキャッシュ

[English / 英語](browser-back-button-cache.md)

---

## これは何？

ブラウザの「戻る」ボタンを押すと、サーバーから再取得する代わりに前のページの保存済みスナップショットが表示されることがあります。これは back-forward cache（bfcache）と呼ばれます。ナビゲーションを瞬時に感じさせるための仕組みですが、セキュリティ上の問題があります。表示されるページが古い可能性があり、認証ページでは既に終了したログインセッションが表示されるかもしれません。

---

## なぜ重要？

図書館の共有パソコンでこんなシナリオを想像してください：

1. Alice がアプリにログインし、個人データのあるダッシュボードを表示
2. Alice が「ログアウト」をクリック。サーバー側のセッションは破棄される
3. Alice がタブを閉じて離席
4. Bob がブラウザを開き、「戻る」ボタンを数回押す
5. Bob に Alice のキャッシュされたダッシュボードが表示される -- 名前、メール、データすべて

Alice は正しいことをしました。ログアウトしました。しかしブラウザの bfcache が古いスナップショットを表示してしまいました。これは仮定の話ではなく、共有パソコンや公共パソコンでよく知られた攻撃経路です。

---

## 簡単な例

保護なしの場合：
```
ログイン     ->  ダッシュボード（bfcache にキャッシュされる）
ログアウト   ->  ログインページ
戻るを押す   ->  bfcache からダッシュボードが表示される！（古いが見える）
```

`Cache-Control: no-store` ありの場合：
```
ログイン     ->  ダッシュボード（キャッシュされない）
ログアウト   ->  ログインページ
戻るを押す   ->  ブラウザがサーバーに取得 -> セッションなし -> ログインにリダイレクト
```

ポイントは `no-store` です。このディレクティブはブラウザに bfcache にもページを保持しないよう指示します。

---

## volta-auth-proxy での使い方

volta はすべての認証済みページに `setNoStore()` を適用し、「戻る」ボタンのキャッシュ攻撃を防いでいます：

```java
private static void setNoStore(Context ctx) {
    ctx.header("Cache-Control", "no-store, no-cache, must-revalidate, private");
    ctx.header("Pragma", "no-cache");
}
```

これにより：

- ログアウト後に「戻る」を押すとログインページが表示される（古いダッシュボードではなく）
- セッション一覧ページ（`/sessions`）はキャッシュされないため、共有パソコンでユーザーを切り替えてもセッションデータが漏洩しない
- 招待承認ページもキャッシュされないため、古い招待状態が表示されない

volta はログアウト時にサーバー側のセッションも破棄し、セッション Cookie もクリアします。万一ブラウザがキャッシュページを表示しても、操作しようとすれば認証に失敗してログインにリダイレクトされます。
