# 今必要なものだけ作る

[English version](build-only-what-you-need.md)

---

ソフトウェアエンジニアリングには、こんな魅惑的なアイデアがあります：「十分に深く考え、十分に注意深く計画し、十分に包括的に設計すれば、一回目で完璧なシステムを作れる。」このアイデアは、失敗したプロジェクトを生み出してきた長く輝かしい歴史を持っています。

---

## 壮大な計画の罠

すべてのエンジニアがこのミーティングを経験したことがあります：

> 「何かを作る前に、すべてを考え尽くしておこう。OIDC、SAML、LDAP、MFA、SCIM、webhook、課金統合、i18n、モバイルSDK、ポリシーエンジンをサポートする必要がある。初日からこれらすべてを処理できるアーキテクチャを設計しよう。」

これは責任感がある、徹底的、プロフェッショナルに感じます。これはまた、プロジェクトが死ぬ方法でもあります。

次に起きること：

1. アーキテクチャが複雑になる。10の機能を同時に収容しなければならないから
2. 各機能が他のすべての機能に制約を加える（「でもSAMLはどうする？セッションモデルでそれを考慮しなければ」）
3. 開発が遅くなる。すべての決定が壮大な設計全体に波及効果を持つから
4. 6ヶ月後、何もエンドツーエンドで動かない。すべてがすべてに依存しているから
5. チームが誰も頼んでいない機能の仮想的なエッジケースについて議論する
6. モチベーションが低下。「いつ出荷できる？」
7. プロジェクトが中止されるか、10の機能のどれもまともに動かないバグだらけのまま出荷

何百万ドル規模のプロジェクトがこのパターンで破壊されるのを見てきました。問題はエンジニアが悪いことではありません -- 問題は多すぎる問題を一度に解こうとすることです。

---

## YAGNI：それは必要にならない

YAGNIはExtreme Programmingの略語で、こう述べています：実際に必要になるまで機能を追加するな。「必要になるかもしれないと思うまで」ではない。「顧客がそれとなく言及するまで」ではない。「競合が持っているまで」ではない。実際に、今、今日、必要になるまで。

YAGNIは怠惰であることではありません。正直であることです。知っていること（将来についてはごくわずか）と知らないこと（将来についてはほぼすべて）について正直であること。

考えてみてください：使われなかった機能をいくつ作りましたか？正直に言えば、答えは「たくさん」です。使われなかったすべての機能は、重要な機能に使えたはずの時間でした。

## voltaのアプローチ：Phase 1は最小限、将来のためのインターフェース

volta-auth-proxyは、フェーズベースのロードマップとインターフェースの使用を通じて「今必要なものだけ作る」哲学を体現しています。

### Phase 1が含むもの（それだけ）

Phase 1は、マルチテナントSaaSが初日に必要なものだけです：

- Google OIDCログイン（すべてのSaaSはGoogleログインから始まるから）
- セッション（ユーザーがログインしたままでいる必要があるから）
- マルチテナンシー（マルチテナントSaaSだから）
- ロール（全員が管理者であるべきではないから）
- 招待（チームがメンバーを追加する必要があるから）
- ForwardAuth（アプリがユーザーが誰か知る必要があるから）
- 内部API（アプリがユーザーとテナントを管理する必要があるから）

それだけです。SAMLなし。LDAPなし。SCIMなし。課金なし。モバイルSDKなし。

### Phase 1に含まれないもの（とその理由）

| 含まれない | 理由 |
|-----------|------|
| SAML SSO | まだエンタープライズ顧客がいない |
| LDAP/AD | まだFortune 500企業に販売していない |
| MFA/TOTP | GoogleログインのMFAはGoogleが処理 |
| SCIM | 顧客にまだ10,000人の従業員はいない |
| 課金 | 課金の前に認証が必要 |
| モバイルSDK | まだモバイルアプリを作っていない |
| ポリシーエンジン | RBACはOWNER > ADMIN > MEMBER > VIEWER |

これらのどれも初日には必要ありません。すべてが後のフェーズで計画されています。そしてアーキテクチャはそれらの追加をサポートします。voltaがそのために設計したからです：

```java
// Phase 1: インターフェースは存在する。Postgres実装のみ出荷。
interface SessionStore {
    void createSession(...);
    Optional<SessionRecord> findSession(...);
    void touchSession(...);
    void revokeSession(...);
    // ...
}

// Phase 1: この実装だけが存在
class PostgresSessionStore implements SessionStore { ... }

// Phase 2: Redis実装が認証ロジック変更ゼロで投入
class RedisSessionStore implements SessionStore { ... }
```

インターフェースはPhase 1で設計されました。Redis実装はPhase 2で構築されました。認証ロジックは一切変更されませんでした。これが核心的な洞察です：**将来を構築せずに、将来に備えることができる。**

## 動くものを出荷して、それから成長する

Phase 1について最も重要なことは、動くということです。「なんとなく動く」ではない。「デモでは動く」ではない。本番で動く。ユーザーはログインし、テナントを作成し、メンバーを招待し、ロールを管理し、アプリケーションを使用できる。システムは安全（PKCE、署名付きセッション、RS256 JWT、CSRF保護）。システムはテスト済み。

これが重要な理由：

1. **本当のフィードバックは本当の使用から来る。** Phase 1を本番で運用することでPhase 2に何が必要か発見する。Redisセッションは同時接続10,000ユーザーに達するまで不要かもしれない。SAMLは予想より早く必要になるかもしれない、主要顧客が要求するから。

2. **勢いが自信を生む。** Phase 1を出荷したチームはPhase 2も出荷できると信じる。6ヶ月間設計してきたチームは何も出荷されないと信じる。

3. **シンプルさはデバッグ可能。** Phase 1にバグがあるとき、探す場所は数箇所だけ。壮大な設計にバグがあるとき、相互接続された10のシステムのどこにでもある可能性がある。

4. **ユーザーはロードマップを気にしない。** 今日動くものを気にする。Phase 1は今日動く。

## ウォーターフォールの罠再訪

ウォーターフォールモデルは言います：すべての要件を収集し、すべてを設計し、すべてを構築し、すべてをテストし、デプロイ。橋を建設するときは機能します（要件が建設中に変わらないから）。ソフトウェアでは失敗します（要件が常に変わるから）。

フェーズベース開発は変装したウォーターフォールではありません。違いは：

```
  ウォーターフォール：
  [全要件を収集] → [すべてを設計] → [すべてを構築] → [出荷]
  ────────────────────────────────────────────────────────────►
  何かが動くまで12ヶ月以上

  フェーズベース：
  [Phase 1: コア要件] → [構築] → [出荷] → [学ぶ]
  ──────────────────────────────────────────────────►
  数週間。何かが動く。

      [Phase 2: スケールニーズ] → [構築] → [出荷] → [学ぶ]
      ──────────────────────────────────────────────────►
      数ヶ月後。もっと動く。

          [Phase 3: エンタープライズニーズ] → [構築] → [出荷]
          ──────────────────────────────────────────────►
          顧客が必要とするとき。その前ではなく。
```

各フェーズがフィードバックループを完了します。出荷し、学び、学んだことに基づいて次のフェーズを計画する。これは壮大な計画より*理論上は*遅い。実際にははるかに速い。壮大な計画が予定通りに出荷されることはないから。

## 実践的なアドバイス

システムを設計していて「すべてを考え尽くしたい」衝動を感じたら：

1. **必要だと思うすべてを書き出す。** 頭の中から出す。
2. **1週目にユーザーが必要とするものに丸をつける。** 容赦なく。
3. **それ以外はすべて「後で」リストへ。** 「絶対しない」ではなく、「後で」。
4. **「後で」リストの項目にインターフェースを設計する。** 実装コードを書かずにアーキテクチャを準備する。
5. **丸をつけた項目を出荷する。** 速く。動く状態で。テスト済みで。
6. **出荷後に「後で」リストを再訪する。** 多くの項目がもはや関連性がないことに驚くはず。

これがvoltaの作り方です。そしてうまくいっています。
