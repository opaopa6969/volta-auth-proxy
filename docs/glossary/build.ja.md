# ビルド

[English version](build.md)

---

## これは何？

ビルドとは、ソースコードを実行可能なアプリケーションに変えるまでの完全なプロセスです。コードの[コンパイル](compile.md)、テストの実行、依存関係の解決、すべてのパッケージングが含まれます。volta-auth-proxyの場合、ビルドは`.java`ファイル、`.jte`テンプレート、`pom.xml`の依存関係リストを受け取り、単一の実行可能な[fat JAR](fat-jar.md)ファイルを生成します。

IKEAの家具を組み立てるのに似ています。原材料（木のパネル = ソースコード）、ネジとボルト（依存関係 = [ライブラリ](library.md)）、組み立て説明書（ビルド設定 = pom.xml）があります。ビルドプロセスは説明書に従って切断、穴あけ、締結、テスト（引き出しはスムーズに開くか？）を行い、完成した使える製品ができるまで続けます。部品が合わなければ（コンパイルエラー）、引き出しが詰まれば（テスト失敗）、プロセスは停止します。

voltaのビルドコマンドは`mvn package`で、ソースコードからデプロイ可能なアーティファクトまでの完全なパイプラインを実行します。

---

## なぜ重要なのか？

ビルドプロセスがなければ：

- 数十のファイルを正しい順序で手動で[コンパイル](compile.md)することになる
- [ライブラリ](library.md)のJARファイルを手動でダウンロードして配置することになる
- デプロイ前にテストを実行し忘れる
- 異なる開発者が同じコードから異なる出力を生成する
- デプロイがエラーを起こしやすく、再現不可能になる

[Maven](maven.md)のようなビルドツールはプロセスを**再現可能**にします -- 同じソースコードは常に同じ出力を、どのマシンでも生成します。これは本番デプロイに不可欠です。

## どう動くのか？

### ビルドパイプライン

```
  ソースコード                 ビルドツール（Maven）           出力
  ───────────                 ──────────────────             ──────

  src/main/java/*.java   ──>  1. 依存関係を解決         ──>  target/
  src/main/jte/*.jte     ──>  2. Javaソースをコンパイル       volta-auth-proxy.jar
  src/main/resources/*   ──>  3. jteテンプレートをコンパイル   （単一fat JAR、
  src/test/java/*.java   ──>  4. ユニットテスト実行           約30MB、実行に
  pom.xml                ──>  5. 統合テスト実行               必要なすべてを
                               6. fat JARにパッケージ          含む）
```

### Mavenビルドフェーズ

Mavenはビルドを順次実行されるフェーズに整理します。各フェーズは以前のすべてのフェーズを含みます：

```
  mvn compile     ──>  1. compile（Java + jteテンプレート）
  mvn test        ──>  1. compile → 2. test
  mvn package     ──>  1. compile → 2. test → 3. package（JAR作成）
  mvn verify      ──>  1. compile → 2. test → 3. package → 4. verify
  mvn install     ──>  1〜4 → 5. ローカルリポジトリにインストール
```

voltaでは`mvn package`が標準のビルドコマンドです。デプロイ可能なfat JARを生成します。

### 各フェーズで何が起こるか

```
  フェーズ1: compile
  ┌─────────────────────────────────────────────────┐
  │  javacが.javaファイルを.classファイルにコンパイル  │
  │  jteプラグインが.jteテンプレートを.classにコンパイル│
  │  リソースがtarget/classes/にコピー               │
  │                                                   │
  │  いずれかのファイルに型エラー → ビルド失敗         │
  └─────────────────────────────────────────────────┘
           │
           ▼
  フェーズ2: test
  ┌─────────────────────────────────────────────────┐
  │  JUnitがsrc/test/java/のすべてのテストクラスを実行│
  │  テスト内容：認証ロジック、JWT作成、               │
  │  ロールチェック、セッション処理など                 │
  │                                                   │
  │  いずれかのテストが失敗 → ビルド失敗               │
  └─────────────────────────────────────────────────┘
           │
           ▼
  フェーズ3: package
  ┌─────────────────────────────────────────────────┐
  │  maven-shade-pluginがfat JARを作成:               │
  │  1. すべてのコンパイル済み.classファイルを取得     │
  │  2. すべての依存関係JARを展開                      │
  │  3. すべてを1つのJARに統合                        │
  │  4. MANIFEST.MFにMain-Classを設定                 │
  │                                                   │
  │  出力: target/volta-auth-proxy.jar                │
  └─────────────────────────────────────────────────┘
```

### ビルドアーティファクト

| アーティファクト | 場所 | 目的 |
|---------------|------|------|
| コンパイル済みクラス | `target/classes/` | 中間バイトコード |
| テスト結果 | `target/surefire-reports/` | テスト合否記録 |
| Fat JAR | `target/volta-auth-proxy.jar` | デプロイ可能なアプリケーション |
| 依存関係ツリー | `mvn dependency:tree`の出力 | 依存関係の監査 |

### 再現可能なビルド

同じソースコード + 同じpom.xml + 同じJDKバージョン = 同じ出力JAR。これは以下の理由です：

- pom.xmlが正確な依存関係バージョンを固定（`5.+`ではなく`5.1.0`）
- JDKバージョンが指定されている（Java 21）
- 外部状態がビルド出力に影響しない

## volta-auth-proxy ではどう使われている？

### 標準的なvoltaビルド

```bash
# フルビルド：コンパイル + テスト + パッケージ
mvn package

# コンパイルのみ（テストなし、パッケージングなし）
mvn compile

# パッケージングなしでテスト実行
mvn test

# テストをスキップ（控えめに -- テストには理由がある）
mvn package -DskipTests
```

### ビルド済みアプリケーションの実行

ビルド後、voltaは単一[プロセス](process.md)として実行されます：

```bash
# ビルド
mvn package

# 実行
java -jar target/volta-auth-proxy.jar
```

この1つのコマンドでアプリケーション全体が起動します：HTTPサーバー、データベース接続、セッション管理、JWTサービス、すべて。

### DockerでのStビルド

voltaの[Docker](docker.md)ビルドはマルチステージDockerfileを使用します：

```dockerfile
# ステージ1: ビルド
FROM maven:3.9-eclipse-temurin-21 AS build
COPY pom.xml .
COPY src/ src/
RUN mvn package -DskipTests

# ステージ2: 実行（小さいイメージ）
FROM eclipse-temurin:21-jre
COPY --from=build target/volta-auth-proxy.jar app.jar
CMD ["java", "-jar", "app.jar"]
```

ステージ1はMavenと完全なJDK（大きいイメージ、約500MB）を持ちます。ステージ2はJREとJARのみ（小さいイメージ、約200MB）。ビルドツールは本番イメージには含まれません。

### ビルド環境の要件

| 要件 | バージョン | 理由 |
|------|---------|------|
| JDK | 21以上 | Java言語機能（レコード、パターンマッチング） |
| Maven | 3.9以上 | ビルドツール |
| PostgreSQL | -- | ビルドには不要、実行時のみ |
| Docker | -- | ビルドには不要、コンテナ化のみ |

## よくある間違いと攻撃

### 間違い1：テストなしでビルド

```bash
mvn package -DskipTests  # 誘惑的だが危険
```

テストをスキップすると数秒節約できますが、壊れたコードをデプロイするリスクがあります。セキュリティバグ（間違ったロールチェックなど）を捕捉するテストは待つ価値があります。

### 間違い2：一貫性のないビルド環境

「自分のマシンでは動く」は、開発者が異なるJDKバージョン、異なるOS設定、異なるMavenバージョンを使っている時に起こります。voltaは以下で緩和しています：

- pom.xmlでJava 21を指定
- 再現可能なビルド用のDockerfileを提供
- 正確な依存関係バージョンを固定

### 攻撃1：ビルド時の依存関係汚染

攻撃者がMaven Centralの依存関係を侵害すると、それをダウンロードするすべてのビルドが悪意のあるコードを取得します。防御策：

- Mavenがダウンロードしたアーティファクトのチェックサムを検証
- voltaが正確なバージョンを固定（バージョン範囲ではなく）
- `mvn dependency:tree`による定期的な依存関係監査

### 間違い3：ビルドアーティファクトをコミットする

`target/`ディレクトリは[git](git.md)に入れるべきではありません。ソースから再構築できる生成ファイルが含まれています。voltaの`.gitignore`は`target/`を除外しています。

### 間違い4：リリースビルド前にクリーンしない

```bash
mvn clean package  # まずtarget/を削除、その後フレッシュにビルド
```

`clean`なしでは、削除されたソースファイルの古いコンパイル済みクラスが`target/`に残り、JARに含まれる可能性があります。
