# ビジネスロジック（Business Logic）

[English version](business-logic.md)

---

## これは何？

ビジネスロジックとは、あなたのビジネスに固有のルールや操作の集まりです -- あなたのアプリケーションをユニークにするもの。インフラストラクチャ（データベース、ネットワーク、サーバー）でもなく、汎用的な機能（ユーザーログイン、ファイルストレージ）でもありません。ソフトウェアの「あなたをあなたたらしめるもの」です。

レストランに例えるとわかりやすいです。厨房設備（オーブン、冷蔵庫、シンク）はインフラストラクチャ -- どのレストランにも必要です。予約を取って客を席に案内するのは汎用的な機能 -- ほとんどのレストランが同じようにやります。でもレシピは？秘密のソースは？地元の農場から仕入れたオーガニック食材のみを使うという決定は？それがビジネスロジックです。このレストランを隣のレストランと違うものにするものです。

---

## ビジネスロジック vs インフラストラクチャの例

```
インフラストラクチャ（汎用的、すべてのアプリに必要）：
  - データベースにデータを保存
  - HTTPレスポンスを送信
  - ネットワーク接続を処理
  - メモリを管理

汎用的な機能（一般的だが、すべてのアプリにすべてが必要ではない）：
  - ユーザー認証（ログイン/ログアウト）
  - ファイルアップロード
  - メール送信
  - 決済処理

ビジネスロジック（あなたのアプリに固有）：
  - 「無料プランのユーザーは最大3プロジェクト作成可能」
  - 「プレミアムユーザーは50GBのストレージ」
  - 「請求書が30日延滞したら、アカウントを停止」
  - 「マネージャーは5,000ドルまでの経費を承認可能；それ以上はVP承認が必要」
  - 「日本のユーザーとアメリカのユーザーで異なる価格を表示」
```

ビジネスロジックはアプリケーションが存在する理由です。これがなければ、汎用的な空の殻があるだけです。

## 認証プロキシにビジネスロジックを含めるべきでない理由

ここからが面白いところです。認証プロキシは1つの質問に答えます：「この人は本人か、そしてこのリソースにアクセスする許可があるか？」それだけです。

認証プロキシは以下のような質問に答えるべきではありません：
- 「このユーザーは無料プランの上限を超えたか？」
- 「このユーザーのサブスクリプションは有効か？」
- 「このユーザーには日本語版を見せるべきか英語版か？」
- 「このマネージャーはこの特定の経費を承認できるか？」

これらはあなたのビジネスルールです。認証レイヤーではなく、あなたのアプリケーションに属します。

### この分離が重要な理由

```
ビジネスロジックを含む認証プロキシ（悪い例）：
┌──────────────────────────────────────┐
│ Auth Proxy                            │
│                                      │
│ - ユーザーはログインしているか？      │ ← 認証（正しい）
│ - ユーザーのロールは何か？            │ ← 認証（正しい）
│ - サブスクリプションは有効か？        │ ← ビジネスロジック（間違い）
│ - API上限を超えたか？                │ ← ビジネスロジック（間違い）
│ - どのティアにいるか？               │ ← ビジネスロジック（間違い）
└──────────────────────────────────────┘

問題：
  - 認証プロキシがサブスクリプション、ティア、API上限を知る必要がある
  - ビジネスルールの変更ごとに認証プロキシの再デプロイが必要
  - 認証プロキシがすべてのビジネスルールのモノリスになる
  - アプリごとにルールが異なり、すべてが1つのプロキシに押し込まれる

ビジネスロジックを含まない認証プロキシ（良い例 / volta）：
┌──────────────────────────────────────┐
│ volta-auth-proxy                      │
│                                      │
│ - ユーザーはログインしているか？      │ ← 認証
│ - ユーザーのロールは何か？            │ ← 認証
│ - どのテナントにいるか？             │ ← 認証
│ - IDヘッダーを返す                   │ ← 認証
└──────────────────────────────────────┘
                │
                │ X-Volta-User-Id, X-Volta-Roles, X-Volta-Tenant-Id
                ▼
┌──────────────────────────────────────┐
│ あなたのアプリ                        │
│                                      │
│ - サブスクリプションは有効か？        │ ← あなたのビジネスロジック
│ - 上限を超えたか？                   │ ← あなたのビジネスロジック
│ - どのティアにいるか？               │ ← あなたのビジネスロジック
└──────────────────────────────────────┘
```

voltaはアプリにユーザーが誰かを伝えます。アプリがユーザーに何ができるかを決めます。

## voltaの規律

voltaは意図的にアイデンティティの関心事に限定しています：

| voltaがやること（認証） | voltaがやらないこと（ビジネスロジック） |
|------|------|
| ユーザーを認証する（OIDCログイン） | サブスクリプション状態のチェック |
| ユーザーを識別する（ユーザーID、メール） | 使用量クォータの強制 |
| テナントを識別する（テナントID、スラグ） | 価格ルールの適用 |
| ロールを決定する（OWNER、ADMIN、MEMBER、VIEWER） | ビジネス権限の評価 |
| IDトークンを発行する（JWT） | コンテンツの決定 |
| アプリレベルのロールアクセスを強制する（allowed_roles） | 請求状態の処理 |

volta-config.yamlの`allowed_roles`はvoltaがビジネスロジックに最も近づくところです -- そしてそれさえ意図的にシンプルです：「このロールはこのアプリにアクセスできるか、はい・いいえ」。アプリ内の細かい権限はアプリの責任です。

この規律により、voltaは小さく、集中し、安定したままです。あなたのビジネスルールが変わってもvoltaを変更する必要はありません。あなたの価格ティア、サブスクリプションモデル、承認ワークフローを知る必要はありません。ただアプリに伝えます：「これは太郎さん、ACMEテナントのADMINです。お好きにどうぞ。」

---

## volta-auth-proxyでは

voltaはアイデンティティとアクセス -- ユーザーが誰か、どのテナントに属するか、どのロールを持つか -- に厳密に限定し、アプリケーション固有のビジネスロジックはすべてvoltaのIDヘッダーを受け取る下流のアプリに委ねています。
