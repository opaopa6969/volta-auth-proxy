# Cache-Control ヘッダ

[English / 英語](cache-control.md)

---

## これは何？

Cache-Control は、ブラウザ（や途中のプロキシ）にキャッシュの扱い方を伝える HTTP レスポンスヘッダです。サーバーのレスポンスに含まれる1行で、「これは60秒間保存していいよ」「これは絶対に保存しないで」といった指示を出します。ブラウザはページを読み込むたびにこのヘッダを確認し、保存済みのコピーを使うか、サーバーに新しく取りに行くかを判断します。

---

## なぜ重要？

キャッシュはウェブを高速にする仕組みです。キャッシュがなければ、内容が変わっていなくても毎回サーバーに問い合わせが必要になります。しかし、キャッシュしてはいけないものをキャッシュすると危険です。「ようこそ、alice@example.com さん」と表示されたページがキャッシュされ、別のユーザーに表示されたら情報漏洩です。Cache-Control は「積極的にキャッシュしていい」と「絶対に保存するな」の境界線を引くための仕組みです。

Cache-Control の設定ミスは、パフォーマンス問題（厳しすぎる）とセキュリティ事故（緩すぎる）の両方で最もよくある原因の一つです。

---

## 簡単な例

サーバーがこのレスポンスヘッダを送る場合：

```
Cache-Control: public, max-age=3600
```

意味は：

- **public** -- どのキャッシュ（ブラウザ、CDN、プロキシ）でも保存してよい
- **max-age=3600** -- 保存したコピーは 3600 秒（1時間）有効

この1時間、ブラウザはサーバーに一切問い合わせず、手元のコピーを使います。

一方、これは：

```
Cache-Control: no-store, private
```

- **no-store** -- このレスポンスをどこにも保存しない。メモリにもディスクにも
- **private** -- 仮にキャッシュする場合でも、エンドユーザーのブラウザだけ（共有 CDN は不可）

個人情報や機密データを含むページにはこちらを使います。

## volta-auth-proxy での使い方

volta は Cache-Control を2つの場面で使い分けています。

**静的アセット**（CSS、JavaScript、画像）には緩いキャッシュを設定：

```java
ctx.header("Cache-Control", "public, max-age=60, stale-while-revalidate=86400");
```

ブラウザや CDN に静的ファイルを60秒間キャッシュさせ、期限切れ後も裏で新しいファイルを取得しつつ古いバージョンを最大1日間提供します。

**すべての認証ページと API レスポンス**には `setNoStore()` ヘルパーメソッドを呼びます：

```java
ctx.header("Cache-Control", "no-store, no-cache, must-revalidate, private");
ctx.header("Pragma", "no-cache");
```

これは「絶対にキャッシュするな」の最も強い設定です。ログインページ、セッションデータ、ユーザー固有のレスポンスがキャッシュに保存されないことを保証します。`Pragma: no-cache` は HTTP/1.0 クライアントとの互換性のためです。
