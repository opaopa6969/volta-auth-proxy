# Caffeine（インメモリキャッシュ）

[English version](caffeine-cache.md)

---

## これは何？

CaffeineはJava向けの高性能インメモリキャッシュライブラリで、頻繁に使うデータをアプリケーションのメモリ（RAM）に保存し、毎回データベースから取得する必要をなくします。

デスクの付箋で考えてみましょう。同僚の内線番号が必要なとき、毎回社内ディレクトリを調べる（データベースにアクセスする）こともできます。または付箋に書いてデスクに貼っておく（キャッシング）こともできます。付箋のほうが速いですが、同僚が席替えすると古くなりますし、付箋が多すぎるとデスクが散らかります。Caffeineは付箋をスマートに管理するシステムです -- 最も役立つものを残し、古いものは自動的に捨て、1時間前に変わった番号を使っていないことを確認してくれます。

---

## なぜ重要なのか？

データベースは速いですが、メモリはもっと速いです。はるかに速い：

```
  データ取得にかかる時間：
  ┌──────────────────────────────────────┐
  │ RAM（Caffeine）から:    ~100ns       │  ← 0.0001ミリ秒
  │ PostgreSQLから:         ~1-5ms       │  ← 10,000〜50,000倍遅い
  │ リモートAPIから:        ~50-500ms    │  ← さらに遅い
  └──────────────────────────────────────┘
```

頻繁に読み込まれるがめったに変わらないデータに対して、キャッシングは大きなパフォーマンス向上です。「このセッションは有効？」と毎秒100回データベースに聞く代わりに、1回聞いて答えを30秒間覚え、次の99リクエストはメモリから返します。

### なぜ特にCaffeineなのか？

Javaにはいくつかのキャッシングライブラリがあります。Caffeineが最良なのは：

| 特徴 | Caffeine |
|------|----------|
| **速度** | 最速のJavaキャッシュ（ほぼ最適なヒット率） |
| **退去** | Window TinyLFUアルゴリズム使用（LRUより賢い） |
| **サイズ制限** | 最大エントリ数または最大メモリ重みを設定 |
| **時間ベースの有効期限** | 書き込み後または最終アクセス後に期限切れ |
| **統計** | 組み込みのヒット/ミス率追跡 |
| **スレッドセーフ** | 複数スレッドから同時に安全に使用可能 |

## voltaでのCaffeineの使い方

volta-auth-proxyはレート制限とパフォーマンスが重要な検索にインメモリキャッシュを使用しています。

### レート制限

ユーザー（または攻撃者）がリクエストを送りすぎると、voltaはIPアドレスまたはユーザーごとにリクエスト数を追跡する必要があります。すべてのリクエストがレートリミッターをチェックするので、極めて高速である必要があります。

```
  キャッシュを使ったレート制限：

  IP 192.168.1.1からリクエストが来る
  ┌─────────────────────────────────────┐
  │ 1. キャッシュ確認: "192.168.1.1" → 47│  ← この1分間で47リクエスト
  │ 2. インクリメント: 47 → 48            │
  │ 3. 48 > 制限(60)？ いいえ → 許可     │
  └─────────────────────────────────────┘
  時間: ~0.001ms（マイクロ秒）

  キャッシュなし（データベース）：
  ┌─────────────────────────────────────┐
  │ 1. SELECT count FROM rate_limits    │
  │    WHERE ip = '192.168.1.1'         │
  │ 2. UPDATE rate_limits SET count=48  │
  │ 3. 48 > 60？ いいえ → 許可          │
  └─────────────────────────────────────┘
  時間: ~2-5ms（何千倍も遅い）
```

レート制限データは再起動後に残る必要がありません。voltaが再起動すれば、カウンターはゼロにリセットされます。それで問題ありません -- 最悪のケースは、レート制限されたユーザーが新しい制限枠をもらうだけです。これがこのユースケースにインメモリキャッシュが最適な理由です。

### セッションキャッシュ

Traefikが[ForwardAuth](forwardauth.ja.md)リクエストをvoltaに送ると、voltaはユーザーのセッションを検索する必要があります。これはすべての保護されたアプリへのすべてのリクエストで発生します。セッションデータを短期間（秒単位）キャッシュすることで、すべてのリクエストでデータベースにアクセスするのを避けます：

```
  セッションキャッシュなし：
  リクエスト1 → DBクエリ(3ms) → セッション有効
  リクエスト2 → DBクエリ(3ms) → セッション有効（同じセッション、1秒後）
  リクエスト3 → DBクエリ(3ms) → セッション有効（同じセッション、2秒後）

  セッションキャッシュあり（30秒TTL）：
  リクエスト1 → DBクエリ(3ms) → 結果をキャッシュ → セッション有効
  リクエスト2 → キャッシュヒット(0.001ms) → セッション有効
  リクエスト3 → キャッシュヒット(0.001ms) → セッション有効
  ... 30秒経過、キャッシュエントリ期限切れ ...
  リクエストN → DBクエリ(3ms) → 結果をキャッシュ → セッション有効
```

トレードオフ：セッションが無効化された場合、キャッシュが期限切れになって無効化が反映されるまで最大30秒かかります。voltaはこのトレードオフを受け入れます。理由は：

1. JWTにはすでに5分間の有効期間がある
2. セッション無効化はまれなイベント（ログアウト、管理者操作）
3. すべてのリクエストでのパフォーマンス向上は、わずかな遅延に見合う

## 「付箋」のアナロジーを詳しく

```
  Caffeineキャッシュはスマートな付箋システム：

  ┌─────────────────────────────────┐
  │ あなたのデスク（RAM）             │
  │                                 │
  │  ┌──────────┐ ┌──────────┐     │
  │  │セッションA │ │レート:    │     │
  │  │有効       │ │IP .1.1   │     │
  │  │午後3時まで │ │= 23リクエ│     │
  │  └──────────┘ └──────────┘     │
  │                                 │
  │  ルール：                        │
  │  - 付箋は最大1000枚             │
  │  - 30秒後に捨てる               │
  │  - よく使う付箋は長く残る        │
  └─────────────────────────────────┘

  付箋があれば → 読む（速い！）
  なければ → ファイリングキャビネット
             （データベース）で調べて、新しい付箋を作る
```

## シンプルなコード例

```java
import com.github.benmanes.caffeine.cache.Caffeine;
import com.github.benmanes.caffeine.cache.Cache;
import java.time.Duration;

// 最大10,000エントリを保持し、
// 書き込みから30秒後にエントリが期限切れになるキャッシュを作成
Cache<String, SessionInfo> sessionCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofSeconds(30))
    .build();

// まずキャッシュを確認し、なければデータベースにフォールバック
SessionInfo session = sessionCache.get(sessionId, key -> {
    // このラムダはキャッシュミス時のみ実行される
    return database.lookupSession(key);
});
```

`get`メソッドがポイントです：まずキャッシュを確認します。エントリがあれば（キャッシュヒット）、即座に返します。なければ（キャッシュミス）、ラムダを呼んでデータベースから読み込み、結果をキャッシュに保存して返します。
