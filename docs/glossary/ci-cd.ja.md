# CI/CD（継続的インテグレーション / 継続的デリバリー）

[English version](ci-cd.md)

---

## これは何？

CI/CDはContinuous Integration（継続的インテグレーション）とContinuous Delivery（継続的デリバリー）またはContinuous Deployment（継続的デプロイメント）の略です。コードをプッシュするたびに、自動的にビルド、テスト、デプロイする実践です。手動でビルドしてデプロイする代わりに、パイプラインが一貫して、確実に、人的エラーなしにそれを行います。

自動洗車機で考えてみましょう。車（コード）を入口に入れます。機械が自動的に石鹸、こすり、すすぎ、乾燥を毎回まったく同じ順序で行います。各ステップを覚える必要もスキップの心配もありません。CI/CDはソフトウェアでの同じことです：コードをプッシュすれば、パイプラインが自動的にコンパイル、テスト、パッケージ、デプロイします。

**CI（継続的インテグレーション）** とは：コードのプッシュごとに自動ビルドとテストが実行される。壊れたコードは即座に検出される。

**CD（継続的デリバリー）** とは：テスト合格後、コードが自動的にパッケージされデプロイ準備完了になる（ボタンをクリック）。**継続的デプロイメント**はさらに進んで、人の承認なしに自動デプロイされる。

---

## なぜ重要なのか？

- **バグを早期に検出。** プッシュ時のテスト失敗は、顧客が[本番](production.ja.md)で発見するバグより修正が安い。
- **デプロイの恐怖を軽減。** デプロイが自動化・テスト済みなら、チームはより頻繁に、より少ないリスクでデプロイできる。
- **一貫性を保証。** 毎回同じビルドプロセスが実行される。「自分のマシンでは動く」問題がなくなる。
- **デリバリーを加速。** 機能が数週間ではなく数時間でユーザーに届く。
- **品質ゲートを強制。** テストに合格しないコードはデプロイされない。以上。

## どう動くのか？

### CI/CDパイプライン

```
  開発者がmainブランチにコードをプッシュ
       │
       ▼
  ┌──────────────────────────────────────────────────┐
  │              CI/CDパイプライン                      │
  │                                                    │
  │  ┌─────────┐  ┌─────────┐  ┌─────────┐          │
  │  │ ビルド   │─▶│ テスト   │─▶│パッケージ│          │
  │  │          │  │         │  │         │          │
  │  │ mvn      │  │ mvn     │  │ mvn     │          │
  │  │ compile  │  │ test    │  │ package │          │
  │  └─────────┘  └─────────┘  └────┬────┘          │
  │                                  │               │
  │                    ┌─────────────┴──────────┐    │
  │                    │                        │    │
  │               ┌────┴────┐            ┌──────┴──┐ │
  │               │ デプロイ │            │ デプロイ │ │
  │               │ステージング│           │  本番   │ │
  │               └─────────┘            └─────────┘ │
  │                                （手動承認）        │
  └──────────────────────────────────────────────────┘
```

### CI：ビルドとテスト

すべてのプッシュがコンパイルとテストをトリガーします：

```
  git push origin main
       │
       ▼
  GitHub Actions（または類似）が開始:

  ステップ1: コードをチェックアウト
  ステップ2: Java 21をセットアップ
  ステップ3: mvn compile        ← コンパイルできるか？
  ステップ4: mvn test           ← すべてのテストが通るか？
  ステップ5: mvn package        ← fat JARをビルド

  ┌────────┐
  │ 結果   │
  ├────────┤
  │ ✓ 合格 │ → デプロイ準備完了
  │ ✗ 失敗 │ → 開発者に通知、デプロイ阻止
  └────────┘
```

### CD：パッケージとデプロイ

テスト合格後、パイプラインがパッケージしてデプロイします：

```
  テスト合格
       │
       ▼
  ┌──────────────────────────┐
  │  アーティファクトをビルド    │
  │  volta-auth-proxy-1.2.0.jar │
  └────────────┬─────────────┘
               │
       ┌───────┴───────┐
       │               │
       ▼               ▼
  ┌─────────┐    ┌─────────┐
  │ デプロイ │    │ 保存    │
  │ステージング│   │アーティファクト│
  │ へ      │    │（バックアップ）│
  └────┬────┘    └─────────┘
       │
  スモークテスト合格？
       │
       ▼
  ┌─────────┐
  │ デプロイ │
  │ 本番へ   │
  └─────────┘
```

### GitHub Actionsワークフロー例

```yaml
name: CI/CD
on:
  push:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Java 21をセットアップ
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      - name: ビルドとテスト
        run: mvn clean verify

      - name: パッケージ
        run: mvn package -DskipTests

      - name: アーティファクトをアップロード
        uses: actions/upload-artifact@v4
        with:
          name: volta-auth-proxy
          path: target/volta-auth-proxy-*.jar

  deploy-staging:
    needs: build-and-test
    runs-on: ubuntu-latest
    steps:
      - name: ステージングにデプロイ
        run: ./deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production  # 手動承認が必要
    steps:
      - name: 本番にデプロイ
        run: ./deploy.sh production
```

## volta-auth-proxy ではどう使われている？

### voltaのCI/CDパイプライン

voltaの一般的なCI/CDパイプラインには以下が含まれます：

```
  ┌────────────────────────────────────────────────────┐
  │  volta-auth-proxy CI/CDパイプライン                  │
  │                                                     │
  │  1. ✓ コンパイル（Java 21 + Maven）                  │
  │  2. ✓ ユニットテスト実行                             │
  │  3. ✓ インテグレーションテスト実行（テストDB + Flyway）│
  │  4. ✓ 依存関係の脆弱性チェック                       │
  │  5. ✓ fat JARビルド                                 │
  │  6. ✓ ステージングにデプロイ                          │
  │  7. ✓ ステージングでスモークテスト実行                 │
  │  8. ◎ 本番にデプロイ（手動ゲート）                    │
  └────────────────────────────────────────────────────┘
```

### CI/CDがvoltaで検証すること

| チェック | 何を検出するか |
|---------|--------------|
| `mvn compile` | 構文エラー、型の不一致 |
| `mvn test` | ロジックバグ、認証フローの退行 |
| Flywayマイグレーションテスト | 既存データを壊すスキーマ変更 |
| 依存関係チェック | ライブラリの既知の脆弱性 |
| インテグレーションテスト | ForwardAuth、OIDCフロー、セッションライフサイクル |
| スモークテスト | デプロイされたインスタンスの基本ヘルスチェック |

### CI/CDでのデータベースマイグレーション

パイプラインはテストデータベースに対して[Flywayマイグレーション](migration.ja.md)を実行し検証します：
- 新しいマイグレーションがクリーンに適用されること
- 既存のマイグレーションが変更されていないこと（チェックサム検証）
- アプリケーションコードが新しいスキーマで動作すること

```
  CIパイプライン:
  ┌────────────────────────────────────────┐
  │ 1. テスト用PostgreSQLを起動（Docker）   │
  │ 2. Flyway migrate（全V*.sqlを適用）     │
  │ 3. マイグレーション済みDBでテスト実行    │
  │ 4. テストDBを破棄                       │
  └────────────────────────────────────────┘
```

## よくある間違いと攻撃

### 間違い1：ローカルで通るがCIで失敗するテスト

「自分のマシンでは動く」は多くの場合CI環境が異なることを意味する -- 異なるJavaバージョン、異なるOS、環境変数の欠如。すべての場所で同じJava [LTS](lts.ja.md)バージョンを使いましょう。

### 間違い2：速くデプロイするためにテストをスキップ

CIパイプラインでの`mvn package -DskipTests`は目的を完全に損なわせます。デプロイパイプラインでは決してテストをスキップしないでください。

### 間違い3：ステージング環境がない

CIから直接本番に行くと、最初の本当のテストが本番で行われることになります。常にまずステージングにデプロイしましょう。

### 間違い4：CI設定にシークレット

データベースパスワード、Google OIDCシークレット、JWT秘密鍵はCIプラットフォームのシークレットマネージャに格納すべきで、ワークフローファイル（gitにコミットされる）には入れません。

### 間違い5：パイプラインにロールバック計画がない

本番デプロイが失敗した場合、パイプラインは自動的に前のバージョンにロールバックするか、手動で簡単にできるようにすべきです。
