# クライアント

[English version](client.md)

---

## これは何？

クライアントとは、[サーバー](server.md)にリクエストを送り、レスポンスを受け取るプログラムやデバイスのことです。Web開発において最も一般的なクライアントは[ブラウザ](browser.md)です -- Webサーバーに[HTTP](http.md)リクエストを送り、結果を表示します。しかしクライアントはモバイルアプリ、`curl`のようなコマンドラインツール、別のサーバー、または[SDK](sdk.md)でもあり得ます。

レストランで食事を注文するのに似ています。あなたがクライアントで、リクエストを出します（「パスタをお願いします」）。キッチンがサーバーで、リクエストを処理して結果を返します。キッチンがどう動くか知る必要はありません。注文の仕方（[API](api.md)）を知っていれば十分です。

クライアント-サーバーモデルはインターネットの基盤です。ウェブサイトを開くたびに、あなたのブラウザ（クライアント）がサーバーにリクエストを送り、サーバーがレスポンスを返します。

---

## なぜ重要なのか？

クライアントとは何かを理解することが重要な理由：

- **セキュリティ信頼境界** -- クライアントは信頼できません。サーバーはクライアントが正直だと絶対に仮定してはいけません。ユーザーがクライアントを制御し、動作を変更できます。
- **複数のクライアントタイプ** -- volta-auth-proxyは異なるタイプのクライアントにサービスを提供します：ブラウザ、[SPA](spa.md)、[ダウンストリームアプリ](downstream-app.md)、Traefik。それぞれ通信方法が異なります。
- **クライアントサイド vs サーバーサイド** -- この区別がコードの実行場所、アクセス可能なデータ、セキュリティ保証を決定します。[フロントエンドとバックエンド](frontend-backend.md)を参照。
- **認証フローはクライアントによって異なる** -- ブラウザは[Cookie](cookie.md)を使います。APIクライアントは[Bearerトークン](bearer-scheme.md)を使います。サーバー間通信は[Internal API](internal-api.md)とJWTを使います。

### 異なるクライアントがvoltaでどう認証するか

| クライアントの種類 | 認証方法 | トークンの場所 |
|-----------------|---------|-------------|
| ブラウザ（人間のユーザー） | セッション[Cookie](cookie.md) | `Cookie: JSESSIONID=...` |
| SPA（JavaScript） | セッションCookie + JWT更新 | セッションにCookie、メモリに[JWT](jwt.md) |
| ダウンストリームバックエンド | [ForwardAuth](forwardauth.md)ヘッダー | `X-Volta-User-Id`, `X-Volta-Tenant-Id` |
| APIクライアント | [Bearerトークン](bearer-scheme.md) | `Authorization: Bearer <jwt>` |
| Traefik | ForwardAuth | 内部ネットワーク呼び出し |

## volta-auth-proxy ではどう使われている？

### ブラウザクライアントに対するサーバーとしてのvolta

ユーザーがvoltaログインページを開くと、ブラウザがクライアントです：

```
  ブラウザ（クライアント）            volta（サーバー）
  ────────────────                    ──────────────
  GET /auth/login  ──────────────────> ログインHTMLページを返す
                   <──────────────────

  ユーザーが「Googleでサインイン」をクリック
  GET /auth/callback?code=... ───────> GoogleでコードSを検証
                                       セッションを作成
                   <────────────────── Set-Cookie: JSESSIONID=abc
                                       /dashboardにリダイレクト
```

### SPAクライアントに対するサーバーとしてのvolta

[SPA](spa.md)がvoltaのAPIを呼ぶとき：

```
  SPA（クライアント）                 volta（サーバー）
  ────────────                        ──────────────
  GET /api/v1/users/me
  Cookie: JSESSIONID=abc  ──────────> セッションを検証
                                      データベースでユーザーを検索
                          <────────── 200 OK {"userId":"...","name":"Taro"}
```

### Traefik（別のサーバーがクライアント）に対するサーバーとしてのvolta

[ForwardAuth](forwardauth.md)パターンでは、Traefikがクライアントです：

```
  Traefik（クライアント）             volta（サーバー）
  ────────────────                    ──────────────
  GET /forwardauth
  X-Forwarded-Uri: /app/dashboard
  Cookie: JSESSIONID=abc  ──────────> セッションを検証
                          <────────── 200 OK
                                      X-Volta-User-Id: 550e...
                                      X-Volta-Tenant-Id: abcd...
                                      X-Volta-Roles: ADMIN
```

### Googleに対するクライアントとしてのvolta

voltaは常にサーバーというわけではありません。[OIDC](oidc.md)認証を行う際、voltaはGoogleのサーバーに対するクライアントとして動作します：

```
  volta（クライアント）               Google（サーバー）
  ──────────────                      ────────────────
  POST /token
  code=xyz&redirect_uri=...  ────────> 認可コードを検証
                             <──────── id_token + access_tokenを返す

  GET /userinfo
  Authorization: Bearer ... ─────────> ユーザープロフィールを返す
                            <────────── {"email":"taro@example.com",...}
```

これが示すのは、クライアントとサーバーは固定された性質ではなく**役割**だということです。同じプログラムがあるやり取りではクライアント、別のやり取りではサーバーになれます。

## よくある間違いと攻撃

### 間違い1：クライアントが提供したデータを信頼する

クライアントは送信内容を制御します。偽のヘッダー、改変されたリクエストボディ、偽造されたCookieを送れます。サーバーはすべてを検証しなければなりません：

```
  決して信頼しない:
  ├── X-Forwarded-Forヘッダー（クライアントがIPを偽装できる）
  ├── User-Agentヘッダー（クライアントが何にでも偽装できる）
  ├── リクエストボディの値（クライアントが任意のJSONを送れる）
  └── クエリパラメータ（クライアントがURLを変更できる）

  常にサーバーで検証する:
  ├── セッションCookie（暗号的に検証）
  ├── JWT署名（RS256検証）
  └── データベースからのロール（リクエストからではなく）
```

### 攻撃1：クライアントのなりすまし

攻撃者がブラウザを装うスクリプトを書きます：

```bash
curl -H "User-Agent: Mozilla/5.0 Chrome/120" \
     -H "Cookie: JSESSIONID=stolen_value" \
     https://volta.example.com/api/v1/users/me
```

サーバーは同じヘッダーを持つ本物のブラウザと`curl`を区別できません。だからこそセッションCookieは`HttpOnly`、`Secure`、`SameSite`フラグで保護しなければなりません -- そもそも[XSS](xss.md)に盗まれるのを防ぐためです。

### 間違い2：1つのクライアントタイプしか想定しない

ブラウザでのみ正しく動くAPI（Cookieとリダイレクトに依存）を作ると、APIクライアント、モバイルアプリ、サーバー間通信では動きません。voltaはCookieベースの認証（ブラウザ用）とJWTベースの認証（APIクライアント用）の両方を提供し、複数のクライアントタイプをサポートしています。

### 攻撃2：自動化クライアントによる濫用

ボットやスクリプトは毎秒何千ものリクエストを送れます。[レート制限](rate-limiting.md)なしでは、悪意のあるクライアントがサーバーを圧倒したり、認証のブルートフォースをしたりできます。voltaは[レート制限](rate-limiting.md)で重要な[エンドポイント](endpoint.md)を保護しています。
