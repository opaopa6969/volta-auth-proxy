# コンパイル

[English version](compile.md)

---

## これは何？

コンパイルとは、人間が読めるソースコードをコンピュータが実行できる形式に変換するプロセスです。[Java](java.md)では、コンパイラが`.java`ファイルを読み、[JVM](jvm.md)が実行できるバイトコードを含む`.class`ファイルを生成します。コードにエラー（型の間違い、存在しないメソッド、構文ミス）があれば、コンパイラは出力の生成を拒否し、何が問題かを正確に教えてくれます。

本を英語から日本語に翻訳するのに似ています。翻訳者（コンパイラ）が英語の原稿（ソースコード）を読み、日本語版（バイトコード）を生成します。英語の文が文法的に意味をなさなければ、翻訳者はそれを翻訳できず、問題を示す赤い印をつけて原稿を返します。翻訳を進めるには、まず英語を修正しなければなりません。

コンパイルはより大きな[ビルド](build.md)プロセスの1ステップです。`mvn compile`はソースコードをコンパイルします。`mvn package`はコンパイル、テスト、[fat JAR](fat-jar.md)へのパッケージングを行います。

---

## なぜ重要なのか？

コンパイルが重要な理由は、バグに対する最初の防衛線だからです：

- **型エラーを早期に捕捉** -- コンパイラが[型安全](type-safe.md)を検証します。`UUID`が期待される場所に`String`を渡した？コンパイルエラー。
- **存在しないメソッドを早期に捕捉** -- メソッドが`user.name()`なのに`user.nmae()`を呼んだ？コンパイルエラー。
- **構文エラーを早期に捕捉** -- セミコロン忘れ、波括弧の不一致、間違ったimport？コンパイルエラー。
- **速いフィードバックループ** -- 問題を数秒で知れます。本番にデプロイしてからではなく。

volta-auth-proxyにとって、コンパイルは[jte](jte.md) [テンプレート](template.md)もチェックします。テンプレートのタイポはユーザーがページを訪れた時ではなく、コンパイル時に捕捉されます。

## どう動くのか？

### Javaコンパイルパイプライン

```
  ソースコード (.java)
       │
       ▼
  ┌──────────────────┐
  │  Javaコンパイラ   │
  │  (javac)          │
  │                   │
  │  1. 構文を解析    │
  │  2. 型をチェック  │
  │  3. 参照を解決    │
  │  4. バイトコード  │
  │     を生成        │
  └──────────────────┘
       │
       ▼
  バイトコード (.class)
       │
       ▼
  ┌──────────────────┐
  │  JVM              │
  │  バイトコードを   │
  │  実行             │
  └──────────────────┘
```

### コンパイラがチェックするもの

| チェック | エラーの例 | 捕捉タイミング |
|---------|-----------|-------------|
| 構文 | `if (x = 5)` → `if (x == 5)` | コンパイル時 |
| 型 | `String x = 42;` | コンパイル時 |
| メソッド解決 | `user.nmae()`（タイポ） | コンパイル時 |
| import解決 | `import com.example.Foo;`（クラス不在） | コンパイル時 |
| アクセス制御 | `private`メソッドを外部から呼び出し | コンパイル時 |
| ジェネリクス | `List<String>.add(42)` | コンパイル時 |

### コンパイル言語 vs インタプリタ言語

| 言語 | モデル | エラー捕捉 | voltaでの役割 |
|------|-------|-----------|-------------|
| [Java](java.md) | バイトコードにコンパイル | コンパイル時 | voltaの言語 |
| Go, Rust, C | ネイティブにコンパイル | コンパイル時 | -- |
| [JavaScript](javascript.md) | インタプリタ | 実行時 | ブラウザ/SPA言語 |
| Python | インタプリタ | 実行時 | -- |
| TypeScript | JavaScriptにコンパイル | コンパイル時（型のみ） | -- |

### インクリメンタルコンパイル

現代のコンパイラは変更されたファイルだけを再コンパイルします：

```
  初回コンパイル:   全50ファイルをコンパイル       → 5秒
  1ファイル変更:    そのファイルだけ再コンパイル   → 0.3秒
  テンプレート変更: そのテンプレートだけ再コンパイル → 0.2秒
```

これにより編集-コンパイル-テストのサイクルが生産的な開発に十分な速さになります。

### コンパイルエラー vs ランタイムエラー

```
  コンパイルエラー（良い -- 実行前に捕捉）:
  ─────────────────────────────────────────────
  src/main/java/UserService.java:42: error: incompatible types
      UUID id = "not-a-uuid";
                ^
      required: UUID
      found:    String

  → 修正: UUID id = UUID.fromString("550e8400-...");


  ランタイムエラー（悪い -- 実行中に捕捉）:
  ─────────────────────────────────────────────
  Exception in thread "main" java.lang.ClassCastException:
      java.lang.String cannot be cast to java.util.UUID
      at UserService.getUser(UserService.java:42)

  → ユーザーはすでにエラーページを見ている。ダメージは完了。
```

## volta-auth-proxy ではどう使われている？

### Mavenでのコンパイル

voltaは[Maven](maven.md)をビルドに使用しています。コンパイルステップ：

```bash
# Javaソースコードをコンパイル
mvn compile

# src/main/java/のすべての.javaファイルにjavacを実行
# 出力はtarget/classes/へ
```

### jteテンプレートのプリコンパイル

voltaはビルド中に[jte](jte.md)テンプレートをプリコンパイルし、テンプレートエラーをコンパイル時に捕捉します：

```bash
# mvn compile中にjte Mavenプラグインが:
# 1. src/main/jte/のすべての.jteファイルを読む
# 2. それらからJavaソースコードを生成
# 3. 生成されたJavaコードをコンパイル
# 4. テンプレートエラー = コンパイル失敗
```

つまり`${user.nmae}`のようなタイポは、ユーザーがページを訪れた時ではなく、ビルドが即座に失敗します。

### 完全なビルドパイプライン

```
  mvn package
       │
       ├── 1. ソースコードをコンパイル (mvn compile)
       │       javac: .java → .class
       │       jte: .jte → .java → .class
       │
       ├── 2. テストを実行 (mvn test)
       │       JUnitテストが動作を検証
       │
       ├── 3. パッケージ (mvn package)
       │       .classファイル + 依存関係を
       │       1つのfat JARにアセンブル
       │
       └── 出力: target/volta-auth-proxy.jar
```

### セキュリティのためのコンパイル時安全性

voltaのセキュリティ上重要な部分はコンパイル時チェックに依存しています：

```java
// Role enum -- ロールのスペルミスはコンパイルエラー
if (role == Role.ADMNI) { }  // ✗ コンパイルエラー: ADMNIが見つかりません

// JWTクレーム -- 型付きレコードがすべてのクレームの存在を保証
public record VoltaClaims(
    UUID sub,
    UUID voltaTid,
    List<Role> voltaRoles,
    String voltaDisplay
) {}
// フィールドの記入忘れ = コンパイルエラー
```

## よくある間違いと攻撃

### 間違い1：コンパイラ警告を無視する

コンパイラはエラーではない警告を出すことがあります -- コードはコンパイルされますが、コンパイラが問題を疑っています。よくある警告：

```
  warning: unchecked cast（未チェックのキャスト）
  warning: deprecated method（非推奨のメソッド）
  warning: unused variable（未使用の変数）
```

警告をエラーとして扱いましょう。voltaのビルドは微妙なバグを防ぐために警告で失敗すべきです。

### 間違い2：「コンパイルが通った」が「動く」を意味すると思い込む

コンパイルは型エラーを捕捉しますが、ロジックエラーは捕捉しません。これはコンパイルは通りますが間違っています：

```java
// コンパイルは通るが、ロジックが逆
if (role.isAtLeast(Role.VIEWER)) {
    // ADMINをチェックすべきだったのに、VIEWERをチェックしている
    deleteAllMembers();
}
```

だからテストが存在します。コンパイル + テストの組み合わせが強力な保証を提供します。

### 間違い3：テンプレートをコンパイルしない

多くのJavaプロジェクトはコンパイル時にチェックされないテンプレートエンジン（JSP、動的モデルのFreeMarker）を使っています。テンプレートエラーは実行時まで隠れます。voltaがjteを選んだのは、まさにテンプレートをコンパイルし、このクラスのバグを排除するからです。

### 間違い4：コンパイルするがテストしない

```
  mvn compile   → 型と構文のみチェック
  mvn test      → JUnitテストも実行
  mvn package   → コンパイル + テスト + パッケージ
  mvn verify    → コンパイル + テスト + 統合テスト + パッケージ
```

`mvn compile`だけの実行は偽の安心感を与えます。デプロイ前に最低でも`mvn package`を実行しましょう。
