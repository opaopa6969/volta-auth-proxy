# コンプライアンス

[English version](compliance.md)

---

## これは何？

コンプライアンスとは、ソフトウェアに適用されるルールを満たすことです。これらのルールは法律（GDPR、HIPAA）、業界標準（SOC 2、PCI DSS、ISO 27001）、契約上の義務（顧客のセキュリティ要件）から来ます。ソフトウェアが個人データ、決済、医療記録を扱うなら、コンプライアンスはオプションではなく、法的に必須です。

車の運転のようなものです。世界最速の車を作れても、シートベルト、ウインカー、排ガス制御がなければ、公道を合法的に走れません。コンプライアンスは、規制環境で運用するためにソフトウェアに必要なシートベルトやウインカーのセットです。

コンプライアンスはセキュリティとは異なります。セキュアだが非準拠（データを適切に保護しているが方法を文書化していない）の場合もあれば、準拠しているが非セキュア（書面上はすべてのチェックボックスを満たしているが脆弱性がある）の場合もあります。目標は両方です。

---

## なぜ重要？

非準拠には実際の結果があります：

- **罰金**：GDPR の罰金はグローバル年間売上の 4% または2,000万ユーロのいずれか高い方に達する可能性があります。これは理論的な話ではなく、企業が数十億の罰金を科されています。
- **契約喪失**：エンタープライズ顧客はコンプライアンス認証を要求します。認証がなければ契約もありません。
- **法的責任**：データ侵害が発生し、準拠していなかった場合、責任は劇的に増大します。
- **評判の損害**：「X社、GDPR違反で罰金」は信頼を失墜させる見出しです。

初期段階のスタートアップにとって、コンプライアンスはスペクトラムです。初日に SOC 2 は不要ですが、どの規制が適用されるかを知り、それを満たす計画を持つ必要があります。

## どう動くのか？

### コンプライアンスの全体像

| 基準 | 正式名称 | 適用対象 | 要件 |
|------|---------|---------|------|
| **GDPR** | 一般データ保護規則 | EU 居住者のデータを扱う者 | プライバシーバイデザイン、同意、削除権、DPO |
| **SOC 2** | Service Organization Control 2 | 米国エンタープライズに販売する SaaS ベンダー | セキュリティコントロールの独立監査 |
| **ISO 27001** | 情報セキュリティ管理 | グローバル企業 | 正式な ISMS（情報セキュリティ管理システム） |
| **HIPAA** | 医療保険の携行性と責任に関する法律 | 米国の医療データ取扱者 | 暗号化、アクセス制御、監査証跡、BAA |
| **PCI DSS** | ペイメントカード業界データセキュリティ基準 | クレジットカードを処理する者 | ネットワークセキュリティ、暗号化、脆弱性管理 |
| **CCPA** | カリフォルニア消費者プライバシー法 | カリフォルニア居住者のデータを扱う企業 | 開示、オプトアウト、削除権 |

### 実際のコンプライアンスの流れ

```
  ステップ1: どの規制が適用されるか特定
            ┌──────────────────────────┐
            │ EU ユーザーデータを       │
            │ 保存？ ──► GDPR 適用     │
            │                          │
            │ 米国エンタープライズに    │
            │ 販売？ ──► SOC 2         │
            │                          │
            │ 決済を処理？             │
            │ ──► PCI DSS             │
            │                          │
            │ 医療データ？             │
            │ ──► HIPAA               │
            └──────────────────────────┘
              │
              ▼
  ステップ2: ギャップ分析
            すでにやっていることは？
            足りないものは？
              │
              ▼
  ステップ3: コントロールの実装
            - 技術的（暗号化、アクセス制御、ログ記録）
            - 手続き的（ポリシー、研修、インシデント対応）
            - 文書化（証拠収集）
              │
              ▼
  ステップ4: 検証 / 監査
            - 自己評価（GDPR）
            - 第三者監査（SOC 2、ISO 27001）
            - 認証機関（PCI DSS）
              │
              ▼
  ステップ5: 継続的に維持
            コンプライアンスは一度きりのイベントではない。
            年次監査、継続的な監視、
            規制の変更に伴うポリシー更新。
```

### SaaS のための GDPR 必須事項

ほとんどの SaaS 製品は EU ユーザーデータを扱うため、GDPR は最も普遍的に関連する規制です：

| GDPR 要件 | 実務での意味 |
|-----------|------------|
| **合法的根拠** | データ処理に法的理由が必要（同意、契約、正当な利益） |
| **データ最小化** | 必要なものだけ収集。「念のため」のデータ蓄積はしない |
| **アクセス権** | ユーザーは保持するすべてのデータを請求できる |
| **削除権** | ユーザーはデータの削除を請求できる（「忘れられる権利」） |
| **データ侵害通知** | 侵害から72時間以内に当局に通知する必要がある |
| **プライバシーバイデザイン** | プライバシーをシステムアーキテクチャに組み込む。後付けではなく |
| **データ処理契約** | ユーザーデータを処理する第三者との必須契約 |

## volta-auth-proxy ではどう使われている？

volta-auth-proxy はコンプライアンスに実用的なアプローチを取ります：技術的なコントロールは今実装し、認証は必要になるまで延期します。

### volta が今行っていること

| コントロール | 実装 | 関連基準 |
|-------------|------|---------|
| **転送時の暗号化** | すべての認証フローに HTTPS/TLS | GDPR、SOC 2、PCI DSS |
| **セッションセキュリティ** | HttpOnly, Secure, SameSite クッキー | SOC 2、OWASP |
| **CSRF 対策** | トークンベース + SameSite | SOC 2、OWASP |
| **データ最小化** | メール、名前、プロバイダー ID のみ保存 | GDPR |
| **監査ログ** | タイムスタンプ付きの認証イベントログ | SOC 2、GDPR |
| **アクセス制御** | ロールベース（OWNER、MEMBER） | SOC 2、ISO 27001 |
| **入力検証** | すべての入力を検証・サニタイズ | OWASP、SOC 2 |

### volta が後のフェーズに延期するもの

| 要件 | フェーズ | 延期理由 |
|------|---------|---------|
| SOC 2 認証 | Phase 3 | 独立監査が必要（$50K〜$200K） |
| GDPR データエクスポート API | Phase 2 | アクセス権に専用エンドポイントが必要 |
| GDPR 削除 API | Phase 2 | 削除権にカスケード削除ロジックが必要 |
| ISO 27001 | Phase 3+ | 正式な ISMS が必要 |
| HIPAA | 予定なし | volta は医療製品ではない |

### セルフホストのコンプライアンス上の利点

volta はセルフホストのため、コンプライアンスの責任は共有されます：

```
  volta（ソフトウェアベンダー）：
    責任範囲：
    - セキュアなコード（脆弱性なし）
    - プライバシーバイデザイン（データ最小化）
    - 文書化（どのデータが保存され、どう流れるか）

  顧客（データコントローラー）：
    責任範囲：
    - インフラセキュリティ（ネットワーク、OS、ファイアウォール）
    - ユーザーとのデータ処理契約
    - 当局への侵害通知
    - volta を実行するサーバーへのアクセス制御
```

このモデルはコンプライアンスフレームワークでよく理解されています。顧客の SOC 2 監査がインフラをカバーし、volta は内部コンポーネントとして扱われます -- PostgreSQL や Nginx と同様です。

## よくある間違いと攻撃

### 間違い1：「うちは小さすぎてコンプライアンスは関係ない」

GDPR は規模に関係なく、EU 居住者のデータを処理するすべての企業に適用されます。EU ユーザーを持つ2人のスタートアップも GDPR の対象です。基本から始めましょう：プライバシーポリシー、データ最小化、暗号化。

### 間違い2：コンプライアンスシアター

実際にコントロールを実装せずにチェックボックスを埋める。「あなたのデータを保護します」というプライバシーポリシーを書きながら、パスワードを平文で保存する。監査人や規制当局はこれを見抜きます。

### 間違い3：コンプライアンスを一度きりのプロジェクトとして扱う

規制は変わります。製品も変わります。新機能は新しいコンプライアンス要件を導入するかもしれません。コンプライアンスは継続的なプロセスであり、期限付きのプロジェクトではありません。

### 間違い4：サードパーティのコンプライアンスを無視する

メールに SendGrid、決済に Stripe を使うなら、それらのコンプライアンス状況はあなたに影響します。プロセッサーとのデータ処理契約が必要で、コンプライアンス要件を満たしていることを確認しなければなりません。

### 間違い5：データの過剰収集

「このデータは後で必要になるかも」は GDPR のデータ最小化原則に違反します。現在の目的に必要なものだけ収集します。volta はメール、表示名、プロバイダー ID のみを保存 -- それ以上は何も保存しません。
