# 同時セッション数制限

[English version](concurrent-session-limit.md)

---

## これは何？

同時セッション数制限とは、一人のユーザーが同時に持てるアクティブなセッション数の上限のことです。動画配信サービスで「同時視聴は3台まで」というのと同じイメージです。4台目はブロックされるか、一番古い視聴が切断されます。

認証では、ノートPC・スマホ・タブレットでログイン中のユーザーが、4台目のデバイスでログインしようとした時に制限がかかります。

---

## なぜ重要？

**セキュリティ：** 攻撃者が認証情報を盗んでも、セッション制限があれば何十台もの端末でセッションを開けません。正規ユーザーは既存のセッションが切断されることで異常に気づきやすくなります。

**ライセンス管理：** B2B 製品ではシート単位で課金することがあります。無制限の同時セッションだと、1アカウントでチーム全体が使えてしまい、課金モデルが成り立ちません。

**リソース管理：** セッションごとにサーバーメモリとデータベース行を消費します。一人のユーザーからの無制限セッションはリソースの無駄遣いになります。

### N+1 回目のログイン時にどうなる？

主に2つの戦略があります：

| 戦略 | 動作 | UX への影響 |
|------|------|-----------|
| **新規ログインをブロック** | 「セッション数が上限です。他の端末でログアウトしてください」 | ストレスが大きい。どこでログインしてたか覚えていないことも |
| **最古のセッションを追い出す** | 一番古いセッションが切断され、新しいログインが通る | スムーズだが、古い端末で突然切れるのは驚く |

最近の製品では**最古セッション追い出し**方式が主流です。ストレスが少ないためです。

## 簡単な例

```
セッション1: ノートPC（09:00 作成）
セッション2: スマホ（10:00 作成）
セッション3: タブレット（11:00 作成）
セッション4: 会社PC（12:00 作成）
セッション5: 自宅PC（13:00 作成）

制限 = 5。全5つがアクティブ。

セッション6: カフェのノートPC（14:00 作成）
  -> セッション1（ノートPC）が追い出される。その端末では再ログインが必要。
```

## volta-auth-proxy では

volta は現在、同時セッション数制限を**設けていません**。`sessions` テーブルはユーザーごとに無制限のアクティブセッションを許可しています。これは Phase 1 の SaaS 製品でシンプルさを優先する一般的なアプローチです。

セッション制限を追加する場合（目標：ユーザーあたり5セッション）、実装は以下のようになります：

1. ログイン時に、そのユーザーのアクティブセッション数をカウント
2. カウントが5以上なら、最も古いセッションを無効化（`invalidated_at = NOW()`）
3. 新しいセッションを通常通り作成

`sessions` テーブルには既に `created_at` と `invalidated_at` があるため、追加は容易です。

**UX の考慮：** セッションが追い出された場合、そのデバイスのユーザーには「別のデバイスでログインしたため、サインアウトされました」という明確なメッセージを表示すべきです。volta のエラーページシステムは `SESSION_REVOKED` のようなカスタムエラーコードを既にサポートしており、この用途に活用できます。
