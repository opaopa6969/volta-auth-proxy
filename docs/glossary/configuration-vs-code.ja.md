# 設定 vs コード（Configuration vs Code）

[English version](configuration-vs-code.md)

---

## これは何？

「設定」と「コード」の境界線は、デプロイメントごとに変わる値（設定）とシステムの動作を定義するロジック（コード）の区別です。設定は「何を」 -- どのデータベースに接続するか、どのドメインを使うか。コードは「どのように」 -- セッションをどう検証するか、JWTをどう署名するか。

車の運転に例えるとわかりやすいです。設定は各種セッティング：シートの位置、ミラーの角度、ラジオの局、GPSの目的地。整備士でなくても変更できます。コードはエンジン、トランスミッション、ブレーキ -- 車が実際にどう動くかの仕組みです。ラジオ局の設定がブレーキシステムに影響することは望みません。そしてラジオ局を変えるたびにエンジンを組み直すことも絶対に望みません。

---

## 境界線はどこか？

純粋な設定から純粋なコードまでのスペクトラムがあります：

```
純粋な設定 ◄─────────────────────────────────► 純粋なコード

.envシークレット  YAML設定      DSLルール     アプリケーションロジック
DATABASE_URL     allowed_roles  状態マシン     Javaのif/else
API_KEY          アプリURL      ガード式       アルゴリズム
                                              エラー処理

「デプロイメント  「機能ごとに   「設計ごとに   「機能ごとに
 ごとに変わる」    変わる」       変わる」       変わる」

コンパイラ       コンパイラ     コンパイラ     コンパイラが
チェックなし     チェックなし   チェックなし   エラーを検出
                               （まだ）
```

左に行くほどシンプルだが非力。右に行くほど強力だが複雑。危険ゾーンは中間：設定が複雑になりすぎて実質的にプログラミングになっているのに、本物のプログラミングが提供するセーフティネットがない状態。

## voltaの姿勢：3層、明確な境界

voltaは境界を明確に引きます：

### 層1：.env -- デプロイメントごとに異なるシークレット

```bash
DATABASE_URL=jdbc:postgresql://localhost:5432/volta_auth
GOOGLE_CLIENT_ID=your-google-client-id
GOOGLE_CLIENT_SECRET=your-google-client-secret
SESSION_SECRET=your-random-secret
BASE_URL=https://auth.example.com
```

これらは純粋な設定です。ロジックは含まれません。ノートPC、ステージングサーバー、本番サーバーで異なります。間違った値は明確なエラー（「データベースに接続できない」）を引き起こし、微妙な動作の変化ではありません。

### 層2：YAML -- システムが保護するものを定義する設定

```yaml
apps:
  - id: app-wiki
    url: https://wiki.example.com
    allowed_roles: [MEMBER, ADMIN, OWNER]
  - id: app-admin
    url: https://admin.example.com
    allowed_roles: [ADMIN, OWNER]
```

これらは構造化された設定です。どのアプリが存在し、誰がアクセスできるかを定義します。命令的（「ロールがADMINなら許可する」）ではなく宣言的（「このアプリはこれらのロールを許可する」）です。アプリの追加は3行の追加であり、新しいコードの記述ではありません。

### 層3：Java -- システムの動作を定義するロジック

```java
// セッションタイムアウト：8時間、スライディングウインドウ
private static final Duration SESSION_TIMEOUT = Duration.ofHours(8);

// JWTアルゴリズム：RS256のみ
private static final String JWT_ALGORITHM = "RS256";

// 最大同時セッション数：ユーザーあたり5
private static final int MAX_SESSIONS = 5;
```

これらはコードです。コードベースに埋め込まれた意見のある決定です。変更するにはソースコードを修正します。つまり以下が得られます：

- コンパイラチェック（タイポが検出される）
- 型安全性（タイムアウトを「バナナ」に設定できない）
- テストカバレッジ（動作がテストされている）
- Git履歴（誰がなぜ変更したか見える）
- コードレビュー（変更が誰かにレビューされる）

## 設定が多すぎるとコンパイラなしのプログラミングになる理由

これが核心的な洞察です。Keycloakのrealm.jsonを考えてみましょう：

```json
{
  "bruteForceProtected": true,
  "maxFailureWaitSeconds": 900,
  "minimumQuickLoginWaitSeconds": 60,
  "waitIncrementSeconds": 60,
  "quickLoginCheckMilliSeconds": 1000,
  "maxDeltaTimeSeconds": 43200,
  "failureFactor": 30
}
```

これは設定ではありません。JSONを通じてブルートフォース検出アルゴリズムをプログラミングしているのです。以下を指定しています：
- アルゴリズムの動作（指数バックオフ）
- タイミングパラメータ（待機時間、チェック間隔）
- 閾値（失敗係数、最大デルタ）

しかし実際のコードと違い、このJSONには：
- 間違いを検出するコンパイラがない（`failureFactor`を-1に設定しても誰も止めない）
- 型システムがない（`maxFailureWaitSeconds`を「yes」に設定、幸運を祈る）
- テストがない（JSONファイルをユニットテストする方法は？）
- IDEサポートがない（オートコンプリートなし、「定義に移動」なし）
- 意味のあるエラーメッセージがない（「無効なレルム設定」 -- どの部分？）

プログラミングをしているのに、プログラミングを安全にするツールが何もない状態です。

## voltaの答え：ロジックならコードに書く

voltaは「設定言語」を作ることを拒否します。設定は最小限です：

| 何 | どこ | なぜ |
|----|------|------|
| データベースURL | `.env` | デプロイメントごとに異なり、シークレット |
| OAuth認証情報 | `.env` | デプロイメントごとに異なり、シークレット |
| セッション署名キー | `.env` | デプロイメントごとに異なり、シークレット |
| ベースURL | `.env` | デプロイメントごとに異なる |
| アプリリスト + ロール | `volta-config.yaml` | 機能ごとに異なり、宣言的 |
| セッションタイムアウト | Javaコード | 設定ではなく設計上の決定 |
| JWTアルゴリズム | Javaコード | 好みではなくセキュリティ上の決定 |
| 最大セッション数 | Javaコード | 設定ではなく設計上の決定 |
| OIDCフローロジック | Javaコード | 設定ではなく複雑なロジック |

テスト：「この値を変更するにはシステムの動作を理解する必要があるか？」はいなら、コードに属します。いいえなら（デプロイの詳細にすぎない）、設定に属します。

`DATABASE_URL=postgres://...`はシステムの動作の理解を必要としません。誰でも設定できます。

`maxDeltaTimeSeconds=43200`はブルートフォース検出アルゴリズムの理解を絶対に必要とします。その理解を持つ人がコードの中で書き、テストし、レビューすべきです。

## volta-auth-proxyでは

voltaはデプロイメントごとに異なるシークレットに.envを、宣言的なアプリ設定にYAMLを、すべての動作ロジックにJavaコードを使用し、コンパイラ、テスト、コードレビューがロジックを保護できる場所にロジックが存在するよう、意図的に設定を最小限に保っています。
