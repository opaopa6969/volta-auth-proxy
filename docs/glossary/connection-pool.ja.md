# コネクションプール

[English version](connection-pool.md)

---

## これは何？

コネクションプールとは、リクエストごとに作成・破棄するのではなく、事前に開いたデータベース接続をリクエスト間で再利用するための接続セットです。新しいデータベース接続を開くのは遅い -- TCPハンドシェイク、認証、プロトコルネゴシエーションが必要です。コネクションプールはこの作業を一度だけ行い、接続をウォーム状態で待機させます。

空港のタクシー乗り場で考えてみましょう。プールがないと、すべての旅行者（リクエスト）はタクシー会社に電話し、車が来るのを待ち、乗車し、車は営業所に戻ります。プール（タクシー乗り場）があれば、タクシーの列がすでに待っています。乗り込み、目的地まで行き、タクシーはすぐに乗り場に戻って次の乗客を待ちます。プールはすぐ乗れるタクシーの待機列です。

volta-auth-proxyでは、コネクションプールはJava最速のコネクションプールライブラリであるHikariCPが管理しています。

---

## なぜ重要なのか？

- **パフォーマンス。** 新しいPostgreSQL接続を開くのに5-50msかかる。プール済みの接続を再利用するのは<1ms。すべてのリクエストで[ForwardAuth](forwardauth.ja.md)チェックが走るため、この差は膨大。
- **リソース管理。** データベースには処理できる最大接続数がある。プールなしでは、トラフィックスパイクで数千の接続が開かれPostgreSQLがクラッシュしかねない。
- **信頼性。** プールは接続障害、古い接続、再接続を自動的に処理する。
- **予測可能性。** 固定プールサイズは予測可能なリソース使用を意味し、キャパシティプランニングを可能にする。

## どう動くのか？

### コネクションプールなし

```
  リクエスト1: 接続を開く(30ms) → クエリ(2ms) → 閉じる(1ms)
  リクエスト2: 接続を開く(30ms) → クエリ(2ms) → 閉じる(1ms)
  リクエスト3: 接続を開く(30ms) → クエリ(2ms) → 閉じる(1ms)

  合計オーバーヘッド: 実作業6msに対して接続セットアップ90ms
```

### コネクションプールあり

```
  起動時: 10接続を開く（合計300ms、一度きりのコスト）

  リクエスト1: 接続を借りる(<0.1ms) → クエリ(2ms) → 返却(<0.1ms)
  リクエスト2: 接続を借りる(<0.1ms) → クエリ(2ms) → 返却(<0.1ms)
  リクエスト3: 接続を借りる(<0.1ms) → クエリ(2ms) → 返却(<0.1ms)

  合計オーバーヘッド: プールなしの90msに対して約0.3ms
```

### プールのライフサイクル

```
  ┌────────────────────────────────────────────────┐
  │               コネクションプール                  │
  │                                                  │
  │  ┌──────────────────────────────────────────┐   │
  │  │  利用可能な接続（アイドル）                 │   │
  │  │                                          │   │
  │  │  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐    │   │
  │  │  │接続│ │接続│ │接続│ │接続│ │接続│    │   │
  │  │  │ 1  │ │ 2  │ │ 3  │ │ 4  │ │ 5  │    │   │
  │  │  └────┘ └────┘ └────┘ └────┘ └────┘    │   │
  │  └──────────────────────────────────────────┘   │
  │                                                  │
  │  ┌──────────────────────────────────────────┐   │
  │  │  使用中の接続（アクティブ）                 │   │
  │  │                                          │   │
  │  │  ┌────┐ ┌────┐ ┌────┐                   │   │
  │  │  │接続│ │接続│ │接続│  ← リクエストを     │   │
  │  │  │ 6  │ │ 7  │ │ 8  │    処理中          │   │
  │  │  └────┘ └────┘ └────┘                   │   │
  │  └──────────────────────────────────────────┘   │
  │                                                  │
  │  プールサイズ: 10 │ アクティブ: 3 │ アイドル: 5  │
  │  最小: 5          │ 最大: 10                     │
  └────────────────────────────────────────────────┘
```

### HikariCPがプールを管理する方法

```
  リクエストが来る:
       │
       ▼
  アイドル接続はあるか？
       │
  ┌────┴────┐
  │         │
  はい      いいえ
  │         │
  ▼         ▼
  借りる    プールは最大サイズ？
  (<0.1ms)  │
            ├── いいえ → 新しい接続を作成
            │          リクエストに返す
            │
            └── はい → 待つ（connectionTimeoutまで）
                      │
                 ┌────┴────┐
                 │         │
              接続が      タイムアウト
              返却された   超過
                 │         │
                 ▼         ▼
              使用      SQLException
                       をスロー
```

### HikariCPの主要設定

| 設定 | デフォルト | 意味 |
|------|----------|------|
| `maximumPoolSize` | 10 | プール内の最大接続数 |
| `minimumIdle` | maxと同じ | ウォーム維持する最小アイドル接続数 |
| `connectionTimeout` | 30000ms | 接続を待つ最大時間 |
| `idleTimeout` | 600000ms | アイドル接続の生存時間 |
| `maxLifetime` | 1800000ms | 接続の最大寿命 |
| `leakDetectionThreshold` | 0（オフ） | 接続が時間内に返されない場合に警告 |

## volta-auth-proxy ではどう使われている？

### HikariCP設定

voltaはHikariCPを使ってPostgreSQL接続を管理します：

```java
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:postgresql://localhost:5432/volta");
config.setUsername("volta");
config.setPassword("secret");
config.setMaximumPoolSize(10);
config.setMinimumIdle(5);
config.setConnectionTimeout(5000);  // 5秒
config.setLeakDetectionThreshold(10000);  // 10秒

HikariDataSource dataSource = new HikariDataSource(config);
```

### voltaでの接続使用

すべてのデータベース操作は接続を借りて返却します：

```java
// セッション検索（すべてのForwardAuthチェックで発生）
try (Connection conn = dataSource.getConnection()) {
    // プールから接続を借りる
    PreparedStatement ps = conn.prepareStatement(
        "SELECT * FROM sessions WHERE token_hash = ?"
    );
    ps.setString(1, tokenHash);
    ResultSet rs = ps.executeQuery();
    // ... 結果を処理 ...
}  // 接続は自動的にプールに返却
```

`try-with-resources`ブロックが例外発生時も接続が返却されることを保証します。

### voltaのプールサイジング

```
  プールサイズの検討:

  ForwardAuthチェック: クエリあたり約2ms
  ピークリクエスト/秒: 約500（例）

  必要接続数 = リクエスト/秒 × クエリ時間（秒）
             = 500 × 0.002
             = 1接続

  ただし: 他のクエリも実行（ユーザー検索、テナント確認）
  さらに: 負荷時にクエリが長くなることも

  経験則:
  ┌─────────────────────────────────┐
  │ 接続数 = (2 × CPUコア数)        │
  │        + ディスク数              │
  │                                 │
  │ 4コアサーバーの場合:             │
  │ 接続数 = (2 × 4) + 1 = 9       │
  │ 切り上げて10                     │
  └─────────────────────────────────┘
```

### Phase 2：複数インスタンスでのプールサイジング

voltaが水平スケールすると、各インスタンスが独自のプールを持ちます：

```
  3 voltaインスタンス × 各10接続 = 合計30接続

  PostgreSQLデフォルトmax_connections: 100
  30接続 = 安全

  ただし: 他のツールも接続（Flyway、監視、管理）
  予約: volta以外の接続に約20
  利用可能: voltaに80

  最大インスタンス: 80 ÷ 10 = 8 voltaインスタンス
```

## よくある間違いと攻撃

### 間違い1：接続リーク

コードが接続を借りて返却しない（`close()`忘れやtry-with-resources未使用）と、プールが徐々に枯渇します。`leakDetectionThreshold`を有効にして検出しましょう。

### 間違い2：プールが大きすぎ

接続が多ければ良いわけではない。PostgreSQLの各接続は約10MBのRAMを使います。4 CPUコアのサーバーでプール100接続は、99接続がどのみち待っています。上記の式を使いましょう。

### 間違い3：プールが小さすぎ

プールが小さすぎると、リクエストが接続待ちの行列に並びます。`connectionTimeout`が短いとSQL例外が発生します。プールのアクティブ数vs合計数を監視しましょう。

### 間違い4：水平スケーリングを考慮しない

PostgreSQLの`max_connections`を増やさずにvoltaインスタンスを追加すると、「接続が多すぎます」エラーが発生します。すべてのインスタンスにわたってプールサイズを計画しましょう。

### 間違い5：長時間のトランザクション

30秒間接続を保持するトランザクションは、その接続を他のすべてのリクエストからブロックします。特にForwardAuthパスではトランザクションを短く保ちましょう。
