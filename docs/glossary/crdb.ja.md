# CockroachDB（CRDB）

[English version](crdb.md)

---

## これは何？

CockroachDB（よくCRDBと略される）は、障害を乗り越え、複数のマシン、データセンター、さらには大陸をまたいで水平にスケールするよう設計された分散SQLデータベースです。PostgreSQLワイヤープロトコルを話すため、Postgresと通信するアプリケーションは最小限の変更でCockroachDBと通信できることが多いです。

銀行のチェーン店のようなものです。1つの支店が火事になっても、お金は他の支店に複製されているから安全です。東京に新しい支店を開けば、日本のお客さんはより速くサービスを受けられます。CockroachDBはデータに対してこれを行います：ノード間で自動的に複製し、障害を乗り越え、最も近い場所からクエリに応答します。

名前はゴキブリ（cockroach）に由来します -- 殺すのが非常に難しいことで有名な生物です。このデータベースも同様に落としにくいように設計されています。

---

## なぜ重要なのか？

CockroachDBが認証の世界で重要なのは、ZITADELという有名なオープンソースのアイデンティティプラットフォームがプライマリデータベースとして使用しているからです。ZITADELを評価するとき、依存関係としてCockroachDBに出会います。CRDBを理解すると、ZITADELのデプロイメントが見た目より重い理由、そしてvolta-auth-proxyがプレーンなPostgreSQLを選んだ理由が分かります。

---

## CockroachDBの強み

| 強み | 詳細 |
|------|------|
| **障害に耐える** | ノードを1つ殺しても、クラスタは動き続ける。手動フェイルオーバー不要。 |
| **水平スケーリング** | ノードを追加してデータとクエリの容量を増やす。アプリにシャーディングロジック不要。 |
| **地理分散** | リージョン間でデータを複製。ユーザーは最も近いノードにアクセス。 |
| **SQL互換** | 標準SQL、PostgreSQLワイヤープロトコル。慣れたツールが使える。 |
| **直列化可能分離** | デフォルトで最も強い一貫性モデル。ダーティリードもファントムリードもなし。 |
| **マスターノードなし** | すべてのノードが読み書きを処理。単一障害点がない。 |

大陸をまたいで数百万ユーザーにグローバルサービスを提供する企業にとって、CockroachDBは本当の問題を解決します。

## ZITADELがCockroachDBを使う理由

ZITADELは、数百万ユーザーにスケールできるクラウドネイティブなアイデンティティプラットフォームとして設計されています。そのアーキテクチャは以下を前提としています：

- 複数リージョンにまたがる複数インスタンス
- 高可用性（認証にダウンタイムなし）
- 設定に応じた結果整合性または強整合性の読み取り
- アイデンティティデータモデルのイベントソーシング

CockroachDBはこのアーキテクチャに合致します。分散合意とマルチリージョンレプリケーションを標準で提供するからです。

```
  ZITADELアーキテクチャ：
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ ZITADEL      │  │ ZITADEL      │  │ ZITADEL      │
  │ (US-East)    │  │ (EU-West)    │  │ (AP-Tokyo)   │
  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
         │                 │                 │
  ┌──────▼─────────────────▼─────────────────▼───────┐
  │            CockroachDB クラスタ                    │
  │  ┌─────┐   ┌─────┐   ┌─────┐   ┌─────┐          │
  │  │Node1│   │Node2│   │Node3│   │Node4│  ...      │
  │  └─────┘   └─────┘   └─────┘   └─────┘          │
  │  （データはノード間で自動的に複製）                  │
  └──────────────────────────────────────────────────┘
```

これは強力です。同時に複雑です。

## voltaがプレーンなPostgreSQLを使う理由

### 1. voltaに分散合意は不要

volta-auth-proxyは中小規模のSaaSアプリケーション向けに設計されています。単一のPostgreSQLインスタンスで数千の同時セッションを楽々処理できます。典型的なvoltaのデプロイメント：

```
  voltaアーキテクチャ：
  ┌──────────────┐
  │ volta-auth-  │
  │ proxy        │
  └──────┬───────┘
         │
  ┌──────▼───────┐
  │ PostgreSQL   │
  │ （単一       │
  │  インスタンス）│
  └──────────────┘
```

2つのコンポーネント。これがデプロイメントの全体です。クラスタ管理なし、ノードディスカバリなし、分散合意なし。

### 2. 運用のシンプルさ

CockroachDBは印象的なテクノロジーですが、運用するということは：

| タスク | PostgreSQL | CockroachDB |
|--------|-----------|-------------|
| インストール | `apt install postgresql`またはDocker一行 | マルチノードクラスタのセットアップ |
| バックアップ | `pg_dump` | 分散バックアップの調整 |
| モニタリング | pgAdmin、標準ツール | CockroachDB固有のメトリクス |
| アップグレード | 標準パッケージアップグレード | ノード間のローリングアップグレード |
| トラブルシューティング | 30年以上のコミュニティ知識 | 若いコミュニティ、部族的知識が少ない |
| チームの知識 | ほぼすべてのバックエンド開発者がPostgresを知っている | CRDBの専門知識は稀 |

2-5人の開発者チームがSaaSを構築するとき、PostgreSQLは「みんなが既に知っているデータベース」を意味します。CockroachDBは「誰かが学ぶ必要があるデータベース」を意味します。

### 3. コスト

小さなVPSの単一PostgreSQLインスタンスは月額$5-20。CockroachDBクラスタ（適切なレプリケーションには最低3ノード）は、コンピュートと運用の注意の両面で大幅にコストが高くなります。

### 4. voltaのスケーリング戦略

voltaがパフォーマンスを必要とするとき、道筋は：

1. **垂直スケーリング：** より大きなPostgresインスタンス。10万ユーザー以上まで大半のSaaSユースケースを処理。
2. **リードレプリカ：** PostgreSQLネイティブのストリーミングレプリケーションで読み取りスケーリング。
3. **Redisセッション：** voltaは既にRedisセッションストレージ（`SESSION_STORE=redis`）をサポートしており、最もホットなクエリをオフロード。

これは退屈なテクノロジーです。退屈は良いことです。退屈とは、予測可能で、よく理解されていて、午前3時にデバッグできることを意味します。

## CockroachDBが適している場合

- 複数リージョンにまたがるグローバル分散サービスを構築している
- 自動フェイルオーバー付きのゼロダウンタイムデプロイメントが必要
- データ量が単一のPostgreSQLインスタンスで処理できる範囲を超える
- 分散システムに経験豊富な運用チームがいる

---

## PostgreSQLがより適している場合

- サービスが単一リージョンで動作する（大半のSaaSアプリケーション）
- チームが小さく、運用の複雑さを最小限にする必要がある
- ツール、拡張機能、コミュニティサポートの最大のエコシステムが欲しい
- 実際には必要ないかもしれない理論上のスケーラビリティよりシンプルさを重視
