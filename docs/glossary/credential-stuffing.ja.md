# クレデンシャルスタッフィング

[English / 英語](credential-stuffing.md)

---

## これは何？

クレデンシャルスタッフィングは、あるウェブサイトから漏洩したユーザー名/パスワードの組み合わせを別のウェブサイトで試す攻撃です。多くの人が複数のサービスで同じパスワードを使い回しているという事実を悪用します。ブルートフォース（ランダムに推測）と異なり、クレデンシャルスタッフィングはどこかで実際に使われている本物の認証情報を使います -- ただしターゲットサイトでも使えるとは限りません。

---

## なぜ重要？

大規模なデータ侵害は定期的に起きています。数十億のメール/パスワードの組み合わせがダークウェブで入手可能です。侵害されたゲームサイトと会社の管理パネルで同じパスワードを使っていたら、攻撃者は漏洩した認証情報で管理パネルにログインできます。ハッキングスキルは不要 -- 漏洩した各ペアをターゲットのログインページに試す自動化ツールだけです。

クレデンシャルスタッフィングはアカウント乗っ取りの最も一般的な原因の一つです。ユーザー側の主な防御策はユニークなパスワード（理想的にはパスワードマネージャー経由）。サーバー側の主な防御策はレート制限、異常検知、多要素認証です。

---

## 簡単な例

```
1. サイト A が侵害される。攻撃者が入手：
   alice@example.com : P@ssw0rd123
   bob@example.com   : qwerty2024

2. 攻撃者がサイト B に対して自動化ツールを実行：
   POST /login  body: {email: "alice@example.com", password: "P@ssw0rd123"}
   -> 200 OK（Alice はパスワードを使い回していた！）

   POST /login  body: {email: "bob@example.com", password: "qwerty2024"}
   -> 401 Unauthorized（Bob はサイト B で別のパスワードを使っていた）
```

攻撃者はサイト B を「ハッキング」していません。サイト A から漏洩した Alice のパスワードを再利用しただけです。

---

## volta-auth-proxy での使い方

volta はパスワードベースのログインがないため、クレデンシャルスタッフィングに対して本質的に耐性があります。ユーザーは Google OIDC を通じてのみ認証します：

- **パスワード保存なし**: volta はパスワードを保存も受け付けもしません。メール/パスワードを受け取る `/login` エンドポイントは存在しません。他サイトから漏洩したパスワードは無関係です
- **Google が認証を担当**: 実際のパスワードチェックは Google のサーバーで行われ、独自の高度なスタッフィング対策（CAPTCHA、リスク分析、不審なログインアラート）を備えています
- **Google 経由の MFA**: ユーザーが Google アカウントで2段階認証を有効にしていれば、パスワードが正しくてもクレデンシャルスタッフィングはブロックされます

それでも volta は関連する攻撃への防御を適用しています：

- **レート制限**: IP ごとに毎分200リクエストの制限で、自動化ツールが volta の OIDC フローに対して漏洩した認証情報を高速で試すことを防止
- **監査ログ**: すべての認証イベント（ログイン、ログアウト、失敗）が IP とユーザーエージェント付きで記録され、1つの IP が多数の異なるアカウントを試すパターンを検出可能
- **セッション制限**: ユーザーごとに最大5つの同時セッション。攻撃者がアクセスを得ても、多数のセッションを無音で維持できない

教訓：認証を大手 ID プロバイダ（Google）に委譲することで、クレデンシャル防御の最も難しい部分を、はるかに多くのリソースを持つチームに任せることができます。
