# Crypto-Random

[日本語版はこちら](crypto-random.ja.md)

---

## What is it?

Crypto-random (cryptographically secure random) refers to random numbers generated from a source that is unpredictable, even to an attacker with significant computational resources. Unlike regular random numbers (which are actually pseudo-random and can be predicted if you know the seed), crypto-random numbers are suitable for security-critical operations like generating session IDs, [encryption](encryption.md) keys, [nonces](nonce.md), and invite codes.

Think of it like the difference between a dice roll and a coin flip pattern. If someone watches you roll a weighted die 100 times, they might start predicting the next roll. But imagine a truly fair die, in a sealed box, with no way to observe or influence the roll -- that is crypto-random. Nobody can predict the next value, no matter how many previous values they have seen.

In Java, `Math.random()` and `new Random()` are NOT crypto-random. They use algorithms that produce predictable sequences if the seed is known. `SecureRandom` IS crypto-random -- it draws entropy from hardware noise, operating system events, and other unpredictable sources.

---

## Why does it matter?

If random values are predictable, an attacker can:

- **Guess session IDs**: If the session ID generator is predictable, an attacker can compute other users' session IDs and hijack their sessions.
- **Predict CSRF tokens**: Predictable CSRF tokens can be generated by the attacker, defeating CSRF protection.
- **Forge nonces**: If [nonces](nonce.md) are predictable, replay protection fails.
- **Recover encryption keys**: If the randomness source for key generation is weak, the key space is effectively smaller, making brute-force feasible.

The security of every random value in volta -- session IDs, invite codes, PKCE verifiers, CSRF tokens, key IDs, AES initialization vectors -- depends on crypto-random generation.

## How does it work?

### Entropy sources

```
  Hardware / OS entropy:
  ┌──────────────────────────────────────────────────┐
  │  Sources of unpredictability:                    │
  │  ├── CPU timing jitter                          │
  │  ├── Disk I/O timing                            │
  │  ├── Network packet arrival times               │
  │  ├── Mouse/keyboard events                      │
  │  ├── Hardware RNG (Intel RDRAND)                │
  │  └── Thermal noise                              │
  │                                                  │
  │  OS collects entropy → /dev/urandom (Linux)     │
  │  JVM reads from OS → java.security.SecureRandom │
  └──────────────────────────────────────────────────┘
```

### Random vs. SecureRandom

```
  java.util.Random (NOT secure):
  ┌──────────────────────────────────────────────┐
  │  seed = System.nanoTime()  (or user-provided)│
  │  next = (seed * 0x5DEECE66DL + 0xBL)        │
  │         & 0xFFFFFFFFFFFFL                    │
  │                                              │
  │  Problem: Given a few outputs, the entire    │
  │  sequence can be reconstructed.              │
  │  NEVER use for security!                     │
  └──────────────────────────────────────────────┘

  java.security.SecureRandom (SECURE):
  ┌──────────────────────────────────────────────┐
  │  Entropy from OS hardware sources            │
  │  CSPRNG algorithm (e.g., SHA1PRNG, DRBG)    │
  │  Periodically re-seeded from OS entropy      │
  │                                              │
  │  Even with billions of outputs observed,     │
  │  the next output is unpredictable.           │
  │  REQUIRED for all security operations.       │
  └──────────────────────────────────────────────┘
```

### Output quality comparison

```
  Predictable (java.util.Random):
  ┌─────────────────────────────────┐
  │  Output 1: 42                   │
  │  Output 2: 17                   │  An attacker who sees
  │  Output 3: 89                   │  these 3 outputs can
  │  Output 4: ???                  │  predict output 4.
  └─────────────────────────────────┘

  Unpredictable (SecureRandom):
  ┌─────────────────────────────────┐
  │  Output 1: a7f2c8e1            │
  │  Output 2: 3b9d4f2a            │  Even seeing millions
  │  Output 3: 8c1e5d7b            │  of outputs, output 4
  │  Output 4: ???                  │  is unpredictable.
  └─────────────────────────────────┘
```

## How does volta-auth-proxy use it?

volta uses `SecureRandom` everywhere randomness is needed:

### 1. Random URL-safe strings

`SecurityUtils.randomUrlSafe()` generates crypto-random bytes and Base64-URL encodes them:

```java
private static final SecureRandom RANDOM = new SecureRandom();

public static String randomUrlSafe(int byteLength) {
    byte[] bytes = new byte[byteLength];
    RANDOM.nextBytes(bytes);
    return Base64.getUrlEncoder().withoutPadding().encodeToString(bytes);
}
```

Used for: PKCE verifiers, [state](state.md) parameters, CSRF tokens.

### 2. Invite codes

```java
public static String inviteCode() {
    return randomUrlSafe(24);  // 24 bytes = 192 bits of entropy
}
```

24 bytes of `SecureRandom` produce 192 bits of entropy. An attacker would need to try 2^192 possibilities to guess an invite code -- more than the number of atoms in the observable universe.

### 3. Session IDs (UUIDs)

```java
public static UUID newUuid() {
    return UUID.randomUUID();  // Uses SecureRandom internally
}
```

Java's `UUID.randomUUID()` produces version-4 UUIDs with 122 bits of randomness from `SecureRandom`.

### 4. AES initialization vectors

`KeyCipher` generates 12-byte random IVs for AES-GCM encryption:

```java
private static final SecureRandom RANDOM = new SecureRandom();

public String encrypt(String plain) {
    byte[] iv = new byte[12];
    RANDOM.nextBytes(iv);  // Crypto-random IV
    Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
    cipher.init(Cipher.ENCRYPT_MODE, keySpec, new GCMParameterSpec(128, iv));
    // ...
}
```

A random IV ensures that encrypting the same plaintext twice produces different ciphertexts.

### 5. RSA key generation

```java
KeyPairGenerator generator = KeyPairGenerator.getInstance("RSA");
generator.initialize(2048);  // Uses SecureRandom by default
KeyPair keyPair = generator.generateKeyPair();
```

Java's `KeyPairGenerator` uses `SecureRandom` by default for all key material.

### Summary of crypto-random usage

```
  ┌───────────────────────────────────────────────────┐
  │  Component            Random source   Bits        │
  │  ─────────────────────────────────────────────── │
  │  Session ID (UUID)    SecureRandom    122 bits    │
  │  Invite code          SecureRandom    192 bits    │
  │  PKCE verifier        SecureRandom    256 bits    │
  │  State parameter      SecureRandom    256 bits    │
  │  CSRF token           SecureRandom    256 bits    │
  │  AES-GCM IV           SecureRandom    96 bits     │
  │  RSA key pair         SecureRandom    2048 bits   │
  │  Key ID (kid)         Timestamp       N/A (not    │
  │                       (not random)    a secret)   │
  └───────────────────────────────────────────────────┘
```

## Common mistakes and attacks

### Mistake 1: Using Math.random() for security values

```java
// WRONG - predictable
String sessionId = String.valueOf(Math.random());

// RIGHT - crypto-random
String sessionId = UUID.randomUUID().toString();
```

`Math.random()` uses a linear congruential generator. Given a few outputs, the entire sequence is recoverable.

### Mistake 2: Using new Random() with a timestamp seed

```java
// WRONG - predictable if attacker knows approximate time
Random rng = new Random(System.currentTimeMillis());

// RIGHT
SecureRandom rng = new SecureRandom();
```

A millisecond-precision timestamp has only ~30 bits of entropy. An attacker can try all possible seeds for a given time window.

### Mistake 3: Reusing IVs/nonces

Using the same IV with the same AES key breaks GCM security entirely. volta generates a fresh random IV for every encryption operation.

### Mistake 4: Insufficient entropy length

A 4-byte random session ID has only 2^32 (~4 billion) possibilities. An attacker can try all of them in seconds. Use at least 16 bytes (128 bits) for security tokens. volta uses 24 bytes (192 bits) for invite codes.

### Attack: Random number generator state recovery

If an attacker can observe enough outputs from a weak RNG (like `java.util.Random`), they can reverse-engineer the internal state and predict all future outputs. This is why `SecureRandom` re-seeds from hardware entropy periodically.
