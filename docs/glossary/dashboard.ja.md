# ダッシュボード（認証コンテキスト）

[English version](dashboard.md)

---

## これは何？

認証の世界で「ダッシュボード」とは、管理者が認証システムを設定・管理するWebベースのグラフィカルインターフェースです。設定ファイルの編集やコーディングの代わりに、ポイント&クリックで操作できます。

車のダッシュボードのようなものです。ボンネットを開けてエンジンケーブルを調整する代わりに、ボタン、ダイヤル、ゲージのあるダッシュボードがあります。ノブを回して温度を調整。ボタンを押してライトを点灯。ダッシュボードは複雑なものを制御するための視覚的インターフェースを提供します。

---

## なぜ重要なのか？

チームが認証ソリューションを評価するとき、「ダッシュボードはあるか？」はしばしば最初の質問の1つです。Auth0のダッシュボードは洗練されていることで有名です。Keycloakには広範な管理コンソールがあります。volta-auth-proxyには... YAMLファイルと環境変数があります。

これは意図的な選択であり、その理由を理解するにはGUI駆動とコード駆動の設定間のトレードオフについて考える必要があります。

## ダッシュボード駆動の設定（Auth0、Keycloak）

### Auth0のダッシュボード

Auth0は`manage.auth0.com`にWebダッシュボードを提供し、以下が可能です：

- アプリケーション（OAuth2クライアント）の作成と設定
- ソーシャルログインプロバイダーのオン/オフ切り替え
- 認証ルールの設定
- ユーザープロファイルとログの閲覧
- Organizationsのセットアップ
- API権限の管理

```
  Auth0ダッシュボード：
  ┌─────────────────────────────────────────────────┐
  │ [Auth0 ロゴ]                         チーム ▼   │
  ├─────────┬───────────────────────────────────────┤
  │         │                                       │
  │ Apps    │  アプリケーション: My SaaS             │
  │ APIs    │  ┌─────────────────────────────────┐  │
  │ Users   │  │ Client ID:  abc123...            │  │
  │ Rules   │  │ 許可URL: https://...             │  │
  │ Orgs    │  │ ソーシャル: [✓] Google [✗] GitHub │  │
  │ Logs    │  │ MFA: 有効                        │  │
  │ ...     │  └─────────────────────────────────┘  │
  │         │                                       │
  │         │  [変更を保存]                         │
  └─────────┴───────────────────────────────────────┘
```

これは視覚的で直感的で、始めやすいです。

### Keycloakの管理コンソール

Keycloakにはさらに多くのオプションを持つ組み込み管理コンソールがあります：

- レルム管理（各レルムがタブ）
- クライアントスコープ、プロトコルマッパー、認証フロー
- ユーザーフェデレーション、IDプロバイダー
- セッション管理、イベントログ

Keycloak管理コンソールには数十のページがあり、各ページに多くの設定があります。

## コード駆動の設定（volta）

volta-auth-proxyには設定用のWebダッシュボードがありません。すべての設定はファイルで行います：

**環境変数（.env）：**
```bash
DATABASE_URL=jdbc:postgresql://localhost:5432/volta_auth
GOOGLE_CLIENT_ID=your-google-client-id
GOOGLE_CLIENT_SECRET=your-google-client-secret
SESSION_SECRET=your-random-secret
BASE_URL=https://auth.example.com
```

**アプリケーション設定（volta-config.yaml）：**
```yaml
apps:
  - id: app-wiki
    url: https://wiki.example.com
    allowed_roles: [MEMBER, ADMIN, OWNER]
  - id: app-admin
    url: https://admin.example.com
    allowed_roles: [ADMIN, OWNER]
```

これが設定の全体です。Webインターフェースはありません。

（voltaにはメンバー、招待、セッションを管理するための**管理UI**はあります -- ただしこれは日常の運用用であり、システム設定用ではありません。）

## GUI vs コードベース設定のトレードオフ

| 観点 | ダッシュボード（GUI） | コード（ファイル） |
|------|---------------------|-------------------|
| **発見可能性** | 視覚的にオプションを探索 | 何が使えるか知る必要あり |
| **学習曲線** | 初期障壁が低い | ドキュメント読む必要あり |
| **バージョン管理** | gitで変更追跡が困難 | 自然なgit統合 |
| **再現性** | 「この7つを順番にクリック」 | `cp .env.example .env` |
| **自動化** | APIラッパーが必要 | 既にコード、スクリプト化可能 |
| **チームレビュー** | 「設定を変えた」（diff なし） | diff付きプルリクエスト |
| **災害復旧** | すべてを再クリック | `git checkout` + デプロイ |
| **複数環境** | 各環境を手動で設定 | 設定コピー、値を変更 |
| **監査証跡** | プロバイダーのログ次第 | Git履歴 |

### ダッシュボードの隠れたコスト

ダッシュボードは最初は簡単に感じます。でもこのシナリオを考えてください：

**新しい環境のセットアップ：**

Auth0のダッシュボードの場合：
1. Auth0ダッシュボードにログイン
2. 新しいアプリケーション作成（5クリック）
3. リダイレクトURI設定（コピー&ペースト、正しいことを祈る）
4. ソーシャル接続の切り替え（プロバイダーごとに3クリック）
5. ルール設定（ダッシュボード内のコードエディタ）
6. Organizationsセットアップ（多くのクリック）
7. API権限設定（さらに多くのクリック）
8. 何も見落としていないことを祈る

voltaの設定ファイルの場合：
1. `.env.example`を`.env`にコピー
2. 5つの値を入力
3. デプロイ

**壊れた環境の再現：**

Auth0の場合：クリックしたすべての設定を覚えている。またはJSONのエクスポート/インポート（本質的にコードベースの設定に余分なステップが加わったもの）。

voltaの場合：`git log`が行われたすべての変更を表示。`git revert`がミスを元に戻す。

## voltaがダッシュボードよりコードを選んだ理由

voltaの哲学は、本番システムにおいて**設定はコードの方が良い**というものです：

1. **コードはレビューできる。** チームメイトがプルリクエストで`.env`の変更をレビューできる。ダッシュボードのクリックを誰もレビューしない。

2. **コードは再現可能。** `cp .env.staging .env.production`で同一の環境を作成。ダッシュボード設定は手動で再作成する必要がある。

3. **コードはバージョン管理可能。** Gitがタイムスタンプと著者付きですべての変更を追跡。ダッシュボードの変更はログに残るかもしれないし残らないかもしれない。

4. **コードはシンプル。** voltaの全設定は1画面に収まる。「発見」するものがない。設定するものがほとんどないから。

5. **設定は日常的な活動であるべきではない。** voltaは一度セットアップする。その後は管理UIやAPIでユーザーとテナントを管理する -- 設定を通じてではなく。
