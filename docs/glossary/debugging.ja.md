# デバッグ（Debugging）

[English version](debugging.md)

---

## これは何？

デバッグとは、バグ -- ソフトウェアのエラーや予期しない動作 -- を見つけて修正するプロセスです。名前は1947年にコンピューターのリレーで実際の蛾が見つかった（真偽は定かではない）話に由来します。今日、「デバッグ」はソフトウェアが期待通りに動作しない理由を調査することを意味します。

探偵のようなものです。何かがうまくいかなかった（事件）。手がかりがあります：エラーメッセージ（目撃者の証言）、ログファイル（監視カメラ映像）、スタックトレース（指紋）。あなたの仕事は何が起きたかをつなぎ合わせ、犯人（バグ）を見つけ、修正する（正義を行う）ことです。手がかりの質が、これに5分かかるか5時間かかるかを決めます。

---

## 「一箇所でデバッグ」が重要な理由

デバッグで最も難しいのは、問題がどこにあるかを特定することです。場所がわかれば、修正は通常簡単です。探索が難しい部分です。

シングルプロセスシステムでは、「どこ」は常に1つのコードベースのどこかです：

```
シングルプロセスのデバッグ：
  1. エラー発生 → スタックトレースがSessionService.java:47を指す
  2. ファイルを開く → コードを読む → バグを理解
  3. 修正 → テスト実行 → デプロイ
  時間：数分から数時間
```

マルチサービスシステムでは、「どこ」が複数のコードベース、ネットワーク、設定のどこにでもあり得ます：

```
マルチサービスのデバッグ：
  1. エラー発生 → 「upstream connect error」
  2. どのサービス？ForwardAuth Serviceのログを確認 → 「session serviceへの接続タイムアウト」
  3. Session Serviceのログを確認 → 「データベースへの接続拒否」
  4. 待って、データベース？ネットワーク？DNS？コネクションプール？
  5. データベースを確認 → 問題なし
  6. ネットワークを確認 → ルーティングはOK
  7. DNSを確認 → あ、DNSキャッシュに古いエントリがあった
  8. DNS設定を修正 → 再起動 → 本当にそれが問題だったことを祈る
  時間：数時間から数日
```

マルチサービスの調査では、どのサービスのコードとも関係のないDNSの問題を見つけるまでに5つの異なるシステムをチェックする必要がありました。これはマイクロサービスアーキテクチャの現実です。

## 複数のサービスがデバッグを難しくする仕組み

### ログの相関

各サービスは独自のログを生成します。サービスをまたいで1つのリクエストを追跡するには、以下が必要です：

1. すべてのサービスを通過するユニークなリクエストID
2. 集中ログ集約（例：ELKスタック、Datadog）
3. すべてのサーバー間の時刻同期
4. 数千行のログを検索する忍耐

シングルプロセスでは、すべてのログが1つの場所に、順序通りに、同じタイムスタンプソースであります。

### エラーコンテキスト

シングルプロセスでエラーが発生すると、スタックトレースに完全な呼び出しチェーンが含まれます。元のリクエスト、通過したすべての関数、正確にどこで失敗したかが見えます。

エラーがネットワーク境界を越えると、コンテキストは失われます。「接続タイムアウト」と見えます -- しかしタイムアウトしたときにリモートサービスが何をしていたかは見えません。その情報を別のログ、別のシステム、別のツールで見つけに行く必要があります。

### 再現

シングルプロセスのバグを再現するには、プロセスをローカルで実行して同じリクエストを送ります。マルチサービスシステムのバグを再現するには、すべてのサービスを実行し、互いに通信するよう設定し、適切なネットワーク条件を整え、タイミング依存のバグが再び現れることを祈る必要があります。

## voltaのシングルプロセスの利点

voltaはすべての認証関連の問題を一箇所で調査できるよう設計されています：

```
voltaでの認証の問題：
  1. voltaのログを確認 → リクエスト、エラー、スタックトレースを見る
  2. スタックトレースが正確なコード行を指す
  3. ローカルで再現：volta + Postgresを実行、同じリクエストを送信
  4. ブレークポイントでデバッグ、コードをステップ実行、変数を検査
  5. 修正 → テスト → デプロイ

ログ集約は不要。
分散トレーシングは不要。
サービスメッシュのデバッグは不要。
「どのサービスが壊れている？」の推測は不要。
```

これは些細な利便性ではありません。認証システムを保守するソロ開発者や小さなチームにとって、「1つのログファイルを開く」と「4つのサービスのログを相関させる」の違いは、生産的な夕方と眠れない夜の違いです。

## 認証のデバッグはとりわけリスクが高い

認証のバグはUIバグとは異なります。CSSのずれは迷惑です。認証のバグは以下を意味する可能性があります：

- ユーザーがすべてのアプリケーションからロックアウト（可用性）
- 機密データへの不正アクセス（セキュリティ）
- テナント間のセッション漏洩（データ漏洩）
- 不正なロール割り当て（権限昇格）

これらのバグが発生したとき（必ず発生します）、迅速に見つけて修正する必要があります。認証ダウンタイムの1分1分が、すべての保護されたアプリケーションがアクセスできない1分です。デバッグの速度がシステムの可用性とセキュリティに直接影響します。

voltaのシングルプロセス設計は認証のバグをより速く見つけることを意味します。より速いデバッグはより短い障害を意味します。より短い障害はより幸せなユーザーとより少ないセキュリティインシデントを意味します。

## volta-auth-proxyでは

voltaはシングルプロセスとして動作し、ログイン障害からセッションバグ、JWTエラーまで、すべての認証の問題を分散トレーシングやマルチサービスログ相関なしに、1つのログ、1つのスタックトレース、1つのコードベースでデバッグできます。
