# Delegation（委譲）

[English version](delegation.md)

---

## これは何？

委譲とは、あるシステムが自分でやるべきタスクの責任を別のシステムに任せることです。すべてのアプリケーションが独自のログインページ、セッション管理、ロールチェックを実装する代わりに、すべてのタスクを1つの認証ゲートウェイに委譲します。アプリケーションはゲートウェイの判定を疑うことなく信頼します。

オフィスビルの受付を雇うようなものです。各フロアの各会社が独自の受付を雇って来訪者のIDをチェックすることもできます。しかし、ロビーに1人の受付を置いて全員をチェックする方がはるかに効率的（かつ安全）です。来訪者が5階に到着したとき、その会社はロビーの受付がすでに確認済みであると信頼します。その信頼こそが委譲です。

volta-auth-proxyでは、下流のアプリケーションが[ForwardAuth](forwardauth.ja.md)パターンを通じて、すべての認証と粗粒度の認可をvoltaに委譲します。アプリはパスワードを見ず、セッションを管理せず、入口でロールをチェックしません。voltaがそれをやります。

---

## なぜ重要なのか？

委譲がなければ、すべてのアプリが認証をゼロから実装する必要があります：

- **コードの重複**：10個のアプリは10個のログインフロー、10個のセッションストア、10個のバグ修正を意味する
- **一貫性のないセキュリティ**：アプリAはセキュアなCookieを使い、アプリBはlocalStorageを使う -- 攻撃者は最も弱いリンクを狙う
- **シングルログアウトなし**：アプリAからログアウトしてもアプリBからはログアウトされない
- **統一された監査証跡なし**：認証イベントが10の異なるログ形式に散らばる
- **オンボーディング地獄**：新しいアプリ開発者全員が認証の専門家にならなければならない

voltaへの委譲があれば：

- **実装は1つ**：認証ロジックはちょうど1か所に存在
- **一貫したセキュリティ**：同じCookieフラグ、同じセッションポリシー、同じCSRF保護
- **シングルログアウト**：voltaがセッションを無効化すれば、すべてのアプリが同時にアクセスを失う
- **統一された監査**：すべての認証イベントがvoltaの[監査ログ](audit-log.ja.md)を通る
- **開発者の自由**：アプリ開発者はビジネスロジックに集中し、認証は気にしない

## どう動くのか？

### 委譲パターン

```
  委譲なし：
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ App A    │  │ App B    │  │ App C    │
  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │
  │ │Login │ │  │ │Login │ │  │ │Login │ │
  │ │Session│ │  │ │Session│ │  │ │Session│ │
  │ │Roles │ │  │ │Roles │ │  │ │Roles │ │
  │ └──────┘ │  │ └──────┘ │  │ └──────┘ │
  │ ビジネス │  │ ビジネス │  │ ビジネス │
  │ ロジック │  │ ロジック │  │ ロジック │
  └──────────┘  └──────────┘  └──────────┘
  3倍の作業、3倍のバグ、3倍の攻撃面

  委譲あり：
  ┌──────────────────────────────────┐
  │ volta-auth-proxy                 │
  │ ┌──────┐ ┌──────┐ ┌──────┐     │
  │ │Login │ │Session│ │Roles │     │
  │ └──────┘ └──────┘ └──────┘     │
  └────────────┬─────────────────────┘
               │ X-Volta-*ヘッダー
  ┌────────────┼────────────┬────────────┐
  │ App A      │ App B      │ App C      │
  │ ビジネス   │ ビジネス   │ ビジネス   │
  │ ロジックのみ│ ロジックのみ│ ロジックのみ│
  └────────────┘────────────┘────────────┘
  1倍の作業、1倍のバグ、1倍の攻撃面
```

### 信頼境界

委譲は信頼境界を作ります。volta背後のアプリはvoltaが送る[ヘッダー](header.ja.md)を信頼します：

```
  信頼境界
  ─────────────────────────────────
  インターネット│ volta        │ 内部ネットワーク
               │              │
  ユーザー ───►│ 認証         │──► X-Volta-User-Id: uuid
               │ 認可         │──► X-Volta-Roles: ADMIN
               │ セッション管理│──► X-Volta-Tenant-Id: uuid
               │              │
  ─────────────────────────────────
  信頼できない │ ゲートウェイ │ 信頼済み（アプリはvoltaを信頼）
```

### 委譲されるもの vs されないもの

| voltaに委譲 | 委譲されない（アプリの責務） |
|------------|--------------------------|
| ログイン/ログアウト | ビジネスロジックの認可（「このユーザーはこの書類を編集できるか？」） |
| セッション管理 | アプリ固有のデータ |
| 粗粒度のRBAC | 細粒度の権限 |
| CSRF保護 | 入力バリデーション |
| JWT発行 | JWT検証（アプリはJWKS経由で検証） |
| 監査ログ（認証イベント） | 監査ログ（ビジネスイベント） |

## volta-auth-proxyではどう使われているか？

### ForwardAuth委譲

下流アプリへのすべてのリクエストはTraefikを通り、TraefikがvoltaにAuthを委譲します：

```
  ブラウザ → Traefik → volta (/auth/verify) → 200? → アプリ
                                              → 401? → /loginにリダイレクト
```

アプリは認証判定に参加しません。事前検証済みのIDヘッダーを受け取ります。

### 内部API委譲

アプリはvoltaの[内部API](internal-api.ja.md)経由でユーザー管理操作も委譲できます：

```
  アプリがチームメンバーを一覧する場合：
  ┌──────────┐                          ┌──────────────────┐
  │ アプリ    │── GET /api/v1/tenants/   │ volta-auth-proxy │
  │          │   {tid}/members          │                  │
  │          │   Authorization: Bearer  │  JWTを検証       │
  │          │   {user_jwt}      ──────►│  ロールをチェック │
  │          │                          │  DBをクエリ       │
  │          │◄── 200 [{members}] ──────│  データを返す     │
  └──────────┘                          └──────────────────┘
```

アプリは独自のユーザーデータベースを保持する代わりに、メンバー一覧をvoltaに委譲します。

### サービストークン委譲

バックグラウンドジョブと[サービストークン](service-token.ja.md)はマシン間の委譲を可能にします：

```java
// AuthService.java
if (token.startsWith("volta-service:")) {
    String provided = token.substring("volta-service:".length());
    if (config.serviceToken().equals(provided)) {
        return Optional.of(new AuthPrincipal(
            new UUID(0L, 0L),
            "service@volta.local",
            "service-token", ...));
    }
}
```

バックグラウンドジョブは静的なサービストークンを使ってID検証をvoltaに委譲します。

## よくある間違いと攻撃

### 間違い1：部分的な委譲

ログインをvoltaに委譲しつつ、アプリで独自のセッション管理を実装する。これで2つのセッションシステムが矛盾する可能性が生じます。各関心事についてすべてを委譲するか、しないかのどちらか。

### 間違い2：委譲先を信頼しない

アプリが毎リクエストでJWTを再検証し、さらにvoltaのAPIでユーザーのロールを二重確認すれば、委譲の意味がなくなります。X-Volta-*ヘッダーを信頼してください。そのために存在しています。

### 間違い3：委譲しすぎ

voltaは認証と粗粒度RBACを扱います。細粒度のビジネスルール（「このユーザーはこの特定の請求書を編集できるか？」）をvoltaに委譲すると、voltaがボトルネックとビジネスロジックの単一障害点になります。

### 攻撃：委譲ヘッダーの偽造

攻撃者がTraefikとvoltaを迂回してアプリに直接リクエストを送り、偽の`X-Volta-User-Id`ヘッダーを付ける。対策：アプリは内部Dockerネットワーク経由でのみ到達可能にし、インターネットに直接公開しない。
