# デプロイメント

[English version](deployment.md)

---

## これは何？

デプロイメントとは、完成したソフトウェアをサーバーに配置し、実際のユーザーがアクセスできるようにするプロセスです。デプロイ前は、アプリケーションはあなたのノートPCにしか存在しません。デプロイ後は、インターネットに接続されたマシン上で動きます。

自宅のキッチンで料理を作ること（開発）と、レストランをオープンすること（デプロイ）の違いだと考えてください。自宅のキッチンでは、あなただけが食べます。デプロイすると、レストランを準備し、「営業中」の看板を出し、一般の人が入れるようになります。キッチンからレストランまでには多くのステップがあります：テーブルの準備、スタッフの雇用、メニューの掲示、保健所の検査合格。デプロイメントはソフトウェアにおけるそれらすべてのステップです。

一般的なWebプロジェクトでは、デプロイとはビルドしたアプリケーション（JARファイル、Dockerイメージなど）を本番サーバーにコピーし、適切な環境変数で設定し、起動し、動き続けることを確認することです。

---

## なぜ重要なのか？

- **デプロイしなければ、誰もソフトウェアを使えない。** ノートPCの中に永遠にとどまる。
- **悪いデプロイはダウンタイムを引き起こす。** 壊れたバージョンをデプロイすると、実際のユーザーにエラーが表示される。
- **セキュリティはデプロイに依存する。** 設定ミスのサーバー、公開されたポート、漏洩したシークレットはデプロイのミス。
- **再現性が重要。** 同じものを2回デプロイして同じ結果が得られなければ、デバッグは悪夢になる。
- **デプロイの速度が俊敏性を決める。** 数分でデプロイできるチームは、数時間かかるチームより速く機能を提供できる。

## どう動くのか？

### デプロイメントパイプライン

一般的なデプロイは複数のステージを経由します：

```
  開発者のノートPC
       │
       ▼
  ┌──────────────────┐
  │  ソースコード管理   │  ← git push
  │  (GitHub等)        │
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │  CI/CDパイプライン  │  ← ビルド、テスト、パッケージ
  │  (GitHub Actions)  │
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │  アーティファクト   │  ← JARファイル、Dockerイメージ
  │  ストア             │
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │  本番サーバー       │  ← アプリが動くサーバー
  │                    │
  └──────────────────┘
```

### デプロイ戦略

| 戦略 | 仕組み | リスク |
|------|--------|--------|
| **ビッグバン** | 旧バージョン停止、新バージョン起動 | 切り替え中にダウンタイム |
| **ローリング** | インスタンスを一つずつ入れ替え | 一時的にバージョン混在 |
| **ブルーグリーン** | 新旧を並行稼働し、トラフィックを切り替え | リソースが2倍必要 |
| **カナリア** | まず5%のトラフィックを新バージョンへ | 複雑なルーティングが必要 |

### 主なデプロイアーティファクト

volta-auth-proxyのようなJavaプロジェクトでは、メインのアーティファクトは**fat JAR**です -- アプリケーションとすべての依存関係を含む単一ファイルです：

```bash
# fat JARをビルド
mvn clean package -DskipTests

# 結果
target/volta-auth-proxy-1.0.0.jar   # 約30MB、すべて含まれる

# サーバー上で実行
java -jar volta-auth-proxy-1.0.0.jar
```

### 環境別の設定

同じJARがすべての環境で動きます。変わるのは設定だけです：

```
  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │   開発       │   │ ステージング  │   │   本番       │
  │              │   │              │   │              │
  │ DB: localhost│   │ DB: staging  │   │ DB: prod-db  │
  │ PORT: 8080  │   │ PORT: 8080   │   │ PORT: 8080   │
  │ LOG: debug  │   │ LOG: info    │   │ LOG: warn    │
  │ GOOGLE_ID:  │   │ GOOGLE_ID:   │   │ GOOGLE_ID:   │
  │  test-xxx   │   │  stage-xxx   │   │  prod-xxx    │
  └─────────────┘   └─────────────┘   └─────────────┘
        │                 │                  │
        └────────────┬────┘──────────────────┘
                     │
              同じJARファイル
```

設定はハードコードではなく、[環境変数](environment-variable.ja.md)で注入します。

## volta-auth-proxy ではどう使われている？

### 現在のデプロイモデル（Phase 1）

volta-auth-proxyは[シングルプロセス](single-process.ja.md)のJavaアプリケーションで、[Traefik](reverse-proxy.ja.md)リバースプロキシの後ろにfat JARとしてデプロイされます：

```
  インターネット
     │
     ▼
  ┌──────────┐
  │ Traefik   │  ← TLS終端、ルーティング
  └────┬─────┘
       │
       ▼
  ┌──────────────────┐
  │ volta-auth-proxy  │  ← 単一JVMプロセス
  │ (JVM上のfat JAR)  │
  └────────┬─────────┘
       │
       ▼
  ┌──────────┐
  │ Postgres  │  ← データベース
  └──────────┘
```

### voltaをデプロイするために必要なもの

1. **Java 21 LTS** ランタイムがサーバーにインストールされていること（[lts.ja.md](lts.ja.md)参照）
2. **PostgreSQL** データベースに[マイグレーション](migration.ja.md)が適用されていること
3. **環境変数** が設定されていること（Google OIDC認証情報、DB接続、JWT鍵）
4. **Traefik** が[ForwardAuth](forwardauth.ja.md)でvoltaを指すよう設定されていること
5. **DNS** レコードがドメインをサーバーに向けていること

### Phase 2のデプロイ（将来）

Phase 2では[Redis](redis.ja.md)と[水平スケーリング](horizontal-scaling.ja.md)が追加され、デプロイ構成が変わります：

```
  インターネット
     │
     ▼
  ┌──────────────┐
  │ ロードバランサ │
  └──┬─────┬─────┘
     │     │
     ▼     ▼
  ┌─────┐ ┌─────┐
  │volta│ │volta│  ← 複数インスタンス
  │  1  │ │  2  │
  └──┬──┘ └──┬──┘
     │     │
     ▼     ▼
  ┌──────┐ ┌──────────┐
  │Postgres│ │  Redis    │  ← 共有セッションストア
  └──────┘ └──────────┘
```

## よくある間違いと攻撃

### 間違い1：デバッグモードのままデプロイ

本番でデバッグログや開発設定のまま実行すると、エラーを発生させた人に内部情報（スタックトレース、SQLクエリ、セッションデータ）が漏れます。

### 間違い2：JAR内にシークレットをハードコード

データベースのパスワードやGoogle OIDCのクライアントシークレットをソースコードに埋め込むと、gitの履歴とアーティファクトに残ります。代わりに環境変数を使いましょう。

### 間違い3：データベースマイグレーションの省略

対応する[Flywayマイグレーション](migration.ja.md)を実行せずに新バージョンのコードをデプロイすると、ランタイムエラーが発生します -- コードがまだ存在しないカラムやテーブルを期待するからです。

### 間違い4：ロールバック計画がない

新バージョンに致命的なバグがある場合、前のバージョンにすばやく戻せる必要があります。以前のJARやDockerイメージを常に利用可能にしておきましょう。

### 間違い5：金曜の午後にデプロイ

真面目な話です。何か壊れたとき、修正できるフルチームがいてほしいはずです。週末の最小限のスタッフではなく。
