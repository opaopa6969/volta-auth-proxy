# Docker ボリューム

[English version](docker-volume.md)

---

## これは何？

Docker ボリュームは Docker コンテナのための永続的なストレージです。コンテナが削除されると、中のすべてが失われます -- ボリュームにデータを保存していない限り。ボリュームはコンテナのファイルシステムの外に存在するので、コンテナの再起動、更新、完全な再構築にも耐えます。

ノートパソコンに差し込んだ USB ドライブのようなものです。ノートパソコンが壊れたら、ハードドライブのデータは失われますが、USB ドライブは無事です。新しいノートパソコンに差し込めば、ファイルはすべてそのまま。Docker ボリュームはコンテナの USB ドライブです。コンテナは破棄して再作成できますが、ボリュームはデータを安全に保ちます。

Docker ボリュームはコンテナ化アプリケーションでデータを永続化する標準的な方法です。データベース、設定ファイル、アップロードファイル、ログ -- コンテナの再起動後も残す必要があるものはすべてボリュームに入れます。

---

## なぜ重要？

ボリュームがなければ、コンテナは設計上エフェメラル（一時的）です。アプリケーションを更新するたびに（新しいイメージをプル、コンテナを再作成）、コンテナ内のすべてのデータが消えます。これは意図的です -- コンテナは使い捨てであるべきです。しかしデータはそうではありません。

ボリュームなしでコンテナ内にデータベースを運用した場合の結果を考えてみましょう：

- アプリケーションの新バージョンをデプロイ：データベース消去
- クラッシュ後に Docker が再起動：データベース消去
- コンテナを1から2にスケール：それぞれが空のデータベースを持つ

ボリュームはデータのライフサイクルをコンテナのライフサイクルから分離することでこれを解決します。

## どう動くのか？

### Docker ストレージの種類

| 種類 | 永続性 | 用途 |
|------|--------|------|
| **コンテナ層** | コンテナと共に削除 | 一時ファイル、ビルド成果物 |
| **バインドマウント** | ホストディレクトリをコンテナにマップ | 開発（ライブコードリロード） |
| **名前付きボリューム** | Docker が管理、独立して永続 | 本番データ（データベース、アップロード） |
| **tmpfs マウント** | メモリのみ、再起動で消失 | シークレット、機密一時ファイル |

### ボリュームの動作

```
  ホストマシンのファイルシステム
  ┌──────────────────────────────────────────┐
  │                                          │
  │  /var/lib/docker/volumes/               │
  │  └── volta-data/                        │
  │      └── _data/                         │
  │          └── volta.db（SQLite ファイル）  │
  │                                          │
  └──────────────────────────────────────────┘
           │
           │  マウント先
           ▼
  コンテナのファイルシステム
  ┌──────────────────────────────────────────┐
  │                                          │
  │  /app/                                   │
  │  ├── volta-auth-proxy.jar（読み取り専用） │
  │  └── data/                              │
  │      └── volta.db ← ボリュームマウント    │
  │                                          │
  └──────────────────────────────────────────┘
```

### Docker ボリュームコマンド

```bash
  # 名前付きボリュームの作成
  docker volume create volta-data

  # ボリューム一覧
  docker volume ls

  # ボリュームの詳細（ディスク上の場所を確認）
  docker volume inspect volta-data

  # ボリュームの削除（注意：全データ削除）
  docker volume rm volta-data

  # ボリューム付きでコンテナを実行
  docker run -v volta-data:/app/data volta-auth-proxy
```

### Docker Compose のボリューム設定

```yaml
  version: "3.8"
  services:
    volta-auth-proxy:
      image: volta-auth-proxy:latest
      volumes:
        - volta-data:/app/data    # 永続化用の名前付きボリューム

  volumes:
    volta-data:                    # ボリューム宣言
      driver: local                # デフォルトドライバー（ホストファイルシステム）
```

### ボリュームのライフサイクル

```
  ボリューム付きでコンテナ作成
       │
       ▼
  コンテナが /app/data/volta.db にデータ書き込み
       │
       ▼
  コンテナの停止 / 再起動
  （ボリューム内のデータは安全）
       │
       ▼
  コンテナを削除（docker rm）
  （ボリューム内のデータはまだ安全）
       │
       ▼
  同じボリュームで新コンテナを作成
  （データはすぐに利用可能）
       │
       ▼
  ボリュームを明示的に削除（docker volume rm）
  （ここでデータが消える）
```

## volta-auth-proxy ではどう使われている？

volta は SQLite データベースファイルを永続化するために Docker ボリュームを使用しています。ボリュームがなければ、コンテナの再起動のたびに新しい空のデータベースが作成され、すべてのユーザー、テナント、セッション、招待が失われます。

### volta のボリューム設定

```yaml
  services:
    volta-auth-proxy:
      image: volta-auth-proxy:latest
      volumes:
        - volta-data:/app/data
      environment:
        VOLTA_DB_PATH: /app/data/volta.db

  volumes:
    volta-data:
```

### ボリュームに保存されるもの

| ファイル | 目的 | 失われた場合の影響 |
|---------|------|-------------------|
| `volta.db` | SQLite データベース（ユーザー、テナント、セッション、招待、ロール） | 完全なデータ損失。全ユーザーが再登録必要。 |
| `volta.db-wal` | SQLite Write-Ahead Log | 進行中のトランザクションが失われる可能性 |
| `volta.db-shm` | SQLite 共有メモリファイル | 再起動時に自動再作成 |

### SQLite + ボリュームが volta で機能する理由

SQLite はすべてを単一ファイルに保存します。これによりボリューム設定が極めてシンプルになります -- 1つのディレクトリをマウントするだけで、データベース全体が永続化されます。以下が不要：

- 別のデータベースコンテナ
- アプリとデータベース間のネットワーク設定
- データベース資格情報の管理
- コネクションプーリングの設定

このシンプルさが Phase 1 の利点です。volta が Phase 2 で PostgreSQL に移行する際、データベースは独自のコンテナと独自のボリュームで動作します。

### バックアップ戦略

データベースが既知のボリューム位置にある単一ファイルなので、バックアップは簡単です：

```bash
  # SQLite データベースのバックアップ
  docker run --rm -v volta-data:/data -v $(pwd):/backup \
    alpine cp /data/volta.db /backup/volta-backup-$(date +%Y%m%d).db

  # または SQLite 組み込みのバックアップコマンドを使用
  docker exec volta-auth-proxy \
    sqlite3 /app/data/volta.db ".backup /app/data/volta-backup.db"
```

## よくある間違いと攻撃

### 間違い1：初回デプロイでボリュームを忘れる

最もよくある問題：`-v` なしで `docker run volta-auth-proxy` を実行する。コンテナが再起動するまですべてが動作し、その後すべてのデータが消えます。最初のユーザーが登録する前に必ずボリュームを設定しましょう。

### 間違い2：バインドマウントと名前付きボリュームの混同

```yaml
  # バインドマウント（ホストディレクトリをマップ）：
  volumes:
    - ./local-data:/app/data    # . はホストパスを意味

  # 名前付きボリューム（Docker 管理）：
  volumes:
    - volta-data:/app/data       # . プレフィックスなし
```

バインドマウントは開発（ライブコード変更）に最適。名前付きボリュームは本番（Docker が場所、権限、クリーンアップを管理）に適しています。

### 間違い3：ボリュームの誤削除

`docker system prune --volumes` は未使用のすべてのボリュームを削除します。データベースが消える可能性があります。`docker volume prune` は慎重に使い、常にバックアップを取りましょう。

### 間違い4：権限の問題

コンテナプロセスは特定のユーザー（多くの場合 root またはカスタムユーザー）で実行されます。ボリュームディレクトリはそのユーザーが書き込み可能でなければなりません。権限の不一致は「読み取り専用ファイルシステム」や「権限拒否」エラーを引き起こします。

### 間違い5：ボリュームをバックアップしない

ボリュームは永続的ですが、自動バックアップされません。ホストマシンのディスクが故障すれば、ボリュームデータは失われます。外部の場所への定期的なバックアップを実装しましょう。
