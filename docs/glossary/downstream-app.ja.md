# 下流アプリ（Downstream App）

[English version](downstream-app.md)

---

## 一言で言うと？

下流アプリとは、volta-auth-proxyの「後ろ」に位置し、リクエストが届く前にvoltaがログイン、本人確認、権限チェックを処理してくれることに頼っているアプリケーションのことです。

---

## 水道管のたとえ

浄水場と近所の家がつながっている様子を想像してください：

```
  水源（雨、川）
       │
       ▼
  ┌──────────────────────┐
  │  浄水場               │  ← 水を浄化して検査する
  │                       │     （volta-auth-proxy）
  └──────────────────────┘
       │
       ├──── 家A（wikiアプリ）         ← きれいな水が届く
       ├──── 家B（管理パネル）         ← きれいな水が届く
       └──── 家C（ダッシュボードアプリ）← きれいな水が届く

  家は「下流」にある -- 浄水場の後にある。
  すでに浄化・検査済みの水を受け取る。
  自分で浄水設備を持つ必要はない。
```

このたとえでは：
- **水源** = ユーザーのブラウザからの生のHTTPリクエスト
- **浄水場** = volta-auth-proxy（本人確認、セッション検証、ロール判定）
- **きれいな水** = 本人情報ヘッダー付きのリクエスト（X-Volta-User-Id、X-Volta-Tenant-Idなど）
- **家** = あなたの下流アプリ（前処理済みの信頼できるリクエストを受け取る）

---

## ソフトウェアにおける「下流」の意味

ソフトウェアアーキテクチャでは、「上流（upstream）」と「下流（downstream）」はデータが流れる方向を表します：

- **上流** = ソースに近い（ユーザー、リクエストの発信元）
- **下流** = ソースから遠い（処理済みデータを受け取る側）

volta-auth-proxyは上流にいます。あなたのアプリは下流にいます。リクエストはユーザーからvoltaを通って、あなたのアプリへと流れていきます。だから「下流」アプリと呼ばれます。

```
  ユーザー（上流）
    → volta-auth-proxy（中間）
      → あなたのアプリ（下流）
```

逆の表現を聞くこともあるかもしれません：「下流アプリが上流のAPIを呼ぶ」。これはデータが反対方向に流れることを意味します。あなたのアプリがvoltaに情報を問い合わせるということです。

---

## なぜこの概念が重要なのか

上流/下流の関係を理解すると、voltaの重要な設計判断が分かります：

**下流アプリはvoltaを完全に信頼する。** wikiアプリが`X-Volta-User-Id: taro-uuid`というヘッダー付きのリクエストを受け取ったとき、それを独自に検証する必要はありません。voltaがすでに検証済みだからです。アプリはヘッダーを読んで使うだけです。

これは、たとえ話の家が浄水場を信頼するのと同じです。水を再検査しません。浄水場が水はきれいだと言えば、家はそれを使います。

この設計には大きなメリットがあります：
- **アプリがシンプルになる** -- ログインページ、セッション管理、認証ロジックが不要
- **セキュリティが一元化される** -- 監査する場所が1つ、脆弱性を修正する場所が1つ
- **一貫性** -- すべてのアプリが同じ認証ルールを適用

ただし、これはつまり：**voltaがダウンすると、すべての下流アプリが認証を失う。** 浄水場は単一障害点です。だからvoltaは信頼性が高く、適切に監視される必要があります。

## 実際のvoltaの下流アプリ

volta-config.yamlで下流アプリを定義します：

```yaml
apps:
  - id: app-wiki
    url: https://wiki.example.com
    allowed_roles: [MEMBER, ADMIN, OWNER]

  - id: app-admin
    url: https://admin.example.com
    allowed_roles: [ADMIN, OWNER]
```

各下流アプリは、すべてのリクエストでvoltaからこれらのヘッダーを受け取ります：

```
  X-Volta-User-Id:    taro-uuid
  X-Volta-Tenant-Id:  acme-uuid
  X-Volta-Roles:      MEMBER
  X-Volta-Email:      taro@acme.com
```

アプリはこれらのヘッダーを読んで、リクエストしているのが誰で何をしてよいかを正確に把握できます。認証の作業を自分で行う必要はありません。

---

## 簡単な例

```
  wikiと管理パネルの2つのアプリを作っているとしましょう。
  どちらもvoltaの下流です。

  太郎（ACME CorpのMEMBER）からのリクエスト：

  太郎のブラウザ
       │
       ▼
  volta-auth-proxy
       │ 「太郎はACME CorpのMEMBER」
       │
       ├──→ wiki.example.com
       │    voltaが追加：X-Volta-User-Id: taro-uuid
       │                 X-Volta-Tenant-Id: acme-uuid
       │                 X-Volta-Roles: MEMBER
       │    Wikiアプリ：「ようこそ太郎さん！ACMEのwikiページです」
       │    結果：200 OK
       │
       └──→ admin.example.com
            voltaが確認：MEMBERは[ADMIN, OWNER]に含まれない
            結果：403 Forbidden（リクエストは管理アプリに届かない）
```

管理パネルでは、voltaがリクエストをアプリに届ける前にブロックしていることに注目してください。下流アプリが太郎を拒否する必要はありません。voltaが無料でやってくれます。
