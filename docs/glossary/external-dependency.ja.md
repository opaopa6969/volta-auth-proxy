# 外部依存関係（External Dependency）

[English version](external-dependency.md)

---

## これは何？

外部依存関係とは、あなたのシステムが依存しているが、あなたがコントロールしていないソフトウェア、サービス、またはインフラです。誰か他の人が書いたコード、誰か他の人が運用しているサーバー、誰か他の人がメンテナンスしているデータベースです。壊れたとき、直接修正はできません -- 待つか、回避策を講じるか、置き換えるしかありません。

家への水道供給のようなものです。市の水道システムが機能することに依存しています。3ブロック先でパイプが破裂すると、キッチンの蛇口が止まります -- 市が修理するのを待つ以外に何もできません。水道システムはあなたの家庭の外部依存関係です。

---

## なぜ重要なのか？

すべての外部依存関係は潜在的な障害点であり、破壊的変更の源であり、アーキテクチャへの制約です。認証の世界では特に重要です。なぜなら認証はクリティカルパス上にあるからです -- 認証が落ちると、アプリケーション全体がアクセス不能になります。

volta-auth-proxyは最小依存関係の哲学で設計されました：**PostgreSQLが唯一の必須外部依存関係です。** これが重要な理由を理解するには、依存関係のコストを理解する必要があります。

## 外部依存関係のコスト

### 運用コスト

すべての依存関係について以下が必要です：

```
  依存関係ごとに：
  ┌──────────────────────────────────────┐
  │ ✓ インストールと設定                 │
  │ ✓ 健全性の監視                       │
  │ ✓ 更新の維持（セキュリティパッチ）   │
  │ ✓ バックアップ（状態を持つ場合）     │
  │ ✓ 壊れた時のトラブルシューティング   │
  │ ✓ 障害モードの学習                   │
  │ ✓ キャパシティプランニング           │
  │ ✓ バージョンアップグレードの対応     │
  └──────────────────────────────────────┘

  依存関係の数 × 依存関係ごとのコスト = 総運用負担

  Keycloakスタック:   Keycloak + Postgres + JVM       = 管理するもの3つ
  Oryスタック:        Oathkeeper + Kratos + Hydra
                      + Keto + Postgres               = 管理するもの5つ
  ZITADELスタック:    ZITADEL + CockroachDB           = 管理するもの2つ
  Auth0:              Auth0（クラウド）                = 1つ（ただしコントロールなし）

  voltaスタック:      volta + Postgres                 = 管理するもの2つ
```

### 障害の乗算

各依存関係に固有の障害率があります。各コンポーネントが99.9%のアップタイム（年間約8.7時間のダウンタイム）の場合：

```
  1コンポーネント:  99.9%アップタイム
  2コンポーネント: 99.9% × 99.9% = 99.8%アップタイム
  3コンポーネント: 99.9% × 99.9% × 99.9% = 99.7%アップタイム
  5コンポーネント: 99.9%^5 = 99.5%アップタイム（年間43.8時間ダウン）
```

コンポーネントを追加するたびにシステムの信頼性が下がります。冗長性を追加しない限り（冗長性を追加すると複雑さが増し、コンポーネントが増え...）。

### アップグレードリスク

依存関係が新バージョンをリリースすると、アップグレードするか評価する必要があります。依存関係の破壊的変更はシステム全体に波及する可能性があります。依存関係が多いほど管理するアップグレードが多くなり、何かが壊れる確率が高まります。

## voltaの最小依存関係の哲学

voltaの本番デプロイメントに必要なのは正確に2つのコンポーネントです：

```
  voltaの依存関係：
  ┌──────────────────────────────┐
  │ volta-auth-proxy             │  認証サービス（あなたがコントロール）
  └──────────┬───────────────────┘
             │
  ┌──────────▼───────────────────┐
  │ PostgreSQL                   │  唯一の外部依存関係
  └──────────────────────────────┘
```

### PostgreSQLだけで他に何もいらない理由

PostgreSQLが処理するもの：
- **ユーザーとテナントデータ** -- users、tenants、memberships、roles
- **セッション** -- セッション作成、検証、期限切れ（オプションでRedisにアップグレード可能）
- **暗号鍵** -- 保存時に暗号化されたRSA鍵ペア
- **監査ログ** -- すべての認証イベント（オプションでKafka/Elasticsearchにアップグレード可能）
- **招待** -- 招待コード、使用追跡、期限切れ
- **マイグレーション** -- Flywayがスキーマ進化を処理

他の認証システムはこれらの関心事を複数のサービスに分割します：

| 関心事 | Keycloak | Ory Stack | volta |
|--------|----------|-----------|-------|
| ユーザーデータ | Postgres | Postgres (Kratos) | Postgres |
| セッション | Infinispan（分散キャッシュ） | Postgres (Kratos) | Postgres |
| 権限 | Keycloakエンジン | Postgres (Keto) | Postgres |
| 鍵ストレージ | Keycloak DB | Hydra DB | Postgres |
| 監査 | Keycloakイベント | カスタム | Postgres |

voltaはすべてを1つのデータベースに入れます。1つのデータベースで十分だからです。単一のPostgreSQLインスタンスで何千もの同時認証リクエストを問題なく処理できます。

### オプションの依存関係（要件ではなく選択により）

voltaは必要なチーム向けにオプションの依存関係をサポートします：

| オプション依存関係 | 目的 | デフォルト（依存関係なし） |
|-------------------|------|--------------------------|
| Redis | より高いスループットのセッションストレージ | PostgreSQLセッション |
| Kafka | 監査イベントストリーミング | Postgres監査テーブル |
| Elasticsearch | 監査の検索と分析 | Postgres監査テーブル |
| SMTPサーバー | メール通知 | メールなし（リンク共有） |
| SendGrid | メール通知（マネージド） | メールなし（リンク共有） |

キーワードは**オプション**です。voltaはこれらのどれがなくても完全に動作します。要件ではなく拡張です。

## 「すべての依存関係は約束」の原則

依存関係を追加するとき、運用チームへの約束をしています：

- 「これを最新に保つことを約束します。」
- 「これを監視することを約束します。」
- 「午前3時に壊れたとき直し方を知っていることを約束します。」
- 「これが本番を落とす原因にならないことを約束します。」

voltaはこの約束を正確に1つの外部システムに対してだけ行います：PostgreSQL。PostgreSQLは世界で最も実戦テストされたオープンソースデータベースです。あなたのチームはほぼ確実に既に知っています。その障害モードは十分にドキュメント化されています。バックアップとリカバリの仕組みは成熟しています。

1つの依存関係。1つの約束。正しくすべき1つのこと。
