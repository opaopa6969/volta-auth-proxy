# 外部 vs 内部

[English version](external-vs-internal.md)

---

「外部」と「内部」という言葉はソフトウェアのあらゆる場所に登場し、文脈によって異なる意味を持ちます。しかし共通の糸があります：**外部はコントロールの外にあるもの、内部はコントロールの中にあるもの。** この区別がvolta-auth-proxyのコアなアーキテクチャ決定の多くを駆動しています。

---

## 外部 vs 内部の3つの意味

### 1. 外部サーバー vs 内部コード

外部サーバーは別のプロセスです -- アプリケーションとは別にデプロイし、実行し、管理するもの。内部実装はアプリケーションプロセスの中に存在するコードです。

```
  外部認証サーバー：
  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │ アプリ    │────►│ Keycloak     │────►│ PostgreSQL   │
  │           │     │ （別の       │     │              │
  │           │     │  サーバー）  │     │              │
  └──────────┘     └──────────────┘     └──────────────┘
  3つのプロセス。デプロイするもの3つ。障害の可能性3つ。

  内部認証コード：
  ┌──────────────────────────────┐     ┌──────────────┐
  │ volta-auth-proxy              │────►│ PostgreSQL   │
  │ （認証ロジックがこの          │     │              │
  │  単一プロセスの中に存在）     │     │              │
  └──────────────────────────────┘     └──────────────┘
  2つのプロセス。よりシンプル。障害の可能性が少ない。
```

Keycloakは外部認証サーバーです。アプリケーションはネットワーク呼び出しで接続します。voltaは違います：認証ロジックがアプリケーションそのものです。認証システムに「接続する」のではなく、あなたが認証システムです。

なぜ重要か？ネットワーク呼び出しは失敗します。外部サーバーは落ちます。アプリと認証サーバー間の設定のずれが不可解なバグを引き起こします。認証が内部コードなら、これらの問題は消えます。

### 2. 外部API（パブリック）vs 内部API（プライベートネットワーク）

外部APIはパブリックインターネットから到達可能です。URLを知る誰でも呼び出しを試みることができます。内部APIはプライベートネットワーク内からのみ到達可能です。

```
  外部API：
  ┌────────────┐
  │ インターネット │──►  https://api.stripe.com/v1/charges
  │ （誰でも）   │     （パブリック、誰でも呼び出しを試行可能）
  └────────────┘

  内部API：
  ┌────────────┐     ┌──────────────┐
  │ アプリ      │──►  volta:7070/api/v1/tenants/...
  │ （バックエンド）│   （プライベート、自分のサービスだけが到達可能）
  └────────────┘     └──────────────┘
  インターネットからは到達不可。
```

voltaの`/auth/*`エンドポイントは外部（ブラウザがログインのために到達する必要がある）。voltaの`/api/v1/*`エンドポイントは内部（バックエンドサービスだけが到達すべき）。この区別はセキュリティにとって決定的です：内部APIは呼び出し元を信頼できます。呼び出し元が既にネットワーク内にいるから。

### 3. 外部依存関係 vs 内部実装

外部依存関係は依存しているが書いておらずコントロールもしていないソフトウェア。内部実装は自分が書いて完全にコントロールするコード。

```
  Keycloakデプロイメントの外部依存関係：
  - Keycloakバイナリ（他人のコード）
  - JVMランタイム（Oracle/Eclipseのコード）
  - Infinispanキャッシュ（JBossのコード）
  - PostgreSQLドライバ（コミュニティのコード）
  - テーマエンジン（Keycloakのコード）

  voltaデプロイメントの外部依存関係：
  - PostgreSQL（コミュニティのコード）
  - それだけ。
```

すべての外部依存関係は、午前3時に壊れたときにデバッグできないコードです。追跡すべきバージョンです。評価すべきアップデートです。監視すべきセキュリティ脆弱性です。voltaはできるだけ多くを内部に保つことで外部依存関係を最小化します。

## voltaが物事を内部に保つ理由

voltaの哲学はこう要約できます：**品質を犠牲にせず内部に保てるなら、内部に保つ。**

| 対象 | 外部アプローチ | voltaの内部アプローチ |
|------|--------------|---------------------|
| 認証サーバー | Keycloak（外部プロセス） | voltaプロセス内の認証ロジック |
| セッションストレージ | Redisクラスタ（外部サービス） | PostgreSQLテーブル（共有DB） |
| テンプレートエンジン | Keycloakの FreeMarker（外部システムのエンジン） | voltaにコンパイルされたjte（内部） |
| 鍵ストレージ | 外部HSMまたはvault | PostgreSQLに暗号化保存 |
| ユーザー管理 | 外部管理コンソール | 組み込み管理UIとAPI |
| 監査ログ | 外部ELKスタック | PostgreSQLテーブル（オプションで外部シンクも） |

利点が複利で効いてきます：

1. **可動部品が少ない** = 障害モードが少ない
2. **理解するプロセスが1つ** = デバッグが速い
3. **1つのデプロイメント** = 運用がシンプル
4. **ネットワークトラフィックが少ない** = レイテンシが低い
5. **バージョンの不一致がない** = 「自分の環境では動く」がない

## 外部が正しい選択のとき

内部が常に良いわけではありません。外部が正しい選択のとき：

- **専門性が重要なとき。** PostgreSQLはvoltaが内部で構築できるどのデータベースよりも良いデータベースです。持つ価値のある外部依存関係です。
- **スケールに分散が必要なとき。** 毎秒何百万のセッションが必要なら、外部Redisクラスタは正当化されます。voltaはこれをサポートします（`SESSION_STORE=redis`）。
- **コンプライアンスが分離を要求するとき。** 一部のセキュリティ基準は専用の監査システムを要求します。voltaは外部監査シンク（Kafka、Elasticsearch）をサポートします。

鍵は意図性です。voltaは教条主義から外部依存関係を避けるのではありません。内部の代替案がよりシンプルで、より信頼性が高く、十分なとき避けます。外部ツールが内部コードでは解決できない問題を本当に解決するとき受け入れます。

## voltaにおけるスペクトラム

voltaのアーキテクチャは完全に内部からオプションで外部のスペクトラム上にあります：

```
  完全に内部（常に）：
  ├── 認証ロジック（OIDCフロー、セッション検証、JWT発行）
  ├── ForwardAuthハンドラー
  ├── ログイン/同意UI（jteテンプレート）
  ├── ロールベースアクセス制御
  └── 招待システム

  デフォルトは内部、外部にアップグレード可能：
  ├── セッションストレージ: PostgreSQL → Redis
  ├── 監査ログ: PostgreSQL → Kafka / Elasticsearch
  └── 通知: なし → SMTP / SendGrid

  常に外部：
  └── PostgreSQL（voltaが受け入れる唯一の依存関係）
```

これが設計原則です：内部から始めて、必要性が証明されたときだけ外部に移行する。
