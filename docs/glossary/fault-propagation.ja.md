# 障害伝播（Fault Propagation）

[English version](fault-propagation.md)

---

## これは何？

障害伝播とは、システムの1つの部分の障害が他の部分の障害を引き起こすこと -- 連鎖反応のようなものです。1つのものが壊れ、他のものがそれに依存しているため、それらも壊れ、さらにそれらに依存するものも壊れ、やがてすべてがダウンします。

ドミノの列のようなものです。最初の1つを倒すと、2番目に当たり、3番目に当たり、数秒で列全体が倒れます。最初のドミノが「障害」。倒れていくドミノの連鎖が「伝播」。列全体が倒れた状態が「カスケード障害」です。

現実の例：発電所がオフラインになり、送電網が過負荷になり、他の発電所が自己保護のためにシャットダウンし、地域全体が停電します。1つの障害が伝播してシステム全体をダウンさせました。

## 認証の障害がとりわけ危険な理由

認証はシステムコンポーネントの中でもユニークです。すべてのリクエストのクリティカルパスにあるからです。検索機能が壊れれば、ユーザーは検索できません -- でもそれ以外はすべて使えます。認証システムが壊れれば、ユーザーは何にもアクセスできません。

```
検索の障害：
  ┌─────────┐     ┌─────────┐     ┌─────────┐
  │ Wiki    │ OK  │ Admin   │ OK  │ Chat    │ OK
  │         │     │         │     │         │
  └─────────┘     └─────────┘     └─────────┘
  検索は壊れているが、他のすべては動作する。

認証の障害：
  ┌─────────┐     ┌─────────┐     ┌─────────┐
  │ Wiki    │ ✗   │ Admin   │ ✗   │ Chat    │ ✗
  │ 401     │     │ 401     │     │ 401     │
  └─────────┘     └─────────┘     └─────────┘
  何も動かない。すべてのアプリが認証に依存している。
```

これは本質的に障害伝播です。認証はすべてのアプリケーションが依存する単一のポイントです。障害が発生すると、すべての保護されたアプリケーションに同時に伝播します。

## コンポーネントが多いほど伝播経路も多い

「認証をマイクロサービスに分割すれば、1つが壊れても他は動き続ける」と主張するアーキテクトもいます。論理的に聞こえますが、認証についての重要な点を見落としています：部品は独立していないのです。

```
マイクロサービス認証：
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ OIDC     │───►│ Session  │───►│ JWT      │
  │ Service  │    │ Service  │    │ Service  │
  └──────────┘    └──────────┘    └──────────┘

  Session Serviceが死んだ場合：
  - OIDC Serviceがセッションを作成できない → ログイン失敗
  - JWT Serviceがセッションを検証できない → ForwardAuth失敗
  - すべてのアプリがアクセス不能に

  3つのサービスに分割しても伝播は防げなかった。
  壊れる可能性のあるものが2つ増えただけ。
```

サービス間の各接続は潜在的な障害ポイントです：ネットワークタイムアウト、DNS障害、コネクションプールの枯渇、シリアライゼーションのバグ。サービスが多いほど接続が多く、壊れるものが多くなります。

```
シングルプロセス（volta）：
  障害ポイント：voltaプロセス + データベース = 2つ

マイクロサービス認証：
  障害ポイント：OIDC Service + Session Service + JWT Service
                + サービス間のネットワーク + ロードバランサー
                + サービスディスカバリ + データベース群 = 7つ以上
```

voltaには壊れる可能性のあるものが2つ。マイクロサービス認証システムには7つ以上。追加の障害ポイントはそれぞれ、チェーンのもう1つのドミノです。

## voltaのアプローチ：コンポーネントを最小化し、伝播を最小化

voltaの障害伝播に対する戦略はシンプルです：壊れる可能性のあるものを減らす。

1. **1プロセス。** 内部ネットワーク障害の可能性なし。プロセス内のメソッド呼び出しにはタイムアウト、DNS障害、コネクションプールの問題がありません。

2. **1データベース。** 分散データ一貫性の問題なし。「Session ServiceにはデータがあるがなぜかはわからないがなぜかJWT Serviceがアクセスできない」シナリオなし。

3. **ランタイムでの外部サービス依存なし。** voltaはForwardAuth検証中にAuth0、Google、その他の外部サービスを呼び出しません。OIDC IdPはログイン中にのみ連絡されます -- すべてのリクエストではなく。

4. **グレースフルデグラデーション。** データベースが一時的にアクセスできなくても、キャッシュされたセッションでForwardAuthチェックを引き続き処理できます。データベースの一時的な問題の爆発半径が縮小されます。

```
voltaの障害伝播サーフェス：
  ┌──────────────┐     ┌──────────┐
  │ volta-auth-  │────►│ Postgres │
  │ proxy        │     │          │
  └──────────────┘     └──────────┘

  壊れる可能性のあるもの：
  1. voltaプロセスがクラッシュ → 再起動（200ms起動）
  2. Postgresにアクセスできない → キャッシュされたセッションが一時的に動作
  3. 両方が失敗 → 認証がダウン（しかしこれはどの認証システムでも同じ）
```

---

## volta-auth-proxyでは

voltaはシングルプロセスと単一データベース依存で動作し、マイクロサービス認証システムで追加の障害パスを作り出すネットワーク接続、サービスディスカバリ、分散調整を排除することで、障害伝播を最小化しています。
