# フレームワーク

[English version](framework.md)

---

## これは何？

フレームワークとは、アプリケーションの骨組みを提供する、あらかじめ構築された構造です。ネットワークリクエストの待ち受け方法、URLの解析方法、レスポンスの送信方法など、すべてをゼロから書く代わりに、フレームワークの構造に自分のコードを差し込めば、配管工事はフレームワークがやってくれます。

家の骨組みに似ています。家を建てるとき、壁をどうやって立てるかから考え始める人はいません。骨組みはすでにそこにあります -- 壁、屋根の構造、床の根太。あなたが決めるのは、キッチンをどこに置くか、壁を何色に塗るか、どんな家具を入れるかです。フレームワークは家の骨組み。あなたのビジネスロジックは家具です。

フレームワークと[ライブラリ](library.md)の重要な違いは**制御の反転**です。ライブラリでは、あなたのコードがライブラリを呼びます。フレームワークでは、フレームワークがあなたのコードを呼びます。ハンドラー関数を書いて登録すると、フレームワークがいつそれを呼び出すかを決めます。これは「ハリウッドの原則」と呼ばれることがあります -- 電話をかけてくるな、こちらからかける。

---

## なぜ重要なのか？

フレームワークがなければ、Webアプリケーションを構築するということは以下を意味します：

- 生の[HTTP](http.md)サーバーをゼロから書く
- リクエストURL、クエリパラメータ、ボディを手動でパースする
- スレッドと並行性を自分で管理する
- ルーティング（URLパスをハンドラーコードに対応付ける）を実装する
- エラーレスポンス、コンテントタイプ、ステータスコードを手動で処理する
- セキュリティの基本（CSRF保護、ヘッダーのサニタイズ）を再発明する

フレームワークは何千行ものボイラープレートを排除し、アプリを独自たらしめているもの -- [ビジネスロジック](business-logic.md)に集中させてくれます。

## どう動くのか？

### 基本パターン

すべてのWebフレームワークはおおよそ同じパターンに従います：

```
  受信HTTPリクエスト
         │
         ▼
  ┌──────────────────────┐
  │    フレームワーク      │
  │                       │
  │  1. リクエストをパース │
  │  2. ルートをマッチ     │──── GET /users/:id  ->  あなたのハンドラー
  │  3. ミドルウェアを実行 │──── 認証チェック、ログ記録など
  │  4. ハンドラーを呼ぶ   │──── あなたのコードがここで実行
  │  5. レスポンスを送信   │
  └──────────────────────┘
         │
         ▼
  HTTPレスポンスをクライアントへ返す
```

ルートとハンドラーを登録します。残りはフレームワークがすべて処理します。

### フレームワーク vs ライブラリ：制御の流れ

```
  ライブラリ（あなたが制御）:            フレームワーク（それが制御）:
  ┌───────────────┐                     ┌───────────────┐
  │  あなたのコード │                     │  フレームワーク │
  │               │                     │               │
  │  result =     │                     │  onRequest -> │──> あなたのhandler()
  │   lib.doX()   │──> ライブラリ        │  onError ->   │──> あなたのerrorHandler()
  │               │                     │  onStart ->   │──> あなたのinit()
  │  結果を使う    │<── 返す             │               │
  └───────────────┘                     └───────────────┘

  あなたがライブラリを呼ぶ。             フレームワークがあなたを呼ぶ。
```

### フレームワークの種類

| 種類 | 例 | 目的 |
|------|---|------|
| Webフレームワーク（軽量） | Javalin, Express, Sinatra, Flask | 最小限のルーティング + HTTP処理 |
| Webフレームワーク（フルスタック） | Spring Boot, Django, Rails | ルーティング + ORM + テンプレート + 認証 + すべて |
| フロントエンドフレームワーク | React, Vue, Angular | ブラウザ側のUIレンダリング |
| テストフレームワーク | JUnit, Jest, pytest | テストの構造化と実行 |

### 軽量 vs フルスタック

これはvolta-auth-proxyの哲学にとって特に重要な区別です：

```
  軽量（Javalin）:                    フルスタック（Spring Boot）:
  ┌─────────────────────┐            ┌─────────────────────────────┐
  │  HTTPルーティング     │            │  HTTPルーティング             │
  │  リクエスト/レスポンス │            │  リクエスト/レスポンス        │
  │  ミドルウェアフック   │            │  ミドルウェア（フィルター）    │
  │  WebSocketサポート   │            │  依存性注入                  │
  │                      │            │  ORM（JPA/Hibernate）        │
  │  以上。               │            │  セキュリティ（Spring Security）│
  │  残りはあなたが選ぶ。  │            │  テンプレート（Thymeleaf）    │
  │                      │            │  キャッシュフレームワーク      │
  └─────────────────────┘            │  ジョブスケジューリング       │
                                     │  50以上の自動設定             │
                                     └─────────────────────────────┘
```

軽量フレームワークはHTTPを処理するために必要最小限のものだけを提供します。[データベース](database.md)ライブラリ、[テンプレート](template.md)エンジン、キャッシュレイヤーは自分で選びます。フルスタックフレームワークはすべてをまとめてバンドルしますが、便利な反面、[設定地獄](config-hell.md)に陥りやすくなります。

### Javalinの例

JavalinはVolta-auth-proxyが使用するフレームワークです。最もシンプルなJavalinアプリはこちらです：

```java
import io.javalin.Javalin;

public class Main {
    public static void main(String[] args) {
        var app = Javalin.create().start(7070);
        app.get("/hello", ctx -> ctx.result("Hello World"));
    }
}
```

4行で、[ポート](port.md) 7070で待ち受け、`GET /hello` に応答する[HTTP](http.md) [サーバー](server.md)が動きます。フレームワークが処理するのは：

- Jettyサーバーの起動
- TCP接続の待ち受け
- HTTPリクエストのパース
- URLパス `/hello` をラムダにマッチング
- 正しいヘッダー付きのレスポンス送信

ネットワークコードはゼロ行です。

## volta-auth-proxy ではどう使われている？

volta-auth-proxyは**Javalin**をWebフレームワークとして使用しています。この選択は意図的です -- Javalinは軽量で、魔法のアノテーションなし、DIコンテナなし、自動設定なし。コードを読めば何をしているか理解できます。

### Main.javaでのアプリケーションセットアップ

```java
var app = Javalin.create(config -> {
    config.staticFiles.add("/public");
    config.jetty.sessionHandler(SessionConfig::fileSessionHandler);
    // テンプレートエンジンなど
});
```

フレームワークは[YAML](yaml.md)ファイルやアノテーションスキャンではなく、明示的なメソッド呼び出しで設定されます。

### ルート登録

ルートは直接登録されます：

```java
app.get("/auth/login",      ctx -> authController.loginPage(ctx));
app.post("/auth/callback",  ctx -> authController.callback(ctx));
app.get("/api/v1/users/me", ctx -> userController.me(ctx));
```

各行がHTTPメソッド + パスをハンドラー関数にマッピングします。隠れたルーティングファイルはありません。convention-over-configurationはありません。どのURLがどのコードをトリガーするか、そのまま見えます。

### before/afterフックによるミドルウェア

Javalinの[ミドルウェア](middleware.md)システムは`before()`と`after()`フックを使います：

```java
app.before("/api/*", ctx -> authMiddleware.requireAuth(ctx));
app.before("/api/v1/admin/*", ctx -> authMiddleware.requireRole(ctx, Role.ADMIN));
```

これらはルートハンドラーの前に実行され、すべての[エンドポイント](endpoint.md)で認証と[ロール](role.md)チェックのコードを繰り返すことなく実行できます。

### なぜJavalinでSpring Bootではないのか？

volta-auth-proxyの哲学（[config-hell.md](config-hell.md)と[native-implementation.md](native-implementation.md)を参照）：

- **アノテーションの魔法なし** -- Spring Bootは`@RestController`、`@Autowired`、`@Configuration`などを使います。これらのアノテーションはコードに見えない動作をトリガーします。Javalinはアノテーションゼロです。
- **依存性注入なし** -- Springはコンテナを通じてオブジェクト生成を管理します。voltaは`new`で明示的にオブジェクトを作成します。
- **自動設定なし** -- Spring Bootはクラスパスをスキャンして自動的に機能を有効にします。voltaはMain.javaで手動で機能を有効にします。
- **デバッグしやすい** -- Javalinで何か問題が起きたとき、スタックトレースにはあなたのコードが表示されます。Springのプロキシの47層ではありません。

## よくある間違いと攻撃

### 間違い1：人気だけでフレームワークを選ぶ

Spring Bootは最も人気のあるJava Webフレームワークです。だからといってすべてのプロジェクトに適しているわけではありません。単一プロセスのアイデンティティゲートウェイにSpringのエンタープライズ機能の複雑さは不要です。ユースケースに対して大きすぎるフレームワークを選ぶと、[設定の複雑さ](complexity-of-configuration.md)と[技術的負債](what-is-tech-debt.md)につながります。

### 間違い2：フレームワークと戦う

すべてのフレームワークには、コードの構造について意見があります。フレームワークのパターンに対するワークアラウンドを書いている自分に気づいたら、間違ったフレームワークを選んでいるかもしれません。voltaがJavalinを選んだのは、Javalinの意見がvoltaの哲学と一致しているからです：暗黙より明示、巧妙よりシンプル。

### 間違い3：フレームワークがセキュリティを処理してくれると思い込む

フレームワークが提供するのはHTTPの配管であって、セキュリティの保証ではありません。JavalinはJWTの検証、CSRFトークンのチェック、認可の強制を自動的には行いません。voltaはすべてのセキュリティチェックを[ミドルウェア](middleware.md)とハンドラーで明示的に実装しています。フレームワークが「面倒を見てくれる」と思い込まないでください。

### 攻撃1：フレームワーク固有の脆弱性

フレームワークはソフトウェアであり、ソフトウェアにはバグがあります。Spring4Shell（CVE-2022-22965）はSpring Frameworkのリモートコード実行脆弱性でした。Express.jsにはプロトタイプ汚染の問題がありました。フレームワークを使うと、その脆弱性サーフェスを引き継ぎます。フレームワークを最新に保つことが不可欠です。

### 間違い4：フレームワークを別の抽象化で包む

カスタムベースクラス、カスタムルーティングレイヤーなど、「フレームワークの上にフレームワーク」を作るチームがあります。これによりフレームワークのアップグレードが難しくなり、新しい開発者がコードを理解しにくくなります。voltaはカスタムラッパーレイヤーなしにJavalinのAPIを直接使用しています。
