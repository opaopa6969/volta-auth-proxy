# フロントエンドとバックエンド

[English version](frontend-backend.md)

---

## これは何？

フロントエンドとバックエンドは、Webアプリケーションの2つの半分です。**フロントエンド**はユーザーの[ブラウザ](browser.md)で動くすべてのもの -- [HTML](html.md)の構造、[CSS](css.md)のスタイリング、[JavaScript](javascript.md)のインタラクティブ機能。**バックエンド**は[サーバー](server.md)で動くすべてのもの -- ビジネスロジック、[データベース](database.md)クエリ、認証チェック、[API](api.md)レスポンス。

レストランに似ています。フロントエンドはダイニングルーム -- メニュー、内装、注文を取るウェイター。お客さんが見て触れるものです。バックエンドはキッチン -- 料理を作るシェフ、食材を保管する冷蔵庫、皿を洗う食洗機。お客さんはキッチンを見ることはありませんが、キッチンがなければ料理はありません。ウェイター（API）がダイニングルームからキッチンへリクエストを運び、結果を持ち帰ります。

この用語は初期のコンピューティングの物理的配置に由来します。「フロント」はユーザーが座る端末、「バック」は別の部屋にあるメインフレームでした。

---

## なぜ重要なのか？

フロントエンドとバックエンドの分離を理解することが不可欠な理由：

- **セキュリティ境界** -- フロントエンドはユーザーのマシンで動きます。信頼できません。あらゆるバリデーション、認可、秘密はバックエンドに置く必要があります。ユーザーはブラウザの開発者ツールでフロントエンドコードを変更できます。
- **異なる言語とツール** -- フロントエンドは通常[JavaScript](javascript.md)、HTML、CSSを使います。バックエンドは[Java](java.md)、Python、Go、Rustなどを使えます。volta-auth-proxyのバックエンドはJava、フロントエンドページは[jte](jte.md)テンプレートを使います。
- **認証アーキテクチャ** -- voltaはフロントエンドとバックエンドの間に位置します。フロントエンドが[Cookie](cookie.md)を送り、voltaのバックエンドがそれを検証し、下流のバックエンド向けに[JWT](jwt.md)や[ヘッダー](header.md)を発行します。
- **デプロイモデル** -- フロントエンドはCDNが配信する静的ファイルであることが多いです。バックエンドはサーバー上で動く[プロセス](process.md)です。スケーリングの方法が異なります。

## どう動くのか？

### 分割

```
  ┌────────────────────────────────────┐     ┌────────────────────────────────┐
  │           フロントエンド            │     │           バックエンド          │
  │      （ブラウザで動く）             │     │      （サーバーで動く）         │
  │                                     │     │                                │
  │  ┌──────────┐  ┌──────────────┐    │     │  ┌─────────────┐              │
  │  │  HTML     │  │ JavaScript   │    │     │  │ ルート       │              │
  │  │（ページの │  │ （振る舞い）  │    │ HTTP│  │ ハンドラー   │              │
  │  │  構造）   │  │              │    │────>│  │              │              │
  │  └──────────┘  │ fetch(url)   │    │     │  └──────┬───────┘              │
  │                │ onClick()    │    │<────│         │                       │
  │  ┌──────────┐  │ render()     │    │     │  ┌──────▼───────┐              │
  │  │  CSS      │  └──────────────┘    │     │  │ ビジネス     │              │
  │  │（スタイル）│                      │     │  │ ロジック     │              │
  │  └──────────┘                       │     │  └──────┬───────┘              │
  │                                     │     │  ┌──────▼───────┐              │
  │                                     │     │  │ データベース  │              │
  │                                     │     │  └──────────────┘              │
  └────────────────────────────────────┘     └────────────────────────────────┘
       ユーザーのコンピュータ                       サーバー（データセンター）
```

### 何がどこにあるか

| 関心事 | フロントエンド | バックエンド |
|--------|-------------|------------|
| ユーザーインターフェース | ボタン、フォーム、色、レイアウト | -- |
| 入力バリデーション | 即座のフィードバック（「メール形式が不正」） | 権威あるチェック（実際に検証） |
| ビジネスルール | -- | 「ADMINは招待可、MEMBERは不可」 |
| データ保存 | -- | PostgreSQL、セッションストア |
| 認証 | リクエストにCookieを送る | Cookieを検証、セッションを作成 |
| 秘密情報（APIキー、DBパスワード） | 絶対に置かない | 常にここ |
| データのフォーマット | JSONをHTMLにレンダリング | JSONを返す |

### フロントエンドとバックエンドの通信

フロントエンドとバックエンドは[HTTP](http.md)で通信します：

```
  フロントエンド（ブラウザ）            バックエンド（サーバー）
  ──────────────────                   ────────────────
  ユーザーが「保存」をクリック
       │
       ▼
  JavaScriptがリクエストを構築：
  POST /api/v1/tenants
  Body: {"name":"ACME Corp"}
  Cookie: JSESSIONID=abc123
       │
       ├──── HTTPリクエスト ──────────>  リクエストを受信
       │                                セッションを検証（Cookie）
       │                                ロールをチェック（OWNER/ADMIN?）
       │                                データベースに挿入
       │                                結果を返す
       │<──── HTTPレスポンス ──────────
       │
  HTTP 201 Created
  Body: {"id":"uuid-here","name":"ACME Corp"}
       │
       ▼
  JavaScriptがページを更新
  （新しいテナントをリストに追加）
```

### サーバーレンダリング vs クライアントレンダリング

HTMLを生成する2つのアプローチがあります：

**サーバーレンダリング（voltaの認証ページ）：**
```
  ブラウザ                             サーバー
  ───────                              ──────
  GET /auth/login  ──────────────────> サーバーがjteテンプレートを実行
                                       データを埋め込む（テナント名など）
                                       完全なHTMLを生成
                   <────────────────── <html>...</html>を返す
  ブラウザが完成したHTMLを表示
```

**クライアントレンダリング（[SPA](spa.md)）：**
```
  ブラウザ                             サーバー
  ───────                              ──────
  GET /            ──────────────────> index.html + app.jsを返す
                   <──────────────────
  JavaScriptが実行
  GET /api/v1/users/me ──────────────> JSONデータを返す
                       <──────────────
  JavaScriptがデータからHTMLを構築
  ブラウザが結果を表示
```

### フルスタック = フロントエンド + バックエンド

「フルスタック開発者」は両方を扱います。「フルスタックフレームワーク」は両方を処理します（例：Next.js、Rails）。volta-auth-proxyはバックエンド専用のアプリケーションです -- [jte](jte.md)経由でいくつかのHTMLページを提供しますが、フロントエンドフレームワークではありません。voltaを使うフロントエンドアプリは別プロジェクトです。

## volta-auth-proxy ではどう使われている？

### voltaは主にバックエンド

volta-auth-proxyは[Java](java.md)バックエンドアプリケーションです。主な仕事はHTTPリクエストの処理、セッション管理、JWT発行、データベースとの通信です。フロントエンドコードは最小限です。

### voltaの小さなフロントエンド：jteテンプレート

voltaは認証フロー用にいくつかのHTMLページを直接提供します：

```
  voltaのフロントエンドページ（jteによるサーバーレンダリング）：
  ├── /auth/login          → 「Googleでサインイン」ボタンのログインページ
  ├── /auth/tenant-select  → ログイン後のテナント選択ページ
  ├── /auth/invite/accept  → 招待承諾ページ
  └── /error               → エラーページ
```

これらのページは[jte](jte.md)[テンプレート](template.md)を使った[サーバーレンダリング](template.md)です。[SPA](spa.md)ではありません。これは意図的です -- 認証ページはJavaScriptなしで動くべきです。

### voltaはフロントエンドとバックエンドの間に位置する

[ダウンストリームアプリ](downstream-app.md)にとって、voltaはフロントエンドとバックエンドの間の認証レイヤーです：

```
  ┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
  │  アプリの          │     │  volta-auth-proxy │     │  アプリの          │
  │  フロントエンド    │     │ （認証ゲートウェイ）│     │  バックエンド      │
  │  (React SPA)      │     │                   │     │  （あなたのAPI）   │
  │                   │     │                   │     │                  │
  │  fetch("/api")────┼────>│  セッションチェック │     │                  │
  │                   │     │  X-Volta-*ヘッダー│────>│  X-Volta-*を読む │
  │                   │     │  を追加           │     │  信頼する          │
  │  <────────────────┼─────┼─────────────────┼─────│  データを返す      │
  └──────────────────┘     └──────────────────┘     └──────────────────┘
      ブラウザ側                 サーバー側               サーバー側
   （フロントエンド）          （バックエンド）          （バックエンド）
```

### 信頼境界

重要なセキュリティ概念：**フロントエンドを決して信頼しない**。

```
  信頼できない                        信頼できる
  ─────────────────────────────────  ─────────────────────────────
  ブラウザ（フロントエンド）          volta-auth-proxy（バックエンド）
  ├── ユーザーがJSを変更できる        ├── セッションCookieを検証
  ├── ユーザーがHTMLを編集できる      ├── 秘密鍵でJWTに署名
  ├── ユーザーがヘッダーを偽造できる  ├── データベースでロールを確認
  ├── ユーザーがリクエストを再送できる├── レート制限を強制
  └── ユーザーが全Cookie*を読める     └── 秘密情報を安全に保管

  * HttpOnly Cookieを除く。JavaScriptから読めない
```

voltaはセッションCookieに`HttpOnly`と`Secure`フラグを設定しているため、フロントエンドのJavaScriptはそれを読み取りも変更もできません。信頼が確立されるのはバックエンドだけです。

## よくある間違いと攻撃

### 間違い1：フロントエンドでのみバリデーション

フロントエンドのフォームが送信前に「このメールは有効か？」をチェックするかもしれません。しかし攻撃者は`curl`や改変されたブラウザでフロントエンドを完全にバイパスできます。バックエンドは常にすべての入力を再検証しなければなりません。

```
  間違い:  フロントエンドがメール形式チェック → バックエンドが信頼
  正しい:  フロントエンドがメール形式チェック → バックエンドもメール形式チェック
```

### 間違い2：フロントエンドに秘密を保存する

フロントエンドコードのすべてはユーザーに見えます。JavaScriptにAPIキー、データベースパスワード、署名鍵を絶対に置かないでください。voltaのRSA秘密鍵はバックエンドにのみ存在します。公開鍵は`/.well-known/jwks.json`で意図的に公開されています。公開鍵は公開されるべきものだからです。

### 攻撃1：フロントエンドの改ざん

攻撃者がフロントエンドのJavaScriptを改変して異なるロールを送ります：

```json
// 攻撃者がリクエストボディを改変
POST /api/v1/tenants
{"name": "Evil Corp", "role": "OWNER"}
```

バックエンドが認証済みセッションからではなくリクエストボディから`role`を読むと、攻撃者は権限を昇格させています。voltaはクライアントリクエストからロールを読みません -- ロールはセッション経由でデータベースから来ます。

### 間違い3：バックエンドエラーをフロントエンドに漏らす

`PSQLException: relation "users" does not exist`のようなバックエンドエラーは、攻撃者にPostgreSQLを使っていることと`users`テーブルが存在しないことを教えます。voltaはフロントエンドに汎用的なエラーメッセージを返し、詳細なエラーはバックエンドにのみログ記録します。

### 攻撃2：CORSの悪用

バックエンドが任意のオリジンを許可すると（`Access-Control-Allow-Origin: *`）、悪意のあるウェブサイトがユーザーに代わってAPIリクエストを送れます。voltaの[CORS](cors.md)設定は特定の信頼されたオリジンのみを許可します。
