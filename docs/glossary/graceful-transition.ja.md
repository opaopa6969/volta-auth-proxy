# グレースフル・トランジション

[English version](graceful-transition.md)

---

## これは何？

グレースフル・トランジション（猶予付き移行）とは、古いバージョンと新しいバージョン（通常は暗号鍵）の両方が同時に受け入れられる切り替え期間のことです。古いバージョンを使っているものすべてを壊す瞬時の切り替えではなく、システムが一定の重複期間は両方をサポートし、古いものが不要になってから引退させます。

ビルの鍵交換を想像してください。強制切り替えは深夜にすべての錠前を交換すること -- 古い鍵を持つ全員が翌朝締め出されます。グレースフル・トランジションは、1週間の間古い鍵と新しい鍵の両方を受け付ける錠前を設置し、全員が新しい鍵を受け取る時間を与えます。1週間後、古い鍵は使えなくなります。

認証では、[鍵ローテーション](key-rotation.md)中のグレースフル・トランジションが重要です。署名鍵を即座に入れ替えると、古い鍵で署名されたすべてのJWTが検証不能になります。グレースフル・トランジションは重複期間中に両方の鍵をアクティブに保ちます。

---

## なぜ重要なのか？

グレースフル・トランジションがなければ、あらゆる運用上の変更が潜在的な障害になります：

- **鍵ローテーション**: 古い鍵で署名された有効なJWTを持つユーザーが401エラーを受ける
- **証明書更新**: クライアントが古い証明書をキャッシュしていると接続が失敗する
- **セッション移行**: セッションストレージの移行中にユーザーがログアウトされる

非グレースフルな移行のコストはアクティブなトークンの数に比例します。鍵が変わったときに10,000人のユーザーがアクティブなJWTを持っていれば、10,000人が同時にエラーに遭遇します。voltaの5分のJWT有効期限なら、グレースフル・トランジションは5分だけ続ければ済みます。

## どう動くのか？

### 重複ウィンドウ

```
  時間 ────────────────────────────────────────────────►

  鍵A:  ████████████████████████░░░░░░░░░░░░░░░░░░░░
        アクティブ              引退（検証には
                                まだ有効）

  鍵B:  ░░░░░░░░░░░░░░░░░░░░░░████████████████████████
                                アクティブ（新JWTに署名）

  重複:                         ▓▓▓▓▓▓▓▓▓▓▓▓
                                両方の鍵を受け入れ。
                                鍵A: 検証のみ
                                鍵B: 署名 + 検証

  重複後:
  鍵AをJWKSから削除。鍵Aのトークンはすべて期限切れ。
```

```
  └──────────────────────────────────┘

  kid "key-B" のJWT:
  ┌──────────────────────────────────┐

  │  署名: 鍵Bで署名                │
  └──────────────────────────────────┘
       │
       ▼  検証者がkidを確認
  ┌──────────────────────────────────┐
  │  JWKSに鍵Aと鍵Bの両方がある     │
  │  kid = "key-B" → 鍵Bを使用     │
  │  検証: 成功                      │
  └──────────────────────────────────┘
```

### 重複期間はどのくらい必要か？

重複は、古い鍵で署名された最も長寿命のトークンと同じかそれ以上必要です：

```
  voltaの計算:
  ┌──────────────────────────────────────────────┐
  │  JWT TTL:          5分                       │
  │  最悪のケース:     T-0でトークン発行、        │
  │                    T+0.001秒でローテーション  │
  │  トークン期限切れ: T + 5分                    │
  │                                              │
  │  最小重複期間:     5分                        │
  │  推奨:             5-10分                     │
  │  重複後:           古い鍵を安全に削除可能      │
  └──────────────────────────────────────────────┘
```

## volta-auth-proxy ではどう使われている？

### グレースフル・トランジション付き鍵ローテーション

`POST /api/v1/admin/keys/rotate`が呼ばれたとき：

```java
public synchronized String rotateKey() {
    RSAKey current = this.rsaKey;
    // 新しい鍵ペアを生成
    KeyPairGenerator generator = KeyPairGenerator.getInstance("RSA");
    generator.initialize(2048);
    KeyPair keyPair = generator.generateKeyPair();
    String kid = "key-" + Instant.now()...;

    // アトミック: 古い鍵の引退 + 新しい鍵の保存
    store.rotateSigningKey(
        current.getKeyID(),     // 古い鍵 → 引退
        kid,                    // 新しい鍵 → アクティブ
        keyCipher.encrypt(...), // 新しい公開鍵（暗号化）
        keyCipher.encrypt(...)  // 新しい秘密鍵（暗号化）
    );
    this.rsaKey = next;  // 新しい鍵で署名を開始
    return kid;
}
```

### 移行タイムライン

```
  T+0:00  ローテーション開始
          ├── 新しい鍵Bを生成
          ├── 古い鍵AをDBで「引退」に設定
          ├── 新しいJWTは鍵Bで署名
          └── JWKSが鍵Bを公開（鍵AはまだDBに参照用として残る）

  T+0:01  クライアントが鍵Aで署名されたJWTを提示
          ├── voltaが鍵Aで検証（まだ利用可能）
          └── 検証成功

  T+2:30  クライアントの鍵A JWTが期限切れ
          ├── クライアントが/api/v1/auth/tokenを呼ぶ（サイレントリフレッシュ）
          ├── voltaが鍵Bで署名された新しいJWTを発行
          └── クライアントは以降鍵B JWTを使用

  T+5:00  すべての鍵A JWTが期限切れ
          ├── 鍵Aはもう必要ない
          └── 鍵Aをシステムから安全に削除可能

  ユーザーはエラーを経験しない。
  ダウンタイムなし。401なし。再ログイン不要。
```

### voltaの設計がグレースフル・トランジションを容易にする理由

1. **短いJWT TTL（5分）**: 重複ウィンドウが自然に短い
2. **JWTヘッダーのkid**: 検証者が推測なしにどの鍵を使うか分かる
3. **セッションの継続性**: セッションは鍵ローテーションの影響を受けない。移行が必要なのはJWTだけ
4. **[サイレントリフレッシュ](silent-refresh.md)**: volta-sdk-jsが現在のJWTの期限切れ時に自動的に新しいJWTを取得し、シームレスに新しい鍵に切り替わる

## よくある間違いと攻撃

### 間違い1: 即座の鍵切り替え

ローテーション直後に古い鍵を削除すると、飛行中のすべてのトークンで検証が失敗します。古い鍵は少なくともJWT TTL期間は利用可能に保ちましょう。

### 間違い2: JWKSに両方の鍵を公開しない

JWKSエンドポイントが新しい鍵だけを公開すると、JWKSを検証に使う下流サービスが古い鍵のトークンを検証できません。重複期間中は両方の鍵をJWKSに含めるべきです。

### 間違い3: 重複が短すぎる

重複がJWT TTLより短いと、一部の有効なトークンが検証不能になります。voltaの5分のJWT TTLは、重複が少なくとも5分必要であることを意味します。

### 間違い4: 重複が長すぎる

引退した鍵をいつまでも残すと攻撃面が増えます。引退した鍵が侵害されていた場合、それで署名されたトークンがまだ受け入れられてしまいます。重複後は速やかに引退した鍵を削除しましょう。

### 間違い5: 移行中の監視なし

ローテーション中に何か問題が起きたら（例：JWKSキャッシュがリフレッシュされない）、すぐに知る必要があります。鍵ローテーション中は401エラー率を監視しましょう。
