# グリーンフィールド

[English version](greenfield.md)

---

## これは何？

グリーンフィールドプロジェクトとは、既存のコードベースも、レガシーの制約も、後方互換性の要件もなく、ゼロから構築するソフトウェアプロジェクトです。空のフォルダから始めて、すべての決定を新たに行います。

この用語は建設業界から来ています。グリーンフィールドサイトは未開発の土地 -- 文字通り緑の野原 -- で、建設者は既存の構造物、配管、ゾーニングの妥協を心配せずに何でも設計できます。反対はブラウンフィールドサイト：以前開発された土地で、汚染されている可能性があり、既存のものを回避して作業しなければなりません。

ソフトウェアにおけるグリーンフィールドは完全な自由を意味します。言語、フレームワーク、データベース、アーキテクチャを自分で選びます。古い決定に制約されません。これは機会であると同時に危険でもあります -- 規律のない自由は、逃げようとしていた混乱と同じものを生み出します。

---

## なぜ重要？

グリーンフィールドの機会は稀で価値があります。ほとんどのソフトウェア作業はブラウンフィールドです：既存システムの保守、拡張、リファクタリング。グリーンフィールドで始められるとき、最初の数週間の決定がプロジェクトを何年も形作ります。

だからこそ、グリーンフィールドプロジェクトにはより少ない規律ではなく、より多くの規律が求められます。ブラウンフィールドプロジェクトでは、アーキテクチャはすでに決まっています（良くも悪くも）。グリーンフィールドプロジェクトでは、すべての悪い初期決定が、将来の開発者が呪うレガシーになります。

グリーンフィールドプロジェクトの核心的な問い：「2年後もこのコードベースがクリーンであるようにするには？」答えは通常：小さなスコープ（MVP/YAGNI）、強い規約、そして過剰設計への衝動を抑えること。

## どう動くのか？

### グリーンフィールド vs. ブラウンフィールド比較

| 観点 | グリーンフィールド | ブラウンフィールド |
|------|-------------------|-------------------|
| 出発点 | 空のフォルダ | 既存コードベース |
| 技術選択 | 無制限 | 既存スタックに制約 |
| アーキテクチャ | ゼロから設計 | 既存パターン内で作業 |
| リスク | 間違ったものを作る | 既存のものを壊す |
| 速度（初期） | 速い -- 制約なし | 遅い -- 既存コードの理解が必要 |
| 速度（6ヶ月後） | 初期決定次第 | 技術的負債の量次第 |
| チーム知識 | 全員がイーブン | 暗黙知が重要 |

### グリーンフィールドのライフサイクル

```
  フェーズ1：陶酔（1-4週目）
  ┌──────────────────────────────┐
  │  「何でもできる！」            │
  │  「レガシーの制約なし！」       │
  │  「最新技術を使おう...」       │
  └──────────────────────────────┘
              │
              ▼
  フェーズ2：決定（2-8週目）
  ┌──────────────────────────────┐
  │  言語は？FWは？DBは？          │
  │  アーキテクチャ？デプロイ？     │
  │  テスト戦略？CI/CD？          │
  └──────────────────────────────┘
              │
              ▼
  フェーズ3：現実（2-6ヶ月目）
  ┌──────────────────────────────┐
  │  初期の決定が制約になった。     │
  │  グリーンフィールドが            │
  │  ブラウンフィールドになりつつある。│
  └──────────────────────────────┘
              │
              ▼
  フェーズ4：新しいレガシー（6ヶ月以降）
  ┌──────────────────────────────┐
  │  「なんでこうしたんだっけ？」   │
  │  次の人にとっては、これは       │
  │  もうブラウンフィールド。       │
  └──────────────────────────────┘
```

### よくあるグリーンフィールドの罠

**「全部新しいものを使おう」の罠**：グリーンフィールドはピカピカの技術を全部採用する口実ではありません。新言語 + 新 FW + 新 DB + 新デプロイ基盤 = 理解していないもの5つを同時に扱うこと。

**「完璧に設計しよう」の罠**：1行もコードを書く前にアーキテクチャに何週間も費やす。完璧に設計するための情報はまだありません。何かを出して、学んで、改善する。

**「後で整理しよう」の罠**：「まだプロトタイプだから」とぐちゃぐちゃなコードを書く。1週目に書いたグリーンフィールドのコードが3年後もまだ本番で動いている。

## volta-auth-proxy ではどう使われている？

volta-auth-proxy はグリーンフィールドプロジェクトです。ブラウンフィールドのアイデンティティシステムでの経験に基づく意図的な技術選択のもと、ゼロから始められました。

### なぜグリーンフィールドが正解だったか

volta を構築する代わりに、既存のアイデンティティプラットフォーム（Keycloak、Auth0 セルフホスト等）を拡張するという選択肢がありました。このブラウンフィールドアプローチは以下の理由で却下されました：

| ブラウンフィールド選択肢 | 問題 |
|------------------------|------|
| Keycloak を拡張 | 30秒の起動、XML 設定地獄、Java EE の複雑さ |
| 既存プロキシをフォーク | 同意できないアーキテクチャ決定を継承 |
| Auth0/Firebase をラップ | ベンダーロックイン、認証ロジックの制御不能 |

ゼロから始めることで volta は主張のある選択ができました：

- **Spring Boot でなく Javalin**：軽量、明示的、起動 ~200ms vs ~5s
- **PostgreSQL でなく SQLite**：Phase 1 は運用ゼロ
- **マイクロサービスでなくシングルプロセス**：デバッグは1箇所で
- **設定ファイルでなく DSL**：型安全、バージョン管理されたポリシー

### volta でのグリーンフィールド規律

volta は設計哲学を通じてよくあるグリーンフィールドの罠を避けています：

1. **YAGNI / フェーズミニマル**：Phase 1 で Phase 2 の機能を作らない。「全部使おう」の罠を防止。
2. **意図的な密結合**：「理解できる地獄を選べ」。過度な抽象化を防止。
3. **制御が王様**：認証ロジックの全行が書かれたもの。フレームワークから生成・インポートされたものではない。「魔法のフレームワーク」の罠を防止。

### ブラウンフィールドの未来

volta はいずれブラウンフィールドプロジェクトになります。今適用している規律 -- クリーンコード、明確な境界、丁寧なドキュメント -- はその未来への投資です。

## よくある間違いと攻撃

### 間違い1：グリーンフィールドを実験許可と見なす

グリーンフィールドプロジェクトは初日から本番システムです。実験的な技術選択は概念実証には問題ありませんが、製品の基盤には不適切です。

### 間違い2：ブラウンフィールドの現実を無視する

グリーンフィールドプロジェクトにも制約はあります：チームのスキル、デプロイ環境、ユーザーの期待。Java 開発者のチームに Haskell を選ぶのは、暴走したグリーンフィールド思考です。

### 間違い3：決定を文書化しない

ブラウンフィールドプロジェクトでは、コードが決定を（不十分ながら）文書化します。グリーンフィールドプロジェクトには歴史がありません。Spring Boot でなく Javalin を選んだ理由を書き残さなければ、次の開発者は疑問に思い、書き直すかもしれません。

### 間違い4：スコープの爆発

「ゼロから始めるんだから、今度こそ正しく作ろう」がすぐに「全部作ろう」になります。グリーンフィールドプロジェクトには、ブラウンフィールド以上に厳格なスコープ制御が必要です。[yagni.md](yagni.md) を参照。
