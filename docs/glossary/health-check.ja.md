# ヘルスチェック

[English version](health-check.md)

---

## これは何？

ヘルスチェックは、サービスが稼働中で正常に動作しているかを報告するエンドポイントです。ヘルスチェック URL にリクエストを送ると、サービスはシンプルなステータスメッセージ -- 通常 `{"status":"ok"}` -- で応答します。サービスがダウンまたは不健全な場合、リクエストは失敗するかエラーステータスを返します。

医者が脈を取るようなものです。生きているか知るのに全身 MRI は不要です -- シンプルなバイタルサインを1つチェックするだけ。ヘルスチェックもソフトウェアに同じことをします：「動いてますか？」という基本的な質問に素早く安価に答えます。

ヘルスチェックは自動運用の基盤です。コンテナオーケストレーター（Docker、Kubernetes）、ロードバランサー（Traefik、Nginx）、監視システム（Prometheus、Datadog）はすべて、サービスにトラフィックをルーティングするか、再起動するか、エンジニアにアラートするかを決定するためにヘルスチェックを使用します。

---

## なぜ重要？

ヘルスチェックがなければ、ユーザーが苦情を言うまでサービスのダウンに気づきません。ヘルスチェックがあれば：

- **自動再起動**：Docker/Kubernetes がクラッシュしたコンテナを数秒以内に再起動可能
- **トラフィックルーティング**：ロードバランサーが不健全なインスタンスへのリクエスト送信を停止
- **アラート**：監視システムがヘルスチェック失敗時にエンジニアに通知
- **デプロイの安全性**：ローリングデプロイが新インスタンスのヘルスチェック通過を待ってから旧インスタンスを停止
- **依存関係の認識**：ヘルスチェックが重要な依存関係（データベース、外部サービス）の到達可能性を検証可能

ヘルスチェックのないサービスは、ダッシュボードのない車のようなものです -- 問題なく走っているかもしれませんが、高速道路で故障するまで知る方法がありません。

## どう動くのか？

### ヘルスチェックの種類

| 種類 | チェック内容 | 速度 | 用途 |
|------|-----------|------|------|
| **Liveness** | プロセスは動いているか？ | 非常に高速（<10ms） | コンテナ再起動の判断 |
| **Readiness** | サービスはリクエストを処理できるか？ | 高速（<100ms） | ロードバランサーのルーティング |
| **Startup** | サービスは初期化を完了したか？ | 一回きり | 早すぎるヘルスチェックの回避 |
| **Deep health** | すべての依存関係は健全か？ | やや遅い（<500ms） | 詳細ステータスページ |

### 典型的なヘルスチェックエンドポイント

```
  GET /healthz          → Liveness チェック（生きてる？）
  GET /readyz           → Readiness チェック（トラフィック捌ける？）
  GET /healthz/detail   → Deep チェック（依存関係は OK？）
```

### ヘルスチェックのレスポンス形式

```json
  // シンプル（liveness）：
  {
    "status": "ok"
  }

  // 詳細（deep health）：
  {
    "status": "ok",
    "checks": {
      "database": { "status": "ok", "latency_ms": 2 },
      "disk_space": { "status": "ok", "free_gb": 42 },
      "memory": { "status": "ok", "used_percent": 65 }
    },
    "uptime_seconds": 86400
  }
```

### オーケストレーターがヘルスチェックを使う方法

```
  Docker Compose / Kubernetes
         │
         │  10秒ごと：
         │  GET /healthz
         │
         ▼
  ┌──────────────────────┐
  │  ヘルスチェック通過   │──► サービスは健全
  │  （HTTP 200）         │    トラフィックルーティング継続
  └──────────────────────┘

  ┌──────────────────────┐
  │  ヘルスチェック失敗   │──► 3回連続失敗？
  │  （タイムアウト/5xx/  │    ├── はい：コンテナ再起動
  │   無応答）            │    └── いいえ：次のインターバルで再試行
  └──────────────────────┘
```

### Docker でのヘルスチェック設定

```yaml
  services:
    volta-auth-proxy:
      image: volta-auth-proxy:latest
      healthcheck:
        test: ["CMD", "curl", "-f", "http://localhost:8080/healthz"]
        interval: 10s
        timeout: 5s
        retries: 3
        start_period: 5s
```

## volta-auth-proxy ではどう使われている？

volta は `GET /healthz` でヘルスチェックエンドポイントを公開し、HTTP 200 で `{"status":"ok"}` を返します。

### volta のヘルスチェック設計

```java
  // Main.java のルート設定内：
  app.get("/healthz", ctx -> {
      ctx.json(Map.of("status", "ok"));
  });
```

これは意図的にシンプルです：

| 設計判断 | 根拠 |
|---------|------|
| **認証不要** | ヘルスチェックはセッションなしでアクセス可能でなければならない |
| **データベースクエリなし** | チェックを高速に保ち、遅いクエリによる偽陰性を回避 |
| **最小限のレスポンス** | `{"status":"ok"}` のみ返す -- バージョン情報も内部詳細もなし |
| **依存関係チェックなし** | Phase 1 YAGNI -- SQLite はローカル、チェックすべきネットワーク依存なし |

### なぜ /healthz（/health ではなく）

`/healthz` の慣例は Google の Borg/Kubernetes エコシステムに由来します。`z` サフィックスはアプリケーションルートとの衝突を避ける慣例（`/readyz`、`/livez` のように）です。事実上の標準になっています。

### volta の Docker セットアップでのヘルスチェック

volta の Docker 設定はヘルスチェックを使ってコンテナの準備完了を判断します：

```yaml
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8080/healthz"]
    interval: 10s
    timeout: 5s
    retries: 3
    start_period: 2s   # volta は ~200ms で起動するので、2秒は十分
```

volta は約 200ms で起動するため（[startup.md](startup.md) 参照）、`start_period` は非常に短くできます。60秒以上の `start_period` が必要な Keycloak と比較してください。

### Traefik との統合

volta が Traefik の背後で動作する場合、Docker ラベルが Traefik にヘルスチェックをルーティング判断に使うよう設定します。詳細は [docker-label.md](docker-label.md) を参照。

## よくある間違いと攻撃

### 間違い1：ヘルスチェックに認証を要求する

ヘルスチェックエンドポイントにセッションや API キーが必要だと、オーケストレーターがチェックできません。ヘルスチェックは公開アクセス可能（または内部ネットワーク内でアクセス可能）でなければなりません。

### 間違い2：ヘルスチェックが高コスト

データベースクエリ、外部 API 呼び出し、チェックサム計算を行うヘルスチェックは遅く、高負荷時に自身が障害の原因になる可能性があります。liveness チェックは安価に。詳細ステータスが必要なら別の deep-health エンドポイントを使いましょう。

### 間違い3：ヘルスチェックが機密情報を公開する

```
  悪い例：
  {
    "status": "ok",
    "version": "1.2.3",
    "database": "sqlite:///var/lib/volta/data.db",
    "java_version": "21.0.1",
    "uptime": "3 days 4 hours"
  }

  良い例：
  {
    "status": "ok"
  }
```

内部の詳細を公開すると、攻撃者がスタックを理解する助けになります。レスポンスは最小限に。

### 間違い4：ヘルスチェックが全くない

ヘルスチェックがないと、オーケストレーターはプロセスが動作している限りコンテナが健全だと仮定します。しかしプロセスは壊れた状態で動作しているかもしれません（デッドロック、メモリ不足、データベース接続の喪失）。ヘルスチェックはこれらのケースを検出します。

### 間違い5：ヘルスチェックが常に 200 を返す

アプリケーションが壊れていても `{"status":"ok"}` を返すヘルスチェックは、ヘルスチェックがないより悪い -- 問題を積極的に隠します。リクエストを処理できないなら、ヘルスチェックはそれを反映しなければなりません。
