# 高可用性（High Availability）

[English version](high-availability.md)

---

## これは何？

高可用性（HA）とは、個々のコンポーネントが故障してもシステムが動き続けるように設計することです。何も壊れないことを祈るのではなく、障害を計画し、障害を通じてもシステムがユーザーにサービスを提供し続けることを保証します。

バックアップ発電機のある病院で考えてみましょう。メインの電力網が故障しても、バックアップがあるので病院は動き続けます。そしてバックアップにもバックアップがあります。患者は電力の切り替えに気づきません。システムが障害を優雅に処理するよう設計されているからです。高可用性はソフトウェアでの同じ概念です：サーバーが死に、データベースが不調になり、ネットワークケーブルが抜かれても、Webサイトは稼働し続けます。

高可用性は「ナイン」で測定されます -- システムが稼働している時間の割合です：

```
  ┌────────────┬─────────────────┬──────────────────┐
  │ ナイン     │ 稼働率 %        │ 年間ダウンタイム  │
  ├────────────┼─────────────────┼──────────────────┤
  │ ツーナイン  │ 99%             │ 3.65日           │
  │ スリーナイン│ 99.9%           │ 8.76時間         │
  │ フォーナイン│ 99.99%          │ 52.6分           │
  │ ファイブナイン│ 99.999%        │ 5.26分           │
  └────────────┴─────────────────┴──────────────────┘
```

---

## なぜ重要なのか？

- **認証はクリティカルインフラ。** volta-auth-proxyがダウンすると、その背後のすべてのアプリにアクセスできなくなる。ログインなし、アクセスなし、ビジネスなし。
- **ユーザーは常時稼働サービスを期待する。** 現代のSaaSユーザーは99.9%以上の稼働率を期待する。ダウンタイムは顧客と信頼を失う。
- **停止にはカスケード効果がある。** 認証がダウンすると、ログインだけでなく、認証を要するすべてのAPI呼び出しが失敗する。
- **ダウンタイムのコストは現実。** SaaSビジネスにとって、1時間のダウンタイムは収益損失、SLA違反、顧客離反を意味しうる。
- **コンプライアンスが要求する場合がある。** エンタープライズ顧客は契約で特定の稼働率保証を要求することが多い。

## どう動くのか？

### 高可用性の3つの柱

```
  ┌───────────────────────────────────────────────────────┐
  │              高可用性                                   │
  │                                                        │
  │  ┌─────────────┐ ┌──────────────┐ ┌───────────────┐  │
  │  │  冗長性     │ │ フェイルオーバ │ │  監視         │  │
  │  │             │ │              │ │               │  │
  │  │ すべてを    │ │ プライマリが  │ │ ユーザーが    │  │
  │  │ 複数コピー  │ │ 故障時に     │ │ 気づく前に    │  │
  │  │             │ │ バックアップ  │ │ 障害を検知    │  │
  │  │             │ │ に自動切替   │ │               │  │
  │  └─────────────┘ └──────────────┘ └───────────────┘  │
  └───────────────────────────────────────────────────────┘
```

### 単一障害点

単一障害点（SPOF）とは、故障するとシステム全体がダウンするコンポーネントのことです：

```
  SPOFの例（Phase 1）:
  ┌──────┐    ┌──────────┐    ┌──────────┐
  │クライアント│───▶│volta (1) │───▶│Postgres  │
  └──────┘    └──────────┘    └──────────┘
                  ↑ SPOF          ↑ SPOF

  HAの例（Phase 2）:
  ┌──────┐    ┌──────────────┐    ┌──────────────┐
  │クライアント│───▶│ロードバランサ │───▶│volta-1       │
  └──────┘    │              │    │volta-2       │
              │              │    │volta-3       │
              └──────────────┘    └──────────────┘
                                       │
                                  ┌────┴────┐
                                  ▼         ▼
                            ┌──────┐  ┌──────┐
                            │PG    │  │PG    │
                            │プライマリ│  │レプリカ│
                            └──────┘  └──────┘
              どの単一インスタンスも死んでよい。
              システムは動き続ける。
```

### ヘルスチェック

ロードバランサはどのインスタンスが健全かを知る必要があります：

```
  ロードバランサがヘルスチェックを送信:

  GET /health → volta-1 → 200 OK     ✓ トラフィックを送り続ける
  GET /health → volta-2 → 200 OK     ✓ トラフィックを送り続ける
  GET /health → volta-3 → タイムアウト  ✗ トラフィック送信停止

  ┌──────────────┐
  │ロードバランサ │
  └──┬─────┬─────┘
     │     │
     ▼     ▼
  ┌─────┐ ┌─────┐ ┌─────┐
  │ v1  │ │ v2  │ │ v3  │
  │ ✓   │ │ ✓   │ │ ✗   │ ← プールから除外
  └─────┘ └─────┘ └─────┘
```

### データベースの高可用性

PostgreSQLのHAは通常、プライマリ-レプリカ レプリケーションを使用します：

```
  ┌───────────────┐   レプリケーション  ┌───────────────┐
  │ PGプライマリ   │ ──────────────▶  │ PGレプリカ     │
  │ (読み書き)     │                  │ (読み取り専用)  │
  └───────────────┘                  └───────────────┘
         │                                  │
         └── プライマリ故障時 ────────────────┘
                                    ↓
                          レプリカがプライマリに
                          昇格
```

## volta-auth-proxy ではどう使われている？

### Phase 1：HAなし（現在、許容範囲）

Phase 1では、voltaは[シングルプロセス](single-process.ja.md)です。ダウンすれば認証がダウンします。シンプルさが稼働率保証より重要な初期段階の製品では、これは許容範囲です。

### Phase 2：HAアーキテクチャ（計画中）

```
  ┌──────────────────────────────────────────────────┐
  │                  HAアーキテクチャ                   │
  │                                                    │
  │  ┌────────────────────────────┐                   │
  │  │    Traefik（ロードバランサ）  │                   │
  │  │    10秒ごとにヘルスチェック   │                   │
  │  └────┬─────────┬─────────┬──┘                   │
  │       │         │         │                       │
  │       ▼         ▼         ▼                       │
  │  ┌────────┐ ┌────────┐ ┌────────┐               │
  │  │volta-1 │ │volta-2 │ │volta-3 │  ← HA最低2    │
  │  └───┬────┘ └───┬────┘ └───┬────┘               │
  │      │          │          │                     │
  │      ▼          ▼          ▼                     │
  │  ┌──────────────────────────────┐                │
  │  │       Redis Sentinel         │                │
  │  │  (Redisの自動フェイルオーバ)   │                │
  │  └──────────────────────────────┘                │
  │      │          │          │                     │
  │      ▼          ▼          ▼                     │
  │  ┌──────────────────────────────┐                │
  │  │  PostgreSQL プライマリ/レプリカ│                │
  │  │  (ストリーミングレプリケーション)│                │
  │  └──────────────────────────────┘                │
  └──────────────────────────────────────────────────┘
```

### HAがvoltaユーザーに意味すること

HAが整備されると：
- voltaインスタンスのクラッシュがユーザーに影響しない（ロードバランサが回避）
- Redis障害時はPostgreSQLにフォールバック（遅いが機能する）
- PostgreSQLプライマリ障害時はレプリカが自動昇格
- デプロイはローリングアップデートで実行（ゼロダウンタイム）

## よくある間違いと攻撃

### 間違い1：アプリケーション層だけのHA

voltaインスタンスを3つ動かしてもPostgreSQLサーバーが1つだけではHAではありません。データベースが単一障害点になります。HAはすべての層で実装する必要があります。

### 間違い2：ヘルスチェックがない、または不適切

データベースに到達できなくても常に200 OKを返すヘルスチェックは目的を果たしません。ヘルスチェックは実際の依存関係を検証すべきです。

### 間違い3：相関障害

すべてのインスタンスを同じ物理サーバーで動かすと、ハードウェア障害ですべてがダウンします。異なるホストやアベイラビリティゾーンにインスタンスを分散させましょう。

### 間違い4：障害シナリオをテストしない

「HAがある」は、ステージングでインスタンスを実際に停止させてシステムが稼働し続けることを検証したことがなければ無意味です。カオスエンジニアリングを実践しましょう。

### 間違い5：HAの早期過剰化

HAは大幅な運用上の複雑さを追加します。10ユーザーの新製品では、適切なバックアップと迅速な復旧を持つシングルインスタンスで十分なことが多い。ビジネスが要求するときにHAを実装しましょう。
