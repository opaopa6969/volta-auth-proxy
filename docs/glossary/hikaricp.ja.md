# HikariCP

[English version](hikaricp.md)

---

## これは何？

HikariCPはJava向けのデータベースコネクションプールです。再利用可能なデータベース接続のセットを管理し、データベースと通信するたびに新しい接続を作る必要をなくします。

コールセンターの電話回線で考えてみましょう。プールがないと、お客さんが電話するたびに新しい電話回線を敷設し、1回の通話に使い、通話が終わったら撤去します。ばかげています -- 電話回線の敷設には時間もお金もかかります。代わりに、コールセンターには10本の電話回線が常設されています。お客さんが電話すると、空いている回線が割り当てられます。通話が終わると、回線は次のお客さんのために使えるようになります。HikariCPはデータベースのためにその電話回線を管理するシステムです。

（「光」を意味する日本語で、このプールが極めて高速で軽量であることを反映しています。）

---

## なぜ重要なのか？

データベース接続の作成はコストが高いです。各接続には以下が必要です：

1. データベースサーバーへのTCPソケットを開く
2. TLSハンドシェイク（暗号化の場合）
3. 認証（ユーザー名/パスワード）
4. プロトコルパラメータのネゴシエーション
5. サーバー側のリソース割り当て

これは接続ごとに50〜200msかかります。WebアプリケーションがすべてのHTTPリクエストで新しい接続を作り、毎秒100リクエストがあると、毎秒100回の接続セットアップ -- 時間とリソースの大きな無駄です。

```
  コネクションプールなし：
  リクエスト1 → 接続を開く(100ms) → クエリ(5ms) → 接続を閉じる
  リクエスト2 → 接続を開く(100ms) → クエリ(5ms) → 接続を閉じる
  リクエスト3 → 接続を開く(100ms) → クエリ(5ms) → 接続を閉じる
  合計：3リクエストで315ms

  HikariCPあり：
  （接続はすでに開かれてプールで待機中）
  リクエスト1 → 接続を借りる(0.1ms) → クエリ(5ms) → プールに返却
  リクエスト2 → 接続を借りる(0.1ms) → クエリ(5ms) → プールに返却
  リクエスト3 → 接続を借りる(0.1ms) → クエリ(5ms) → プールに返却
  合計：3リクエストで15.3ms
```

HikariCPは事前に確立された接続のプールをすぐ使えるように保持します。コードが接続を必要とするとき、プールから1つ借ります（マイクロ秒）。終わったら、閉じる代わりにプールに接続を返します。接続は次のリクエストのために生き続けます。

## なぜ特にHikariCPなのか？

Javaにはいくつかのコネクションプールがあります：DBCP2、C3P0、Tomcat Pool、HikariCP。HikariCPが最も人気なのは：

| 特徴 | HikariCP |
|------|----------|
| **速度** | 最速のJavaコネクションプール（ベンチマーク済み） |
| **サイズ** | ~130KBのJAR、最小限の依存関係 |
| **信頼性** | 何百万ものアプリの本番で使用 |
| **デフォルト値** | そのままで機能する妥当なデフォルト |
| **監視** | Micrometer/Dropwizard経由の組み込みメトリクス |

Spring Bootのデフォルトコネクションプールであり、その品質と信頼性を物語っています。

## voltaでのHikariCPの使い方

volta-auth-proxyはHikariCPを使ってPostgreSQLへの接続を管理しています。すべてのデータベース操作 -- セッション検索、ユーザークエリ、テナント解決、監査ログ -- がHikariCPプールを経由します。

```
  volta-auth-proxy
  ┌─────────────────────────────────────┐
  │                                     │
  │  ForwardAuthハンドラ ──┐            │
  │  ログインハンドラ ──────┤            │
  │  APIハンドラ ──────────┤            │
  │  Flywayマイグレーション ┤            │
  │                        ▼            │
  │  ┌─────────────────────────────┐    │
  │  │  HikariCPコネクションプール  │    │
  │  │  ┌────┐ ┌────┐ ┌────┐      │    │
  │  │  │接続│ │接続│ │接続│ ...   │    │
  │  │  └────┘ └────┘ └────┘      │    │
  │  └─────────────┬───────────────┘    │
  │                │                    │
  └────────────────┼────────────────────┘
                   │
                   ▼
  ┌─────────────────────────────────────┐
  │  PostgreSQL                         │
  │  (sessions、users、tenants、        │
  │   audit logsなど)                   │
  └─────────────────────────────────────┘
```

### voltaのプール設定

voltaはHikariCPを妥当なデフォルトで設定しています：

```java
HikariConfig config = new HikariConfig();
config.setJdbcUrl(databaseUrl);
config.setUsername(user);
config.setPassword(password);
config.setMaximumPoolSize(10);  // 最大10接続
```

主要な設定：

- **最大プールサイズ：** 同時に開ける接続数。voltaは軽量で多くの同時クエリを必要としないため、小さなプール（デフォルト10）を使う。
- **接続タイムアウト：** プールから接続を取得するまでの最大待ち時間。
- **アイドルタイムアウト：** 未使用の接続がプールに残る時間。

### 小さなプールで十分な理由

voltaは効率的に設計されています。各リクエストのデータベース操作は最小限です：

- ForwardAuth: 1〜2クエリ（セッション検索 + オプションのテナントチェック）
- ログイン: 2〜3クエリ（ユーザー検索 + セッション作成）
- API呼び出し: 1〜3クエリ

10接続でクエリが約5msなら、voltaはたった10のプール接続で毎秒約2,000リクエストを処理できます。これがvoltaが約30MB RAMで動作する理由です -- 大きなプールは必要ありません。

## シンプルな例

コードでHikariCPを使うとこうなります：

```java
// プールを作成（アプリケーション起動時に1回だけ）
HikariDataSource ds = new HikariDataSource(config);

// 接続を使う（プールから借りる）
try (Connection conn = ds.getConnection()) {
    PreparedStatement stmt = conn.prepareStatement(
        "SELECT * FROM sessions WHERE id = ?"
    );
    stmt.setObject(1, sessionId);
    ResultSet rs = stmt.executeQuery();
    // ... 結果を使う ...
}
// try-with-resourcesブロックが終わると接続は自動的にプールに返却
```

`try-with-resources`ブロックにより、例外が発生しても接続がプールに返却されることが保証されます。これは重要です -- リークした接続（借りたまま返さない）は最終的にプールを使い果たします。
