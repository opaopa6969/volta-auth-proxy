# 水平スケーリング

[English version](horizontal-scaling.md)

---

## これは何？

水平スケーリングとは、1台のサーバーを大きくするのではなく、サーバーインスタンスを追加してより多くのトラフィックを処理することです。アプリケーションの1コピーから2コピー、4コピー、10コピーへと増やし、各インスタンスがリクエスト全体の一部を処理します。

スーパーマーケットで考えてみましょう。垂直スケーリングは1人のレジ担当者をより速く働かせること（より良いハードウェア）。水平スケーリングはレジの列を増やすこと（より多くのインスタンス）。1人のレジ担当者がスキャンできる速度には限界がありますが、もう1つ列を開けることは常にできます。課題は調整です：お客様が誤って「閉鎖中」の列に行かないこと、在庫システムがすべての列で一貫していることです。

水平スケーリングは、現代のWebアプリケーションが数百万ユーザーを処理する方法です。1台の巨大で高価なサーバーを買う代わりに、多数の小型で安価なサーバーを実行します。

---

## なぜ重要なのか？

- **垂直スケーリングには上限がある。** お金で買える最大のサーバーにもCPUとRAMの上限がある。水平スケーリングには理論上の限界がない。
- **水平スケーリングは冗長性を可能にする。** 1つのインスタンスが死んでも、他がトラフィックを処理し続ける。これが[高可用性](high-availability.ja.md)の基盤。
- **コスト効率。** 3台の小さいサーバーは、同等の合計容量を持つ1台の大きいサーバーより安いことが多い。
- **ゼロダウンタイムデプロイ。** インスタンスを1つずつ更新（ローリングデプロイ）でき、システム全体をオフラインにしない。
- **地理的分散。** インスタンスをユーザーに近い異なるデータセンターで実行できる。

## どう動くのか？

### 垂直 vs 水平スケーリング

```
  垂直スケーリング:                 水平スケーリング:
  (より大きいサーバー)              (より多くのサーバー)

  前:            後:               前:              後:
  ┌──────┐     ┌──────────┐       ┌──────┐      ┌──────┐ ┌──────┐
  │ 2 CPU│     │  8 CPU   │       │ 2 CPU│      │ 2 CPU│ │ 2 CPU│
  │ 4 GB │     │  32 GB   │       │ 4 GB │      │ 4 GB │ │ 4 GB │
  │      │     │          │       │      │      │      │ │      │
  └──────┘     └──────────┘       └──────┘      └──────┘ └──────┘
  1サーバー     1台の大きいサーバー   1サーバー     2サーバー
```

### 水平スケーリングの要件

同じアプリケーションの複数インスタンスを実行するには：

| 要件 | 理由 | 解決策 |
|-----|------|--------|
| **ステートレスアプリ** | インスタンスがローカルに状態を保持できない | 共有サービスに状態を格納 |
| **共有データベース** | すべてのインスタンスが同じデータを読み書き | PostgreSQL（すでに共有） |
| **共有セッションストア** | インスタンス1のセッションをインスタンス2が読める必要 | [Redis](redis.ja.md) |
| **ロードバランサ** | リクエストをインスタンス間に分配 | [Traefik](load-balancer.ja.md) |
| **共有キャッシュ** | レート制限が全インスタンスでカウントされる必要 | [Redis](redis.ja.md) |

### スケーリングのフロー

```
  シングルインスタンス（Phase 1）:

  全トラフィック → ┌──────────┐ → PostgreSQL
                  │ volta (1) │
                  └──────────┘

  水平スケール（Phase 2）:

                  ┌──────────────┐
  全トラフィック → │ロードバランサ │
                  └──┬───┬───┬──┘
                     │   │   │
                  ┌──┴┐ ┌┴──┐ ┌┴──┐
                  │ v1│ │ v2│ │ v3│  ← 同一インスタンス
                  └─┬─┘ └─┬─┘ └─┬─┘
                    │     │     │
                    ▼     ▼     ▼
                 ┌──────────────────┐
                 │    PostgreSQL    │ ← 共有データベース
                 └──────────────────┘
                 ┌──────────────────┐
                 │      Redis       │ ← 共有セッション+キャッシュ
                 └──────────────────┘
```

### 「ステートレス」とは何か

アプリケーションがステートレスとは、どのインスタンスでも、その特定インスタンスのメモリにしかないデータを必要とせずに、どんなリクエストでも処理できることです。

```
  ステートフル（水平スケール不可）:
  ┌──────────┐
  │インスタンス1│
  │           │
  │ セッションA│ ← インスタンス1だけがセッションAを知っている
  │ セッションB│
  └──────────┘

  セッションAのリクエストがインスタンス2に到達 → 失敗（セッション見つからず）

  ステートレス（水平スケール可能）:
  ┌──────────┐  ┌──────────┐
  │インスタンス1│  │インスタンス2│
  │           │  │           │
  │（ローカル  │  │（ローカル  │
  │ セッション │  │ セッション │
  │  なし）    │  │  なし）    │
  └─────┬─────┘  └─────┬─────┘
        │              │
        ▼              ▼
  ┌──────────────────────┐
  │ Redis（全セッション）  │
  └──────────────────────┘

  セッションAのリクエストがインスタンス2に到達 → Redis確認 → 発見
```

## volta-auth-proxy ではどう使われている？

### Phase 1：シングルインスタンス（現在）

voltaは現在[シングルプロセス](single-process.ja.md)として動作します。セッション状態はPostgreSQLに存在し、[Caffeine](caffeine-cache.ja.md)のローカルキャッシュがあります。運用がシンプルで、現在の規模には十分です。

### Phase 2：水平スケーリング（計画中）

Phase 2は複数のvoltaインスタンスの実行に必要なインフラを追加します：

```
  Phase 2アーキテクチャ:

  ┌──────────────────────────────────────────┐
  │              インターネット                │
  └─────────────────┬────────────────────────┘
                    │
                    ▼
  ┌──────────────────────────────────────────┐
  │      Traefik（ロードバランサ）             │
  │   voltaインスタンス間でラウンドロビン       │
  └────┬──────────┬──────────┬──────────────┘
       │          │          │
       ▼          ▼          ▼
  ┌────────┐ ┌────────┐ ┌────────┐
  │volta-1 │ │volta-2 │ │volta-3 │
  │        │ │        │ │        │
  │Caffeine│ │Caffeine│ │Caffeine│  ← L1キャッシュ（ローカル）
  └───┬────┘ └───┬────┘ └───┬────┘
      │          │          │
      ▼          ▼          ▼
  ┌──────────────────────────────────┐
  │           Redis                   │  ← L2キャッシュ（共有）
  │   セッション、レート制限            │
  └──────────────────────────────────┘
      │          │          │
      ▼          ▼          ▼
  ┌──────────────────────────────────┐
  │         PostgreSQL                │  ← 真のデータソース
  │   ユーザー、テナント、メンバーシップ │
  └──────────────────────────────────┘
```

### Phase 1からPhase 2への変更点

| コンポーネント | Phase 1 | Phase 2 |
|--------------|---------|---------|
| インスタンス数 | 1 | 2以上 |
| セッションキャッシュ | Caffeineのみ | Caffeine + Redis |
| レート制限 | Caffeineカウンター | Redisカウンター |
| セッション無効化 | ローカルキャッシュクリア | Redis Pub/Subブロードキャスト |
| ロードバランシング | 不要 | Traefikラウンドロビン |
| デプロイ | 停止-起動 | ローリングアップデート |

## よくある間違いと攻撃

### 間違い1：状態を削除する前にスケール

アプリケーションがローカルメモリ（Caffeineのみ）にセッションを格納したまま2番目のインスタンスを追加すると、ユーザーの半分がランダムに「セッションが見つかりません」エラーを受けます。まずローカル状態への依存を除去しましょう。

### 間違い2：データベース接続を忘れる

各voltaインスタンスは独自の[コネクションプール](connection-pool.ja.md)を開きます。10接続ずつの3インスタンスはPostgreSQLに30接続です。PostgreSQLが合計を処理できることを確認しましょう。

### 間違い3：不均等な負荷分配

設定ミスのロードバランサがトラフィックの90%を1インスタンスに、10%を別のインスタンスに送るかもしれません。適切なヘルスチェックとロードバランシングアルゴリズムを使いましょう。

### 間違い4：複数インスタンスでローカルテストしない

「1インスタンスで動く」は「3インスタンスで動く」を意味しません。ステージングで複数インスタンスと現実的なトラフィックパターンで水平スケーリングをテストしましょう。

### 間違い5：最適化の代替としてスケール

1インスタンスが100リクエスト/秒を処理し、200が必要な場合、複雑さを追加する前に最適化で1インスタンスで200を達成できないか確認しましょう。水平スケーリングは運用オーバーヘッドを追加します。
