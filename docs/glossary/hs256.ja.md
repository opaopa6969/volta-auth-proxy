# HS256 (HMAC with SHA-256)

[English version](hs256.md)

---

## これは何？

HS256は、単一の共有秘密鍵を使ってデータに署名する方法です。署名者と検証者がまったく同じ鍵を使います。HMAC（Hash-based Message Authentication Code）とSHA-256ハッシュを組み合わせたものです。鍵ペアを使うRS256とは異なり、HS256は対称的です。同じ秘密が署名の作成と検証の両方を行います。

金庫の共有パスワードのようなものです。2人が同じ番号の組み合わせを知っていれば、どちらもロックとアンロックができます。しかし、もっと多くの人がアクセスする必要があれば、全員に同じ組み合わせを教えなければなりません。誰か1人が漏らせば、全員のセキュリティが崩壊します。

---

## なぜ重要なのか？

HS256はシンプルで高速です。自分でトークンに署名し自分で検証する単一のアプリケーションには問題ありません。しかし、1つのサービスが署名し多くのサービスが検証する分散システムでは、HS256は危険です。

**volta-auth-proxyはHS256を使用しません。RS256のみを使用します。** この記事ではHS256が拒否される理由と、それが可能にする攻撃を説明します。

## どう動くのか？

### HS256での署名

```
  共有秘密: "super-secret-key-12345"

  署名者とすべての検証者がこの同じ秘密を持たなければならない。

  署名：
    signature = HMAC_SHA256("super-secret-key-12345",
                            ヘッダー + "." + ペイロード)

  検証：
    expected = HMAC_SHA256("super-secret-key-12345",
                           ヘッダー + "." + ペイロード)
    valid = (expected == 受信した署名)
```

### 問題：共有秘密はスケールしない

```
  分散システムでHS256を使う場合：

  ┌──────────────────────────────┐
  │     volta-auth-proxy         │
  │                              │
  │  持っている: 共有秘密         │
  │  すること: JWTに署名          │
  └──────────────────────────────┘
                 │
      JWTを検証するすべてのサービスに
      同じ秘密を配布しなければならない
                 │
    ┌────────────┼────────────┐
    │            │            │
    ▼            ▼            ▼
  ┌──────┐   ┌──────┐   ┌──────┐
  │App A │   │App B │   │App C │
  │      │   │      │   │      │
  │持って │   │持って │   │持って │
  │いる:  │   │いる:  │   │いる:  │
  │同じ   │   │同じ   │   │同じ   │
  │秘密   │   │秘密   │   │秘密   │
  └──────┘   └──────┘   └──────┘

  App Bが侵害された場合：
  - 攻撃者が得るもの：共有秘密
  - 攻撃者ができること：任意のユーザーのJWTを偽造
  - 攻撃者ができること：App AとApp Cに対して誰にでもなりすまし
  - App Bだけでなく、システム全体が侵害される
```

RS256との比較（[rs256.md](rs256.md)参照）：

```
  RS256の場合：
  - App B侵害 → 攻撃者が得るのは公開鍵（偽造には役立たない）
  - リスクにさらされるのはApp Bのデータだけ
  - 他のアプリは影響を受けない
```

### アルゴリズム混乱攻撃

これはHS256とRS256に関する最も有名な攻撃です。仕組みは以下の通り：

```
  通常の動作：
  1. voltaがRS256で秘密鍵を使ってJWTに署名
  2. アプリがRS256で公開鍵を使ってJWTを検証

  攻撃：
  1. 攻撃者が有効なJWTを入手
  2. ペイロードを変更（ユーザーをadminに変更など）
  3. ヘッダーを{"alg":"RS256"}から{"alg":"HS256"}に変更
  4. 公開鍵をHS256の秘密として使って変更済みJWTに署名

  なぜ動くのか（脆弱なライブラリで）：
  ┌────────────────────────────────────────────────────┐
  │  JWTライブラリがalg=HS256を見る                      │
  │  検証に「鍵」が必要                                  │
  │  開発者の設定: verify_key = 公開鍵                    │
  │  （RS256を期待していたので、                          │
  │   検証鍵は公開鍵であるべき）                          │
  │                                                     │
  │  ライブラリが公開鍵をHMACの秘密として使用              │
  │  HMAC(公開鍵, ヘッダー + ペイロード)が一致！           │
  │  偽造JWTが受け入れられる！                            │
  └────────────────────────────────────────────────────┘

  根本原因：
  - RS256では: 検証鍵 = 公開鍵（意図通り）
  - HS256では: 検証鍵 = 共有秘密（署名鍵と同じ）
  - ライブラリがalgヘッダーを盲目的に信頼し、両方の
    アルゴリズムに同じ「鍵」変数を使うと、攻撃が成功する。
```

**voltaの防御：** RS256のみ受け入れます。検証の前にアルゴリズムがチェックされます：

```java
if (!JWSAlgorithm.RS256.equals(jwt.getHeader().getAlgorithm())) {
    throw new IllegalArgumentException("Unsupported JWT algorithm");
}
```

## volta-auth-proxyはHS256をどう扱うか？

**完全に拒否します。** voltaは：

1. RS256でのみJWTに署名する
2. RS256でのみJWTを検証する
3. `alg`がRS256以外に設定されたJWTをすべて拒否する
4. nimbus-jose-jwtライブラリを使用しており、RSAとHMACの検証パスが分離されている（明示的なチェックがなくてもアルゴリズム混乱に脆弱でない）

### なぜHS256をまったく使わないのか？

| 観点 | HS256 | RS256 |
|------|-------|-------|
| 鍵の配布 | すべての検証者に秘密を安全に共有する必要がある | 公開鍵のみ必要（公開で共有可能） |
| 侵害の影響 | 1つの鍵漏洩 = システム全体が侵害 | 公開鍵の漏洩 = 影響なし（既に公開されている） |
| 鍵のローテーション | すべてのサービスで同時に秘密を更新する必要 | 新しい公開鍵をJWKSに公開、サービスは自動更新 |
| パフォーマンス | 高速（対称暗号） | やや遅い（約1ms多い） |
| 偽造リスク | 秘密を持つどのサービスもトークンを偽造可能 | 署名サービスだけがトークンを偽造可能 |

パフォーマンスの差はごくわずか（マイクロ秒単位）で、特にvoltaの5分JWT有効期限モデルではトークン発行頻度が低いため問題になりません。

## よくある間違いと攻撃

### 間違い1：マイクロサービスアーキテクチャでHS256を使う

サービスAがトークンに署名し、サービスB、C、Dが検証する場合、4つのサービスすべてが同じ秘密を共有します。どれか1つの侵害ですべてが壊れます。

### 間違い2：HS256の秘密をハードコーディングする

ソースコードに保存された秘密はバージョン管理、CIログ、開発者マシンに行き着きます。最低限でも環境変数を使い、本番では秘密管理（Vault、AWS Secrets Manager）を使ってください。

### 間違い3：弱いHS256秘密を使う

HS256の秘密は少なくとも256ビット（32バイト）の暗号学的にランダムなデータであるべきです。短いまたは予測可能な秘密はブルートフォース攻撃を可能にします。

### 間違い4：RS256とHS256の両方を許可する

一部のシステムはどちらのアルゴリズムも受け入れるよう設定されています。これは上述のアルゴリズム混乱攻撃を可能にします。1つを選んで強制してください。
