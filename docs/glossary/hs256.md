# HS256 (HMAC with SHA-256)

[日本語版はこちら](hs256.ja.md)

---

## What is it?

HS256 is a method for signing data using a single shared secret key. Both the signer and the verifier use the exact same key. It combines HMAC (Hash-based Message Authentication Code) with SHA-256 hashing. Unlike RS256 which uses a key pair, HS256 is symmetric -- the same secret both creates and verifies signatures.

Think of it like a shared password for a safe. If two people both know the combination, either one can lock or unlock it. But if more people need access, you have to give the same combination to everyone. If any one person leaks it, everyone's security is broken.

---

## Why does it matter?

HS256 is simple and fast. For a single application that both signs and verifies its own tokens, it works fine. But in a distributed system where one service signs and many verify, HS256 is dangerous.

**volta-auth-proxy does not use HS256. It only uses RS256.** This article explains why HS256 is rejected and what attacks it enables.

---

## How does it work?

### Signing with HS256

```
  The Shared Secret: "super-secret-key-12345"

  Both the signer AND every verifier must have this same secret.

  Signing:
    signature = HMAC_SHA256("super-secret-key-12345",
                            header + "." + payload)

  Verifying:
    expected = HMAC_SHA256("super-secret-key-12345",
                           header + "." + payload)
    valid = (expected == received_signature)
```

### The problem: shared secrets do not scale

```
  With HS256 in a distributed system:

  ┌──────────────────────────────┐
  │     volta-auth-proxy         │
  │                              │
  │  Has: Shared Secret          │
  │  Does: Signs JWTs            │
  └──────────────────────────────┘
                 │
      Must distribute the SAME secret
      to every service that verifies JWTs
                 │
    ┌────────────┼────────────┐
    │            │            │
    ▼            ▼            ▼
  ┌──────┐   ┌──────┐   ┌──────┐
  │App A │   │App B │   │App C │
  │      │   │      │   │      │
  │Has:  │   │Has:  │   │Has:  │
  │SAME  │   │SAME  │   │SAME  │
  │Secret│   │Secret│   │Secret│
  └──────┘   └──────┘   └──────┘

  If App B is compromised:
  - Attacker gets: the shared secret
  - Attacker CAN: forge JWTs for ANY user
  - Attacker CAN: impersonate anyone to App A and App C
  - The ENTIRE SYSTEM is compromised, not just App B
```

Compare this with RS256 (see [rs256.md](rs256.md)):

```
  With RS256:
  - App B compromised → attacker gets public key (useless for forging)
  - Only App B's data is at risk
  - Other apps are unaffected
```

### The algorithm confusion attack

This is the most famous attack involving HS256 and RS256. Here is how it works:

```
  Normal operation:
  1. volta signs JWT with RS256 using PRIVATE KEY
  2. Apps verify JWT with RS256 using PUBLIC KEY

  The attack:
  1. Attacker takes a valid JWT
  2. Modifies the payload (changes user to admin, etc.)
  3. Changes the header from {"alg":"RS256"} to {"alg":"HS256"}
  4. Signs the modified JWT using the PUBLIC KEY as the HS256 secret

  Why it works (on vulnerable libraries):
  ┌────────────────────────────────────────────────────┐
  │  The JWT library sees alg=HS256                     │
  │  It needs a "key" to verify                        │
  │  The developer configured: verify_key = PUBLIC_KEY  │
  │  (because they expected RS256, where the            │
  │   verify key IS the public key)                    │
  │                                                     │
  │  The library uses PUBLIC_KEY as the HMAC secret     │
  │  HMAC(PUBLIC_KEY, header + payload) matches!        │
  │  The forged JWT is accepted!                        │
  └────────────────────────────────────────────────────┘

  The root cause:
  - In RS256: verify key = public key (intended)
  - In HS256: verify key = shared secret (same as signing key)
  - If the library blindly trusts the alg header and uses the
    same "key" variable for both algorithms, the attack works.
```

**volta's defense:** Only RS256 is accepted. The algorithm is checked before any verification occurs:

```java
if (!JWSAlgorithm.RS256.equals(jwt.getHeader().getAlgorithm())) {
    throw new IllegalArgumentException("Unsupported JWT algorithm");
}
```

## How does volta-auth-proxy handle HS256?

**It rejects it completely.** volta:

1. Only signs JWTs with RS256
2. Only verifies JWTs with RS256
3. Rejects any JWT with `alg` set to anything other than RS256
4. Uses the nimbus-jose-jwt library which separates RSA and HMAC verification paths (not vulnerable to algorithm confusion even without the explicit check)

### Why not use HS256 at all?

| Aspect | HS256 | RS256 |
|--------|-------|-------|
| Key distribution | Must securely share secret with every verifier | Only public key needed (can be shared openly) |
| Compromise impact | One leaked key = entire system compromised | One leaked public key = no impact (it is already public) |
| Key rotation | Must update secret on every service simultaneously | Publish new public key to JWKS; services auto-update |
| Performance | Faster (symmetric crypto) | Slightly slower (~1ms more) |
| Forgery risk | Any service with the secret can forge tokens | Only the signing service can forge tokens |

The performance difference is negligible (microseconds), especially for volta's 5-minute JWT expiry model where tokens are issued infrequently.

## Common mistakes and attacks

### Mistake 1: Using HS256 in a microservices architecture

If Service A signs tokens and Services B, C, and D verify them, all four services share the same secret. Any one compromise breaks everything.

### Mistake 2: Hardcoding the HS256 secret

Secrets stored in source code end up in version control, CI logs, and developer machines. Use environment variables at minimum, secrets management (Vault, AWS Secrets Manager) for production.

### Mistake 3: Using a weak HS256 secret

HS256 secrets should be at least 256 bits (32 bytes) of cryptographically random data. Using a short or predictable secret makes brute-force attacks feasible.

### Mistake 4: Allowing both RS256 and HS256

Some systems are configured to accept either algorithm. This enables the algorithm confusion attack described above. Pick one and enforce it.
