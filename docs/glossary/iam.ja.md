# IAM（ID・アクセス管理）

[English version](iam.md)

---

## これは何？

IAMはIdentity and Access Management（ID・アクセス管理）の略で、誰かが何者であるか（ID）と何を許可されているか（アクセス管理）に関するすべてを包括する用語です。

建物の全セキュリティシステムのようなものです。IAMは正面ドアの鍵だけではなく、IDバッジ、カードキー、来訪者台帳、「関係者以外立入禁止」の看板、カメラ、そして誰がどこに行けるかのルール、すべてを含みます。一つのパーツではなく、システム全体です。

---

## なぜ重要なのか？

ユーザーがいるすべてのアプリケーションは何らかの形でIAMが必要です。ログインページがあるシンプルなアプリでも、ID管理（あなたは誰？）とアクセス管理（何が見られる？）をしています。

混乱しやすいのは「IAM」が以下を表すために使われることです：

1. **概念** -- IDとアクセスを管理するという一般的なアイデア
2. **ソフトウェアカテゴリ** -- IDとアクセス管理を実装するのに役立つツール群
3. **特定のクラウドサービス** -- AWS IAM、Google Cloud IAMなど

初心者として最も大切なのは、IAMが全体像だと理解することです。他のすべて -- OIDC、OAuth、JWT、セッション、RBAC -- はIAMという大きな傘の下にある具体的なツールと技術です。

### IAMの全体像

```
  IAM（大きな傘）
  ├── 認証（「あなたは誰？」）
  │   ├── パスワード
  │   ├── OIDC / OAuth（ソーシャルログイン、SSO）
  │   ├── SAML（エンタープライズSSO）
  │   ├── MFA（多要素認証）
  │   ├── WebAuthn / パスキー
  │   └── 証明書
  │
  ├── 認可（「何ができる？」）
  │   ├── RBAC（ロールベースアクセス制御）
  │   ├── ABAC（属性ベースアクセス制御）
  │   ├── ACL（アクセス制御リスト）
  │   └── ポリシーエンジン（OPA、Cedar）
  │
  ├── ユーザー管理
  │   ├── 登録/サインアップ
  │   ├── プロフィール管理
  │   ├── パスワードリセット
  │   └── アカウント無効化
  │
  ├── セッション管理
  │   ├── Cookie/トークン
  │   ├── セッション有効期限
  │   └── 同時セッション制限
  │
  └── 監査/コンプライアンス
      ├── ログインログ
      ├── アクセスログ
      └── コンプライアンスレポート
```

## voltaはIAMの世界でどこに位置するか？

volta-auth-proxyはIAMソリューションですが、すべてをカバーしようとはしません。以下がその位置づけです：

| IAM領域 | voltaのカバー範囲 |
|---------|-----------------|
| **認証** | Google OIDC、セッションCookie、JWT発行。IdP設定によるSAML（Phase 4）。 |
| **認可** | テナント・アプリごとの[RBAC](rbac.ja.md)（OWNER / ADMIN / MEMBERロール）。 |
| **ユーザー管理** | 初回ログイン時のユーザー作成、API経由のプロフィール、管理UI。 |
| **セッション管理** | 8時間スライディングウィンドウ、最大5同時セッション、セッション一覧/無効化。 |
| **マルチテナンシー** | コア設計機能：テナント解決、テナント切り替え、メンバー管理。 |
| **監査** | 設定可能なシンク（Postgres、Kafka、Elasticsearch）付き監査ログ。 |

voltaがカバーしないもの：

- パスワード管理（voltaはGoogle OIDCを使うので、Googleがパスワードを管理）
- LDAP/Active Directoryフェデレーション
- サードパーティアプリ向けの汎用OIDCプロバイダ
- きめ細かいポリシーエンジン（ABAC、OPA）

これは意図的です。voltaは一つのユースケース -- マルチテナントSaaSの認証と認可 -- のために構築され、それをしっかりやります。フルスペクトルのIAMプラットフォームが必要なら、[Keycloak](keycloak.ja.md)やOktaのようなエンタープライズソリューションを検討してください。

## IAM製品の比較

| 製品 | タイプ | 強み | 弱み |
|------|--------|------|------|
| **Auth0 / Okta** | クラウド[IDaaS](idaas.ja.md) | 素早い立ち上げ、すべて管理される | スケール時に高額、ロックイン |
| **Keycloak** | オープンソースIAMサーバー | 機能が完全、エンタープライズ対応 | 重い、設定が複雑 |
| **AWS IAM** | クラウドプロバイダIAM | AWSとの深い統合 | AWS専用、エンドユーザー認証向きではない |
| **Ory Stack** | オープンソースIAM | モジュラー、APIファースト | 複数サービスの管理が必要 |
| **ZITADEL** | オープンソースIAM | ネイティブマルチテナンシー | 新しく、コミュニティが小さい |
| **volta-auth-proxy** | 専用認証ゲートウェイ | 軽量、シンプル、マルチテナント | 機能範囲が狭い |

## シンプルな例

Webアプリにログインするとき、IAMはすべてのステップで裏側で動いています：

```
  1. app.example.com にアクセス
     → IAMの質問：「この人は認証済み？」
     → 回答：セッションCookieなし → ログインにリダイレクト

  2. 「Googleでサインイン」をクリック
     → IAMの動作：OIDC認証フローを開始

  3. Googleがあなたの身元を確認
     → IAMの動作：セッション作成、JWT発行

  4. 管理パネルにアクセスしようとする
     → IAMの質問：「この人は認可されている？」
     → 回答：ユーザーはMEMBERロール、管理にはADMINが必要 → 403 Forbidden

  5. 管理者があなたをADMINに昇格
     → IAMの動作：データベースのロールを更新

  6. もう一度管理パネルにアクセス
     → IAMの質問：「この人は認可されている？」
     → 回答：ユーザーはADMINロール → 200 OK
```

これらのステップすべてがID（あなたは誰？）とアクセス（何ができる？）に関わっています。これがIAMです。
