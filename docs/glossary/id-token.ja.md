# id_token (OIDC アイデンティティトークン)

## これは何？

id_token は OIDC（OpenID Connect）標準で定義された特別な種類の JWT（JSON Web Token）です。答える質問はひとつだけ: 「この人は誰？」 認証プロバイダ（Google など）がユーザーのログイン後に id_token を作成・署名し、アプリケーションがそれをユーザーの本人確認の証拠として受け取ります。

id_token は政府発行の身分証明書のようなものです。名前、写真、政府の公式な印鑑が付いています。政府を信頼する人なら誰でも、その身分証明書を見てあなたが誰かを確認できます。ただし、身分証明書は特定の行動の許可を与えるものではなく、身元を証明するだけです。

id_token をデコードすると、こんな感じです:

```
ヘッダ:
{
  "alg": "RS256",
  "kid": "google-key-abc123",
  "typ": "JWT"
}

ペイロード:
{
  "iss": "https://accounts.google.com",
  "sub": "110248495921238986420",
  "aud": "your-google-client-id.apps.googleusercontent.com",
  "exp": 1711900000,
  "iat": 1711896400,
  "nonce": "random-nonce-value",
  "email": "alice@example.com",
  "email_verified": true,
  "name": "Alice Smith",
  "picture": "https://lh3.googleusercontent.com/..."
}

署名:
(Google の秘密鍵による RSA-SHA256 デジタル署名)
```

重要なフィールド（「claims」と呼ばれます）:

| Claim | 意味 |
|-------|------|
| `iss` | Issuer（発行者）-- このトークンを作ったのは誰か（Google） |
| `sub` | Subject（主体）-- Google でのユーザーの一意な ID |
| `aud` | Audience（対象者）-- このトークンが発行された対象のアプリ |
| `exp` | Expiration（有効期限）-- このトークンが無効になる時刻 |
| `iat` | Issued At（発行時刻）-- このトークンが作成された時刻 |
| `nonce` | アプリが送ったランダムな値。リプレイ攻撃を防止するため |
| `email` | ユーザーのメールアドレス |
| `email_verified` | Google がメールアドレスの実在を確認したかどうか |
| `name` | ユーザーの表示名 |

## なぜ重要？

OIDC と id_token が登場する前は、OAuth プロバイダからユーザーの身元情報を取得する標準的な方法がありませんでした。素の OAuth 2.0 では、access_token を取得してから、別途「userinfo」エンドポイントに API コールして、ユーザーが誰かを調べる必要がありました。この追加のステップは遅く、信頼性に欠け（userinfo エンドポイントがダウンしていたら？）、プロバイダごとに仕様が異なりました。

id_token は、署名済みトークンに身元情報を直接含めることでこの問題を解決します。追加のネットワーク通信なしに、アプリケーションがローカルで検証できます。

### id_token と access_token の違い

これは非常に重要な区別です:

| | id_token | access_token |
|---|---|---|
| **目的** | ユーザーが誰かを証明する（認証） | リソースへのアクセスを許可する（認可） |
| **対象者** | あなたのアプリケーション | API やリソースサーバー |
| **形式** | 常に JWT | JWT またはランダムな文字列 |
| **API に送るべき？** | いいえ | はい |
| **含む情報** | ユーザーの身元に関する claims | スコープと権限 |
| **定義元** | OIDC | OAuth 2.0 |

よく使われるたとえ: id_token はあなたの身分証明書（あなたが誰かを証明する）、access_token はホテルの部屋の鍵（あなたが誰であるかに関係なく、特定の部屋へのアクセスを許可する）です。

## どう動くの？

id_token は OIDC の認可コードフローの一部です:

```
  ブラウザ        あなたのアプリ        Google
  |              |                   |
  | ログインクリック|                   |
  |------------->|                   |
  |              |                   |
  | Google の     |                   |
  | ログインへ     |                   |
  | リダイレクト   |                   |
  |<-------------|                   |
  |                                  |
  | ユーザーがログイン                  |
  |--------------------------------->|
  |                                  |
  | code 付きでリダイレクトで戻す        |
  |<---------------------------------|
  |                                  |
  | code           |                 |
  |--------------->|                 |
  |                |                 |
  |                | code を          |
  |                | トークンに交換    |
  |                |---------------->|
  |                |                 |
  |                | 受け取る:        |
  |                | - id_token      |
  |                | - access_token  |
  |                |<----------------|
  |                |                 |
  |                | id_token を検証: |
  |                | 1. 署名確認      |
  |                |    (JWKS 経由)  |
  |                | 2. iss 確認     |
  |                | 3. aud 確認     |
  |                | 4. exp 確認     |
  |                | 5. nonce 確認   |
  |                |                 |
  | セッション作成  |                 |
  |<---------------|                 |
```

### id_token の検証

適切な検証には以下のチェックが含まれます（順序通り）:

1. **JWT をパース**して、ヘッダ、ペイロード、署名に分離する。
2. 発行者の公開鍵（JWKS エンドポイントから取得）を使って**署名を検証**する。
3. **アルゴリズムを確認** -- 想定されるアルゴリズム（RS256 など）のみ受け入れる。`none` は絶対に受け入れない。
4. **発行者（`iss`）を確認** -- 想定される認証プロバイダと一致する必要がある。
5. **対象者（`aud`）を確認** -- アプリケーションのクライアント ID を含む必要がある。
6. **有効期限（`exp`）を確認** -- トークンが期限切れでないこと。
7. **nonce を確認** -- ログインフロー中にアプリが送った nonce と一致する必要がある。
8. **追加のチェック** -- 要件に応じて `email_verified` など。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は OIDC のコールバックフローで Google の id_token を受け取り、徹底的に検証します。

**トークン交換。** Google が認可コード付きでユーザーをリダイレクトで戻した後、volta はそのコードを Google のトークンエンドポイント（`https://oauth2.googleapis.com/token`）でトークンに交換します。レスポンスには id_token が含まれます。volta は id_token だけを取り出します。Google の access_token は一切使いません。

**検証。** volta は Nimbus JOSE+JWT ライブラリを使って Google の id_token を検証します。検証内容:

1. **署名検証** -- Google の JWKS エンドポイント（`https://www.googleapis.com/oauth2/v3/certs`）を使用。ライブラリが Google の公開鍵を取得し、RSA-SHA256 署名を検証する。
2. **発行者チェック** -- `https://accounts.google.com` と `accounts.google.com` の両方を受け入れる（Google は両方の形式を使う）。
3. **対象者チェック** -- `aud` claim が volta の設定された Google クライアント ID を含む必要がある。
4. **有効期限チェック** -- トークンが期限切れでないこと。
5. **nonce チェック** -- `nonce` claim が、ログインフロー開始時に volta が保存した値と一致する必要がある。
6. **メール検証チェック** -- `email_verified` が `true` であること。volta は未検証のメールアドレスを受け入れない。

**身元情報の取り出し。** 検証後、volta は id_token の claims からユーザーの身元情報を取り出します:
- `sub` -- Google のユニークなユーザー識別子
- `email` -- ユーザーのメールアドレス
- `name` -- ユーザーの表示名

この情報は、volta のデータベースでユーザーを検索・作成し、セッションを確立するために使われます。

**volta の JWT は id_token ではない。** volta はその後、下流のアプリケーション向けに独自の JWT を発行します。volta が発行するこの JWT は技術的にはアクセストークン（下流のサービスへのアクセスを認可する）であり、id_token ではありません。`volta_tid`（テナント ID）、`volta_roles`、`volta_display` のような、認可に関連する volta 固有の claims を含んでいます。

全体のフロー:

```
  Google の id_token                volta の JWT
  (アイデンティティトークン)          (下流向けアクセストークン)
  +---------------------+         +------------------------+
  | iss: accounts.google |  --->   | iss: volta-auth         |
  | sub: google-user-id  |  これを  | sub: volta-user-uuid    |
  | email: alice@...     |  使って  | volta_tid: tenant-uuid  |
  | email_verified: true |  作成   | volta_roles: [ADMIN]    |
  | nonce: xyz           |         | volta_display: Alice    |
  +---------------------+         +------------------------+
  Alice が誰かを証明する            volta 内で Alice を認可する
```

## よくある間違い

**1. id_token をアクセストークンとして使う。**
id_token を Bearer トークンとして API に送ってはいけません。id_token はアプリケーションが消費するためのものであり、API 向けではありません。API コールにはアクセストークンを使いましょう。

**2. id_token を全く検証しない。**
HTTPS 経由のトークンエンドポイントから来たからといって、id_token を無条件に信頼する開発者がいます。しかし、攻撃者がリダイレクト URI の処理を侵害できた場合、偽造トークンを注入できてしまいます。署名、発行者、対象者、有効期限、nonce を必ず検証しましょう。

**3. 対象者チェックをスキップする。**
`aud` が自分のクライアント ID と一致するか確認しないと、全く別のアプリケーション向けに発行された id_token を受け入れてしまう可能性があります。攻撃者が自分のアプリ用の有効な Google id_token を取得して、あなたのアプリに提示できてしまいます。

**4. `email_verified` を確認しない。**
Google は未検証のメールアドレスの id_token を発行できます。`email_verified` を確認せずにメールを信頼すると、攻撃者が他人のメールアドレスで Google アカウントを作り、その人としてログインできてしまいます。

**5. id_token を長期保存する。**
id_token はログイン時に一度検証してから破棄するものです。保存して後で再検証するものではありません。検証後にセッションを作成し、以降はそのセッションを使いましょう。

**6. Google の id_token と volta の JWT を混同する。**
Google の id_token は身元を証明します。volta の JWT は volta エコシステム内での認可を証明します。目的が異なり、含まれる claims も異なります。
