# IDaaS（Identity as a Service）

[English version](idaas.md)

---

## これは何？

IDaaSはIdentity as a Service（サービスとしてのID管理）の略で、自分で運用する代わりに、他者に料金を払って運用してもらうクラウドホスト型の認証・ユーザー管理システムです。

管理付きの賃貸マンションと持ち家の違いで考えてみましょう。管理付きマンション（IDaaS）なら、大家さんが配管、電気、外構、修理を担当します。家賃を払って住むだけです。持ち家（セルフホスティング）なら、すべてをコントロールできます -- 好きなようにリノベーションでき、誰も家賃を上げられません -- でも深夜にトイレが壊れたら？それはあなたの問題です。

IDaaSは認証における「管理付きマンション」モデルです。

---

## なぜ重要なのか？

認証を正しく構築するのは難しいです。いたるところにセキュリティの脆弱性があります：XSS、CSRF、セッションハイジャック、トークン窃取、ブルートフォース攻撃など。間違えるとユーザーのデータが漏洩します。

[Auth0](auth0.ja.md)、Clerk、OktaなどのIDaaSプロバイダはこう言います：「認証を自分で作らないでください。フルタイムでこれをやっているセキュリティ専門家のチームがいます。うちのサービスを使ってください。」

これは正当な価値提案です。問題は、トレードオフ（コスト、ロックイン、限定的なコントロール）があなたの状況に見合うかどうかです。

## 主要なIDaaSプロバイダ

| プロバイダ | 特徴 | 料金モデル |
|-----------|------|-----------|
| **Auth0**（Okta） | 最も人気。優秀なSDKとクイックスタートガイド。 | [MAU](mau.ja.md)課金 |
| **Clerk** | 開発者フレンドリー、美しいプリビルトUIコンポーネント。 | MAU課金 |
| **Okta** | エンタープライズ向け。SAML、SCIM、LDAP。 | ユーザー数（プロビジョニング済み）課金 |
| **Firebase Auth** | Google Cloudの一部。シンプルで、モバイルに最適。 | 50K MAUまで無料、以降認証ごと |
| **AWS Cognito** | AWSの一部。AWSとの緊密な統合。 | MAU課金 |
| **Supabase Auth** | Supabaseの一部。PostgreSQLベース。 | Supabaseプランに含まれる |

### すべてに共通すること

- **クラウドホスト型：** サーバーを運用するのはあなたではなく、彼らです。
- **API/SDK駆動：** 彼らのライブラリまたはREST APIで統合。
- **使用量ベース課金：** ユーザー数に応じて支払い。
- **管理されたセキュリティ：** セキュリティパッチ、インフラ、稼働率は彼らが担当。

## IDaaS対セルフホスト

```
  IDaaS（Auth0、Clerk、Okta）：
  ┌───────────────────────────────────────┐
  │  + 始めるのが速い（数分〜数時間）       │
  │  + 運用負荷ゼロ                        │
  │  + セキュリティチームがメンテナンス     │
  │  - コストがユーザー数に比例（MAU）      │
  │  - ベンダーロックイン                   │
  │  - カスタマイズが限定的                 │
  │  - データは彼らのサーバーに             │
  └───────────────────────────────────────┘

  セルフホスト（volta、Keycloak）：
  ┌───────────────────────────────────────┐
  │  + ユーザー数に関係なく$0               │
  │  + すべてを完全にコントロール           │
  │  + データは自分のサーバーに             │
  │  + ベンダーロックインなし               │
  │  - 最初のセットアップ作業が多い         │
  │  - 運用とセキュリティは自分で           │
  │  - 認証を理解している人が必要           │
  └───────────────────────────────────────┘
```

### IDaaSを好むチームの理由

- **小さなチーム：** 2人のスタートアップは認証インフラを管理したくない。機能をリリースしたい。
- **市場投入の速さ：** IDaaSならログインを数時間で追加できる。MVPにはこれが大事。
- **コンプライアンスのチェックボックス：** 一部のエンタープライズ顧客はSOC 2認証のあるAuth0/Oktaの名前を聞きたがる。
- **広いIdPサポート：** IDaaSプロバイダは20以上のソーシャルログインプロバイダとエンタープライズSAML接続を標準でサポートすることが多い。

### セルフホストを好むチームの理由

- **スケール時のコスト：** 10,000+ MAUを超えると、セルフホスティングの節約効果が大きくなる。
- **コントロール：** ログイン体験、セッション動作、ユーザーデータモデルのあらゆる面をカスタマイズできる。
- **プライバシー/コンプライアンス：** 一部の規制はユーザーデータを特定の法域に置くことを要求する。
- **独立性：** プロバイダの値上げ、条件変更、サービス終了のリスクがない。

## voltaの位置づけ

volta-auth-proxyはIDaaSではありません。その逆です：自分のインフラで実行するセルフホスト型の認証ゲートウェイです。

voltaはIDaaSオプションを検討し、マルチテナントSaaSに対してトレードオフが受け入れられないと判断した人々によって構築されました：

| IDaaSのトレードオフ | voltaの回答 |
|-------------------|------------|
| MAU課金 | どんなスケールでも$0 |
| ベンダーロックイン | 標準HTTPヘッダー、あなたのPostgreSQL |
| 限定的なUIコントロール | 完全な[jte](jte.ja.md)テンプレート |
| 彼らのサーバーにあるデータ | あなたのPostgresにあるデータ |
| 彼らの稼働率への依存 | あなたの稼働率、あなたの責任 |
