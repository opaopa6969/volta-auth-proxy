# 冪等性（べきとうせい）

[English version](idempotency.md)

---

## これは何？

ある操作が**冪等**とは、1回実行しても複数回実行しても同じ結果になることです。エレベーターのボタンを押すのと同じです。5回押してもエレベーターが5台来るわけではなく、1回押した結果と同じです。

HTTP の観点では：
- `GET /users/42` -- 冪等。同じユーザーを10回読んでも何も変わらない。
- `DELETE /users/42` -- 冪等。既に削除されたユーザーを再度削除しても追加の影響なし。
- `POST /invitations` -- **冪等でない。** 同じ招待リクエストを2回送ると、2つの招待が作成される可能性がある。

---

## なぜ重要？

ネットワークは不安定です。リクエストが消失し、タイムアウトが起き、クライアントがリトライします。冪等でない操作は、リトライで重複データを生む可能性があります：

```
クライアント: POST /invitations {email: "alice@example.com"}
サーバー: 招待を作成、レスポンス送信開始...
ネットワーク: 接続切断！
クライアント: 「成功した？もう一度やろう...」
クライアント: POST /invitations {email: "alice@example.com"}
サーバー: もう一つ招待を作成
結果: Alice に招待メールが2通届く。混乱。
```

冪等な操作はリトライしても安全です。冪等でない操作には特別な対処が必要です。

---

## 簡単な例

### HTTP メソッドと冪等性

| メソッド | 冪等？ | 理由 |
|---------|--------|------|
| GET | はい | 読み取りは状態を変えない |
| PUT | はい | 「これを X に設定」-- 2回やっても結果は X |
| DELETE | はい | 既に削除されたものを削除 = 変化なし |
| POST | **いいえ** | 「新しいものを作成」-- 2回やると2つできる |
| PATCH | 場合による | 「フィールド X を Y に変更」は冪等。「1を加算」は冪等でない |

### POST を安全にリトライする方法

一般的な戦略：
1. **冪等性キー：** クライアントがリクエストに一意の ID を付与。サーバーがその ID が処理済みか確認。
2. **一意制約：** データベースが重複を拒否（例：email + tenant でユニーク）。
3. **存在確認して作成：** 作成前に既に存在するか確認。

## volta-auth-proxy では

volta は招待の冪等性を**データベースレベルの制約**で処理しています。招待システムは `ON CONFLICT` 句とユニーク制約を使用します：

招待経由でメンバーシップを作成する場合：
```sql
INSERT INTO memberships(user_id, tenant_id, role)
VALUES (?, ?, ?)
ON CONFLICT(user_id, tenant_id) DO UPDATE SET is_active = true, role = EXCLUDED.role
```

これは、ユーザーが既にこのテナントのメンバーである場合、重複を作成せず、ロールを更新して再アクティブ化することを意味します。「招待を受け入れる」操作が冪等になります。ユーザーは招待リンクを複数回クリックしても、重複メンバーシップは作成されません。

招待作成自体については、volta は `max_uses` と `used_count` フィールドを使って招待の利用回数を制御し、単一の招待コードの無制限使用を防いでいます。

GET を使う API エンドポイント（メンバー一覧、テナント一覧、監査ログ）は本質的に冪等です。POST エンドポイント（招待作成、テナント停止）はデータベース制約と一意キーを使って有害な重複を防いでいます。
