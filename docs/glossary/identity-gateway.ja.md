# アイデンティティゲートウェイ

[English version](identity-gateway.md)

---

## 一言で言うと？

アイデンティティゲートウェイとは、アプリケーションの前に立って「あなたは誰で、何をしてよいか」というロジックをすべて処理するサービスのことです。アプリケーション自身がそれをやる必要がなくなります。

---

## セキュリティガードのたとえ

多くの会社が入っている大きなオフィスビルを想像してください。各会社がそれぞれ自前の警備員を雇う代わりに、ビルの入り口に1つの受付デスクがあります。入館するとき：

1. 警備員があなたのIDを確認する（認証 -- 「あなたは誰？」）
2. 警備員があなたが訪問できるフロアを確認する（認可 -- 「何にアクセスできる？」）
3. 警備員があなたの名前とフロア番号が書かれた入館バッジを渡す
4. 目的地に向かうと、中にいる全員があなたのバッジを見ることができる

これがまさにアイデンティティゲートウェイのやっていることです。ソフトウェアのための受付デスクです。

- **オフィスビル** = アプリケーションのシステム全体
- **警備員** = アイデンティティゲートウェイ（volta-auth-proxy）
- **あなたのID** = ログイン情報（メール、パスワード、Googleアカウント）
- **入館バッジ** = アプリにあなたが誰かを伝えるヘッダーやトークン
- **中にいる会社** = 下流のアプリケーション（wiki、管理パネルなど）

---

## なぜアプリが自分で認証しないの？

こう思うかもしれません：「各アプリにログインロジックを入れればいいのでは？」良い質問です。その理由がこちら：

**5つのアプリがそれぞれ自前で認証する場合：**

```
  App 1 (Wiki)      → 独自のログインページ、セッション管理、Google連携を構築
  App 2 (Admin)     → 独自のログインページ、セッション管理、Google連携を構築
  App 3 (Dashboard) → 独自のログインページ、セッション管理、Google連携を構築
  App 4 (API)       → 独自のログインページ、セッション管理、Google連携を構築
  App 5 (Chat)      → 独自のログインページ、セッション管理、Google連携を構築

  問題：
  - メンテナンスするコードが5倍
  - 見つけるべきセキュリティバグが5倍
  - ユーザーが5回別々にログイン
  - 「誰が何にアクセスできるか」が5つのデータベースに散らばる
  - ログインフローの変更で5つのアプリを更新
```

**1つのアイデンティティゲートウェイがある場合：**

```
  volta-auth-proxy → すべてのログイン、セッション、ロール、テナントを処理
       │
       ├── App 1 (Wiki)      → ヘッダーを読むだけ、wiki機能に集中
       ├── App 2 (Admin)     → ヘッダーを読むだけ、管理機能に集中
       ├── App 3 (Dashboard) → ヘッダーを読むだけ、ダッシュボード機能に集中
       ├── App 4 (API)       → ヘッダーを読むだけ、API機能に集中
       └── App 5 (Chat)      → ヘッダーを読むだけ、チャット機能に集中

  メリット：
  - 認証ロジックが1箇所
  - 監査するセキュリティ面が1つ
  - ユーザーは1回ログイン（シングルサインオン）
  - 「誰が何にアクセスできるか」の一元管理
  - ログインフローの変更が1箇所で済み、全アプリに反映
```

## アイデンティティゲートウェイとしてのvolta-auth-proxy

volta-auth-proxyはSaaSアプリケーションのためのアイデンティティゲートウェイです。以下のことを行います：

1. **ログイン** -- ログインフロー全体を処理（Googleログイン、SAML、メール/パスワード）
2. **セッション** -- ユーザーセッションを管理してログイン状態を維持
3. **テナント解決** -- ユーザーがどの会社/ワークスペースに属しているかを特定
4. **ロール確認** -- ユーザーがOWNER、ADMIN、MEMBER、VIEWERのいずれかを判定
5. **トークン作成** -- ユーザーの本人情報を運ぶJWT（安全なトークン）を作成
6. **ヘッダー注入** -- リクエストをアプリに渡す前に、本人情報のヘッダーを追加

あなたのアプリはパスワードを見ることがありません。ログインページを扱うこともありません。すでに本人情報が付いたリクエストを受け取るだけです。入館バッジを読むようなものです。

---

## 簡単な例

voltaで保護されたwikiアプリにユーザーがアクセスするとこうなります：

```
  1. ユーザーがブラウザでwiki.example.comと入力

  2. リクエストはまずvolta-auth-proxyに到着
     volta：「有効なセッションCookieはある？」
     ユーザー：「いいえ、初めての訪問です」
     volta：「OK、ログインページにリダイレクトします...」

  3. ユーザーがGoogleでログイン
     volta：「Googleによると、あなたはtaro@acme.comですね。確認します...」
     volta：「見つけました！あなたはACME CorpのMEMBERです」

  4. voltaがセッションを作成し、wiki.example.comにリダイレクト
     今度はリクエストにセッションCookieが含まれます。

  5. voltaがセッションCookieを確認し、ヘッダーを追加：
     X-Volta-User-Id: taro-uuid
     X-Volta-Tenant-Id: acme-uuid
     X-Volta-Roles: MEMBER
     X-Volta-Email: taro@acme.com

  6. wikiアプリがこれらのヘッダー付きのリクエストを受信
     Wiki：「ああ、ACMEの太郎さん、MEMBERだ。
            ACMEのwikiページを表示しよう」
```

wikiアプリはGoogleログイン、セッション、テナントロジックを一切扱う必要がありませんでした。voltaが提供するヘッダーを読むだけです。
