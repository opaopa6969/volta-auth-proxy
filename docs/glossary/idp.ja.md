# IdP (Identity Provider / アイデンティティプロバイダ)

## これは何？

Identity Provider（IdP）とは、ユーザーの ID 情報を保存し検証するサービスです。ウェブサイトで「Google でサインイン」をクリックしたとき、Google が IdP として機能しています。ウェブサイトはあなたのパスワードを知りません。あなたが本人であることの確認を Google に信頼して委ねています。

IdP はパスポートオフィスのようなものです。パスポートオフィスがあなたの身元を確認し（出生証明書、写真、生体認証）、パスポートを発行します。海外旅行のとき、渡航先の国はあなたの身元をゼロから調査しません。発行機関を信頼しているからパスポートを信頼するのです。

```
  ユーザー       アプリケーション      IdP (Google)
  |             |                   |
  | "私は Alice" |                   |
  |------------>|                   |
  |             | "Alice を知らない。  |
  |             |  Google に聞こう。" |
  |             |------------------>|
  |             |                   |
  |   Google ログインページ           |
  |<--------------------------------|
  |   認証情報を入力                  |
  |-------------------------------->|
  |             |                   |
  |             | "はい、この方は     |
  |             |  alice@gmail.com  |
  |             |  です。証拠です。"  |
  |             |<------------------|
  |             |                   |
  | "ようこそ、  |                   |
  |  Alice さん!" |                   |
  |<------------|                   |
```

代表的な Identity Provider:

| IdP | 使用場面 | プロトコル |
|-----|---------|-----------|
| Google | コンシューマー・ビジネスアプリ | OIDC |
| Microsoft Azure AD (Entra ID) | エンタープライズ、Microsoft 365 | OIDC, SAML |
| Okta | エンタープライズ | OIDC, SAML |
| Auth0 | 開発者向けアプリ | OIDC |
| Keycloak | セルフホスト、オープンソース | OIDC, SAML |
| GitHub | 開発者ツール | OAuth 2.0 |

## なぜ重要？

**アプリがパスワードを管理する必要がなくなる。** パスワードの安全な保存は難しく、データ漏洩は起こります。認証を IdP に委任することで、パスワードの保存を完全に回避できます。Google のセキュリティが数十億人のユーザーに十分なら、あなたのアプリにも十分なはずです。

**ユーザーの体験が良くなる。** また別のアカウントとパスワードを作る代わりに、ボタンを 1 つクリックするだけでログインできます。新しいことを覚える必要がありません。

**エンタープライズの要件。** 大規模組織は社員のアクセスを一元管理したいと考えています。Azure AD や Okta のような IdP を使って、誰がどのアプリケーションにアクセスできるか制御します。あなたのアプリが彼らの IdP をサポートしていなければ、プロダクトを使ってもらえません。

**セキュリティが向上する。** IdP はセキュリティの専門家です。MFA の実装、不審なログインの検出、パスワードポリシーの適用、アカウント復旧の処理を行います。あなたのアプリはこれらを無料で得られます。

## どう動くの？

アプリケーション（「リライングパーティ」または「サービスプロバイダ」）と IdP の間のやり取りは標準プロトコル（OIDC または SAML）に従います:

```
  1. ユーザーがアプリにアクセスしようとする
  2. アプリがユーザーを IdP にリダイレクト
  3. ユーザーが IdP で認証（パスワード、MFA など）
  4. IdP がユーザーの身元を証明する署名済みトークン/アサーションを作成
  5. IdP が証拠とともにユーザーをアプリにリダイレクトで戻す
  6. アプリが証拠を検証し、ローカルセッションを作成
```

### IdP の信頼モデル

アプリは暗号学的な検証に基づいて IdP を信頼します:

- **OIDC:** IdP が秘密鍵で id_token (JWT) に署名する。アプリは IdP の公開鍵（JWKS エンドポイントから取得）で署名を検証する。
- **SAML:** IdP が X.509 証明書で SAML アサーションに署名する。アプリは IdP の公開証明書で署名を検証する。

アプリはユーザーのパスワードを一切見ません。IdP からの署名済みの証拠だけを見ます。

### IdP ブローカーパターン

アプリケーションが複数の IdP をサポートする必要がある場合もあります。それぞれと直接統合する代わりに、「IdP ブローカー」-- 複数の IdP と統合し、アプリに統一されたインターフェースを提供する中間サービスを使えます。

```
  ブローカーなし:                  ブローカーあり:

  App --> Google (OIDC)           App --> ブローカー --> Google
  App --> Azure AD (SAML)                          --> Azure AD
  App --> Okta (OIDC)                              --> Okta
  App --> GitHub (OAuth)                           --> GitHub

  4 つの統合を維持                 1 つの統合を維持
  4 つの異なるプロトコル            ブローカーがプロトコルの違いを吸収
```

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は主要な IdP として Google と通信し、マルチ IdP サポートのロードマップを持っています。

**フェーズ 1（現在）: Google を IdP として使用。**

volta は OIDC を使って Google と統合します:

1. volta がユーザーを Google の認可エンドポイント（`https://accounts.google.com/o/oauth2/v2/auth`）に必要なパラメータ（client_id, redirect_uri, scope, state, nonce, PKCE challenge）とともにリダイレクト。
2. ユーザーが Google で認証。
3. Google が認可コード付きで volta の `/callback` エンドポイントにリダイレクトで戻す。
4. volta がコードを Google のトークンエンドポイントで id_token に交換。
5. volta が Google の JWKS（`https://www.googleapis.com/oauth2/v3/certs`）で id_token を検証。
6. volta が検証済み id_token からユーザーの身元情報（メール、名前、Google subject ID）を取り出す。
7. volta が自身のデータベースでユーザーを作成・更新し、セッションを確立。

```
  volta-auth-proxy                    Google (IdP)
  +------------------+               +------------------+
  |                  |  1. 認可 URL   |                  |
  |  OIDC Service    |--------------->|  認可             |
  |                  |                |  エンドポイント    |
  |                  |  2. コールバック|                  |
  |  コールバック     |<---------------|  トークン         |
  |  ハンドラ        |                |  エンドポイント    |
  |                  |  3. id_token   |                  |
  |  トークン        |<---------------|  JWKS            |
  |  検証            |  4. JWKS      |  エンドポイント    |
  |                  |--------------->|                  |
  +------------------+               +------------------+
  |
  | 5. 身元情報を取り出す
  | 6. ユーザーを作成/更新
  | 7. セッションを作成
  v
  PostgreSQL
```

volta の Google IdP 設定:
- `GOOGLE_CLIENT_ID` -- Google Cloud Console からの OAuth クライアント ID
- `GOOGLE_CLIENT_SECRET` -- OAuth クライアントシークレット
- `GOOGLE_REDIRECT_URI` -- 認証後に Google がユーザーを送る先

**SAML サポート（基盤構築済み）。**

volta には基本的な SAML ログインエンドポイント（`/auth/saml/login`）とコールバック（`/auth/saml/callback`）があります。現在の実装は RelayState パラメータ付きで SAML エントリーポイントにリダイレクトします。これがフェーズ 2-3 のマルチ IdP サポートの基盤です。

**volta における IdP ブローカーパターン。**

volta 自体が下流のアプリケーションにとっての IdP ブローカーとして機能します。`app-wiki` や `app-admin` の視点では、volta が IdP そのものです。ユーザーが Google、Azure AD、SAML のどれで認証したかは知りません。検証済みの ID 情報が `X-Volta-*` ヘッダで届くだけです。

```
  下流アプリから見ると:             volta が処理する:

  +--------+                       +--------+     +--------+
  |  Wiki  |--- volta を信頼 ---    | volta  |---->| Google |
  +--------+                       |        |     +--------+
                                   |        |
  +--------+                       | (IdP   |     +----------+
  | Admin  |--- volta を信頼 ---    | ブロー  |---->| Azure AD |
  +--------+                       | カー)   |     +----------+
                                   |        |
  +--------+                       |        |     +--------+
  |  Chat  |--- volta を信頼 ---    |        |---->|  Okta  |
  +--------+                       +--------+     +--------+

  アプリは 1 つの IdP              volta が複数の IdP
  (volta) と統合                   と統合
```

**フェーズ 2-3 マルチ IdP 計画。** volta はテナントごとの IdP 設定をデータベースに保存します（`idp_config` レコード経由）。各テナントが独自の SAML または OIDC プロバイダを設定できます。そのテナントのユーザーがログインすると、volta が正しい IdP にルーティングします。テナントの IT 管理者が volta の管理画面から IdP の詳細（発行者 URL、証明書、クライアント ID）を設定します。

## よくある間違い

**1. 単一の IdP をハードコーディングする。**
アプリを Google（または Okta のみ）でしか動かないように作ると、市場が限定されます。エンタープライズ顧客は自社の IdP を要求します。最初は 1 つしか実装しなくても、マルチ IdP サポートを前提に設計しましょう。

**2. IdP のレスポンスを検証しない。**
トークンが IdP のリダイレクトから来たからといって、有効とは限りません。署名の検証、発行者、対象者、有効期限、nonce の確認を必ず行いましょう。volta はこれらのチェックを全て実行します。

**3. email_verified を確認せずに email claim を信頼する。**
一部の IdP は未検証のメールアドレスでの登録を許可しています。`email_verified` を確認せずにメールを信頼すると、攻撃者が他人のメールで IdP にアカウントを作り、アクセスを得る可能性があります。volta は `email_verified: true` を必須としています。

**4. IdP の障害を想定しない。**
Google の認証サーバーがダウンすると、ユーザーはログインできません。アプリが優雅に縮退する必要があるか（メンテナンスページを表示）、複数の IdP をフォールバックとしてサポートするか検討しましょう。

**5. IdP とディレクトリサービスを混同する。**
IdP はユーザーを認証します（誰であるかを証明する）。ディレクトリサービス（Active Directory や LDAP）はユーザー情報と組織構造を保存します。しばしば組み合わされますが（Azure AD は両方を行う）、概念的には異なります。

**6. IdP からローカルユーザーへのマッピングを考えない。**
ユーザーが IdP 経由で認証するとき、IdP の ID をローカルのユーザーレコードにマッピングする必要があります。IdP でメールアドレスを変更したら？ ある日は Google で、別の日は Azure AD でサインインしたら？ volta はメール変更の問題を避けるために、Google の subject ID（`sub` claim）でユーザーをマッピングしています。
