# インメモリ

[English version](in-memory.md)

---

## これは何？

インメモリとは、ハードドライブやSSDではなく、コンピュータのRAM（ランダムアクセスメモリ）にデータを格納することです。RAMは高速だが揮発性のメモリで、電源が切れると内容が失われます。ディスクストレージは遅いが永続的。RAMは速いが一時的です。

デスクとファイリングキャビネットで考えてみましょう。ファイリングキャビネット（ディスク）はすべてを永続的に保管します -- オフィスを出ても、すべてが引き出しにあります。デスクトップ（RAM）は今作業中の書類を置く場所です。デスクから紙を取るのは一瞬。ファイリングキャビネットまで歩いて引き出しを探すのはずっと長い。でも夜帰るとき（サーバー再起動）、デスクの上のすべてが消えます。

インメモリストレージは、速度が耐久性より重要なとき、またはデータが本質的に一時的なとき（毎分リセットするレート制限カウンターなど）に使われます。

---

## なぜ重要なのか？

- **速度。** RAMアクセスはディスクアクセスの100,000倍速い。すべてのリクエストで発生する操作（セッションチェック、レート制限）では、実際に大きな違いになる。
- **データベース負荷の軽減。** 頻繁にアクセスするデータをメモリにキャッシュすることで、PostgreSQLへのクエリが減り、データベースが書き込みと複雑なクエリに集中できる。
- **一時データに永続性は不要。** レート制限カウンター、短命なキャッシュ、計算値は再起動を生き延びる必要がない。インメモリに最適。
- **トレードオフの認識。** インメモリを理解することは、再起動で何を失い、なぜそれが許容されるか（されないか）を理解すること。

## どう動くのか？

### メモリ階層

```
  ┌─────────────────────────────────────────────────────┐
  │              メモリ階層                               │
  │                                                      │
  │  速度                           サイズ               │
  │  ◀────────────────────────────────▶                  │
  │  最速                         最大                   │
  │                                                      │
  │  ┌──────────┐                                       │
  │  │CPUキャッシュ│ ~1ns    ~8MB   ← 極小、最速          │
  │  └──────────┘                                       │
  │  ┌──────────┐                                       │
  │  │   RAM    │  ~100ns  ~16-64GB ← インメモリはここ    │
  │  └──────────┘                                       │
  │  ┌──────────┐                                       │
  │  │   SSD    │  ~100μs  ~1-4TB  ← PostgreSQLデータ   │
  │  └──────────┘                                       │
  │  ┌──────────┐                                       │
  │  │   HDD    │  ~10ms   ~4-16TB ← バックアップ、ログ  │
  │  └──────────┘                                       │
  │                                                      │
  │  1ns = 0.000001ms                                    │
  │  RAMはSSDの約1,000倍速い                              │
  │  RAMはHDDの約100,000倍速い                            │
  └─────────────────────────────────────────────────────┘
```

### 「揮発性」の意味

```
  サーバー稼働中:                  サーバー再起動:
  ┌────────────────────┐        ┌────────────────────┐
  │ RAM（インメモリ）    │        │ RAM（インメモリ）    │
  │                    │        │                    │
  │ session_cache: {   │        │ （空）              │
  │   "abc": 有効      │───X───▶│                    │
  │   "def": 有効      │  消失! │                    │
  │ }                  │        │                    │
  │                    │        │                    │
  │ rate_limits: {     │        │                    │
  │   "1.2.3.4": 47   │───X───▶│                    │
  │ }                  │  消失! │                    │
  └────────────────────┘        └────────────────────┘

  ┌────────────────────┐        ┌────────────────────┐
  │ ディスク（PostgreSQL）│       │ ディスク（PostgreSQL）│
  │                    │        │                    │
  │ sessionsテーブル:   │        │ sessionsテーブル:   │
  │   abc: 有効        │────────▶│   abc: 有効        │
  │   def: 有効        │  安全! │   def: 有効        │
  └────────────────────┘        └────────────────────┘
```

### インメモリ vs ディスク：いつどちらを使うか

| 基準 | インメモリ（RAM） | ディスク（データベース） |
|------|-----------------|---------------------|
| 速度 | マイクロ秒 | ミリ秒 |
| 再起動を生き延びる | いいえ | はい |
| インスタンス間で共有 | いいえ（ローカル） | はい |
| 容量 | 制限あり（GB） | 大きい（TB） |
| GBあたりコスト | 高い | 安い |
| **最適な用途** | キャッシュ、カウンター、ホットデータ | 永続レコード、真のデータソース |

## volta-auth-proxy ではどう使われている？

### Caffeine：voltaのインメモリキャッシュ

voltaはインメモリキャッシングに[Caffeine](caffeine-cache.ja.md)を使用します。CaffeineはJVMのヒープメモリにデータを格納します：

```
  ┌─────────────────────────────────────┐
  │  JVMヒープ（volta-auth-proxy）       │
  │                                     │
  │  ┌───────────────────────────────┐  │
  │  │  Caffeineキャッシュ            │  │
  │  │                               │  │
  │  │  セッションキャッシュ:          │  │
  │  │  ┌────────────────────────┐   │  │
  │  │  │ "sess_abc" → {userId,  │   │  │
  │  │  │              role,     │   │  │
  │  │  │              tenantId} │   │  │
  │  │  └────────────────────────┘   │  │
  │  │                               │  │
  │  │  レート制限キャッシュ:          │  │
  │  │  ┌────────────────────────┐   │  │
  │  │  │ "192.168.1.1" → 47    │   │  │
  │  │  │ "10.0.0.5" → 12       │   │  │
  │  │  └────────────────────────┘   │  │
  │  │                               │  │
  │  │  TTL: 30秒（セッション）       │  │
  │  │  TTL: 60秒（レート制限）       │  │
  │  └───────────────────────────────┘  │
  └─────────────────────────────────────┘
```

### ユースケース1：セッションキャッシング

すべての[ForwardAuth](forwardauth.ja.md)リクエストがユーザーのセッションを検証する必要があります。キャッシュなしでは、すべてのリクエストがPostgreSQLにアクセスします：

```
  インメモリキャッシュなし:
  100リクエスト/秒 × 3ms/クエリ = 毎秒300msのDB時間

  インメモリキャッシュあり（30秒TTL）:
  約3キャッシュミス/秒 × 3ms = 9msのDB時間
  約97キャッシュヒット/秒 × 0.001ms = 0.097ms

  削減: データベースクエリを97%削減
```

### ユースケース2：レート制限

レート制限にはリクエストごとにインクリメントするカウンターが必要です。これらはインメモリに最適です：
- すべてのリクエストで変更される（毎回DBに書くのは遅い）
- 一時的（リセットされる1分間のウィンドウ）
- 再起動で失われても許容される（ユーザーは新しい枠をもらう）

```java
// Caffeineを使ったインメモリレート制限
Cache<String, AtomicInteger> rateLimitCache = Caffeine.newBuilder()
    .expireAfterWrite(Duration.ofMinutes(1))
    .build();

int count = rateLimitCache
    .get(clientIp, k -> new AtomicInteger(0))
    .incrementAndGet();

if (count > MAX_REQUESTS_PER_MINUTE) {
    throw new RateLimitExceededException();
}
```

### Phase 2：L1キャッシュとしてのインメモリ

voltaが水平スケールすると、インメモリのCaffeineはローカル「L1」キャッシュになり、[Redis](redis.ja.md)が共有「L2」キャッシュになります：

```
  リクエストがvolta-2に到達:

  1. Caffeineを確認（L1、ローカル、約0.001ms）
     │
     ├── ヒット？ → 即座に返却
     │
     └── ミス？ → Redisを確認（L2、共有、約0.5ms）
                  │
                  ├── ヒット？ → Caffeineに格納、返却
                  │
                  └── ミス？ → PostgreSQLにクエリ（約3ms）
                               Redis + Caffeineに格納
                               返却
```

## よくある間違いと攻撃

### 間違い1：インメモリを永続ストレージとして扱う

重要なデータの唯一のコピーがメモリ内にある場合、サーバー再起動で破壊されます。常にインメモリは耐久性のあるバッキングストア（PostgreSQL）を持つキャッシュとして使いましょう。

### 間違い2：キャッシュサイズ無制限

サイズ制限なしでは、キャッシュはJVMがメモリを使い果たして`OutOfMemoryError`でクラッシュするまで成長します。Caffeineでは必ず`maximumSize`を設定しましょう。

### 間違い3：古いキャッシュによるセキュリティ問題

セッションがデータベースで無効化されても、インメモリキャッシュがまだ「有効」と言っていれば、キャッシュエントリが期限切れになるまでユーザーは認証されたままです。セキュリティに敏感なデータではTTLを短く保ちましょう。

### 間違い4：マルチインスタンスデプロイでキャッシュを無視

インメモリデータは1つのインスタンスにローカルです。volta-1がセッションをキャッシュしても、volta-2はそれを見ません。これがPhase 2で[Redis](redis.ja.md)を共有キャッシュ層として追加する理由です。

### 間違い5：キャッシュしすぎ

すべてを長いTTLでキャッシュすると、アプリケーションは速く感じますが、古いデータを返す可能性があります。データ型に基づいて速度と鮮度をバランスしましょう。
