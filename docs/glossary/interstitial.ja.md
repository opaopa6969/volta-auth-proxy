# Interstitial（インタースティシャル）

[English version](interstitial.md)

---

## これは何？

インタースティシャルとは、フロー内の2つのアクション間に表示される中間ページです。出発点でも最終目的地でもなく、途中の短い立ち寄りです。インタースティシャルは、重要な情報をユーザーに見せたり、確認を求めたり、実際の目的地に進む前の技術的な遷移を処理するために使われます。

空港のトランジットラウンジのようなものです。出発ゲートを離れましたが、目的地にはまだ着いていません。何か（税関検査、搭乗券のスキャン）が行われている中間の空間にいます。トランジットラウンジは行きたい場所ではなく、行き先にたどり着くための必要なステップです。

volta-auth-proxyでは、認証フローの複数のポイントにインタースティシャルが現れます：OIDCコールバックページ、招待同意画面、テナント選択ページ、「テナントなし」ページ。それぞれが認証とユーザーの最終目的地の間の短い一時停止です。

---

## なぜ重要なのか？

インタースティシャルがなければ：

- **コンテキストの喪失**：OIDCコールバックは`code`パラメータを返す。何かがユーザーをリダイレクトする前にそれをトークンと交換する必要がある。インタースティシャルがなければ、このロジックの居場所がない
- **同意なし**：ユーザーが何に参加するかを見ることなく、招待によって自動的にテナントに追加される
- **選択なし**：複数テナントのユーザーが聞かれることなくランダムなテナントに放り込まれる
- **唐突なUX**：ユーザーにはログインページかアプリのどちらかしか見えず、間の遷移や説明がない

## どう動くのか？

### インタースティシャルの種類

```
  ┌────────────────────────────────────────────────────────┐
  │ 種類       │ 目的                  │ ユーザーに見えるもの │
  │────────────│───────────────────────│───────────────────│
  │ 技術的     │ OIDCコードをトークンに│ 短いローディングページ│
  │            │ 交換                  │ （または何も見えない）│
  │────────────│───────────────────────│───────────────────│
  │ 同意       │ ユーザーにアクションの│ 「ワークスペース      │
  │            │ 確認を求める          │ Acme Corpに参加？」  │
  │────────────│───────────────────────│───────────────────│
  │ 選択       │ ユーザーが選択肢から  │ ワークスペースの      │
  │            │ 選ぶ                  │ リスト               │
  │────────────│───────────────────────│───────────────────│
  │ 情報提供   │ 重要なことをユーザーに│ 「ワークスペースなし。│
  │            │ 伝える                │ 招待をもらってください」│
  └────────────────────────────────────────────────────────┘
```

### インタースティシャルフローパターン

```
  アクションA            インタースティシャル       アクションB
  ┌──────────┐           ┌──────────────┐           ┌──────────┐
  │ OIDC     │  リダイレクト│ コールバック │  リダイレクト│ アプリ   │
  │ プロバイダ│ ─────────►│ 処理         │ ─────────►│ ダッシュ │
  │ (Google) │           │(インタースティシャル)│     │ ボード   │
  └──────────┘           └──────────────┘           └──────────┘
                          │
                          ├── コードをトークンに交換
                          ├── ユーザーをDBにUpsert
                          ├── セッションを作成
                          └── 次の目的地を決定
```

## volta-auth-proxyではどう使われているか？

### OIDCコールバックインタースティシャル

Googleがユーザーを認証した後、`/callback?code=...&state=...`にリダイレクトします。これは技術的インタースティシャルで、voltaが以下を行う間、ユーザーは短時間だけ（または全く）見ます：

1. stateとnonceパラメータの検証
2. 認可コードをトークンに交換
3. ユーザーレコードの作成または更新
4. セッションの作成
5. 次の目的地の決定（アプリ、テナント選択、招待同意、またはテナントなし）

```yaml
# auth-machine.yaml — AUTH_PENDING状態
callback_success:
  trigger: "GET /callback"
  guard: "oidc_flow.state_valid && oidc_flow.nonce_valid && oidc_flow.email_verified"
  actions:
    - { type: side_effect, action: upsert_user }
    - { type: side_effect, action: delete_oidc_flow }
    - { type: side_effect, action: create_session }
    - { type: audit, event: LOGIN_SUCCESS }
  next_if:
    - guard: "invite.present && invite.valid"
      next: INVITE_CONSENT          # → 招待インタースティシャル
    - guard: "user.tenant_count == 1"
      next: AUTHENTICATED           # → アプリに直行（インタースティシャルなし）
    - guard: "user.tenant_count > 1"
      next: TENANT_SELECT           # → 選択インタースティシャル
    - guard: "user.tenant_count == 0"
      next: NO_TENANT               # → 情報提供インタースティシャル
```

### 招待同意インタースティシャル

保留中の招待があるユーザーは参加前に同意ページを見ます：

```
  ┌─────────────────────────────────────────┐
  │                                         │
  │   以下のワークスペースへの招待です       │
  │                                         │
  │   ┌─────────────────────────────────┐   │
  │   │  Acme Corporation               │   │
  │   │  ロール: MEMBER                 │   │
  │   │  招待者: admin@acme.com        │   │
  │   └─────────────────────────────────┘   │
  │                                         │
  │   [ 承認 ]     [ 辞退 ]                │
  │                                         │
  └─────────────────────────────────────────┘
```

### テナント選択インタースティシャル

複数テナントのユーザーは選択ページを見ます：

```
  ┌─────────────────────────────────────────┐
  │                                         │
  │   ワークスペースを選択                   │
  │                                         │
  │   ┌─────────────────────────────────┐   │
  │   │  ● Acme Corporation            │   │
  │   │  ○ サイドプロジェクト LLC       │   │
  │   │  ○ オープンソース Org           │   │
  │   └─────────────────────────────────┘   │
  │                                         │
  │   [ 続ける ]                            │
  │                                         │
  └─────────────────────────────────────────┘
```

### テナントなしインタースティシャル

テナントメンバーシップのないユーザーは情報提供ページを見ます：

```
  ┌─────────────────────────────────────────┐
  │                                         │
  │   ワークスペースがまだありません         │
  │                                         │
  │   どのワークスペースのメンバーでも       │
  │   ありません。管理者に招待を             │
  │   送ってもらってください。               │
  │                                         │
  │   連絡先: support@example.com           │
  │                                         │
  └─────────────────────────────────────────┘
```

## よくある間違いと攻撃

### 間違い1：スキップ可能なインタースティシャル

ユーザーがアプリに直接ナビゲートして招待同意ページをバイパスできれば、同意インタースティシャルは無用です。voltaは状態マシン遷移を強制します。招待が保留中の場合、AUTHENTICATEDに到達するにはINVITE_CONSENTを通過しなければなりません。

### 間違い2：永久にブロックするインタースティシャル

OIDCコールバックインタースティシャルには600秒のタイムアウト（AUTH_PENDINGで定義）があります。これがなければ、コールバックを完了しなかったユーザーのOIDCフローが永久にハングします。

### 間違い3：インタースティシャルでの情報漏洩

招待同意ページはテナント名と招待者を表示します。他のメンバー、内部詳細、消費後のURLの招待コードは表示すべきではありません。voltaはユーザーが判断に必要な情報のみを表示します。

### 攻撃：同意インタースティシャルへのCSRF

攻撃者が招待承認フォームを自動送信するページを作成する。voltaはPOSTアクションでの[CSRF](csrf.ja.md)トークンチェック（`guard_check: csrf_token_valid`）でこれを防ぎます。
