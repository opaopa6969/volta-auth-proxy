# 無効化（Invalidation）

[English version](invalidation.md)

---

## これは何？

無効化（Invalidation）とは、何かをもはや有効でない状態にする行為です。認証では、セッション、トークン、または資格情報が技術的にはまだ期限切れでなくても受け入れられるべきではないと宣言することを意味します。「これは終わり」の能動的な形です。

小切手の無効化を思い浮かべてください。小切手自体は紙として存在し、有効な日付があり、見た目は正当です。しかし「VOID」と書かれており、銀行は処理を拒否します。無効化とは、セッション、トークン、鍵に「VOID」と書くことです。

無効化は[有効期限](session-expiry.md)（受動的 -- 時間切れ）とも[失効](revoke.md)（鍵やトークンに特化した無効化の一形態）とも異なります。無効化はより広い概念で、何かが受け入れられなくするあらゆるメカニズムです。

---

## なぜ重要なのか？

無効化がなければ、有効でなくなる唯一の方法は期限切れを待つことです。これは問題を引き起こします：

- **ログアウトが嘘になる**: 「ログアウト」をクリックするとCookieは消えるが、セッションレコードは残る。セッションIDをコピーした攻撃者はまだ使える
- **侵害からの回復が遅い**: 漏洩した鍵やトークンは自然な期限切れまで有効なまま
- **アクセス制御の変更が遅延する**: adminから降格されたユーザーがJWTの期限切れまでadmin権限のJWTを持ち続ける

無効化は認証のライフサイクルに対するアクティブな制御を与えます。「いずれ機能しなくなる」と「今すぐ機能しなくなる」の違いです。

## どう動くのか？

### 無効化の種類

```
  ┌────────────────────────────────────────────────────┐
  │  種類              方法                  速度      │
  │  ──────────────────────────────────────────────── │
  │  セッション削除    DELETE FROM sessions  即座       │
  │  セッションログアウト status=revoked設定  即座       │
  │  鍵ローテーション  鍵を引退に設定        即座*      │
  │  JWT期限切れ       expクレームを待つ      最大TTL   │
  │  Cookieクリア      ブラウザから削除      クライアント側│
  │  トークンブロックリスト jtiをリストに追加 即座       │
  └────────────────────────────────────────────────────┘
  * 鍵ローテーションは新トークンには即座だが、既存の
    トークンはexpまで有効。
```

### セッション無効化フロー

```
  アクティブなセッション:
  ┌─────────────────┐     ┌──────────────────────┐
  │  ブラウザ       │     │  データベース         │
  │  Cookie: abc123 │ ──► │  sessions:            │
  │                 │     │  id=abc123 ✓ ACTIVE   │
  └─────────────────┘     └──────────────────────┘
  リクエスト受理。

  無効化後:
  ┌─────────────────┐     ┌──────────────────────┐
  │  ブラウザ       │     │  データベース         │
  │  Cookie: abc123 │ ──► │  sessions:            │
  │  (まだ存在)     │     │  (abc123 削除済み)    │
  └─────────────────┘     └──────────────────────┘
  リクエスト拒否 → 401 → ログインページ。
```

### カスケード無効化

```
  ユーザー無効化:
  ユーザーアカウントを無効化
       │
       ├── このユーザーのすべてのセッションを削除
       │   └── すべてのデバイスで即座にログアウト
       │
       └── 既存のJWTは最大5分間まだ有効
           └── その後すべてのアクセスが完全終了

  テナント無効化:
  テナント全体を無効化
       │
       ├── テナント内のすべてのユーザーのすべてのセッションを削除
       │   └── テナント内の全ユーザーがログアウト
       │
       └── 既存のJWTは5分以内に期限切れ
           └── テナント完全分離
```

## volta-auth-proxy ではどう使われている？

### セッション無効化（ログアウト）

ユーザーがログアウトすると、voltaはデータベースからセッションを削除します：

```
  POST /api/v1/auth/logout
       │
       ▼
  1. __volta_session Cookie読み取り → セッションID
  2. DELETE FROM sessions WHERE id = session_id
  3. Set-Cookie: __volta_session=; Max-Age=0（Cookie消去）
  4. 200 OK を返す
```

この無効化はサーバー側では即座です。ブラウザにまだCookieが残っていても、セッションがデータベースに存在しないため次のリクエストは失敗します。

### JWT無効化（間接的）

JWTは直接無効化できません（自己完結型のため）。voltaは以下の組み合わせで対処します：

1. **セッション無効化**: セッション削除後は新しいJWTを発行不可
2. **短いTTL**: 既存のJWTは最大5分で期限切れ
3. **[鍵ローテーション](key-rotation.md)**: 緊急時、署名鍵をローテーションしてすべてのトークンを無効化

```
  無効化の伝搬タイムライン:
  ┌──────────────────────────────────────────────┐
  │  T+0:00  セッション無効化（ログアウト）        │
  │  T+0:00  新JWT要求 → 401（セッションなし）    │
  │  T+0:01  古いJWTはまだ有効（残り4:59）        │
  │  T+5:00  古いJWT期限切れ。完全に無効化。      │
  │                                              │
  │  ギャップ: 0-5分の残余JWT有効性               │
  │  これはvoltaのトレードオフ: ステートレスJWT   │
  │  検証 vs 即座の無効化。                       │
  └──────────────────────────────────────────────┘
```

### Cookie無効化

ブラウザのCookie消去は`Max-Age=0`で行います：

```
Set-Cookie: __volta_session=; Max-Age=0; Path=/; HttpOnly; Secure
```

ただしこれはクライアント側のみ。サーバーもセッションレコードを削除する必要があります。完全な無効化には両方が必要です。

## よくある間違いと攻撃

### 間違い1: Cookieだけをクリア（クライアント側無効化のみ）

サーバー側のセッションを削除せずにCookieだけをクリアすると、セッションは有効なままです。ログアウト前にセッションIDをキャプチャした攻撃者がまだ使えます。

### 間違い2: ハード削除ではなくソフト削除

セッションを`status=inactive`にマークしてもクエリで受け入れ続けるのは無効化ではありません。セッション参照クエリが非アクティブなセッションをフィルタアウトすることを確認しましょう。またはハード削除がベターです。

### 間違い3: パスワード変更時にすべてのセッションを無効化しない

ユーザーがパスワードを変更したら、既存のすべてのセッション（現在のものを除く）を無効化すべきです。そうしなければ、古いパスワードを侵害した攻撃者がアクティブなセッションを保持し続けます。

### 間違い4: ロール変更時の無効化を忘れる

ユーザーがadminから降格されても、既存のJWTには期限切れまで`volta_roles: ["admin"]`が含まれたままです。機密操作では、JWTだけでなく常にデータベースに対してロールを検証しましょう。

### 攻撃: 競合状態の悪用

攻撃者が有効なJWTで長時間の操作を開始します。操作中にセッションが無効化されます。操作完了時に認可を再チェックしなければ、無効化にもかかわらず攻撃者の操作が成功します。
