# Invariant（不変条件）

[English version](invariant.md)

---

## これは何？

不変条件とは、何が起きても常に真でなければならないルールです。個々の遷移に対するガードではなく、システム全体のグローバルな性質です。不変条件が破られたら、システムにバグがあります。不変条件は設計上の欠陥を実行時のバグになる前に捕捉します。

建物の物理法則のようなものです。「すべての部屋に少なくとも1つの出口がなければならない」は不変条件です。建物がどう設計されていようと、このルールは成り立たなければなりません。出口のない部屋があれば防火違反です。ドアのところでチェックするのではなく、建物の設計時に検証します。

volta-auth-proxyでは、不変条件は`dsl/auth-machine.yaml`に[状態マシン](state-machine.ja.md)が満たすべき形式的性質として宣言されています。デッドロックを防ぎ、到達可能性を保証し、すべての参照が有効であることを保証します。

---

## なぜ重要なのか？

ガードは個々の遷移を守ります。不変条件はシステム全体を守ります。不変条件がなければ：

- **デッドロック**：ユーザーが出力遷移のない状態に到達し、永久に詰む
- **到達不能な状態**：DSLに状態が存在するが、そこへの遷移がない（無駄なコード、混乱するドキュメント）
- **宙ぶらりんの参照**：遷移が`errors.yaml`に存在しないエラーコード`FOOBAR`を参照
- **安全網の欠如**：ログアウト遷移がその状態から偶然除外され、認証済みユーザーがログアウトできない

## どう動くのか？

### 不変条件 vs ガード vs 制約

```
  ┌────────────────────────────────────────────────────────────┐
  │ ガード：     実行時にチェック、遷移ごと                     │
  │              "session.valid && tenant.active"               │
  │              偽の場合 → 遷移がブロック                      │
  │                                                            │
  │ 制約：       実行時にチェック、操作ごと                     │
  │              "テナントには最低1人のOWNERが必要"              │
  │              違反した場合 → 操作が拒否（400エラー）          │
  │                                                            │
  │ 不変条件：   設計時にチェック、グローバル                   │
  │              "すべての状態に少なくとも1つの出口がある"       │
  │              違反した場合 → DSLにバグがある                  │
  └────────────────────────────────────────────────────────────┘
```

### 不変条件の種類

**構造的不変条件**（状態マシンの形状について）：
- すべての状態に出力遷移がある
- すべての`next`値が定義された状態を参照
- 孤立した状態がない

**到達可能性の不変条件**（マシンを通るパスについて）：
- どの状態からもAUTHENTICATEDかUNAUTHENTICATEDに到達可能
- 認証済みのすべての状態からログアウトに到達可能

**参照整合性の不変条件**（ファイル間の一貫性について）：
- アクション中のすべてのエラーコードが`errors.yaml`に存在
- すべてのテンプレート名が既存のテンプレートを参照

### 不変条件の検証方法

不変条件はDSLファイルの静的解析で検証できます。実行コード不要です：

```
  1. auth-machine.yamlをパース
  2. 状態と遷移の有向グラフを構築
  3. 各不変条件について：
     ┌─────────────────────────────────────────────────────┐
     │ no_deadlock:                                        │
     │   各非終端状態Sについて：                            │
     │     Sからの出力遷移を数える                          │
     │     ASSERT 数 > 0                                   │
     │                                                     │
     │ reachable_auth:                                     │
     │   各状態Sについて：                                  │
     │     SからBFS/DFS                                    │
     │     ASSERT AUTHENTICATEDまたはUNAUTHENTICATEDがパス内│
     │                                                     │
     │ no_undefined_refs:                                  │
     │   各遷移Tについて：                                  │
     │     ASSERT T.nextが定義された状態に含まれる          │
     │                                                     │
     │ error_codes_defined:                                │
     │   error_refを持つ各アクションAについて：             │
     │     ASSERT A.error_refがerrors.yamlに存在            │
     └─────────────────────────────────────────────────────┘
  4. いずれかのアサーションが失敗 → DSLバグ、デプロイ前に修正
```

## volta-auth-proxyではどう使われているか？

### `dsl/auth-machine.yaml`で宣言された不変条件

```yaml
invariants:
  - id: no_deadlock
    rule: "非終端状態には到達可能な出力遷移が少なくとも1つ必要"

  - id: reachable_auth
    rule: "どの状態からもAUTHENTICATEDかUNAUTHENTICATEDへのパスが存在する必要がある"

  - id: logout_always_possible
    rule: "認証済みのどの状態からもlogoutに到達可能（global_transitions経由）"

  - id: no_undefined_refs
    rule: "すべての'next'値が定義された状態を参照する必要がある"

  - id: error_codes_defined
    rule: "アクションで使用されるすべてのエラーコードがdsl/errors.yamlに存在する必要がある"
```

### `no_deadlock`が実際のバグを捕捉した例

DSLバージョン1で、INVITE_CONSENTの`cancel`遷移が`AUTHENTICATED_NO_TENANT`（存在しない状態）への遷移でした。`no_undefined_refs`不変条件がこれを捕捉したでしょう。バージョン2でcancel遷移は`NO_TENANT`に修正されました。

同様に、`callback_nonce_invalid`遷移はバージョン3でデッドロック修正のために追加されました。これがなければ、stateは有効だがnonceが無効な場合、AUTH_PENDINGでどの遷移にも一致せず、ユーザーが詰みます。

### `dsl/policy.yaml`の制約

`policy.yaml`は実行時の制約を定義します（関連するが異なる概念）：

```yaml
constraints:
  - id: last_owner
    rule: "テナントには常に最低1人のOWNERが必要"
    enforcement: "OWNERが0人になるロール変更/削除をブロック"
    error: "LAST_OWNER_CANNOT_CHANGE"

  - id: promote_limit
    rule: "自分より上のロールにはプロモートできない"

  - id: concurrent_sessions
    rule: "ユーザーは最大MAX_CONCURRENT_SESSIONSのアクティブセッションを持てる"
    default: 5
```

これらはデータモデルの不変条件で、Javaコードにより実行時に強制されます。

### 実践的な不変条件の検証

```
  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
  │ 開発者が      │     │ CIパイプライン│     │ 本番          │
  │ DSLを編集    │────►│ 不変条件を    │────►│ デプロイ      │
  │               │     │ 検証          │     │              │
  └──────────────┘     └──────┬───────┘     └──────────────┘
                              │
                        失敗？│ デプロイをブロック。
                              │ "no_deadlock違反:
                              │  MFA_REQUIREDに出力遷移
                              │  がありません"
```

## よくある間違いと攻撃

### 間違い1：強制なき不変条件

コメントに不変条件を書いてチェックしないのは、図面に非常口を描いて実際には建てないようなもの。voltaはDSLに不変条件を宣言し、自動検証できるようにしています。

### 間違い2：不変条件が少なすぎる

`no_deadlock`だけチェックすると、エラーコードが存在しないエントリを参照していることを見逃すかもしれません。各カテゴリの不変条件が異なるクラスのバグを捕捉します。

### 間違い3：不変条件とガードの混同

不変条件は「これは常に真でなければならない」。ガードは「この特定の遷移でこれが真でなければならない」。不変条件レベルのチェックをガードに入れると、間違ったレベルでチェックしていることになります。遅すぎて、散らばりすぎです。

### 間違い4：システムの進化時に不変条件を更新しない

Phase 3でMFA_REQUIREDが追加されたとき、`reachable_auth`不変条件は依然として成り立たなければなりません。MFA_REQUIREDにAUTHENTICATEDへのパスがなければ、不変条件が即座にバグを捕捉します。
