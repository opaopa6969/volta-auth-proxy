# 招待コード

[English version](invitation-code.md)

---

## これは何？

招待コードは、volta-auth-proxyの特定の[テナント](tenant.md)に参加するための、暗号学的にランダムな一意のシークレットです。テナント管理者が誰かを招待すると、voltaが32バイトのランダム値を生成し、[base64url](base64.md)でエンコードして招待者に送ります。招待者は登録時にこのコードを提示して、招待されたことを証明します。

プライベートクラブへの黄金のチケットのようなものです。クラブのオーナーがカードにユニークなコードを書いてあなたに渡します。ドアに着いたらカードを提示します。ドアマンがコードが台帳と一致するか、期限切れでないか、使用済みでないかを確認します。すべてOKなら入場できます。コードはランダムなので推測できず、一度限りなので再利用できません。

招待コードはvoltaの管理された[プロビジョニング](provisioning.md)モデルの重要な部分です。オープン登録（誰でもサインアップ可能）とは異なり、招待により明示的に招待された人のみがテナントに参加できます。

---

## なぜ重要なのか？

マルチテナントSaaSアプリケーションでは、誰でも任意のテナントに参加させるわけにはいきません：

```
  オープン登録（B2Bには危険）：
  ┌──────────┐
  │ インター  │──── サインアップ ────► 「Acme Corp」テナントに参加?!
  │ ネット上の│                      （誰が承認した？）
  │ 誰でも   │
  └──────────┘

  招待ベース（管理された）：
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Acme     │────►│ volta    │────►│ Alice    │
  │ 管理者   │     │ コード   │     │ 招待を   │
  │ Aliceを  │     │ 生成     │     │ 受け取り │
  │ 招待     │     │          │     │ 参加     │
  └──────────┘     └──────────┘     └──────────┘
  （承認済み）     （安全なコード）   （検証済み）
```

主な利点：

- **アクセス制御**：招待された人だけが参加可能
- **テナント分離**：コードは特定のテナントに紐づく
- **監査証跡**：誰が誰を招待したか、いつ、承認されたかどうか
- **有効期限**：コードは期限切れになり、数ヶ月前の招待が使われることを防ぐ
- **使い捨て**：各コードは1回しか使えない

## どう動くのか？

### コード生成

voltaは暗号学的に安全なランダムバイトを使って招待コードを生成します：

```
  ステップ1：32バイトのランダムデータを生成
    SecureRandom.getBytes(32)
    → [0x7A, 0x3F, 0xB2, 0x91, ... 合計32バイト]

  ステップ2：base64urlでエンコード（URLセーフ、パディングなし）
    Base64.getUrlEncoder().withoutPadding().encode(bytes)
    → "ej-ykYH3mQ_7vKxN2bF..."

  なぜ32バイト？
    32バイト = 256ビットのエントロピー
    可能なコード数：2^256 ≈ 1.16 × 10^77
    比較：宇宙の原子数：~10^80
    ブルートフォースは不可能。
```

### 招待フロー

```
  管理者                   volta-auth-proxy              招待者
  ======                   ================              ======

  1. POST /api/v1/tenants/{tid}/invitations
     {"email": "alice@example.com", "role": "MEMBER"}
  ──────────────────────────────────────►

                           2. コード生成：
                              code = base64url(SecureRandom(32))
                              DBに保存：
                              ┌────────────────────────────┐
                              │ invitation_id: inv-uuid     │
                              │ tenant_id: acme-uuid        │
                              │ email: alice@example.com    │
                              │ code_hash: SHA256(code)     │
                              │ role: MEMBER                │
                              │ invited_by: admin-uuid      │
                              │ expires_at: +7日            │
                              │ used: false                 │
                              └────────────────────────────┘

                           3. リンク付き招待メールを送信：
                              https://app.com/invite?code=ej-ykYH3mQ_7vKxN...
                           ──────────────────────────────────────►

                                                          4. リンクをクリック
                                                          5. 登録を完了

                           6. POST /auth/accept-invitation
                              {"code": "ej-ykYH3mQ_7vKxN..."}
  ◄────────────────────────────────────────────────────────────

                           7. 検証：
                              - 提示されたコードをハッシュ
                              - code_hashで招待を検索
                              - 期限切れでないか確認
                              - 使用済みでないか確認
                              - メールが一致するか確認
                              → テナントにユーザー作成
                              → 招待を使用済みにマーク
                              → invitation.accepted webhook発火
```

### なぜデータベースでコードをハッシュするか？

voltaはデータベースに生のコードではなく`SHA256(code)`を保存します。パスワードハッシュと同じ原則です：

```
  攻撃者がデータベースアクセスを取得した場合：

  生コードを保存：
    攻撃者がコードを読む → そのユーザーとして参加可能 → 危険

  ハッシュされたコードを保存：
    攻撃者がハッシュを読む → SHA256を逆算不可 → 安全
    （本物のコードはメール内にしかなかった）
```

### コードの特性

| 特性 | 値 | 理由 |
|------|-----|------|
| 長さ | 32バイト（256ビット） | ブルートフォース不可能 |
| エンコーディング | [base64url](base64.md) | URL内で安全、パディング問題なし |
| 保存形式 | SHA256ハッシュのみ | DB侵害でもコード漏洩なし |
| 有効期限 | 7日（設定可能） | 古い招待の使用を防止 |
| 使用回数 | 一度限り | コードの共有/再利用を防止 |
| 紐づけ | メール+テナント | 意図された受信者のみ使用可能 |

## volta-auth-proxyではどう使われている？

### 招待API

```
  招待作成：
    POST /api/v1/tenants/{tid}/invitations
    Authorization: Bearer <admin-session>
    {
      "email": "alice@example.com",
      "role": "MEMBER"
    }

    レスポンス：
    {
      "invitation_id": "inv-uuid",
      "email": "alice@example.com",
      "role": "MEMBER",
      "expires_at": "2026-04-08T12:00:00Z",
      "invite_url": "https://app.com/invite?code=ej-ykYH3mQ_7vKxN..."
    }

  招待承認：
    POST /auth/accept-invitation
    {
      "code": "ej-ykYH3mQ_7vKxN...",
      "name": "Alice Smith",
      "password": "str0ng-p@ssword"
    }
```

### 招待の状態

```
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ PENDING  │────►│ ACCEPTED │     │ EXPIRED  │
  │          │     │          │     │          │
  │ 作成済み  │     │ ユーザー  │     │ TTL     │
  │ 応答待ち  │     │ テナント  │     │ 超過    │
  │          │     │ に参加   │     │          │
  └──────────┘     └──────────┘     └──────────┘
       │                                  ▲
       │           ┌──────────┐           │
       └──────────►│ REVOKED  │           │
                   │          │     （有効期限後に
                   │ 管理者   │      自動遷移）
                   │ がキャンセル│
                   └──────────┘
```

### 他のvolta機能との統合

- **[ロール](role.md)**：招待が新規ユーザーのロールを指定
- **[テナント](tenant.md)**：コードは特定のテナントに紐づく
- **[Webhook](webhook.md)**：`invitation.accepted`イベントが[Outbox](outbox-pattern.md)経由で発火
- **[セッション](session.md)**：承認後、ユーザーは即座にセッションを取得
- **[SCIM](scim.md)**：SCIM[プロビジョニング](provisioning.md)はエンタープライズ向けの招待の代替

## よくある間違いと攻撃

### 間違い1：短いまたは予測可能なコードを使う

6桁の数字コードは100万通りしかなく、簡単にブルートフォースできます。常に十分なエントロピー（256ビット）を持つ暗号学的にランダムなコードを使ってください。

### 間違い2：データベースに生のコードを保存する

データベースがダンプされると、保留中のすべての招待コードが取得されます。常にSHA256ハッシュを保存してください。

### 間違い3：有効期限がない

6ヶ月前の招待がまだ有効であるべきではありません。合理的な有効期限を設定してください（volta：デフォルト7日）。

### 間違い4：メールに紐づけない

コードがメールに紐づいていない場合、傍受した誰でも使用できます。voltaは登録ユーザーのメールが招待メールと一致することを検証します。

### 間違い5：再利用を許可する

コードを複数回使用できると、ソーシャルメディアで共有すれば誰でもテナントに参加できてしまいます。

### 攻撃：ブルートフォースコード推測

攻撃者が招待承認エンドポイントに対してランダムなコードを試す。防御：32バイトのエントロピーでこれは不可能（2^256通り）、さらにエンドポイントのレート制限。

### 攻撃：メール傍受

招待メールが傍受された場合（例：侵害されたメールアカウント）、攻撃者がコードを使用できます。防御：コードの早期期限切れ、管理者が保留中の招待を取り消し可能、承認された招待が監視用のWebhookをトリガー。
