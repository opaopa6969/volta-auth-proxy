# 招待フロー（Invitation Flow）

[English version](invitation-flow.md)

---

## 一言で言うと？

招待フローとは、誰かがワークスペースへの参加に招待されてから、チームの正式なメンバーになるまでの段階的なプロセスのことです。

---

## 引っ越しパーティーのたとえ

新しいマンションに誰かを招待することを考えてみてください：

1. **招待状を送る** -- 「遊びに来て！住所とドアのコードはこれだよ」
2. **来訪する** -- 建物に到着して、ドアマンに身分を示す
3. **家のルールに同意する** -- 「靴を脱いでね、猫には触らないでね」
4. **中に入れる** -- くつろいで、お菓子を食べて、トイレも使える

ソフトウェアでも、ワークスペースへの招待はとてもよく似たパターンをたどります：

1. **管理者が招待を送る** -- 特別なコード付きのメールやリンク
2. **本人がログインする** -- 自分が誰かを証明する（認証）
3. **招待を受け入れる** -- ワークスペースへの参加に同意する
4. **メンバーになる** -- 与えられたロールでワークスペースを使えるようになる

---

## なぜ招待フローが重要なの？

「誰でもどのワークスペースにも参加できるようにすればいいのでは？」と思うかもしれません。それが大惨事になる理由：

- **セキュリティ** -- 見知らぬ人に会社のデータにアクセスされたくない。明示的に招待された人だけが参加できるべき。
- **ロールの割り当て** -- 招待する人が新メンバーのロールを決める。インターンが誤ってOWNERアクセスを取得すべきではない。
- **監査証跡** -- 誰がいつ誰を招待したかを知る必要がある。問題が起きたとき、追跡できる。
- **成長の制御** -- ワークスペースにはメンバー上限がある場合がある（voltaはテナントプランごとの`max_members`を強制する）。

## voltaでの招待フロー、ステップごとに

voltaのワークスペースに誰かが招待されたとき、正確に何が起きるか：

```
  ステップ1：管理者が招待を作成
  ──────────────────────────────
  けんじ（ACME CorpのADMIN）が管理パネルを開いて入力：
    メール：mika@example.com
    ロール：MEMBER

  voltaが招待レコードを作成：
    - ユニークな招待コード
    - 対象のメール
    - 割り当てるロール
    - 有効期限（招待は永遠には続かない）
    - 送信者（けんじ）

  ステップ2：招待が送信される
  ──────────────────────────
  voltaがmika@example.comにメールを送信：
    「けんじがACME Corpへの参加に招待しています。
     ここをクリックして受け入れてください：https://volta.example.com/invite/abc123」

  ステップ3：みかがリンクをクリック
  ─────────────────────────────────
  みかがメール内のリンクをクリック。
  volta：「みかはログインしている？」
    - はい → ステップ4へ
    - いいえ → ログインページにリダイレクトし、戻ってくる

  ステップ4：みかがログイン（必要な場合）
  ──────────────────────────────────────
  みかがGoogle（または設定された方法）でログイン。
  volta：「OK、あなたはmika@example.com。確認済み。」

  ステップ5：同意 / 受け入れ
  ──────────────────────────
  voltaがページを表示：
    「ACME CorpにMEMBERとして招待されました。
     参加しますか？」
    [受け入れる]  [辞退する]

  みかが[受け入れる]をクリック。

  ステップ6：メンバーシップが作成される
  ──────────────────────────────────────
  voltaがメンバーシップレコードを作成：
    user_id:    mika-uuid
    tenant_id:  acme-uuid
    role:       MEMBER
    invited_by: kenji-uuid

  招待は「使用済み」としてマーク（再利用不可）。

  ステップ7：みかが参加完了！
  ──────────────────────────
  みかがACME Corpのワークスペースにリダイレクトされる。
  MEMBERロールを許可するすべてのアプリを使えるようになる。
```

## セキュリティは？

招待フローにはいくつかのセキュリティ保護が組み込まれています：

- **有効期限** -- 招待リンクは一定時間後に期限切れ。メール履歴で見つかった古いリンクは使えない。
- **一回限りの使用** -- 受け入れ後、招待コードは再利用不可。誰かがリンクを傍受しても、みかがすでに受け入れた後は使えない。
- **メール確認** -- 招待は特定のメールに紐付いている。別の人がリンクを使おうとしても、voltaはログイン中のユーザーのメールが招待と一致するか確認する。
- **ロール制限** -- ADMINはADMINレベルまでのロールでしか招待できない。OWNERだけが別の人をOWNERにできる。
- **メンバー制限** -- voltaは新メンバーを許可する前に、テナントの`max_members`制限を確認する。

## 簡単な例

```
  ACME Corpワークスペース：
    ゆき（OWNER）- ワークスペースを作った人
    けんじ（ADMIN）- チームマネージャー

  けんじが新しいインターンのりくを追加したい：

  1. けんじ → POST /api/v1/tenants/acme-uuid/invitations
     ボディ：{ "email": "riku@example.com", "role": "VIEWER" }

  2. volta → riku@example.comにリンク付きメールを送信

  3. りくがリンクをクリック → Googleでログイン → 「ACME Corpに参加？」と表示

  4. りくが受け入れをクリック → voltaがメンバーシップを作成：
     りく - ACME CorpのVIEWER、けんじが招待

  5. りくはACME Corpのアプリでコンテンツを閲覧できるようになる
     （ただし編集はできない、VIEWERは読み取り専用だから）
```
