# JVM（Java仮想マシン）

[English version](jvm.md)

---

## これは何？

Java Virtual Machine（JVM）は、Javaアプリケーションを実行するプログラムです。Javaコードを書くと「バイトコード」にコンパイルされます。これは特定のOSに依存しない中間形式です。JVMがこのバイトコードを読み、あなたのコンピュータが理解する命令に変換します。

万能翻訳機のようなものです。Javaで指示を書くと、JVMがコンピュータの言語 -- Windows、Mac、Linuxなど -- に翻訳してくれます。

---

## なぜ重要なのか？

JVMが認証の世界で重要なのは、一つの言葉に集約されます：**Keycloak**。Keycloakは最も有名なオープンソースのアイデンティティサーバーで、JVM上で動作します。JVMを理解すると、Keycloakがなぜ起動するだけで~512MBのRAMを使うのか、なぜ起動に~30秒かかるのか、そしてvolta-auth-proxyが同じ[Java](java.ja.md)で書かれているにもかかわらず~30MBしか使わず~200msで起動するのかが分かります。

違いは言語ではありません。JVMの上に何を載せるかの違いです。

---

## 「JVM上で動く」の意味

「このアプリケーションはJVM上で動く」と言うとき、それは：

1. アプリケーションがJava（またはKotlin、Scala、Clojure、Groovy -- すべてJVM言語）で書かれている
2. 実行にJVMのインストールが必要
3. アプリケーションコードが動き始める前にJVM自体がメモリとCPUを消費する

```
  JVMアプリケーション起動時の流れ：

  1. JVM起動                           [JVMだけで~50-100MB]
  2. アプリケーションクラスのロード     [クラスローディング、検証]
  3. フレームワークの初期化             [本当のコストはここ]
  4. ホットパスのJITコンパイル          [時間とともに性能向上]
  5. アプリケーション準備完了           [やっとリクエスト受付開始]

  ステップ1-4はJavalinで~200ms。
  ステップ1-4はKeycloak/Quarkus（全拡張機能付き）で~30秒。
```

JVM自体は問題ではありません。JVMは実際に驚くほど高速で効率的です。問題はステップ3でフレームワークが何をするかです。

---

## Keycloakが重い理由（JVMオーバーヘッド + フレームワークオーバーヘッド）

Keycloakは以前WildFly（フルJava EEアプリケーションサーバー）上で動作し、現在はQuarkus上で動作しています。どちらの場合も、起動時に膨大な機能をロードします：

```
  Keycloak起動時のロード内容：
  ┌─────────────────────────────────┐
  │ JVMベース                       │  ~50MB
  │ Quarkusランタイム               │  ~30MB
  │ Hibernate ORM                   │  ~20MB
  │ RESTEasy (JAX-RS)               │  ~15MB
  │ Infinispan（分散キャッシュ）     │  ~30MB
  │ Keycloakアイデンティティエンジン │  ~50MB
  │ 管理コンソール (React SPA)       │  （メモリから提供）
  │ テーマエンジン (FreeMarker)      │  ~10MB
  │ プロトコルマッパー (OIDC, SAML)  │  ~20MB
  │ イベントリスナー                 │  ~10MB
  │ ユーザーフェデレーション SPI     │  ~10MB
  │ その他数十のサブシステム...       │  ~???MB
  ├─────────────────────────────────┤
  │ 合計: アイドル時~512MB+          │
  └─────────────────────────────────┘
```

KeycloakはJavaが原因で遅いのではありません。Keycloakが遅いのは、必要かどうかに関わらずすべてをロードする**フル機能のエンタープライズIDプラットフォーム**だからです。スイスアーミーナイフのたとえが当てはまります：パンを切りたいだけでも、30個の道具をすべて持ち運んでいるのです。

## voltaがJavaなのに軽い理由

volta-auth-proxyもJavaで書かれています。JVM上で動作します。しかし[Javalin](https://javalin.io/)という最小限のWebフレームワークを使います：

```
  volta起動時のロード内容：
  ┌─────────────────────────────────┐
  │ JVMベース                       │  ~15MB（-Xmxチューニング済み）
  │ Javalin（組み込みJetty）        │  ~5MB
  │ HikariCP（コネクションプール）  │  ~2MB
  │ JJWT（JWTライブラリ）           │  ~1MB
  │ voltaアプリケーションコード     │  ~5MB
  │ jteテンプレート（コンパイル済み）│  ~2MB
  ├─────────────────────────────────┤
  │ 合計: アイドル時~30MB            │
  └─────────────────────────────────┘
```

違いは歴然です：

| 指標 | Keycloak | volta |
|------|----------|-------|
| アイドル時RAM | ~512MB+ | ~30MB |
| 起動時間 | ~30秒 | ~200ms |
| フレームワーク | Quarkus + 数十の拡張機能 | Javalin（最小限） |
| アプローチ | すべてロード | 必要なものだけロード |

voltaはこれを意見の強い選択で実現しています：

- **ORMなし。** JDBC経由の生SQL。Hibernateのクラスローディングもエンティティスキャンもレイジープロキシ生成もなし。
- **フルアプリケーションサーバーなし。** JavalinはJettyの薄いラッパー。CDIもJAX-RSもBean検出もなし。
- **分散キャッシュなし。** セッションはPostgreSQL（またはRedis）。Infinispanクラスタ管理なし。
- **プラグインシステムなし。** voltaはプラットフォームではなく1つのアプリケーション。SPIスキャンも拡張機能ロードもなし。

---

## JVMは敵ではない

「Javaは遅い」「JVMアプリケーションは重い」とよく聞きます。これは不正確です。JVM自体はこれまでに作られた最も最適化されたソフトウェアの一つです：

- **JITコンパイル**により、長時間稼働するJavaアプリケーションは多くの「コンパイル言語」より速くなる
- **G1 / ZGCガベージコレクタ**が最小限の停止でメモリを効率的に処理
- **仮想スレッド**（Java 21+）により並行プログラミングが軽量に

重さの原因は**フレームワーク**であり、JVMではありません。最小限のフレームワーク（Javalin、Spark、Helidon SE）を選ぶか、フルスタックフレームワーク（Spring Boot、拡張機能付きQuarkus、全機能のMicronaut）を選ぶかは、異なるプログラミング言語を選ぶより大きな違いを生みます。

voltaがそれを証明しています：同じJVM、同じ言語で、Keycloakの17分の1のメモリ。
