# JWKS / JWK (JSON Web Key Set)

## これは何？

JWK (JSON Web Key) とは、公開鍵などの暗号鍵を JSON 形式で表現する標準的な方法です。JWKS (JSON Web Key Set) は、その鍵をまとめたコレクションで、決まった URL で公開されるため、他のアプリケーションが自動的に取得できます。

分かりやすく言うと、誰かからサイン入りの手紙を受け取ったとき、そのサインが本物かどうか確認するには、相手のサインを知っている必要があります。JWKS は、アプリケーションがオンラインで参照できる「サインの台帳」のようなものです。JWT（JSON Web Token）が本物かどうかを確認するために使います。

鍵を公開する標準的なエンドポイントはこちらです:

```
https://your-auth-server/.well-known/jwks.json
```

JWKS のレスポンスは、だいたいこんな感じです:

```json
{
  "keys": [
    {
      "kty": "RSA",
      "kid": "key-2026-03-15T10-00",
      "alg": "RS256",
      "use": "sig",
      "n": "0vx7agoebGcQ...(公開鍵のデータ)...",
      "e": "AQAB"
    }
  ]
}
```

各フィールドの意味:

- **kty** - Key Type（鍵の種類）。"RSA" は RSA 鍵ペアという意味です。
- **kid** - Key ID（鍵の識別子）。どの鍵でトークンに署名したかを示すラベルです。
- **alg** - Algorithm（アルゴリズム）。"RS256" は RSA 署名 + SHA-256 ハッシュの意味です。
- **use** - Usage（用途）。"sig" は署名用（暗号化用ではない）という意味です。
- **n** と **e** - RSA 公開鍵の実際の数学的な値です。

ここで重要なのは、公開されるのは**公開鍵だけ**ということです。署名に使う秘密鍵は、認証サーバーの外には絶対に出ません。

## なぜ重要？

JWKS がなかったら、JWT を検証したいアプリケーションは、公開鍵を手作業でやり取りしなければなりません。ファイルをコピーしたり、チャットで秘密を共有したり、設定ファイルに直接鍵を書き込んだり。この方法は、鍵を変更（ローテーション）する必要が出た瞬間に破綻します。

JWKS は 3 つの大きな問題を解決します:

1. **鍵の自動検出。** アプリケーションは URL から鍵を取得するだけ。手作業の共有は不要です。
2. **ダウンタイムなしの鍵ローテーション。** 古い鍵を廃止する前に、新しい鍵を JWKS に追加できます。移行期間中は両方の鍵がセットに含まれているので、どちらの鍵で署名されたトークンも検証できます。
3. **疎結合。** 認証サーバーとアプリケーションサーバーは、秘密を共有する必要がありません。公開鍵は文字通り公開情報です。誰でも読めますが、署名できるのは対応する秘密鍵を持つ認証サーバーだけです。

検証の流れはこうなります:

```
  アプリが JWT を受け取る
        |
        v
  JWT ヘッダから "kid" を読む
        |
        v
  /.well-known/jwks.json から JWKS を取得
        |
        v
  一致する "kid" の鍵を探す
        |
        v
  その公開鍵で署名を検証
        |
        v
  有効?  --> JWT の中身（claims）を信頼する
  無効?  --> リクエストを拒否する
```

## どう動くの？

認証サーバーが JWT を発行する際、秘密鍵で署名し、JWT ヘッダに `kid`（Key ID）を含めます。後でアプリケーションがその JWT を受け取ったとき:

1. JWT ヘッダを解析して `kid` を取得する。
2. 認証サーバーの well-known URL から JWKS を取得する。
3. `keys` 配列の中から、`kid` が一致する JWK を探す。
4. その公開鍵を使って、数学的に署名を検証する。

アプリケーションは通常、JWKS レスポンスをキャッシュして、毎回のリクエストのたびに取得しなくて済むようにします。よくあるパターンは、鍵をキャッシュしておいて、知らない `kid` のJWT が来たときだけ再取得する（鍵のローテーションが起きた可能性があるため）というものです。

### 鍵のローテーション

鍵のローテーションとは、署名鍵を定期的に新しいものに交換する運用のことです。万が一秘密鍵が漏洩した場合の被害を限定するために行います。典型的なローテーションの流れ:

```
時間 ----->

  フェーズ 1: 鍵 A だけが存在
  JWKS: [鍵 A]
  全トークンを鍵 A で署名

  フェーズ 2: 鍵 B を作成。両方の鍵を JWKS に掲載。
  JWKS: [鍵 A, 鍵 B]
  新しいトークンは鍵 B で署名
  鍵 A で署名された古いトークンもまだ検証可能

  フェーズ 3: 鍵 A を廃止
  JWKS: [鍵 B]
  鍵 A のトークンはもう全部有効期限切れ
  全ての新しいトークンは鍵 B で署名
```

重なり期間（フェーズ 2）が大事です。古い鍵で署名されたトークンが有効期限切れになるまで、問題なく検証できることを保証します。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は独自の JWT 発行者として機能します。RSA 2048 ビットの鍵ペアを生成し、JWKS エンドポイントを提供しています:

```
GET /.well-known/jwks.json
```

裏側で起きていること:

1. **鍵の作成。** 初回起動時、volta は RSA 鍵ペアを生成し、公開鍵と秘密鍵の両方をデータベースに保存します。秘密鍵は保存前に AES で暗号化されます。
2. **Key ID の形式。** volta はタイムスタンプベースの Key ID を使います（例: `key-2026-03-15T10-00`）。いつ作られた鍵かが一目で分かります。
3. **JWKS エンドポイント。** 下流のアプリケーション（または誰でも）が `/.well-known/jwks.json` をリクエストすると、volta は現在の鍵の公開部分だけを返します。キャッシュヘッダ（`Cache-Control: public, max-age=60`）が付いているので、クライアントがエンドポイントに負荷をかけすぎることを防ぎます。
4. **鍵のローテーション。** volta は管理者 API による鍵ローテーションに対応しています。ローテーションが実行されると、新しい鍵ペアが生成され、古い鍵はデータベースで廃止とマークされ、新しい鍵がアクティブな署名鍵になります。
5. **トークンの検証。** volta 自身が JWT を受け取った場合（例: API リクエストの Bearer トークン）、現在の RSA 公開鍵で署名を検証します。

全体の流れ:

```
  volta-auth-proxy                    下流アプリ（例: wiki）
  +-----------------+                 +-------------------+
  |                 |  1. JWKS 取得   |                   |
  | RSA 秘密鍵      |<----------------|  ユーザーから      |
  | RSA 公開鍵      |----JWKS JSON--->|  JWT を受け取る    |
  |                 |                 |                   |
  | 秘密鍵で         |                 |  公開鍵で          |
  | JWT に署名       |                 |  JWT を検証        |
  +-----------------+                 +-------------------+
```

`volta-config.yaml` で定義された `app-wiki` や `app-admin` などの下流アプリケーションは、JWKS エンドポイントから公開鍵を取得して、volta が発行した JWT を検証できます。秘密鍵を知る必要は一切ありません。

## よくある間違い

**1. JWKS で秘密鍵を公開してしまう。**
JWKS エンドポイントは公開鍵だけを返すべきです。うっかり秘密鍵を含めてしまうと、誰でも有効なトークンを偽造できてしまいます。volta はエンドポイント提供時に明示的に公開鍵だけの JWK を構築することで、これを防いでいます。

**2. JWKS レスポンスをキャッシュしない。**
毎回のリクエストごとに JWKS を取得すると、遅延が増え、認証サーバーに過負荷がかかります。レスポンスをキャッシュして、知らない `kid` に遭遇したときだけ再取得するようにしましょう。

**3. `kid` の検証をスキップする。**
JWT ヘッダの `kid` を JWKS と照合しないと、間違った鍵を使ったり、攻撃者が提供した鍵を盲目的に信頼してしまう可能性があります。検証する前に必ず `kid` を照合してください。

**4. 鍵のローテーションを忘れる。**
同じ署名鍵をずっと使い続けると、もし漏洩した場合、発行された全てのトークンが危険にさらされます。定期的に鍵をローテーションし、既存のトークンが有効期限切れになるまで重なり期間を十分に取りましょう。

**5. JWT の `alg` ヘッダを無制限に受け入れる。**
攻撃者が `"alg": "none"`（署名なし）や `"alg": "HS256"`（公開鍵を秘密鍵として使う対称方式）の JWT を送ってくる可能性があります。volta は RS256 だけを受け入れ、それ以外は全て拒否します。

**6. JWKS を使わずに公開鍵をハードコーディングする。**
鍵をローテーションするまでは動きますが、ローテーションした瞬間に全てが壊れます。ローテーションをシームレスにするために JWKS エンドポイントを使いましょう。
