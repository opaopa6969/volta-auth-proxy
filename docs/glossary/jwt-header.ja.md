# JWT ヘッダー

[English version](jwt-header.md)

---

## これは何？

JWT（JSON Web Token）はドットで区切られた3つの部分で構成されます：`ヘッダー.ペイロード.署名`。**ヘッダー**は最初の部分です。小さな JSON オブジェクトで、受信者に2つの重要なことを伝えます：トークンが*どのように*署名されたか、*どの種類*のトークンか。

封筒のラベルのようなものです。手紙（ペイロード）を開ける前に、ラベルを見てどんな書類が入っていて、どんな封印で閉じられているか確認します。

---

## なぜ重要？

ヘッダーがないと、受信者はトークンの検証方法がわかりません。署名アルゴリズムは多数あり（RS256、HS256、ES256 など）、それぞれ検証方法が異なります。ヘッダーが「このトークンは RS256 で、鍵 `key-2025-01-01` を使って署名されました」と伝えることで、検証者は何をすべきか正確にわかります。

`kid`（Key ID）フィールドは**鍵ローテーション**時に特に重要です。署名鍵が変更されると、古いトークン（旧鍵で署名）と新しいトークン（新鍵で署名）が共存する可能性があります。`kid` が検証者にどの鍵を使うべきか教えてくれます。

---

## 簡単な例

デコードされた JWT ヘッダー：

```json
{
  "alg": "RS256",
  "kid": "key-2025-03-31T09-00",
  "typ": "JWT"
}
```

| フィールド | 意味 | 例 |
|-----------|------|-----|
| `alg` | 署名アルゴリズム | `RS256`（RSA + SHA-256） |
| `kid` | どの鍵で署名したか | `key-2025-03-31T09-00` |
| `typ` | トークンの種類 | `JWT` |

`typ: "JWT"` は冗長に見えるかもしれませんが、複数のトークン形式を扱うシステム（JWT と SAML トークンなど）で「これは JWT であり、他のものではない」と明示するのに役立ちます。

## volta-auth-proxy では

volta は `JwtService.java` で JWT ヘッダーを構築します：

```java
new JWSHeader.Builder(JWSAlgorithm.RS256)
    .keyID(rsaKey.getKeyID())
    .type(JOSEObjectType.JWT)
    .build()
```

ポイント：
- **アルゴリズムは常に RS256。** volta は検証時に異なるアルゴリズムのトークンを拒否します：`if (!JWSAlgorithm.RS256.equals(jwt.getHeader().getAlgorithm()))` -- これにより、攻撃者がアルゴリズムを "none" に設定して署名チェックを回避する「alg: none 攻撃」を防止します。
- **kid のフォーマット：** `key-` の後にタイムスタンプが続きます（例：`key-2025-03-31T09-00`）。鍵がいつ作成されたか一目でわかります。
- **typ は常に "JWT"。** 明確化のための標準的なプラクティス。
