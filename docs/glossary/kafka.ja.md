# Kafka

[English version](kafka.md)

---

## これは何？

Apache Kafkaは、LinkedInで開発され2011年にオープンソース化された分散イベントストリーミングプラットフォームです。アプリケーションが耐障害性のある高スループットの方法でイベント（メッセージ）のストリームをパブリッシュ・サブスクライブできます。Kafkaは毎秒数百万のイベントを処理し、設定可能な期間（数時間から永久まで）それらを保持するように設計されています。

Kafkaは、新聞印刷機と図書館が合体したようなものです。印刷機（プロデューサー）は継続的に新聞（イベント）を印刷します。図書館（Kafkaブローカー）はすべての版を順番に保管します。読者（コンシューマー）はいつでも来て、中断した場所から読むことができます -- 何日も不在でも、見逃したすべての版を読んで追いつけます。複数の読者が同じ新聞を独立して、自分のペースで読めます。

これは、メッセージが一度消費されると消える従来のメッセージキュー（RabbitMQなど）とは根本的に異なります。Kafkaでは、イベントは永続的に保存され、複数のコンシューマーが独立して読むことができます。

---

## なぜ重要なのか？

volta-auth-proxyにとって、Kafkaは監査ログとシステム統合の**イベントストリーミングバックボーン**として重要です。認証イベント（ログイン、ログアウト、失敗、権限変更）が発生すると、複数のシステムが知る必要があるかもしれません：

- 監査ログストア（[Elasticsearch](elasticsearch.md)）がイベントをインデックスする必要がある
- アラートシステムが異常をチェックする必要がある
- 課金システムがアクティブユーザーをカウントする必要がある
- 分析ダッシュボードがリアルタイムメトリクスを更新する必要がある
- コンプライアンスシステムがイベントをアーカイブする必要がある

Kafkaがなければ、voltaは各システムに直接イベントを送信する必要があります。いずれかのシステムがダウンしていると、イベントが失われます。Kafkaがあれば、voltaはイベントを一度パブリッシュし、各システムが独立して、自分のペースで、配信保証付きで消費します。

## どう動くのか？

### コア概念

| 概念 | 説明 |
|------|------|
| **プロデューサー** | Kafkaにイベントをパブリッシュするアプリケーション（例：volta-auth-proxy） |
| **コンシューマー** | Kafkaからイベントを読むアプリケーション（例：Elasticsearchシンク） |
| **トピック** | 名前付きのイベントストリーム（例：`volta.audit.events`） |
| **パーティション** | トピックは並列性のためにパーティションに分割。パーティション内のイベントは順序付き。 |
| **オフセット** | パーティション内の各イベントの順序ID。コンシューマーがオフセットを追跡。 |
| **ブローカー** | イベントを保存するKafkaサーバー。複数のブローカーでクラスターを形成。 |
| **コンシューマーグループ** | 負荷分散のためにパーティションを分け合うコンシューマーのグループ。 |

### イベントフロー

```
  volta-auth-proxy（プロデューサー）
       │
       │  イベントをパブリッシュ
       ▼
  ┌──────────────────────────────────┐
  │           Kafka クラスター         │
  │                                   │
  │  トピック: volta.audit.events      │
  │  ┌─────────┐ ┌─────────┐        │
  │  │ Part. 0  │ │ Part. 1  │       │
  │  │ offset 0 │ │ offset 0 │       │
  │  │ offset 1 │ │ offset 1 │       │
  │  │ offset 2 │ │ ...      │       │
  │  └─────────┘ └─────────┘        │
  └──────────────────────────────────┘
       │              │
       ▼              ▼
  コンシューマーグループA  コンシューマーグループB
  (Elasticsearch)       (アラートシステム)
```

### 直接統合よりKafkaを使う理由

```
  Kafkaなし：                       Kafkaあり：
  volta ──► Elasticsearch           volta ──► Kafka ──► Elasticsearch
  volta ──► アラートシステム                        ──► アラート
  volta ──► 分析                                   ──► 分析
  volta ──► コンプライアンス                         ──► コンプライアンス

  問題：Elasticsearchがダウンすると   Kafkaはイベントを永続的に保存。
  voltaがブロックまたはイベントを失う。 コンシューマーは準備ができたら追いつく。
```

### Kafka vs 代替手段

| 機能 | Kafka | RabbitMQ | Redis Streams | Amazon SQS |
|------|-------|----------|---------------|------------|
| 永続性 | ディスクベース、保持期間設定可能 | メモリ + ディスク | メモリ + オプション永続化 | クラウドマネージド |
| スループット | 非常に高い（数百万/秒） | 高い（10万/秒） | 非常に高い | 高い |
| メッセージリプレイ | 可能（コンシューマーがオフセットを追跡） | 不可（一度消費） | 可能（限定的） | 不可 |
| 順序保証 | パーティション内 | キュー内 | ストリーム内 | ベストエフォート |
| 複数コンシューマー | 可能（独立グループ） | 可能（ただしキュー内で一度消費） | 可能 | 不可（SNSファンアウトを除く） |
| 複雑さ | 高い（クラスター、ZooKeeper/KRaft） | 中程度 | 低い | 低い（マネージド） |

## volta-auth-proxy ではどう使われている？

Kafkaは[Elasticsearch](elasticsearch.md)と並んで、volta-auth-proxyの**オプションの外部監査ログシンク**です。voltaは認証イベントをKafkaトピックにパブリッシュし、下流のコンシューマーがそれらを独立して処理します。

### アーキテクチャ

```
  volta-auth-proxy
       │
       │  監査イベント
       │
       ├──► PostgreSQL（常時 -- プライマリストレージ）
       │
       └──► Kafkaトピック: volta.audit.events（オプション）
                │
                ├──► コンシューマー: Elasticsearchインデクサー
                ├──► コンシューマー: アラート / 異常検知
                ├──► コンシューマー: 分析パイプライン
                └──► コンシューマー: コンプライアンスアーカイブ（S3など）
```

### パーティショニング戦略

voltaは`tenant_id`でイベントをパーティショニングします。これにより、単一テナントのすべてのイベントが（パーティション内で）順序付けられ、テナント固有のプロセッサーで効率的に消費できます。

### なぜオプションで必須ではないのか？

[Elasticsearch](elasticsearch.md)と同じ理由です：voltaの哲学は最小限の依存関係。多くのデプロイでは監査ログにPostgresだけで十分です。Kafkaは以下を追加します：

- **運用の複雑さ**：クラスター（本番には最低3ブローカー）
- **リソース使用量**：Kafkaブローカーには相当なRAMとディスクが必要
- **専門知識**：Kafka運用（リバランシング、パーティション管理、コンシューマーラグ）には専門知識が必要

Kafkaが意味を持つのは：

- 複数の下流システムが監査イベントを消費する必要がある
- 下流システムが一時的にオフラインでも配信保証が必要
- 他のサービスですでにKafkaを運用している
- イベントボリュームがリアルタイムクエリでPostgresが処理できる量を超える

## よくある間違いと攻撃

### 間違い1：トピックのパーティション不足

パーティションが少なすぎるとコンシューマーの並列性が制限されます。パーティションが1つだと、グループ内の1つのコンシューマーしか同時に読めません。期待するコンシューマーインスタンス数以上のパーティションで開始してください。

### 間違い2：コンシューマーラグを監視しない

コンシューマーラグは、最新のイベントオフセットとコンシューマーの現在のオフセットの差です。ラグが増加するとコンシューマーが遅れています。監査ログの場合、増大するラグはElasticsearchやアラートシステムが最近のイベントを見逃していることを意味します。ラグ監視とアラートを設定してください。

### 間違い3：リテンションが短すぎる

Kafkaリテンションを24時間に設定し、下流コンシューマーが48時間ダウンした場合、それらのイベントは失われます。監査ログには少なくとも7日、コンプライアンスが要求する場合はそれ以上のリテンションを設定してください。

### 間違い4：Kafkaを保護しない

KafkaにはオプションのAuthentication（SASL）と暗号化（TLS）がありますが、デフォルトでは無効です。Kafkaがネットワーク上で到達可能な場合、誰でも偽のイベントを生成したり機密の監査データを消費できます。本番ではSASL + TLSを有効にしてください。

### 攻撃：監査ログインジェクション

攻撃者がKafka監査トピックにメッセージを生成できる場合、偽イベントを注入して：
- 行動の隠蔽（ノイズで溢れさせる）
- 他のユーザーのフレーム（偽の「ログイン」イベントを注入）
- 誤ったアラートのトリガー（大量の「login_failed」イベントを注入）

Kafkaプロデューサー ACLを制限して、volta-auth-proxyのみが監査トピックに書き込めるようにしてください。
