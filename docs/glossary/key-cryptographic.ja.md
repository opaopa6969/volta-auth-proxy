# 暗号鍵

[English version](key-cryptographic.md)

---

## これは何？

暗号鍵とは、情報をロック（暗号化）、アンロック（復号）、または署名するために使われる秘密のデータです。正しい鍵がなければ、暗号化されたデータは読めず、署名の作成も検証もできません。

物理的な鍵と錠前を思い浮かべてください。家の鍵は自分の玄関を開けますが、他の誰の玄関も開けません。暗号鍵も同じです -- 暗号操作を機能させる特定のビット列です。1ビット変えただけで「錠前」は開きません。鍵を失えば、それが保護していたすべてへのアクセスを失います。

大きく2つの種類があります：**対称鍵**（ロックとアンロックの両方に1つの鍵を使う、南京錠の鍵のようなもの）と**非対称鍵ペア**（公開鍵と秘密鍵のペアが連携する、誰でも投函口から手紙を入れられるが鍵の持ち主だけが開けられるポストのようなもの）。voltaは両方を使います：AES-256対称鍵は[保存時の暗号化](encryption-at-rest.md)に、RSA非対称鍵は[JWT署名](digital-signature.md)に使います。

---

## なぜ重要なのか？

鍵はすべての暗号セキュリティの基盤です。暗号化されたデータベースフィールド、署名されたJWT、HTTPS接続のすべてが鍵に依存しています。鍵が漏洩すると：

- **暗号化が破れる**: AES鍵を持つ攻撃者はデータベースに保存されたすべての署名鍵を復号できる
- **署名が偽造可能になる**: RSA秘密鍵を持つ攻撃者は検証を通過する偽のJWTを発行できる
- **信頼が崩壊する**: 下流のサービスは正当なトークンと偽造されたものを区別できなくなる

鍵は安全に生成し（[暗号的乱数](crypto-random.md)ソースを使用）、安全に保存し（暗号化、ログに出さない）、定期的にローテーションする（[鍵ローテーション](key-rotation.md)）必要があります。システムのセキュリティは、最も弱い鍵管理の実践と同じレベルでしかありません。

## どう動くのか？

### 対称鍵

1つの鍵、2つの操作：

```
  対称鍵: K
  ┌──────────────────────────────────────────────┐
  │                                              │
  │  暗号化:  平文 + K → 暗号文                    │
  │  復号:    暗号文 + K → 平文                    │
  │                                              │
  │  両方向で同じ鍵を使用。                        │
  │  両者がKを秘密に共有する必要あり。              │
  └──────────────────────────────────────────────┘

  例: AES-256-GCM（voltaのKeyCipherが使用）
  ┌─────────────────────────────────────────┐
  │  鍵:  32バイト（256ビット）SHA-256から   │
  │  IV:  12バイト（操作ごとにランダム）     │
  │  タグ: 16バイト（認証タグ）              │
  │                                         │
  │  "v1:" + base64(IV) + ":" + base64(     │
  │     暗号化データ + 認証タグ)             │
  └─────────────────────────────────────────┘
```

### 非対称鍵ペア

2つの鍵、補完的な操作：

```
  秘密鍵: sk     公開鍵: pk
  ┌──────────────────────────────────────────────┐
  │                                              │
  │  署名:    メッセージ + sk → 署名              │
  │  検証:    メッセージ + 署名 + pk → yes/no     │
  │                                              │
  │  暗号化:  平文 + pk → 暗号文                  │
  │  復号:    暗号文 + sk → 平文                  │
  │                                              │
  │  秘密鍵は秘密。公開鍵は共有。                 │
  └──────────────────────────────────────────────┘

  例: RSA-2048（voltaのJwtServiceが使用）
  ┌─────────────────────────────────────────┐
  │  秘密鍵: 2048ビット、DBに保存           │
  │          （AESで暗号化）                 │
  │  公開鍵: 秘密鍵から導出                 │
  │          JWKSエンドポイントで公開        │
  └─────────────────────────────────────────┘
```

### 鍵のサイズと強度

| アルゴリズム | 鍵サイズ | 強度 | 用途 |
|-------------|---------|------|------|
| AES-256 | 256ビット | 非常に強い | 保存時の署名鍵暗号化 |
| RSA-2048 | 2048ビット | 強い | JWT署名 |
| HMAC-SHA256 | 256ビット | 非常に強い | Cookie署名、Webhook検証 |

### 鍵のライフサイクル

```
  生成 ──→ 保存（暗号化） ──→ 使用 ──→ ローテーション ──→ 引退
   │           │              │           │              │
  SecureRandom AES-256-GCM  JWTに署名    新しい鍵       古い鍵を
  またはKeyPair signing_keys  または      作成           inactive
  Generator   テーブルに     暗号化       古い鍵を       に設定
                                         引退
```

## volta-auth-proxy ではどう使われている？

voltaは3種類の暗号鍵を管理しています：

### 1. RSA署名鍵ペア（非対称）

Javaの`KeyPairGenerator`を使って`JwtService.createKey()`で生成：

```java
KeyPairGenerator generator = KeyPairGenerator.getInstance("RSA");
generator.initialize(2048);
KeyPair keyPair = generator.generateKeyPair();
// 公開鍵: /.well-known/jwks.json で共有
// 秘密鍵: JWTの署名に使用、サーバー外には出ない
```

### 2. AES暗号化鍵（対称）

`KeyCipher`で`JWT_KEY_ENCRYPTION_SECRET`環境変数から導出：

```java
MessageDigest sha = MessageDigest.getInstance("SHA-256");
byte[] key = sha.digest(secret.getBytes(StandardCharsets.UTF_8));
this.keySpec = new SecretKeySpec(key, "AES");
// データベースに保存されたRSA鍵の暗号化/復号に使用
```

### 3. HMAC鍵（対称）

`SecurityUtils.hmacSha256Hex()`で[署名Cookie](signed-cookie.md)やWebhook[検証](verification.md)に使用：

```java
Mac mac = Mac.getInstance("HmacSHA256");
mac.init(new SecretKeySpec(secret.getBytes(UTF_8), "HmacSHA256"));
byte[] sig = mac.doFinal(payload.getBytes(UTF_8));
```

### 鍵の保存アーキテクチャ

```
  環境変数:
  JWT_KEY_ENCRYPTION_SECRET="my-secret-passphrase"
          │
          ▼  SHA-256ハッシュ
  AES-256鍵（メモリ上のみ）
          │
          ▼  暗号化/復号
  ┌─────────────────────────────────────────┐
  │  signing_keys テーブル                   │
  │  ┌───────────────────────────────────┐  │
  │  │ kid: "key-2026-04-01T12-00"      │  │
  │  │ public_pem: "v1:base64IV:base64" │  │  ← AES-256-GCM暗号化済み
  │  │ private_pem: "v1:base64IV:base64"│  │  ← AES-256-GCM暗号化済み
  │  │ status: "active"                 │  │
  │  └───────────────────────────────────┘  │
  └─────────────────────────────────────────┘
```

## よくある間違いと攻撃

### 間違い1: ソースコードに鍵をハードコード

ソースコード内の鍵はgit履歴、CIログ、開発者のマシンに残ります。環境変数やシークレットマネージャーを使いましょう。voltaは`JWT_KEY_ENCRYPTION_SECRET`を環境変数から読み取ります。

### 間違い2: 弱い鍵生成の使用

`Math.random()`や`new Random()`は予測可能な出力を生成します。常に`SecureRandom`（Java）や同等の[暗号的乱数](crypto-random.md)ソースを使いましょう。voltaはすべての鍵生成に`SecureRandom`を使用しています。

### 間違い3: 鍵をローテーションしない

永遠に使われる鍵はリスクが蓄積します。一度でも（たとえ短時間でも）漏洩していれば、過去と未来のすべてのデータが危険にさらされます。[鍵ローテーション](key-rotation.md)を参照。

### 間違い4: 秘密鍵を暗号化せずに保存

データベースが侵害された場合、暗号化されていない秘密鍵があると攻撃者がJWTを偽造できます。voltaは`KeyCipher`経由でAES-256-GCMですべての鍵を保存時に暗号化しています。

### 攻撃: メモリからの鍵抽出

サーバーアクセスを持つ攻撃者はプロセスメモリをダンプして鍵を抽出できます。緩和策：サーバーアクセスの制限、短寿命鍵の使用、異常なアクセスパターンの監視。
