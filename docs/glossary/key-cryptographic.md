# Cryptographic Key

[日本語版はこちら](key-cryptographic.ja.md)

---

## What is it?

A cryptographic key is a piece of secret data used to lock (encrypt), unlock (decrypt), or sign information. Without the correct key, encrypted data is unreadable and signatures cannot be created or verified.

Think of it like a physical key for a lock. Your house key lets you open your front door but nobody else's. A cryptographic key works the same way -- it is a specific sequence of bits that makes a cryptographic operation work. Change one bit, and the "lock" will not open. Lose the key, and you lose access to everything it protected.

There are two main families: **symmetric keys** (one key for both locking and unlocking, like a padlock key) and **asymmetric key pairs** (a public key and a private key that work together, like a mailbox where anyone can drop mail in through the slot but only the key holder can open it). volta uses both: AES-256 symmetric keys for [encryption at rest](encryption-at-rest.md) and RSA asymmetric keys for [JWT signing](digital-signature.md).

---

## Why does it matter?

Keys are the foundation of all cryptographic security. Every encrypted database field, every signed JWT, every HTTPS connection depends on keys. If a key is compromised:

- **Encryption breaks**: An attacker with the AES key can decrypt all signing keys stored in the database.
- **Signatures become forgeable**: An attacker with the RSA private key can issue fake JWTs that pass verification.
- **Trust collapses**: Downstream services cannot distinguish legitimate tokens from forged ones.

Keys must be generated securely (using [crypto-random](crypto-random.md) sources), stored safely (encrypted, never in logs), and rotated regularly ([key rotation](key-rotation.md)). A system is only as secure as its weakest key management practice.

## How does it work?

### Symmetric keys

One key, two operations:

```
  Symmetric key: K
  ┌──────────────────────────────────────────────┐
  │                                              │
  │  Encrypt:  plaintext + K → ciphertext        │
  │  Decrypt:  ciphertext + K → plaintext        │
  │                                              │
  │  Same key for both directions.               │
  │  Both parties must share K secretly.         │
  └──────────────────────────────────────────────┘

  Example: AES-256-GCM (used by volta's KeyCipher)
  ┌─────────────────────────────────────────┐
  │  Key:  32 bytes (256 bits) from SHA-256 │
  │  IV:   12 bytes (random per operation)  │
  │  Tag:  16 bytes (authentication tag)    │
  │                                         │
  │  "v1:" + base64(IV) + ":" + base64(     │
  │     encrypted_data + auth_tag)          │
  └─────────────────────────────────────────┘
```

### Asymmetric key pairs

Two keys, complementary operations:

```
  Private key: sk     Public key: pk
  ┌──────────────────────────────────────────────┐
  │                                              │
  │  Sign:    message + sk → signature           │
  │  Verify:  message + signature + pk → yes/no  │
  │                                              │
  │  Encrypt: plaintext + pk → ciphertext        │
  │  Decrypt: ciphertext + sk → plaintext        │
  │                                              │
  │  Private key is SECRET. Public key is SHARED.│
  └──────────────────────────────────────────────┘

  Example: RSA-2048 (used by volta's JwtService)
  ┌─────────────────────────────────────────┐
  │  Private key: 2048 bits, kept in DB     │
  │               (encrypted with AES)      │
  │  Public key:  Derived from private key  │
  │               Published at JWKS endpoint│
  └─────────────────────────────────────────┘
```

### Key sizes and strength

| Algorithm | Key size | Strength | Used for |
|-----------|----------|----------|----------|
| AES-256 | 256 bits | Very strong | Encrypting signing keys at rest |
| RSA-2048 | 2048 bits | Strong | Signing JWTs |
| HMAC-SHA256 | 256 bits | Very strong | Cookie signing, webhook verification |

### Key lifecycle

```
  Generate ──→ Store (encrypted) ──→ Use ──→ Rotate ──→ Retire
      │              │                │          │          │
  SecureRandom   AES-256-GCM     Sign JWTs   New key    Old key
  or KeyPairGen  in signing_keys  or encrypt  created    marked
                 table                        old key    inactive
                                              retired
```

## How does volta-auth-proxy use it?

volta manages three types of cryptographic keys:

### 1. RSA signing key pair (asymmetric)

Generated by `JwtService.createKey()` using Java's `KeyPairGenerator`:

```java
KeyPairGenerator generator = KeyPairGenerator.getInstance("RSA");
generator.initialize(2048);
KeyPair keyPair = generator.generateKeyPair();
// Public key: shared via /.well-known/jwks.json
// Private key: used to sign JWTs, never leaves the server
```

### 2. AES encryption key (symmetric)

Derived from `JWT_KEY_ENCRYPTION_SECRET` environment variable in `KeyCipher`:

```java
MessageDigest sha = MessageDigest.getInstance("SHA-256");
byte[] key = sha.digest(secret.getBytes(StandardCharsets.UTF_8));
this.keySpec = new SecretKeySpec(key, "AES");
// Used to encrypt/decrypt RSA keys stored in the database
```

### 3. HMAC key (symmetric)

Used by `SecurityUtils.hmacSha256Hex()` for [signed cookies](signed-cookie.md) and webhook [verification](verification.md):

```java
Mac mac = Mac.getInstance("HmacSHA256");
mac.init(new SecretKeySpec(secret.getBytes(UTF_8), "HmacSHA256"));
byte[] sig = mac.doFinal(payload.getBytes(UTF_8));
```

### Key storage architecture

```
  Environment variable:
  JWT_KEY_ENCRYPTION_SECRET="my-secret-passphrase"
          │
          ▼  SHA-256 hash
  AES-256 key (in memory only)
          │
          ▼  Encrypts/decrypts
  ┌─────────────────────────────────────────┐
  │  signing_keys table                     │
  │  ┌───────────────────────────────────┐  │
  │  │ kid: "key-2026-04-01T12-00"      │  │
  │  │ public_pem: "v1:base64IV:base64" │  │  ← AES-256-GCM encrypted
  │  │ private_pem: "v1:base64IV:base64"│  │  ← AES-256-GCM encrypted
  │  │ status: "active"                 │  │
  │  └───────────────────────────────────┘  │
  └─────────────────────────────────────────┘
```

## Common mistakes and attacks

### Mistake 1: Hard-coding keys in source code

Keys in source code end up in git history, CI logs, and developer machines. Use environment variables or secret managers. volta reads `JWT_KEY_ENCRYPTION_SECRET` from the environment.

### Mistake 2: Using weak key generation

`Math.random()` or `new Random()` produce predictable output. Always use `SecureRandom` (Java) or equivalent [crypto-random](crypto-random.md) sources. volta uses `SecureRandom` for all key generation.

### Mistake 3: Never rotating keys

A key used forever accumulates risk. If it was ever leaked (even briefly), all past and future data is compromised. See [key-rotation](key-rotation.md).

### Mistake 4: Storing private keys unencrypted

If the database is breached, unencrypted private keys let the attacker forge JWTs. volta encrypts all keys at rest with AES-256-GCM via `KeyCipher`.

### Attack: Key extraction from memory

An attacker with server access can dump process memory to extract keys. Mitigations: limit server access, use short-lived keys, monitor for unusual access patterns.
