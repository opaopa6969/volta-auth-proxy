# レイテンシ（Latency）

[English version](latency.md)

---

## これは何？

レイテンシとは、何かを要求してからレスポンスが返ってくるまでの時間 -- 待ち時間です。ソフトウェアではミリ秒（ms）で測定され、ユーザーがボタンをクリックしてから結果が表示されるまでの遅延です。

コーヒーを注文するようなものです。カウンターに歩いて行き、注文して、待ちます。「ラテを1つお願いします」と言ってからカップを受け取るまでの時間がレイテンシです。バリスタが30秒で作れば低レイテンシ。20人の列があって10分かかれば高レイテンシ。コーヒーの味はどちらも同じですが、体験はまったく異なります。

---

## 認証でミリ秒が重要な理由

認証はすべてのリクエストのクリティカルパスにあります。voltaのようなForwardAuthセットアップでは、保護されたアプリケーションへのすべてのリクエストがこのフローを通ります：

```
ユーザーがリンクをクリック
    → ブラウザがTraefikにリクエスト送信       ~1ms
    → TraefikがvoltaにForwardAuthチェック送信  ~1ms
    → voltaがセッション検証 + JWT発行          ~Xms  ← これがレイテンシ
    → Traefikがアプリにリクエスト転送          ~1ms
    → アプリが処理してレスポンス               ~Yms
    → ユーザーがページを見る

認証によって追加される合計：~2msネットワーク + Xms volta処理
```

voltaの処理が5msなら、ユーザーはほとんど気づきません。500msなら、すべてのページ読み込みが遅く感じます。そしてこれはログインだけでなくすべてのリクエストで発生するため、小さな増加でもユーザー体験全体にわたって乗算されます。

```
5msの認証レイテンシ × ページ読み込みあたり20リクエスト = 100ms追加
50msの認証レイテンシ × ページ読み込みあたり20リクエスト = 1,000ms（1秒！）追加
```

ページ読み込みあたり1秒のレイテンシがあると、ユーザーはアプリケーションから離れます。

## ネットワークホップがレイテンシを追加する仕組み

データがネットワーク境界を越える -- あるサーバーから別のサーバーに移動する -- たびに、レイテンシが追加されます。これを「ネットワークホップ」と呼びます。

```
マイクロサービス認証（多くのホップ）：
  Traefik → ForwardAuth Service → Session Service → JWT Service
            ~1ms                  ~1ms               ~1ms
  ネットワークオーバーヘッド合計：最低~3ms
  各サービスでの処理時間を加算
  各ホップでのシリアライゼーション/デシリアライゼーションを加算
  現実的な合計：15-50ms

volta（最小ホップ）：
  Traefik → volta-auth-proxy（すべて内部で処理）
            ~1ms
  ネットワークオーバーヘッド合計：~1ms
  処理時間を加算（1プロセス、シリアライゼーションなし）
  現実的な合計：3-10ms
```

この差は重要です。3ms vs 50msは、voltaがForwardAuthチェックをユーザーが認証レイヤーを遅いと感じないほど高速に処理できることを意味します。

## voltaの10ms ForwardAuthターゲット

voltaはForwardAuth検証で10ms未満のレスポンスタイムを目標としています。その方法：

1. **シングルプロセス。** 内部コンポーネント間のネットワークホップなし。セッション検索、ロールチェック、JWT発行はHTTP呼び出しではなくメソッド呼び出し。

2. **インメモリキャッシュ。** 頻繁にアクセスされるセッションはCaffeine（Javaのインメモリキャッシュ）にキャッシュでき、ホットなセッションのデータベースラウンドトリップを回避。

3. **リクエストごとの最小作業。** ForwardAuth検証はセッションクッキーを読み、検証し、ヘッダーを返します。リクエストボディの解析、複雑なポリシーの解決、外部サービスの呼び出しは行いません。

4. **事前計算されたJWT。** JWTは事前にロードされたRSA鍵で署名されます。リクエスト処理中の鍵取得や外部JWKS呼び出しはありません。

```
ForwardAuth /auth/verify の内訳：
  ヘッダーからクッキー読み取り：         ~0.01ms
  セッション検索（キャッシュヒット）：    ~0.1ms
  セッション検索（DB、ローカルPostgres）：~2-5ms
  ロール認可チェック：                    ~0.01ms
  JWT作成 + RS256署名：                   ~1-2ms
  ヘッダー組み立て + レスポンス：          ~0.01ms
  ─────────────────────────────────────────
  合計（キャッシュヒット）：              ~1-3ms
  合計（DBヒット）：                      ~3-8ms
```

どちらも10ms以下です。比較すると、すべてのリクエストで外部IdPを呼び出すOAuth2プロキシは簡単に100-500msかかることがあります。

## レイテンシ vs スループット

レイテンシとスループットは関連していますが異なるものです：

- **レイテンシ：** 1つのリクエストにかかる時間（リクエストあたりの時間）
- **スループット：** システムが1秒あたりに処理するリクエスト数

低レイテンシだが低スループット（高速だが一度に1つのリクエストしか処理できないシステム）もありえます。高スループットだが高レイテンシ（毎秒数千リクエストを処理するが、各リクエストに500msかかるシステム）もありえます。

voltaは両方を最適化します：低レイテンシ（リクエストあたり10ms未満）と適切なスループット（JVMが単一インスタンスで並行リクエストを効率的に処理）。

---

## volta-auth-proxyでは

voltaは内部ネットワークホップのないシングルプロセスとして動作し、セッション検索にインメモリキャッシュを使用し、セッションの検証とJWTの発行に必要な最小限の作業を行うことで、10ms未満のForwardAuthレスポンスタイムを目標としています。
