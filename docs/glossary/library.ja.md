# ライブラリ

[English version](library.md)

---

## これは何？

ライブラリとは、特定の問題を解決する再利用可能なコードパッケージです。プロジェクトにインポートし、必要な時に関数を呼び出すと、結果が返ってきます。あなたのコードが制御権を持ち続けます -- ライブラリは必要な時に手に取って使う道具であって、中に住む構造物ではありません。

電動工具セットに似ています。本棚を作るとき、プロジェクトの主導権はあなたにあります。何をいつ作るかはあなたが決めます。穴を開ける必要があるとき、ドリルを手に取り、使い、戻します。ドリルが何を作るか、いつ作るかを指示することはありません。それがライブラリです -- 必要な時に手を伸ばして取る道具。

これは[フレームワーク](framework.md)との根本的な違いです。フレームワークはあなたの作業を整理し、何をどこに置くかを指示する工房です。ライブラリは、あなたが手に取るまで棚に置かれている道具です。技術用語で言えば：ライブラリでは**あなた**が流れを制御します。フレームワークでは**フレームワーク**が流れを制御します。

---

## なぜ重要なのか？

ライブラリはすべての現代アプリケーションの構成要素です。ライブラリがなければ：

- プロジェクトごとにJSONパースを書き直すことになる
- 自分で暗号アルゴリズムを実装することになる（最悪のアイデア）
- 自分でデータベース接続プーリングを書くことになる
- 生のソケットから自分でHTTPクライアントを構築することになる
- 開発時間が桁違いに増える

ライブラリは専門家の肩の上に立たせてくれます。[HikariCP](hikaricp.md)を書いた人たちはデータベース接続プーリングの最適化に何年も費やしました。nimbus-jose-jwtを書いた人たちは攻撃に対するJWTパースの堅牢化に何年も費やしました。依存関係を追加するだけで、その専門知識を無料で手に入れられます。

## どう動くのか？

### ライブラリの使い方

[Java](java.md)では、ライブラリはJARファイル（Java Archive -- コンパイル済みクラスのZIPファイル）として配布されます。[Maven](maven.md)のpom.xmlで依存関係として宣言します：

```xml
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
    <version>5.1.0</version>
</dependency>
```

Mavenが[リポジトリ](repository.md)からJARをダウンロードし、クラスパスに追加し、使えるようになります：

```java
// あなたのコードがライブラリを呼ぶ
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:postgresql://localhost:5432/volta");
HikariDataSource ds = new HikariDataSource(config);

// ライブラリが結果を返す -- それをどうするかはあなたが決める
Connection conn = ds.getConnection();
```

注目：**あなたのコード**が制御しています。プールをいつ作るか、接続をいつ取得するか、それをどうするかを決めるのはあなたです。

### 依存関係ツリー

ライブラリはしばしば他のライブラリに依存します。これがツリーを作ります：

```
  volta-auth-proxy
  ├── javalin（フレームワークだが、依存関係でもある）
  │   ├── jetty-server
  │   ├── jetty-webapp
  │   └── slf4j-api
  ├── hikaricp（ライブラリ）
  │   └── slf4j-api
  ├── nimbus-jose-jwt（ライブラリ）
  │   └── jcip-annotations
  ├── caffeine（ライブラリ）
  │   └── checker-qual
  ├── postgresql（ライブラリ - JDBCドライバ）
  └── jte（ライブラリ - テンプレートエンジン）
```

Mavenはこのツリーを自動的に解決し、推移的依存関係をダウンロードします。直接の依存関係が10個のプロジェクトでも、最終的な[fat JAR](fat-jar.md)には50以上のJARが含まれることがあるのはこのためです。

### ライブラリ vs フレームワーク vs SDK

| 概念 | 制御 | 例 | 関係 |
|------|------|---|------|
| ライブラリ | あなたが呼ぶ | HikariCP, Caffeine, nimbus-jose-jwt | 棚の上の道具 |
| [フレームワーク](framework.md) | それがあなたを呼ぶ | Javalin, Spring Boot | 工房の構造 |
| [SDK](sdk.md) | 混合 | volta-sdk-js | ライブラリ + プラットフォーム用ツールの集合 |

### ライブラリの種類

| 種類 | voltaでの例 | 目的 |
|------|------------|------|
| データベースドライバ | postgresql JDBC | [Postgres](database.md)との通信 |
| 接続プール | [HikariCP](hikaricp.md) | 再利用可能なDB接続の管理 |
| キャッシュ | [Caffeine](caffeine-cache.md) | インメモリキャッシュ |
| 暗号/JWT | nimbus-jose-jwt | JWTの作成と検証 |
| テンプレートエンジン | [jte](jte.md) | データを使ったHTMLレンダリング |
| マイグレーション | [Flyway](flyway.md) | データベーススキーマのバージョニング |
| HTTPクライアント | java.net.http（組み込み） | 外部API呼び出し（Google OIDC） |
| ロギング | SLF4J + Logback | 構造化ログ出力 |

## volta-auth-proxy ではどう使われている？

volta-auth-proxyは、すべてをバンドルした大きなフレームワークの代わりに、小さくて焦点の絞られたライブラリを意図的に選んでいます。各ライブラリは1つの問題をうまく解決します。

### voltaのライブラリスタック

```
  ┌─────────────────────────────────────────────┐
  │              volta-auth-proxy                │
  │                                              │
  │  ┌──────────┐  ┌──────────┐  ┌───────────┐  │
  │  │ HikariCP │  │ Caffeine │  │ nimbus-   │  │
  │  │ (DBプール)│  │(キャッシュ)│  │ jose-jwt  │  │
  │  └──────────┘  └──────────┘  └───────────┘  │
  │                                              │
  │  ┌──────────┐  ┌──────────┐  ┌───────────┐  │
  │  │ Flyway   │  │  jte     │  │ PostgreSQL│  │
  │  │(マイグレ) │  │(テンプレ) │  │ (ドライバ) │  │
  │  └──────────┘  └──────────┘  └───────────┘  │
  │                                              │
  │  ┌──────────────────────────────────────┐    │
  │  │         Javalin（フレームワーク）      │    │
  │  │         Jetty（HTTPサーバー）          │    │
  │  └──────────────────────────────────────┘    │
  └─────────────────────────────────────────────┘
```

### なぜSpring Bootの代わりに小さなライブラリなのか？

Spring Bootはデータベースアクセス（Spring Data JPA）、キャッシュ（Spring Cache）、セキュリティ（Spring Security）、テンプレートなどを1つのパッケージにバンドルした[フレームワーク](framework.md)です。voltaは正反対のアプローチを取ります：

- Spring Dataの接続管理の代わりに**HikariCP** -- 明示的なプール設定、魔法の`@Transactional`アノテーションなし
- Spring Cacheの代わりに**Caffeine** -- `cache.get(key, loader)`のような直接的なAPI呼び出し、`@Cacheable`アノテーションなし
- Spring SecurityのJWTサポートの代わりに**nimbus-jose-jwt** -- アルゴリズムチェックとクレーム検証の完全な制御
- Hibernate auto-DDLの代わりに**Flyway** -- 明示的な[SQL](sql.md)マイグレーション、ORMによるスキーマ生成なし

各ライブラリは独立しています。1つを置き換える必要があっても、他には影響しません。すべてが相互接続されているフレームワークの逆です。

### pom.xmlでの依存関係管理

すべてのライブラリはプロジェクトのpom.xmlに明示的なバージョンで宣言されています。voltaはバージョン管理にMaven BOM（Bill of Materials）や親POMの継承を使わず、依存関係リストを自己完結的で監査可能に保っています。

## よくある間違いと攻撃

### 間違い1：理解せずにライブラリを使う

JWTを処理するライブラリを追加しても、JWTが安全になるわけではありません。どのアルゴリズムを受け入れるか、どのクレームを検証するか、期限切れをどう処理するかを理解する必要があります。voltaのJwtService.javaはnimbus-jose-jwtを明示的なRS256強制でラップしています -- ライブラリのデフォルトを盲目的に信頼していません。

### 間違い2：単純なタスクにライブラリを使いすぎる

ライブラリを追加するたびに依存関係ツリーが大きくなり、[fat JAR](fat-jar.md)のサイズと攻撃サーフェスが増加します。ランダムな文字列を生成する必要があるなら、Apache Commons Langを追加する代わりに`java.util.UUID.randomUUID()`を使いましょう。voltaはJava標準ライブラリの機能で十分な場合はそれを使います。

### 攻撃1：サプライチェーン攻撃

悪意のある行為者が人気のあるライブラリに似た名前でライブラリを公開したり（タイポスクワッティング）、既存のライブラリのビルドパイプラインを侵害したりします。その依存関係を追加すると、彼らのコードが実行されます。voltaが明示的なバージョンを固定し、npmではなく[Maven](maven.md)（チェックサムを使用する）を使う理由です。

### 間違い3：ライブラリを更新しない

ライブラリにはセキュリティパッチが提供されます。古いバージョンのnimbus-jose-jwtを使い続けると、数ヶ月前に修正されたJWTパース攻撃に対して脆弱になるかもしれません。ただし、盲目的な更新も危険です -- 必ずchangelogを読みましょう。

### 攻撃2：推移的依存関係の悪用

直接の依存関係が安全でも、それらの依存関係は安全でないかもしれません。ロギングライブラリの脆弱性（Log4jのLog4Shellなど）は、間接的にであっても、それに依存するすべてのアプリケーションに影響を与えます。`mvn dependency:tree`を使って完全な依存関係ツリーを監査しましょう。

### 間違い4：ライブラリとフレームワークを混同する

ライブラリをあまりに多くの抽象化でラップして、コードの流れを制御し始めたら、うっかり[フレームワーク](framework.md)を作ってしまっています。ライブラリの使用はシンプルで直接的に保ちましょう。
