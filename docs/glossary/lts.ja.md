# LTS（長期サポート）

[English version](lts.md)

---

## これは何？

LTSはLong Term Support（長期サポート）の略です。メンテナーがセキュリティパッチとバグ修正を長期間 -- 通常は数年間 -- サポートすることを約束するソフトウェアリリースを意味します。非LTSリリースは数ヶ月間だけサポートされ、その後次のバージョンにアップグレードしなければなりません。

車の購入で考えてみましょう。通常モデル（非LTS）はメーカーサポートが2年で、その後は新車を買うことが期待されます。LTSモデルは5年以上のサポート -- オイル交換、保証修理、安全リコール。メーカーに忘れられる心配なく何年も乗れます。認証を扱うソフトウェア（セキュリティパッチが重要）では、LTSの車が欲しいはずです。

volta-auth-proxyは**Java 21 LTS**を使用しており、少なくとも2029年9月までセキュリティアップデートを受け取ります。これにより、6ヶ月ごとにJavaをアップグレードするプレッシャーなく、安定した安全な基盤をvoltaに提供します。

---

## なぜ重要なのか？

- **何年もセキュリティパッチが来る。** 認証コードは脆弱性に対してパッチされ続ける必要がある。サポートされていないJavaバージョンはパッチなし。
- **安定性。** LTSリリースは何千もの組織で実戦検証済み。非LTSバージョンは使用者が少なく、実世界でのテストも少ない。
- **依存関係の互換性。** Javalin、HikariCP、FlywayなどのライブラリはLTSバージョンを優先してテストする。非LTSの互換性は多くの場合「ベストエフォート」。
- **エンタープライズ要件。** 規制産業の顧客はコンプライアンス上の理由でLTSソフトウェアを要求することが多い。
- **アップグレードの混乱を軽減。** 6ヶ月ごとにJavaをアップグレード（非LTSサイクル）するのは破壊的。LTSなら代わりに2-3年ごと。

## どう動くのか？

### Javaのリリーススケジュール

Oracleは6ヶ月ごとに新しいJavaバージョンをリリースします。4回に1回（2年ごと）のリリースのみがLTSに指定されます：

```
  Javaリリースタイムライン:
  ──────────────────────────────────────────────────────▶ 時間

  2021    2022    2023    2024    2025    2026    2027
   │       │       │       │       │       │       │
   17      18      19      20      21      22      23
   LTS     │       │       │       LTS     │       │
   │       │       │       │       │       │       │
   │      6ヶ月   6ヶ月   6ヶ月    │      6ヶ月   6ヶ月
   │     サポート サポート サポート  │     サポート サポート
   │                               │
   └── サポート期限                 └── サポート期限
       2029年9月                       2031年9月（推定）

  非LTS（18, 19, 20, 22, 23）:
  ┌──────────────────┐
  │ 6ヶ月のサポート   │ → その後アップグレード必須
  └──────────────────┘

  LTS（17, 21, 25）:
  ┌──────────────────────────────────────────────┐
  │ 5年以上のセキュリティパッチ                     │
  └──────────────────────────────────────────────┘
```

### LTSに含まれるもの

| 得られるもの | LTS | 非LTS |
|-------------|-----|-------|
| **セキュリティパッチ** | 数年間 | 6ヶ月 |
| **バグ修正** | 数年間 | 6ヶ月 |
| **パフォーマンス更新** | あり | あり |
| **コミュニティテスト** | 広範 | 限定的 |
| **ライブラリサポート** | 保証 | ベストエフォート |
| **Dockerイメージ** | 常にメンテナンス | すぐに放棄される |
| **本番準備** | 初日から | 安定化後 |

### より広いエコシステムでのLTS

LTSはJavaだけの概念ではありません。多くのプロジェクトが同じパターンに従います：

| ソフトウェア | LTSバージョン | サポート期間 |
|------------|-------------|-------------|
| **Java** | 17, 21, 25 | 5年以上 |
| **Node.js** | 18, 20, 22 | 30ヶ月 |
| **Ubuntu** | 22.04, 24.04 | 5年 |
| **PostgreSQL** | 各メジャーバージョン | 5年 |
| **.NET** | 6, 8 | 3年 |

### LTSアップグレードパス

```
  voltaをJava 17 LTSで使用（2029年9月までサポート）
       │
       │  Java 25 LTSリリース時（2025年9月）:
       │  - voltaをJava 25でテスト
       │  - 依存関係の動作を検証
       │  - 自分のペースで移行
       │  - Java 21はまだサポート中
       │
       ▼
  voltaをJava 21 LTSで使用 ← 現在
       │
       │  Java 29 LTSリリース時（2027年頃）:
       │  - 同じプロセス
       │
       ▼
  voltaをJava 25 LTSで使用（将来）
```

## volta-auth-proxy ではどう使われている？

### voltaの基盤としてのJava 21 LTS

volta-auth-proxyはJava 21を必要とします。これは2023年9月にLTSバージョンとしてリリースされました。voltaに提供されるもの：

```
  ┌────────────────────────────────────────────────┐
  │  volta向けJava 21 LTSの利点                     │
  │                                                 │
  │  セキュリティパッチ:   2029年9月以降まで          │
  │  仮想スレッド:         より良い並行処理           │
  │  パターンマッチング:   よりクリーンなコード        │
  │  レコードクラス:       ボイラープレート削減        │
  │  テキストブロック:     読みやすいSQL/テンプレート   │
  │  シールドクラス:       型安全性                   │
  │  ライブラリサポート:   全主要ライブラリがテスト済み │
  │  Dockerイメージ:       eclipse-temurin:21-jre    │
  └────────────────────────────────────────────────┘
```

### pom.xml設定

```xml
<properties>
    <java.version>21</java.version>
    <maven.compiler.source>21</maven.compiler.source>
    <maven.compiler.target>21</maven.compiler.target>
</properties>
```

### なぜJava 22、23、24ではないのか？

| バージョン | タイプ | サポート終了 | なぜ使わないか |
|----------|--------|------------|--------------|
| Java 22 | 非LTS | 2025年3月 | すでにサポート終了 |
| Java 23 | 非LTS | 2025年9月 | パッチは6ヶ月だけ |
| Java 24 | 非LTS | 2026年3月 | パッチは6ヶ月だけ |
| **Java 21** | **LTS** | **2029年9月以降** | **安定、長期パッチ** |

セキュリティ上重要な操作を扱う認証プロキシでは、安定性と長期セキュリティパッチが最新の言語機能へのアクセスより重要です。

### LTSとMaven依存関係の安定性

LTS Javaバージョンを使うことは、より安定した[依存関係](dependencies.ja.md)も意味します：

```
  Javalin 6.x    → Java 21 LTSでテスト済み ✓
  HikariCP 5.x   → Java 21 LTSでテスト済み ✓
  Flyway 10.x    → Java 21 LTSでテスト済み ✓
  Caffeine 3.x   → Java 21 LTSでテスト済み ✓

  これらのライブラリはLTS互換性を優先する。
  非LTS Javaバージョンにはテストされていないエッジケースがある可能性。
```

## よくある間違いと攻撃

### 間違い1：本番で非LTS Javaを使用

本番でJava 23を実行すると、セキュリティパッチは6ヶ月だけです。その後、発見された脆弱性（JVMエスケープや暗号の欠陥など）はあなたのバージョンではパッチされません。

### 間違い2：LTSバージョンを決してアップグレードしない

LTSは「永遠に使える」という意味ではありません。Java 11 LTSはサポート終了に近づいています。LTSアップグレードを計画しましょう -- 古いバージョンがサポートされなくなるまで待たないでください。

### 間違い3：環境間でJavaバージョンが異なる

開発でJava 24、本番でJava 21だと、ローカルでコンパイルできるコードが本番にない機能を使っているかもしれません。すべての環境で同じLTSバージョンを固定しましょう。

### 間違い4：LTSを「更新不要」と混同

LTSでも四半期ごとの更新（例：21.0.1、21.0.2、21.0.3）がリリースされます。これらにはセキュリティパッチが含まれ、適用が必要です。LTSはメジャーバージョンが安定しているという意味で、決して更新しないという意味ではありません。

### 間違い5：間違った理由でLTSを選ぶ

ユーザーのいない個人プロジェクトを作るなら、最新の非LTSバージョンで問題ありません。LTSが重要なのは、実際のユーザーとセキュリティ要件がある本番ソフトウェアです。
