# M2M（マシン間通信）

[English version](m2m.md)

---

## これは何？

M2M（Machine-to-Machine）通信は、2つのソフトウェアシステムが人間の関与なしに通信することです。ブラウザなし、ログインフォームなし、「許可する」クリックなし。あるサーバーが別のサーバーにリクエストを送り、事前共有された認証情報で認証されます。

工場の組立ラインにいる2台のロボットを想像してください。ロボットAがドアの塗装を終えると、ロボットBに取り付け開始のシグナルを送ります。間に人間の監督者はいません。お互いが理解するプロトコルで直接連携します。インターネット上のM2M通信も同様です。サービスが認証情報で認証し、自動的にデータを交換します。

OAuth 2.0の世界では、M2M通信は[Client Credentials](client-credentials.md)グラントを使います。呼び出し側サービスが`client_id`と`client_secret`を提示し、短命の[JWT](jwt.md)を受け取り、そのトークンでAPIにアクセスします。

---

## なぜ重要なのか？

現代のSaaSアプリケーションはモノリスではなく、多くのサービスで構成されています：

```
  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
  │  課金         │    │  通知         │    │  分析         │
  │  サービス     │    │  サービス     │    │  サービス     │
  └──────┬───────┘    └──────┬───────┘    └──────┬───────┘
         │                   │                   │
         ▼                   ▼                   ▼
  ┌─────────────────────────────────────────────────────┐
  │              volta-auth-proxy                        │
  │         （すべてのM2Mコールを認証）                    │
  └─────────────────────────────────────────────────────┘
         │                   │                   │
         ▼                   ▼                   ▼
  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
  │  ユーザー     │    │  テナント     │    │  ロール       │
  │  API         │    │  API         │    │  API         │
  └──────────────┘    └──────────────┘    └──────────────┘
```

M2M認証がないと：

- ネットワーク上のどのプロセスでも[内部API](internal-api.md)を呼べてしまう
- ログで課金サービスと分析サービスを区別できない
- 権限をスコープする方法がない。すべてのサービスがフルアクセス
- 認証情報のローテーションが悪夢

適切なM2M認証があれば、各サービスが独自のアイデンティティ、スコープされた権限、監査可能なアクセスを持ちます。

## どう動くのか？

### M2M認証フロー

```
  課金サービス                          volta-auth-proxy
  ============                          ================

  1. テナントデータを読みたい。
     ユーザーコンテキストがない。私はマシン。

  2. POST /oauth/token
     grant_type=client_credentials
     &client_id=billing-service
     &client_secret=my-secret-key
     &scope=read:tenants read:billing

  ─────────────────────────────────────────────────►

                                        3. 認証情報を検証：
                                           - client_id登録済み？ ✓
                                           - client_secret一致？ ✓
                                           - スコープ許可？ ✓

                                        4. M2M JWTを発行：
                                           {
                                             "sub": "billing-svc-uuid",
                                             "volta_client": true,
                                             "volta_client_id":
                                               "billing-service",
                                             "volta_tid": "acme-uuid",
                                             "volta_roles": [
                                               "read:tenants",
                                               "read:billing"
                                             ],
                                             "exp": +5分
                                           }

  ◄─────────────────────────────────────────────────

  5. トークンを使ってAPIを呼ぶ：
     GET /api/v1/tenants/acme-uuid
     Authorization: Bearer eyJhbGci...

  ─────────────────────────────────────────────────►

                                        6. JWTを検証：
                                           - volta_client=true → M2M
                                           - スコープにread:tenants含む
                                           - テナントデータを返す

  ◄─────────────────────────────────────────────────
```

### M2M vs 人間の認証

| 側面 | 人間（Authorization Code） | マシン（Client Credentials） |
|------|--------------------------|------------------------------|
| 認証する主体 | ブラウザ経由の人 | APIコール経由のサービス |
| 認証情報 | ユーザー名+パスワード（+MFA） | client_id + client_secret |
| [セッション](session.md) | あり、[Cookie](cookie.md)付き | なし、短命トークンのみ |
| 同意画面 | あり（「このアプリに許可...」） | なし（事前設定されたスコープ） |
| トークン有効期間 | 15分〜1時間 | 5分（voltaデフォルト） |
| リフレッシュトークン | あり | なし（再認証するだけ） |

### トークンキャッシュ

M2Mトークンは短命ですが、APIコールのたびに新しいトークンをリクエストするのは無駄です。サービスはトークンをキャッシュし、期限切れ時のみ再認証すべきです：

```
  ┌─────────────────────────────────────────────┐
  │  トークンキャッシュ戦略                       │
  │                                              │
  │  1. キャッシュで有効なトークンを確認            │
  │  2. トークンがあり未期限切れ → 使用            │
  │  3. 期限切れまたは未取得：                      │
  │     a. POST /oauth/token                     │
  │     b. 新しいトークンをキャッシュ               │
  │     c. キャッシュTTL = expires_in - 30秒       │
  │  4. キャッシュしたトークンを使用                │
  └─────────────────────────────────────────────┘
```

`-30秒`のバッファにより、トークンが実際に期限切れになる前にリフレッシュされ、レースコンディションを回避します。

## volta-auth-proxyではどう使われている？

### Phase 1：静的サービストークン（現在）

voltaは現在シンプルなM2Mメカニズムをサポートしています：

```
  Authorization: Bearer volta-service:<VOLTA_SERVICE_TOKEN>
```

汎用サービスプリンシパルとして認証されます。動作しますが制限があります：全サービスで1つのトークン、スコープなし、サービスごとの監査証跡なし。

### Phase 2：完全なClient Credentials（計画）

voltaが`oauth_clients`テーブルと`/oauth/token`エンドポイントを追加予定：

```
  クライアント登録：
    POST /api/v1/tenants/{tid}/oauth-clients
    {
      "name": "billing-service",
      "scopes": ["read:tenants", "read:billing"]
    }

  認証：
    POST /oauth/token
    grant_type=client_credentials
    &client_id=<返されたclient-id>
    &client_secret=<返されたsecret>
```

各M2Mクライアントは[テナント](tenant.md)にスコープされ、許可されたスコープのみリクエストでき、監査ログで固有のアイデンティティを取得します。

### [Webhook](webhook.md)でのM2Mトークン

voltaの[Outboxワーカー](outbox-pattern.md)が外部サービスにWebhookを配信する際、受信側がBearer認証を要求する場合はM2M認証を使うことがあります。ワーカーが自身を認証し、短命トークンを取得して、Webhookリクエストに含めます。

### アップストリームアプリがM2Mコールを検出する方法

voltaがM2Mリクエストを特別な[ヘッダー](header.md)付きで[アップストリーム](upstream.md)アプリに転送します：

```
  X-Volta-User-Id:    00000000-0000-0000-0000-000000000000
  X-Volta-Email:      m2m@billing-service
  X-Volta-Roles:      read:tenants,read:billing
  X-Volta-M2M:        true
  X-Volta-Client-Id:  billing-service
```

アプリは`X-Volta-M2M: true`をチェックして、マシンコールに異なるロジックを適用できます（例：レート制限をスキップ、レスポンス形式を調整）。

## よくある間違いと攻撃

### 間違い1：ユーザー操作にM2Mトークンを使う

M2Mトークンにはユーザーアイデンティティがありません。レコード作成に使うと、人間への監査証跡がなくなります。ユーザーが開始したアクションには常にユーザートークンを使ってください。

### 間違い2：複数のサービスで1つのクライアントを共有する

すべてのサービスが同じ`client_id`を共有すると、サービスごとのスコーピングと監査が失われます。各サービスに別のクライアントを登録してください。

### 間違い3：クライアントシークレットをローテーションしない

クライアントシークレットは定期的にローテーションすべきです。voltaはデュアルシークレットローテーションをサポート：新しいシークレットを登録、サービスを更新、古いシークレットを取り消し。

### 間違い4：トークンを永久にキャッシュする

M2Mトークンには有効期限があります。キャッシュが期限を無視すると、古いトークンを使って401エラーが発生します。常に`expires_in`を尊重してください。

### 攻撃：client_secretの窃取

攻撃者がサービスのclient_secretを取得すると、そのサービスになりすませる。防御：短命トークン（5分）、サービスごとのスコーピング、M2M使用パターンの異常検知。

### 攻撃：スコープ操作による権限昇格

攻撃者がscopeパラメータを変更して許可以上の権限をリクエストする。防御：voltaがリクエストされたスコープをクライアントの登録スコープに対して検証し、不正なリクエストを拒否。
