# MFA / 2FA（多要素認証）

[English version](mfa.md)

---

## これは何？

多要素認証（MFA）とは、身元を証明するために複数の種類の証拠を要求することです。パスワードだけでなく、スマホのコードや指紋スキャンなど、もう1つ別のものも必要になります。二要素認証（2FA）は、要素がちょうど2つのMFAです。

銀行の金庫室のようなものです。2人の異なる人が持つ2つの鍵が必要です。泥棒が1つの鍵を盗んでも、もう1つがなければ金庫は開けられません。MFAも同じです。パスワードが盗まれても、攻撃者はスマホや指紋がまだ必要です。

---

## なぜ重要なのか？

パスワードだけでは不十分です。パスワードは：

- データ漏洩で**盗まれる**（Verizon DBIRによると、侵害の83%が認証情報の窃取を含む）
- 偽のログインページで**フィッシングされる**
- ブルートフォースや一般的なパスワードリストで**推測される**
- 複数のサイトで**使い回される**（1つが漏洩すると全部）

MFAはパスワード侵害の影響を劇的に減少させます。攻撃者がパスワードを持っていても、第二要素がまだ必要です。第二要素は通常、攻撃者が持っていない物理的なものです。

---

## どう動くのか？

### 3つの要素

認証要素はその性質で分類されます：

```
  ┌─────────────────────────────────────────────────────────────┐
  │                    認証の要素                                  │
  │                                                              │
  │  1. 知っているもの               2. 持っているもの              │
  │  ┌──────────────────────┐     ┌──────────────────────┐      │
  │  │ - パスワード          │     │ - スマホ（TOTPアプリ） │      │
  │  │ - PIN                │     │ - セキュリティキー     │      │
  │  │ - 秘密の質問         │     │ - スマートカード       │      │
  │  │ - パターン           │     │ - ハードウェアトークン  │      │
  │  └──────────────────────┘     └──────────────────────┘      │
  │                                                              │
  │  3. 自分自身であるもの                                         │
  │  ┌──────────────────────┐                                   │
  │  │ - 指紋               │                                   │
  │  │ - 顔認証             │                                   │
  │  │ - 虹彩スキャン        │                                   │
  │  │ - 声紋               │                                   │
  │  └──────────────────────┘                                   │
  │                                                              │
  │  真のMFAは異なるカテゴリの要素を要求する。                       │
  │  パスワード + 秘密の質問 = 1要素のまま（両方「知っている」）      │
  │  パスワード + TOTPコード = 2要素（「知っている」+「持っている」）  │
  └─────────────────────────────────────────────────────────────┘
```

### 一般的なMFA方式

| 方式 | 要素タイプ | 仕組み | セキュリティレベル |
|------|----------|--------|-----------------|
| **TOTP**（Google Authenticator） | 持っている | アプリからの時間ベースワンタイムパスワード | 良い。[totp.md](totp.md)参照 |
| **SMSコード** | 持っている | テキストメッセージで送られるコード | 弱い。SIMスワッピング攻撃が一般的 |
| **メールコード** | 持っている | メールで送られるコード | 弱い。メールアカウントはしばしば侵害される |
| **WebAuthn/パスキー** | 持っている + 自分自身 | セキュリティキーまたは生体認証 | 優秀。[webauthn.md](webauthn.md)参照 |
| **プッシュ通知** | 持っている | スマホで承認 | 良い。ただし「疲労攻撃」に脆弱 |

### MFAフローの例

```
  ユーザー                       volta-auth-proxy             Google
  ========                       ================             ======

  1. 「Googleでログイン」をクリック
  ─────────────────────────────►
                                  Google OIDCにリダイレクト
                                ──────────────────────────────►
  2. Googleで認証                                               検証
  ─────────────────────────────────────────────────────────────►
                                ◄──────────────────────────────
                                  id_tokenを受信

                                3. チェック：このテナントでMFAが必要か？
                                   （管理者設定：「全メンバーにMFA必須」）

                                   ├── いいえ：通常通りログイン完了
                                   └── はい：ステップ4へ

  ◄─────────────────────────────
  4. MFAチャレンジページを表示
     「6桁のコードを入力してください」

  5. TOTPコードを入力: 847293
  ─────────────────────────────►
                                6. TOTPコードを検証
                                   ├── 無効：エラー表示、再試行
                                   └── 有効：ログイン完了

                                7. セッションCookieを設定
                                8. セッションに認証方法を記録：
                                   amr = ["google_oidc", "totp"]

  ◄─────────────────────────────
  9. アプリにリダイレクト
```

## volta-auth-proxyではどう使われているか？

### 現在の状態（Phase 1）

Phase 1では、voltaはMFAを**実装していません**。認証はGoogle OIDCに依存しており、Googleには独自のMFA（Googleの2段階認証）があります。つまり：

- ユーザーがGoogleの2FAを有効にしていれば、Google経由でMFAを得る
- voltaはGoogleの認証の上に追加の層を加えない
- voltaはテナントごとのMFA要件を強制しない

### Phase 2-3の計画

voltaは段階的にMFAを追加する予定です：

**Phase 2：パスキー（WebAuthn/FIDO2）**
- パスキーをオプションの第二認証方法として追加
- ユーザーはセキュリティキーを登録するか生体認証を使用可能
- 詳細は[webauthn.md](webauthn.md)を参照

**Phase 3：TOTPとMFAの強制**
- TOTP（Google Authenticator、Authyなど）を追加
- テナント管理者が「全メンバーにMFA必須」を強制可能
- リスクベース認証（新しいデバイス/IPでの追加検証）
- `amr`（Authentication Methods References）JWTクレームが認証の強度を反映

```json
// 将来のvolta JWTとMFAクレーム
{
  "iss": "volta-auth",
  "sub": "user-uuid",
  "volta_v": 1,
  "volta_tid": "tenant-uuid",
  "volta_roles": ["ADMIN"],
  "amr": ["google_oidc", "totp"],
  "auth_time": 1711899700
}
```

### MFAのアーキテクチャ

```
  ┌──────────────────────────────────────────────┐
  │               volta-auth-proxy                │
  │                                               │
  │  Google OIDC ──► 一次認証                      │
  │       │                                       │
  │       ▼                                       │
  │  MFA必須？ ──► テナントポリシーをチェック         │
  │       │                                       │
  │       ├── TOTPチャレンジ（Phase 3）             │
  │       │   └── 共有秘密に対して検証              │
  │       │                                       │
  │       ├── WebAuthnチャレンジ（Phase 2）         │
  │       │   └── 登録済み鍵に対して検証            │
  │       │                                       │
  │       └── MFAなし ──► ログイン完了              │
  │                                               │
  │  JWT: amrクレームがどの方法を使ったか記録        │
  │  アプリは機密操作でamrをチェック可能              │
  └──────────────────────────────────────────────┘
```

## よくある間違いと攻撃

### 間違い1：SMSを唯一の第二要素にする

SMSコードはSIMスワッピング（攻撃者が電話会社を説得して番号を攻撃者のSIMに移す）で傍受可能。TOTPまたはWebAuthnを使ってください。

### 間違い2：MFAをいつまでもオプションにする

MFAが常にオプションなら、ユーザーは有効にしません。テナント管理者が全メンバーにMFAを強制できるべきです。

### 間違い3：リカバリコードを提供しない

ユーザーがスマホを紛失するとTOTPコードにアクセスできなくなります。MFA設定時に必ずワンタイムリカバリコードを提供してください。

### 攻撃：MFA疲労（プッシュ通知スパム）

パスワードを盗んだ攻撃者がプッシュ通知MFAを繰り返しトリガー。疲れたユーザーが誤って1つを承認してしまう。対策：MFA試行を回数制限、番号マッチング（画面に表示されるコードをユーザーが入力）を使用。

### 攻撃：リアルタイムフィッシング

高度な攻撃者が本物のログインページをプロキシし、パスワードとMFAコードの両方をリアルタイムでキャプチャして本物のサイトにリプレイ。WebAuthn（パスキー）はドメインに紐づいているためこれに耐性がある。[webauthn.md](webauthn.md)参照。
