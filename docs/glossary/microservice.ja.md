# マイクロサービス（Microservice）

[English version](microservice.md)

---

## これは何？

マイクロサービスとは、1つの特定の仕事をする小さな独立したプログラムで、他の小さなプログラムとネットワーク越しに通信して完全なシステムを形成するものです。1つの大きなアプリケーションの代わりに、連携する多くの小さなものがあります。

レストランの厨房に例えるとわかりやすいです。小さなレストランでは、1人のシェフがすべてをやります：下ごしらえ、調理、盛り付け、デザート。これはモノリスです -- 1人、すべての仕事。大きなレストランでは、ソーシエ（ソース担当）、グリヤルダン（グリル担当）、パティシエ（デザート担当）、シェフ・ド・キュイジーヌ（統括）がいます。それぞれが1つの仕事を上手にこなします。これがマイクロサービスです -- 多くの専門家、それぞれが集中。

大きな厨房はより多くの注文を処理でき、個別のステーションをスケールできます。しかし、より多くのコミュニケーション（「7番テーブルのステーキは準備完了、ソースを待っています！」）、より多くの調整も必要で、1つのステーションが遅れると混乱が起きます。

## マイクロサービスが人気な理由

マイクロサービスは大きなチームが直面する実際の問題を解決するため人気になりました：

1. **独立したデプロイ。** 検索チームは決済チームに影響を与えずにデプロイできます。
2. **技術選択。** 検索チームはPythonを使い、決済チームはJavaを使えます。
3. **独立したスケーリング。** 検索が決済の100倍のトラフィックを受ける場合、決済をスケールせずに検索だけスケールできます。
4. **障害の分離（理論上）。** 検索サービスがクラッシュしても、決済は動き続けます。

何百人ものエンジニアがNetflixやAmazonを構築している会社にとって、マイクロサービスは理にかなっています。調整コストは支払う価値があります。なぜなら代替案 -- 500人のエンジニアが1つの巨大なコードベースで作業する -- の方がもっと悪いからです。

## voltaがマイクロサービスを選ばなかった理由

volta-auth-proxyはマイクロサービスとして構築することもできました：

```
マイクロサービスアプローチ（voltaがやらないこと）：
┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐
│ OIDC      │  │ Session   │  │ JWT       │  │ ForwardAuth│
│ Service   │──│ Service   │──│ Service   │──│ Service    │
│ (ログイン)│  │ (ストレージ)│  │ (署名)    │  │ (検証)     │
└───────────┘  └───────────┘  └───────────┘  └───────────┘
     ネットワークホップ   ネットワークホップ   ネットワークホップ

問題：
- 4つのサービスをデプロイ、監視、デバッグ
- 認証チェックごとに3つのネットワークホップ（レイテンシ）
- Session Serviceがクラッシュするとすべて停止
- エラー：「JWT Serviceへの接続タイムアウト」-- バグはどこ？
- 4つの別々のログストリームを相関させる必要
```

```
voltaのアプローチ（シングルプロセス）：
┌──────────────────────────────────────────┐
│ volta-auth-proxy                          │
│                                          │
│  OIDC → Session → JWT → ForwardAuth     │
│  （すべて1つのプロセス、メソッド呼び出し）  │
│                                          │
└──────────────────────────────────────────┘
     ネットワークホップなし。ログは1つ。スタックトレースは1つ。
```

### voltaが受け入れたトレードオフ

| マイクロサービスの利点 | voltaの回答 |
|-----------------------|-------------|
| 独立したスケーリング | 「ターゲットは数千ユーザーで、数百万ではない。1プロセスで十分。」 |
| サービスごとの技術選択 | 「すべてJava。それがポイント。」 |
| 独立したデプロイ | 「1つのjarをデプロイする。それはより簡単であり、劣っていない。」 |
| 障害の分離 | 「認証の障害はどのみち波及する。セッションストレージが死ねば、すべての認証が死ぬ。サービスに分割しても助けにならない。」 |

### voltaが避けたトレードオフ

| マイクロサービスの問題 | voltaは回避 |
|-----------------------|-------------|
| サービス間のネットワークレイテンシ | ゼロ -- HTTP呼び出しではなくメソッド呼び出し |
| 分散デバッグ | 1プロセス、1スタックトレース、1ログ |
| デプロイの複雑さ | 1つのjar、1つのDockerコンテナ |
| データ一貫性 | 1つのデータベース、分散トランザクションなし |
| 運用オーバーヘッド | 監視するプロセスは4つではなく1つ |

## マイクロサービスが理にかなう時（とそうでない時）

**マイクロサービスが理にかなう時：**
- 独立して作業する必要がある50人以上のエンジニアがいる
- システムの異なる部分で大幅に異なるスケーリング要件がある
- 異なるコンポーネントに異なる技術スタックが必要
- システムが大きすぎて1つのコードベースでは管理不能

**マイクロサービスが理にかなわない時：**
- 1-10人のエンジニア
- 1人が理解できる程度の小さなシステム
- すべてのコンポーネントが同じ技術を使用
- コンポーネントが密接に関連している（ログインフローの認証ステップのように）

voltaの認証フロー -- OIDCログイン、セッション作成、JWT発行、ForwardAuth検証 -- は各ステップが前のステップに依存するパイプラインです。サービスに分割しても価値を追加せずに複雑さを追加します。1人のシェフが玉ねぎを切り、それを別のシェフに渡して鍋に入れるようなものです。受け渡しの方が作業より時間がかかります。

---

## volta-auth-proxyでは

voltaはマイクロサービスではなく意図的にシングルプロセスとして動作します。すべての認証操作（OIDC、セッション、JWT、ForwardAuth）が1つの密接に接続されたパイプラインの一部であり、別々のサービスに分割するとvoltaのターゲット規模では意味のある利点なしにネットワークレイテンシとデバッグの複雑さが増すだけだからです。
