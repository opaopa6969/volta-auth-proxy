# ミドルウェア

[English version](middleware.md)

---

## これは何？

ミドルウェアとは、リクエストを受け取ってから最終的なハンドラーを実行するまでの間に実行されるコードです。[サーバー](server.md)が[HTTP](http.md)リクエストを受け取った後、ビジネスロジックが処理する前の「真ん中」に位置します。ミドルウェアはリクエストを検査し、変更し、拒否し、または通過させることができます。

空港のセキュリティチェックポイントに似ています。搭乗ゲート（ハンドラー）に向かっているとします。そこに着く前に、いくつかのチェックポイントを通過します：チケット確認、身分証明書チェック、手荷物検査、場合によっては税関チェック。各チェックポイントは通過させるか、別の場所に誘導するか、完全に止めることができます。ゲートスタッフ（あなたのハンドラー）は、すべてのチェックポイントを通過した乗客だけを見ます。ミドルウェアも同じように機能します -- 各ピースが1つのことをチェックし、リクエストを次に渡すか拒否します。

ミドルウェアは[フレームワーク](framework.md)によって「フィルター」「インターセプター」「フック」とも呼ばれます。Javalin（voltaが使用）では`before()`と`after()`ハンドラーです。

---

## なぜ重要なのか？

ミドルウェアがなければ、すべてのハンドラーで同じチェックを繰り返すことになります：

- ユーザーは認証されているか？（すべてのエンドポイントでチェック）
- ユーザーは適切な[ロール](role.md)を持っているか？（すべてのエンドポイントでチェック）
- このリクエストをログに記録すべきか？（すべてのエンドポイントでログ）
- CSRFトークンは有効か？（すべてのエンドポイントでチェック）

これは以下につながります：

- **コードの重複** -- 30のハンドラーに同じ10行の認証チェック
- **チェック漏れ** -- ある開発者が1つのハンドラーで認証チェックを忘れ、セキュリティホールができる
- **ロジックの混在** -- ビジネスロジックとセキュリティロジックが混ざり、両方とも理解しにくくなる

ミドルウェアは横断的関心事を一箇所に集約します。認証チェックを一度書き、必要なすべてのルートに適用します。

## どう動くのか？

### ミドルウェアチェーン

ミドルウェアはチェーンを形成します。各ピースが順番に実行され、次に進むか停止するかを決定できます。

```
  HTTPリクエスト
       │
       ▼
  ┌─────────────────┐
  │ ミドルウェア1    │  ログ記録：メソッド、パス、タイムスタンプを記録
  │ (ログ)          │
  └────────┬────────┘
           │ 通過
           ▼
  ┌─────────────────┐
  │ ミドルウェア2    │  認証：セッションCookieをチェック、無効なら拒否
  │ (認証)          │
  └────────┬────────┘
           │ 通過（または401拒否）
           ▼
  ┌─────────────────┐
  │ ミドルウェア3    │  RBAC：このパスにロール >= ADMIN をチェック
  │ (認可)          │
  └────────┬────────┘
           │ 通過（または403拒否）
           ▼
  ┌─────────────────┐
  │ ルートハンドラー │  あなたの実際のビジネスロジック
  │ (あなたのコード) │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 後処理ミドルウェア│  セキュリティヘッダー追加、レスポンス時間記録
  │ (レスポンスフック)│
  └────────┬────────┘
           │
           ▼
  HTTPレスポンス
```

### beforeとafterミドルウェア

ほとんどのフレームワークは2種類をサポートします：

| 種類 | 実行タイミング | 一般的な用途 |
|------|-------------|------------|
| Before（前処理） | ルートハンドラーの前 | 認証、認可、レート制限、リクエストパース |
| After（後処理） | ルートハンドラーの後 | レスポンスヘッダー追加、レスポンス時間記録、クリーンアップ |

### パススコープのミドルウェア

ミドルウェアはすべてのルートに適用することも、特定のパスにのみ適用することもできます：

```java
// すべてのリクエストに適用
app.before(ctx -> {
    logger.info("{} {}", ctx.method(), ctx.path());
});

// /api/* パスにのみ適用
app.before("/api/*", ctx -> {
    authMiddleware.requireAuth(ctx);
});

// /api/v1/admin/* パスにのみ適用
app.before("/api/v1/admin/*", ctx -> {
    authMiddleware.requireRole(ctx, Role.ADMIN);
});
```

これにより階層型のセキュリティモデルが作られます：

```
  パス                          適用されるミドルウェア
  ─────────────────────────     ────────────────────────────
  /auth/login                   ログのみ
  /auth/callback                ログのみ
  /api/v1/users/me              ログ + 認証
  /api/v1/admin/tenants         ログ + 認証 + 管理者チェック
```

### ショートサーキット

ミドルウェアがリクエストを拒否すると、チェーンを「ショートサーキット」します -- 残りのミドルウェアとハンドラーは実行されません：

```
  リクエスト: GET /api/v1/users/me（セッションCookieなし）
       │
       ▼
  ┌─────────────────┐
  │ ログMW          │  ✓ リクエストを記録
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 認証MW          │  ✗ 有効なセッションなし！
  │                 │  → 401 Unauthorizedを返す
  └─────────────────┘
           │
           ╳（ここでチェーン停止）

  ルートハンドラーは実行されない。
  認可MWは実行されない。
```

これがミドルウェアが[エンドポイント](endpoint.md)を保護する方法です。エンドポイント自身はセキュリティについて知る必要がありません。

## volta-auth-proxy ではどう使われている？

### voltaのミドルウェアレイヤー

volta-auth-proxyはJavalinの`before()`ハンドラーを使って階層型のセキュリティモデルを実装しています：

```
  ┌──────────────────────────────────────────────┐
  │              すべてのリクエスト                 │
  │  before("/*"): リクエストログ記録              │
  ├──────────────────────────────────────────────┤
  │            /api/* リクエスト                   │
  │  before("/api/*"): セッション/JWT検証          │
  │  → コンテキストにX-Volta-User-Idを設定        │
  │  → コンテキストにX-Volta-Tenant-Idを設定      │
  ├──────────────────────────────────────────────┤
  │         /api/v1/admin/* リクエスト             │
  │  before("/api/v1/admin/*"): ロール >= ADMIN   │
  ├──────────────────────────────────────────────┤
  │            ForwardAuthリクエスト               │
  │  before("/forwardauth"): セッションチェック    │
  │  → TraefikにX-Volta-*ヘッダーを返す          │
  └──────────────────────────────────────────────┘
```

### 認証ミドルウェア

コア認証ミドルウェアは有効な[セッション](session.md)をチェックします：

```java
app.before("/api/*", ctx -> {
    var session = ctx.sessionAttribute("user");
    if (session == null) {
        throw new UnauthorizedResponse();  // 401
    }
    // ハンドラー用にユーザー情報をリクエストコンテキストに付加
    ctx.attribute("userId", session.userId());
    ctx.attribute("tenantId", session.tenantId());
});
```

下流のハンドラーは、すでに検証済みユーザーが付加されたリクエストだけを受け取ります。セッションチェックを扱うことはありません。

### 認可ミドルウェア

認証の後、ロールチェックミドルウェアが[RBAC](rbac.md)を強制します：

```java
app.before("/api/v1/admin/*", ctx -> {
    var role = ctx.attribute("role");
    if (!role.isAtLeast(Role.ADMIN)) {
        throw new ForbiddenResponse();  // 403
    }
});
```

### ForwardAuthミドルウェア

Traefikが[ForwardAuth](forwardauth.md)リクエストを送ると、ミドルウェアがセッションを検証し、ユーザー情報を[ヘッダー](header.md)として返します：

```java
app.before("/forwardauth", ctx -> {
    var session = validateSession(ctx);
    ctx.header("X-Volta-User-Id", session.userId().toString());
    ctx.header("X-Volta-Tenant-Id", session.tenantId().toString());
    ctx.header("X-Volta-Roles", String.join(",", session.roles()));
});
```

[ダウンストリームアプリ](downstream-app.md)はこれらのヘッダーを読み、信頼します。Traefikが転送前にクライアント提供のX-Volta-*ヘッダーを除去するためです。

### セキュリティヘッダーのafterミドルウェア

voltaは`after()`フックを使って、すべてのレスポンスにセキュリティ[ヘッダー](header.md)を追加します：

```java
app.after(ctx -> {
    ctx.header("X-Content-Type-Options", "nosniff");
    ctx.header("X-Frame-Options", "DENY");
    ctx.header("Cache-Control", "no-store");
});
```

## よくある間違いと攻撃

### 間違い1：ミドルウェアの順序が間違っている

ミドルウェアは登録順に実行されます。ログミドルウェアを認証ミドルウェアの後に置くと、拒否されたリクエストはログに記録されません：

```java
// 間違った順序 -- 認証失敗が見えない
app.before("/api/*", ctx -> requireAuth(ctx));      // 拒否 → ログなし
app.before("/api/*", ctx -> logRequest(ctx));        // 実行されない

// 正しい順序 -- すべてがログに記録される
app.before("/api/*", ctx -> logRequest(ctx));        // まずログ
app.before("/api/*", ctx -> requireAuth(ctx));       // 次に認証チェック
```

### 間違い2：ミドルウェアのスコープを忘れる

認証ミドルウェアを広く適用しすぎると、公開[エンドポイント](endpoint.md)が壊れます：

```java
// 間違い -- ログインページ自体がブロックされる
app.before("/*", ctx -> requireAuth(ctx));

// 正しい -- APIルートだけを保護
app.before("/api/*", ctx -> requireAuth(ctx));
```

### 攻撃1：パス操作によるミドルウェアのバイパス

攻撃者はパストリックでパススコープのミドルウェアをバイパスしようとするかもしれません：

```
  /api/v1/admin/../public/resource    （パストラバーサル）
  /API/v1/admin/tenants               （大文字小文字操作）
  /api/v1/admin/tenants/              （末尾スラッシュ）
  /api/v1/admin/tenants;jsessionid=x  （パスパラメータインジェクション）
```

Javalinはマッチング前にパスを正規化するため、ほとんどを防ぎます。ただし、パスベースのミドルウェアセキュリティはフレームワークのパスマッチングの正確さに依存することを理解しておくことが重要です。

### 攻撃2：ミドルウェア経由のヘッダーインジェクション

ミドルウェアがクライアントからのヘッダーを検証なしに下流に渡すと、攻撃者が任意の値を注入できます。voltaの[ForwardAuth](forwardauth.md)ミドルウェアは安全です。Traefikが認証エンドポイントを呼ぶ前に、元のリクエストからX-Volta-*ヘッダーを除去するためです。

### 間違い3：ミドルウェアで重い処理をする

ミドルウェアは高速であるべきです。すべてのリクエストで実行されるミドルウェアにデータベースクエリを入れると、すべてのリクエストにレイテンシが追加されます -- 後のミドルウェアで拒否されるものにも。ミドルウェアは軽量に保ちましょう：セッション検索（理想的にはキャッシュ済み）、ロールチェック（セッションデータから）、ヘッダー検証。
