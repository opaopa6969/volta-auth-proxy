# 最小構成（Minimum Viable Architecture）

[English version](minimum-viable-architecture.md)

---

MVP -- Minimum Viable Product（実用最小限の製品）は誰もが聞いたことがあります。顧客が実際に使う最小の製品を作り、そこから学び、反復する。この概念はプロダクト開発を変革しました。

ではアーキテクチャはどうでしょうか？多くのチームは「行き止まりに陥る」ことを恐れて、デフォルトで複雑なアーキテクチャを採用します。初日からメッセージキュー、分散キャッシュ、別々のマイクロサービス、コンテナオーケストレーションを追加します -- 念のために。このエッセイはMVA（Minimum Viable Architecture：実用最小限のアーキテクチャ）を主張します。動作する本番品質のシステムを提供するコンポーネントの最小セットです。

---

## voltaのPhase 1アーキテクチャ：Gateway + Postgres

Phase 1でのvoltaの本番アーキテクチャは2つのコンポーネントです：

```
  ┌──────────────────────────────────────────────┐
  │                                                │
  │  ┌──────────────┐     ┌──────────────┐       │
  │  │ volta-auth-  │────►│ PostgreSQL   │       │
  │  │ proxy        │     │              │       │
  │  │              │◄────│              │       │
  │  └──────────────┘     └──────────────┘       │
  │                                                │
  │  これだけ。                                    │
  │  箱2つ。矢印1本。                             │
  │                                                │
  └──────────────────────────────────────────────┘
```

Redisなし。Kafkaなし。Elasticsearchなし。メッセージキューなし。分散キャッシュなし。別々のマイクロサービスなし。Kubernetesなし。1つのアプリケーションプロセスが1つのデータベースと通信するだけです。

これはデモ環境ではありません。本番アーキテクチャです。実際のユーザーがこれを通じて認証します。実際のセッションがこれで管理されます。実際のJWTがこれで署名されます。そして~30MBのRAMと~200msの起動時間ですべてを処理します。

## 2コンポーネントが5コンポーネントより良い理由

### 「すべてのコンポーネントが障害点」の原則

アーキテクチャを比較しましょう：

```
  アーキテクチャA: volta（2コンポーネント）
  ┌──────┐     ┌──────────┐
  │ volta │────►│ Postgres │
  └──────┘     └──────────┘
  障害点: 2
  ネットワーク接続: 1
  監視するもの: 2
  バックアップするもの: 1

  アーキテクチャB: Oryスタック（5コンポーネント）
  ┌────────────┐  ┌────────┐  ┌───────┐  ┌──────┐  ┌──────────┐
  │ Oathkeeper  │  │ Kratos │  │ Hydra │  │ Keto │  │ Postgres │
  └──────┬─────┘  └───┬────┘  └───┬───┘  └──┬───┘  └────┬─────┘
         │            │           │          │           │
         └────────────┴───────────┴──────────┴───────────┘
  障害点: 5
  ネットワーク接続: 4+（各サービスがPostgresと、場合によっては相互に通信）
  監視するもの: 5
  バックアップするもの: 1（ただし4サービスが依存）

  アーキテクチャC: Keycloak + エコシステム（3+コンポーネント）
  ┌──────────┐  ┌───────────┐  ┌──────────┐
  │ Keycloak  │  │ Infinispan│  │ Postgres │
  └────┬─────┘  └─────┬─────┘  └────┬─────┘
       │              │              │
       └──────────────┴──────────────┘
  障害点: 3
  ネットワーク接続: 2+
  監視するもの: 3
  バックアップするもの: 1+
```

コンポーネントが追加されるたびに以下が乗算されます：
- **デプロイメントの複雑さ** -- より多くのコンテナ、設定、ポートマッピング
- **モニタリングの負担** -- より多くのヘルスチェック、ダッシュボード、アラート
- **デバッグの難しさ** -- 「どのサービスが500エラーを引き起こしている？」
- **アップグレードリスク** -- サービス間のバージョン互換性
- **リソース使用量** -- 各サービスがCPU、RAM、ディスクを使用

### 「$10のVPSで動かせるか？」テスト

voltaはこのテストに合格します。$10/月のVPS（1 CPU、1GB RAM）でvolta（~30MB）+ PostgreSQL（~50MB）が余裕を持って快適に動きます。SaaSの認証インフラ全体がコーヒー1杯より安い。

$10のVPSでOryスタックを動かしてみてください。またはKeycloakを。少なくとも$40-80/月のサーバーが必要で、それは実際のアプリケーションの前の話です。

これはケチという話ではありません。効率性の話です。認証システムが$80/月のインフラを必要とするなら、全体のホスティングコストがそれに比例して増えます。voltaの最小フットプリントは、認証がインフラ予算を支配しないことを意味します。

## MVAが「ない」もの

MVAは以下では「ない」：
- **手を抜くこと。** voltaの認証は本番品質：PKCE、RS256、暗号化されたキーストレージ、CSRF保護、レート制限、監査ログ。最小限のアーキテクチャは最小限のセキュリティを意味しません。
- **スケーリングの拒否。** voltaのアーキテクチャはPostgreSQLの垂直スケーリング、Redisセッション（Phase 2）、ロードバランサー背後のvoltaインスタンスの水平展開による成長をサポートします。アーキテクチャは成長するよう設計されています -- ただ成長した状態から始まらないだけ。
- **将来を無視すること。** voltaはインターフェース（`SessionStore`、`AuditSink`、`NotificationService`）を定義し、必要なときに追加コンポーネントのプラグインを可能にします。アーキテクチャは拡張に備えていますが、それに重荷を負っていません。

MVAは以下：
- 本番品質を提供する最も少ないコンポーネントから始める
- 実際の必要性が示されたときにのみコンポーネントを追加する
- シンプルさを制限ではなく機能として扱う

## 「何がうまくいかないか」エクササイズ

有用なエクササイズ：追加したい各コンポーネントに対して「このコンポーネントが障害を起こしたら何が起きるか？」と尋ねる。

```
  コンポーネント: Redis（セッション用）
  Redisが障害を起こしたら？
  - すべてのセッションが失われる
  - すべてのユーザーが再ログインする必要がある
  - RedisとPostgres間のデータ不整合の可能性

  Redisなし（volta Phase 1）：
  Postgresが障害を起こしたら？
  - セッションが失われる（他のすべてと同じデータベース）
  - voltaが起動できない
  - しかし：Postgres障害はどのみちすべてがダウンすることを意味

  Redisを追加してもシステムはより信頼性が高くなったわけではない。
  2つ目の障害モードを追加しただけ。
```

これはRedisが悪いという意味ではありません。Redisはその仕事に優れています。しかしアーキテクチャに追加することにはコストがあり、そのコストは仮想的な将来の負荷ではなく、実際の必要性（測定されたスループットの限界）によって正当化されるべきです。

## コンポーネントを追加するタイミング

Phase 2-4のロードマップがコンポーネントを追加するタイミングを教えてくれます：

| コンポーネント | 追加するとき | 必要だというシグナル |
|---------------|------------|---------------------|
| Redis | Phase 2 | セッションクエリがPostgres負荷の50%以上 |
| Kafka | Phase 2+ | 監査イベントが外部システムへのリアルタイムストリーミングを必要とする |
| Elasticsearch | Phase 3+ | 数百万の監査イベントの全文検索が必要 |
| SMTPサーバー | Phase 3 | 顧客がメール通知を望む |
| 2つ目のvoltaインスタンス | 必要時 | 負荷下でリクエストレイテンシが増加 |

キーフレーズは「必要だというシグナル」。「いつか必要になるかもしれないと思う」ではない。「ベストプラクティスでは持つべきと言っている」ではない。「アーキテクトがホワイトボードに描いた」ではない。実際の、測定可能なシグナル。

## 哲学的なポイント

最小構成は規律の行為です。仮想的な問題のために構築する衝動に抵抗することです。適切に設計された基盤（インターフェース、クリーンなデータモデル、バージョン管理されたAPI）が、手戻りなしに後からコンポーネントを追加できると信頼することです。

こう言うことです：「6ヶ月後に何が必要かは分からない。今日何が必要かは分かる。今日のシステムは今日作り、明日のシステムは明日作ろう。」

voltaのアーキテクチャはこれを体現しています：今日は2つのコンポーネント、必要性が証明されたときにそれ以上への明確なパス。そして注目すべきは、大半のSaaS製品にとって、今日のアーキテクチャはとても長い間十分だということです。
