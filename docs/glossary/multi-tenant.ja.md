# マルチテナンシー

[English version](multi-tenant.md)

---

## 一言で言うと？

マルチテナンシーとは、1つのアプリケーションが多くの別々のユーザーグループ（「テナント」と呼ばれる）に同時にサービスを提供しながら、各グループのデータを他のグループから完全に非公開に保つ設計のことです。

---

## マンションのたとえ

大きなマンションを想像してください。建物自体は1つの構造物です。壁も屋根もエレベーターも1つで、配管や電気は共有されています。でも中に入れば、各部屋は完全にプライベートな空間です。あなたは自分の鍵を持っています。隣の住人があなたの部屋に入って冷蔵庫を覗くことはできません。管理人が誰にどの部屋の鍵を渡すかを決めます。

ソフトウェアでも、これがまさにマルチテナンシーです：

- **建物** = アプリケーション（volta-auth-proxy）
- **各部屋** = テナント（ソフトウェアを使う会社、チーム、組織）
- **管理人** = 認証・認可システム
- **部屋の鍵** = ログイン情報とアクセス権限

全員が同じアプリケーションコード、同じサーバー、同じデータベースを共有しています。しかし各テナントのデータは他のテナントから遮断されていて、それぞれが自分だけの部屋を持っているようなものです。

---

## なぜSaaSにマルチテナンシーが必要なの？

Slack、Notion、Google Workspaceを使ったことがあれば、マルチテナントアプリケーションを使ったことがあります。あなたの会社のSlackワークスペースは1つのテナントです。別の会社のワークスペースは別のテナントです。全員が同じSlackソフトウェアを使っていますが、お互いのメッセージを見ることはできません。

マルチテナンシーがなければ、すべての顧客に対してソフトウェアの完全に別のコピーを動かす必要があります。Slackが何百万もの企業それぞれに別のサーバーを用意しなければならなかったら、どうなるでしょう：

- **ものすごく高コスト** -- 各顧客に専用のサーバー、データベース、メンテナンスが必要
- **アップデートが不可能** -- 何百万もの別々のインストールを更新しなければならない
- **管理が悪夢** -- あるコピーのバグが別のコピーには存在しないかもしれない

マルチテナンシーなら、1つのアプリケーションで全員にサービスを提供しつつ、全員のデータを安全に分離できます。

---

## データはどうやって分離されるの？

一般的な方法は3つあります。分離のレベルとして考えてみてください：

**レベル1：ラベル付きの共有テーブル（voltaが使用する方式）**
全員の書類が一緒に保管されているファイルキャビネットを想像してください。ただし、すべての書類には所有者を示す色付きのシールが貼ってあります。キャビネットを開けるとき、自分のシールの色の書類しか見ることができません。

データベースの用語では、すべてのテーブルのすべての行に`tenant_id`というカラムがあります。アプリがデータを検索するとき、常にこのカラムでフィルタリングします。シンプルで効率的です。

**レベル2：別々の引き出し**
各テナントがファイルキャビネット内に自分専用の引き出しを持ちます。より強い分離ですが、管理が大変です。

**レベル3：別々のキャビネット**
各テナントが完全に別のファイルキャビネットを持ちます。最も強い分離ですが、非常に高コストです。

---

## volta-auth-proxyでのマルチテナンシーの仕組み

voltaはレベル1（tenant_id付きの共有テーブル）を使用しています。実際にはこのようになります：

```
  「projects」というデータベーステーブルを想像してください：

  ┌────────────┬───────────┬─────────────────┐
  │ tenant_id  │ id        │ project_name    │
  ├────────────┼───────────┼─────────────────┤
  │ acme-uuid  │ proj-1    │ Website Redesign│
  │ acme-uuid  │ proj-2    │ Mobile App      │
  │ globex-uuid│ proj-3    │ Rocket Engine   │
  │ globex-uuid│ proj-4    │ Moon Base       │
  └────────────┴───────────┴─────────────────┘

  ACMEのユーザーがログインすると、voltaはすべての
  クエリに WHERE tenant_id = 'acme-uuid' を含めます。
  なのでACMEのユーザーには「Website Redesign」と
  「Mobile App」しか見えません。Globexの存在すら知りません。
```

voltaはゲートウェイレベルでこの分離を強制します。リクエストが来ると、voltaはユーザーがどのテナントに属しているかを特定し、その情報を特別なヘッダー（`X-Volta-Tenant-Id`）に入れて、下流のアプリに送ります。アプリはそのテナントIDを使って、データベースクエリをすべてフィルタリングします。

---

## 簡単な例

voltaの後ろにあるwikiアプリを作っているとしましょう。「ACME Corp」のユーザーがwikiにアクセスすると：

1. ユーザーのブラウザがwikiにリクエストを送る
2. リクエストはまずvolta-auth-proxy（「管理人」）に到着する
3. voltaが確認する：「この人はログインしている？はい。どのテナント？ACME Corp。」
4. voltaがリクエストに`X-Volta-Tenant-Id: acme-uuid`を追加する
5. wikiアプリがリクエストを受け取り、`acme-uuid`を使ってACMEのwikiページだけを検索する
6. ユーザーには自分の会社のwikiだけが表示される。他の会社のものは一切見えない
