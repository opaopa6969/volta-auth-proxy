# MVP（Minimum Viable Product：実用最小限の製品）

[English version](mvp.md)

---

## これは何？

MVP は Minimum Viable Product（実用最小限の製品）の略です。コアとなる問題を解決できる、最もシンプルなバージョンの製品です。プロトタイプではありません。デモでもありません。本物の製品です -- ただし、可能な限り小さな機能セットで。

レストランの開業を想像してください。MVP のレストランにはミシュランスターのメニュー、ワインセラー、バレーパーキングは不要です。美味しい料理、清潔なテーブル、支払い方法があればいい。お客さんが料理を目当てにリピートすれば、拡張する価値があるとわかります。リピートしなければ、誰も欲しがらなかったワインセラーのコストを節約できたことになります。

「Viable（実用的）」という言葉がポイントです。MVP は実際に動かなければなりません。誰もログインできないログインページは MVP ではなく、壊れた製品です。MVP は機能が少ないですが、ある機能は正しく確実に動く必要があります。

---

## なぜ重要？

ソフトウェア開発は高コストで、最大のリスクは間違ったものを作ることです。MVP はこのリスクを軽減します -- 本物の製品を本物のユーザーの前にできるだけ早く届けることで。1週間の実際の利用から得られる学びは、6ヶ月の計画よりも多いです。

代替案 -- 誰も使う前にフル機能の製品を何ヶ月もかけて作る -- は、ほとんどのスタートアップが失敗する方法です。ユーザーが欲しいと思うものを作り、華々しくローンチし、ユーザーが別のものを欲しがっていたことに気づくのです。

MVP は優先順位付けも強制します。5つの機能しか作れないなら、最も重要な5つを決めなければなりません。この明確さは MVP フェーズの後も続きます。

## どう動くのか？

### MVP プロセス

```
  ステップ1: コア問題を特定
            「ユーザーは Google 経由で認証し、
             チームメンバーを管理する必要がある」

  ステップ2: 作りうるすべての機能をリスト化
            - Google ログイン
            - GitHub ログイン
            - メール/パスワードログイン
            - チーム管理
            - ロールベースアクセス
            - 監査ログ
            - SSO/SAML
            - API キー管理
            - 請求統合
            - 管理ダッシュボード
            ...

  ステップ3: MVP ラインを引く
            ─────────────────────────────────
            上: 今作る（MVP）
            - Google ログイン
            - チーム管理（招待、削除）
            - ロールベースアクセス（OWNER, MEMBER）
            ─────────────────────────────────
            下: 後で作る（検証後）
            - その他すべて
            ─────────────────────────────────

  ステップ4: 作る、出す、学ぶ
            │
            ▼
         MVP を出荷 ──► 実ユーザーが試す
            │                │
            ▼                ▼
         計測 ◄──── フィードバック
            │
            ▼
         決定：拡張、ピボット、または停止
```

### 良い MVP の条件

| 良い MVP | 悪い MVP |
|---------|---------|
| 1つの問題を完全に解決 | 多くの問題を部分的に解決 |
| 確実に動作 | 「自分のマシンでは動く」 |
| エラーハンドリングあり | 不正入力でクラッシュ |
| ユーザーがデプロイ可能 | 開発者がセットアップ必要 |
| 明確な境界（「X はやる、Y はやらない」） | すべてを約束、何も届けない |

### MVP vs. プロトタイプ vs. 概念実証

| 概念 | 対象者 | 品質 | 目的 |
|------|--------|------|------|
| **概念実証** | 開発者 | 使い捨て | 「これ、そもそも動く？」 |
| **プロトタイプ** | ステークホルダー | 低い | 「これが正しい UX？」 |
| **MVP** | 実ユーザー | 本番品質 | 「これは実際の問題を解決する？」 |

## volta-auth-proxy ではどう使われている？

volta-auth-proxy 自体が MVP です。より大きなビジョンの Phase 1 であり、意図的に1つの問題をうまく解決するようにスコープが絞られています：小規模チーム向けのマルチテナント認証。

### volta の MVP に含まれるもの

| 機能 | 状態 | MVP に含まれる理由 |
|------|------|-------------------|
| Google OIDC ログイン | 構築済 | コア認証フロー -- 存在意義そのもの |
| テナント管理 | 構築済 | マルチテナンシーが差別化要因 |
| ロールベースアクセス（OWNER/MEMBER） | 構築済 | 最小限のアクセス制御 |
| 招待システム | 構築済 | チームにメンバー追加が必要 |
| セッション管理 | 構築済 | セキュリティのベースライン |
| CSRF 対策 | 構築済 | セキュリティはオプションではない |
| ヘルスチェックエンドポイント | 構築済 | デプロイ可能性の要件 |
| DSL ベースの設定 | 構築済 | コア開発者体験 |

### volta の MVP に含まれないもの

| 機能 | 除外理由 |
|------|---------|
| GitHub/Azure/Okta ログイン | Google OIDC でモデルを証明；他は Phase 2 |
| PostgreSQL サポート | 小規模チームには SQLite で十分 |
| 高可用性 | ターゲット層にはシングルノードで十分 |
| SOC2 準拠 | エンタープライズ要件、インディーハッカーの要件ではない |
| SLA 保証 | 有料顧客がまだいない、SLA 不要 |
| 請求統合 | Stripe ウェブフック は Phase 2 で予定 |
| 管理ダッシュボード UI | API ファースト；UI は後で |

### MVP 検証ループ

volta は早期ユーザーからのフィードバックを使って次に何を作るかを決定します：

```
  volta Phase 1（MVP）
        │
        ▼
  インディーハッカー / 社内ツールビルダーに出荷
        │
        ▼
  シグナルを収集：
    - 最初に何を求めるか？
    - どこで詰まるか？
    - 何を回避策で対処しているか？
        │
        ▼
  Phase 2 スコープ = 上位リクエスト + ブロッカー修正
```

## よくある間違いと攻撃

### 間違い1：MVP は低品質を意味する

最も一般的な誤解です。MVP は機能が少ないことであり、品質が低いことではありません。時々失敗するログインシステムは MVP ではなく、バグです。MVP はログインオプションが少ない（Google のみ、Google + GitHub + SAML ではない）ですが、あるものは完璧に動きます。

### 間違い2：フィーチャークリープの罠

「ローンチ前にもう1つだけ機能が必要」。この一文は、どんな技術的課題よりも多くの製品を遅延させてきました。MVP のスコープは早期にロックし、積極的に守るべきです。

### 間違い3：ユーザーと話さずに MVP を作る

仮定に基づいて作った MVP はやはり推測です。コードを書く前に5人の潜在ユーザーと話しましょう。volta の DGE プロセス（Dialogue-driven Gap Extraction）は、構造化された対話を通じて実際の要件を浮かび上がらせるよう設計されています。

### 間違い4：MVP を最終製品として扱う

MVP は出発点であり、目的地ではありません。MVP が成功したら、拡張に投資しなければなりません。MVP の品質でずっと運用し続ければ、ユーザーは離れていきます。
