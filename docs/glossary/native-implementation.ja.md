# 自前実装 / ネイティブ実装

[English version](native-implementation.md)

---

ソフトウェアにおける「ネイティブ」「自前実装」とは、他人のソリューションを使う代わりに自分で書いたことを意味します。volta-auth-proxyがOIDCをネイティブに処理すると言うとき、OIDCフローがvolta自身のコードで実装されていることを意味します -- Keycloakに委譲せず、SDKで処理せず、サードパーティサービスを経由しません。コードはあなたのリポジトリにあり、すべての行を読めます。

このエッセイは、チームがいつ、なぜ自前構築を選ぶか、そしてそのコストについてです。

---

## 認証を自前で作る理由

自前で認証を構築する決断は、通常3つの場所のいずれかから来ます：

**フラストレーション。** チームが6ヶ月間Keycloakを使って設定の海で溺れている。またはAuth0を使って月額$2,400の請求が来た。またはログインページをカスタマイズしようとして1週間FreeMarkerテンプレートと格闘した。ある時点で誰かが言います：「このツールと戦う時間より短い時間で自分たちで作れるのでは。」

**コントロール。** チームが特定のもの -- 特定のマルチテナンシーモデル、特定の招待フロー、カスタム同意画面 -- を必要とし、既存のツールではきれいな回避策なしにはできない。すべての決定を自分で持ちたい。

**理解。** 認証は重要なインフラです。チームの中には、認証システムを自分で構築できるほど深く理解していないなら、安全に運用できるほど深く理解していないと考える人もいます。「すべての行を説明できないなら、すべての行を信頼できない。」

voltaは3つすべての動機から生まれました。

---

## 自前構築の利点

### コントロール

自分で認証を構築すると、すべての決定はあなたのものです：

- セッションタイムアウト：あなたが選ぶ
- JWTクレーム：あなたが選ぶ
- マルチテナンシーモデル：あなたが選ぶ
- ログインページ：ピクセル単位のコントロール
- セキュリティヘッダー：あなたが設定する

「申し訳ありませんが、その機能にはエンタープライズプランが必要です」はありません。「この設定は構成で公開されていません」はありません。コードだけがあり、コードは何でもできます。

### 理解

自分でOIDCフローを書くと、理解できます。「ドキュメントを読んだ」レベルの理解ではなく、「stateパラメータを実装し、なぜCSRFを防ぐか正確に知っている」レベルの理解です。この深さは問題が起きたときに重要です。自分で認証を構築したチームは午前3時にデバッグできます。他人の認証を設定したチームは午前3時にStack Overflowを読みます。

### 速度（開発ではなく、システムの）

自前の認証はあなたの正確なユースケースに最適化できます。voltaは200msで起動し30MBのRAMを使います。必要なことだけをするから。Keycloakは30秒で起動し512MBを使います。誰かが必要とするかもしれないすべてをするから。

### ベンダー依存なし

自前で構築すると、誰もあなたの料金を上げたり、機能を廃止したり、ベンダーを買収してロードマップを変更したりできません。認証システムはあなたが望む限り存在し続けます。

---

## 自前構築の欠点

### 責任

これが最大のものです。自分で認証を構築すると、すべてのバグ、すべての脆弱性、すべてのインシデントを所有します。Auth0のセキュリティチームがJWTライブラリのCVEを監視してくれるわけではありません。KeycloakコミュニティがOIDCフローのパッチをリリースしてくれるわけではありません。あなたのチームだけです。

```
  JJWTライブラリにセキュリティ脆弱性が発見された場合：

  Auth0を使っている場合：
  - Auth0のセキュリティチームがパッチする
  - あなたは何もしない

  volta（自前構築）を使っている場合：
  - CVEのアナウンスを見る
  - システムへの影響を評価する
  - 依存関係を更新する
  - 更新をテストする
  - 再デプロイする
  - 本番で修正を検証する
```

これは実際の仕事です。警戒が必要です。セキュリティアドバイザリを読むチームの誰かが必要です。

### 自分で作るバグ

Auth0は何千もの企業が何年もかけてテストしています。あなたの自前の認証はチームが数ヶ月テストしたものです。実装に微妙なバグ（タイミング攻撃、セッション処理のレースコンディション、CSRFのエッジケース）がある確率は、Auth0が同じバグを持つ確率より高いです。

これは自前の認証が安全でないことを意味しません。自前の認証にはより多くのテスト、より多くのレビュー、より多くの謙虚さが必要だということを意味します。「これは安全だと思う」は「プロフェッショナルがペネトレーションテストした」とは同じではありません。

### メンテナンス

認証は「作って忘れる」コンポーネントではありません。OAuth2の標準は進化します。ブラウザのセキュリティポリシーは変わります。新しい攻撃ベクトルが出現します。自前の認証システムには継続的なメンテナンスが必要です -- 機能開発だけでなく、セキュリティの衛生管理も。

## voltaの哲学：「理解できる地獄を選べ」

すべての選択肢にはデメリットがあります。認証にフリーランチはありません：

- **Auth0の地獄：** ベンダーロックイン、増大するコスト、限定的なコントロール、他社のロードマップへの依存。
- **Keycloakの地獄：** 設定の複雑さ、リソースの重さ、FreeMarkerテンプレート、スケール時のレルム管理。
- **自前構築の地獄：** セキュリティ責任、バグの所有、メンテナンス負担。

voltaの主張はシンプルです：**理解できる地獄を選べ。**

自分で認証を構築するとき（voltaを基盤として使って）、コードのすべての行を理解しています。何かが壊れたら修正できる。脆弱性が発見されたらパッチできる。顧客が機能を必要としたら構築できる。自前認証の「地獄」は本物ですが、地形を理解しているからナビゲートできる地獄です。

Keycloakの500行realm.jsonの「地獄」も本物ですが、`quickLoginCheckMilliSeconds`が何をするか理解していないからナビゲートできない地獄です。Auth0のベンダーロックインの「地獄」も本物ですが、コード、ユーザー、ワークフローがすべて彼らのプラットフォームと絡み合っているから逃げられない地獄です。

少なくとも自前の認証なら、問題はあなたの問題です。誰かが修正するのを待っていない。誰かに修正料金を払っていない。誰かが修正してくれることを祈っていない。

---

## voltaは基盤であり、白紙ではない

重要な注意点：voltaは「認証をゼロから構築する」ではありません。voltaは既に難しい決定を行っています：

- PKCE付きOIDC Authorization Codeフロー：実装済み・テスト済み
- RS256とキーローテーションによるJWT発行：実装済み・テスト済み
- スライディングウインドウ期限切れによるセッション管理：実装済み・テスト済み
- ロールベースアクセスを持つマルチテナントデータモデル：実装済み・テスト済み
- ForwardAuthプロキシパターン：実装済み・テスト済み
- CSRF保護、レート制限、監査ログ：実装済み・テスト済み

voltaを使うとき、認証をゼロから書いているのではありません。動作する、テスト済みの基盤から始めてカスタマイズしているのです。「ネイティブ実装」の利点は、カスタマイズできることです -- すべてをゼロから構築しなければならないことではありません。

これが違いです：
- 「認証をゼロから構築」（数ヶ月の作業、高リスク）
- 「voltaを基盤にしてコードを所有」（数日のセットアップ、低リスク、完全なコントロール）
