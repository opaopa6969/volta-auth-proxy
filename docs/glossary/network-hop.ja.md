# ネットワークホップ（Network Hop）

[English version](network-hop.md)

---

## これは何？

ネットワークホップとは、データがネットワーク接続を通じてあるサーバー（またはプロセス）から別のサーバーに移動するたびの1回の転送です。各ホップは時間を追加し、潜在的な障害ポイントを導入し、誰かがデータを傍受できる場所を作ります。

手紙を仲介者に順番に渡すようなものです。あなたが手紙をアリスに渡し、アリスがボブに渡し、ボブがキャロルに渡し、キャロルが受取人に渡します。各受け渡しが「ホップ」です。各ホップには時間がかかります（アリスはボブを見つけなければなりません）。各ホップは失敗する可能性があります（ボブは休暇中かもしれません）。そして各ホップは誰かが手紙を読むチャンスです（ボブはおせっかい）。

手紙を受取人に直接送る -- 1ホップ -- の方が速く、信頼性が高く、プライベートです。

---

## ホップが少ない方がいい理由

### 速度

すべてのネットワークホップはレイテンシを追加します。高速なローカルネットワークでも、各ホップは往復で約0.5-2ms追加します。微小に聞こえますが、積み重なります：

```
1ホップ：  ~1ms    （見えないほど高速）
3ホップ：  ~3-6ms  （規模が大きいと顕著）
5ホップ：  ~5-10ms （ユーザーが感じ始める）
10ホップ： ~10-20ms（認証には明らかに遅い）
```

そしてこれはネットワーク転送時間だけです。各ホップにはさらに以下が必要です：
- シリアライゼーション（送信用にデータをバイトに変換）
- デシリアライゼーション（到着時にバイトをデータに変換）
- 接続オーバーヘッド（TCPハンドシェイク、TLSネゴシエーション）
- 各停止点での処理

処理を含む現実的なホップあたりのコスト：2-10ms。

### 信頼性

各ホップは潜在的な障害ポイントです。ネットワークは完全に信頼できるわけではありません。接続がタイムアウトします。DNSが誤って解決します。ロードバランサーにバグがあります。ファイアウォールがパケットを静かにドロップします。

各ホップの成功率が99.9%（これは良い方）の場合：

```
1ホップ：  99.9%の成功率
3ホップ：  99.9% × 99.9% × 99.9% = 99.7%
5ホップ：  99.9%^5 = 99.5%
10ホップ： 99.9%^10 = 99.0%
```

10ホップでは、100リクエスト中1リクエストがネットワークの問題で失敗します。すべてのリクエストを処理する認証システムにとって、100ページ読み込み中1回が失敗することを意味します。

### セキュリティ

すべてのホップはデータが転送中で潜在的に露出する場所です。TLS暗号化があっても、各ホップは以下のポイントです：
- データが復号化され、処理され、再暗号化される
- 受信サーバーが平文データにアクセスできる
- 設定ミスのサーバーが機密情報をログに記録したりリークしたりする可能性
- 侵害されたサーバーがデータを改ざんできる可能性

ホップが少ないほど、認証トークン、セッションID、ユーザーデータが露出する場所が少なくなります。

## voltaのForwardAuth vs リバースプロキシ：ホップの比較

```
フルリバースプロキシ（voltaではない）：
  ブラウザ → リバースプロキシ → 認証サービス → バックエンドアプリ
  ホップ1：ブラウザからプロキシ
  ホップ2：プロキシから認証サービス
  ホップ3：認証サービスからプロキシに返答
  ホップ4：プロキシからバックエンドアプリ
  合計：4ホップ、認証サービスがすべてのトラフィックを見る（リクエストボディ含む）

ForwardAuth（volta）：
  ブラウザ → Traefik → volta（認証チェックのみ）
                     → バックエンドアプリ（実際のリクエスト）
  ホップ1：ブラウザからTraefik
  ホップ2：Traefikからvolta（ヘッダーのみ、ボディなし）
  ホップ3：voltaのレスポンスがTraefikに
  ホップ4：Traefikからバックエンドアプリ（追加ヘッダー付きの元リクエスト）
  合計：4ホップ、ただしvoltaはヘッダーのみを見る（完全なリクエストではない）
```

ホップ数は似ていますが、重要な違いはそれらのホップで何が転送されるかです。フルリバースプロキシでは、リクエスト全体（大きなPOSTボディ、ファイルアップロード含む）が認証サービスを通過します。ForwardAuthでは、voltaはヘッダーのみを受信し返却します -- メガバイトではなく数百バイト。

そしてvolta内部ではゼロの内部ホップです：

```
マイクロサービス認証（内部ホップ）：
  ForwardAuthエンドポイント → Session Service → JWT Service
  リクエストあたり2つの追加内部ホップ

volta（内部ホップなし）：
  ForwardAuthエンドポイント → セッションチェック → JWT発行
  0内部ホップ（すべて1プロセス内のメソッド呼び出し）
```

---

## volta-auth-proxyでは

voltaはすべての認証ロジックを単一プロセスで実行することで内部ネットワークホップを排除し、認証チェックでヘッダーのみ（リクエストボディではなく）が通過するForwardAuthパターンを使用することで外部ホップで転送されるデータを最小化しています。
