# ネットワーク分離

[English version](network-isolation.md)

---

## 一言で言うと？

ネットワーク分離とは、コンピュータのグループ間に壁を設けて、必要なもの同士だけが通信できるようにし、不正アクセスを防ぐことです。

---

## レストランの「関係者以外立入禁止」エリア

レストランに例えると：

- **客席** -- お客さんはここに座れる（パブリック[ネットワーク](network.ja.md)）
- **厨房** -- スタッフだけ入れる（プライベートネットワーク）
- **厨房との間のドア** -- ウェイターだけが通る（リバースプロキシ）

| レストラン | volta デプロイ |
|---|---|
| お客さん（食事する人） | インターネット上のユーザーの[ブラウザ](browser.ja.md) |
| 客席 | 公開向け[リバースプロキシ](reverse-proxy.ja.md) |
| 厨房のドア（ウェイターのみ） | [ForwardAuth](forwardauth.ja.md) 検証 |
| 厨房 | [Docker](docker.ja.md) 内部ネットワーク |
| シェフ | volta-auth-proxy |
| 冷蔵庫 | PostgreSQL データベース |
| お客さんは冷蔵庫に触れない | ブラウザはデータベースに直接クエリできない |

もし誰でも厨房に入れたら、料理を改ざん（データ変更）したり、食材を盗んだり（秘密を読む）、友達のために裏口を開けたり（バックドア作成）できてしまいます。

---

## なぜ必要なの？

ネットワーク分離がなければ：

- **データベースがインターネットに露出** -- 誰でも PostgreSQL に直接接続してパスワードをブルートフォースできる
- **内部 API が露出** -- volta の Internal API（信頼済み、認証なし）が誰でも呼び出せる
- **攻撃対象が倍増** -- すべてのサービスが潜在的な侵入口になる
- **横方向の移動** -- 1つのサービスが侵害されたら、攻撃者がすべてに到達できる

ネットワーク分離は**最小権限の原則**に従います：各コンポーネントは必要なものとだけ通信できます。

## volta-auth-proxy でのネットワーク分離

volta は [Docker](docker.ja.md) ネットワークで分離を実現します：

```
  インターネット
  ═══════════════════════════════════════
       │（ポート 443 のみ公開）
       ▼
  ┌─────────────────────────────────────┐
  │  リバースプロキシ（Traefik/Nginx）    │  ← 両方のネットワークに接続
  └─────────────────────────────────────┘
       │
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
       │  Docker 内部ネットワーク
       ▼
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ volta-auth   │  │  アプリ       │  │  PostgreSQL   │
  │ :8080        │  │  :3000       │  │  :5432        │
  └──────────────┘  └──────────────┘  └──────────────┘
       インターネットからアクセス不可
```

何が何と通信できるか：

| 送信元 | 送信先 | 許可? | 理由 |
|---|---|---|---|
| インターネット | リバースプロキシ (:443) | はい | これが正面玄関 |
| インターネット | volta (:8080) | **いいえ** | volta はプロキシの背後に置くべき |
| インターネット | PostgreSQL (:5432) | **いいえ** | DB は絶対にパブリックにしない |
| リバースプロキシ | volta | はい | ForwardAuth に必要 |
| リバースプロキシ | アプリ | はい | リクエストのプロキシ |
| volta | PostgreSQL | はい | セッション/ユーザーデータの保存 |
| volta | Google API | はい | [OIDC](oidc.ja.md) 認証 |
| アプリ | volta Internal API | はい | [ヘッダー](header.ja.md)や API 経由でユーザー情報取得 |
| アプリ | PostgreSQL | **場合による** | アプリ独自の DB がある場合も |

## 具体的な例

Docker Compose でのネットワーク分離設定：

```yaml
# docker-compose.yml（簡略化）
services:
  traefik:
    networks:
      - public      # インターネットからアクセス可能
      - internal    # volta とアプリに到達可能

  volta-auth-proxy:
    networks:
      - internal    # public ネットワークには接続しない
    # "ports:" セクションなし = ホストに公開されない

  app:
    networks:
      - internal
    # "ports:" セクションなし = ホストに公開されない

  postgres:
    networks:
      - internal
    # "ports:" セクションなし = ホストに公開されない

networks:
  public:
    # Traefik がここで 443 をバインド
  internal:
    # インターネットから完全に分離
```

ネットワーク分離がない場合に起きること：

1. 攻撃者がサーバーの IP をスキャンし、[ポート](port.ja.md) 5432 が開いているのを発見
2. 攻撃者が `psql -h your-server.com -U postgres` を実行 -- デフォルトパスワードを試す
3. 成功すると、すべてのセッションデータ、ユーザーデータ、テナントデータにフルアクセス
4. ゲームオーバー

適切な分離があれば、ステップ1で何も見つからない -- ポート 5432 は Docker ネットワーク内でのみ利用可能で、インターネットからのルートは存在しません。
