# no-store と no-cache の違い

[English / 英語](no-store-vs-no-cache.md)

---

## これは何？

`no-store` と `no-cache` は Cache-Control ヘッダに指定できるディレクティブですが、名前が似ているのに動作はまったく違います。`no-store` は「このレスポンスをどこにも保存するな」という意味です。`no-cache` は「保存してもいいけど、使う前に必ずサーバーに確認しろ」という意味です。この違いはセキュリティにとって非常に重要です。

---

## なぜ重要？

多くの開発者が `no-cache` を「キャッシュするな」という意味だと思い込んでいます。違います。`no-cache` が指定されたレスポンスは、ブラウザのディスクに保存される可能性があります。使うたびにサーバーへの確認が必要なだけです。つまりデータは物理的にブラウザのキャッシュフォルダに存在しています。共有パソコンなら、別のユーザーがそれを見つけるかもしれません。

認証ページや機密データには `no-cache` では不十分です。`no-store` が必要です。

---

## 簡単な例

| ディレクティブ | ブラウザがディスクに保存？ | 使う前にサーバーに確認？ | 認証ページに安全？ |
|-----------|----------------------|-------------------------------------|---------------------|
| `no-store` | しない | 該当なし（何も保存されない） | はい |
| `no-cache` | する | する（毎回） | いいえ -- データがディスクに残る |
| `max-age=0` | する | する | いいえ -- データがディスクに残る |
| (ヘッダなし) | する | たぶん | いいえ |

こう考えてください：

- **no-store** = 「読んだら書類をシュレッダーにかけろ」
- **no-cache** = 「書類はファイルキャビネットに保管していいが、使う前に電話して最新か確認しろ」

どちらもサーバーへの問い合わせは発生しますが、データがディスクに残らないのは `no-store` だけです。

---

## volta-auth-proxy での使い方

volta は最大限の安全のため両方のディレクティブを一緒に使います：

```java
ctx.header("Cache-Control", "no-store, no-cache, must-revalidate, private");
```

意図的に冗長にしています。各部分の意味：

- **no-store** -- 本命。レスポンスを一切保存しないようブラウザに指示
- **no-cache** -- 念のため。古いキャッシュは `no-store` を無視しても `no-cache` は尊重する場合がある
- **must-revalidate** -- エラー時でもキャッシュが古い内容を提供するのを防止
- **private** -- 共有プロキシや CDN がレスポンスを保存しないことを保証

volta はすべての認証関連エンドポイント（ログインページ、コールバックハンドラ、セッション一覧、ユーザーデータを含む API レスポンス）にこれを適用しています。静的アセット（CSS, JS）には別の緩いポリシーを使います。
