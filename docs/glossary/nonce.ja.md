# Nonce（ナンス / 一回限りの数値）

## これは何？

nonce とは、一回だけ使えるランダムな値のことです。名前は「number used once（一度だけ使われる数値）」に由来します。OIDC（OpenID Connect）の文脈では、nonce はアプリケーションが生成し、ログイン時に認証プロバイダに送り、戻ってくる id_token の中に埋め込まれているランダムな文字列です。トークン内の nonce とアプリが最初に送ったものが一致することを確認することで、このトークンがこの特定のログイン試行のために発行されたことを証明できます。

映画のチケットのシリアルナンバーのようなものです。映画館はチケットにユニークなシリアルナンバーを印刷します。入場時にシリアルナンバーを確認してチケットをちぎります。もし誰かがチケットをコピーしていても、シリアルナンバーは既に使用済みなので、コピーは拒否されます。

## なぜ重要？

nonce は id_token に対する**リプレイ攻撃**を防ぎます。リプレイ攻撃とは、有効なトークンを傍受して、後で（あるいは別の場面で）不正アクセスに再利用しようとする攻撃です。

nonce がない場合の攻撃シナリオ:

```
  1. Alice が OIDC でログインする
  2. Google が Alice 用の id_token を発行する
  3. 攻撃者がこの id_token を傍受する（ブラウザ履歴、ログファイル、
     ネットワーク盗聴など）
  4. その後、攻撃者がこの id_token をアプリに送る
  5. アプリがトークンを検証 -- 署名は正しい、期限内、
     発行者も一致、対象者も一致
  6. アプリが Alice のセッションを作成
  7. 攻撃者が Alice としてログインした状態になる
```

nonce があると、ステップ 5 で失敗します:

```
  アプリが確認: id_token 内の nonce が、
  このログイン試行のために保存した nonce と一致する？

  攻撃者のリクエストには、対応する nonce がサーバーに保存されていない
  （攻撃者はログインフローを開始していないから）。

  結果: 拒否。
```

### nonce と state の違いは？

よく混同されるポイントです。どちらも OIDC フローのランダムな値ですが、異なるレベルで異なるものを保護します:

| | State | Nonce |
|---|---|---|
| **保護対象** | リダイレクトフロー（HTTP レベル） | id_token（トークンレベル） |
| **防止する攻撃** | CSRF / ログイン CSRF 攻撃 | トークンのリプレイ攻撃 |
| **通る場所** | URL のクエリパラメータ | id_token の内部（claim として） |
| **検証タイミング** | トークン交換の前 | トークン交換の後 |
| **IdP での扱い** | 変更せずそのまま通す | 署名済みトークンの中に埋め込む |

こう考えると分かりやすいです:
- **State** は、リダイレクトがアプリが開始したログインフローから来たことを確認する。
- **Nonce** は、id_token がそのログインフロー専用に発行されたことを確認する。

両方必要です。フローの異なるポイントで起こり得る異なる攻撃から保護するためです。

## どう動くの？

```
  あなたのアプリ             Google                あなたのアプリ
  (ログイン開始)             (認証プロバイダ)        (コールバック)
  +----------------+       +----------------+     +----------------+
  |                |       |                |     |                |
  | 1. nonce=XYZ   |       |                |     | 5. id_token を  |
  |    を生成       |       |                |     |    パース       |
  |                |       |                |     |                |
  | 2. nonce を     |       |                |     | 6. トークンの   |
  |    DB に保存    |       |                |     |    claims から  |
  |                |       |                |     |    nonce を取得 |
  | 3. Google に    | ----> | 4. ユーザーが   |     |                |
  |    nonce=XYZ   |       |    ログイン。    | --> | 7. 比較:       |
  |    を含めて     |       |    Google が    |     |    トークンの   |
  |    リダイレクト  |       |    nonce=XYZ   |     |    nonce ==    |
  |                |       |    を id_token  |     |    保存した     |
  |                |       |    に埋め込む   |     |    nonce?      |
  +----------------+       +----------------+     +----------------+
                                                        |
                                                   一致? -> OK
                                                   不一致? -> 拒否
```

ステップごとの説明:

1. アプリが暗号学的にランダムな文字列（nonce）を生成する。
2. アプリが nonce をサーバーサイドに保存し、このログイン試行に関連付ける。
3. アプリが Google に送る認可 URL のパラメータとして `nonce=XYZ` を含める。
4. Google がユーザーを認証し、id_token を作成する。Google は署名する前に nonce の値を id_token の claim として埋め込む。nonce は署名済みトークンの中にあるので、署名を無効にせずに変更することはできない。
5. アプリがコールバックで id_token を受け取る。
6. アプリが id_token をパースし、中の `nonce` claim を読む。
7. アプリがこの nonce をステップ 2 で保存したものと比較する。一致すれば、トークンは新鮮で、この特定のログイン試行のために発行されたものだと分かる。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は Google OIDC 連携の一部として nonce の生成と検証を行います。

**生成。** `OidcService.createAuthorizationUrl()` で、volta は `SecurityUtils.randomUrlSafe(32)` を使って nonce を生成します。state パラメータと同じ暗号学的に強力な乱数生成器を使用しています。32 バイト（256 ビット）のランダム値を URL セーフな Base64 文字列としてエンコードします。

**保存。** nonce は `OidcFlowRecord` の一部として、state、PKCE 検証値、リターン URL、有効期限とともにデータベースに保存されます。state の値がレコード全体の検索キーになります。

**認可 URL への含め方。** nonce はクエリパラメータとして認可 URL に追加されます:
```
https://accounts.google.com/o/oauth2/v2/auth?
  ...
  &nonce=<ランダムな値>
  &state=<ランダムな値>
  ...
```

**検証。** コールバックが届くと、volta は:
1. state パラメータを使ってデータベースから `OidcFlowRecord` を検索する。
2. 認可コードをトークンと交換し、Google から id_token を受け取る。
3. Google の JWKS を使って id_token の署名を検証する。
4. カスタム claims 検証の中で、id_token の `nonce` claim が `flow.nonce()`（データベースレコードの nonce）と一致するか確認する。
5. トークンに nonce がないか、一致しない場合、volta は「Invalid nonce」エラーを投げてログイン失敗となる。

検証コードはこのチェックを行います:
```
String nonce = (String) claims.getClaim("nonce");
if (nonce == null || !nonce.equals(expectedNonce)) {
    throw new IllegalArgumentException("Invalid nonce");
}
```

**一回限りの使用。** state の検証時に `OidcFlowRecord` が消費（データベースから削除）されるため、nonce も自動的に消費されます。同じ id_token を volta に対してリプレイすることはできません。その id_token に含まれる nonce は、もうデータベースに存在しないからです。

## よくある間違い

**1. nonce を使わない。**
一部の OAuth/OIDC 実装は nonce を省略しています。これでは id_token がリプレイ攻撃に対して脆弱になります。OIDC 仕様はインプリシットフローで nonce を必須とし、認可コードフローでも推奨しています。volta は常に使います。

**2. state と nonce に同じ値を使う。**
技術的には OIDC 仕様で禁止されていませんが、同じ値を使うと、一方が漏洩した場合に両方の保護が侵害されます。volta はそれぞれ独立に生成します。

**3. id_token を受け取った後に nonce を検証しない。**
nonce を生成して送るのは半分の仕事です。トークンが返ってきたときに確認しなければ、意味がありません。

**4. nonce をブラウザだけに保存する。**
nonce が localStorage や通常の Cookie にあると、XSS アクセスを持つ攻撃者が読み取って有効なリプレイを作れてしまいます。volta は nonce をサーバーサイドのデータベースに保存します。

**5. 予測可能な nonce を使う。**
nonce は推測不可能でなければなりません。タイムスタンプ、カウンター、既知のデータのハッシュを使うと、攻撃者が予測できてしまいます。volta は 32 バイトの暗号学的ランダムネスを使います。

**6. 使用後に nonce を消費しない。**
最初の使用後も nonce が有効なままだと、id_token をリプレイできてしまいます。volta の検索時消費パターンにより、各 nonce は正確に一回だけ機能します。
