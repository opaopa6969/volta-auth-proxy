# OPA (Open Policy Agent)

[English version](opa.md)

---

## これは何？

Open Policy Agent（OPA、「オーパ」と発音）は、Cloud Native Computing Foundation（CNCF）の汎用ポリシーエンジンです。**Rego**という専用言語で認可ルールを記述し、構造化データに対して評価できます。OPAはスタンドアロンサービス（またはサイドカー）として動作し、HTTP APIを公開します。アプリケーションが「誰が何をしたいか」を記述するJSONリクエストを送信し、OPAが「許可」または「拒否」で応答します。

OPAは法廷の裁判官のようなものです。アプリケーション（検察官）が事件の事実を提示します：「ユーザーaliceがテナントacmeのリソース/api/users/42をDELETEしたい」。裁判官（OPA）は法律書（Regoポリシー）を参照して判決を下します：許可か拒否か。重要な違いは、裁判官は検察官のために働く人ではない別の人物であること -- OPAはアプリケーションとは別のプロセスです。

これは[jCasbin](jcasbin.md)のようなインプロセスライブラリとは根本的に異なります。jCasbinでは認可ロジックがアプリケーション内で実行されますが、OPAは常にネットワーク呼び出しを挟みます。

---

## なぜ重要なのか？

認可の決定はあらゆるところで行われます -- API、Kubernetesアドミッションコントロール、CI/CDパイプライン、データベースクエリ、インフラプロビジョニング。統一されたポリシーエンジンがなければ、各システムが独自の言語で独自の認可ロジックを実装し、「誰が何をできるか」を監査する単一の場所がありません。

OPAは単一のポリシー言語（Rego）と単一の評価エンジンをこれらすべてのコンテキストに提供します。ポリシーを一度書けば、どこでも適用できます。多くのサービスを持ち厳格なコンプライアンス要件がある組織にとって強力です。

OPAはCNCF **卒業**プロジェクト（Kubernetes、Prometheus、Envoyと並ぶ最高の成熟度レベル）です。これは幅広い業界での採用と長期的な安定性を示しています。

ただし、OPAの力には複雑さが伴います。RegoはJava、Python、JavaScriptとは似ても似つかない宣言型論理プログラミング言語です。学習曲線は急で、Regoポリシーのデバッグは困難な場合があります。

## どう動くのか？

### アーキテクチャ：サイドカーパターン

```
  ┌─────────────────┐         ┌──────────────┐
  │  あなたのアプリ    │  HTTP   │     OPA      │
  │                  │────────►│              │
  │  「aliceは       │         │  Regoポリシー │
  │   /api/users/42  │         │  を評価       │
  │   をDELETEできる？」│        │              │
  │                  │◄────────│  { "allow":  │
  │                  │         │    true }     │
  └─────────────────┘         └──────────────┘
```

OPAは別プロセスとして動作します（多くの場合Kubernetesのサイドカーコンテナ）。アプリケーションは入力データを含むHTTP POSTリクエストをOPAのREST APIに送信し、OPAはその入力に対してポリシーを評価します。

### Rego言語

RegoはOPAのポリシー言語です。宣言型で、計算方法ではなく何が真であるべきかを記述します。

```rego
package volta.authz

default allow = false

# 管理者は何でもできる
allow {
    input.user.role == "admin"
}

# ユーザーは自分のテナントのリソースを読める
allow {
    input.action == "GET"
    input.user.tenant_id == input.resource.tenant_id
}

# テナントオーナーは自テナントのユーザーを管理できる
allow {
    input.action in ["GET", "POST", "DELETE"]
    input.resource.path == "/api/users"
    input.user.role == "owner"
    input.user.tenant_id == input.resource.tenant_id
}
```

### 入力とデータ

OPAは**入力**（評価する特定のリクエスト）と**データ**（ロール割り当て、リソースメタデータなどの参照情報）を分離します：

| 概念 | 説明 | 例 |
|------|------|-----|
| **入力** | 評価するリクエスト。HTTPで各リクエストごとに提供。 | `{"user": {"email": "alice@acme.com", "role": "admin"}, ...}` |
| **データ** | 事前にロードされる参照データ。バンドル、ファイル、APIから取得可能。 | ロール定義、テナント設定、リソース階層 |
| **ポリシー** | 認可ロジックを定義するRegoルール。起動時またはバンドル経由でロード。 | 上記の`.rego`ファイル |

### OPA vs jCasbin vs Cedar

| 機能 | OPA | [jCasbin](jcasbin.md) | Cedar (AWS) |
|------|-----|---------|-------|
| デプロイ | サイドカー / デーモン | インプロセスライブラリ | ライブラリまたはサービス |
| 言語 | Rego（宣言型、論理ベース） | PERMモデル + CSV/DB | Cedar（型付き、構造化） |
| 開発言語 | Go | Java | Rust |
| レイテンシ | 1-5ms（HTTP呼び出し） | サブミリ秒 | サブミリ秒 |
| CNCFステータス | 卒業 | サンドボックス（Casbin） | N/A |
| 適用範囲 | 汎用（authz、K8s、CI/CDなど） | 認可のみ | 認可のみ |
| 学習コスト | 高い（Rego） | 低い（モデルDSL） | 中程度 |

## volta-auth-proxy ではどう使われている？

OPAはvolta-auth-proxyのPhase 4の**代替ポリシーエンジン**として評価されています。[jCasbin](jcasbin.md)が推奨オプションですが、OPAはインフラの他の場所ですでにOPAを使用しており単一のポリシーエンジンを維持したい組織のために文書化されています。

### voltaがOPAよりjCasbinを推奨する理由

voltaの核心的な哲学はシンプルさとコントロールです：

1. **ネットワークホップなし**：jCasbinはインプロセスで動作。OPAは認可判断のたびにHTTP呼び出しが必要。
2. **新しい言語なし**：jCasbinのPERMモデルはシンプルな設定。Regoは独自の学習曲線を持つ完全なプログラミング言語。
3. **サイドカー管理なし**：jCasbinはMaven依存関係。OPAはデプロイ、監視、保守が必要な別のコンテナ。
4. **デバッグ**：jCasbinはIntelliJでステップ実行可能。OPAには別のツールが必要。

### OPAがvoltaで意味を持つケース

- 組織がすでにKubernetesアドミッションコントロールにOPAを使用している
- volta専用ではなくインフラ全体で単一のポリシーエンジンが欲しい
- ポリシーがRegoの表現力から恩恵を受けるほど複雑
- OPAを管理する専任のプラットフォームチームがある

### 統合パターン（選択した場合）

```
  ブラウザ ──► Traefik ──► volta-auth-proxy
                               │
                               │  1. 認証（セッション/JWT）
                               │  2. 認可コンテキストを構築
                               │
                               ▼
                          ┌────────┐
                          │  OPA   │  （サイドカー）
                          │        │
                          │  POST  │  /v1/data/volta/authz
                          │  input:│  {user, tenant, resource, action}
                          │        │
                          │  resp: │  {allow: true/false}
                          └────────┘
                               │
                               ▼
                          3. 許可: 200 + ヘッダー
                             拒否: 403
```

## よくある間違いと攻撃

### 間違い1：デフォルトを許可にする

Regoでは、ルールに一致しない場合、結果は`undefined`です。アプリケーションが`undefined`を「許可」として扱うと、未処理のすべてのケースがセキュリティホールになります。常に`default allow = false`を設定してください。

### 間違い2：ポリシーをテストしない

Regoポリシーには微妙なバグがあり得ます。OPAには組み込みテストフレームワークがあります -- 使いましょう。すべてのポリシールール、特にエッジケースのテストケースを書いてください。

### 間違い3：OPAのレイテンシを無視する

すべての認可チェックはHTTPラウンドトリップです。1リクエストあたり10回の認可チェックで1回2msなら、20msのレイテンシが追加されます。判断のバッチ処理や短期間の結果キャッシュを検討してください。

### 間違い4：OPAのAPIを外部に公開する

OPAのREST APIは任意の入力で任意のポリシーをクエリできます。インターネットに公開すると、攻撃者が認可ロジックを調査してギャップを見つけられます。OPAは内部ネットワークに置いてください。

### 攻撃：バンドル経由のポリシー改ざん

攻撃者がバンドルサーバーを侵害すると、すべてを許可するポリシーをプッシュできます。バンドルに署名し、OPAで署名を検証してこれを防いでください。
