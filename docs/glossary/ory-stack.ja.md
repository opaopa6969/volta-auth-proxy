# Ory Stack

[English version](ory-stack.md)

---

## これは何？

Ory Stackは、Go言語で構築されたオープンソースのIDインフラストラクチャで、4つの独立しつつ補完的なサービスで構成されています：**Kratos**（ID/ユーザー管理）、**Hydra**（OAuth2/OIDCサーバー）、**Oathkeeper**（ID認識リバースプロキシ）、**Keto**（認可/権限）。これらを合わせると、[Keycloak](keycloak.md)や[ZITADEL](zitadel.md)のようなモノリシックなIDプラットフォームに対するモジュラーな代替手段となります。

総合建設会社を1社雇う代わりに、専門業者を個別に雇って家を建てるようなものです。ある業者は配管を担当（Kratos -- ユーザーアカウント）、別の業者は電気を担当（Hydra -- OAuth2）、別の業者はセキュリティシステムを担当（Oathkeeper -- アクセスプロキシ）、さらに別の業者はどの部屋に誰が入れるかのルールを担当（Keto -- 権限）。各業者は専門分野で優秀ですが、自分で調整する必要があります。

このモジュラーアプローチがOryの決定的な特徴です。Hydraなしでユーザー管理のためにKratosだけを使用できます。Kratosなしでハイドラ OAuth2サーバーとしてHydraだけを使用できます。各サービスは独自のデータベース、独自のAPI、独自のデプロイライフサイクルを持っています。

## なぜ重要なのか？

volta-auth-proxyの文脈では、Oryは**競合製品**であり、代替アーキテクチャ哲学の例として重要です。

ID管理の世界には2つの陣営があります：

1. **モノリシックプラットフォーム**：Keycloak、ZITADEL、Auth0 -- すべてを行う1つの大きなシステム
2. **モジュラーツールキット**：Ory Stack -- 組み合わせる個別のサービス

volta-auth-proxyは第3のアプローチを取ります：認証と基本的な認可を行う**単一の特化サービス**で、それ以外はすべて既存のインフラに委任します。

Oryを理解するとトレードオフがわかります：

| アプローチ | メリット | デメリット |
|-----------|---------|-----------|
| モノリシック（Keycloak） | すべて統合、1つのデプロイ | 複雑、リソース大、カスタマイズ困難 |
| モジュラー（Ory） | 必要なものを選択、不要なものは置換 | 調整のオーバーヘッド、複数デプロイ |
| 特化型（volta） | シンプル、軽量、完全制御 | 自分で構築する部分が多い、箱出し機能が少ない |

## どう動くのか？

### 4つのサービス

#### Kratos -- ID管理

Kratosはユーザー登録、ログイン、パスワード回復、プロフィール管理、MFAを処理します。OAuth2やトークン発行は処理しません -- それはHydraの仕事です。

重要な設計決定：Kratosは**UIを提供しません**。表示すべきフィールドを記述するJSONを返すAPIを公開します。UIは自分で構築します。見た目の完全な制御を得られますが、より多くの作業を意味します。

#### Hydra -- OAuth2/OIDCサーバー

Hydraは認定済みのOAuth 2.0およびOpenID Connectサーバーです。トークン発行、同意フロー、クライアント管理を処理します。ユーザー管理は行いません -- ログインと同意を外部サービス（Kratosなど）に委任します。

#### Oathkeeper -- ID認識プロキシ

Oathkeeperはサービスの前に立ち、アクセスルールを適用します。volta-auth-proxyの[ForwardAuth](forwardauth.md)パターンに似ていますが、独自のルール設定システムを持っています。

#### Keto -- 権限/認可サービス

KetoはGoogleのZanzibar権限モデルを実装しています -- 「aliceは組織acmeのメンバーか？」のような関係をチェックするグラフベースの認可システムです。

### Ory Stack vs 競合製品

| 観点 | Ory Stack | [Keycloak](keycloak.md) | [ZITADEL](zitadel.md) | volta-auth-proxy |
|------|-----------|---------|---------|-----------------|
| アーキテクチャ | 4マイクロサービス | モノリス | モノリス（イベントソース） | 単一サービス |
| 言語 | Go | Java | Go | Java |
| デプロイの複雑さ | 高い（4サービス） | 中程度（1サービス） | 中程度（1サービス） | 低い（1サービス） |
| UI提供 | なし（ヘッドレス） | 組み込み（FreeMarker） | 組み込み（Console） | jteテンプレート |
| OAuth2/OIDC | Hydra（認定済み） | 組み込み | 組み込み | Googleに直接 |
| ユーザー管理 | Kratos | 組み込み | 組み込み | DB + 管理API |
| 認可モデル | Keto（Zanzibar） | レルムロール | Organizations | [jCasbin](jcasbin.md)（Phase 4） |
| 合計メモリ | ~500MB-1GB（全4つ） | ~512MB-2GB | ~200-500MB | ~30-50MB |
| 哲学 | モジュラーマイクロサービス | エンタープライズモノリス | クラウドネイティブモノリス | 構成可能なライブラリ |

## volta-auth-proxy ではどう使われている？

volta-auth-proxyはOry Stackを**使用しません**。根本的に異なる哲学を持つ競合製品です。

### Oryが存在するのにvoltaが存在する理由

Oryは印象的なエンジニアリングですが、voltaはその前提のいくつかを拒否しています：

1. **4サービスは多すぎる**：Kratos + Hydra + Oathkeeper + Ketoの管理は、4つのDockerコンテナ、4つのデータベース（または共有）、4セットの設定、4つのアップグレードサイクル、4セットの監視すべきログを意味します。voltaは1つのJARです。

2. **ヘッドレスUIはより多くの作業を意味する**：Kratosはログインを提供しません。セルフサービスフローAPIを使って自分で構築する必要があります。voltaはすぐに使えるjteテンプレートを提供し、直接カスタマイズできます。

3. **Goサイドカーパターンが合わない**：OryのサービスはGoの別プロセスとして動作します。Javaショップにとって、これはJavaアプリケーションと並行してGoバイナリを管理し、言語境界を超えてデバッグし、Oryの設定形式を学ぶことを意味します。voltaは最初から最後までJavaです。

4. **ZanzibarはほとんどのサaaSにはオーバーキル**：KetoはGoogleのZanzibar権限モデルを実装していますが、これはGoogleスケールの関係ベース認可用に設計されています。ほとんどのSaaSアプリケーションにはテナント付きRBACが必要で、[jCasbin](jcasbin.md)がはるかに少ない複雑さで処理します。

### voltaよりOryを選ぶべきケース

- 認定済みOAuth2/OIDCサーバー（Hydra）が必要
- Google Zanzibarスタイルの認可が欲しい
- すでにマイクロサービスアーキテクチャを運用していて、4つ追加するのが普通
- ヘッドレス認証が必要（サーバーレンダリングUIなし、すべてAPI駆動）
- Go指向のチームがある

### OryよりvoltaをChooseするケース

- 4つではなく1つのサービスが欲しい
- サーバーレンダリングのログインページ（jteテンプレート）が欲しい
- Javaを好み、すべてをIDEでデバッグしたい
- 最小限のリソース使用量が必要
- テナント付きRBACが認可ニーズに十分
- 認証コードのすべての行を理解・制御したい

## よくある間違いと攻撃

### 間違い1：1つしか必要ないのに4つすべてをデプロイする

多くのチームがKratos（ユーザー管理）やHydra（OAuth2）だけが必要な時にOry Stack全体をデプロイします。各サービスは独立しています -- 必要なものだけを使ってください。

### 間違い2：調整の複雑さを過小評価する

4つのサービスは相互に通信する必要があります。KratosはOAuth2フローのためにHydraを知る必要があります。Oathkeeperは両方を知る必要があります。サービス間の設定エラーは、認証は通るが認可がスキップされるという微妙なセキュリティホールを作りかねません。

### 間違い3：UIレイヤーを構築しない

Kratosはヘッドレスです -- ログイン/登録UIを提供しません。これを計画しないチームは、KeycloakやvoltaがすぐExperience提供するフォーム、エラー処理、MFAフロー、パスワード回復UIの構築に何週間も費やします。

### 間違い4：移行の複雑さを無視する

各Oryサービスは独自のDBスキーマとマイグレーションシステムを持っています。4つのサービスを同期してアップグレードし、4つのDBのスキーママイグレーションを処理し、1つが失敗した場合にロールバックするのは運用上複雑です。

### 攻撃：サービス間のなりすまし

Oryサービス間の内部ネットワークが保護されていない場合、1つのサービスを侵害した攻撃者が別のサービスになりすますことができます。例えば、HydraへのKratosログイン応答を偽造する。サービス間通信にはmTLSまたはサービスメッシュを使用してください。
