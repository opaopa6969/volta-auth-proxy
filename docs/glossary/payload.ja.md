# ペイロード

[English version](payload.md)

---

## これは何？

ペイロードは、メッセージの中に運ばれる実際のデータコンテンツです。メッセージを説明するメタデータ（ヘッダー、封筒情報）とは対照的です。ネットワーキングでは、ペイロードは「貨物」です。実際に関心のあるもの。それ以外（ヘッダー、ルーティング情報、プロトコルフレーミング）は貨物を目的地に届けるための包装にすぎません。

配送パッケージのようなものです。外側のラベル（送信者住所、受取人住所、追跡番号、「壊れ物」シール）はメタデータです。箱の中のもの、実際に注文した品物がペイロードです。パッケージを受け取ったとき、中身に関心があり、ラベルには関心がありません。

volta-auth-proxyでは、ペイロードはあらゆる場所に現れます：JWTペイロードはユーザークレームを運び、HTTPペイロードはリクエスト/レスポンスボディを運び、[Webhook](webhook.md)ペイロードはイベントデータを運び、[SCIM](scim.md)ペイロードはユーザープロビジョニングコマンドを運びます。

---

## なぜ重要なのか？

ペイロードとメタデータの区別を理解することは、セキュリティ、デバッグ、統合に不可欠です：

```
  HTTPリクエストの構造：
  ┌──────────────────────────────────────────────┐
  │  メタデータ（ヘッダー、ルーティング）：         │
  │  ┌──────────────────────────────────────────┐│
  │  │ POST /api/v1/tenants/acme/members        ││
  │  │ Host: volta.example.com                   ││
  │  │ Content-Type: application/json            ││
  │  │ Authorization: Bearer eyJhbGci...         ││
  │  │ X-Volta-Request-Id: req-uuid              ││
  │  └──────────────────────────────────────────┘│
  │                                               │
  │  ペイロード（実際のデータ）：                   │
  │  ┌──────────────────────────────────────────┐│
  │  │ {                                         ││
  │  │   "email": "alice@acme.com",             ││
  │  │   "role": "MEMBER"                        ││
  │  │ }                                         ││
  │  └──────────────────────────────────────────┘│
  └──────────────────────────────────────────────┘
```

- **セキュリティ**：ペイロードに暗号化や署名が必要な機密データが含まれる場合がある
- **バリデーション**：処理前にペイロードを検証する必要がある（型チェック、サイズ制限）
- **ログ記録**：メタデータは自由にログ記録できるが、ペイロードは注意（PIIを含む可能性）
- **デバッグ**：ほとんどのバグはヘッダー処理ではなくペイロード処理にある

## どう動くのか？

### JWTペイロード

[JWT](jwt.md)は3つの部分があります：ヘッダー、ペイロード、署名。ペイロードは[base64url](base64.md)でエンコードされたJSONオブジェクトです：

```
  JWT: eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOi...  .signature
       ├── header ────────┤ ├── payload ──┤  ├── sig ──┤

  デコードされたペイロード：
  {
    "iss": "volta-auth",           ← 発行者
    "sub": "user-uuid",            ← 主体（誰）
    "aud": ["volta-apps"],         ← 対象者（誰のため）
    "exp": 1711900000,             ← 有効期限
    "iat": 1711899700,             ← 発行日時
    "volta_v": 1,                  ← voltaバージョン
    "volta_tid": "acme-uuid",      ← テナントID
    "volta_roles": ["ADMIN"],      ← ユーザーロール
    "volta_email": "a@acme.com"    ← メール
  }
```

JWTペイロードの各フィールドは「クレーム」と呼ばれます。voltaは標準クレームとの衝突を避けるため`volta_`プレフィックス付きのカスタムクレームを追加します。

### HTTPリクエストペイロード

HTTPリクエストボディがペイロードです：

```
  POST /scim/v2/Users
  Content-Type: application/json
  Content-Length: 187

  {                          ←─┐
    "schemas": [...],           │
    "userName": "alice@a.com",  │  ペイロード
    "name": {                   │
      "givenName": "Alice",     │
      "familyName": "Smith"     │
    },                          │
    "active": true              │
  }                          ←─┘
```

### Webhookペイロード

voltaが[Webhook](webhook.md)を送信するとき、HTTPボディがペイロードです：

```
  POST https://app.com/webhooks/volta
  Content-Type: application/json
  X-Volta-Signature: sha256=abc123    ← メタデータ
  X-Volta-Event: user.created         ← メタデータ

  {                                    ←─┐
    "event": "user.created",              │
    "timestamp": "2026-04-01T12:00:00Z",  │ ペイロード
    "tenant_id": "acme-uuid",             │
    "data": {                             │
      "user_id": "user-uuid",            │
      "email": "alice@acme.com",          │
      "roles": ["MEMBER"]                 │
    }                                     │
  }                                    ←─┘
```

ヘッダー内のHMAC署名はこのペイロードに対して計算され、完全性を保証します。

### ペイロードサイズの考慮事項

```
  ┌──────────────────────────────────────────────┐
  │  voltaのペイロードサイズ制限：                  │
  │                                               │
  │  JWTペイロード：   典型的に ~2 KB              │
  │    （Bearerトークンとしてヘッダーに収まる必要）  │
  │                                               │
  │  Webhookペイロード：最大64 KB                  │
  │    （イベントデータは簡潔であるべき）            │
  │                                               │
  │  SCIMペイロード：  最大256 KB                  │
  │    （ユーザー/グループオブジェクトは大きくなり得る）│
  │                                               │
  │  APIリクエスト：   最大1 MB                    │
  │    （一般的なリクエストボディ制限）              │
  └──────────────────────────────────────────────┘
```

## volta-auth-proxyではどう使われている？

### JWTペイロードクレーム（volta固有）

voltaはJWTペイロードにカスタムクレームを追加します：

| クレーム | 型 | 例 | 目的 |
|--------|------|---------|---------|
| `volta_v` | int | `1` | ペイロード形式バージョン |
| `volta_tid` | string | `"acme-uuid"` | [テナント](tenant.md)ID |
| `volta_roles` | string[] | `["ADMIN"]` | [ロール](role.md) |
| `volta_email` | string | `"a@acme.com"` | ユーザーメール |
| `volta_client` | boolean | `true` | [M2M](m2m.md)フラグ |
| `volta_client_id` | string | `"billing-svc"` | M2Mクライアント名 |

### Webhookペイロード構造

すべてのvolta Webhookペイロードは同じ構造に従います：

```json
{
  "event": "event.type",
  "timestamp": "ISO-8601",
  "delivery_id": "uuid",
  "tenant_id": "uuid",
  "data": {
    // イベント固有のフィールド
  }
}
```

この一貫した構造により、Webhookハンドラは同じ基本ロジックで任意のvoltaイベントをパースできます。

### ペイロードバリデーション

voltaは複数のレベルで受信ペイロードを検証します：

```
  1. Content-Typeチェック
     → application/jsonであること（SCIMはapplication/scim+json）

  2. サイズチェック
     → 制限を超えるペイロードを拒否

  3. JSONパース
     → 有効なJSONであること（末尾カンマなし、コメントなし）

  4. スキーマバリデーション
     → 必須フィールドの存在、正しい型

  5. ビジネスバリデーション
     → メールが有効な形式、ロールが存在、テナントがアクティブ
```

### [アップストリーム](upstream.md)転送でのペイロード

voltaがリクエストをアップストリームアプリに転送する際、元のペイロードを変更せずに通過させます。voltaはリクエストボディを変更しません。認証情報付きの[ヘッダー](header.md)を追加するだけです：

```
  クライアント → volta（X-Volta-*ヘッダーを追加） → アップストリーム
                ペイロードは変更なしで通過
```

## よくある間違いと攻撃

### 間違い1：完全なペイロードをログに記録する

ペイロードにはPII（メール、名前、トークン）が含まれることが多い。メタデータ（メソッド、パス、ステータスコード）はログに記録し、本番ログではペイロードの内容を編集または省略してください。

### 間違い2：ペイロードサイズを検証しない

攻撃者が100 MBのペイロードを送ってメモリを枯渇させる可能性。パース前に常にサイズ制限を適用してください。

### 間違い3：署名検証なしでJWTペイロードを信頼する

JWTペイロードは[base64url](base64.md)エンコードされているだけで、暗号化されていません。誰でもデコードして読めます。攻撃者も変更できます。クレームを信頼する前に常に署名を検証してください。

### 間違い4：ペイロードにシークレットを含める

Webhookペイロードやクレームにパスワード、APIキー、その他のシークレットを入れないでください。値の代わりに参照（ID）を使ってください。

### 攻撃：ペイロードインジェクション

攻撃者が予期しないフィールド（例：`"role": "ADMIN"`）を含むJSONペイロードを作成し、サーバーが盲目的にすべてのフィールドをマージすることを期待する。防御：期待されるフィールドのみ読み取り、余分なものを無視する厳密なデシリアライゼーションを使用。

### 攻撃：巨大ペイロード（DoS）

攻撃者が数ギガバイトのペイロードを送ってサーバーをクラッシュさせる。防御：ボディを読み取る前に`Content-Length`制限を適用。voltaはデフォルトでリクエストボディを制限。
