# ペネトレーションテスト

[English version](penetration-test.md)

---

## これは何？

ペネトレーションテスト（ペンテスト）は、実際の攻撃者より先にセキュリティ脆弱性を見つけるための、認可されたシステムへの模擬攻撃です。訓練されたセキュリティ専門家が、実際の攻撃者と同じ手法 -- SQLインジェクション、XSS、認証バイパス、権限昇格 -- を使ってアプリケーションへの侵入を試み、見つかったすべてをレポートにまとめます。

鍵師を雇って自宅への侵入を試みてもらうようなものです。許可を与え、すべての窓、すべての鍵、すべての侵入経路を試してもらいます。終わったらレポートを渡されます：「裏口の鍵は30秒でピッキングできます。ガレージのコードは工場出荷時のデフォルトです。でも正面のデッドボルトは堅固です。」これで本物の泥棒が来る前に何を直すべきかわかります。

キーワードは「認可された」です。許可なしにこれを行えばハッキングです。許可と正式なスコープ合意があれば、ペネトレーションテストです。

---

## なぜ重要？

すべてのアプリケーションには脆弱性があります。問題は、あなたが先に見つけるか攻撃者が見つけるかです。ペネトレーションテストが重要な理由：

- **開発者はコードに近すぎる。** システムがどう動くべきかを考えます。ペンテスターはどう壊せるかを考えます。
- **自動スキャナーはロジックの欠陥を見逃す。** スキャナーは欠落したヘッダーを見つけられますが、「招待が期限切れ後も受理できる」や「ユーザーが自分のロールを昇格できる」は見つけられません。
- **コンプライアンスが要求する。** SOC 2、PCI DSS、多くのエンタープライズ契約は定期的なペネトレーションテストを要求します。
- **信頼を構築する。** クリーンな結果のペンテスト後、顧客に「独立したセキュリティ企業がシステムをテストした」と伝えられます。

## どう動くのか？

### ペネトレーションテストの種類

| 種類 | テスターの知識 | シミュレート対象 |
|------|-------------|----------------|
| **ブラックボックス** | システムの知識なし。URL のみ提供 | 内部情報なしの外部攻撃者 |
| **グレーボックス** | 部分的な知識。ユーザーアカウントと API ドキュメント提供 | 基本的なアクセスを持つ攻撃者（例：顧客） |
| **ホワイトボックス** | ソースコード、アーキテクチャ、DB スキーマへのフルアクセス | 内部脅威または執念深い攻撃者 |

### ペネトレーションテストのプロセス

```
  フェーズ1：スコーピング
  ┌────────────────────────────────────┐
  │  スコープ内は？                     │
  │  - Web アプリケーション             │
  │  - API エンドポイント               │
  │  - 認証フロー                       │
  │                                    │
  │  スコープ外は？                     │
  │  - サードパーティサービス（Google） │
  │  - 物理インフラ                     │
  │  - ソーシャルエンジニアリング       │
  └────────────────────────────────────┘
              │
              ▼
  フェーズ2：偵察
  ┌────────────────────────────────────┐
  │  - すべてのエンドポイントをマップ   │
  │  - 使用技術を特定                   │
  │  - 侵入ポイントを発見              │
  │  - 認証フローをレビュー            │
  └────────────────────────────────────┘
              │
              ▼
  フェーズ3：脆弱性の発見
  ┌────────────────────────────────────┐
  │  - インジェクションテスト（SQL, XSS）│
  │  - 認証バイパステスト              │
  │  - 認可テスト（IDOR、権限昇格）    │
  │  - セッション管理テスト            │
  │  - CSRF 対策テスト                 │
  │  - 情報漏洩テスト                  │
  └────────────────────────────────────┘
              │
              ▼
  フェーズ4：エクスプロイト
  ┌────────────────────────────────────┐
  │  発見した脆弱性の悪用を試行：      │
  │  - 別テナントのデータにアクセス    │
  │    できるか？                      │
  │  - MEMBER から OWNER に昇格        │
  │    できるか？                      │
  │  - 認証プロキシをバイパス          │
  │    できるか？                      │
  └────────────────────────────────────┘
              │
              ▼
  フェーズ5：レポート
  ┌────────────────────────────────────┐
  │  各発見事項について：              │
  │  - 深刻度（Critical/High/Med/Low） │
  │  - 説明                            │
  │  - 概念実証                        │
  │  - 修正の推奨事項                  │
  └────────────────────────────────────┘
```

### 深刻度の評価

| 深刻度 | 例 | 影響 |
|--------|-----|------|
| **Critical** | データ抽出可能な SQL インジェクション | 全データ漏洩 |
| **High** | 認証バイパス | すべてのアカウントへの不正アクセス |
| **Medium** | ロール変更エンドポイントの CSRF | 攻撃者が権限を変更可能 |
| **Low** | スタックトレースを露出する詳細なエラーメッセージ | 情報漏洩 |
| **Informational** | セキュリティヘッダーの欠如 | 軽微なハードニング機会 |

## volta-auth-proxy ではどう使われている？

volta-auth-proxy はまだ正式なペネトレーションテストを受けていません。Phase 2/3 で計画されています。しかし、volta のアーキテクチャはペンテスト対応を念頭に設計されています。

### volta のペンテストで狙われるもの

| 攻撃面 | ペンテスターがテストすること | volta の防御 |
|--------|---------------------------|-------------|
| **OIDC コールバック** | コールバックを偽造できるか？State の改ざんは？ | State パラメーター、nonce 検証 |
| **セッション管理** | セッション固定？セッションハイジャック？ | ログイン時の再生成、HttpOnly/Secure/SameSite |
| **CSRF** | フォームをクロスオリジンから送信できるか？ | トークンベース + SameSite 防御 |
| **テナント分離** | ユーザー A がテナント B のデータにアクセスできるか？ | `enforceTenantMatch()`、行レベルフィルタリング |
| **ロール昇格** | MEMBER が OWNER になれるか？ | すべてのリクエストでサーバーサイドのロールチェック |
| **招待システム** | 期限切れの招待を使えるか？招待をブルートフォースできるか？ | 有効期限チェック、暗号化招待コード |
| **API エンドポイント** | インジェクション？マスアサインメント？IDOR？ | 入力検証、明示的フィールドマッピング |
| **ヘルスチェック** | /healthz が情報を漏洩するか？ | `{"status":"ok"}` のみ返す |

### volta の設計がペンテスターに優しい理由

volta のシングルプロセス・密結合哲学はペネトレーションテストを容易にします：

```
  マイクロサービスのペンテスト：
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │Svc A │──│Svc B │──│Svc C │──│Svc D │
  └──────┘  └──────┘  └──────┘  └──────┘
  4サービス、6インターフェース、4認証境界
  テスト困難、ギャップを見逃しやすい

  volta のペンテスト：
  ┌──────────────────────────────────┐
  │        volta-auth-proxy          │
  │  （シングルプロセス、1境界）       │
  └──────────────────────────────────┘
  1サービス、1インターフェース、1認証境界
  徹底的にテストしやすい
```

### ペンテスト準備チェックリスト

volta がペンテストフェーズに達した際の準備：

1. すべての API エンドポイントと期待される動作を文書化
2. 各ロールレベル（OWNER、MEMBER）のテストアカウントを提供
3. 本番環境と同一のステージング環境をセットアップ
4. スコープを定義（Web アプリ + API、Google OIDC プロバイダーは除外）
5. 重要な発見事項のコミュニケーションチャネルを確立
6. レポート後の修正スプリントの予算を確保

## よくある間違いと攻撃

### 間違い1：ペンテストを1回だけ行う

1回のペンテストはスナップショットです。新機能は新しい脆弱性をもたらします。少なくとも年1回、および主要な機能リリース後にペンテストを行いましょう。

### 間違い2：早すぎるペンテスト

半分しかできていないアプリケーションのペンテストはお金の無駄です。開発中に発見されるはずの問題を見つけるだけです。現在のフェーズで機能が完成するまで待ちましょう。

### 間違い3：レポートを無視する

発見事項が修正されなければ、ペンテストレポートは無意味です。Critical と High の発見は即座に修正を優先。Medium の発見は30日以内に対処すべきです。

### 間違い4：脆弱性スキャンとペンテストを混同する

自動脆弱性スキャナー（Nessus、OWASP ZAP）は既知の脆弱性パターンを見つけます。有用ですが、ロジックの欠陥、連鎖攻撃、ビジネスロジックの脆弱性を見つけられる人間のペンテスターの代わりにはなりません。

### 間違い5：スコープを定義しない

明確なスコープなしでは、ペンテスターが意図しないもの（本番データベース、サードパーティサービス）をテストしたり、重要なもの（特定の API エンドポイント）を見逃したりする可能性があります。必ずスコープ合意書に署名しましょう。
