# フェーズベース開発（Phase-Based Development）

[English version](phase-based-development.md)

---

## これは何？

フェーズベース開発は、最小限の実行可能な機能セットから始めて段階的に拡張していく戦略です。各フェーズは動く製品です -- 半完成のプロトタイプではなく、最終ビジョンより少ない機能を持つ完全なシステムです。

家を建てるようなものです。Phase 1：しっかりした基礎、壁、屋根、配管を作る。住める。Phase 2：2つ目のバスルームとガレージを追加。Phase 3：ホームオフィスとデッキを追加。各フェーズで住めるものが生まれます。デッキの設計が終わるまで基礎を打つのを待つことはありません。

---

## なぜ重要なのか？

多くのソフトウェアプロジェクトは、すべてを一度に作ろうとして失敗します。チームは何ヶ月も包括的なシステムを設計し、何ヶ月もかけて構築し、そして機能の半分は不要で残りの半分は間違っていたことに気づきます。フェーズベース開発は、動くソフトウェアを早期に提供し、実際のニーズに基づいて拡張することでこれを回避します。

volta-auth-proxyは4フェーズのロードマップに従い、このアプローチを実証しています。Phase 1は完全で動作する認証システムです。後続の各フェーズは、実際のユーザーが要求した機能、またはアーキテクチャがサポートするよう設計された機能を追加します。

---

## アンチパターン：最初にすべて作る

```
  「何かを作る前にすべてを設計しよう」アプローチ：

  1-3ヶ月:   OIDC + SAML + LDAP + MFA + SCIM + Billingを設計
  4-8ヶ月:   すべてを同時に構築
  9ヶ月目:   まだ何も動かない。すべてがすべてに依存。
  10ヶ月目:  チームの士気が低下。「いつ出荷できる？」
  12ヶ月目:  締め切りのプレッシャーでバグだらけのまま出荷。

  結果：10個のことをすべて中途半端にやるシステム。
```

これはウォーターフォールの罠です。失敗する理由：

1. **要件が変わる** -- 8ヶ月目には元の要件が時代遅れ。
2. **統合地獄** -- 隔離して構築された10のコンポーネントはスムーズに連携することが稀。
3. **フィードバックループがない** -- 12ヶ月間ユーザーから何も学べない。
4. **モチベーションの死** -- 進捗を見せられないとチームのエネルギーが失われる。

## voltaのアプローチ：フェーズベースのデリバリー

### Phase 1: Core（最小限の実行可能な認証）

```
  出荷するもの：
  ┌──────────────────────────────────────────────────────┐
  │ Google OIDCログイン（Authorization Code + PKCE）      │
  │ セッション管理（スライディングウインドウ、ユーザーあたり最大5）│
  │ マルチテナンシー（テナント作成、解決、切り替え）      │
  │ ロールベースアクセス（OWNER > ADMIN > MEMBER > VIEWER）│
  │ 招待システム（コード、期限切れ、同意画面）            │
  │ ForwardAuth（ダウンストリームアプリへのX-Volta-*ヘッダー）│
  │ 内部API（アプリ委譲用/api/v1/*）                     │
  │ JWT発行（RS256、5分有効期限、JWKSエンドポイント）     │
  │ ログインUI（jteテンプレート、完全コントロール）       │
  └──────────────────────────────────────────────────────┘

  コンポーネント：volta-auth-proxy + PostgreSQL。
  それだけ。動く。出荷する。
```

Phase 1はデモではありません。マルチテナントSaaS向けの本番対応の認証です。チームはこの上に実際のアプリケーションを構築できます。

### Phase 2: Scale

```
  追加されるもの：
  ┌──────────────────────────────────────────────────────┐
  │ 複数IdP（GitHub、Microsoft）                         │
  │ M2M OAuth（Client Credentialsフロー）                │
  │ Redisセッションストレージ（より高いスループット）     │
  │ Webhooks（サービスへのイベント通知）                  │
  │ パスキー（WebAuthn/FIDO2）                           │
  └──────────────────────────────────────────────────────┘
```

Phase 2はスケーリングのニーズに対応します。より多くのログインオプション、マシン間通信、より速いセッション検索。これらはPhase 1が本番稼働した後に必要だと気づく機能です。

### Phase 3: Enterprise

```
  追加されるもの：
  ┌──────────────────────────────────────────────────────┐
  │ SAML SSO（エンタープライズIdP統合）                   │
  │ メール通知（SMTP、SendGrid）                         │
  │ MFA/2FA（TOTP、WebAuthn）                            │
  │ i18n（国際化）                                       │
  │ 条件付きアクセス（リスクベース認証）                  │
  │ 不正検知/アラート                                    │
  └──────────────────────────────────────────────────────┘
```

Phase 3はエンタープライズ要件を追加します。SAMLはActive Directoryを使う企業に販売するときに必要。MFAはセキュリティポリシーが要求するときに必要。これらはエンタープライズ顧客がいるときに追加する機能であり、その前ではありません。

### Phase 4: Platform

```
  追加されるもの：
  ┌──────────────────────────────────────────────────────┐
  │ SCIM（自動ユーザープロビジョニング）                  │
  │ ポリシーエンジン                                      │
  │ 課金統合（Stripe）                                   │
  │ GDPRデータエクスポート/削除                          │
  │ デバイストラスト                                      │
  │ モバイルSDK（iOS/Android）                           │
  └──────────────────────────────────────────────────────┘
```

Phase 4はvoltaをプラットフォームに変えます。SCIMは何千人もの従業員を持つ企業向け。課金統合はサブスクリプションをテナントに紐づけるSaaS製品向け。これらはスケール時に必要な機能であり、ローンチ時ではありません。

## すべてを一度に作らないことがなぜ実は速いか

### 1. より早く出荷できる

Phase 1は数週間で出荷できます。「すべて作る」アプローチは数ヶ月（出荷できない場合も）。実際のユーザーがシステムを使うことで、Phase 2を形作るフィードバックが得られます。

### 2. インターフェースが将来を守る

voltaはPhase 1で代替実装が後から来る場合でもインターフェース（`SessionStore`、`AuditSink`など）を定義します。アーキテクチャは早すぎるコードに負担をかけることなく、拡張の準備ができています：

```
  Phase 1: SessionStoreインターフェース + PostgresSessionStore
  Phase 2: + RedisSessionStore（投入、認証ロジックの変更ゼロ）
```

### 3. 各フェーズは独立してテスト可能

Phase 1は完全なテストを持つ完全なシステムです。Phase 2はPhase 1を壊さずに機能を追加します。同時に構築された10の機能をテストするよりはるかに簡単です。

### 4. 間違ったものを作ることを避けられる

SAMLを使う顧客がいる前に、SAMLが必要だとどうやって分かりますか？分かりません。フェーズベース開発は、顧客が求めたときにSAMLを作ることを意味し、必要になるかもしれないと想像したときではありません。

---

## 重要な要件：アーキテクチャが拡張をサポートしなければならない

フェーズベース開発は、Phase 1のアーキテクチャが書き直しなしに将来のフェーズを受け入れられる場合にのみ機能します。意図的な設計が必要です：

- 代替実装を持つコンポーネントへの**インターフェース**
- 外部キーと拡張性を考慮して設計された**データベーススキーマ**
- 将来のAPIが既存クライアントを壊さないための**APIバージョニング**（`/api/v1/`）
- 環境変数ベースで簡単に拡張可能な**設定**

voltaのPhase 1はPhase 2-4を念頭に設計されました。コードは書かれていませんが、アーキテクチャは準備されていました。
