# プロセス

[English version](process.md)

---

## これは何？

プロセスとは、プログラムの実行中のインスタンスです。`java -jar volta-auth-proxy.jar`を実行すると、オペレーティングシステムがプロセスを作成します。そのプロセスは独自のメモリ空間、独自のCPU時間、独自のプロセスID（PID）を取得します。同じマシン上の他のプロセスとは独立して動作します。

机で働く作業員に似ています。プログラム（JARファイル）は職務記述書のようなもので、作業員が何をすべきかを記述します。プロセスは実際に机に座って仕事をしている作業員です。同じ職務記述書に従う複数の作業員（同じプログラムを実行する複数のプロセス）を持つことができ、各作業員は自分の机と書類（メモリ空間）を持っています。

volta-auth-proxyを実行すると、単一のプロセスとして起動します。そのプロセス内部で、[JVM](jvm.md)が複数のスレッド（サブタスク）を管理しますが、OSの観点からは1つのプロセスです。これが[シングルプロセス](single-process.md)アーキテクチャです。

---

## なぜ重要なのか？

プロセスを理解することが重要な理由：

- **リソース管理** -- 各プロセスはメモリとCPUを消費します。voltaは単一プロセスとして動作し、リソース使用を予測可能に保ちます。
- **ライフサイクル管理** -- voltaの起動、停止、監視は1つのプロセスの管理を意味します。`kill <PID>`で停止。`docker stop`でコンテナプロセスを停止。
- **隔離** -- プロセスは互いに隔離されています。voltaがクラッシュしても、同じサーバー上の他のアプリケーションを巻き添えにしません。
- **環境** -- 各プロセスは独自の[環境変数](environment-variable.md)を持ちます。voltaはプロセス環境から設定を読み取ります。
- **ポートバインディング** -- 1つのプロセスだけが特定の[ポート](port.md)でリッスンできます。voltaがポート7070で動作中なら、他のプロセスはそのポートを使えません。

## どう動くのか？

### プロセスのライフサイクル

```
  プログラム（ディスク上のファイル）     プロセス（メモリ上で実行中）
  ──────────────────────              ─────────────────────────────

  volta-auth-proxy.jar                PID: 12345
  （ただのファイル、何もしていない）    状態: 実行中
                                      メモリ: 256MB
       │                              CPU: 2.3%
       │                              ポート: 7070
       │    java -jar volta.jar       スレッド: 24
       └────────────────────────────> （HTTPリクエストを受付中）
                                           │
                                           │  kill 12345 / Ctrl+C
                                           │
                                           ▼
                                      状態: 終了
                                      メモリ: 解放
                                      ポート: 解放
```

### プロセス vs スレッド

```
  ┌───────────────────────────────────────────┐
  │              プロセス（PID: 12345）         │
  │              volta-auth-proxy              │
  │                                            │
  │  ┌────────┐  ┌────────┐  ┌────────┐      │
  │  │スレッド1│  │スレッド2│  │スレッド3│ ... │
  │  │(main)  │  │(HTTP   │  │(HTTP   │       │
  │  │        │  │ req 1) │  │ req 2) │       │
  │  └────────┘  └────────┘  └────────┘       │
  │                                            │
  │  共有メモリ空間                              │
  │  （すべてのスレッドが同じデータを見る）        │
  │                                            │
  │  ┌─────────────────────────────────────┐   │
  │  │  HikariCPプール（共有）              │   │
  │  │  Caffeineキャッシュ（共有）           │   │
  │  │  RSA鍵ペア（共有）                   │   │
  │  └─────────────────────────────────────┘   │
  └───────────────────────────────────────────┘
```

プロセスはコンテナです。スレッドはそのコンテナ内の作業員です。複数のスレッドが同じメモリを共有するため、[Caffeineキャッシュ](caffeine-cache.md)と[HikariCP](hikaricp.md)プールが効率的に動作します -- すべてのスレッドが同じキャッシュとプールにアクセスします。

### プロセスの識別

すべてのプロセスは一意のPID（プロセスID）を取得します：

```bash
# voltaのプロセスを見つける
ps aux | grep volta-auth-proxy
# 出力: user  12345  2.3  5.1  java -jar volta-auth-proxy.jar

# Dockerで実行中の場合
docker ps
# 出力: CONTAINER_ID  IMAGE  COMMAND  STATUS  PORTS
#         abc123        volta  java...  Up 2h   0.0.0.0:7070->7070
```

### プロセス環境

各プロセスは親から環境変数を継承します：

```bash
# これらの環境変数はvoltaプロセス内部で利用可能
export VOLTA_PORT=7070
export GOOGLE_CLIENT_ID=abc123.apps.googleusercontent.com
export DATABASE_URL=jdbc:postgresql://localhost:5432/volta

java -jar volta-auth-proxy.jar
# voltaはSystem.getenv("VOLTA_PORT")などで読み取る
```

### シグナル

OSはシグナルを通じてプロセスと通信します：

| シグナル | 意味 | voltaへの影響 |
|---------|------|-------------|
| SIGTERM (kill) | 「正常にシャットダウンしてください」 | voltaが接続を閉じ、状態を保存し、終了 |
| SIGKILL (kill -9) | 「今すぐ死ね」 | プロセスが即座に終了、クリーンアップなし |
| SIGINT (Ctrl+C) | 「ユーザーが停止を望んでいる」 | SIGTERMと同じ |
| SIGHUP | 「端末が切断された」 | サーバーでは通常無視 |

## volta-auth-proxy ではどう使われている？

### シングルプロセスアーキテクチャ

voltaは単一の[JVM](jvm.md)プロセスとして動作します。HTTPサーバー、セッション管理、JWTサービス、データベース接続、キャッシュ -- すべてが1つのプロセス内で動きます：

```
  ┌─────────────────────────────────────────────────┐
  │         volta-auth-proxy（単一プロセス）          │
  │                                                  │
  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
  │  │  Javalin  │  │ セッション │  │  JwtService  │  │
  │  │  (HTTP)   │  │ マネージャ │  │  (RS256)     │  │
  │  └──────────┘  └──────────┘  └──────────────┘  │
  │                                                  │
  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
  │  │ HikariCP │  │ Caffeine │  │  Flyway       │  │
  │  │ (DBプール)│  │(キャッシュ)│  │(マイグレーション)│  │
  │  └──────────┘  └──────────┘  └──────────────┘  │
  │                                                  │
  │  1つのPID。1つのJVM。1つのプロセス。              │
  └─────────────────────────────────────────────────┘
```

このシンプルさは意図的です。メッセージキューなし、ワーカープロセスなし、プロセスマネージャなし。1つのプロセスがすべてを処理します。

### プロセス起動シーケンス

`java -jar volta-auth-proxy.jar`が実行されると：

```
  1. JVM起動
  2. Main.main()が実行
  3. 環境変数を読み取り（設定）
  4. HikariCPを初期化（DB接続プール）
  5. Flywayマイグレーションを実行（テーブル作成/更新）
  6. RSA鍵ペアを読み込み（JWT署名用）
  7. Caffeineキャッシュを初期化
  8. Javalinを起動（ポート7070でHTTPサーバー）
  9. ルートとミドルウェアを登録
  10. プロセス準備完了 -- リクエスト受付中
```

### Dockerでのプロセス

[Docker](docker.md)で実行する場合、voltaプロセスはコンテナ内のPID 1です：

```bash
docker run -d \
  -e VOLTA_PORT=7070 \
  -e DATABASE_URL=jdbc:postgresql://db:5432/volta \
  -p 7070:7070 \
  volta-auth-proxy

# コンテナ内部:
# PID 1 = java -jar volta-auth-proxy.jar
```

Dockerはホストの[ポート](port.md) 7070をコンテナ内のポート7070にマッピングし、voltaプロセスにトラフィックを転送します。

### グレースフルシャットダウン

voltaがSIGTERMを受信すると：

```
  SIGTERM受信
       │
       ▼
  1. 新しいHTTPリクエストの受付を停止
  2. 処理中のリクエストを完了（タイムアウトまで）
  3. HikariCP接続を閉じる
  4. 保留中の書き込みをフラッシュ
  5. コード0で終了
```

## よくある間違いと攻撃

### 間違い1：同じポートで複数プロセスを実行

```bash
java -jar volta-auth-proxy.jar &  # プロセス1がポート7070で
java -jar volta-auth-proxy.jar    # プロセス2が失敗：ポートは使用中！
```

1つのプロセスだけがポートをバインドできます。複数インスタンスが必要なら、異なるポートかロードバランサーを使ってください。

### 間違い2：kill -9をデフォルトとして使う

`kill -9`（SIGKILL）はクリーンアップなしでプロセスを即座に終了します。データベース接続は閉じられず、処理中のリクエストはレスポンスを得られず、セッションデータが失われる可能性があります。常にまず`kill`（SIGTERM）を使い、プロセスが本当にスタックしている場合のみ`-9`を使ってください。

### 間違い3：プロセスを監視しない

プロセスは静かにクラッシュすることがあります。監視なしでは、ユーザーが苦情を言うまで誰も気づきません。プロセスマネージャ（systemd、再起動ポリシー付きDockerなど）を使って、クラッシュしたプロセスを自動的に再起動してください。

### 攻撃1：リソース枯渇

攻撃者が何千ものリクエストを送り、プロセスのメモリやスレッドを枯渇させます：

```
  攻撃者 → 10,000の同時接続
         → スレッドプールが枯渇
         → 正規ユーザーがレスポンスを得られない
```

防御：接続制限、[レート制限](rate-limiting.md)、Javalin/Jettyでのタイムアウト設定。

### 間違い4：rootとして実行

voltaプロセスをrootユーザーとして実行すると、voltaの脆弱性がサーバー全体を危険にさらす可能性があります。常にroot以外のユーザーとして実行してください。Dockerのベストプラクティス：Dockerfileに`USER 1000`を追加。
