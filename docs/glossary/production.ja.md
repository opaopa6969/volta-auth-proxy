# 本番環境（Production）

[English version](production.md)

---

## これは何？

本番環境（Production）とは、実際のユーザーがソフトウェアを利用するライブ環境のことです。開発、テスト、ステージングを経た最終的な到達先です。「本番に入った」と言うとき、それは実際の人々が今まさに使っていて、実際のデータがあり、何か壊れたら実際に影響があるという意味です。

舞台で考えてみましょう。開発はリハーサルです -- いろいろ試して、間違えても、観客には見えません。ステージングはゲネプロ（最終リハーサル）です -- すべて本番同様ですが、まだ非公開です。本番は初日公演です -- 幕が上がり、観客が見ていて、すべてのミスが見えます。本番を一時停止して誤字を直すことはできません。

ほとんどのプロジェクトには少なくとも3つの環境があります：**開発**（あなたのノートPC）、**ステージング**（本番を模したサーバーだが偽データ）、**本番**（本物）。コードは環境間で同一です。違うのはデータ、ユーザー、そして結果の重大さです。

---

## なぜ重要なのか？

- **本番はビジネスが動く場所。** 収益、ユーザーの信頼、評判は本番の安定性に依存する。
- **本番のバグは実際の人々に影響する。** 開発でのログインバグは不便だが、本番でのログインバグは有料顧客を締め出す。
- **本番データは神聖。** テストデータベースの誤削除は肩をすくめるだけ。本番データベースの誤削除は大惨事。
- **セキュリティ要件がエスカレートする。** 開発では弱いパスワードや自己署名証明書でOK。本番では強力なシークレット、TLS、適切なアクセス制御が必要。
- **パフォーマンスがより重要。** 開発で100msのレイテンシは問題ない。1000同時ユーザーの本番で100msはサーバーが瀕死かもしれない。

## どう動くのか？

### 環境の比較

```
  ┌──────────────────────────────────────────────────────────┐
  │                      環境                                │
  ├──────────────┬──────────────┬──────────────┬─────────────┤
  │              │    開発      │ ステージング  │    本番      │
  ├──────────────┼──────────────┼──────────────┼─────────────┤
  │ ユーザー      │ 自分だけ     │ QAチーム     │ 全員         │
  │ データ       │ 偽/シード    │ 偽/コピー    │ 本物         │
  │ URL          │ localhost    │ staging.x    │ app.x.com    │
  │ DB           │ ローカルPG   │ 共有PG       │ マネージドPG │
  │ シークレット  │ テスト用鍵   │ テスト用鍵   │ 本物の鍵     │
  │ エラー       │ スタックトレース│ スタックトレース│ ユーザー向け │
  │ ログレベル   │ DEBUG        │ INFO         │ WARN/ERROR   │
  │ 壊れたら     │ ¯\_(ツ)_/¯   │ あとで直す   │ 今すぐ直せ   │
  └──────────────┴──────────────┴──────────────┴─────────────┘
```

### 本番が他と違う理由

**1. 本物のデータ、本物の結果**

本番データベースには実際のユーザー情報 -- メール、セッション、テナント設定、ロール割り当てが含まれます。データの消失や破損はシードスクリプトからは復元できません。

**2. アップタイムの期待値**

| 稼働率 % | 年間ダウンタイム | 一般的な対象 |
|----------|-----------------|-------------|
| 99%      | 3.65日          | 趣味プロジェクト |
| 99.9%    | 8.76時間        | SaaS製品 |
| 99.99%   | 52.6分          | エンタープライズ |
| 99.999%  | 5.26分          | 銀行/医療 |

**3. 可観測性**

開発ではターミナルでログを読みます。本番では以下が必要です：
- 構造化ログ（ログ集約用のJSON形式）
- ヘルスチェックエンドポイント
- メトリクス（リクエスト率、エラー率、レイテンシ）
- アラート（PagerDuty、Slack通知）

**4. セキュリティ強化**

```
  開発:                           本番:
  ┌───────────────────┐          ┌────────────────────────┐
  │ HTTP (TLSなし)    │          │ HTTPS (TLS必須)        │
  │ CORS: すべて許可  │          │ CORS: 特定のオリジン   │
  │ デバッグエンドポイント│       │ デバッグ無効           │
  │ 弱いシークレット   │          │ 強力なローテーション済み│
  │ レート制限なし     │          │ レート制限有効         │
  └───────────────────┘          └────────────────────────┘
```

## volta-auth-proxy ではどう使われている？

### 本番設定

volta-auth-proxyは[環境変数](environment-variable.ja.md)を使って本番と他の環境を区別します。重要な本番設定には以下が含まれます：

```bash
# 本番環境変数（例）
VOLTA_BASE_URL=https://auth.yourdomain.com
DATABASE_URL=jdbc:postgresql://prod-db:5432/volta
GOOGLE_CLIENT_ID=<本番用Google OAuthクライアントID>
GOOGLE_CLIENT_SECRET=<本番用シークレット>
JWT_PRIVATE_KEY_PATH=/etc/volta/keys/private.pem
VOLTA_COOKIE_SECURE=true
VOLTA_COOKIE_SAMESITE=Lax
```

### voltaの本番チェックリスト

公開前に確認すること：

1. **データベース** -- PostgreSQLが稼働中、[マイグレーション](migration.ja.md)が適用済み、バックアップがスケジュール済み
2. **シークレット** -- Google OIDC認証情報が本番用Google Cloudプロジェクトのもの
3. **JWT鍵** -- [RS256](rs256.ja.md)鍵ペアが生成・保護されていること。開発用の鍵ではないこと
4. **Traefik** -- [ForwardAuth](forwardauth.ja.md)がvoltaに正しくルーティングされ、TLS証明書が有効
5. **Cookie** -- `Secure`フラグが`true`、`SameSite`が`Lax`または`Strict`
6. **レート制限** -- [ブルートフォース](brute-force.ja.md)攻撃から守るために有効化
7. **ログ** -- WARNまたはERRORレベルに設定。機密データを漏らすデバッグ出力なし
8. **コネクションプール** -- [HikariCP](hikaricp.ja.md)が予想負荷に適切にサイズ設定

### 本番アーキテクチャ

```
  DNS: auth.yourdomain.com
          │
          ▼
  ┌──────────────┐
  │   Traefik     │  ← TLS終端
  │   (リバース    │  ← voltaへのForwardAuth
  │    プロキシ)   │  ← ダウンストリームアプリへのルーティング
  └──────┬───────┘
         │
         ▼
  ┌──────────────────┐
  │ volta-auth-proxy  │  ← セッション検証
  │                    │  ← JWT発行
  │                    │  ← X-Volta-*ヘッダー設定
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │    PostgreSQL     │  ← ユーザー、テナント、セッション、
  │                    │     ロール、招待
  └──────────────────┘
```

## よくある間違いと攻撃

### 間違い1：本番で開発用シークレットを使用

本番のGoogle OIDCクライアントIDがlocalhostで使っているものと同じなら、何かが大きく間違っています。本番のシークレットは別に管理し、定期的にローテーションし、gitにコミットしてはいけません。

### 間違い2：デバッグエンドポイントを公開したまま

`/debug`、冗長な出力のある`/health`、エラーレスポンスのスタックトレースなどは、攻撃者に内部アーキテクチャを漏らします。

### 間違い3：バックアップ戦略がない

本番データベースには自動バックアップが必要です。ユーザーデータの唯一のコピーが稼働中のデータベースだけなら、`DELETE FROM users`一つで大惨事です。

### 間違い4：本番でテスト

「本番でちょっと試してみるだけ」は障害の始まりです。ステージングを使いましょう。そのためにあるのです。

### 間違い5：環境間で同じ認証情報

開発と本番が同じデータベースパスワードを共有していると、侵害された開発マシンが攻撃者に本番データへのアクセスを与えます。
