# 伝搬（Propagation）

[English version](propagation.md)

---

## これは何？

伝搬（Propagation）とは、変更がシステム全体に時間をかけて広がるプロセスです。認証では、変更が起きたとき（セッションが失効、鍵がローテーション、ロールが変更）から、システムのすべての部分がその変更を反映するまでの遅延を指します。

ニュース放送を思い浮かべてください。大きな出来事が14:00に起きます。テレビ局が14:05に報道します。ライブで見ている視聴者はすぐに知ります。朝の番組を録画して遅れて見ている視聴者は、ライブに切り替えるまで知りません。「伝搬遅延」とは、出来事と全員が知るまでの時間です。

voltaでは、主要な伝搬遅延はJWTの有効期限ウィンドウです。セッションを失効させると、サーバー側のチェックは即座に変更を反映します。しかし既に発行されたJWTは期限切れまで（最大5分）下流サービスに受け入れられ続けます。その5分間が伝搬遅延です。

---

## なぜ重要なのか？

伝搬遅延は意図と現実の間にギャップを作ります：

- **セキュリティ**: 侵害されたセッションを14:00に失効。攻撃者のJWTは14:05まで有効。5分間のアクセス継続
- **アクセス制御**: 14:00にadminロールを削除。ユーザーのJWTは14:05まで`volta_roles: ["admin"]`のまま
- **鍵ローテーション**: 侵害された鍵を14:00にローテーション。古い鍵で署名されたトークンは14:05まで受け入れられる

伝搬を理解することで、情報に基づいたトレードオフができます。voltaが5分のJWT有効期限を選んだのは、セキュリティには十分短く、ステートレス検証（リクエストごとのDB呼び出し不要）を可能にするバランスだからです。

## どう動くのか？

### 伝搬遅延の発生源

```
  変更内容            伝搬時間             理由
  ────────────────────────────────────────────────────
  セッション削除      即座（サーバー）     リクエストごとのDB参照
                      0-5分（JWT）        発行済みJWTがまだ有効

  鍵ローテーション    即座（新トークン）   新JWTは新鍵を使用
                      0-5分（旧トークン） 旧鍵JWTが期限切れ

  ロール変更          即座（DB）          DBは変更を反映
                      0-5分（JWT）        JWTクレームが古い

  JWKSキャッシュ更新  0-N分              下流のキャッシュTTL
```

### voltaの伝搬モデル

```
  時間 ──────────────────────────────────────────────────►

  14:00:00  管理者がユーザーのセッションを失効
            │
            ├── サーバー側: 即座
            │   次のセッションチェック → 401
            │   新しいJWTは発行不可
            │
            └── JWTベース: 遅延（0-5分）
                既存のJWTはまだ有効
                │
  14:00:01  下流アプリがJWTチェック → 有効（残り4:59）
  14:02:30  下流アプリがJWTチェック → 有効（残り2:30）
  14:05:00  JWT期限切れ → 無効
            │
            ▼
            完全に伝搬完了。ユーザーは全箇所でロックアウト。
```

### 伝搬の方程式

```
  最大伝搬遅延 = JWT TTL

  volta:  JWT TTL = 5分
          → 最大遅延 = 5分

  JWT TTLが1時間の場合:
          → 最大遅延 = 1時間
          → 攻撃者に1時間のウィンドウ

  JWT TTLが24時間の場合:
          → 最大遅延 = 24時間
          → ほとんどのアプリケーションで受け入れ不可

  トレードオフ:
  ┌──────────────────────────────────────────────┐
  │  短いJWT TTL:                                │
  │  ✓ より速い伝搬                              │
  │  ✓ より高いセキュリティ                       │
  │  ✗ より頻繁なサイレントリフレッシュ            │
  │  ✗ トークンエンドポイントへの負荷増            │
  │                                              │
  │  長いJWT TTL:                                │
  │  ✓ リフレッシュ回数減少                       │
  │  ✓ サーバー負荷減少                           │
  │  ✗ 遅い伝搬                                  │
  │  ✗ 大きな攻撃ウィンドウ                       │
  │                                              │
  │  voltaは5分を選択: 良いバランス。              │
  └──────────────────────────────────────────────┘
```

### 伝搬が即座の場合と遅延する場合

```
  ┌────────────────────────────────────────────────┐
  │  即座（サーバー側の状態チェック）:              │
  │  ├── データベースでのセッション参照            │
  │  ├── Cookie検証                               │
  │  └── トークンエンドポイント（新JWT発行）       │
  │                                                │
  │  遅延（ステートレスJWTチェック）:              │
  │  ├── 下流アプリのJWT署名検証                  │
  │  ├── APIゲートウェイのJWTクレーム確認          │
  │  └── マイクロサービスのJWTロール信頼           │
  └────────────────────────────────────────────────┘

  volta-auth-proxy自体はセッションをチェック（即座）。
  下流アプリはJWTをチェック（最大5分遅延）。
```

## volta-auth-proxy ではどう使われている？

### 二重検証モデル

voltaは異なる伝搬特性を持つ2つのレイヤーを使います：

```
  レイヤー1: セッション（即座の伝搬）
  ┌──────────────────────────────────────────┐
  │  voltaへのすべてのリクエストがセッション  │
  │  をチェック                              │
  │  セッション削除 → 即座に401              │
  │  伝搬遅延なし                            │
  └──────────────────────────────────────────┘

  レイヤー2: JWT（遅延伝搬、最大5分）
  ┌──────────────────────────────────────────┐
  │  下流アプリがJWT署名を検証               │
  │  JWTクレームはある時点のスナップショット   │
  │  JWTが期限切れになると変更が伝搬          │
  │  クライアントがリフレッシュで新JWT取得    │
  └──────────────────────────────────────────┘
```

### 鍵ローテーションの伝搬

`POST /api/v1/admin/keys/rotate`が呼ばれたとき：

```
  T+0:00  新鍵アクティブ。旧鍵引退。
          新JWTは新鍵で署名。

  T+0:00  キャッシュ済みJWKSの下流:
    から   旧鍵トークンもまだ検証可能 ✓
  T+5:00  （移行中、旧鍵はまだJWKSに含まれる）

  T+5:00  旧鍵JWTすべて期限切れ。
          旧鍵をJWKSから削除可能。

  伝搬完了。
```

### 伝搬メカニズムとしてのサイレントリフレッシュ

[サイレントリフレッシュ](silent-refresh.md)はJWTの変更がクライアントに伝搬する方法です：

```
  ロール変更:  admin → member（T+0:00で）

  T+0:00  古いJWT: volta_roles=["admin"]（まだ有効）
  T+4:00  古いJWTが期限切れ間近
          volta-sdk-jsがトークンエンドポイントを呼ぶ
          新JWT: volta_roles=["member"]（変更を反映）
  T+4:00+ 以降のすべてのリクエストが更新されたJWTを使用

  サイレントリフレッシュがロール変更を伝搬した。
```

## よくある間違いと攻撃

### 間違い1: JWTの即座の伝搬を仮定する

開発者は`セッション削除 → 全箇所で即座のロックアウト`を期待します。これはサーバー側のセッションチェックにのみ当てはまります。JWTベースのチェックにはJWT TTLに等しい伝搬遅延があります。

### 間違い2: 非常に長いJWT TTL

24時間のJWTは24時間の伝搬遅延を意味します。鍵が侵害されたら攻撃者は丸一日使えます。JWT TTLを短く保ちましょう（volta: 5分）。

### 間違い3: JWKSキャッシュTTLを考慮しない

下流サービスがJWKSエンドポイントを1時間キャッシュすると、鍵ローテーションの伝搬が最大1時間+JWT TTL遅延します。短いJWKSキャッシュTTL（5-15分）を推奨します。

### 間違い4: 機密操作がJWTクレームだけに依存

「テナントデータをすべて削除」や「ユーザーをadminに昇格」のような操作では、JWTクレーム（古い可能性あり）だけに頼らないでください。重要な操作ではデータベースに対して検証しましょう。

### 攻撃: 伝搬ウィンドウの悪用

セッションが失効した攻撃者がまだ有効なJWT（最大5分）を使ってできるだけ多くのアクションを実行しようと急ぎます。緩和策：重要なアクションでは、JWT有効性だけでなく常にサーバー側でセッション有効性をチェック。
