# プロビジョニング

[English version](provisioning.md)

---

## これは何？

プロビジョニングは、外部のIDソースからのコマンドに基づいて、アプリケーション内のユーザーアカウントを自動的に作成、更新、無効化するプロセスです。従業員が入社すると、プロビジョニングが接続されたすべてのシステムにアカウントを作成します。退職すると、デプロビジョニングがそれらのアカウントをすべて無効化します。

ホテルの予約システムのようなものです。部屋を予約すると、ホテルは名前をリストに載せるだけではありません。滞在をプロビジョニングします：ルームキーがプログラムされ、ミニバーが補充され、ウェルカムスクリーンに名前が表示され、Wi-Fiパスワードが生成されます。チェックアウト時にはすべてがデプロビジョニングされます：キーは使えなくなり、名前は削除され、部屋は次のゲストのために準備されます。ソフトウェアのプロビジョニングも同じ仕組みです。ユーザーが作業を始めるために必要なすべてをセットアップします。

ID管理の世界では、プロビジョニングは通常[SCIM](scim.md)（標準プロトコル）またはJIT（Just-In-Time）プロビジョニング（初回ログイン時）で自動化されます。

---

## なぜ重要なのか？

手動のアカウント管理はスケールせず、セキュリティリスクを生みます：

```
  手動プロビジョニング（悪夢）：
  ┌──────────┐
  │ 新入社員  │
  │ 「Alice」 │
  └─────┬────┘
        │
        ▼
  ┌──────────────────────────────────────────────┐
  │ IT管理者のToDoリスト：                         │
  │                                               │
  │ □ メールシステムにアカウント作成                │
  │ □ Slackにアカウント作成                        │
  │ □ GitHubにアカウント作成                       │
  │ □ volta（あなたのSaaSアプリ）にアカウント作成   │
  │ □ あと12システムにアカウント作成...             │
  │                                               │
  │ 所要時間：従業員1人あたり2時間                   │
  │ エラー率：「GitHub忘れた」が毎週発生            │
  └──────────────────────────────────────────────┘

  自動プロビジョニング：
  ┌──────────┐     ┌──────┐     ┌───────────────┐
  │ 新入社員  │────►│ Okta │────►│ 全システム     │
  │ 「Alice」 │     │ (IdP)│     │ （SCIM経由）   │
  └──────────┘     └──────┘     └───────────────┘
  所要時間：30秒。エラー率：0%。
```

主な利点：

- **スピード**：数時間ではなく数分でオンボーディング
- **セキュリティ**：退職者が即座にアクセスを失う。忘れられたアカウントなし
- **一貫性**：すべてのユーザーが毎回同じセットアップを受ける
- **コンプライアンス**：すべてのプロビジョニング/デプロビジョニングアクションの監査証跡
- **スケール**：10ユーザーでも10,000ユーザーでも同じように動作

## どう動くのか？

### プロビジョニングライフサイクル

```
  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
  │ プロビジョニング│────►│   更新       │────►│デプロビジョニング│
  │              │     │              │     │              │
  │ ユーザー作成  │     │ ロール変更    │     │ 無効化       │
  │ ロール割当    │     │ 名前更新     │     │ アクセス取消  │
  │ アクセス付与  │     │ チーム移動    │     │ クリーンアップ │
  └──────────────┘     └──────────────┘     └──────────────┘
       │                     │                    │
       ▼                     ▼                    ▼
  ┌──────────────────────────────────────────────────────┐
  │                volta-auth-proxy                       │
  │  usersテーブル：active=true   ロール変更  active=false │
  │  セッション：作成             有効       取り消し       │
  │  webhook：user.created      role       user.deleted  │
  └──────────────────────────────────────────────────────┘
```

### プロビジョニング方法

| 方法 | トリガー | プロトコル | ユースケース |
|------|---------|----------|------------|
| SCIMプロビジョニング | IdPが変更をプッシュ | [SCIM](scim.md) 2.0 REST API | エンタープライズSSO+ユーザー管理 |
| JITプロビジョニング | ユーザーの初回ログイン | [OAuth2](oauth2.md)コールバック | セルフサービスアプリ、小規模チーム |
| [招待](invitation-code.md) | 管理者が招待送信 | volta招待API | 管理されたオンボーディング、B2B |
| APIプロビジョニング | 管理者がユーザー作成 | volta REST API | カスタム統合 |

### SCIMプロビジョニングフロー

```
  Okta                          volta-auth-proxy
  ====                          ================

  管理者がアプリにユーザーを割り当て
        │
        ▼
  POST /scim/v2/Users ─────────────────────────►
  {                                              volta：
    "userName": "alice@acme.com",                1. ユーザーレコード作成
    "name": {...},                               2. テナントに割り当て
    "active": true                               3. デフォルトロール設定
  }                                              4. user.created webhook発火
                               ◄─────────────────
                               201 Created
                               {"id": "volta-uuid", ...}

  管理者がユーザーのグループを変更
        │
        ▼
  PATCH /scim/v2/Users/volta-uuid ──────────────►
  {                                              volta：
    "Operations": [{                             1. ロール更新
      "op": "replace",                           2. role_changed webhook発火
      "path": "groups",
      "value": [{"value": "admin-group"}]
    }]
  }
                               ◄─────────────────
                               200 OK

  管理者がアプリからユーザーを削除
        │
        ▼
  PATCH /scim/v2/Users/volta-uuid ──────────────►
  {                                              volta：
    "Operations": [{                             1. active=falseに設定
      "op": "replace",                           2. 全セッション取り消し
      "path": "active",                          3. user.deleted webhook発火
      "value": false
    }]
  }
                               ◄─────────────────
                               200 OK
```

### JIT（Just-In-Time）プロビジョニングフロー

```
  ユーザー              volta-auth-proxy          IDプロバイダ
  ======              ================          ===========

  初回ログイン
  ──────────────────►
                     IdPにリダイレクト
                     ──────────────────────────────────►
                                               ユーザーが認証
                     ◄──────────────────────────────────
                     ユーザー情報付きOAuthコールバック受信

                     ユーザーはvoltaに存在する？
                       NO → ユーザー作成（JITプロビジョニング）
                            デフォルトロール割り当て
                            user.created webhook発火
                       YES → 属性変更があれば更新

  ◄──────────────────
  ログイン完了（新しいアカウント準備完了）
```

## volta-auth-proxyではどう使われている？

### 現在のプロビジョニング方法

voltaは3つのプロビジョニング方法をサポートしています：

1. **招待ベース**（Phase 1）：管理者が[招待コード](invitation-code.md)を送信、ユーザーがそれで登録
2. **SCIMベース**（Phase 3）：IDプロバイダが[SCIM](scim.md)エンドポイント経由でユーザーをプッシュ
3. **JIT**（Phase 1）：初回OAuthログインでテナントにユーザーを自動作成

### プロビジョニングとテナントメンバーシップ

すべてのプロビジョニングされたユーザーは[テナント](tenant.md)に所属します。voltaは「浮遊する」ユーザーを許可しません：

```
  プロビジョニングは常に2つの質問に答える：
  1. ユーザーは誰か？（メール、名前、アイデンティティ）
  2. どこに所属するか？（tenant_id）

  ┌──────────────────────────────────────┐
  │  User: alice@acme.com               │
  │  Tenant: acme-corp (tenant_id)       │
  │  Role: MEMBER（デフォルト）            │
  │  Source: SCIM / Invitation / JIT     │
  │  Active: true                        │
  └──────────────────────────────────────┘
```

### デプロビジョニングとセキュリティ

ユーザーがデプロビジョニングされると（SCIMの`active: false`または管理者アクション経由）：

1. ユーザーレコードが`active: false`に設定（[停止](suspension.md)）
2. すべてのアクティブな[セッション](session.md)が即座に取り消し
3. すべての[Cookie](cookie.md)が無効化
4. `user.deleted` [Webhook](webhook.md)が[Outbox](outbox-pattern.md)経由で発火
5. 再プロビジョニングされるまでログイン不可

### プロビジョニングイベントとWebhook

すべてのプロビジョニングアクションが[Webhook](webhook.md)イベントを発火し、[アップストリーム](upstream.md)アプリが対応できます：

| プロビジョニングアクション | Webhookイベント | アプリがすべきこと |
|------------------------|---------------|-----------------|
| 新規ユーザー作成 | `user.created` | アプリレベルのプロファイル作成 |
| ユーザーロール変更 | `user.role_changed` | 権限更新 |
| ユーザー無効化 | `user.deleted` | ユーザーデータのアーカイブ |
| ユーザー再有効化 | `user.created` | ユーザープロファイルの復元 |

## よくある間違いと攻撃

### 間違い1：デプロビジョニングプロセスがない

デプロビジョニングなしのプロビジョニングはセキュリティホールです。元従業員がシステムへのアクセスを保持します。ライフサイクルの両方を常に実装してください。

### 間違い2：テナント割り当てなしのJITプロビジョニング

ユーザーがOAuth経由でログインしてテナントなしでJITプロビジョニングすると、コンテキストなしでシステムに存在します。プロビジョニング中に常にテナント割り当てが必要です（招待コード、ドメインマッチング、または管理者承認経由）。

### 間違い3：デプロビジョニング時にセッションを取り消さない

既存のセッションが有効なままでは`active: false`に設定するだけでは不十分です。デプロビジョニング時に常にすべてのセッションを取り消してください。

### 間違い4：制御なしの自己登録を許可する

誰でもプロビジョニングできると（例：パブリックサインアップ）、スパムアカウントや不正アクセスが発生する可能性があります。招待またはドメインベースの制限を使ってください。

### 攻撃：プロビジョニングエスカレーション

SCIMアクセスを持つ攻撃者が自身をADMINとしてプロビジョニングする。防御：SCIMでプロビジョニングされたユーザーはデフォルトで最小[ロール](role.md)を取得。管理者昇格には別の承認フローが必要。

### 攻撃：ゴーストアカウントの悪用

データがクリーンアップされていないデプロビジョニング済みユーザーが、残存するアクセストークンやAPIキーを持っている可能性。防御：デプロビジョニング時に`active: false`に設定するだけでなく、すべてのトークンとセッションを取り消す。
