# プロキシの種類：フォワード、リバース、認証（Proxy Types）

[English version](proxy-types.md)

---

## これは何？

プロキシとは仲介者として機能するサーバーです -- 2つの当事者の間に位置し、一方の代わりに通信を処理します。プロキシにはいくつかの種類があり、誰を代表するか、何をするかが異なります。

会社の人に例えるとわかりやすいです：

- **フォワードプロキシ = あなたの秘書。** 外部に手紙を送りたいとき、秘書があなたの代わりに送ります。受取人には秘書の差出人住所が見え、あなたの住所は見えません。秘書はあなたの身元を守ります。

- **リバースプロキシ = 会社の受付。** 外部の誰かが社内の部署に連絡したいとき、受付に話し、受付が適切な人に取り次ぎます。訪問者は社内のオフィス配置を見ることはありません。受付は会社の内部構造を守ります。

- **認証プロキシ = 警備員。** 誰かがビルに入る前に、警備員がバッジをチェックします。警備員は人を案内したり身元を隠したりしません -- 「入る許可があるか？」を確認するだけです。

---

## フォワードプロキシ

フォワードプロキシはクライアント（ユーザー）の代わりに動作します。ユーザーとインターネットの間に位置します。

```
ユーザー → [フォワードプロキシ] → インターネット → サーバー

サーバーは本当のユーザーが誰か知りません。
プロキシがユーザーの身元を隠します。
```

**実例：**
- 従業員が会社のネットワーク経由でインターネットにアクセスする企業VPN
- 地域的なコンテンツ制限を回避するウェブプロキシ
- 複数のプロキシを経由してユーザーを匿名化するTor

**守る対象：** クライアント（ユーザー）。

volta-auth-proxyはフォワードプロキシではありません。サーバーからユーザーの身元を隠すのではなく、X-Volta-*ヘッダーを通じて明らかにします。

---

## リバースプロキシ

リバースプロキシはサーバー（アプリケーション）の代わりに動作します。インターネットとサーバーの間に位置します。

```
ユーザー → インターネット → [リバースプロキシ] → サーバーA、B、C

ユーザーはどのサーバーと話しているか知りません。
プロキシがサーバーの内部構造を隠します。
```

**実例：**
- `wiki.example.com`を1つのサーバーに、`admin.example.com`を別のサーバーにルーティングするTraefik
- 10台の同一アプリサーバーにトラフィックを分散するNginx（ロードバランシング）
- 攻撃をブロックしコンテンツをキャッシュするためにウェブサイトの前に位置するCloudflare

**守る対象：** サーバー（アプリケーション）。

voltaのアーキテクチャでは、Traefikがリバースプロキシです。すべての着信トラフィックを受け取り、正しいバックエンドアプリケーションにルーティングします。

## 認証プロキシ

認証プロキシは1つの質問に答える専門サービスです：「このリクエストは認可されているか？」トラフィックをルーティングしません。身元を隠しません。認証情報をチェックして、はい・いいえと答えるだけです。

```
ユーザー → [リバースプロキシ] → 質問 → [認証プロキシ]：「これはOK？」
                                        認証プロキシ：「はい、この人です。」
           [リバースプロキシ] → アプリに転送（IDヘッダー付き）
```

認証プロキシには2つのパターンがあります：

### パターン1：認証付きフルリバースプロキシ（voltaではない）

認証プロキシがトラフィックパスに位置します。すべてのデータが通過します：

```
ユーザー → [認証プロキシ] → アプリ

認証プロキシがすべてを見る：ヘッダー、ボディ、レスポンス。
リバースプロキシと認証チェッカーの両方。
問題：ボトルネック、プライバシーリスク、複雑さ。
```

例：Ory Oathkeeper（プロキシモード）、一部のAPIゲートウェイ。

### パターン2：ForwardAuthサイドカー（volta）

認証プロキシはリバースプロキシの横に位置し、トラフィックパスには入りません。ヘッダーのみ見ます：

```
ユーザー → [Traefik] → アプリ
               │
          [volta-auth-proxy]
          （ヘッダーのみ）
```

認証プロキシは相談されるだけで、通過されません。トラフィックはTraefikからアプリに直接流れます。voltaはヘッダーを受け取り、ID情報付きのはい・いいえを返すだけです。

## voltaの位置づけ

```
                    フォワード  リバース    認証
                    プロキシ    プロキシ    プロキシ
守る対象：          クライアント サーバー    アクセス
トラフィックを見る：すべて       すべて      ヘッダーのみ（ForwardAuth）
voltaの構成では：   N/A         Traefik    volta-auth-proxy
```

volta-auth-proxyはForwardAuthパターンを使用する認証プロキシです。トラフィックをルーティングしません（それはTraefikの仕事）。身元を隠しません（明らかにします）。リクエストボディを見ません（ヘッダーのみ）。唯一の仕事は「このリクエストは認証・認可されたユーザーからか？」に答え、答えが「はい」のときにID情報を提供することです。

---

## volta-auth-proxyの「プロキシ」の意味

「volta-auth-proxy」という名前は紛らわしいかもしれません。voltaは従来のプロキシではないからです。トラフィックパスには位置しません。より正確な名前は「volta-auth-service」や「volta-auth-checker」かもしれません。しかし「auth proxy」は、トラフィックパスにあるかどうかに関係なく、アプリケーションの代わりに認証判断を処理するサービスの業界標準用語です。

volta-auth-proxyの「プロキシ」の意味：「voltaが認証を処理するので、あなたのアプリは処理する必要がない。」プロキシしているのはトラフィックではなく責任です。

---

## volta-auth-proxyでは

voltaはForwardAuthパターンを使用する認証プロキシです -- Traefik（リバースプロキシ）の横に位置し、ヘッダーのみを受け取り、実際のトラフィックを見たりルーティングしたりすることなく「このリクエストは認可されているか？」に答えます。
