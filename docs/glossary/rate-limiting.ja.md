# Rate Limiting（レート制限）

## これは何？

レート制限とは、クライアントが一定の時間内にサーバーに送れるリクエスト数を制御する手法です。制限を超えたクライアントには、時間枠がリセットされるまで追加のリクエストを拒否します（通常は 429 "Too Many Requests" レスポンス）。

クラブの入場制限をするドアマンのようなものです。クラブが満員になったら、誰かが出るまで新しい人は入れません。レート制限は API に対して同じことをしますが、グローバルではなくクライアントごとです。

```
  クライアントが毎分 200 リクエスト送信（制限は 200）:

  リクエスト 1   --> 200 OK
  リクエスト 2   --> 200 OK
  ...
  リクエスト 200 --> 200 OK
  リクエスト 201 --> 429 Too Many Requests (ブロック！)
  リクエスト 202 --> 429 Too Many Requests (ブロック！)

  (次の 1 分が開始)
  リクエスト 203 --> 200 OK (カウンターリセット)
```

## なぜ重要？

レート制限は以下の脅威からアプリケーションを保護します:

**ブルートフォース攻撃。** パスワードを推測しようとする攻撃者は何千もの組み合わせを試します。レート制限はこれを劇的に遅くします。毎分 200 回の制限があると、数秒で解読される 6 文字のパスワードも、はるかに長い時間がかかります。

**サービス拒否攻撃。** レート制限がないと、1 つのクライアント（またはボットネット）がリクエストでサーバーを圧倒し、正規のユーザーが使えなくなります。

**リソースの枯渇。** 悪意のないクライアントでも偶然に過大な負荷を生むことがあります。無限ループのバグがあるスクリプト、設定ミスのクーロンジョブ、失敗したリクエストを過剰にリトライするモバイルアプリなど。

**クレデンシャルスタッフィング。** 攻撃者はデータ漏洩から盗んだユーザー名/パスワードの組み合わせリストを使って、ログインエンドポイントに試行します。レート制限はこれを大規模には非実用的にします。

**コスト保護。** アプリが外部の有料 API（AI サービスや SMS プロバイダなど）を呼び出す場合、暴走したクライアントが莫大な請求を発生させる可能性があります。

## どう動くの？

### 一般的なアルゴリズム

**固定ウィンドウカウンター**（volta が使用）:
時間を固定のウィンドウ（例: 1 分間隔）に分割します。各ウィンドウでクライアントごとのリクエストをカウントします。各ウィンドウの開始時にリセットします。

```
  時間:       |  1 分目    |  2 分目     |  3 分目     |
  リクエスト:  |  ||||||||| |  ||||      |  ||||||    |
  カウント:    |  150       |  40        |  60        |
  制限:       |  200       |  200       |  200       |
  状態:       |  OK        |  OK        |  OK        |
```

欠点: 境界でのバースト。クライアントが 1 分目の終わりに 200 リクエスト、2 分目の開始に 200 リクエストを送ると、短期間で実質 400 リクエストになります。

**スライディングウィンドウ:**
固定の境界の代わりに、ローリングする時間期間でリクエストを追跡します。より正確ですが、メモリ使用量が多くなります。

**トークンバケット:**
各クライアントにトークンを保持する「バケット」があります。トークンは一定の速度で追加されます（例: 毎秒 10 個）。各リクエストがトークンを 1 つ消費します。バケットが空ならリクエストは拒否されます。バケットには最大容量があり、短いバーストを許可します。

```
  トークンバケット（容量=10、補充=5/秒）:

  時間 0:  バケット = 10 トークン
  バースト: 10 リクエスト --> バケット = 0（10 個全て即座に処理）
  時間 1秒: バケット = 5（5 トークン補充）
  時間 2秒: バケット = 10（さらに 5 補充、10 で上限）
```

**リーキーバケット:**
トークンバケットに似ていますが、リクエストは一定の速度で処理されます。超過リクエストは即座に拒否されるのではなく、（上限まで）キューに入ります。

### キーとして何を使うか

| キー | いつ使うか | 利点 | 欠点 |
|------|-----------|------|------|
| IP アドレス | 未認証エンドポイント（ログインページ） | シンプル、認証前に動作 | 共有 IP（NAT、VPN）で複数ユーザーが影響を受ける |
| ユーザー ID | 認証済みエンドポイント（API） | 正確なユーザーごとの制限 | 先に認証が必要 |
| API キー | マシン間 API | 明確な責任追跡 | キー共有で無効化される |
| IP + エンドポイント | 細かい制御 | 特定エンドポイントの乱用を防止 | 管理が複雑 |

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は `ConcurrentHashMap` を使った固定ウィンドウカウンターでレート制限を実装しています。

**実装。** `RateLimiter` クラスは固定の 1 分ウィンドウを使います。各分の境界でキーごとに新しいカウンターバケットが作成されます:

- 同じ分内のリクエストは共有カウンターをインクリメントする。
- 新しい分が始まると、カウンターが 1 にリセットされる。
- カウンターが設定された上限（毎分 200 リクエスト）を超えると、リクエストが拒否される。

**設定。** volta は毎分 200 リクエストの制限でレートリミッターを作成します:
```java
RateLimiter rateLimiter = new RateLimiter(200);
```

**キーの選択。** volta はレート制限のキーとしてクライアントの IP アドレスを使います。レート制限はユーザー ID がまだない未認証エンドポイント（ログインフローなど）で最も重要なため、これが適切です。

**適用箇所。** レートリミッターはログイン試行のような機密操作の処理前にチェックされ、OIDC フローへのブルートフォース攻撃を抑制します。

フロー:

```
  クライアント (IP: 203.0.113.42)
  |
  | /login へのリクエスト
  |
  v
  RateLimiter.allow("203.0.113.42")
  |
  |-- 現在の分のバケットが存在する?
  |   |
  |   いいえ: 新しいバケットを作成、count=1、許可
  |   はい:   カウントをインクリメント
  |            |
  |            count <= 200? --> リクエストを許可
  |            count > 200?  --> 429 で拒否
  |
  v
  リクエストを処理（許可された場合）
```

**スレッドセーフティ。** volta のレートリミッターはアトミックなカウンター更新のために `ConcurrentHashMap.compute()` を使用しています。Web サーバーは複数のリクエストを同時に処理するため、スレッドセーフな操作がないとカウンターが不正確になる可能性があります。

**メモリ管理。** 古いバケットは新しい分が始まると自然に置き換えられます（`compute()` コールバックがバケットの分が変わると新しい `Counter` を作成する）。つまり、クライアントがリクエストを続ける限り、古いエントリはガベージコレクションされます。

## よくある間違い

**1. ログインエンドポイントにレート制限をかけない。**
ログインエンドポイントはブルートフォース攻撃の最大のターゲットです。API エンドポイントにだけレート制限をかけて、玄関のドアを開けっ放しにしないでください。

**2. レート制限が寛大すぎる。**
毎分 10,000 リクエストの制限は、ないのと同じです。正規のユーザーが実際に必要とするものに基づいて制限を設定しましょう。ログインエンドポイントなら、IP あたり毎分 10-20 回の試行が妥当です。

**3. リバースプロキシの背後で IP ベースのレート制限だけを使う。**
全てのリクエストがロードバランサーやリバースプロキシを通ると、全て同じ IP から来ているように見える場合があります。実際のクライアント IP を取得するために `X-Forwarded-For` ヘッダを使いましょう。volta は `clientIp()` ヘルパーでこれを行います。

**4. 適切な 429 レスポンスを返さない。**
レート制限が発動したら、`429 Too Many Requests` ステータスと、クライアントにいつ再試行すべきか伝える `Retry-After` ヘッダを返しましょう。これがないと、クライアントが即座にリトライして状況を悪化させる可能性があります。

**5. 1 つのレイヤーだけでレート制限する。**
複数のレベルでレート制限を適用しましょう: リバースプロキシ（Traefik/Nginx）、アプリケーションレベル（volta）、場合によってはデータベースクエリレベルでも。各レイヤーが異なる種類の乱用をキャッチします。

**6. 分散デプロイメントを忘れる。**
アプリが複数のサーバーで動作する場合、インメモリのレートリミッター（volta の `ConcurrentHashMap` のような）は特定のサーバーにヒットしたリクエストだけをカウントします。分散レート制限には Redis のような共有ストアを使いましょう。volta の現在の実装は単一インスタンスのデプロイメント向けに設計されています。

**7. 正当なバーストを考慮しない。**
一部の正当な使用パターンは短いバーストを伴います（ページ読み込みで同時に 20 の API コールが発生するなど）。トークンバケットアルゴリズムは最大容量までのバーストを許可するため、厳密な固定ウィンドウカウンターよりもこのケースをうまく処理できます。
