# RBAC（ロールベースアクセス制御）

[English version](rbac.md)

---

## これは何？

ロールベースアクセス制御（RBAC）は、ユーザーに直接権限を与える代わりに、ロール（役割）を割り当てて権限を管理する方法です。「太郎はページを編集でき、ユーザーを作成でき、ログを見られる」と言う代わりに、「太郎はADMINだ」と言い、ADMINが何をできるかを定義します。権限を変更する必要があるとき、個々のユーザーではなくロールの定義を変更します。

会社の肩書きのようなものです。各従業員ができるすべてのタスクをリストアップする代わりに、ロール（マネージャー、エンジニア、インターン）を定義し、ロールに権限を割り当てます。新しいエンジニアが入社すると、自動的にすべてのエンジニア権限を得ます。エンジニアがマネージャーに昇進すると、権限が自動的に変わります。

---

## なぜ重要なのか？

RBACがないと：

- ユーザーごとリソースごとに権限を管理する必要がある。これはスケールしない
- 新しい従業員は、セットアップした人によって一貫性のない権限を得る
- 「誰が何をできるか」の監査には、すべてのユーザーを個別にチェックする必要がある
- 退職者のアクセス削除には、すべての個別権限を覚えている必要がある

RBACはこれらすべてをシンプルにします。少数のロールを管理するだけで、ユーザーはロールから権限を継承します。

## どう動くのか？

### 基本モデル

```
  ┌─────────────────────────────────────────────────────────┐
  │                       RBACモデル                          │
  │                                                          │
  │  ユーザー ──── 持つ ──── ロール ──── 付与する ──── 権限    │
  │                                                          │
  │  太郎 ──────── ADMIN ──────── メンバー管理可能             │
  │                                ユーザー招待可能             │
  │                                設定変更可能                │
  │                                                          │
  │  花子 ──────── MEMBER ─────── コンテンツ閲覧可能           │
  │                                自分のコンテンツ編集可能     │
  │                                                          │
  │  ゲスト ────── VIEWER ────── コンテンツ閲覧のみ            │
  └─────────────────────────────────────────────────────────┘
```

### voltaの4段階ロール階層

voltaは4つのロールのシンプルな順序付き階層を使います：

```
  OWNER > ADMIN > MEMBER > VIEWER

  ┌─────────┐
  │  OWNER  │  すべてができる。テナントの削除、所有権の移譲。
  │         │  テナントには常に少なくとも1人のOWNERが必要。
  ├─────────┤
  │  ADMIN  │  人の管理ができる：メンバー招待、ロール変更（ADMINまで）、
  │         │  メンバー削除、テナント設定変更。
  ├─────────┤
  │  MEMBER │  通常の利用。MEMBERロールを許可するすべてのアプリを使用可能。
  │         │  他のユーザーの管理はできない。
  ├─────────┤
  │  VIEWER │  読み取り専用アクセス。コンテンツの閲覧はできるが変更はできない。
  │         │  強制は個々のアプリに委ねられる。
  └─────────┘
```

### ロール階層は「上位は下位を含む」を意味する

```
  アプリがMEMBERロールを要求する場合：

  OWNER  → ✓（OWNER > ADMIN > MEMBERなので、OWNERはMEMBERを含む）
  ADMIN  → ✓（ADMIN > MEMBERなので、ADMINはMEMBERを含む）
  MEMBER → ✓（完全一致）
  VIEWER → ✗（VIEWER < MEMBER）
```

### テナントスコープのロール

voltaでは、ロールはグローバルではなく**テナントごと**です。ユーザーは異なるテナントで異なるロールを持てます：

```
  ユーザー: taro@example.com

  ┌─────────────────────────────────┐
  │ テナント: ACME Corp             │
  │ ロール: OWNER                   │
  │                                 │
  │ 太郎がACMEを作った。所有者。     │
  │ すべてを完全にコントロール。      │
  └─────────────────────────────────┘

  ┌─────────────────────────────────┐
  │ テナント: Globex Inc            │
  │ ロール: MEMBER                  │
  │                                 │
  │ 太郎はGlobexに招待された。       │
  │ アプリは使えるがメンバー管理は   │
  │ できない。                       │
  └─────────────────────────────────┘

  ┌─────────────────────────────────┐
  │ テナント: Initech               │
  │ ロール: VIEWER                  │
  │                                 │
  │ 太郎は読み取り専用アクセス。     │
  │ 閲覧はできるが編集はできない。   │
  └─────────────────────────────────┘
```

これは`memberships`テーブルに保存されます：

```sql
-- 太郎のメンバーシップ
SELECT tenant_id, role FROM memberships WHERE user_id = 'taro-uuid';

-- 結果:
-- acme-uuid    | OWNER
-- globex-uuid  | MEMBER
-- initech-uuid | VIEWER
```

## volta-auth-proxyではどう使われているか？

### JWT内のロール

voltaがJWTを発行するとき、現在のテナントでのユーザーのロールが含まれます：

```json
{
  "sub": "taro-uuid",
  "volta_tid": "acme-uuid",
  "volta_roles": ["OWNER"],
  "volta_tname": "ACME Corp"
}
```

アプリは`volta_roles`を読んで認可の判断をします：

```java
app.delete("/api/projects/:id", ctx -> {
    VoltaUser user = VoltaAuth.getUser(ctx);
    if (!user.hasRole("ADMIN")) {
        throw new ForbiddenResponse("ADMINまたはOWNERのみプロジェクトを削除できます");
    }
    // ... プロジェクトを削除
});
```

### ForwardAuthでのロール

`volta-config.yaml`で、各アプリがアクセスできるロールを指定します：

```yaml
apps:
  - id: app-wiki
    url: https://wiki.example.com
    allowed_roles: [MEMBER, ADMIN, OWNER]

  - id: app-admin
    url: https://admin.example.com
    allowed_roles: [ADMIN, OWNER]
```

ForwardAuth（`/auth/verify`）はアクセスを許可する前に、ユーザーのロールをアプリの`allowed_roles`と照合します。つまりVIEWERはapp-adminに到達することすらできません。ゲートウェイレベルでブロックされます。

### ロール管理

ロールは以下の方法で管理されます：

1. **管理UI：** `/admin/members` -- ロール変更のビジュアルインターフェース
2. **Internal API：** `PATCH /api/v1/tenants/{tid}/members/{uid}` -- プログラムによるロール変更
3. **招待：** 招待作成時にADMINが招待者が受け取るロールを指定

### OWNERの保護

voltaには重要な安全ルールがあります：**テナントの最後のOWNERは降格も削除もできません。** これにより、テナントを管理できる人がいなくなる状況を防ぎます。

```
  テナント: ACME Corp
  メンバー:
    太郎 (OWNER)  ← 最後のOWNER
    花子 (ADMIN)

  太郎が自分をMEMBERに降格しようとする：
  → volta: 「降格できません。あなたが最後のOWNERです。
            先に他の誰かに所有権を移譲してください。」

  花子が太郎を削除しようとする：
  → volta: 「最後のOWNERは削除できません。」
```

### ロール変更のルール

| 操作者 | 変更先に設定可能 | 制約 |
|--------|-----------------|------|
| OWNER | OWNER, ADMIN, MEMBER, VIEWER | OWNERへの昇格が可能（所有権の移譲） |
| ADMIN | ADMIN, MEMBER, VIEWER | OWNERへの昇格は不可。他のADMINの降格は不可。 |
| MEMBER | （ロール変更不可） | |
| VIEWER | （ロール変更不可） | |

## よくある間違いと攻撃

### 間違い1：UIだけでロールをチェックする

ロールチェックがフロントエンドだけ（例：非管理者にはボタンを非表示）なら、攻撃者がAPIを直接呼べます。必ずサーバーサイドでロールをチェックしてください。

```
  悪い例：フロントエンドが非管理者に「削除」ボタンを非表示。
          APIエンドポイントにはロールチェックなし。
          → 攻撃者がDELETE /api/projects/1を直接呼ぶ。

  良い例：フロントエンドがボタンを非表示にし、かつ
          APIエンドポイントがuser.hasRole("ADMIN")をチェック。
```

### 間違い2：ロール階層が一貫して強制されていない

一部のエンドポイントが`role == "ADMIN"`（完全一致）をチェックし、他が`role >= "ADMIN"`（階層）をチェックすると、OWNERが管理者エンドポイントからブロックされる可能性があります。常に階層的な比較を使ってください。

### 間違い3：ロール変更をログに記録しない

ロール変更はセキュリティ上の重要な操作です。攻撃者が自分をOWNERに昇格した場合、監査証跡が必要です。voltaはすべてのロール変更を`audit_logs`テーブルに記録します。

### 攻撃：権限昇格

攻撃者が自分のロールを変更する方法を見つける（例：操作者の権限を検証しないロール変更APIのバグ）。防御：リクエストが認証されているだけでなく、操作者がロール変更を行う権利を持っていることを常に検証する。

### 攻撃：混乱した代理人

ADMIN権限で動作するアプリ（サービストークン経由）が、VIEWERの代わりにアクションを実行するよう騙される可能性がある。防御：操作を委任する際、サービストークンのロールではなく、元のユーザーのロールを常にチェックする。
