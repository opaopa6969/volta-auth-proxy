# レルム（Keycloak Realm）

[English version](realm.md)

---

## これは何？

Keycloak Realmは、[Keycloak](keycloak.ja.md)サーバー内の独立したセキュリティドメインです。各レルムは独自のユーザー、ロール、クライアント（アプリケーション）、IDプロバイダー、ログインページ、設定を持ちます。レルム間でデータは共有されません -- レルムAのユーザーは、レルムBに別途登録されない限りログインできません。

同じ管理会社が管理する別々のマンションのようなものです。各マンションには独自の鍵、郵便受け、ロビーのセキュリティルール、入居者リストがあります。マンションAに住んでいても、同じ会社が管理していてもマンションBにはアクセスできません。

---

## なぜ重要なのか？

レルムはKeycloakの[マルチテナンシー](multi-tenant.ja.md)への回答です。チームがKeycloakをマルチテナントSaaS（各顧客組織が別のテナント）に使おうとすると、通常テナントごとに1つのレルムを作成します。少数のテナントなら機能しますが、スケールすると深刻な問題が生じます。

レルムを理解すると、volta-auth-proxyが異なるマルチテナンシーモデルで構築された核心的な理由の1つが見えてきます。

---

## Keycloakでのレルムの仕組み

```
  Keycloakサーバー
  ┌───────────────────────────────────────────────────────┐
  │                                                       │
  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
  │  │ Realm: Acme  │  │ Realm: Beta  │  │ Realm: Gamma│  │
  │  │              │  │              │  │             │  │
  │  │ ユーザー:    │  │ ユーザー:    │  │ ユーザー:   │  │
  │  │  - alice     │  │  - bob       │  │  - charlie  │  │
  │  │  - dave      │  │  - eve       │  │  - frank    │  │
  │  │              │  │              │  │             │  │
  │  │ クライアント: │  │ クライアント: │  │ クライアント:│  │
  │  │  - wiki-app  │  │  - wiki-app  │  │  - wiki-app │  │
  │  │  - admin-app │  │  - admin-app │  │             │  │
  │  │              │  │              │  │             │  │
  │  │ ロール:      │  │ ロール:      │  │ ロール:     │  │
  │  │  - admin     │  │  - admin     │  │  - admin    │  │
  │  │  - user      │  │  - user      │  │  - user     │  │
  │  └─────────────┘  └─────────────┘  └─────────────┘  │
  │                                                       │
  │  各レルムは完全に別世界。何も共有されない。             │
  └───────────────────────────────────────────────────────┘
```

各レルムは完全に独立しています：
- ユーザーは各レルムに別々に作成する必要がある
- クライアントアプリケーション（OAuth2クライアント）は各レルムに別々に設定する必要がある
- ロール、グループ、権限はレルムごとに定義
- ログインページとテーマはレルムごとに設定
- 各レルムに独自の[realm.json](realm-json.ja.md)設定（各500行以上）

## マルチテナンシーにおけるレルムの限界

### 問題1: 一人のユーザー、複数テナント

多くのSaaSプラットフォームでは、一人の人が複数の組織に所属します。AliceはAcme CorpではADMIN、Beta IncではVIEWERかもしれません。Keycloakレルムでは：

```
  alice@gmail.com がAcmeとBetaの両方に入りたい：

  選択肢A: 別々のアカウントを作成
  ┌─────────────┐  ┌─────────────┐
  │ Realm: Acme  │  │ Realm: Beta  │
  │ alice (admin)│  │ alice (viewer)│  ← 2つのアカウント、2つのパスワード、
  └─────────────┘  └─────────────┘    2つのログインセッション。紛らわしい。

  選択肢B: レルム間フェデレーションを構築
  ┌─────────────┐  ┌─────────────┐
  │ Realm: Acme  │──│ Realm: Beta  │  ← 複雑。ネイティブ非対応。
  │ alice (admin)│  │ alice (viewer)│    カスタムSPI開発が必要。
  └─────────────┘  └─────────────┘
```

どちらも良くありません。SaaSユーザーは1つのアカウント、1回のログイン、複数のワークスペースを期待します。

### 問題2: スケール

100以上のレルムでKeycloakの管理コンソールが遅くなります。1,000以上のレルムで管理が深刻な運用課題になります。各レルムは設定のオーバーヘッド、メモリ使用量、起動時間を増やします。

### 問題3: テナント作成がセルフサービスでない

Keycloakレルムの作成は管理者操作です。通常Keycloak管理者アクセスが必要です。SaaS製品では、テナント作成をセルフサービスにしたい -- ユーザーがサインアップすると組織が自動的に作成される。Keycloakでは、Admin API経由でレルムを作成するカスタム自動化が必要です。

### 問題4: 設定の重複

すべてのレルムに独自のクライアント設定、ロール定義、IDプロバイダー設定、テーマ割り当てが必要です。100テナントがあり、全テナントに新しいOAuth2クライアントを追加したい場合、100のレルム設定を更新する必要があります。

## voltaのテナントモデル vs Keycloakのレルムモデル

voltaは根本的に異なるアプローチを取ります：

```
  voltaデータモデル：
  ┌──────────────────────────────────────────────────┐
  │ PostgreSQL                                        │
  │                                                    │
  │  users              tenants          memberships   │
  │  ┌──────────┐      ┌──────────┐    ┌───────────┐ │
  │  │ alice     │──┐   │ Acme     │    │ alice     │ │
  │  │ bob       │  ├──►│ Beta     │◄───│  → Acme   │ │
  │  │ charlie   │  │   │ Gamma    │    │    (ADMIN) │ │
  │  └──────────┘  │   └──────────┘    │  → Beta   │ │
  │                │                    │    (VIEWER)│ │
  │                │                    │ bob       │ │
  │                └───────────────────►│  → Acme   │ │
  │                                     │    (MEMBER)│ │
  │                                     └───────────┘ │
  └──────────────────────────────────────────────────┘
```

| 観点 | Keycloak Realm | volta Tenant |
|------|---------------|--------------|
| ストレージ | 独立した設定ドメイン | `tenants`テーブルの行 |
| ユーザーID | レルムごと（重複） | グローバル（1アカウント、複数テナント） |
| 複数所属 | ネイティブ非対応 | コア機能（テナントごとにロール） |
| 作成 | Admin API、複雑 | API呼び出し、セルフサービス |
| 設定 | テナントごとにフルrealm.json | 共有アプリ設定、テナントは単なるデータ |
| スケール上限 | ~100で問題発生 | データベースの行数のみが制限 |
| テナント横断 | 困難 | ネイティブ（セッション内でテナント切り替え） |

### voltaがレルムの機能をどう処理するか

| レルムの機能 | voltaの対応 |
|-------------|------------|
| ユーザー分離 | すべてのクエリの`tenant_id`カラム + 行レベル分離 |
| ロール割り当て | `(user_id, tenant_id, role)`の`memberships`テーブル |
| クライアント/アプリ設定 | `volta-config.yaml`（全テナント共有） |
| ログインテーマ | 全テナント共通のログインページ（設定でテナントブランディング可能） |
| IDプロバイダー | テナント間で共有のOIDCプロバイダー（Googleなど） |

voltaのモデルがよりシンプルなのは、テナントを**設定ドメイン**ではなく**データ**として扱うからです。テナントは500行のJSON設定ファイルではなく、データベーステーブルの行です。これにより、テナント作成は瞬時に、マルチテナント所属は自然に、テナント横断操作は簡単になります。
