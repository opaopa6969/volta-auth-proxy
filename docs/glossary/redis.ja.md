# Redis

[English version](redis.md)

---

## これは何？

Redisはオープンソースのインメモリデータストアで、データベース、キャッシュ、メッセージブローカーとして使用できます。データをRAMに格納するため、極めて高速です -- PostgreSQLのような従来のディスクベースデータベースよりはるかに速い。

オフィスの共有ホワイトボードで考えてみましょう。オフィスの全員がすぐに読み書きできます。全同僚に素早く伝えたいこと -- 例えば「会議室Aは使用中」-- があれば、ホワイトボードに書きます。通りかかった人はファイリングキャビネットを探すことなく確認できます。PostgreSQLはファイリングキャビネット（永続的、整理済み、だが遅い）。Redisはホワイトボード（速い、全員に見える、だが消すと消える）。

アプリケーションレベルのキャッシュ（[Caffeine](caffeine-cache.ja.md)など）との大きな違いは、Redisが別プロセスとして動くことです。複数のアプリケーションインスタンスが同じRedisに読み書きでき、[水平スケーリング](horizontal-scaling.ja.md)に不可欠です。

---

## なぜ重要なのか？

- **インスタンス間の共有状態。** [ロードバランサ](load-balancer.ja.md)の後ろにアプリケーションの複数コピーを動かすとき、すべてがセッション状態について合意する必要がある。Redisがこの共有メモリを提供。
- **速度。** Redis操作はマイクロ秒で完了する。セッション検証、レート制限、キャッシュ参照はほぼ瞬時。
- **永続化オプション。** 純粋なインメモリキャッシュとは異なり、Redisはオプションでデータをディスクに永続化でき、再起動に耐える。
- **データ構造。** Redisは単なるキーバリューではない。文字列、ハッシュ、リスト、セット、ソート済みセットなどをサポート -- レート制限カウンターやセッションデータに最適。
- **Pub/Subメッセージング。** Redisはアプリケーションインスタンス間でメッセージをブロードキャストできる（例：「セッションXが無効化された、ローカルキャッシュを無効にせよ」）。

## どう動くのか？

### Redisアーキテクチャ

```
  ┌─────────────────────────────────────────┐
  │              Redisサーバー                │
  │                                          │
  │  ┌──────────────────────────────────┐   │
  │  │           RAM                     │   │
  │  │                                   │   │
  │  │  "session:abc123" → {user, role}  │   │
  │  │  "session:def456" → {user, role}  │   │
  │  │  "rate:192.168.1.1" → 47          │   │
  │  │  "tenant:acme:config" → {...}     │   │
  │  │                                   │   │
  │  └──────────────────────────────────┘   │
  │                                          │
  │  オプション: スナップショットをディスクに永続化│
  └─────────────────────────────────────────┘
        ▲           ▲           ▲
        │           │           │
  ┌─────┴──┐  ┌────┴───┐  ┌───┴────┐
  │ volta  │  │ volta  │  │ volta  │
  │ inst.1 │  │ inst.2 │  │ inst.3 │
  └────────┘  └────────┘  └────────┘
```

### Redisのデータ型

| 型 | ユースケース | 例 |
|----|------------|-----|
| **String** | 単純な値、カウンター | `SET rate:1.2.3.4 "47"` |
| **Hash** | 構造化オブジェクト | `HSET session:abc user_id "u1" role "ADMIN"` |
| **List** | 順序付きシーケンス | 監査ログエントリ |
| **Set** | ユニークなコレクション | ユーザーごとのアクティブセッションID |
| **Sorted Set** | ランク付きデータ | リーダーボード、有効期限キュー |
| **キー有効期限** | 自動削除 | `EXPIRE session:abc 3600`（1時間） |

### 基本的なRedis操作

```
  SET session:abc123 '{"userId":"u1","role":"ADMIN"}' EX 3600
  │                    │                                │
  │                    │                                └─ 1時間で期限切れ
  │                    └─ 値（JSON文字列）
  └─ キー

  GET session:abc123
  → '{"userId":"u1","role":"ADMIN"}'

  DEL session:abc123
  → （キー削除、セッション無効化）

  INCR rate:192.168.1.1
  → 48（アトミックインクリメント、レート制限に最適）
```

### Caffeine（ローカル）vs Redis（共有）

```
  Caffeine (Phase 1):              Redis (Phase 2):
  ┌────────────┐                   ┌────────────────┐
  │ volta (1)  │                   │    Redis       │
  │            │                   │  (共有)        │
  │ ┌────────┐ │                   └──┬──────┬──────┘
  │ │Caffeine│ │                      │      │
  │ │ cache  │ │                   ┌──┴──┐ ┌─┴───┐
  │ └────────┘ │                   │volta│ │volta│
  └────────────┘                   │  1  │ │  2  │
                                   └─────┘ └─────┘
  1インスタンスのみが               すべてのインスタンスが
  キャッシュデータを見れる。         同じデータを見れる。

  inst.1にキャッシュされたセッション？ Redisにキャッシュされたセッション？
  inst.2はそれを知らない。          すべてのインスタンスが読める。
```

## volta-auth-proxy ではどう使われている？

### Phase 1：Redisなし（現在）

Phase 1では、voltaは[シングルプロセス](single-process.ja.md)として動作します。セッションはPostgreSQLに格納され、[Caffeine](caffeine-cache.ja.md)にキャッシュされます。インスタンスが1つしかないため、Redisは必要ありません。

### Phase 2：共有セッション用Redis

voltaが複数インスタンスにスケールすると、インメモリのCaffeineキャッシュが問題になります。ユーザーAがインスタンス1でログインしても、インスタンス2のローカルキャッシュにそのセッションはありません。Redisがこれを解決します：

```
  Phase 2のセッションフロー：

  1. ユーザーがログイン → voltaインスタンス1がセッション作成
  2. セッションをPostgreSQLに格納（永続）
  3. セッションをRedisにキャッシュ（高速共有アクセス）
  4. 次のリクエストがvoltaインスタンス2に到達
  5. インスタンス2がRedisを確認 → セッション発見！
  6. PostgreSQLにアクセスせずにユーザーが認証される

  ┌──────────────────────────────────────────────┐
  │  Redisありのリクエストフロー：                  │
  │                                               │
  │  ブラウザ → ロードバランサ → voltaインスタンス2  │
  │                              │                │
  │                              ▼                │
  │                         Redisを確認           │
  │                              │                │
  │                     ┌────────┴────────┐       │
  │                     │                 │       │
  │                  キャッシュヒット   キャッシュミス│
  │                     │                 │       │
  │                     ▼                 ▼       │
  │               セッション返却    Postgres照会   │
  │               (高速: <1ms)     Redisにキャッシュ│
  │                                セッション返却   │
  └──────────────────────────────────────────────┘
```

### Phase 2：レート制限用Redis

レート制限カウンターは共有されなければなりません。インスタンス1がIPから30リクエストを数え、インスタンス2が30を数えると、攻撃者は60リクエストしたのに各インスタンスは30しかしていないと思います。

```
  Redisなし（壊れている）:        Redisあり（正しい）:
  ┌──────┐  ┌──────┐            ┌──────┐  ┌──────┐
  │volta1│  │volta2│            │volta1│  │volta2│
  │ 30   │  │ 30   │            │      │  │      │
  └──────┘  └──────┘            └──┬───┘  └───┬──┘
  合計: 60だが各自30と認識          │          │
  攻撃者が制限を回避！              ▼          ▼
                                 ┌──────────────┐
                                 │    Redis      │
                                 │  count: 60    │
                                 └──────────────┘
                                 正しい合計、制限が強制される
```

### Phase 2：キャッシュ無効化用Redis Pub/Sub

セッションがインスタンス1で無効化されたとき、インスタンス2もそれを知る必要があります。Redis Pub/Subが無効化をブロードキャストします：

```
  管理者がインスタンス1でセッションを無効化
       │
       ▼
  インスタンス1: PostgresとRedisからDELETE
  インスタンス1: PUBLISH "session:revoked" "abc123"
       │
       ▼
  Redisがすべてのサブスクライバにブロードキャスト
       │
       ▼
  インスタンス2: メッセージ受信、ローカルCaffeineキャッシュから削除
```

## よくある間違いと攻撃

### 間違い1：Redisをインターネットに公開

Redisはデフォルトで認証がありません。Redisがインターネットからアクセス可能だと、誰でもすべてのセッションを読み、任意のユーザーになりすませます。Redisは内部ネットワークのみに配置すべきです。

### 間違い2：永続化もバックアップもなし

Redisが唯一のセッションストアでクラッシュすると、すべてのセッションが失われます -- 全ユーザーがログアウト。voltaはRedisをキャッシュ層として使い、PostgreSQLを真のデータソースとします。

### 間違い3：Redisをプライマリデータベースとして使う

Redisはキャッシュと高速ストアであり、PostgreSQLの代替ではありません。セッションデータはPostgresに永続的に存在し、Redisは高速アクセスを提供します。

### 間違い4：Redisのメモリ制限を無視

RedisはすべてをRAMに格納します。`maxmemory`を設定しないと、Redisが利用可能なRAMをすべて使い切りクラッシュします。退去ポリシーを設定しましょう（`maxmemory-policy allkeys-lru`）。

### 間違い5：Redisの高可用性がない

単一Redisインスタンスは単一障害点です。[高可用性](high-availability.ja.md)のために、Redis SentinelまたはRedis Clusterを使いましょう。
