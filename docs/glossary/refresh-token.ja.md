# Refresh Token（リフレッシュトークン）

## これは何？

リフレッシュトークンとは、ユーザーに再ログインを要求せずに新しいアクセストークンを取得するための、有効期間の長い認証情報です。アクセストークンが期限切れになったとき、ユーザーをログインページに戻す代わりに、アプリケーションがリフレッシュトークンを認証サーバーに提示して、新しいアクセストークンを受け取ります。

ジムの会員証のようなものです。1 日パス（アクセストークン）で今日はドアを通れます。期限が切れたら、会員証（リフレッシュトークン）をフロントに見せると、新しい 1 日パスをもらえます。毎回会員登録をし直す必要はありません。

```
  時間 --->

  [  access_token (5分)  ]  期限切れ
                              |
                              | refresh_token を使用
                              v
  [  新しい access_token (5分)  ]  期限切れ
                                    |
                                    | refresh_token を使用
                                    v
  [  新しい access_token (5分)  ]
```

## なぜ重要？

アクセストークンは、盗まれた場合の被害を制限するために意図的に短命です。しかし短命トークンには問題があります: アクセストークンが 5 分ごとに期限切れになると、ユーザーは 5 分ごとにログインし直すことになります。体験として最悪です。

リフレッシュトークンは 2 つの関心事を分離することでこれを解決します:

| アクセストークン | リフレッシュトークン |
|---|---|
| 毎回の API コールで使用 | 更新時のみ使用 |
| 短命（分単位） | 長命（時間、日、週単位） |
| 多くのサービスに送信 | 認証サーバーにのみ送信 |
| 露出が多い | 露出が少ない |
| 盗まれた場合: 限られた期間 | 盗まれた場合: 長い期間、ただし盗みにくい |

リフレッシュトークンは認証サーバーにのみ送られる（全ての API には送らない）ため、攻撃の表面積が少なくなります。アクセストークンは素早く期限切れになるため、API で傍受されても被害の窓は小さいです。

### トークンのローテーション

現代のベストプラクティスは**リフレッシュトークンのローテーション**です: リフレッシュトークンを使うたびに、認証サーバーが新しいアクセストークンと新しいリフレッシュトークンの両方を発行し、古いリフレッシュトークンを無効化します。

```
  リフレッシュトークンのローテーション:

  refresh_token_A  -->  認証サーバー  -->  access_token_2
                                      -->  refresh_token_B（A は無効に）

  refresh_token_B  -->  認証サーバー  -->  access_token_3
                                      -->  refresh_token_C（B は無効に）

  攻撃者が refresh_token_A を盗んで使おうとした場合:
  認証サーバーが既に使用済みだと検出 --> 全て失効
```

盗まれたリフレッシュトークンがリプレイされた場合、認証サーバーは再利用を検出し（トークンは既に消費済みなので）、トークンファミリー全体を失効させることができます。安全策としてユーザーをログアウトさせます。

## どう動くの？

標準的なリフレッシュフロー:

```
  アプリ                 認証サーバー               API
  |                     |                          |
  | access_token で      |                          |
  | API コール            |                          |
  |---------------------------------------------------->
  |                     |                          |
  |                     |   401 Token Expired      |
  |<----------------------------------------------------|
  |                     |                          |
  | POST /token         |                          |
  | grant_type=         |                          |
  |   refresh_token     |                          |
  | refresh_token=      |                          |
  |   <old_refresh>     |                          |
  |-------------------->|                          |
  |                     |                          |
  | {                   |                          |
  |   access_token: 新  |                          |
  |   refresh_token: 新 |  (ローテーション)          |
  | }                   |                          |
  |<--------------------|                          |
  |                     |                          |
  | 新しい access_token  |                          |
  | で API コールをリトライ|                          |
  |---------------------------------------------------->
  |                     |                          |
  |                     |   200 OK                 |
  |<----------------------------------------------------|
```

### リフレッシュトークンの保存場所

リフレッシュトークンは長命で強力なため、安全な保存が必要です:

- **サーバーサイドアプリケーション:** バックエンドに保存。ブラウザは見ない。
- **シングルページアプリ (SPA):** これが難しい部分。localStorage は XSS に脆弱。HttpOnly Cookie が最も安全。
- **モバイルアプリ:** プラットフォームのセキュアストレージに保存（iOS は Keychain、Android は Keystore）。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy はリフレッシュトークンを**使いません**。代わりに、同じ目標を達成する別のパターンを使います: **サーバーサイドセッション + 短命 JWT。**

なぜそうするか、どう動くか:

**リフレッシュトークンが解決する問題:** アクセストークンを短命に保ちつつ、再認証なしでユーザーのログイン状態を維持する。

**volta の代替ソリューション:**

```
  従来のアプローチ:                volta のアプローチ:

  ブラウザが保存:                  ブラウザが保存:
  - access_token（短命）          - セッション Cookie（HttpOnly）
  - refresh_token（長命）
                                  サーバーが保存:
                                  - PostgreSQL のセッションレコード
                                  - （ユーザー、テナント、有効期限など）

  access_token が期限切れ:         JWT が期限切れ:
  refresh_token を認証サーバーに   セッション Cookie を
  送り、新しい access_token を取得 /auth/verify か /auth/refresh に送信
                                  volta がセッションを検証し、
                                  新しい JWT を発行
```

重要な洞察: volta のセッション Cookie は機能的にはリフレッシュトークンと同等ですが、いくつかの利点があります:

1. **HttpOnly と SameSite。** セッション Cookie には `HttpOnly` と `SameSite=Lax` フラグが付いており、JavaScript からアクセスできず、クロスサイト攻撃にも耐性があります。localStorage に保存されたリフレッシュトークンにはこれらの保護がありません。

2. **サーバーサイドの状態。** セッション ID は意味のない UUID です。全てのセッションデータ（ユーザーは誰か、どのテナントか、いつ期限切れか、失効されたか）はデータベースにあります。これにより volta は:
   - セッションを即座に失効できる（データベース更新）
   - ユーザーごとの全アクティブセッションを追跡できる
   - 最大同時セッション数を強制できる（volta は 5 に制限）
   - 不審なセッション使用パターンを検出できる

3. **ブラウザから盗むトークンがない。** リフレッシュトークンの場合、攻撃者が XSS の脆弱性を悪用すると、localStorage からリフレッシュトークンを盗んで自分のマシンから使えます。volta の HttpOnly セッション Cookie では、XSS は Cookie を全く読めません。

4. **シンプルなメンタルモデル。** ブラウザが必要な認証情報は 1 つだけ（セッション Cookie）です。JWT は ForwardAuth フローで volta がリクエストごとに発行し、ブラウザには触れません。

フロー:

```
  ブラウザ         volta            下流アプリ
  |               |                |
  | ページリクエスト |                |
  | Cookie: __volta_session=uuid   |
  |-------------->|                |
  |               |                |
  |  セッション有効? DB を確認。      |
  |  session.expiresAt > 現在? はい。|
  |  セッション延長（touch）。       |
  |  新しい JWT を発行。             |
  |               |                |
  |               | X-Volta-JWT: eyJ... (5分 TTL)
  |               |--------------->|
  |               |                |
  |               | レスポンス       |
  |<------------------------------|
  |               |                |
  | (5分後、新しいリクエスト)         |
  | Cookie: __volta_session=uuid   |
  |-------------->|                |
  |               | 新しい JWT 発行  |
  |               |--------------->|
```

**セッションの有効期間。** volta のセッションはデフォルトで 8 時間（`SESSION_TTL_SECONDS=28800`）です。各リクエストがセッションの有効期限を延長します（「スライディング有効期限」）。アクティブなユーザーはログイン状態が維持されます。

**セッションの失効。** ユーザーがログアウトすると、volta は `store.revokeSession()` を呼び出し、データベースでセッションを無効化とマークします。ブラウザに Cookie がまだ存在していても、次のリクエストでセッションは拒否されます。

## よくある間違い

**1. リフレッシュトークンを localStorage に保存する。**
従来のリフレッシュトークンアーキテクチャで最もよくある間違いです。localStorage は XSS に対する保護がゼロです。volta は HttpOnly Cookie を使うことで、この問題を完全に回避しています。

**2. リフレッシュトークンをローテーションしない。**
リフレッシュトークンが永遠に（または手動で失効するまで）使えると、盗まれたトークンで攻撃者が永続的なアクセスを得ます。使用するたびに必ずローテーションしましょう。

**3. リフレッシュトークンの有効期間が長すぎる。**
30 日のリフレッシュトークンは、盗まれた場合に 30 日分のアクセスを意味します。利便性とリスクのバランスを取りましょう。volta の 8 時間セッション + スライディング有効期限は合理的な中間点です。

**4. 同時セッション数を制限しない。**
制限がないと、リフレッシュトークン（またはセッション）を盗んだ攻撃者が、正規のユーザーと並行して無期限に使えます。volta はユーザーあたりの同時セッションを 5 に制限し、上限を超えると最も古いものを失効させます。

**5. リフレッシュトークンが常に必要だと思い込む。**
リフレッシュトークンは「ユーザーのログイン状態を維持する」問題の解決策のひとつです。HttpOnly Cookie を使ったサーバーサイドセッションも別の解決策です。リバースプロキシの背後にある Web アプリケーション（volta のアーキテクチャのような）では、セッション Cookie のアプローチの方がシンプルで安全なことが多いです。

**6. 即時失効を提供しない。**
純粋な JWT ベースのリフレッシュトークンでサーバーサイドの状態がないと、失効にはトークンの期限切れを待つ必要があります。volta のサーバーサイドセッションは、データベースを更新することで即時失効が可能です。
