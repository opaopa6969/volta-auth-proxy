# 再生成（Regenerate）

[English version](regenerate.md)

---

## これは何？

再生成（Regeneration）とは、古いバージョンの再利用を防ぐために新しいバージョンを作成して置き換えることです。認証で最も重要な例はセッションIDの再生成です -- ログイン後に新しいセッションIDを生成して[セッション固定化](session-fixation.md)攻撃を防ぎます。

ルームメイトが引っ越した後に新しいアパートの鍵をもらうようなものです。アパートは同じ、荷物もそのまま、でも鍵が交換されます。古い鍵はもう使えません。古い鍵をコピーした人は締め出されます。鍵の交換が再生成です -- アパート（セッションデータ）はそのまま、でも識別子（鍵）は完全に新しくなります。

再生成はローテーション（システム全体の暗号鍵を交換する）や作成（新しいものを始める）とは異なります。再生成は関連するデータと状態を保持しながら識別子を交換します。

---

## なぜ重要なのか？

セッションIDの再生成がなければ、攻撃者は[セッション固定化](session-fixation.md)という有名な攻撃を悪用できます：

```
  セッション固定化攻撃（再生成なし）:
  ┌──────────────────────────────────────────────────┐
  │  1. 攻撃者がサイトにアクセス → セッションID "abc" │
  │  2. 攻撃者が被害者にsid=abc付きリンクを送信      │
  │  3. 被害者がリンクをクリック → ブラウザがsid=abc  │
  │  4. 被害者がログイン → セッションabcが被害者として │
  │     認証済みになる                                │
  │  5. 攻撃者がセッションabcを使用 → 被害者になる    │
  └──────────────────────────────────────────────────┘

  再生成あり:
  ┌──────────────────────────────────────────────────┐
  │  1-3. 上と同じ                                   │
  │  4. 被害者がログイン → サーバーがセッションIDを    │
  │     "abc"から"xyz"に再生成                        │
  │     → 被害者のブラウザが新Cookie "xyz"を取得      │
  │  5. 攻撃者がセッション"abc"を使用 → 無効          │
  │     → 攻撃阻止！                                 │
  └──────────────────────────────────────────────────┘
```

再生成は実装コストがほぼゼロでありながら、攻撃の一分類全体を防ぐ重要な防御策です。

## どう動くのか？

### セッションIDの再生成

```
  ログイン前:
  ┌──────────────┐     ┌───────────────────────┐
  │  ブラウザ    │     │  サーバー              │
  │  Cookie: abc │ ──► │  session abc:          │
  │              │     │  authenticated: false   │
  │              │     │  user: null             │
  └──────────────┘     └───────────────────────┘

  ユーザーが認証情報を送信（メール + パスワード）...
  サーバーが認証情報を検証 → 成功

  セッションIDを再生成:
  ┌──────────────┐     ┌───────────────────────┐
  │  ブラウザ    │     │  サーバー              │
  │  Cookie: xyz │ ──► │  session xyz:          │  ← 新ID
  │  (新しい!)   │     │  authenticated: true   │
  │              │     │  user: alice            │
  └──────────────┘     └───────────────────────┘
                       │  session abc: 削除済み  │  ← 旧IDは消滅
                       └───────────────────────┘
```

### いつ再生成すべきか

| イベント | 再生成？ | 理由 |
|---------|---------|------|
| ログイン（認証） | はい | セッション固定化を防止 |
| 権限昇格 | はい | セッション昇格攻撃を防止 |
| パスワード変更 | はい | 古いセッションが侵害されている可能性 |
| ロール変更 | 推奨 | 異なる信頼レベル = 新しいセッション |
| アイドルタイムアウト後 | はい（再認証時） | 新しい認証には新しいセッション |

### 再生成プロセス

```
  1. 新しいセッションIDを生成（暗号ランダムUUID）
  2. セッションデータを新IDにコピー
  3. 古いセッションIDをデータベースから削除
  4. 新しいセッションIDで新しいCookieを設定
  5. 古いセッションIDは無効

  ┌─────────────────────────────────────────────┐
  │  旧:  abc → { user: alice, tenant: acme }   │
  │                     │                       │
  │                     │ データコピー           │
  │                     ▼                       │
  │  新:  xyz → { user: alice, tenant: acme }   │
  │                                             │
  │  削除: abc → (消滅)                          │
  │  Cookie: __volta_session=xyz                │
  └─────────────────────────────────────────────┘
```

## volta-auth-proxy ではどう使われている？

### ログイン時のセッション再生成

ユーザーが認証に成功すると、voltaは既存のセッションIDを再利用せず、まったく新しいセッションを作成します：

```
  ログインフロー:
  1. ユーザーが認証情報を送信
  2. voltaが認証情報を検証（パスワードチェック）
  3. voltaが新しいセッションIDを生成: SecurityUtils.newUuid()
  4. voltaがsessionsテーブルに新しい行を挿入
  5. voltaが新しいIDで__volta_session Cookieを設定
  6. そのブラウザの既存セッションは再利用されない
```

### 安全なランダム生成

voltaはセッションIDに`java.util.UUID.randomUUID()`を使用します。これは内部的に`SecureRandom`を使うため、新しいセッションIDが予測不能であることが保証されます：

```java
public static UUID newUuid() {
    return UUID.randomUUID();
}
```

これにより、攻撃者は再生成後の新しいセッションIDを予測できません。

### 招待コードの再生成

voltaは招待コードにも再生成を使います。各招待は`SecurityUtils.inviteCode()`経由で[暗号ランダム](crypto-random.md)なコードを生成します：

```java
public static String inviteCode() {
    return randomUrlSafe(24);  // 24バイトのSecureRandom、base64urlエンコード
}
```

招待を再送信する必要がある場合、古いコードを再利用するのではなく新しいコードが生成されます。これにより以前に傍受されたコードの使用を防ぎます。

## よくある間違いと攻撃

### 間違い1: ログイン時に再生成しない

最も一般的な間違いです。セッションIDがログインを生き延びると、セッション固定化が可能になります。認証成功後は常に新しいセッションIDを生成しましょう。

### 間違い2: IDを再生成するが古いものを削除しない

古いセッションIDがデータベースにまだ存在すると、攻撃者はまだ使えます。新しいものが作成されたとき、古いレコードは削除（または無効化）されなければなりません。

### 間違い3: 予測可能な新ID

再生成されたセッションIDが予測可能（連番カウンタ、タイムスタンプのみ、弱い乱数）なら、攻撃者は新しいIDを推測できます。常に[暗号ランダム](crypto-random.md)生成を使いましょう。

### 間違い4: 権限変更時に再生成しない

memberからadminに昇格したユーザーは新しいセッションIDを得るべきです。そうしなければ、ユーザーの権限が低かったときに取得されたセッションIDが、信頼度の低い第三者に観察されている可能性があります。

### 攻撃: 再生成中の競合状態

攻撃者が「古いIDが無効化」されてから「新しいIDが設定」されるまでの短い間に高速リクエストを送ります。サーバーが注意していないと、両方のIDが短時間同時に動作する可能性があります。再生成にはアトミックな操作（データベーストランザクション）を使いましょう。
