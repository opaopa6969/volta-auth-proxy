# リトライ（再試行）

[English version](retry.md)

---

## これは何？

リトライは、失敗した操作を通常は短い遅延の後にもう一度試みることです。分散システムでは一時的な障害は避けられません。サーバーが一時的に過負荷になったり、ネットワーク接続が切れたり、データベースが一時的に利用不可になったりします。即座に諦める代わりに、少し待ってからもう一度試みます。

電話に出ない友人に電話するようなものです。1回かけて出なかったからといって友情が終わったと宣言しません。1分待ってもう一度かけ、5分待ってもう一度試します。何回かの試行後もつながらなければ、留守番電話にメッセージを残します（デッドレター）。重要なのは、試行の間隔を空けて、最終的には諦めることです。1,000回連続で電話しません。

リトライはサービス間の信頼性ある通信の基本であり、特に[Webhook](webhook.md)配信、[M2M](m2m.md)トークンリフレッシュ、[Outbox](outbox-pattern.md)イベント処理で重要です。

---

## なぜ重要なのか？

分散システムでは、障害は例外ではなく常態です：

```
  リトライなし：
  ┌──────────┐     POST /webhook     ┌──────────┐
  │  volta   │ ─────────────────────►│ あなたの  │
  │          │                       │ アプリ    │
  │          │     接続拒否           │ （ダウン!）│
  │          │ ◄─────────────────────│          │
  │          │                       │          │
  │ 「仕方ない│                      │          │
  │  イベント │                      │          │
  │  消失」   │                      │          │
  └──────────┘                       └──────────┘

  リトライあり：
  ┌──────────┐     POST /webhook     ┌──────────┐
  │  volta   │ ─────────────────────►│ あなたの  │
  │          │     接続拒否           │ アプリ    │
  │          │ ◄─────────────────────│ （ダウン!）│
  │          │                       │          │
  │ (10秒待機)│                      │ (再起動) │
  │          │     POST /webhook     │          │
  │          │ ─────────────────────►│          │
  │          │     200 OK            │ （復帰!）│
  │          │ ◄─────────────────────│          │
  │          │                       │          │
  │ 「2回目で │                      │「受け取った│
  │  配信成功」│                      │!」       │
  └──────────┘                       └──────────┘
```

リトライがないと、一瞬のネットワーク障害やサービス再起動で永久的なデータ損失が発生します。リトライがあれば、一時的な障害に対してシステムが回復力を持ちます。

## どう動くのか？

### リトライ戦略

#### 1. 固定間隔

```
  試行1 → 失敗 → 10秒待機
  試行2 → 失敗 → 10秒待機
  試行3 → 失敗 → 10秒待機
  試行4 → 成功！

  問題：サーバーが過負荷の場合、全クライアントが
  同じ間隔でリトライ → サンダリングハード
```

#### 2. 指数バックオフ

```
  試行1 → 失敗 → 10秒待機
  試行2 → 失敗 → 30秒待機   (10 * 3)
  試行3 → 失敗 → 90秒待機   (30 * 3)
  試行4 → 失敗 → 270秒待機  (90 * 3)
  試行5 → 成功！

  より良い：サーバーに段階的により多くの
  回復時間を与える
```

#### 3. ジッター付き指数バックオフ

```
  試行1 → 失敗 → 10秒 + ランダム(0-5秒)待機
  試行2 → 失敗 → 30秒 + ランダム(0-15秒)待機
  試行3 → 失敗 → 90秒 + ランダム(0-45秒)待機

  最良：複数クライアントがまったく同じタイミングで
  リトライすることを防止（サンダリングハード対策）
```

### リトライすべき場合とすべきでない場合

| HTTPステータス | リトライ？ | 理由 |
|-------------|---------|------|
| 200-299 | いいえ | 成功！ |
| 400 Bad Request | **いいえ** | リクエストが間違っている。リトライしても直らない。 |
| 401 Unauthorized | **1回** | トークンが期限切れかも。リフレッシュして1回リトライ。 |
| 403 Forbidden | **いいえ** | 権限がない。リトライしても無駄。 |
| 404 Not Found | **いいえ** | リソースが存在しない。 |
| 408 Request Timeout | **はい** | サーバーがビジー。再試行。 |
| 429 Too Many Requests | **はい** | レート制限。`Retry-After`ヘッダーを尊重。 |
| 500 Internal Server Error | **はい** | サーバーバグ、一時的かも。 |
| 502 Bad Gateway | **はい** | アップストリーム問題、一時的な可能性。 |
| 503 Service Unavailable | **はい** | サーバー過負荷、後で再試行。 |
| 接続拒否 | **はい** | サーバー再起動中かも。 |
| タイムアウト | **はい** | ネットワーク問題、一時的かも。 |

### リトライバジェット

リトライバジェットはリトライの総数を制限してカスケード障害を防ぎます：

```
  ┌─────────────────────────────────────────────┐
  │  リトライバジェット：総リクエストの最大10%     │
  │                                              │
  │  voltaが1000 webhook/分を送信する場合：       │
  │    最大リトライ許容：100/分                    │
  │                                              │
  │  500 webhookが失敗（サーバーダウン）の場合：   │
  │    即座にリトライされるのは100のみ             │
  │    残り400は次のサイクルにキューイング          │
  │                                              │
  │  これによりvoltaが既に苦しんでいる             │
  │  サーバーを圧倒することを防ぐ                  │
  └─────────────────────────────────────────────┘
```

## volta-auth-proxyではどう使われている？

### Webhook配信リトライ（Outboxワーカー）

[Outbox](outbox-pattern.md)ワーカーは指数バックオフで失敗したWebhook配信をリトライします：

```
  ┌────────────────────────────────────────────┐
  │  volta Webhookリトライスケジュール            │
  │                                            │
  │  試行1：即座                                │
  │  試行2：+10秒                               │
  │  試行3：+30秒                               │
  │  試行4：+90秒                               │
  │  試行5：+270秒（約4.5分）                    │
  │  試行6：+810秒（約13.5分）                   │
  │  試行7：+2430秒（約40分）                    │
  │  試行8：+3600秒（1時間、最大）               │
  │                                            │
  │  最大試行回数：10                            │
  │  最大超過後：status = FAILED、運用にアラート   │
  └────────────────────────────────────────────┘
```

```sql
-- Outboxワーカーのリトライロジック
UPDATE webhook_outbox
SET attempts = attempts + 1,
    next_retry = NOW() + INTERVAL '1 second' *
      LEAST(POWER(3, attempts) * 10, 3600)
WHERE id = :event_id;
```

### volta-sdk-jsの401リトライ

volta JavaScript SDKは401 Unauthorizedに対して[セッション](session.md)をリフレッシュして自動リトライします：

```
  volta-sdk-js                    volta-auth-proxy
  =============                   ================

  1. GET /api/data
     Cookie: volta_session=abc123
  ──────────────────────────────────────────►

                                  セッション期限切れ
  ◄──────────────────────────────────────────
     401 Unauthorized

  2. (自動) POST /auth/refresh
     Cookie: volta_refresh=xyz789
  ──────────────────────────────────────────►

                                  新しいセッション発行
  ◄──────────────────────────────────────────
     Set-Cookie: volta_session=new456

  3. (自動リトライ) GET /api/data
     Cookie: volta_session=new456
  ──────────────────────────────────────────►

                                  成功
  ◄──────────────────────────────────────────
     200 OK + data
```

これは透過的に行われます。アプリケーションコードはトークンリフレッシュを処理する必要がありません。

### Stripe Webhookインジェスションのリトライ

voltaの[インジェスション](ingestion.md)エンドポイントが一時的にダウンしている場合、Stripeが独自のスケジュールで配信をリトライします。voltaはこれらのリトライを正しく処理するために冪等でなければなりません。

### M2Mトークンリトライ

[M2M](m2m.md)サービスが401を受け取った場合、[Client Credentials](client-credentials.md)で再認証し、元のリクエストをリトライすべきです。SDK の動作と同様です。

## よくある間違いと攻撃

### 間違い1：リトライすべきでないエラーをリトライする

400 Bad Requestや403 Forbiddenをリトライしても成功しません。リソースを浪費し、レート制限を引き起こす可能性があります。一時的な障害（5xx、タイムアウト、接続エラー）のみリトライしてください。

### 間違い2：最大リトライ制限がない

永久にリトライするとキューが溢れ、接続が枯渇し、他のシステムにカスケードする可能性があります。常に最大リトライ回数を設定し、失敗したアイテムをデッドレターキューに移動してください。

### 間違い3：バックオフがない

過負荷のサーバーに対して即座にフル速度でリトライすると、過負荷が悪化します。常にジッター付き指数バックオフを使ってください。

### 間違い4：冪等でない

Webhookハンドラが冪等でない場合、リトライが重複した副作用を引き起こします（例：同じメールを2回送信、二重課金）。同じイベントを複数回受信しても耐えられるようにハンドラを設計してください。

### 攻撃：リトライ増幅（DDoS）

攻撃者が常に失敗するリクエストを送り、リトライが負荷を増幅する。防御：リトライバジェット、サーキットブレーカー、リトライレートの監視。

### 攻撃：リトライ遅延の悪用

攻撃者がリトライスケジュールを知り、リトライストームに合わせて攻撃のタイミングを計る。防御：リトライ間隔にジッターを追加してリトライを予測不可能にする。
