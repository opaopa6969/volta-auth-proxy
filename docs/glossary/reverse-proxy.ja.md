# リバースプロキシ（Reverse Proxy）

[English version](reverse-proxy.md)

---

## 一言で言うと？

リバースプロキシとは、アプリケーションの前に立って、インターネットからのすべてのリクエストを受け取り、裏側で適切なアプリケーションに転送するサーバーのことです。訪問者を正しいオフィスに案内する受付のようなものです。

---

## 受付のたとえ

多くの部門が入っている大きな会社のビルを想像してください。ただし受付は1つだけです：

```
  訪問者が到着：「エンジニアリングチームに会いたいです」

  受付（リバースプロキシ）：
    1. 「いらっしゃいませ！エンジニアリングが何階か確認します」
    2. 「エンジニアリングは3階です。ご案内します」
    3. 訪問者は3階に行ってエンジニアリングチームと話す

  訪問者は階数を知る必要がなかった。
  受付がルーティングを処理した。
```

別の訪問者を想像してください：「CEOに会いたいです」

```
  受付：
    1. 「予約はありますか？」（認可を確認）
    2. 「ない？申し訳ございませんが、予約なしではお通しできません」
    3. 訪問者はCEOのフロアに到達する前に断られる
```

リバースプロキシはWebアプリケーションに対して同じことをします：
- ユーザーからのすべてのリクエストを受け取る
- 各リクエストをどのアプリケーションが処理すべきか決める
- 不正なリクエストをアプリケーションに届く前にブロックできる
- ユーザーはアプリケーションが実際にどこにあるか知る必要がない

---

## なぜ「リバース」と呼ばれるの？

プロキシには2種類あります：

**フォワードプロキシ** -- ユーザーの代わりに動作する。例：Webサイトを訪問するときに身元を隠すVPN。Webサイトはあなたが誰か分からない。

**リバースプロキシ** -- サーバーの代わりに動作する。例：アプリの前に立つTraefik。ユーザーはどのサーバーと通信しているか分からない。

```
  フォワードプロキシ（ユーザーを保護）：
  ユーザー → [フォワードプロキシ] → インターネット → サーバー
  「サーバーはユーザーが本当は誰か分からない」

  リバースプロキシ（サーバーを保護）：
  ユーザー → インターネット → [リバースプロキシ] → サーバーA、B、C
  「ユーザーはどのサーバーと通信しているか分からない」
```

フォワードプロキシと反対方向だから「リバース」と呼ばれます。

## Traefikとvoltaはどうやって連携するの？

典型的なvoltaのセットアップでは、[Traefik](https://traefik.io/)がリバースプロキシとして使われます。Traefikは人気のある軽量リバースプロキシで、特にDockerとの連携に優れています。

各パーツがどのように組み合わさるか：

```
  ユーザーのブラウザがwiki.example.comにアクセス
       │
       ▼
  Traefik（リバースプロキシ）
       │
       │  「wiki.example.com？ルーティングルールを確認...」
       │  「ルール：wiki.example.com → ポート8080のwikiアプリに転送」
       │
       │  でもまず！「このルートにForwardAuthが有効」
       │  「まずvolta-auth-proxyに確認しよう...」
       │
       ├──→ volta-auth-proxy（ForwardAuth確認）
       │    volta：「このユーザーはログインしている？テナントは？ロールは？」
       │    volta：「はい、太郎、ACMEのMEMBER。ヘッダーはこちら」
       │    voltaが返す：200 OK + 本人情報ヘッダー
       │
       │  Traefik：「voltaが承認！リクエストに本人情報ヘッダーを追加」
       │
       └──→ wikiアプリ（ポート8080）
            X-Volta-User-Id、X-Volta-Tenant-Idなどが付いたリクエストを受信
```

Traefikがネットワーク（どのリクエストがどこへ行くか）を処理し、voltaが本人確認（誰がリクエストしていて、許可されているか）を処理します。

## なぜアプリはプロキシの後ろにいるの？

アプリケーションを直接インターネットに公開しない理由はいくつかあります：

1. **セキュリティ** -- プロキシが悪意のあるリクエストをアプリに届く前にブロックできる。アプリはスクリーニング済みのトラフィックだけを受け取る。

2. **認証** -- voltaのForwardAuthにより、プロキシがリクエストを転送する前にユーザーがログイン済みか確認できる。未認証ユーザーはアプリに到達しない。

3. **ルーティング** -- 1つのドメインで多くのアプリをホストできる。`wiki.example.com`はwikiへ、`admin.example.com`は管理パネルへ。プロキシがURLに基づいてトラフィックをルーティング。

4. **SSL/TLS** -- プロキシがHTTPS暗号化を処理する。各アプリが個別に証明書を管理する必要がない。

5. **ロードバランシング** -- アプリの複数コピーが動いている場合、プロキシがリクエストを均等に分配できる。

## 簡単な例

wikiと管理パネルの2つのアプリがあるとします。リバースプロキシなしでは、各アプリに独自の公開アドレスが必要です：

```
  リバースプロキシなし：
  wiki.example.com:8080   → Wikiアプリ
  admin.example.com:9090  → 管理アプリ
  （ユーザーがポート番号を知る必要がある。見た目が悪く、安全でない）
```

リバースプロキシ（Traefik）ありの場合：

```
  リバースプロキシあり：
  wiki.example.com    → Traefik → Wikiアプリ（ポート8080、非公開）
  admin.example.com   → Traefik → 管理アプリ（ポート9090、非公開）
  （ユーザーはきれいなURLにアクセスするだけ。Traefikが残りを処理）
```

さらにvoltaのForwardAuthを追加：

```
  リバースプロキシ + volta：
  wiki.example.com    → Traefik → volta確認 → Wikiアプリ
  admin.example.com   → Traefik → volta確認 → 管理アプリ
  （すべてのリクエストがアプリに届く前に認証される）
```
