# 無効化（Revoke）

[English version](revoke.md)

---

## これは何？

失効（Revocation）とは、鍵、トークン、セッションを自然な有効期限が来る前にキャンセルする行為です。何かが自然に期限切れになるのを待つ代わりに、「これはもう有効ではない、即座に」と能動的に宣言します。

クレジットカードの解約を思い浮かべてください。カードには有効期限（2028/12）が印刷されていますが、盗まれたら銀行に電話して今すぐ解約します。カードは物理的にまだ存在し、有効期限は変わっていませんが、銀行のシステムが次のスワイプで拒否します。その電話が失効です。

失効は認証システムの緊急ブレーキです。通常の運用は有効期限に依存しますが（トークンが期限切れ、セッションがタイムアウト）、何かが間違ったとき -- 鍵が漏洩、アカウントが侵害、従業員が退職 -- 即座にアクセスを遮断する能力が必要です。

---

## なぜ重要なのか？

失効がなければ、時間に身を委ねるしかありません。JWTの有効期限が24時間で署名鍵が漏洩したら、攻撃者は24時間の無制限アクセスを得ます。セッションに失効メカニズムがなければ、解雇された従業員はセッションが自然に期限切れになるまでログインし続けます。

失効が提供するもの：

- **即座の対応**: 侵害が検出された瞬間にアクセスを遮断
- **被害の限定**: 悪用の時間枠を数時間/数日から数秒に短縮
- **コンプライアンス**: 多くの規制が即座のアクセス終了能力を要求（GDPR削除権、SOC 2アクセス制御）

難しいのは、すべてが同じように失効可能ではないことです。不透明トークン（セッションID）はレコードを削除するだけで簡単に失効できます。自己完結型トークン（JWT）は自身で有効性を持ち運ぶため失効が困難です。

## どう動くのか？

### 失効可能性のスペクトラム

```
  失効しやすい                          失効しにくい
  ◄─────────────────────────────────────────────────────►
  │                      │                              │
  セッションID           リフレッシュトークン             JWT
  （DBから削除）          （DBから削除）            （自己完結型、
                                                  削除するDBがない
                                                  -- 期限切れを
                                                  待つしかない）
```

### セッションの失効（サーバー側の状態）

```
  失効前:
  ┌─────────────────────────────────────────┐
  │  sessions テーブル                       │
  │  ┌───────────────────────────────────┐  │
  │  │ id: abc-123      status: ACTIVE  │  │
  │  │ user_id: alice   expires: 17:00  │  │
  │  └───────────────────────────────────┘  │
  └─────────────────────────────────────────┘
       ↑ Cookie "abc-123" → 見つかった → アクセス許可

  失効後（DELETE FROM sessions）:
  ┌─────────────────────────────────────────┐
  │  sessions テーブル                       │
  │  （空 -- 行が削除された）                │
  └─────────────────────────────────────────┘
       ↑ Cookie "abc-123" → 見つからない → 401
```

### JWTの失効（難しい問題）

JWTは自己完結型です。削除するデータベース行がありません。選択肢：

```
  選択肢1: 短い有効期限（voltaのアプローチ）
  ┌──────────────────────────────────────────┐
  │  JWTは5分で期限切れ。                     │
  │  セッションを失効 → 新しいJWTは発行不可   │
  │  既存のJWTは最大5分で消滅。               │
  │  許容できる伝搬遅延。                     │
  └──────────────────────────────────────────┘

  選択肢2: トークンブロックリスト（voltaは不使用）
  ┌──────────────────────────────────────────┐
  │  失効したJTIのリストを管理。              │
  │  すべてのJWTをリストと照合。              │
  │  リクエストごとにレイテンシと状態が増加。  │
  │  ステートレスJWTの意味がなくなる。         │
  └──────────────────────────────────────────┘

  選択肢3: 鍵ローテーション
  ┌──────────────────────────────────────────┐
  │  署名鍵をローテーション。                 │
  │  古い鍵で署名されたすべてのトークンが      │
  │  古い鍵が削除されると検証不能に。          │
  │  核オプション -- すべてのユーザーに影響。   │
  └──────────────────────────────────────────┘
```

### 署名鍵の失効

```
  POST /api/v1/admin/keys/rotate

  前:  鍵A（アクティブ） → JWTに署名
  後:  鍵A（引退） → 鍵B（アクティブ） → JWTに署名
       鍵Aのトークンは5分の期限切れまで有効
       5分後、鍵Aを完全に削除可能
```

## volta-auth-proxy ではどう使われている？

voltaは複数のレベルで失効を実装しています：

### 1. セッション失効（ログアウト）

`sessions`テーブルからセッション行を削除すると、即座にセッションが無効化されます。ユーザーの`__volta_session` Cookieは何も指さないダングリングポインタになります：

```
  ユーザーが「ログアウト」をクリック
       │
       ▼
  DELETE FROM sessions WHERE id = ?
       │
       ▼
  Cookieはブラウザに残るが、セッションを指していない
       │
       ▼
  次のリクエスト: セッション参照失敗 → 401 → ログインにリダイレクト
```

### 2. JWT伝搬遅延

JWTは直接失効できませんが、voltaの5分の有効期限により失効が素早く[伝搬](propagation.md)します：

```
  時刻 0:00  管理者がユーザーセッションを失効
  時刻 0:00  新しいJWTは発行不可（セッションがない）
  時刻 0:01  既存のJWTはまだ有効（残り4分）
  時刻 5:00  JWT期限切れ。ユーザー完全にロックアウト。

  最大伝搬遅延: 5分
```

### 3. ローテーションによる鍵の失効

[署名鍵](signing-key.md)の侵害が疑われる場合：

```
  POST /api/v1/admin/keys/rotate
       │
       ▼
  新しい鍵を生成、古い鍵を引退
       │
       ▼
  古い鍵で署名されたJWTは最大5分で期限切れ
       │
       ▼
  新しい鍵でシステム完全復旧
```

### 4. テナントレベルの失効

テナントの全セッションを失効（例：テナント侵害時）：

```
  DELETE FROM sessions WHERE tenant_id = ?
       │
       ▼
  テナント内の全ユーザーが即座にログアウト
  既存のJWTはすべて5分以内に期限切れ
```

## よくある間違いと攻撃

### 間違い1: 失効メカニズムがまったくない

トークンの有効期限だけに頼るシステムは緊急事態に対応できません。JWTが24時間有効で鍵が漏洩したら、24時間の脆弱性ウィンドウになります。voltaはJWTを5分に保ってこれを最小化しています。

### 間違い2: Cookieは失効するがセッションはしない

ブラウザのCookieをクリアしてもサーバー側のセッションは無効化されません。セッションIDをコピーした攻撃者はまだ使えます。常にデータベースからセッションを削除しましょう。

### 間違い3: 遅い失効伝搬

下流のサービスがJWTやJWKSを積極的にキャッシュすると、失効が効果を発揮するまで数時間かかる場合があります。キャッシュTTLを短く保ちましょう。voltaの5分のJWT有効期限が最大伝搬遅延です。

### 攻撃: 伝搬前の失効済みトークンの使用

盗まれたJWTを持つ攻撃者は、セッションが失効した後でも最大5分（voltaのJWT TTL）使えます。そのためvoltaはセッション失効（サーバー側チェックでは即座）と短いJWT有効期限（ステートレスチェックでの被害を制限）を組み合わせています。
