# ロール（Role）

[English version](role.md)

---

## 一言で言うと？

ロールとは、ワークスペース内でユーザーが何をしてよいかを決めるために割り当てられるラベルのことです。特定の権限セットが付いてくる肩書きのようなものです。

---

## オフィスのたとえ

典型的なオフィスを考えてみてください。人々にはそれぞれ異なる肩書きがあり、各肩書きには異なるアクセスレベルが付いています：

- **CEO（Owner）**は何でもできる：雇用と解雇、すべての部屋へのアクセス、契約の締結、会社の閉鎖さえもできる。
- **オフィスマネージャー（Admin）**は新しい従業員を招待したり、デスクを割り当てたり、オフィスを管理できる。でも会社を閉鎖することはできない。
- **一般社員（Member）**は自分の仕事をし、会議室を使い、共有ファイルにアクセスできる。でも人を雇ったり、オフィスのポリシーを変えたりはできない。
- **来客（Viewer）**はロビーに座って公開資料を見ることができる。でも何かに触れたり、プライベートなエリアに入ったりはできない。

各人に50個の個別権限のリストを渡すのではなく、肩書き（ロール）を与えると、それに付随する権限が付いてきます。「一般社員」ができることを変える必要があれば、ロール定義を一度変更するだけで、そのロールを持つ全員に適用されます。

---

## voltaの4つのロール

voltaはシンプルな4段階のロール階層を使います。「階層」とは、上位のロールが下位のロールのすべての権限を自動的に含むということです：

```
  OWNER   （最上位 -- すべてができる）
    │
  ADMIN   （人と設定を管理できる）
    │
  MEMBER  （通常利用）
    │
  VIEWER  （読み取り専用、最下位）
```

各ロールができること：

### OWNER -- 「自分の会社です」
- ADMINができることすべて、さらに：
- テナント（ワークスペース）全体の削除
- オーナーシップの他の人への移譲
- テナントには常に少なくとも1人のOWNERが必要（オーナー不在のワークスペースは作れない）

### ADMIN -- 「チームを管理します」
- MEMBERができることすべて、さらに：
- ワークスペースに新しい人を招待
- ワークスペースから人を削除
- 人のロールを変更（ADMINレベルまで、OWNERにはできない）
- ワークスペースの設定変更

### MEMBER -- 「ここで働いています」
- MEMBERロールを許可するすべてのアプリを使用
- コンテンツの作成と編集
- 他のユーザーの管理や設定変更はできない

### VIEWER -- 「見ているだけです」
- VIEWERロールを許可するアプリでコンテンツを閲覧
- 何も作成、編集、削除できない
- 「読み取り専用」アクセスと考えてください

## なぜロールが重要なのか

ロールがなければ、すべてのユーザーの権限を個別に管理する必要があります。200人のワークスペースを想像してみてください：

**ロールなし（悪夢）：**
```
  太郎：can_view, can_edit, can_invite, can_remove_users, can_change_settings
  花子：can_view, can_edit
  次郎：can_view
  ... あと197人分を繰り返す ...

  新方針：「編集者はPDFエクスポートができるように」
  → 150人を個別に更新。3人を見落とす。バグ報告が来る。
```

**ロールあり（シンプル）：**
```
  太郎：ADMIN
  花子：MEMBER
  次郎：VIEWER
  ... あと197人分を繰り返すが、各自1ワードだけ ...

  新方針：「MEMBERはPDFエクスポートができるように」
  → MEMBERのロール定義を一度更新。完了。
```

## voltaでのロールの使い方

### volta-config.yamlにて

volta背後の各アプリが、許可するロールを指定します：

```yaml
apps:
  - id: app-wiki
    url: https://wiki.example.com
    allowed_roles: [MEMBER, ADMIN, OWNER]   # Viewerはアクセス不可

  - id: app-admin
    url: https://admin.example.com
    allowed_roles: [ADMIN, OWNER]           # 管理者とオーナーのみ
```

太郎（MEMBER）がwikiにアクセスしようとすると、voltaが確認します：「MEMBERは許可リストにある？はい。」太郎はアクセスできます。太郎が管理パネルにアクセスしようとすると、voltaが確認します：「MEMBERは許可リストにある？いいえ。」太郎は403 Forbiddenエラーを受け取ります。

### JWTトークンにて

voltaがユーザーを認証すると、ユーザーのロールを含むトークン（JWT）を作成します：

```json
{
  "sub": "taro-uuid",
  "volta_tid": "acme-uuid",
  "volta_roles": ["MEMBER"],
  "volta_tname": "ACME Corp"
}
```

アプリはこのトークンを読んで、独自の権限判断ができます。例えば、wikiアプリはMEMBERにページの編集は許可するが、削除は許可しないかもしれません。

## 簡単な例

```
  ACME Corpワークスペースには3人のメンバーがいます：

  ┌──────────────────────────────────────────────┐
  │ ゆき   - OWNER  - ワークスペースを作った人    │
  │ けんじ - ADMIN  - チームを管理する人          │
  │ みか   - MEMBER - 一般ユーザー               │
  │ 取引先 - VIEWER - 外部コンサルタント          │
  └──────────────────────────────────────────────┘

  場面：取引先がwikiページを編集しようとする
  → volta：「取引先はVIEWER。Viewerは読み取り専用アクセス。」
  → 結果：編集がブロックされる。

  場面：みかがけんじをワークスペースから削除しようとする
  → volta：「みかはMEMBER。ADMINまたはOWNERのみがメンバーを削除できる。」
  → 結果：操作がブロックされる。

  場面：けんじが自分をOWNERにしようとする
  → volta：「けんじはADMIN。OWNERのみがOWNERに昇格できる。」
  → 結果：操作がブロックされる。
```
