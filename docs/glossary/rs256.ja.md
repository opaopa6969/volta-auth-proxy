# RS256 (RSA Signature with SHA-256)

[English version](rs256.md)

---

## これは何？

RS256は、データ（JWTなど）をデジタル署名するための方法で、**秘密鍵**（署名者だけが持つ）と**公開鍵**（誰でも持てる）の鍵ペアを使います。秘密鍵が署名を作り、公開鍵がそれを検証します。RS256はRSA暗号アルゴリズムとSHA-256ハッシュを組み合わせたものです。

手紙の蝋印のようなものです。王様だけが印章指輪（秘密鍵）を持ち、蝋印を作れます。でも王国の誰もが蝋印を見て本物だと認識できます（公開鍵）。指輪なしに蝋印を偽造することはできず、蝋印の検証に指輪は必要ありません。

---

## なぜ重要なのか？

複数のサービスがJWTを検証する必要があるシステムでは、署名アルゴリズムの選択が極めて重要です。RS256（非対称）とHS256（対称）が最も一般的な2つの選択肢で、間違った選択は深刻なセキュリティ脆弱性を生みます。

RS256が重要な理由：

1. **秘密を必要とするシステムは1つだけ。** volta-auth-proxyが秘密鍵を保持します。他のすべてのサービスは公開鍵だけで十分です。
2. **公開鍵は自由に共有できる。** `/.well-known/jwks.json`で公開されています。秘密の配布問題がありません。
3. **侵害が封じ込められる。** アプリのサーバーがハッキングされても、攻撃者が得るのは公開鍵（既に公開されているもの）です。JWTを偽造できません。

代替手段（HS256）がなぜ危険かについては[hs256.md](hs256.md)を参照。

## どう動くのか？

### 鍵ペア

```
  鍵の生成（volta-auth-proxy起動時に1回だけ実行）：

  ┌──────────────────────────────────────────────────────┐
  │  RSA鍵ペアジェネレーター（2048ビット）                  │
  │                                                       │
  │  数学的にリンクした2つの鍵を生成：                       │
  │                                                       │
  │  秘密鍵                          公開鍵               │
  │  ┌────────────────────┐         ┌────────────────────┐│
  │  │ 非常に長い数値      │         │ 非常に長い数値      ││
  │  │（秘密にする）       │────────►│（自由に共有）       ││
  │  │                    │  導出   │                    ││
  │  │ データに署名できる   │         │ 署名を検証できる    ││
  │  │ データを復号できる   │         │ データを暗号化できる ││
  │  └────────────────────┘         └────────────────────┘│
  │                                                       │
  │  公開鍵から秘密鍵を導出することはできない                │
  └──────────────────────────────────────────────────────┘
```

### RS256でJWTに署名する

```
  ステップ1：JWTの内容を作成

    ヘッダー:  {"alg":"RS256","typ":"JWT","kid":"key-2026-03-31T09-00"}
    ペイロード: {"sub":"user-uuid","volta_tid":"tenant-uuid",...}

  ステップ2：両パートをエンコード

    encoded = BASE64URL(ヘッダー) + "." + BASE64URL(ペイロード)

  ステップ3：秘密鍵で署名

    signature = RSA_SIGN(SHA256(encoded), 秘密鍵)

  ステップ4：JWTを組み立てる

    jwt = encoded + "." + BASE64URL(signature)

    結果: eyJhbGci...ヘッダー...eyJzdWIi...ペイロード...SflKxwRJ...署名
```

### RS256でJWTを検証する

```
  ステップ1：JWTを3パート（ヘッダー、ペイロード、署名）に分解

  ステップ2：アルゴリズムをチェック
    - header.algはRS256でなければならない
    - HS256、none、その他すべてを拒否

  ステップ3：公開鍵を取得
    - ヘッダーから"kid"を読む
    - JWKSエンドポイントから公開鍵を取得（またはキャッシュ）

  ステップ4：署名を検証
    - 計算: RSA_VERIFY(SHA256(ヘッダー + "." + ペイロード), 署名, 公開鍵)
    - trueの場合: JWTは秘密鍵の持ち主が署名した
    - falseの場合: JWTは改変または偽造された

  ステップ5：クレームをチェック（exp, iss, audなど）
```

### なぜRS256が分散システムで安全か

```
  ┌──────────────────────────────┐
  │     volta-auth-proxy         │
  │                              │
  │  持っている: 秘密鍵（秘密）   │
  │  すること: JWTに署名          │
  │                              │
  │  公開もする:                  │
  │  /.well-known/jwks.json      │
  │  (公開鍵、秘密ではない)       │
  └──────────────┬───────────────┘
                 │
       JWKSで公開鍵を共有
                 │
    ┌────────────┼────────────┐
    │            │            │
    ▼            ▼            ▼
  ┌──────┐   ┌──────┐   ┌──────┐
  │App A │   │App B │   │App C │
  │      │   │      │   │      │
  │持って │   │持って │   │持って │
  │いる:  │   │いる:  │   │いる:  │
  │公開鍵 │   │公開鍵 │   │公開鍵 │
  │のみ   │   │のみ   │   │のみ   │
  └──────┘   └──────┘   └──────┘

  App Bが侵害された場合：
  - 攻撃者が得るもの: 公開鍵（既に公開 -- 価値なし）
  - 攻撃者ができないこと: JWTの偽造（秘密鍵が必要）
  - 攻撃者ができないこと: 他のユーザーのなりすまし
  - 被害はApp B自身のデータに封じ込められる
```

## volta-auth-proxyではどう使われているか？

### 鍵の生成

voltaは初回起動時に2048ビットのRSA鍵ペアを生成します：

```java
// JwtService.java
KeyPairGenerator generator = KeyPairGenerator.getInstance("RSA");
generator.initialize(2048);
KeyPair keyPair = generator.generateKeyPair();
```

### 鍵の保存

秘密鍵はAES-256-GCMで暗号化してから`signing_keys`データベーステーブルに保存されます。`KeyCipher.java`を参照。

### 鍵のローテーション

管理者は`POST /api/v1/admin/keys/rotate`で鍵をローテーションできます。古い鍵は移行期間中、検証用（JWKSエンドポイント内）に有効なまま残りますが、新しいJWTは新しい鍵で署名されます。

### JWKSエンドポイント

公開鍵は`GET /.well-known/jwks.json`で提供されます：

```json
{
  "keys": [
    {
      "kty": "RSA",
      "kid": "key-2026-03-31T09-00",
      "alg": "RS256",
      "n": "0vx7agoebGcQ...(公開モジュラス)...",
      "e": "AQAB"
    }
  ]
}
```

アプリはこのエンドポイントを取得し、鍵をキャッシュし、JWT署名をローカルで検証するために使います。

### アルゴリズムの強制

voltaはRS256以外のJWTを明示的に拒否します：

```java
// JwtService.java - verify()
if (!JWSAlgorithm.RS256.equals(jwt.getHeader().getAlgorithm())) {
    throw new IllegalArgumentException("Unsupported JWT algorithm");
}
```

これにより`alg:none`攻撃とRS256からHS256への混乱攻撃を防ぎます。

## よくある間違いと攻撃

### 攻撃：alg:none攻撃

詳細は[jwt.md](jwt.md)を参照。要約：サーバーがJWTの`alg`ヘッダーを信頼し`none`を許可する場合、攻撃者は署名を完全に削除できます。voltaは明示的なアルゴリズムホワイトリストでこれを防ぎます。

### 攻撃：アルゴリズム混乱（RS256 -> HS256）

完全な説明は[hs256.md](hs256.md)を参照。要約：攻撃者がアルゴリズムをRS256からHS256に変更し、公開鍵（公開されている！）をHMACの秘密として使います。一部のJWTライブラリはこれを受け入れます。voltaはRS256のみを受け入れることでこれを防ぎます。

### 間違い：短いRSA鍵の使用

2048ビット未満のRSA鍵は安全でないとされています。voltaは2048ビットの鍵を使用しており、これが現在の標準最小値です。

### 間違い：鍵をローテーションしない

秘密鍵を永久に使い続けて最終的に漏洩した場合、その鍵で発行されたすべてのJWTが偽造可能になります。voltaは鍵のローテーションをサポートし、古い鍵を無効化できます。

### 間違い：秘密鍵を配布する

秘密鍵をすべてのサービスの環境変数に入れるチームがあります。これはRS256の目的を無にします。署名サービス（volta-auth-proxy）だけが秘密鍵を持つべきです。
