# RS256 (RSA Signature with SHA-256)

[日本語版はこちら](rs256.ja.md)

---

## What is it?

RS256 is a method for digitally signing data (like a JWT) using a pair of keys: a **private key** that only the signer has, and a **public key** that anyone can have. The private key creates the signature, and the public key verifies it. RS256 specifically combines the RSA cryptographic algorithm with SHA-256 hashing.

Think of it like a wax seal on a letter. Only the king has the signet ring (private key) to create the seal. But anyone in the kingdom can look at the seal and recognize it is authentic (public key). Nobody can forge the seal without the ring, and nobody needs the ring to verify the seal.

---

## Why does it matter?

In systems where multiple services need to verify JWTs, the choice of signing algorithm is critical. RS256 (asymmetric) and HS256 (symmetric) are the two most common options, and choosing wrong can create serious security vulnerabilities.

RS256 matters because:

1. **Only one system needs the secret.** volta-auth-proxy keeps the private key. Every other service only needs the public key.
2. **Public keys can be shared freely.** They are published at `/.well-known/jwks.json`. No secret distribution problem.
3. **Compromise is contained.** If an app's server is hacked, the attacker gets a public key -- which is already public. They cannot forge JWTs.

See [hs256.md](hs256.md) for why the alternative (HS256) is dangerous in distributed systems.

## How does it work?

### The key pair

```
  Key Generation (happens once, at volta-auth-proxy startup):

  ┌──────────────────────────────────────────────────────┐
  │  RSA Key Pair Generator (2048-bit)                    │
  │                                                       │
  │  Generates two mathematically linked keys:            │
  │                                                       │
  │  Private Key                    Public Key            │
  │  ┌────────────────────┐        ┌────────────────────┐ │
  │  │ VERY long number   │        │ VERY long number   │ │
  │  │ (kept secret)      │───────►│ (shared freely)    │ │
  │  │                    │ derived│                    │ │
  │  │ Can SIGN data      │  from  │ Can VERIFY sigs    │ │
  │  │ Can DECRYPT data   │        │ Can ENCRYPT data   │ │
  │  └────────────────────┘        └────────────────────┘ │
  │                                                       │
  │  You CANNOT derive the private key from the public key│
  └──────────────────────────────────────────────────────┘
```

### Signing a JWT with RS256

```
  Step 1: Create the JWT content

    Header:  {"alg":"RS256","typ":"JWT","kid":"key-2026-03-31T09-00"}
    Payload: {"sub":"user-uuid","volta_tid":"tenant-uuid",...}

  Step 2: Encode both parts

    encoded = BASE64URL(header) + "." + BASE64URL(payload)

  Step 3: Sign with the private key

    signature = RSA_SIGN(SHA256(encoded), private_key)

  Step 4: Assemble the JWT

    jwt = encoded + "." + BASE64URL(signature)

    Result: eyJhbGci...header...eyJzdWIi...payload...SflKxwRJ...signature
```

### Verifying a JWT with RS256

```
  Step 1: Parse the JWT into 3 parts (header, payload, signature)

  Step 2: Check the algorithm
    - header.alg MUST be RS256
    - Reject HS256, none, or anything else

  Step 3: Get the public key
    - Read "kid" from header
    - Fetch public key from JWKS endpoint (or cache)

  Step 4: Verify the signature
    - Compute: RSA_VERIFY(SHA256(header + "." + payload), signature, public_key)
    - If true: the JWT was signed by the holder of the private key
    - If false: the JWT was modified or forged

  Step 5: Check claims (exp, iss, aud, etc.)
```

### Why RS256 is safe for distributed systems

```
  ┌──────────────────────────────┐
  │     volta-auth-proxy         │
  │                              │
  │  Has: Private Key (SECRET)   │
  │  Does: Signs JWTs            │
  │                              │
  │  Also publishes:             │
  │  /.well-known/jwks.json      │
  │  (public key, NOT secret)    │
  └──────────────┬───────────────┘
                 │
       Public key shared via JWKS
                 │
    ┌────────────┼────────────┐
    │            │            │
    ▼            ▼            ▼
  ┌──────┐   ┌──────┐   ┌──────┐
  │App A │   │App B │   │App C │
  │      │   │      │   │      │
  │Has:  │   │Has:  │   │Has:  │
  │Public│   │Public│   │Public│
  │Key   │   │Key   │   │Key   │
  │ONLY  │   │ONLY  │   │ONLY  │
  └──────┘   └──────┘   └──────┘

  If App B is compromised:
  - Attacker gets: public key (already public -- no value)
  - Attacker CANNOT: forge JWTs (needs private key)
  - Attacker CANNOT: impersonate other users
  - Damage is contained to App B's own data
```

## How does volta-auth-proxy use RS256?

### Key generation

volta generates a 2048-bit RSA key pair on first startup:

```java
// JwtService.java
KeyPairGenerator generator = KeyPairGenerator.getInstance("RSA");
generator.initialize(2048);
KeyPair keyPair = generator.generateKeyPair();
```

### Key storage

The private key is encrypted with AES-256-GCM before being stored in the `signing_keys` database table. See `KeyCipher.java`.

### Key rotation

Admins can rotate keys via `POST /api/v1/admin/keys/rotate`. The old key remains valid for verification (in the JWKS endpoint) during a transition period, but new JWTs are signed with the new key.

### JWKS endpoint

The public key is served at `GET /.well-known/jwks.json`:

```json
{
  "keys": [
    {
      "kty": "RSA",
      "kid": "key-2026-03-31T09-00",
      "alg": "RS256",
      "n": "0vx7agoebGcQ...(public modulus)...",
      "e": "AQAB"
    }
  ]
}
```

Apps fetch this endpoint, cache the key, and use it to verify JWT signatures locally.

### Algorithm enforcement

volta explicitly rejects any JWT that does not use RS256:

```java
// JwtService.java - verify()
if (!JWSAlgorithm.RS256.equals(jwt.getHeader().getAlgorithm())) {
    throw new IllegalArgumentException("Unsupported JWT algorithm");
}
```

This prevents the `alg:none` attack and the RS256-to-HS256 confusion attack.

## Common mistakes and attacks

### Attack: The alg:none attack

See [jwt.md](jwt.md) for details. In short: if a server trusts the `alg` header in the JWT and allows `none`, an attacker can remove the signature entirely. volta prevents this with an explicit algorithm whitelist.

### Attack: Algorithm confusion (RS256 -> HS256)

See [hs256.md](hs256.md) for a full explanation. In short: an attacker changes the algorithm from RS256 to HS256 and uses the public key (which is public!) as the HMAC secret. Some JWT libraries will accept this. volta prevents this by only accepting RS256.

### Mistake: Using short RSA keys

RSA keys shorter than 2048 bits are considered insecure. volta uses 2048-bit keys, which is the current standard minimum.

### Mistake: Not rotating keys

If a private key is used forever and eventually leaked, all JWTs ever issued with it can be forged. volta supports key rotation, and old keys can be revoked.

### Mistake: Distributing the private key

Some teams put the private key in environment variables on every service. This defeats the purpose of RS256. Only the signing service (volta-auth-proxy) should have the private key.
