# ランタイム

[English version](runtime.md)

---

## これは何？

ランタイムにはソフトウェアで2つの意味があり、どちらもvolta-auth-proxyに関連します。

**意味1：実行のフェーズ。** ランタイムとは、コードが実際に実行されているときのことで、コンパイル時（コードがバイトコードにコンパイルされるとき）とは対照的です。「ランタイムエラー」はプログラムの実行中に発生するエラーで、ビルド中ではありません。

**意味2：コードを実行する環境。** Java仮想マシン（JVM）はランタイムです -- コンパイルされたバイトコードを読み、オペレーティングシステム上で実行するソフトウェアです。ランタイムがなければ、コードはディスク上にあるただのファイルです。

オルゴールで考えてみましょう。コンパイル時は、金属のシリンダーに正しい突起を付けて製造するとき（コードをバイトコードにコンパイル）です。ランタイムはオルゴールのメカニズムそのもの -- ゼンマイ、櫛歯、筐体で、シリンダーから実際に音楽を奏でるものです。メカニズム（ランタイム）がなければ、シリンダー（バイトコード）はただの装飾的な金属片です。

---

## なぜ重要なのか？

- **ランタイムエラーは本番エラー。** コンパイル時エラーはデプロイ前にコンパイラが検出する。ランタイムエラーは実際のユーザーの前で発生する。
- **ランタイムがパフォーマンスを決める。** JVMのガベージコレクタ、JITコンパイラ、メモリ管理がvoltaのリクエスト応答速度に直接影響する。
- **ランタイムバージョンが重要。** voltaはJava 21 [LTS](lts.ja.md)が必要。Java 17や8では言語機能の欠如により失敗する。
- **ランタイム設定が安定性に影響。** JVMのヒープサイズ、GC戦略、スレッドプールサイズはランタイム上の判断で、本番の動作に影響する。
- **ランタイムは攻撃対象面。** JVM、OS、ランタイムライブラリはすべて攻撃の潜在的ターゲット。

## どう動くのか？

### コンパイル時 vs ランタイム

```
  ソースコード (.javaファイル)
       │
       │  コンパイル時
       │  (javacコンパイラ)
       │
       ▼
  バイトコード (.classファイル / JAR)
       │
       │  ランタイム
       │  (JVMがバイトコードを実行)
       │
       ▼
  実行中のアプリケーション
  (HTTPリクエストを処理中)

  ┌───────────────────────────────────────┐
  │            コンパイル時                 │
  │                                       │
  │  検出されるエラー:                     │
  │  - 構文エラー                         │
  │  - 型の不一致                         │
  │  - インポート不足                     │
  │                                       │
  │  ツール: javac、Maven、IDE            │
  └───────────────────────────────────────┘
                    │
                    ▼
  ┌───────────────────────────────────────┐
  │              ランタイム                │
  │                                       │
  │  検出されるエラー:                     │
  │  - NullPointerException              │
  │  - データベース接続拒否               │
  │  - OutOfMemoryError                  │
  │  - HTTP 500レスポンス                 │
  │                                       │
  │  ツール: JVM、監視、ログ               │
  └───────────────────────────────────────┘
```

### ランタイムとしてのJVM

JVMは現存する最も洗練されたランタイムの一つです：

```
  ┌──────────────────────────────────────┐
  │            JVMランタイム              │
  │                                      │
  │  ┌────────────────┐                 │
  │  │ クラスローダー   │  .classファイル  │
  │  └───────┬────────┘  を読み込み      │
  │          │                           │
  │          ▼                           │
  │  ┌────────────────┐                 │
  │  │ バイトコード     │  最初は         │
  │  │ インタプリタ     │  解釈実行       │
  │  └───────┬────────┘                 │
  │          │                           │
  │          ▼                           │
  │  ┌────────────────┐                 │
  │  │ JITコンパイラ   │  ホットパスを    │
  │  │                │  ネイティブ      │
  │  │                │  コードに変換    │
  │  └───────┬────────┘                 │
  │          │                           │
  │          ▼                           │
  │  ┌────────────────┐                 │
  │  │ ガベージ        │  未使用メモリを  │
  │  │ コレクタ        │  回収           │
  │  └────────────────┘                 │
  │                                      │
  │  ┌────────────────┐                 │
  │  │ スレッド管理     │  並行処理を     │
  │  │                │  管理           │
  │  └────────────────┘                 │
  └──────────────────────────────────────┘
          │
          ▼
  ┌──────────────────────────────────────┐
  │       オペレーティングシステム         │
  │  (Linux, macOS, Windows)             │
  └──────────────────────────────────────┘
```

### JVMランタイムの主要機能

| 機能 | 何をするか | voltaへの影響 |
|------|----------|--------------|
| **JITコンパイル** | ホットなバイトコードをネイティブコードに変換 | ウォームアップ後にvoltaが高速化 |
| **ガベージコレクション** | 未使用メモリを自動解放 | 一時停止がリクエストレイテンシに影響 |
| **スレッド管理** | JavaスレッドをOSスレッドにマッピング | 並行ForwardAuthリクエストを処理 |
| **メモリ管理** | ヒープとスタックの割り当て | キャッシュ可能なセッション数を決定 |
| **セキュリティマネージャ** | コード実行のサンドボックス | (Java 17+で非推奨) |

### ランタイム設定

```bash
# ランタイム設定付きでvoltaを起動
java \
  -Xms256m \              # 初期ヒープサイズ
  -Xmx512m \              # 最大ヒープサイズ
  -XX:+UseG1GC \          # G1ガベージコレクタを使用
  -XX:MaxGCPauseMillis=200 \  # GC一時停止の目標時間
  -jar volta-auth-proxy.jar
```

## volta-auth-proxy ではどう使われている？

### Java 21 LTSランタイム

voltaはJava 21が必要で、これは長期サポートリリースです。voltaが恩恵を受けるJava 21の主要機能：

- **仮想スレッド（Project Loom）** -- 多数の同時リクエストを効率的に処理する軽量スレッド
- **パターンマッチング** -- 型チェックのためのよりクリーンなコード
- **レコードクラス** -- ボイラープレートが少ない不変データキャリア
- **シールドクラス** -- 型を拡張できるクラスを制限
- **テキストブロック** -- SQLやテンプレート用の複数行文字列

### 本番でのランタイム動作

```
  volta-auth-proxy JVMライフサイクル:

  1. JVM起動
     └─ クラスロード、Javalin初期化
     └─ Flywayがマイグレーション実行
     └─ HikariCPがコネクションプールを開く
     └─ Caffeineキャッシュ初期化（空）

  2. ウォームアップフェーズ（約30秒）
     └─ JITコンパイラがまだ活発でない
     └─ 最初のリクエストがやや遅い
     └─ Caffeineキャッシュが蓄積中

  3. 定常状態
     └─ JITがホットパスをコンパイル済み
     └─ コネクションプールが最適サイズ
     └─ キャッシュヒット率が安定
     └─ GC一時停止が最小

  4. シャットダウン
     └─ Javalinがリクエスト受付を停止
     └─ 処理中のリクエストが完了
     └─ HikariCPが接続を閉じる
     └─ JVMが終了
```

### voltaが処理するランタイムエラー

```java
// ランタイムでデータベース接続が切断
try {
    return dataSource.getConnection();
} catch (SQLException e) {
    // ランタイムエラー: DBに到達不能
    // ログ記録、503返却、運用チームに通知
}

// ランタイムでGoogle OIDCに到達不能
try {
    return httpClient.send(tokenRequest);
} catch (HttpTimeoutException e) {
    // ランタイムエラー: Googleが遅い/ダウン
    // ユーザーフレンドリーなエラーページを返す
}
```

## よくある間違いと攻撃

### 間違い1：間違ったJavaバージョンで実行

voltaはJava 21が必要です。Java 17や11で実行すると、起動時に`UnsupportedClassVersionError`が発生します。常に確認：`java -version`。

### 間違い2：ヒープメモリ不足

JVMがヒープメモリを使い果たすと、`OutOfMemoryError`が発生しプロセスがクラッシュします。ヒープ使用量を監視し、負荷に応じて`-Xmx`を適切に設定しましょう。

### 間違い3：GC一時停止を無視

長いガベージコレクション一時停止はリクエストタイムアウトを引き起こします。ForwardAuthチェックが時々200ms以上かかる場合、`-Xlog:gc`でGCログを確認しましょう。

### 間違い4：ランタイムメトリクスを監視しない

ヒープ使用量、スレッド数、GC頻度を監視しなければ、盲目で飛ぶようなものです。JMXまたはPrometheusメトリクスでJVMの健全性を追跡しましょう。

### 間違い5：コンパイル時の安全性をランタイムの安全性と混同

Javaの型システムはコンパイル時に多くのエラーを検出しますが、`NullPointerException`、データベース障害、ネットワークタイムアウトは検出できません。ランタイムのエラーハンドリングは依然として不可欠です。
