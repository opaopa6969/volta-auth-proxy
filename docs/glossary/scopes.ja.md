# スコープ

[English / 英語](scopes.md)

---

## これは何？

スコープは、アプリケーションが ID プロバイダにどの情報やアクセスを要求しているかを定義するラベルです。同意画面で「このアプリがあなたのメールアドレスの閲覧を希望しています」と表示されるのは、アプリが `email` スコープを要求したからです。スコープは認可リクエストの一部として送信され、ID プロバイダはそれに基づいてレスポンスに含めるデータを決定します。

---

## なぜ重要？

スコープは最小権限の原則を実現します。アプリケーションは実際に必要なデータだけを要求すべきです。ユーザーが誰かとメールアドレスだけ知ればよいアプリが、Google ドライブのファイルへのアクセスまで要求すべきではありません。過剰なスコープ要求はユーザーの信頼を損ない（同意画面が怖くなる）、リスクを増大させます（アプリが侵害された場合、より多くのデータが漏洩する）。

ID プロバイダにとって、スコープは機密データを明示的なユーザー同意の背後に守る手段でもあります。

---

## 簡単な例

最も一般的な3つの OIDC スコープ：

| スコープ | 提供されるもの |
|-------|-----------------|
| `openid` | OIDC に必須。id_token に `sub`（サブジェクト識別子）を返す。これがないと OIDC ではなく通常の OAuth 2.0 |
| `email` | ユーザーのメールアドレスと検証済みかどうか（`email`、`email_verified` クレーム） |
| `profile` | ユーザーの名前、写真、その他のプロフィール情報（`name`、`picture`、`locale` クレーム） |

これらのスコープを含む認可リクエスト：

```
scope=openid email profile
```

Google に伝えていること：「この人が誰か知りたい（openid）、メールアドレスが欲しい（email）、表示名が欲しい（profile）」

## volta-auth-proxy での使い方

volta はちょうど3つのスコープを要求します：

```java
params.put("scope", "openid email profile");
```

各スコープが必要な理由：

- **openid**: OIDC フローにする（OAuth だけではない）。Google がユーザーを一意に識別する `sub` クレームを含む `id_token` を返すことを保証
- **email**: テナントメンバーシップや招待との照合にメールアドレスが必要。`email_verified` フラグもチェック -- volta は未検証のメールを拒否
- **profile**: `name` クレームをユーザーの表示名として使用。users テーブルに保存し、JWT にも `volta_display` として含める

volta は `https://www.googleapis.com/auth/drive` や `https://www.googleapis.com/auth/calendar` のようなより広いスコープは要求しません。ユーザーの代わりに Google API にアクセスする必要がないためです。ID 情報だけが必要です。

この最小限のスコープセットにより、Google の同意画面は短く脅威的でなくなります：「volta があなたのメールアドレスと基本的なプロフィール情報の閲覧を希望しています」 広範な権限を要求するアプリと比べて、ユーザーはこれを信頼し承認しやすくなります。
