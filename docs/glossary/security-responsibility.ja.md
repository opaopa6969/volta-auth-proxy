# セキュリティの責任

[English version](security-responsibility.md)

---

自分で認証を構築すると、セキュリティを自分で所有します。この文は言うのは簡単で、実践するのは難しい。このエッセイは「セキュリティを所有する」が実際に何を意味するか -- 理論ではなく、自前の認証システムを運用する日常の現実において -- について書いています。

---

## 何にサインアップしているのか

volta-auth-proxy（または自前ホスト型の認証ソリューション）をAuth0の代わりに選ぶとき、暗黙的に以下に同意しています：

**「依存関係のセキュリティ脆弱性を監視し、速やかに修正します。」**

voltaはライブラリに依存しています：JWT処理のJJWT、HTTPのJetty（Javalin経由）、データベース接続のHikariCPなど。これらのライブラリには潜在的な脆弱性があります。Auth0はこれをあなたのために監視します。voltaでは、自分でやります。

**「実装しているプロトコルのセキュリティアドバイザリを追い続けます。」**

OAuth2とOIDCには既知の攻撃ベクトルがあります。新しいものが定期的に発見されます。PKCEに対する新しい攻撃が公開されたら、voltaが影響を受けるか評価し、必要なら修正する必要があります。

**「インシデントに迅速に対応します。」**

ユーザーがアカウント侵害を報告したら、調査し、原因を特定し、対応する必要があります。Auth0にはインシデント対応チームがあります。voltaでは、あなたのチームがインシデント対応チームです。

## セキュリティ責任の月ごとの姿

### 1ヶ月目：セットアップ

すべてが新しい。voltaのコードをレビューし、認証フローを理解し、デプロイする。セキュリティは管理可能に感じます。今作ったばかりだから。

### 3ヶ月目：依存関係の更新

GitHubのDependabotが`io.jsonwebtoken:jjwt-impl`に新バージョンがあると通知。セキュリティ修正？リリースノートを確認。特定のJWTパースのエッジケースに対するマイナー修正。更新を決定。テスト。デプロイ。

```
  Dependabotアラート：
  ┌──────────────────────────────────────────────────────┐
  │ jjwt-impl 0.12.5 → 0.12.6                           │
  │ 重大度: 中                                            │
  │ 「修正: 特定の不正なヘッダーを持つJWTがまれなケースで │
  │  署名検証をバイパスする可能性があった」               │
  └──────────────────────────────────────────────────────┘

  あなたの対応：
  1. CVEの詳細を読む（30分）
  2. システムへの影響を評価（1時間）
  3. pom.xmlを更新（5分）
  4. テスト実行（10分）
  5. ステージングにデプロイ（30分）
  6. ステージングで検証（1時間）
  7. 本番にデプロイ（30分）
  8. 本番で検証（30分）

  合計：約4時間の集中的なセキュリティ作業
```

Auth0なら、これはあなたにとってゼロ時間です。彼らのチームが処理します。

### 6ヶ月目：ペネトレーションテスト

ペネトレーションテスト会社を雇います。1週間かけて認証システムを破ろうとします。発見事項：

1. 特定の管理エンドポイントでCSRFトークンが検証されていない（中程度の重大度）
2. バーストトラフィック時にレートリミッターが設定より若干多くのリクエストを許可（低重大度）
3. ログインページのエラーメッセージがメールアドレスの存在を露呈（情報開示、低重大度）

3つすべてを修正。時間：2-3日の集中作業。

### 9ヶ月目：実際のインシデント

ユーザーが要求していないパスワードリセットメールを受信したと報告。（voltaはGoogle OIDCを使うのでパスワードリセットはありませんが、メール/パスワードログインを追加したとして。）調査：

- ユーザーアカウントの監査ログを確認
- 不審なIPアドレスを探す
- 正当な間違いか攻撃の試みか判断
- 影響を受けたユーザーとコミュニケーション
- インシデントレポートを作成

時間：1-2日。

### 12ヶ月目：プロトコルの進化

新しいOAuth2仕様が公開（例：ベストプラクティスを統合したOAuth 2.1）。評価：

- voltaは既にこれらのプラクティスに従っているか？（ほぼはい。PKCEは常に必須だったから。）
- 実装すべき新しい推奨事項はあるか？（新しいヘッダーやパラメータかもしれない。）
- 更新が必要か？（おそらく緊急ではないが、計画に入れる。）

時間：4-8時間の調査と評価。

## コスト比較：ドル vs 責任

100,000 MAUでのAuth0は約月額$2,400（年間$28,800）。何を買っている？

```
  年間$28,800でAuth0から買えるもの：
  ┌────────────────────────────────────────────────┐
  │ ✓ 24/7セキュリティモニタリング                  │
  │ ✓ 自動依存関係パッチ                            │
  │ ✓ インシデント対応チーム                        │
  │ ✓ SOC 2 Type IIコンプライアンス                 │
  │ ✓ 定期的なペネトレーションテスト                │
  │ ✓ DDoS保護                                     │
  │ ✓ アップタイムSLA                               │
  │ ✓ 脆弱性開示プログラム                          │
  │ ✓ セキュリティリサーチチーム                    │
  └────────────────────────────────────────────────┘

  voltaで年間$0があなたに要求すること：
  ┌────────────────────────────────────────────────┐
  │ ✓ 依存関係の脆弱性を監視                        │
  │ ✓ パッチを速やかに適用                          │
  │ ✓ セキュリティインシデントに対応                │
  │ ✓ ペネトレーションテストを実施または委託        │
  │ ✓ DDoS保護を設定（Cloudflare等）               │
  │ ✓ アップタイムを監視                            │
  │ ✓ セキュリティアドバイザリをレビュー            │
  │ ✓ 認証プロトコルの知識を最新に保つ              │
  └────────────────────────────────────────────────┘
```

これがトレードオフです。年間$28,800で他の誰かに心配してもらうか、年間$0プラスチームの時間と注意力。どちらの選択肢も客観的に優れているわけではありません。チームの能力と優先順位次第です。

## voltaのトレードオフ：$0だがあなたがセキュリティチーム

voltaの立場は透明です：

1. **十分にテストされた安全なコードを提供します。** voltaはセキュリティのベストプラクティスに従います：PKCE、RS256、署名付きCookie、CSRFトークン、レート制限、暗号化された鍵ストレージ。

2. **コードを理解可能にします。** voltaのコードベースは1人の開発者がすべてを読んで理解できるほど小さい。理解できないものを監査することはできません。

3. **依存関係を最小限にします。** 依存関係が少ない = 脆弱性の表面が少ない。voltaのコアは数個の依存関係で、数百ではありません。

4. **残りはあなたの責任です。** モニタリング。パッチ適用。テスト。インシデント対応。これらはあなたの責任です。

これはvoltaのモデルのバグではありません。これがモデルです。年間$28,800を節約します。その節約をエンジニアリングの時間と注意に使います。強いエンジニアリング文化を持つチームにとって、これは良い取引です。純粋にプロダクトに集中し認証セキュリティについて考えたくないチームにとって、Auth0がより良い選択です。

## volta運用者のための実践的セキュリティチェックリスト

volta-auth-proxyを運用するなら、これが最低限のセキュリティ衛生管理です：

- [ ] voltaリポジトリで**Dependabot**（またはRenovate）を有効化
- [ ] 通知から48時間以内に**依存関係の更新をレビュー**
- [ ] JJWT、Jetty、PostgreSQL JDBCドライバの**CVEフィードを購読**
- [ ] 少なくとも年1回**ペネトレーションテストを実施**
- [ ] 不審なパターン（ブルートフォース、異常なIP）の**監査ログを監視**
- [ ] **Javaランタイムを最新に保つ**（セキュリティパッチ）
- [ ] **PostgreSQLを最新に保つ**（セキュリティパッチ）
- [ ] **あらゆる場所でHTTPSを使用**（TraefikでのTLS終端）
- [ ] voltaの内部APIへの**ネットワークアクセスを制限**
- [ ] **データベースをバックアップ**し、定期的にリカバリをテスト
- [ ] 閾値を超える認証失敗に対する**アラートを設定**
