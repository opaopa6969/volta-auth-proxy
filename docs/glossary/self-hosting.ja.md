# セルフホスティング

[English version](self-hosting.md)

---

## これは何？

セルフホスティングとは、他人のクラウドでソフトウェアを動かしてもらう代わりに、自分が所有または管理するサーバーでソフトウェアを実行することです。

自炊とデリバリーの違いで考えてみましょう。自炊なら、材料を自分で買い、レシピをコントロールし、何が入っているか正確に分かり、1食あたりは安くなります。でも皿洗いもキッチンの掃除も自分でやらなければなりませんし、料理を覚える必要もあります。デリバリーなら全部やってもらえます。でもお金は多くかかり、カスタマイズはあまりできず、利用できるレストランに制限されます。

セルフホスティングは、テクノロジースタックにおける「自炊」モデルです。

---

## なぜ重要なのか？

SaaSの世界では、ほとんどのツールがクラウドサービスとして提供されます。月額料金を払えば、その会社がソフトウェアを運用してくれます。多くのことにはうまく機能します。ただし、一部のもの -- 特に認証 -- ではセルフホスティングに大きな利点があります。

### セルフホスティングの利点

| 利点 | なぜ重要か |
|------|-----------|
| **コスト管理** | ユーザー単位の課金なし。コストはサーバーだけ（月$10-50のVMでかなり対応できる）。 |
| **データ所有権** | ユーザーデータは自分のサーバーに残る。第三者がユーザーデータベースのコピーを持たない。 |
| **プライバシー/コンプライアンス** | 一部の規制（GDPR、医療、政府）ではデータを特定の場所に置く必要がある。セルフホスティングならデータの置き場所を正確に選べる。 |
| **ベンダーロックインなし** | ソフトウェアを変更、フォーク、置換できる。許可は不要。[vendor-lock-in.ja.md](vendor-lock-in.ja.md)を参照。 |
| **カスタマイズ** | ログインUI、メールテンプレート、セッション動作、すべての設定を完全にコントロール。 |
| **予想外の請求なし** | インフラコストは予測可能。[MAU](mau.ja.md)に連動しない。 |

### セルフホスティングの欠点

| 欠点 | 現実 |
|------|------|
| **運用責任** | 稼働率、監視、インシデント対応は自分の責任。午前3時にサーバーが落ちたら自分の問題。 |
| **セキュリティパッチ** | ソフトウェアを最新に保つ必要がある。自動的にセキュリティ修正を適用してくれる人はいない。 |
| **バックアップ** | データベースのバックアップは自分で設定。忘れてディスクが壊れたらデータは消える。 |
| **セットアップの手間** | 初期設定はクラウドサービスで「アカウント作成」をクリックするよりも時間がかかる。 |
| **サポートチームなし** | ベンダーのサポート窓口はない。ドキュメント、コミュニティ、自分のスキルが頼り。 |

### 正直なトレードオフ

```
  クラウドホスト型認証（Auth0/Clerk）：
  ┌──────────────────────────────────┐
  │  始めるのは簡単                   │
  │  運用は他人がやってくれる          │
  │  スケールすると高額に             │
  │  時間とともにコントロールを失う    │
  └──────────────────────────────────┘

  セルフホスト型認証（volta）：
  ┌──────────────────────────────────┐
  │  最初のセットアップは手間が多い    │
  │  運用は自分で                     │
  │  どんなスケールでも安い           │
  │  完全なコントロールをずっと維持    │
  └──────────────────────────────────┘
```

## volta-auth-proxyのセルフホスティングへのアプローチ

voltaはセルフホスト専用です。クラウドホスト型のvoltaサービスはありません。これは制限ではなく、意図的な設計判断です。

セルフホスティングを実用的にするため、voltaは運用負荷を最小化しています：

| 懸念事項 | voltaのアプローチ |
|---------|-----------------|
| **依存関係** | PostgreSQLだけ。Redis不要（オプション）、Kafka不要（オプション）、Elasticsearch不要（オプション）。 |
| **メモリ** | ~30MB RAM。小さなインスタンスで動く。 |
| **起動** | ~200ms。高速な再起動、高速なデプロイ。 |
| **設定** | `.env`ファイル1つ + `volta-config.yaml` 1つ。500行の設定ファイルは不要。[config-hell.ja.md](config-hell.ja.md)を参照。 |
| **データベースマイグレーション** | [Flyway](flyway.ja.md)による自動実行。アプリを起動すればスキーマ更新が行われる。 |
| **デプロイ** | 単一のJavaプロセス。複雑なオーケストレーションは不要。 |

### 最小限のセルフホスティング構成

```yaml
# docker-compose.yml -- これだけで十分
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: volta_auth
      POSTGRES_USER: volta
      POSTGRES_PASSWORD: volta

  volta:
    image: volta-auth-proxy:latest
    environment:
      DATABASE_URL: jdbc:postgresql://postgres:5432/volta_auth
      GOOGLE_CLIENT_ID: your-google-client-id
      GOOGLE_CLIENT_SECRET: your-google-client-secret
      SESSION_SECRET: your-random-secret
    ports:
      - "7070:7070"
```

約15行の設定で本番運用可能な認証システムです。

## セルフホスティングと他のデプロイモデルの比較

| モデル | 例 | 自分がコントロール | 支払い |
|--------|------|-------------|--------|
| **クラウドSaaS** | Auth0、Clerk | 何も（彼らのサーバー、彼らのデータ、彼らのルール） | MAU連動の月額料金 |
| **クラウドVMでのセルフホスト** | AWSのEC2でvolta | ソフトウェア、データ、設定。AWSはハードウェアを管理。 | VMコスト（月~$10-50） |
| **オンプレミスでのセルフホスト** | 自社の物理サーバーでvolta | 物理ハードウェアも含めてすべて | ハードウェア + 電気代 |

ほとんどのvoltaユーザーはクラウドVM（AWS、GCP、またはシンプルなVPS）でセルフホストします。物理ハードウェアを管理せずに、セルフホスティングのコストとコントロールの利点を得られます。
