# 自己署名

[English version](self-signed.md)

---

## これは何？

自己署名の証明書や鍵とは、信頼された第三者機関（認証局など）ではなく、自身の作成者によって署名されたものです。作成者が「自分で自分を保証します」と言っているのです。身元を確認する独立した第三者はいません。

パーティーで誰かが「MegaCorp CEO」と書かれた名刺を渡してきたと想像してください。MegaCorpが実際にその名刺を印刷したのでしょうか？それとも自宅で印刷したのでしょうか？自己署名証明書は、自分で名刺を印刷するようなものです -- 情報は正確かもしれませんが、外部の検証はありません。認証局（CA）からの証明書は、MegaCorpの公式印刷所がホログラムシール付きで印刷した名刺のようなものです。

自己署名は安全でないという意味ではありません。信頼を別のチャネルで確立する必要があるという意味です。自分で鍵を生成し、通信の両端を制御しているなら、自己署名で全く問題ありません。voltaは初回起動時に独自のRSA鍵ペアを生成します -- voltaが署名者でもあり検証者でもあるため、これは自己署名でありながら安全です。

---

## なぜ重要なのか？

自己署名とCA署名の区別は信頼のために重要です：

- **CA署名**: CAを信頼するクライアントは自動的にその証明書を信頼する。公開向けHTTPSで使用
- **自己署名**: この特定の証明書を信頼するよう明示的に設定されたクライアントのみが受け入れる。内部サービス、開発、およびvoltaのように同じシステムが署名と検証を行う場合に使用

CA署名が期待される場所で自己署名証明書を使うと、ブラウザは怖い警告を表示し、APIクライアントは接続を拒否します。しかし内部のJWT署名では、自己署名が標準的なアプローチです -- 署名鍵がシステムから出ないため、CAを介入させる理由がありません。

## どう動くのか？

### CA署名 vs 自己署名

```
  CA署名の証明書:
  ┌──────────────────────────────────────────────┐
  │  「私はapi.example.comです」                   │
  │                                              │
  │  署名者: Let's Encrypt (CA)                   │
  │  CAの署名者: ISRG Root X1                     │
  │  ルートの信頼者: 地球上のすべてのブラウザ       │
  │                                              │
  │  信頼の連鎖:                                  │
  │  ブラウザ → ルートCA → 中間CA → 証明書         │
  └──────────────────────────────────────────────┘

  自己署名の証明書/鍵:
  ┌──────────────────────────────────────────────┐
  │  「私はvolta-auth-proxyです」                  │
  │                                              │
  │  署名者: volta-auth-proxy（自分自身）           │
  │  信頼者: volta-auth-proxy（自分自身）           │
  │                                              │
  │  信頼の連鎖は不要:                             │
  │  voltaがJWTに署名 → voltaがJWTを検証           │
  │  （同じシステム、同じ鍵ペア）                   │
  └──────────────────────────────────────────────┘
```

### 自己署名が適切な場面

| シナリオ | 自己署名OK？ | 理由 |
|---------|------------|------|
| 公開HTTPSサイト | いいえ | ブラウザがCAの信頼の連鎖を必要とする |
| 内部JWT署名 | はい | 同じシステムが署名と検証を行う |
| 開発/テスト | はい | 実ユーザーなし、開発者のみ |
| サービス間mTLS | 場合による | 信頼を手動で管理するならOK |
| voltaのRSA鍵ペア | はい | voltaが両側を制御 |

### 信頼モデル

```
  外部信頼（CA署名）:
  ┌────────┐     ┌──────┐     ┌────────┐     ┌────────┐
  │ルートCA│────→│中間CA│────→│  証明書 │────→│クライアント│
  └────────┘     └──────┘     └────────┘     └────────┘
  ブラウザに      ルートCAが    中間CAが       ルートを信頼
  プリインストール 署名         署名           → 証明書を信頼

  自己信頼（自己署名）:
  ┌────────────────────────────────────────┐
  │  volta-auth-proxy                      │
  │                                        │
  │  秘密鍵 ──署名──→ JWT                  │
  │  公開鍵 ──検証──→ JWT                  │
  │                                        │
  │  外部の信頼の連鎖は不要。               │
  │  公開鍵はJWKSエンドポイントで共有。     │
  └────────────────────────────────────────┘
```

## volta-auth-proxy ではどう使われている？

voltaのRSA鍵ペアは完全に自己署名です。初回起動時に`JwtService.loadOrCreateKey()`が`signing_keys`テーブルを確認します。鍵が存在しなければ、新しい2048ビットRSA鍵ペアを生成します：

```java
private RSAKey createKey() {
    KeyPairGenerator generator = KeyPairGenerator.getInstance("RSA");
    generator.initialize(2048);
    KeyPair keyPair = generator.generateKeyPair();
    String kid = "key-" + Instant.now()...;
    // データベースに暗号化して保存
    store.saveSigningKey(kid,
        keyCipher.encrypt(base64(publicKey)),
        keyCipher.encrypt(base64(privateKey)));
    return rsaKey;
}
```

これが自己署名である理由：

1. **voltaが鍵を生成する** -- CAは関与しない
2. **voltaが秘密鍵でJWTに署名する** -- `RSASSASigner`を使用
3. **volta（と下流サービス）が公開鍵でJWTを検証する** -- `/.well-known/jwks.json`で公開
4. **信頼は設定によって確立される** -- 下流アプリはCAではなくvoltaのJWKSエンドポイントを信頼するよう設定される

### なぜJWT署名にCAを使わないのか？

- JWTはTLS証明書ではない。ブラウザの信頼は不要
- CAを追加すると[設定地獄](config-hell.md)が増えるだけでセキュリティ上の利点なし
- [署名鍵](signing-key.md)はvoltaから出ない。署名レベルで「中間者」を防ぐ必要がない
- voltaの[自動鍵生成](auto-key-generation.md)により手動の証明書管理はゼロ

## よくある間違いと攻撃

### 間違い1: 公開HTTPSに自己署名を使用

ブラウザが「この接続はプライベートではありません」と警告を表示します。ユーザーは離脱します。公開向けTLSにはLet's Encryptなどのを使いましょう。

### 間違い2: 本番環境で証明書検証を無効化

開発者は自己署名証明書での開発中に`verify_ssl=False`や`InsecureSkipVerify`を設定し、再有効化を忘れがちです。これはTLSの意味を完全に無にします。

### 間違い3: 自己署名と安全でないを混同する

JWT署名のための自己署名鍵は業界標準です。セキュリティはCAの認証ではなく、秘密鍵を秘密に保つことから来ます。voltaのアプローチはAuth0、Firebase、その他のIDプロバイダーと同じです。

### 間違い4: 自己署名鍵をローテーションしない

CAの有効期限がローテーションを強制しないため、自己署名鍵は永遠に残りがちです。常に[鍵ローテーション](key-rotation.md)を実装しましょう。voltaは`POST /api/v1/admin/keys/rotate`でローテーションをサポートしています。
