# SendGrid

[English version](sendgrid.md)

---

## これは何？

SendGrid（現在Twilioの一部）は、アプリケーションがAPIを通じてトランザクションメールやマーケティングメールを送信できるクラウドベースのメール配信サービスです。自分でメールサーバーを運用する（悪名高く難しい）代わりに、メールをSendGridに渡せば、配信、スパムフィルタリング、バウンス管理、配信到達性レピュテーションを処理してくれます。

SendGridは手紙のプロの宅配サービスのようなものです。各受取人の家まで自分で歩いて行く（自分のSMTPサーバーを運用する）こともできますが、ロックされたゲート（スパムフィルター）、間違った住所（バウンス）、間違った地域に配達した場合の悪い評判に対処することになります。宅配サービス（SendGrid）はすべての郵便局（メールプロバイダー）との信頼関係を確立しており、ルートを知り、すべての配達を追跡します。手紙と住所を渡すだけです。

メール配信は簡単に聞こえますが、実際には最も難しいインフラ問題の一つです。スパムフィルターを通過するには、IPレピュテーション管理、SPF/DKIM/DMARC認証、フィードバックループ処理、スパム対策規制への準拠が必要です。SendGridはこれらすべてを処理します。

---

## なぜ重要なのか？

volta-auth-proxyは、いくつかの認証関連フローでメールを送信する必要があります：

| メールタイプ | 送信タイミング |
|-----------|-------------|
| **招待** | テナント管理者が新しいユーザーを参加に招待 |
| **パスワードリセット** | ユーザーがパスワードリセットをリクエスト（メール/パスワード認証が有効な場合） |
| **MFA回復** | ユーザーがMFAアクセスの回復が必要 |
| **セキュリティアラート** | 異常なログインを検出（新しいデバイス、新しい場所） |
| **アカウント通知** | サブスクリプション変更、プランアップグレード（[Stripe](stripe.md) webhookから） |
| **管理者アラート** | ブルートフォース検出、レート制限超過、テナント問題 |

これらは**トランザクションメール**です -- ユーザーアクションによってトリガーされ、一斉マーケティングではありません。迅速かつ確実に届く必要があります。30分かかる（または届かない）パスワードリセットメールは、壊れた製品です。

## どう動くのか？

### SendGrid APIによるメール送信

```
  volta-auth-proxy
       │
       │  POST https://api.sendgrid.com/v3/mail/send
       │  Authorization: Bearer SG.xxxxxxxx
       │  {
       │    "personalizations": [{
       │      "to": [{"email": "alice@acme.com"}],
       │      "dynamic_template_data": {
       │        "invite_link": "https://auth.example.com/invite/abc123",
       │        "tenant_name": "Acme Corp"
       │      }
       │    }],
       │    "from": {"email": "noreply@example.com"},
       │    "template_id": "d-abc123template"
       │  }
       │
       ▼
  SendGrid
       │
       │  1. APIキーを検証
       │  2. 動的データでテンプレートをレンダリング
       │  3. DKIMで署名
       │  4. 配信キューに追加
       │  5. ソフトバウンス時にリトライ
       │  6. 開封/クリック/バウンスを追跡
       │
       ▼
  alice@acme.comの受信箱
```

### メール認証（SPF、DKIM、DMARC）

メールが受信箱（スパムではなく）に届くには、3つの認証メカニズムを設定する必要があります：

| メカニズム | 機能 | 設定方法 |
|----------|------|---------|
| **SPF** | どのIPがドメインのメール送信を許可されているか受信サーバーに通知 | DNS TXTレコード：`v=spf1 include:sendgrid.net ~all` |
| **DKIM** | メールに暗号署名して改ざんを防止 | SendGridが提供するDNS CNAMEレコード |
| **DMARC** | SPF/DKIMが失敗した場合に受信サーバーが何をすべきか通知 | DNS TXTレコード：`v=DMARC1; p=quarantine; ...` |

### SendGrid vs 代替手段

| 機能 | SendGrid | Amazon SES | Mailgun | Postmark |
|------|----------|-----------|---------|----------|
| 料金 | 無料（100通/日）、$15+/月〜 | $0.10/1000通 | $0.80/1000通 | $1.25/1000通 |
| API品質 | 良好 | 基本的 | 良好 | 優秀 |
| テンプレート | 動的テンプレート（Handlebars） | 基本的 | あり | あり |
| 配信到達性 | 良好 | 良好（ただしレピュテーション管理が必要） | 良好 | 優秀（トランザクション専用） |
| Webhook（配信イベント） | あり | あり（SNS経由） | あり | あり |
| 専用IP | 利用可能（有料） | デフォルト | 利用可能（有料） | 共有（マネージド） |
| セットアップの容易さ | 簡単 | 中程度（サンドボックスモード、検証） | 簡単 | 簡単 |

## volta-auth-proxy ではどう使われている？

SendGridはvolta-auth-proxyの**オプションの通知チャネル**です。voltaはSendGrid経由でメールを送信するように設定できますが、他のプロバイダーもサポートし、メールを完全にスキップすることもできます（メールが不要なデプロイ向け）。

### 統合

voltaは信頼性と追跡のためにSendGridのREST API（SMTPではなく）を使用します：

```env
# .env設定
EMAIL_PROVIDER=sendgrid
SENDGRID_API_KEY=SG.xxxxxxxxxxxxxxxxxxxxxxxx
EMAIL_FROM=noreply@yourdomain.com
EMAIL_FROM_NAME=あなたのSaaS名
```

### メールテンプレート

voltaはHandlebars構文のSendGrid動的テンプレートを使用します。各メールタイプにSendGridのテンプレートがあります：

| テンプレート | 変数 | 目的 |
|-----------|------|------|
| 招待 | `{{invite_link}}`、`{{tenant_name}}`、`{{inviter_name}}` | テナントへのユーザー招待 |
| セキュリティアラート | `{{user_name}}`、`{{event_type}}`、`{{ip}}`、`{{location}}` | 異常なアクティビティの通知 |
| 管理者通知 | `{{tenant_name}}`、`{{alert_type}}`、`{{details}}` | テナント管理者へのアラート |

### なぜ特にSendGridか？

voltaはSendGridを義務付けていません -- いくつかの選択肢の一つです。SendGridがデフォルトとして言及される理由：

1. **無料ティア**：毎日100通無料（小規模デプロイに十分）
2. **APIのシンプルさ**：1回のREST呼び出しでメール送信
3. **テンプレートシステム**：動的テンプレートでメールフォーマットをvoltaのコードから分離
4. **配信追跡**：バウンス、開封、クリックのWebhook

voltaのメール統合はインターフェースの背後に抽象化されているため、SendGridからAmazon SES、Mailgun、またはローカルSMTPサーバーへの切り替えは、コードではなく設定の変更のみで可能です。

## よくある間違いと攻撃

### 間違い1：共有送信ドメインを使用する

SPF、DKIM、DMARCを設定せずに`noreply@yourdomain.com`から送信すると、メールはスパムに入ります。SendGridのダッシュボードで必ず送信ドメインを認証してください。

### 間違い2：バウンスを処理しない

バウンスする（ハードバウンス）メールアドレスに送信し続けると、SendGridが送信者レピュテーションを下げます。voltaはSendGridバウンスWebhookを処理し、無効なアドレスをマークすべきです。

### 間違い3：APIキーの漏洩

SendGrid APIキーはあなたのドメインとしてメール送信を許可します。漏洩した場合（コード、ログ、クライアント側）、攻撃者があなたのドメインから来たように見えるフィッシングメールを送信できます。キーは環境変数に保存し、ソースコードには絶対に入れないでください。

### 間違い4：メールを送りすぎる

招待や通知メールはテナントごとにレート制限すべきです。「ユーザー招待」機能にアクセスした攻撃者が数千の招待をスパム送信し、SendGridのクォータを使い果たしドメインがスパムとしてフラグされる可能性があります。

### 攻撃：メールベースのアカウント乗っ取り

攻撃者がパスワードリセットメールをトリガーし、それを傍受できる場合（例：受信者のドメインのDNSハイジャック経由）、パスワードをリセットしてアカウントを乗っ取れます。以下で軽減：
- 短命のリセットトークン（10〜15分）
- ワンタイムトークン
- ユーザーの他のチャネルへの通知（例：アプリ内通知）
