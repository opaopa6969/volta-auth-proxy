# セッション有効期限

[English version](session-expiry.md)

---

## これは何？

セッション有効期限とは、ログインセッションが無効になりユーザーが再認証する必要がある時点のことです。すべてのセッションには限られた寿命があり、時間切れになるとセッションは「死に」、ユーザーはログインページにリダイレクトされます。

パーキングメーターを思い浮かべてください。2時間分を支払います。2時間後、駐車は期限切れです -- メーターに再度コインを入れる（再認証）か、駐車違反切符をもらう（401エラー）かです。コインを入れるたびにタイマーがリセットされるメーターもあります（スライディングウィンドウ）。何があっても固定時刻で期限切れになるものもあります（絶対タイムアウト）。

セッション有効期限はセキュリティと使いやすさのバランスです。短すぎると、ユーザーは常に再ログインを求められます。長すぎると、忘れられたセッション（共有コンピュータなど）が何時間も何日もアクセス可能なまま残ります。

---

## なぜ重要なのか？

セッション有効期限がなければ：

- **共有コンピュータ**: ユーザーが図書館でログインして立ち去り、別の人がそのセッションを無期限に使用できる
- **セッションの窃取**: ネットワーク傍受で盗まれたセッションCookieが永遠に使える
- **リソース枯渇**: 二度と使われないセッションがメモリやデータベースの行を消費し続ける
- **コンプライアンス違反**: PCI DSSなどの規制がセッションタイムアウトを要求

セッション有効期限は安全網です。他のすべての防御が失敗しても（HttpOnlyバイパス、HTTPS侵害、XSS悪用）、被害は時間的に制限されます。

## どう動くのか？

### 2種類の有効期限

```
  絶対タイムアウト:
  ┌──────────────────────────────────────────────┐
  │  セッション作成: 09:00                        │
  │  絶対タイムアウト: 8時間                      │
  │  死亡: 17:00（何があっても）                  │
  │                                              │
  │  09:00 ═══════════════════════════ 17:00     │
  │  ログイン                          無効       │
  │                                              │
  │  ユーザーがアプリを使い続けていても、         │
  │  セッションは17:00に期限切れ。                │
  └──────────────────────────────────────────────┘

  スライディングウィンドウ（アイドルタイムアウト）:
  ┌──────────────────────────────────────────────┐
  │  セッション作成: 09:00                        │
  │  スライディングウィンドウ: 8時間              │
  │                                              │
  │  09:00  リクエスト → 期限を17:00にリセット    │
  │  11:00  リクエスト → 期限を19:00にリセット    │
  │  14:00  リクエスト → 期限を22:00にリセット    │
  │  ...ユーザーが帰宅...                        │
  │  22:00  14:00以降リクエストなし → 期限切れ    │
  │                                              │
  │  タイマーはすべてのアクティビティでリセット。 │
  │  ユーザーがアイドルのときだけ期限切れ。       │
  └──────────────────────────────────────────────┘
```

### voltaはスライディングウィンドウを使用

```
  voltaのセッション有効期限: 8時間スライディングウィンドウ

  時間 ──────────────────────────────────────────►

  09:00  ログイン → session.expires_at = 17:00
         │
  10:30  API呼び出し → session.expires_at = 18:30
         │                （現在時刻から8時間先にスライド）
  12:00  ページ読み込み → session.expires_at = 20:00
         │
  14:00  フォーム送信 → session.expires_at = 22:00
         │
  ...ユーザーが退勤...
         │
  22:00  アクティビティなし → セッション期限切れ
         │
  翌朝: ユーザーがサイトにアクセス → 401 → ログインページ
```

### 有効期限チェックフロー

```
  __volta_session Cookie付きリクエストが到着
         │
         ▼
  データベースでセッションを参照
         │
         ├── 見つからない? → 401（セッション削除済みか存在しない）
         │
         ├── 見つかった。expires_atを確認:
         │   │
         │   ├── expires_at < 現在 → 期限切れ → セッション削除 → 401
         │   │
         │   └── expires_at > 現在 → 有効
         │       │
         │       ▼
         │   expires_at = 現在 + 8時間 に更新（ウィンドウをスライド）
         │       │
         │       ▼
         │   リクエスト処理を続行
         │
         └── 更新されたMax-AgeでSet-Cookie
```

## volta-auth-proxy ではどう使われている？

### 設定

voltaのセッション寿命は8時間（28800秒）で、`__volta_session` Cookieに`Max-Age=28800`として設定されます。

### スライディングウィンドウの実装

認証済みのリクエストごとに、voltaは：

1. `sessions`テーブルからセッションを読み取る
2. `expires_at`が過去か（期限切れ）を確認
3. 有効なら`expires_at`を`現在 + 8時間`に更新
4. 新しい`Max-Age=28800`でCookieを再送信

```
  sessions テーブル:
  ┌────────────────────────────────────────────────┐
  │ id:         550e8400-e29b-41d4-a716-...        │
  │ user_id:    alice-uuid                         │
  │ tenant_id:  acme-uuid                          │
  │ expires_at: 2026-04-01T22:00:00Z  ← スライド   │
  │ created_at: 2026-04-01T09:00:00Z  ← 固定       │
  └────────────────────────────────────────────────┘
```

### JWT有効期限との関係

voltaでは2つの有効期限の時計が同時に動いています：

```
  ┌─────────────────────────────────────────────┐
  │  セッション: 8時間スライディングウィンドウ    │
  │  JWT:        5分絶対期限                     │
  │                                             │
  │  セッション期限切れ → 新しいJWTを発行不可    │
  │  → ユーザーは再ログインが必要               │
  │                                             │
  │  JWT期限切れ → クライアントがサイレント      │
  │  リフレッシュ → 新しいJWT発行               │
  │  （セッションがまだ有効な場合）              │
  │                                             │
  │  セッション有効期限:「まだログイン中？」     │
  │  JWT有効期限:「このトークンは新鮮？」        │
  └─────────────────────────────────────────────┘
```

### クリーンアップ

期限切れセッションは蓄積を防ぐために定期的にデータベースから削除すべきです：

```sql
DELETE FROM sessions WHERE expires_at < NOW();
```

## よくある間違いと攻撃

### 間違い1: セッション有効期限がまったくない

永遠に生きるセッションは恒久的なセキュリティホールです。セッションCookieを盗んだ攻撃者が無期限のアクセスを得ます。常に有効期限を設定しましょう。

### 間違い2: クライアント側だけの有効期限

サーバー側で`expires_at`を確認せずにCookieに`Max-Age`を設定するだけでは、変更された有効期限でCookieを手動で再送信されるとセッションが生き続けます。常にサーバー側で検証しましょう。

### 間違い3: 絶対タイムアウトが長すぎる

銀行アプリケーションで30日の絶対タイムアウトは危険です。データの機密性に合わせてタイムアウトを設定しましょう。voltaの8時間スライディングウィンドウはビジネスアプリケーションに適切です。

### 間違い4: ウィンドウをスライドしない

8時間の絶対タイムアウトでは、09:00にログインしたユーザーが一日中アクティブでも17:00に追い出されます。対話型アプリケーションでは[スライディングウィンドウ](sliding-window-expiry.md)のほうがユーザーフレンドリーです。

### 攻撃: ログアウト後のセッション持続

ログアウト時にセッションレコードを削除せずCookieだけをクリアすると、セッションIDをキャプチャした攻撃者が有効期限まで使い続けられます。ログアウト時は常にデータベースからセッションを削除しましょう。[無効化](invalidation.md)を参照。
