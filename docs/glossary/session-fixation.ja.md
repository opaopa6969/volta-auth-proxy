# Session Fixation（セッション固定攻撃）

## これは何？

セッション固定攻撃とは、被害者がログインする**前に**、攻撃者が被害者のセッション ID を攻撃者が既に知っている値に設定（「固定」）する攻撃です。被害者が認証すると、サーバーはその既存のセッションを認証済み状態にアップグレードします。攻撃者はセッション ID を知っているので、被害者のアカウントにアクセスできるようになります。

こう考えてください。ホテルに行くと、チェックインする前に見知らぬ人がルームキーを渡してきます。あなたはそのキーをフロントデスクで使い、フロントがそのキーをあなたの部屋で有効化します。これで、あなたと見知らぬ人の両方があなたの部屋の有効なキーを持っています。

この攻撃が巧妙なのは、被害者は全て正常に操作するからです。自分の認証情報でログインし、自分のダッシュボードを見て、他の誰かも自分のセッションにアクセスしていることに気づきません。

## なぜ重要？

セッション固定攻撃が危険な理由:

1. **認証をバイパスする。** 攻撃者は被害者のパスワードを知る必要がありません。被害者に攻撃者が制御するセッション ID を使わせるだけです。
2. **検出が難しい。** サーバーから見ると、セッションは正常に見えます。ユーザーは正しく認証されています。セッション自体に不審な点はありません。
3. **安全なシステムでも機能する。** 強力なパスワード、多要素認証、暗号化された接続を使っていても、ログイン時にセッション ID を再生成しなければセッション固定攻撃は成功します。

## どう動くの？

攻撃のステップ:

```
  攻撃者                         サーバー                     被害者
  |                             |                           |
  | 1. 有効なセッション ID を取得  |                           |
  |  GET /login                 |                           |
  |---------------------------->|                           |
  |  Set-Cookie: session=EVIL123|                           |
  |<----------------------------|                           |
  |                             |                           |
  | 2. 被害者に                  |                           |
  |    session=EVIL123 を使わせる |                           |
  |    (リンク、XSS、            |                           |
  |     メタタグ注入で)           |                           |
  |----------------------------------------------------->  |
  |                             |                           |
  |                             | 3. 被害者が               |
  |                             |    session=EVIL123 で      |
  |                             |    ログイン                |
  |                             |<--------------------------|
  |                             |                           |
  |                             | 4. サーバーがユーザーを認証  |
  |                             |    して同じ session=EVIL123 |
  |                             |    を維持（認証済みに）      |
  |                             |                           |
  | 5. 攻撃者が                  |                           |
  |    session=EVIL123 を使用    |                           |
  |---------------------------->|                           |
  |                             |                           |
  |    サーバーは有効な            |                           |
  |    認証済みセッションとして認識  |                           |
  |    --> 攻撃者が侵入！         |                           |
```

### 攻撃者がセッションを「固定」する方法

- **URL パラメータ:** `https://your-app.com/login?sessionid=EVIL123`（アプリが URL のセッション ID を受け入れる場合）
- **XSS による Cookie 注入:** 被害者のブラウザで `document.cookie = "session=EVIL123"` を実行
- **メタタグやヘッダの注入:** 攻撃者がページに HTML を注入できる場合
- **サブドメイン Cookie の設定:** 攻撃者がサブドメインを制御している場合、親ドメインの Cookie を設定できる

### 対策: ログイン時にセッションを再生成する

防止策はシンプルです: ユーザーが認証に成功したら、完全に新しいセッション ID を作成します。認証前のセッション ID を認証後のセッションに絶対に再利用しないでください。

```
  脆弱:                              安全:

  ログイン前:                        ログイン前:
  session = EVIL123                  session = EVIL123

  ユーザーが認証                      ユーザーが認証

  ログイン後:                        ログイン後:
  session = EVIL123（同じ！）         session = NEW_RANDOM_456
  攻撃者の ID がまだ使える             EVIL123 は無効化
                                     攻撃者の ID は役立たず
```

## volta-auth-proxy ではどう使っている？

volta-auth-proxy はそのアーキテクチャによりセッション固定攻撃に対して免疫を持つよう設計されています。

**ログインのたびに新しいセッション。** ユーザーが OIDC 認証を完了すると、volta は常に新しい UUID で新しいセッションを作成します:

```java
UUID sessionId = SecurityUtils.newUuid();  // 暗号学的にランダムな UUID
store.createSession(sessionId, ...);       // 新しいデータベースレコード
setSessionCookie(ctx, sessionId, ...);     // 新しい Cookie をブラウザに送信
```

古いセッション（もしあれば）は引き継がれません。新しいランダムなセッション ID が生成され、新しいデータベースレコードが作成されます。

**サーバーサイドのセッション保存。** volta のセッションは Cookie 自体ではなく、PostgreSQL に保存されます。Cookie にはランダムな UUID だけが含まれます。つまり:
- 攻撃者は Cookie の値を捏造して有効なセッションを作れない。
- セッション ID は volta のコードがデータベースに明示的に作成したときだけ有効になる。
- 存在しないセッション ID は拒否されるため、ログイン前にセッションを「固定」する方法がない。

**テナント切り替え時の古いセッション無効化。** ユーザーがテナントを切り替えるとき、volta は古いセッションを失効させて新しいものを作成します:

```java
// 古いセッションを失効
store.revokeSession(UUID.fromString(oldSessionRaw));

// 新しい ID で新しいセッションを作成
UUID sessionId = authService.issueSession(switched, ...);
setSessionCookie(ctx, sessionId, ...);
```

**同時セッション数の制限。** volta は各ユーザーの同時セッションを 5 に制限しています。制限を超えると、最も古いセッションが自動的に失効されます。追加の保護を提供します: たとえ攻撃者がセッションを固定できたとしても、ユーザーが十分な数の他のセッションを持っていれば、自動的にクリーンアップされます。

防御の層:

```
  攻撃者がセッションを固定しようとする:

  層 1: セッション ID は暗号学的にランダムな UUID
        --> 攻撃者は有効な ID を予測・捏造できない

  層 2: セッションは volta のコードが作成したときだけ存在する
        --> 乗っ取る既存のセッションがない

  層 3: ログインは常に新しいセッションを作成する
        --> ログイン前のセッション ID は無関係になる

  層 4: ログイン/切り替え時に古いセッションを失効する
        --> 古いセッションが存在しても無効化される

  層 5: ユーザーあたり最大 5 同時セッション
        --> 古いセッションは自動的に整理される
```

## よくある間違い

**1. 認証後にセッション ID を再利用する。**
これがセッション固定の根本原因です。修正はシンプル: ユーザーの権限レベルが変わるとき（ログイン、ロール変更、テナント切り替え）に常に新しいセッション ID を生成する。

**2. URL パラメータからセッション ID を受け入れる。**
URL でセッション ID を許可する（`?JSESSIONID=...` のように）と、攻撃者が細工したリンクで簡単にセッションを固定できます。volta は HttpOnly Cookie からのみセッション ID を受け入れます。

**3. 古いセッションを無効化しない。**
新しいセッションの作成は必要ですが十分ではありません。古いセッションも無効化（削除または非アクティブとマーク）すべきです。そうしないと、古いセッション ID と新しいセッション ID の両方が機能し、攻撃者は古いものを通じてアクセスできます。

**4. 連番や予測可能なセッション ID を使う。**
セッション ID が予測可能（インクリメントする整数など）だと、攻撃者はセッションを固定する必要すらなく、推測するだけで済みます。volta は `UUID.randomUUID()` を使い、122 ビットのランダム性を提供します。

**5. HTTPS がセッション固定を防ぐと思い込む。**
HTTPS はネットワーク盗聴を防ぎますが、攻撃者が被害者を事前設定されたセッション ID 付きの URL に誘導したり、他の注入テクニックを使うことは防ぎません。セッション固定はロジックの欠陥であり、通信の欠陥ではありません。

**6. ログイン時だけセッションを再生成し、権限昇格時はしない。**
一般ユーザーが管理者になる場合も権限変更であり、新しいセッションを必要とします。volta はテナント切り替え時（ロールが変わる可能性がある）にセッションを再生成します。
