# セッション管理

[English](session-management.md) | [日本語](session-management.ja.md)

> 知らなくても大丈夫です！みんな毎日セッションを使っているけど、意識していないだけ。

## これは何？

**セッション管理とは、「この人はログイン済みです」をページを移動しても覚えておく仕組み全体のこと。** セッションそのものだけでなく、作成・保存・検証・期限切れ・破棄の全てを含みます。

ホテルのフロントに例えると:
- **チェックイン** = セッション作成（ログイン）
- **ルームキー** = セッション Cookie（宿泊客である証明）
- **宿泊者名簿** = セッションストア（誰が泊まっているかのデータベース）
- **カードキーの有効期限** = セッションタイムアウト
- **チェックアウト** = セッション破棄（ログアウト）
- **全室カードキー無効化** = 管理者による強制ログアウト

## なぜ大事？

セッション管理がないと、ユーザーは**ページを開くたびに毎回ログイン**しなければなりません。それは明らかにひどい体験です。でも間違ったやり方はセキュリティの大事故になります:

- 期限切れしないセッション → 盗まれたセッションが永久に使える
- 安全でない保存方法 → 誰でもユーザーになりすませる
- 同時セッション制限なし → 攻撃者のログインに気づけない
- セッション無効化機能なし → 漏洩したアカウントを強制ログアウトできない

## どう動く？

```
ログイン:
  ユーザーが認証 → サーバーがセッション作成 → Cookie をブラウザに送信
                         ↓
                    DB にセッション保存
                    (id, user_id, tenant_id, expires_at)

毎リクエスト:
  ブラウザが Cookie を送信 → サーバーがセッション検証
                                  ↓
                             有効？ → 有効期限を延長（スライディング）
                             期限切れ？ → 401、ログインへリダイレクト
                             無効化済み？ → 401、ログインへリダイレクト

ログアウト:
  サーバーがセッションを無効化 → Cookie 削除
```

## volta-auth-proxy では

volta のセッション管理は以下の全てを含みます:

| 機能 | 実装 |
|------|------|
| **作成** | Google OIDC コールバック成功時 |
| **保存** | Postgres の `sessions` テーブル |
| **Cookie** | `__volta_session`、HttpOnly、Secure、SameSite=Lax |
| **署名** | HMAC-SHA256（Cookie の改ざん防止） |
| **有効期限** | 8 時間の[スライディングウインドウ](sliding-window-expiry.ja.md) |
| **同時制限** | ユーザーあたり最大 5 セッション |
| **無効化** | `sessions.invalidated_at` フィールド |
| **固定攻撃対策** | ログイン成功時にセッション ID 再生成 |
| **マルチデバイス** | `/settings/sessions` でセッション一覧・個別失効 |
| **テナント紐付け** | 各セッションは 1 つの[テナント](tenant.ja.md)に紐付く |
