# Session（セッション）

[English version](session.md)

---

## これは何？

セッションは、複数のリクエストにわたってサーバーがあなたを覚えておくための仕組みです。ログインすると、サーバーがセッション（ログイン状態の記録）を作成し、ブラウザにセッションID（通常はCookieで）を渡します。その後のすべてのリクエストでブラウザがセッションIDを送り、サーバーがセッションを検索してあなたが誰かを知ります。

ホテルのキーカードのようなものです。チェックイン時にフロントデスクが記録（部屋番号、チェックアウト日、名前）を作成し、キーカードを渡します。キーカード自体にはあなたの個人情報は含まれていません。ただの番号です。でもドアでかざすと、ホテルのシステムがその番号を検索して入室を許可するか判断します。

---

## なぜ重要なのか？

HTTPは「ステートレス」です。各リクエストが独立しています。セッションがなければ、サーバーはページ読み込みのたびにあなたを忘れます。持続的なログインを可能にするのがセッションです。

適切なセッション管理がないと：

- ユーザーはすべてのリクエストでログインし直す必要がある
- ログアウトが機能しない（無効化するものがない）
- 同時セッション数の制限が不可能
- 現在ログイン中のユーザーを追跡したりアクセスを取り消したりできない

## どう動くのか？

### サーバーサイドセッション（voltaが使用するもの）

```
  ログイン：
  ════════
  1. ユーザーが認証（Google OIDC）
  2. サーバーがデータベースにセッションレコードを作成：
     ┌─────────────────────────────────────────────┐
     │  id:           550e8400-e29b-41d4-...        │
     │  user_id:      user-uuid                     │
     │  tenant_id:    tenant-uuid                   │
     │  created_at:   2026-03-31T09:00:00Z          │
     │  expires_at:   2026-03-31T17:00:00Z (8時間後)│
     │  ip_address:   192.168.1.100                 │
     │  user_agent:   Chrome/120 on macOS           │
     │  csrf_token:   Kj8mX2pQ...                   │
     └─────────────────────────────────────────────┘
  3. サーバーがセッションIDをCookieとして送信：
     Set-Cookie: __volta_session=550e8400-e29b-41d4-...

  後続のリクエスト：
  ════════════════
  1. ブラウザが送信: Cookie: __volta_session=550e8400-...
  2. サーバーがデータベースでセッションを検索
  3. 有効で期限切れでなければ：
     - 有効期限を延長（スライディングウィンドウ）
     - ユーザーIDを返す
  4. 無効または期限切れなら：
     - 401 Unauthorizedを返す

  ログアウト：
  ══════════
  1. サーバーがセッションを無効化（invalidated_atを設定）
  2. サーバーがブラウザからCookieを削除
  3. 古いセッションIDでの後続リクエストは401になる
```

### サーバーサイド vs Cookieベースのセッション

根本的に異なる2つのアプローチがあります：

```
  サーバーサイドセッション（voltaのアプローチ）：
  ┌─────────────────┐     ┌──────────────────────────────┐
  │ Cookie:          │     │ データベース（sessionsテーブル）：│
  │ session_id=UUID  │ ──► │ - user_id                    │
  │                  │     │ - tenant_id                  │
  │ (キーだけ、       │     │ - expires_at                 │
  │  データなし)      │     │ - ip_address                 │
  └─────────────────┘     │ - csrf_token                 │
                          └──────────────────────────────┘

  長所: ✓ 即座に無効化可能（DBから削除）
        ✓ セッションデータはプライベート（Cookieに含まれない）
        ✓ Cookieが極小（UUIDだけ）
        ✓ アクティブセッションの一覧/管理が可能

  短所: ✗ リクエストごとにデータベース検索が必要
        ✗ データベースが依存関係


  Cookieベースセッション（voltaは使用しない）：
  ┌──────────────────────────────────────────────┐
  │ Cookie:                                       │
  │ session=BASE64(ENCRYPTED({                    │
  │   user_id: "user-uuid",                      │
  │   tenant_id: "tenant-uuid",                  │
  │   expires_at: "2026-03-31T17:00:00Z",        │
  │   roles: ["ADMIN"]                           │
  │ }))                                           │
  │                                               │
  │ (すべてのデータがCookieに、署名または暗号化)    │
  └──────────────────────────────────────────────┘

  長所: ✓ セッション検索にデータベース不要
        ✓ スケールしやすい（共有状態なし）

  短所: ✗ ブロックリストなしでは無効化できない
        ✗ Cookieサイズ制限（4KB）
        ✗ クライアントにデータが見える（署名のみで暗号化なしの場合）
        ✗ アクティブセッションの一覧ができない
```

voltaがサーバーサイドセッションを選んだのは、無効化とセッション管理が必須の機能だからです。

### スライディングウィンドウ有効期限

voltaはセッション有効期限に「スライディングウィンドウ」を使用します：

```
  最初のログイン（午前9:00）：
  ├── セッション作成
  └── 有効期限：午後5:00（8時間後）

  午前10:30にユーザーがリクエスト：
  ├── セッションが見つかり有効
  ├── 有効期限を延長：午後6:30（現在から8時間後）
  └── レスポンス送信

  午後2:00にユーザーがリクエスト：
  ├── セッションが見つかり有効
  ├── 有効期限を延長：午後10:00（現在から8時間後）
  └── レスポンス送信

  ユーザーが帰宅、8時間以上何もしない：
  ├── セッションが午後10:00に期限切れ
  └── 次のリクエストで401 → ログインにリダイレクト
```

つまり、アクティブなユーザーはセッション期限切れに中断されることがなく、非アクティブなセッションは8時間の無操作後に期限切れになります。

## volta-auth-proxyではどう使われているか？

### セッションのライフサイクル

```
  ログイン（Google OIDCコールバック）
  │
  ├── チェック：ユーザーのアクティブセッションが5つ以上？
  │   └── はい：最も古いセッションを無効化
  │
  ├── 新しいセッションID（UUID）を生成
  ├── CSRFトークン（ランダム）を生成
  ├── sessionsテーブルに以下を保存：
  │   - user_id, tenant_id
  │   - IPアドレス、ユーザーエージェント
  │   - 8時間の有効期限
  │   - CSRFトークン
  │
  ├── Cookieを設定：__volta_session=<uuid>
  └── アプリにリダイレクト

  リクエストごと：
  │
  ├── __volta_session Cookieを読む
  ├── DBでセッションを検索
  ├── チェック：有効？期限切れでない？無効化されていない？
  │   ├── いいえ：401を返す
  │   └── はい：続行
  │
  ├── チェック：ユーザーはセッションのテナントのメンバーか？
  │   ├── いいえ：401を返す
  │   └── はい：続行
  │
  ├── セッション有効期限を延長（スライディングウィンドウ）
  └── ユーザーID（AuthPrincipal）を返す

  ログアウト：
  │
  ├── DBでセッションを無効化
  ├── ブラウザからCookieを削除
  └── ログインにリダイレクト
```

### 同時セッション制限

voltaは各ユーザーの同時セッションを5つに制限します。これにより以下を防ぎます：

- 認証情報の共有（デバイスが多すぎる = 共有アカウント）
- セッションの蓄積（古いセッションが永久に残る）

5つのセッションが存在する状態でログインすると、最も古いセッションが無効化されます：

```java
// AuthService.java
int current = store.countActiveSessions(principal.userId());
if (current >= MAX_CONCURRENT_SESSIONS) {
    int revokeCount = (current - MAX_CONCURRENT_SESSIONS) + 1;
    store.revokeOldestActiveSessions(principal.userId(), revokeCount);
}
```

### セッション管理UI

ユーザーは`/settings/sessions`でセッションを表示・管理できます：

- すべてのアクティブセッションを確認（デバイス、IP、最終アクティブ時刻）
- 個別セッションの無効化（例：「スマホを無くした」）
- すべてのセッションの無効化（緊急の「すべてのデバイスからログアウト」）

### セッション固定攻撃と防止

**セッション固定とは？**

```
  1. 攻撃者がvoltaにアクセスし、セッションIDを取得：session=攻撃者のセッション
  2. 攻撃者が被害者にこのセッションIDを使わせるよう仕向ける
     （例：XSSや特別に作成したリンクでCookieを設定）
  3. 被害者がログイン。サーバーが攻撃者のセッションを認証済みにアップグレード。
  4. 攻撃者が攻撃者のセッションを使用 -- 被害者としてログイン成功。
```

**voltaの防止策：**

voltaはログインのたびに完全に新しいセッションIDを生成します。古いセッションID（もしあれば）は再利用されません：

```java
// AuthService.java - issueSession()
UUID sessionId = SecurityUtils.newUuid();  // 常に新しいUUID
```

攻撃者がセッションIDを仕込んでも、被害者のログイン時に新しいものに置き換わります。攻撃者の古いセッションIDは無効になります。

## よくある間違いと攻撃

### 間違い1：予測可能なセッションID

セッションIDが連番（1, 2, 3...）や予測可能だと、攻撃者が他のユーザーのセッションIDを推測できる。voltaは`UUID.randomUUID()`を使用しており、暗号学的にランダムです。

### 間違い2：セッションを期限切れにしない

期限切れにならないセッションは、盗まれたセッションが永久に有効ということ。voltaは8時間のスライディング有効期限を使用。

### 間違い3：ログアウト時に無効化しない

一部のシステムはCookieを削除するがデータベースのセッションは有効なまま。攻撃者がセッションIDを持っていれば（ログアウト前にコピー）、まだ使える。voltaはデータベースでセッションを無効化済みにマーク。

### 間違い4：ログイン時にセッションを再生成しない

上記の「セッション固定」を参照。ログイン時には必ず新しいセッションを作成すること。

### 攻撃：セッションハイジャック

攻撃者が有効なセッションIDを入手した場合（XSS、ネットワーク盗聴など）、ユーザーになりすませる。防御策：

- **HttpOnly** Cookie（XSSでの持ち出しを防止）
- **Secure**フラグ（HTTPでの盗聴を防止）
- **IP/UserAgent追跡**（不審な変更を検知）
- **短い有効期限**（悪用の機会を制限）
- **セッション無効化**（ユーザーが侵害されたセッションを無効化可能）

voltaはこれらすべてを実装しています。
