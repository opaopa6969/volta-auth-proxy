# サイレントリフレッシュ

[English version](silent-refresh.md)

---

## これは何？

サイレントリフレッシュとは、ユーザーが認証情報を再入力したり、何かが起きたことに気づくことなく、自動的に新しいアクセス[トークン](token.md)を取得するプロセスです。現在のトークンが期限切れになると、クライアント側のコードがバックグラウンドで静かに新しいトークンをリクエストします。

自動車のETCカード自動更新を思い浮かべてください。ETCカードは毎月期限切れになりますが、ETC会社が自動的にカードを引き落としてカウンターに立ち寄ることなく新しいカードを有効化します。料金所を中断なく通過し続けます。カードの更新は「サイレント」に行われ、ログイン画面や「再認証してください」メッセージは見えません。

voltaのアーキテクチャでは、volta-sdk-jsがJWTの期限切れが近い（または既に期限切れで401レスポンスをトリガーした）ことを検出し、自動的にトークンエンドポイントを呼んで新しいJWTを取得します -- ユーザーにローディングスピナーが表示されたりログインページにリダイレクトされたりすることは一切ありません。

---

## なぜ重要なのか？

短寿命トークン（voltaのJWT: 5分）はセキュリティのベストプラクティスです -- トークンが盗まれた場合の被害を制限します。しかしサイレントリフレッシュがなければ、5分のトークンはユーザーが5分ごとに再認証を中断されることを意味します。実際のアプリケーションでは受け入れられません。

サイレントリフレッシュはセキュリティと使いやすさの間のギャップを埋めます：

- **セキュリティ**: トークンは短寿命のまま（5分）。盗まれたトークンは最大5分しか使えない
- **使いやすさ**: ユーザーは8時間（セッション寿命）中断なしにログイン状態を維持
- **透明性**: トークンの更新はユーザーにもアプリケーションコードにも見えない

サイレントリフレッシュがなければ、開発者は悪い選択を迫られます：長寿命トークン（安全でない）か、常時の再認証（使えない）か。

## どう動くのか？

### リフレッシュサイクル

```
  時間 ──────────────────────────────────────────────────►

  0:00   ユーザーがログイン
         voltaがJWT発行（0:05に期限切れ）
         volta-sdk-jsがJWTをメモリに保存

  0:04   SDKがJWTの期限切れまで1分未満を検出
         SDKがPOST /api/v1/auth/tokenを呼ぶ
         （__volta_session Cookie付き）
         voltaが新しいJWT発行（0:09に期限切れ）
         SDKが古いJWTを新しいものに差し替え

  0:08   SDKがJWTの期限切れまで1分未満を検出
         SDKがPOST /api/v1/auth/tokenを呼ぶ
         voltaが新しいJWT発行（0:13に期限切れ）
         ...

  これが8時間のセッション全体で、見えないところで
  約4分ごとに繰り返される。
```

### 401ベースのリフレッシュ

```
  期限切れJWTでのAPI呼び出し:
  ┌──────────────────────────────────────────────────┐
  │  アプリコード: fetch("/api/data", { jwt })       │
  │                                                  │
  │  サーバー: JWT期限切れ → 401 Unauthorized        │
  │                                                  │
  │  volta-sdk-jsが401を傍受:                        │
  │  1. POST /api/v1/auth/tokenを呼ぶ              │
  │     （Cookieベース、JWT不要）                    │
  │  2. 新しいJWTを受け取る                         │
  │  3. 元のリクエストを新しいJWTでリトライ         │
  │  4. レスポンスをアプリコードに返す              │
  │                                                  │
  │  アプリコードは401を見ない。                     │
  │  少し遅れるが、データを受け取るだけ。            │
  └──────────────────────────────────────────────────┘
```

### プロアクティブ vs リアクティブリフレッシュ

```
  プロアクティブ（期限切れ前）:
  ┌──────────────────────────────────────────┐
  │  JWTの期限切れ: T+5:00                   │
  │  SDKのリフレッシュ: T+4:00（1分前）      │
  │  401は発生しない                         │
  │  リトライ不要                            │
  │  最高のユーザー体験                      │
  └──────────────────────────────────────────┘

  リアクティブ（期限切れ後）:
  ┌──────────────────────────────────────────┐
  │  JWTの期限切れ: T+5:00                   │
  │  SDKのAPI呼び出し: T+5:30                │
  │  サーバーが401を返す                     │
  │  SDKがリフレッシュ、リトライ             │
  │  わずかな遅延、でもユーザーは気づかない   │
  └──────────────────────────────────────────┘

  volta-sdk-jsは両方の戦略をサポート。
```

### セッションCookieが可能にすること

```
  リフレッシュトークンなしでサイレントリフレッシュが機能する理由:

  ┌─────────────────────────────────────────────────┐
  │  従来のアプローチ:                               │
  │  アクセストークン(5分) + リフレッシュトークン(7日)│
  │  リフレッシュトークンの保存先は...どこ？          │
  │  localStorage？（XSSリスク）                     │
  │  httpOnly Cookie？（それならそれを使えばいい）    │
  │                                                 │
  │  voltaのアプローチ:                              │
  │  JWT(5分) + セッションCookie(8時間)              │
  │  セッションCookieはhttpOnly、Secure、SameSite   │
  │  これがリフレッシュの仕組みそのもの。             │
  │  別のリフレッシュトークンは不要。                 │
  └─────────────────────────────────────────────────┘

  POST /api/v1/auth/token
  Cookie: __volta_session=abc123  ← これが本人確認
  Response: { "token": "eyJhbG..." }  ← 新しいJWT
```

## volta-auth-proxy ではどう使われている？

### サーバー側: トークンエンドポイント

voltaは`POST /api/v1/auth/token`（または同等）を公開し：

1. `__volta_session` [署名Cookie](signed-cookie.md)を読み取る
2. セッションを検証（存在する、期限切れでない、署名が有効）
3. 更新された`iat`と`exp` [クレーム](claim.md)で新しいJWTを発行
4. レスポンスボディでJWTを返す

このコールでセッションの[スライディングウィンドウ](sliding-window-expiry.md)も延長されます。

### クライアント側: volta-sdk-js

SDKがサイレントリフレッシュを透過的に処理します：

```
  volta-sdk-jsライフサイクル:
  ┌────────────────────────────────────────────┐
  │  1. SDKを初期化                           │
  │  2. トークンエンドポイントから初期JWT取得  │
  │  3. リフレッシュタイマー開始(JWT exp - 60秒)│
  │  4. タイマー発火時:                       │
  │     └── トークンエンドポイントを呼ぶ      │
  │     └── メモリ内JWTを更新                 │
  │     └── タイマーをリセット                │
  │  5. 401レスポンス時:                      │
  │     └── トークンエンドポイントを呼ぶ      │
  │     └── 失敗したリクエストをリトライ      │
  │  6. トークンエンドポイント失敗時:         │
  │     └── セッション期限切れ→ログインへ遷移 │
  └────────────────────────────────────────────┘
```

### セッションが期限切れになったとき

```
  JWT期限切れ → SDKがトークンエンドポイントを呼ぶ
  セッションも期限切れ → サーバーが401を返す
  SDKが新しいJWTを取得できない → セッション終了
  SDKが発動: ログインページにリダイレクト

  ユーザーの画面:「セッションが期限切れです。再度ログインしてください。」
  これは8時間無操作の後にのみ発生。
```

## よくある間違いと攻撃

### 間違い1: リフレッシュのためにJWTをlocalStorageに保存

`localStorage`はページ読み込みをまたいで永続化しますが、XSSからアクセス可能です。volta-sdk-jsはJWTをメモリのみ（JavaScript変数）に保存します。ページリロード時にSDKはトークンエンドポイントから新しいJWTを取得します。

### 間違い2: リフレッシュ失敗を処理しない

トークンエンドポイントがエラーを返した場合（セッション期限切れ、サーバーダウン）、SDKはそれを適切に処理すべきです -- ログインにリダイレクト、エラーメッセージを表示、無限ループしない。

### 間違い3: 並行リクエストの競合状態

5つのAPI呼び出しがすべて同時に401を受けた場合、SDKは5回リフレッシュを呼ぶべきではありません。すべての待機中リクエストが共有する単一の進行中リフレッシュPromiseを使いましょう。

### 間違い4: セッション検証なしのリフレッシュ

トークンエンドポイントは毎回セッションを検証しなければなりません。セッションの有効性を確認せずにJWTを発行するリフレッシュエンドポイントは、セッション有効期限の意味を無にします。

### 攻撃: リフレッシュ経由のセッション固定化

セッションIDが適切に[再生成](regenerate.md)されなければ、事前にセッションをセットアップした攻撃者が被害者のそのセッションへの認証を待ち、サイレントリフレッシュでアクセスを維持できます。voltaはログイン時にセッションIDを再生成してこれを防いでいます。
