# シングルプロセス（Single Process）

[English version](single-process.md)

---

## これは何？

シングルプロセスアプリケーションとは、必要なことをすべて自分自身の中で行う1つの実行中のプログラムです。作業の一部を処理するために他の実行中のプログラムに頼りません。すべてのロジック、すべてのデータ処理、すべての判断が1つのプログラム内で行われます。

スイスアーミーナイフとフルツールボックスの違いのようなものです。スイスアーミーナイフにはブレード、ドライバー、栓抜き、はさみがあります -- すべて1つのツールに。持ち歩くのは1つ。はさみが必要なら折り畳みを開きます。フルツールボックスには個別のツールがあり、それぞれ仕事は得意ですが、箱を持ち歩き、正しいツールを見つけ、戻す必要があります。キャンプ（小規模）ならスイスアーミーナイフの勝ち。家を建てる（大規模）ならツールボックスの勝ちです。

## 代替案は何か？

代替案はネットワーク越しに通信する複数のプロセスです：

```
シングルプロセス（volta）：
┌─────────────────────────────────┐
│ volta-auth-proxy                 │
│                                 │
│ [OIDC] → [Session] → [JWT]     │
│          → [ForwardAuth]        │
│          → [Internal API]       │
│                                 │
│ すべてメソッド呼び出し。ネットワークなし。│
└─────────────────────────────────┘

複数プロセス（例：Oryスタック）：
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ Kratos   │──►│ Hydra    │──►│ Keto     │──►│ Oathkeeper│
│(アイデンティティ)│  │ (OAuth)  │   │ (権限)   │   │ (プロキシ)│
└──────────┘   └──────────┘   └──────────┘   └──────────┘
  プロセス間のHTTP呼び出し。各自が独立して失敗できる。
```

## シングルプロセスがシンプルな理由

### デバッグ

シングルプロセスで何かがうまくいかないとき、正確に何が起きたかを示す1つのスタックトレースが得られます：

```
シングルプロセスでのエラー：
  NullPointerException at SessionService.java:47
  called from ForwardAuthHandler.java:23
  called from Router.java:15

  → SessionService.javaを開き、47行目。バグを発見。
```

複数プロセスにまたがって何かがうまくいかないとき、得られるのは... タイムアウトです：

```
プロセスをまたがるエラー：
  "Connection timed out after 5000ms connecting to session-service:8080"

  → どのプロセスが壊れている？Session Service？ネットワーク？DNS？
    ロードバランサー？呼び出し元のサービス？4つのサービスすべてのログを確認。
    タイムスタンプを照合。ログが同期されていることを祈る。
```

### より少ない障害ポイント

シングルプロセスにはクラッシュする可能性のあるものが正確に1つ：自分自身。マルチプロセスシステムにはN個のクラッシュする可能性のあるものに加え、それらの間のネットワーク接続があります：

```
シングルプロセスの障害ポイント：
  1. voltaプロセス
  2. データベース接続
  合計：2

マルチプロセスの障害ポイント：
  1. プロセスA
  2. プロセスB
  3. プロセスC
  4. プロセスD
  5. ネットワーク A→B
  6. ネットワーク B→C
  7. ネットワーク C→D
  8. DNS解決
  9. ロードバランサー
  合計：9以上
```

### より高速

プロセス内のメソッド呼び出しはナノ秒。プロセス間のHTTP呼び出しはミリ秒。1,000倍の差です：

```
内部メソッド呼び出し：        ~0.0001ms（100ナノ秒）
別サービスへのHTTP呼び出し：  ~1-5ms

セッション検索 + JWT発行を含むForwardAuthチェックの場合：
  シングルプロセス：内部呼び出しに~0.0002ms
  マルチプロセス：サービス間HTTP呼び出しに~2-10ms
```

### よりシンプルなデプロイ

1プロセスは1つのDockerコンテナ、1つのヘルスチェック、1つのログストリーム、1つの監視対象を意味します：

```
シングルプロセスのデプロイ：
  docker run volta-auth-proxy
  完了。

マルチプロセスのデプロイ：
  docker-compose up
    - oidc-service
    - session-service
    - jwt-service
    - forwardauth-service
    - redis（サービス間通信用）
  さらに：各サービスのヘルスチェック、ログ集約、サービスディスカバリ。
```

## トレードオフ

シングルプロセスは個別のコンポーネントを独立してスケールできないことを意味します。JWT署名がCPU集約的でセッション検索がIO集約的な場合、マルチプロセスシステムではそれらを別々にスケールできます。シングルプロセスでは、すべてを一緒にスケールします。

voltaのターゲットユースケース（数百万ではなく数千のユーザー）では、これは問題になりません。1つのPostgreSQLインスタンスの前の1つのvoltaインスタンスで十分すぎる負荷を処理します。もっと必要なら、もう1つのvoltaインスタンスを追加し（水平スケーリング）、データベースを共有します。

---

## volta-auth-proxyでは

voltaはOIDCログイン、セッション管理、JWT発行、ForwardAuth検証、内部APIをすべて1つのJVM内のメソッド呼び出しとして処理する単一のJavaプロセスとして動作し、よりシンプルなデバッグ、より少ない障害ポイント、サブミリ秒の内部通信を実現しています。
