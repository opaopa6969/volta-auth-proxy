# Single Source of Truth（唯一の情報源）

[English version](single-source-of-truth.md)

---

## これは何？

唯一の情報源（SSOT）とは、データやルールの決定版が存在する1つの権威ある場所のことです。複数のシステムが同じ情報を必要とするとき、それぞれが独自のコピーを保持するのではなく、すべてがこの1つのソースから読み取ります。矛盾や疑問が生じたら、唯一の情報源が勝ちます。

スポーツの試合の公式スコアボードのようなものです。スタンドのファン、テレビの解説者、チームのベンチがそれぞれスコアを記録しているかもしれません。しかし意見が食い違ったら、公式スコアボードが真実です。誰もスコアボードに異論を唱えません。他の全員がスコアボードに合わせます。

ソフトウェアでは、SSOTの反対は同じルールが3か所に定義されている状態です：設定ファイル、データベースマイグレーション、コードのコメント。それらが必然的にずれていくと、診断がほぼ不可能なバグが生まれます。

---

## なぜ重要なのか？

唯一の情報源がなければ：

- **矛盾**：コードはセッションタイムアウト8時間、設定は4時間、ドキュメントは12時間と言っている。どれが正しい？
- **古いコピー**：誰かがJavaコードのエラーメッセージを更新するが、フロントエンドのローカライゼーションファイルの更新を忘れる
- **セキュリティの隙間**：権限チェックがAPIレイヤーにはあるが管理UIにはない。異なるチームが異なる参照を使って書いたため
- **オンボーディングの苦痛**：新しい開発者がどのソースを信頼すべきかわからない

## どう動くのか？

### 問題：データの重複

```
  SSOTなし：
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ Javaコード    │  │ 設定ファイル  │  │ ドキュメント  │
  │               │  │              │  │              │
  │ timeout=8h    │  │ timeout=4h   │  │ timeout=12h  │
  │ (更新済み)    │  │ (忘れられた)  │  │ (古い)       │
  └──────────────┘  └──────────────┘  └──────────────┘
      どれが正しい？ 誰にもわからない。

  SSOTあり：
  ┌──────────────────────────────────────────────┐
  │ dsl/policy.yaml（唯一のソース）               │
  │                                              │
  │ session:                                     │
  │   expiry:                                    │
  │     duration_seconds: 28800  # 8時間         │
  └──────────────────────────────────────────────┘
      │              │               │
      ▼              ▼               ▼
  Javaが読む    ドキュメント生成   テストが検証
```

### SSOTのパターン

| パターン | 仕組み | 例 |
|---------|--------|-----|
| **正規ファイル** | 1つのファイルがルールを定義、他はすべてそれを読む | voltaの4つのYAML DSLファイル |
| **データベースが真実** | DBスキーマが権威、コードが適応 | PostgreSQLのユーザーレコード |
| **生成コード** | ソースファイルが派生成果物を生成 | OpenAPI仕様がクライアントSDKを生成 |
| **環境変数** | 設定はコード内ではなく環境に存在 | `SESSION_TIMEOUT_SECONDS` |

### SSOTは「コピーが1つだけ」という意味ではない

コピーやキャッシュは問題ありません。パフォーマンスのために必要です。ルールは：コピーがソースと矛盾したら、ソースが勝ち、コピーはリフレッシュされなければならない。

```
  情報源：            dsl/errors.yaml
  キャッシュコピー：   起動時にロードされるJava enum
  生成コピー：         すべてのエラーコードを一覧するドキュメントページ

  errors.yamlが変更された場合：
    → Javaアプリが再起動時に新しい値を読む
    → ドキュメントページがCIで再生成
    → すべてのコピーが唯一の真実に収束
```

## volta-auth-proxyではどう使われているか？

### SSOTとしての4つのDSLファイル

voltaの認証システム全体には、正確に1つの情報源があります：`dsl/`ディレクトリの4つの[DSL](dsl.ja.md)ファイルです。

```
  ┌──────────────────────────────────────────────────────┐
  │ 唯一の情報源                                          │
  │                                                      │
  │ dsl/auth-machine.yaml → すべての状態と遷移            │
  │ dsl/protocol.yaml     → API契約とJWT仕様             │
  │ dsl/policy.yaml       → ロール、権限、制限            │
  │ dsl/errors.yaml       → すべてのエラーコードとメッセージ│
  └───────────────┬──────────────────────────────────────┘
                  │
     ┌────────────┼────────────┬────────────┐
     ▼            ▼            ▼            ▼
  Javaコード    テスト     用語集ドキュメント AIツール
  が実装       が検証     が参照           がここから生成
```

### ファイル横断のSSOTとしてのerrors.yaml

`auth-machine.yaml`と`protocol.yaml`の両方が`errors.yaml`のエラーコードを参照します：

```yaml
# auth-machine.yaml
errors_ref: "dsl/errors.yaml"
# ...
actions:
  - { type: http, action: json_error, error_ref: AUTHENTICATION_REQUIRED }

# protocol.yaml
errors_ref: "dsl/errors.yaml"
# ...
response_error:
  unauthenticated:
    status: 401
    code: AUTHENTICATION_REQUIRED   # errors.yamlで定義

# errors.yaml（唯一のソース）
AUTHENTICATION_REQUIRED:
  status: 401
  message:
    en: "Login is required"
    ja: "ログインが必要です"
```

エラーメッセージを変更したければ、1か所で変更します：`errors.yaml`。状態マシンもプロトコルも自動的に更新されたメッセージを使います。

### 一度だけ定義されるロール階層

```yaml
# dsl/policy.yaml（ロールの唯一のソース）
roles:
  hierarchy:
    - OWNER     # 最上位
    - ADMIN
    - MEMBER
    - VIEWER    # 最下位
```

この階層は以下から参照されます：
- `auth-machine.yaml`のガード：`"membership.role in ['ADMIN', 'OWNER']"`
- `protocol.yaml`のAPI認証：`"ADMIN+"`
- `AppRegistry.java`のForwardAuthロールチェック
- `AuthService.java`のメンバー管理

### DSLにないもの

デプロイごとに異なるため、環境変数や実行時設定として意図的に保持されるものがあります：

```
  DSLのSSOT：                  環境のSSOT：
  ┌────────────────────────┐  ┌────────────────────────┐
  │ ロール階層              │  │ GOOGLE_CLIENT_ID       │
  │ エラーメッセージ        │  │ DATABASE_URL           │
  │ 状態遷移               │  │ VOLTA_SERVICE_TOKEN    │
  │ APIエンドポイント契約   │  │ COOKIE_DOMAIN          │
  │ ガード式               │  │ SESSION_TIMEOUT_SECONDS│
  └────────────────────────┘  └────────────────────────┘
  すべてのデプロイで同一       デプロイごとに異なる
```

## よくある間違いと攻撃

### 間違い1：「影の」情報源

開発者が`policy.yaml`に反映されていない権限チェックをJavaコードに追加。これで権限の情報源が2つになります。食い違うと、ある環境では動くが別の環境では動かなくなります。

### 間違い2：SSOTを読み取り専用のドキュメントとして扱う

DSLを書いてもコードやテストから読まなければ、本当の情報源ではなく、ずれていく可能性のあるドキュメントにすぎません。voltaのJavaコードとテストはDSLファイルを参照し、正直さを保っています。

### 間違い3：情報源が多すぎる

「エラー用に1つ、ロール用に1つ、状態用に1つ、設定用に1つ...」とすると、すぐに15の「唯一の」ソースになります。voltaは4つのDSLファイルと環境変数に制限しています。以上です。

### 間違い4：SSOTのバージョン管理をしない

voltaのDSLファイルには`version`フィールドがあります。スキーマが変更されるとバージョンが上がります。古いコードがバージョンの不一致を検出し、新しいフォーマットを静かに誤解するのではなく、大きく失敗できます。
