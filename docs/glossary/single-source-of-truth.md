# Single Source of Truth

[日本語版はこちら](single-source-of-truth.ja.md)

---

## What is it?

A single source of truth (SSOT) is one authoritative place where the definitive version of data or rules lives. When multiple systems need the same information, they all read from this one source instead of maintaining their own copies. If there is ever a conflict or question, the single source wins.

Think of it like the official scoreboard at a sports game. Fans in the stands, TV commentators, and the team benches might all keep track of the score. But if there is a disagreement, the official scoreboard is the truth. Nobody argues with the scoreboard -- everyone else syncs to it.

In software, the opposite of SSOT is having the same rule defined in 3 places: a config file, a database migration, and a code comment. When they inevitably drift apart, you have bugs that are nearly impossible to diagnose.

---

## Why does it matter?

Without a single source of truth:

- **Contradictions**: The code says session timeout is 8 hours, the config says 4 hours, the docs say 12 hours. Which is correct?
- **Stale copies**: Someone updates the error message in the Java code but forgets to update it in the frontend localization file
- **Security gaps**: A permission check exists in the API layer but not in the admin UI, because they were written by different teams using different references
- **Onboarding pain**: New developers do not know which source to trust

## How does it work?

### The problem: data duplication

```
  WITHOUT SSOT:
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ Java code     │  │ Config file  │  │ Documentation│
  │               │  │              │  │              │
  │ timeout=8h    │  │ timeout=4h   │  │ timeout=12h  │
  │ (updated)     │  │ (forgotten)  │  │ (outdated)   │
  └──────────────┘  └──────────────┘  └──────────────┘
      Which is correct?  Nobody knows.

  WITH SSOT:
  ┌──────────────────────────────────────────────┐
  │ dsl/policy.yaml (THE source)                 │
  │                                              │
  │ session:                                     │
  │   expiry:                                    │
  │     duration_seconds: 28800  # 8 hours       │
  └──────────────────────────────────────────────┘
      │              │               │
      ▼              ▼               ▼
  Java reads it   Docs generated   Tests verify it
```

### Patterns for SSOT

| Pattern | How it works | Example |
|---------|-------------|---------|
| **Canonical file** | One file defines the rules, everything else reads it | volta's 4 YAML DSL files |
| **Database as truth** | DB schema is authoritative, code adapts | User records in PostgreSQL |
| **Generated code** | Source file generates derived artifacts | OpenAPI spec generates client SDKs |
| **Environment variables** | Config lives in the environment, not in code | `SESSION_TIMEOUT_SECONDS` |

### SSOT does NOT mean "only one copy exists"

Copies and caches are fine -- they are necessary for performance. The rule is: when copies disagree with the source, the source wins and copies must be refreshed.

```
  Source of truth:    dsl/errors.yaml
  Cached copy:        Java enum loaded at startup
  Generated copy:     Docs page listing all error codes

  If errors.yaml changes:
    → Java app reads new values on restart
    → Docs page regenerated by CI
    → All copies converge to the single truth
```

## How does volta-auth-proxy use it?

### The 4 DSL files as SSOT

volta's entire auth system has exactly one source of truth: the 4 [DSL](dsl.md) files in the `dsl/` directory.

```
  ┌──────────────────────────────────────────────────────┐
  │ SINGLE SOURCE OF TRUTH                               │
  │                                                      │
  │ dsl/auth-machine.yaml → All states and transitions   │
  │ dsl/protocol.yaml     → API contract and JWT spec    │
  │ dsl/policy.yaml       → Roles, permissions, limits   │
  │ dsl/errors.yaml       → All error codes and messages │
  └───────────────┬──────────────────────────────────────┘
                  │
     ┌────────────┼────────────┬────────────┐
     ▼            ▼            ▼            ▼
  Java code    Tests      Glossary docs   AI tools
  implements   verify     reference       generate from
```

### errors.yaml as cross-file SSOT

Both `auth-machine.yaml` and `protocol.yaml` reference error codes from `errors.yaml`:

```yaml
# auth-machine.yaml
errors_ref: "dsl/errors.yaml"
# ...
actions:
  - { type: http, action: json_error, error_ref: AUTHENTICATION_REQUIRED }

# protocol.yaml
errors_ref: "dsl/errors.yaml"
# ...
response_error:
  unauthenticated:
    status: 401
    code: AUTHENTICATION_REQUIRED   # defined in errors.yaml

# errors.yaml (THE source)
AUTHENTICATION_REQUIRED:
  status: 401
  message:
    en: "Login is required"
    ja: "ログインが必要です"
```

If you want to change the error message, you change it in ONE place: `errors.yaml`. Both the state machine and the protocol automatically use the updated message.

### Role hierarchy defined once

```yaml
# dsl/policy.yaml (THE source for roles)
roles:
  hierarchy:
    - OWNER     # highest
    - ADMIN
    - MEMBER
    - VIEWER    # lowest
```

This hierarchy is referenced by:
- `auth-machine.yaml` guards: `"membership.role in ['ADMIN', 'OWNER']"`
- `protocol.yaml` API auth: `"ADMIN+"`
- `AppRegistry.java` for ForwardAuth role checks
- `AuthService.java` for member management

### What is NOT in the DSL

Some things are deliberately kept as environment variables or runtime config because they vary per deployment:

```
  SSOT in DSL:                    SSOT in environment:
  ┌────────────────────────┐     ┌────────────────────────┐
  │ Role hierarchy          │     │ GOOGLE_CLIENT_ID       │
  │ Error messages          │     │ DATABASE_URL           │
  │ State transitions       │     │ VOLTA_SERVICE_TOKEN    │
  │ API endpoint contracts  │     │ COOKIE_DOMAIN          │
  │ Guard expressions       │     │ SESSION_TIMEOUT_SECONDS│
  └────────────────────────┘     └────────────────────────┘
  Same across all deploys         Different per deployment
```

## Common mistakes and attacks

### Mistake 1: "Shadow" sources of truth

A developer adds a permission check in Java code that is not reflected in `policy.yaml`. Now there are two sources of truth for permissions. When they disagree, one environment works and another does not.

### Mistake 2: Treating the SSOT as read-only documentation

If you write the DSL but never read it in code or tests, it is not really a source of truth -- it is just documentation that can drift. volta's Java code and tests reference the DSL files, keeping them honest.

### Mistake 3: Too many sources of truth

"We will have one source for errors, one for roles, one for states, one for config, one for..." -- soon you have 15 "single" sources. volta limits it to 4 DSL files plus environment variables. That is it.

### Mistake 4: Not versioning the SSOT

volta's DSL files include `version` fields. When the schema changes, the version increments. Old code can detect version mismatches and fail loudly instead of silently misinterpreting the new format.
