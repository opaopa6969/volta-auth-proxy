# SLA（Service Level Agreement：サービスレベル契約）

[English version](sla.md)

---

## これは何？

SLA（Service Level Agreement）は、サービスプロバイダーと顧客の間の正式な契約で、保証されるサービスレベルを定義します -- 主に稼働時間ですが、応答時間、サポート可用性、プロバイダーが保証を満たせなかった場合の対応も含みます。

ピザのデリバリー保証のようなものです。「30分以内にお届け、遅れたら無料」。ピザ店はコミットメント（30分）を行い、失敗した場合のペナルティ（返金）があります。SLA も同じです：「99.9% の稼働時間、未達の場合はサービスクレジット」。

SLA は理想的な目標ではありません。金銭的ペナルティを伴う契約上の義務です。エンタープライズ顧客が 99.9% 稼働時間の SLA で契約を結ぶとき、その数字が達成されることを期待します。達成されなければ、補償を期待し -- 契約を解除する可能性があります。

---

## なぜ重要？

エンタープライズ顧客にとって、SLA は最低条件です。彼らの製品はあなたのサービスに依存しています。認証システムがダウンすれば、ユーザーはログインできず、ビジネスは止まり、損失が出ます。SLA は以下を提供します：

- **予測可能性**：保証された稼働時間を基に計画できる
- **説明責任**：失敗した場合、定義された結果がある
- **法的保護**：SLA は契約の一部であり、法的に強制可能

サービスプロバイダーにとって、SLA は諸刃の剣です。顧客との信頼を構築しますが、インフラ、監視、オンコールチーム、冗長性への大幅な投資を必要とする義務も生み出します。

## どう動くのか？

### 「ナイン」を理解する

稼働時間は「ナイン」で測定されます。ナインが1つ増えるごとに、達成は指数関数的に困難になります：

```
  稼働時間   年間ダウンタイム     月間ダウンタイム
  ────────   ──────────────────   ──────────────────
  99%        3.65 日              7.3 時間
  99.9%      8.77 時間            43.8 分
  99.95%     4.38 時間            21.9 分
  99.99%     52.6 分              4.38 分
  99.999%    5.26 分              26.3 秒
```

ほとんどの SaaS 製品は 99.9%（「スリーナイン」）を提供します。銀行や重要インフラは 99.99%（「フォーナイン」）を目指します。ファイブナインは実質的に「決してダウンしない」であり、莫大な投資が必要です。

### SLA の構成要素

| 要素 | 説明 | 例 |
|------|------|-----|
| **稼働時間保証** | サービスが利用可能な時間の割合 | 月間 99.9% 稼働時間 |
| **測定ウィンドウ** | 稼働時間の計算方法 | 月次ローリングウィンドウ |
| **除外事項** | ダウンタイムに数えないもの | 計画的メンテナンス、顧客起因の問題 |
| **サービスクレジット** | SLA 違反時の補償 | 目標を 0.1% 下回るごとに 10% クレジット |
| **応答時間 SLA** | サポートの応答速度 | P1: 15分、P2: 1時間、P3: 4時間 |
| **解決時間 SLA** | 問題の修正速度 | P1: 4時間、P2: 24時間 |

### SLA の適用方法

```
  月末
      │
      ▼
  実際の稼働時間を計算
  （総分数 - ダウンタイム分数）/ 総分数
      │
      ▼
  SLA 目標と比較
      │
  ┌───┴───┐
  達成    未達
  │       │
  ▼       ▼
  何も     サービスクレジットを計算
  起きない     │
               ▼
           次回請求書に
           クレジット適用
           （または返金）
```

### SLA を提供するために必要なもの

99.9% の SLA を提供するだけでも以下が必要です：

- **冗長性**：単一障害点なし。複数サーバー、複数アベイラビリティゾーン
- **監視**：顧客が気づく前にダウンを検知。数秒ごとのヘルスチェック
- **オンコールローテーション**：24時間365日、問題修正のために連絡可能な人員
- **インシデント対応**：一般的な障害モードの文書化された手順
- **自動フェイルオーバー**：コンポーネント障害時に自己修復するシステム
- **負荷テスト**：顧客が限界に達する前に限界を把握
- **カオスエンジニアリング**：回復力を証明するために意図的に壊す

## volta-auth-proxy ではどう使われている？

volta-auth-proxy は SLA を**提供していません**。Phase 1 のスコープに沿った意図的な決定です。

### SLA がない理由（現時点）

| 理由 | 詳細 |
|------|------|
| **シングルノードアーキテクチャ** | volta はシングルプロセスで動作。冗長性なし = SLA なし |
| **SQLite データベース** | 単一ファイル DB はマルチノードレプリケーション非対応 |
| **オンコールチームなし** | 意味のある SLA には 24/7 の人的サポートが必要 |
| **ターゲットユーザー** | インディーハッカーや社内ツールは時折のダウンタイムを許容 |
| **セルフホスト** | 稼働時間を制御するのは顧客であり volta ではない |

### セルフホストの SLA モデル

ソフトウェアがセルフホストの場合、SLA の構図は大きく変わります：

```
  マネージド SaaS（Auth0）：
    Auth0 ──► 顧客への SLA 保証 ──► 99.99% 稼働時間
    Auth0 が稼働時間に責任を持つ。

  セルフホスト（volta）：
    顧客 ──► 自社インフラで volta を運用 ──► 顧客の SLA
    顧客の運用チームが稼働時間に責任を持つ。
    volta はソフトウェア品質を提供し、稼働時間保証は提供しない。
```

volta の責任は信頼性の高いソフトウェアを出荷すること。顧客の責任はそれを信頼性高く運用すること。これは PostgreSQL、Nginx、その他のセルフホストインフラと同じモデルです。

### volta の信頼性への取り組み

正式な SLA はなくても、volta は信頼性を重視して設計されています：

- **高速起動（~200ms）**：プロセスがクラッシュしても、ほぼ即座に再起動
- **ヘルスチェックエンドポイント**：`GET /healthz` が `{"status":"ok"}` を返し、オーケストレーターの監視に対応
- **グレースフルシャットダウン**：処理中のリクエストが完了してからプロセス終了
- **SQLite WAL モード**：クラッシュ時もデータベースの一貫性を維持
- **最小限の依存関係**：故障しうるものが少ない

### Phase 3 SLA ロードマップ

volta が Phase 3（エンタープライズ）に移行する際、SLA には以下が必要：

- PostgreSQL バックエンド（マルチノードレプリケーション）
- ロードバランサー背後のマルチインスタンスデプロイ
- 自動ヘルスチェックとフェイルオーバー
- オンコールエンジニアリングサポート
- リアルタイム稼働時間レポートのステータスページ
- 定義されたサービスクレジット構造

## よくある間違いと攻撃

### 間違い1：達成できない SLA を提供する

SLA は契約です。99.9% の稼働時間を保証しながら、監視なしの単一サーバーで運用すれば、SLA に違反し、クレジットを支払い、顧客の信頼を失います。裏付けるインフラが整うまで SLA を提供してはいけません。

### 間違い2：細則を読まない

SLA の除外事項は重要です。「月最大8時間の計画的メンテナンスウィンドウを除く 99.9% 稼働時間」は「99.9% 稼働時間」とは大きく異なります。常に測定方法を確認しましょう。

### 間違い3：SLA と SLO と SLI を混同する

| 用語 | 内容 | 見る人 |
|------|------|--------|
| **SLI**（サービスレベル指標） | 実際の測定値（例：レイテンシ、エラー率） | エンジニアリング |
| **SLO**（サービスレベル目標） | 内部目標（例：「99.95% 稼働時間目標」） | エンジニアリング + 経営 |
| **SLA**（サービスレベル契約） | ペナルティ付き外部契約（例：「99.9% 未達ならクレジット」） | 顧客 |

SLA は SLO より低く設定し、SLO は実際の達成値より低く設定すべきです。これが安全バッファになります。

### 間違い4：ダウンタイムをバイナリに捉える

「サービスは稼働していた」は単純すぎます。稼働していたが応答に 300ms ではなく 30秒かかっていたら？90% のユーザーには稼働していたが 10% にはダウンしていたら？SLA は「利用可能」の意味を正確に定義しなければなりません。
