# スライディングウインドウ

[English version](sliding-window.md)

---

スライディングウインドウは、適切なたとえ話を聞くまでは複雑に聞こえる概念の1つです。でも聞けば明白になります。

---

## 図書館の本のたとえ

図書館で本を借りたとします。図書館には2つの貸出方針があり得ます：

**方針A：絶対タイムアウト**
「本は14日間で返却してください。以上。13日目に読み始めたばかりでも関係ありません。14日目に返してください。」

**方針B：スライディングウインドウ**
「本は最後に開いてから14日後に返却してください。章を読むたびに、返却期限が14日先に延びます。14日間読まなかったら、もう読まないものとみなして返却をお願いします。」

方針Aは予測可能ですが不満が出ます。返却期限のときに章の途中かもしれません。方針Bは寛容です -- 積極的な利用を報酬にします。読み続ける限り本はあなたのもの。でも忘れたら図書館に戻ります。

volta-auth-proxyはセッションに方針Bを使います。これがスライディングウインドウです。

## voltaの8時間スライディングウインドウの仕組み

ユーザーがログインすると、voltaは現在から8時間後に期限切れになるセッションを作成します：

```
  09:00  ユーザーがログイン。
         セッション作成。
         期限切れ: 17:00（今から8時間後）
```

ユーザーが何かをするたび（ページの読み込み、API呼び出し、リンクのクリック）、voltaのForwardAuthハンドラーがセッションを「タッチ」して有効期限を先に押します：

```
  09:00  ログイン。           期限切れ 17:00
  10:30  /dashboard 読み込み。期限切れ 18:30（今から8時間）
  12:00  /settings 読み込み。 期限切れ 20:00（今から8時間）
  14:00  /team 読み込み。     期限切れ 22:00（今から8時間）
  14:15  /billing 読み込み。  期限切れ 22:15（今から8時間）
```

インタラクションのたびに時計がリセットされます。アクティブなユーザーはセッションタイムアウトに中断されることなく一日中作業できます。セッションが死ぬのは、ユーザーが連続8時間インタラクションを停止したときだけです。

```
  14:15  最後のインタラクション。
  14:16  ユーザーが帰宅。
  ...
  22:15  セッション期限切れ。誰も更新しなかった。
  22:16  次の訪問：「もう一度ログインしてください。」
```

## スライディングウインドウ vs 絶対タイムアウト

これが2つの基本的なタイムアウト戦略です。多くのシステムは一方または両方を使います：

```
  スライディングウインドウ：
  ─────────────────────────────────────────────►
  ログイン  アクティビティ  アクティビティ  アクティビティ   8h無操作→期限切れ
  09:00    10:30          12:00          14:00          22:00

  ウインドウは各インタラクションで前にスライドする。


  絶対タイムアウト：
  ─────────────────────────────────────────────►
  ログイン                                   常に17:00で期限切れ
  09:00                                      17:00
  （ユーザーがどんなにアクティブでも）

  期限はログイン時に固定。
```

| 観点 | スライディングウインドウ | 絶対タイムアウト |
|------|----------------------|----------------|
| 期限切れのタイミング | **無操作**の8時間後 | **ログイン**の8時間後 |
| アクティブなユーザー | 一日中作業可能 | 固定時間でログアウトされる |
| 非アクティブなユーザー | 無操作8時間後に期限切れ | 8時間後に関係なく期限切れ |
| UX | 良い（作業中の予期しないログアウトなし） | 悪い（アクティブ使用中のログアウト） |
| セキュリティ | やや緩い（常にアクティブなら「永久に」生存可能） | 厳密に制限（最大寿命を保証） |

### スライディングがUXに良い理由

voltaで保護されたWikiアプリケーションで作業する開発者を想像してください。ドキュメントを書いています。段落を打ち、プレビューし、別の段落を打ち、保存する。数分ごとにアプリとインタラクションします。

スライディングウインドウなら、セッションは延長され続けます。作業セッション中にログインプロンプトが表示されることはありません。フローは中断されません。

絶対タイムアウトなら、ログインからぴったり8時間後に突然ログアウトされます。段落を書いている途中かもしれません。フォームデータが失われるかもしれません。再度ログインし、ページに戻り、作業が保存されていることを祈る必要があります。

日常的に使用するSaaSツールにとって、スライディングウインドウが正しい選択です。

### セキュリティの考慮事項

スライディングウインドウの懸念は、ユーザー（またはセッションを盗んだ攻撃者）がアクティブに保てば、セッションが理論的に永遠に生存できることです。実際にはvoltaは以下で緩和しています：

1. **最大同時セッション数**（ユーザーあたり5）-- アクティブセッション数を制限
2. **セッション無効化** -- ユーザーがアクティブセッションを確認して終了可能
3. **IP/ユーザーエージェント追跡** -- 不審なセッションを識別可能
4. **管理者セッション無効化** -- 管理者が任意のユーザーのセッションを無効化可能

voltaは現在、スライディングウインドウの上に絶対タイムアウトの上限を強制していません。大半のSaaSアプリケーションでは、同時セッション制限付きスライディングウインドウが良いバランスを提供します。セキュリティ要件が絶対的な上限を求める場合（例：「アクティビティに関わらずセッションは24時間以上存続不可」）、コード変更として追加可能です -- [absolute-timeout.ja.md](absolute-timeout.ja.md)参照。

## 実装

voltaのコードでは、スライディングウインドウは`touchSession()`で実装されています：

```java
// ForwardAuthがセッションを検証するたびに：
store.touchSession(session.id(), Instant.now().plusSeconds(config.sessionTtlSeconds()));
// これがexpires_atをSESSION_TTL_SECONDS（デフォルト：28800 = 8時間）分先に押す
```

背後のSQLはシンプルなUPDATEです：

```sql
UPDATE sessions SET expires_at = $2, last_active_at = NOW() WHERE id = $1
```

すべての認証済みリクエストがセッションを「タッチ」します。タッチは軽量です -- インデックス付き主キーに対する単一のUPDATE文。ForwardAuthチェックにほぼ無視できるオーバーヘッドしか追加しません。

## ウインドウサイズの選択

voltaのデフォルトは8時間（`SESSION_TTL_SECONDS=28800`）で、典型的な勤務日に合わせています：

- ユーザーが午前9時にログイン
- 午後5時まで作業（8時間のアクティビティ）
- 帰宅
- セッションが午前1時に期限切れ（午後5時頃の最後のインタラクションから8時間後）
- 翌朝、再度ログイン

異なるウインドウが必要なら、`SESSION_TTL_SECONDS`環境変数を変更してください。短いウインドウ（例：銀行向け1時間）はより安全ですが頻繁なログインが必要。長いウインドウ（例：24時間）はより便利ですがユーザーが離れた後もセッションが長く持続します。
