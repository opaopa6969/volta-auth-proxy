# SOC 2（Service Organization Control 2）

[English version](soc2.md)

---

## これは何？

SOC 2 はアメリカ公認会計士協会（AICPA）が作成したセキュリティ監査フレームワークです。企業が顧客データを安全に取り扱っていることを認証します。企業が「SOC 2 準拠です」と言うとき、独立した監査人がシステムが特定のセキュリティ、可用性、プライバシー基準を満たしていることを検証したことを意味します。

レストランの保健所検査のようなものです。レストランは素晴らしい料理を作っているかもしれませんが、保健所の検査官が来て、厨房、冷蔵庫、手洗い場をチェックし、報告書を書きます。合格は「この厨房は安全基準を満たしている」ことを意味します。SOC 2 も同じです -- ただし、データを扱うソフトウェア会社が対象です。

SOC 2 は法律ではなく、任意の基準です。しかし実際には、エンタープライズ顧客（銀行、医療企業、大企業）は契約を結ぶ前にベンダーに SOC 2 認証を要求します。エンタープライズ向け販売の入場券です。

---

## なぜ重要？

SaaS 企業にとって、SOC 2 は「スタートアップにスタートアップが売る」と「企業にソフトウェアを売る」の境界線です。SOC 2 がないと：

- エンタープライズの調達チームが製品を承認しない
- セキュリティレビュー質問票に合格できない
- 政府や規制産業の契約に入札できない
- 大規模顧客がデータを信頼しない

裏を返せば：SOC 2 認証は高額で時間がかかります。通常 $50,000〜$200,000 のコストと、初回監査に向けて6〜12ヶ月の準備が必要です。初期段階のスタートアップがスキップする理由です。

## どう動くのか？

### 5つの信頼サービス基準

SOC 2 は5つの原則を中心に構築されています（全5つは必須ではありません）：

| 基準 | カバー範囲 | 必須？ |
|------|-----------|--------|
| **セキュリティ** | 不正アクセスからの保護 | 常に（必須） |
| **可用性** | システムの稼働時間とパフォーマンス | オプション |
| **処理の完全性** | データが正しく完全に処理される | オプション |
| **機密性** | 機密データが保護される | オプション |
| **プライバシー** | 個人情報がポリシーに従って扱われる | オプション |

### 監査プロセス

```
  ステップ1: 含める基準を決定
            （セキュリティは必須、他はオプション）
              │
              ▼
  ステップ2: コントロールを実装
            - アクセス管理（誰が何にアクセスできるか）
            - 暗号化（保存時と転送時のデータ）
            - ログ記録と監視
            - インシデント対応手順
            - 従業員セキュリティ研修
            - 変更管理（コードが本番に至る方法）
            - ベンダー管理（依存関係）
              │
              ▼
  ステップ3: 一定期間コントロールを運用
            ┌─────────────────────────────┐
            │  Type I:  ある時点           │
            │  「今日、コントロールが存在」  │
            │  （スナップショット、容易）    │
            ├─────────────────────────────┤
            │  Type II: 一定期間にわたって  │
            │  「6-12ヶ月間、コントロールが  │
            │   継続的に機能した」          │
            │  （はるかに困難、はるかに     │
            │   顧客にとって価値が高い）    │
            └─────────────────────────────┘
              │
              ▼
  ステップ4: 独立監査人がレビュー
            - 証拠を調査
            - コントロールをテスト
            - 従業員にインタビュー
              │
              ▼
  ステップ5: 監査人が SOC 2 レポートを発行
            - 合格：「コントロールは有効」
            - 不合格：「これらのコントロールにギャップあり」
```

### 実際のコントロール例

| コントロール領域 | 要件の例 |
|-----------------|---------|
| アクセス制御 | 全本番システムで MFA 必須 |
| 変更管理 | すべてのコード変更にピアレビュー必須 |
| 暗号化 | 保存時（AES-256）と転送時（TLS 1.2+）のデータ暗号化 |
| ログ記録 | 顧客データへのすべてのアクセスをログに記録し90日間保持 |
| インシデント対応 | セキュリティインシデントを文書化し SLA 内に解決 |
| ベンダー管理 | サードパーティライブラリの脆弱性レビュー |
| 従業員オンボーディング | 身元調査、30日以内のセキュリティ研修 |

## volta-auth-proxy ではどう使われている？

volta-auth-proxy は SOC 2 認証を**取得していません**。これは見落としではなく、意図的な Phase 1 の決定です。

### volta が SOC 2 をスキップする理由（現時点）

| 理由 | 詳細 |
|------|------|
| **ターゲットユーザー** | インディーハッカーや初期スタートアップは SOC 2 を要求しない |
| **コスト** | $50K〜$200K は Phase 1 製品には正当化できない |
| **オーバーヘッド** | SOC 2 コントロールは高速な反復と相容れないプロセスオーバーヘッドを追加 |
| **セルフホスト** | volta はセルフホスト -- 顧客が自分のデータを制御 |

### セルフホストの利点

volta は顧客のインフラストラクチャ上で動作するため、多くの SOC 2 の懸念は顧客側に移ります：

```
  マネージド SaaS（Auth0 等）：
    顧客データ ──► ベンダーのサーバー ──► ベンダーが SOC 2 必要

  セルフホスト（volta）：
    顧客データ ──► 顧客のサーバー ──► 顧客の SOC 2 がカバー
```

顧客の既存の SOC 2 認証が、データベースやWebサーバーをカバーするのと同様に、社内ツールとしての volta をカバーできます。

### volta が代わりに行っていること

正式な SOC 2 はなくても、volta は SOC 2 コントロールに沿ったセキュリティベストプラクティスに従っています：

- **転送時の暗号化**：すべての認証フローが HTTPS/TLS を使用
- **セッションセキュリティ**：HttpOnly, Secure, SameSite クッキー
- **CSRF 対策**：トークンベース + SameSite 防御
- **入力検証**：すべての入力を検証・サニタイズ
- **ログ記録**：認証イベントを監査証跡としてログ
- **最小限の依存関係**：サードパーティライブラリが少ない = サプライチェーンリスクが小さい

### Phase 3 ロードマップ

SOC 2 認証は Phase 3（エンタープライズ対応）で計画されています。その時点で volta に必要なもの：

- 正式なアクセス制御ポリシー
- 継続的な監視
- インシデント対応ランブック
- ペネトレーションテスト（[penetration-test.md](penetration-test.md) 参照）
- 監査ログの保持
- 独立した監査人

## よくある間違いと攻撃

### 間違い1：SOC 2 があれば安全だと思う

SOC 2 はコントロールが存在し、遵守されていることを認証します。コントロールが優れているとも、ソフトウェアに脆弱性がないとも意味しません。SOC 2 認証を持つ企業でも侵害は起きます。

### 間違い2：SOC 2 を早すぎる段階で追求する

顧客がインディーハッカーやスタートアップなら、$100K を SOC 2 に費やすのは無駄金です。まず製品を作る。エンタープライズ顧客が実際に求めたときに SOC 2 を取る。

### 間違い3：SOC 2 を一回きりのイベントとして扱う

SOC 2 Type II は継続的なコンプライアンスを要求します。コントロールは監査シーズンだけでなく、毎日機能しなければなりません。ツール、プロセス、研修への継続的な投資が必要です。

### 間違い4：SOC 2 を他の基準と混同する

SOC 2 は ISO 27001（国際セキュリティ管理）ではありません。HIPAA（医療データ）でもありません。PCI DSS（決済カードデータ）でもありません。異なる顧客が異なる認証を要求します。
