# SPA（シングルページアプリケーション）

[English version](spa.md)

---

## これは何？

シングルページアプリケーション（SPA）とは、1つの[HTML](html.md)ページを読み込み、[JavaScript](javascript.md)を使ってコンテンツを動的に書き換えるWebアプリケーションです。[サーバー](server.md)から新しいページを読み込むことはありません。SPAでリンクをクリックしても、ブラウザは従来の意味で新しいURLに移動しません -- JavaScriptがクリックを傍受し、[API](api.md)からデータを取得し、ページをその場で更新します。

パラパラ漫画と絵はがきの束の違いに似ています。従来のウェブサイトは絵はがきの束のようなもので、各ページが別々のカードで、新しいものを見るには今のカードを捨てて次のカードを取り上げます。SPAはパラパラ漫画のようなもので、1冊の本があり、ページをめくることで内容が変わりますが、本を置くことはありません。「入れ物」は同じまま、中身だけが変わります。

一般的なSPAフレームワークにはReact、Vue、Angular、Svelteがあります。すべて同じ原則に従います：ブラウザが1つのHTMLファイルと大きなJavaScriptバンドルをダウンロードし、それ以降はJavaScriptがすべてを処理します -- ルーティング、レンダリング、API呼び出し、状態管理。

---

## なぜ重要なのか？

SPAは現代のWebアプリケーションフロントエンドを構築する主流の方法になっています。volta-auth-proxyにとって理解が重要な理由：

- **SPAはvoltaの[Internal API](internal-api.md)の主要な消費者** -- JavaScriptで`/api/v1/users/me`、`/api/v1/tenants`などを呼ぶ
- **SPAは認証の処理方法が異なる** -- サーバーサイドの[セッション](session.md)を直接使えず、[JWT](jwt.md)と[Cookie](cookie.md)に依存する
- **SPAは独自のセキュリティ課題を生む** -- [XSS](xss.md)攻撃でブラウザからトークンが盗まれる可能性、[CORS](cors.md)を正しく設定する必要がある
- **SPAにはvolta-sdk-jsが必要** -- [SDK](sdk.md)が自動JWT更新を処理する。SPAには不可欠

voltaを認証に使う[ダウンストリームアプリ](downstream-app.md)を構築しているなら、おそらくSPAです。

## どう動くのか？

### 従来のウェブサイト vs SPA

```
  従来型（マルチページアプリケーション）:

  ブラウザ                         サーバー
  ──────                           ──────
  GET /dashboard    ─────────────>  完全なHTMLをレンダリング
                    <─────────────  <html>...</html>
  （ページ全体が読み込まれる）

  「設定」をクリック ─────────────>  完全なHTMLをレンダリング
                    <─────────────  <html>...</html>
  （再びページ全体が読み込まれる）


  SPA（シングルページアプリケーション）:

  ブラウザ                         サーバー
  ──────                           ──────
  GET /             ─────────────>  index.html + app.jsを返す
                    <─────────────  <html><script src="app.js">
  （ページは一度だけ読み込まれる）

  「設定」をクリック ─────────────>  GET /api/v1/settings (JSON)
  （JSが傍受）       <─────────────  {"theme":"dark","lang":"ja"}
  JSがDOMを更新      （ページ再読み込みなし）

  「プロフィール」   ─────────────>  GET /api/v1/users/me (JSON)
  をクリック（JSが傍受）<────────────  {"name":"Taro","role":"ADMIN"}
  JSがDOMを更新      （ページ再読み込みなし）
```

### SPAアーキテクチャ

```
  ┌──────────────────────────────────────────────────────┐
  │                    ブラウザ                            │
  │                                                       │
  │  ┌─────────────────────────────────────────────────┐  │
  │  │              SPA（JavaScript）                   │  │
  │  │                                                  │  │
  │  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │  │
  │  │  │ ルーター  │  │  状態    │  │ コンポーネント │  │  │
  │  │  │ /dash    │  │ 管理     │  │  Dashboard    │  │  │
  │  │  │ /settings│  │ (Redux/  │  │  Settings     │  │  │
  │  │  │ /profile │  │  Pinia)  │  │  Profile      │  │  │
  │  │  └──────────┘  └──────────┘  └──────────────┘  │  │
  │  │                      │                           │  │
  │  │               fetch("/api/...")                   │  │
  │  └──────────────────────┼──────────────────────────┘  │
  │                         │                              │
  └─────────────────────────┼──────────────────────────────┘
                            │ HTTP (JSON)
                            ▼
  ┌──────────────────────────────────────────────────────┐
  │              volta-auth-proxy (API)                   │
  │  /api/v1/users/me                                    │
  │  /api/v1/tenants                                     │
  │  /api/v1/admin/members                               │
  └──────────────────────────────────────────────────────┘
```

### クライアントサイドルーティング

従来のウェブサイトでは`/settings`に移動するとブラウザがサーバーに新しいリクエストを送ります。SPAではURLは変わりますが、ブラウザは新しいページのリクエストを送りません：

```
  URL: /dashboard           URL: /settings           URL: /profile
  ┌──────────────────┐      ┌──────────────────┐     ┌──────────────────┐
  │  ┌────────────┐  │      │  ┌────────────┐  │     │  ┌────────────┐  │
  │  │  ナビバー   │  │      │  │  ナビバー   │  │     │  │  ナビバー   │  │
  │  ├────────────┤  │      │  ├────────────┤  │     │  ├────────────┤  │
  │  │            │  │      │  │            │  │     │  │            │  │
  │  │ ダッシュ   │  │ ───> │  │ 設定      │  │ ──> │  │ プロフィール│  │
  │  │ ボード内容 │  │      │  │ 内容      │  │     │  │ 内容       │  │
  │  │            │  │      │  │            │  │     │  │            │  │
  │  ├────────────┤  │      │  ├────────────┤  │     │  ├────────────┤  │
  │  │  フッター   │  │      │  │  フッター   │  │     │  │  フッター   │  │
  │  └────────────┘  │      │  └────────────┘  │     │  └────────────┘  │
  └──────────────────┘      └──────────────────┘     └──────────────────┘

  同じシェル（ナビバー、フッター）。真ん中のコンテンツだけが変わる。
  ページは再読み込みされない。JavaScriptがコンテンツを入れ替える。
```

### SPA vs MPA vs SSR

| アプローチ | レンダリング | ページ読み込み | 例 |
|-----------|------------|-------------|---|
| MPA（マルチページアプリ） | サーバーが完全なHTMLをレンダリング | ナビゲーション時に全ページ再読み込み | volta-auth-proxy管理ページ（jte） |
| SPA（シングルページアプリ） | ブラウザがJSでレンダリング | 再読み込みなし、JSがDOMを更新 | React/Vueダッシュボードアプリ |
| SSR（サーバーサイドレンダリング） | サーバーがレンダリング、JSがハイドレート | 初回はサーバーレンダリング、その後SPA的 | Next.js, Nuxt.js |

## volta-auth-proxy ではどう使われている？

### volta自体はSPAではない

volta-auth-proxy自身のページ（ログイン、テナントセレクター、招待承諾）は[jte](jte.md) [テンプレート](template.md)を使ったサーバーレンダリングです。これは意図的です：

- ログインページはJavaScriptなしで動く必要がある（アクセシビリティ、信頼性）
- サーバーレンダリングページにはクライアントサイドレンダリングによる[XSS](xss.md)リスクがない
- 認証フローは[リダイレクト](redirect-uri.md)を伴い、従来のページ読み込みの方がうまく動く

### SPAはvoltaの主要な消費者

voltaの背後に位置するアプリケーション（[ダウンストリームアプリ](downstream-app.md)）は通常SPAです。典型的なアーキテクチャ：

```
  ┌──────────────────────────────────────────────────────────┐
  │                        ブラウザ                           │
  │                                                           │
  │  ┌───────────────────────────────────────────────────┐   │
  │  │        あなたのSPA（React / Vue / etc.）           │   │
  │  │                                                    │   │
  │  │  import { VoltaClient } from 'volta-sdk-js';      │   │
  │  │  const volta = new VoltaClient();                  │   │
  │  │                                                    │   │
  │  │  // SDKがJWT更新を自動処理                          │   │
  │  │  const user = await volta.getUser();               │   │
  │  │  const members = await volta.fetch('/api/members');│   │
  │  └───────────────────────────────────────────────────┘   │
  │         │                           │                     │
  │         │ /auth/refresh             │ /api/v1/users/me   │
  └─────────┼───────────────────────────┼─────────────────────┘
            │                           │
            ▼                           ▼
  ┌──────────────────────────────────────────────────────────┐
  │                    volta-auth-proxy                       │
  │  セッションCookieでユーザーを検証                         │
  │  短命JWT（5分）を発行                                    │
  │  API経由でユーザー情報を返す                              │
  └──────────────────────────────────────────────────────────┘
```

### voltaとのSPA認証フロー

```
  1. ユーザーがyour-app.example.comを開く
     ├── SPAが読み込まれる（index.html + app.js）
     └── SPAがvolta /api/v1/users/meを呼ぶ
              │
              ▼
  2. セッションCookieなし → voltaが401を返す
              │
              ▼
  3. volta-sdk-jsが401を検知 → voltaログインにリダイレクト
              │
              ▼
  4. ユーザーがGoogle OIDCで認証
              │
              ▼
  5. voltaがセッションCookieを作成 → SPAにリダイレクトバック
              │
              ▼
  6. SPAが再読み込み、/api/v1/users/meを呼ぶ
     ├── セッションCookieが有効 → voltaがユーザーデータを返す
     └── SPAが認証済みUIをレンダリング
              │
              ▼
  7. 5分後、JWTが期限切れ
     ├── volta-sdk-jsが401を検知
     ├── /auth/refreshを呼ぶ（セッションCookieはまだ有効）
     ├── 新しいJWTを取得
     └── 元のリクエストを透過的にリトライ
```

### SPAとForwardAuthパターン

SPAがTraefikの[ForwardAuth](forwardauth.md)経由で提供される場合、SPAのバックエンドへのすべてのリクエストがvoltaを通ります：

```
  SPA → fetch("/api/data") → Traefik → ForwardAuth → voltaがセッションチェック
                                                      │
                                              ┌───────┴───────┐
                                              │ 有効なセッション │
                                              │ X-Volta-*     │
                                              │ ヘッダーを追加  │
                                              └───────┬───────┘
                                                      │
                                              あなたのバックエンドAPI
                                              がX-Volta-*ヘッダーを読む
```

## よくある間違いと攻撃

### 間違い1：JWTをlocalStorageに保存する

SPAはJWTをどこかに保存する必要があります。`localStorage`はページ再読み込みを跨いで永続化するため、つい選びがちです。しかし`localStorage`はページ上のあらゆるJavaScript（[XSS](xss.md)攻撃で注入されたスクリプトを含む）からアクセス可能です。volta-sdk-jsはJWTをメモリにのみ保存し、必要に応じて[セッション](session.md)[Cookie](cookie.md)経由で更新します。

### 間違い2：トークン更新を処理しない

JWTを取得して永遠に使い続けるSPAは、JWTの期限が切れると壊れます（voltaでは5分）。SPAはvolta-sdk-js（自動更新を処理する）を使うか、独自の更新ロジックを実装する必要があります。

### 攻撃1：SPAにおけるXSS

SPAはアプリケーション全体がJavaScriptとして実行されるため、[XSS](xss.md)に対して特に脆弱です。攻撃者が`<script>`タグを注入できると：

- SPAがアクセスできるあらゆるデータを読める
- 認証済みユーザーとしてAPI呼び出しができる
- メモリ内のJWTを盗める

防御：すべてのユーザー入力をサニタイズ、Content Security Policyヘッダーを使用、JWTを短命にする（voltaの5分期限が被害の時間枠を限定）。

### 間違い3：CORSの設定ミス

SPAはクロスオリジンAPI呼び出しを行います（SPAは`app.example.com`、APIは`auth.example.com`）。[CORS](cors.md)が正しく設定されていないと、ブラウザがリクエストをブロックします。CORSの設定が緩すぎると（`Access-Control-Allow-Origin: *`）、どのウェブサイトでもユーザーに代わってAPI呼び出しができてしまいます。

### 間違い4：ディープリンクルーティングの失敗

ユーザーが`/dashboard/settings`をブックマークして後でアクセスすると、サーバーは`/dashboard/settings`へのリクエストを受け取ります。サーバーが`/`でのみ`index.html`を提供している場合、ユーザーは404を受け取ります。サーバーはすべてのSPAルートに対して`index.html`を返すよう設定し、クライアントサイドルーターにパスを処理させる必要があります。
