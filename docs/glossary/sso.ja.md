# SSO (Single Sign-On / シングルサインオン)

## これは何？

シングルサインオン（SSO）とは、ユーザーが一度ログインするだけで、それぞれのアプリに再度ログインすることなく複数のアプリケーションにアクセスできる仕組みです。1 回のログインで、多くのアプリを利用可能。

Gmail にログインした後、Google ドライブ、YouTube、Google カレンダーをパスワードなしで開いたことがあれば、それが SSO です。1 回の Google ログインで Google の全サービスにアクセスできるようになっています。

```
  SSO なし:                       SSO あり:

  Wiki  --> ログイン               Wiki  --> ログイン済み！
  Admin --> また ログイン           Admin --> ログイン済み！
  Chat  --> また ログイン           Chat  --> ログイン済み！
  CRM   --> また ログイン           CRM   --> ログイン済み！

  4 つのパスワードを覚える         1 回のログインで 4 アプリ利用可能
  4 つのログインフォームに入力      ログインフォームは合計 1 つ
```

## なぜ重要？

**ユーザーにとって:**
- 手間が少ない。1 回のログインでどこでもアクセス。
- 管理するパスワードが減る（忘れにくく、使い回しも減る）。
- 特に内部ツールの多い組織で、体験が向上する。

**セキュリティにとって:**
- 認証の一元化により、強力なパスワード、MFA、アカウントポリシーを一箇所で適用できる。
- 社員が退職したとき、1 つのアカウントを無効にするだけで全てのアクセスを遮断できる。
- パスワードが減ることでパスワードの使い回しが減り、データ漏洩で侵害されるアカウントも減る。

**IT チームにとって:**
- 多くの代わりに 1 つの ID システムを維持するだけ。
- コンプライアンスが容易（誰が何にアクセスできるかを一箇所で追跡）。
- オンボーディングとオフボーディングがシンプルに。

## どう動くの？

SSO は全てのアプリケーションが信頼する中央の認証サービス（Identity Provider / IdP）に依存します。主に 2 つのパターンがあります:

### IdP 起点の SSO

ユーザーが IdP（Okta のダッシュボードや Google Workspace のランチャーなど）から始めます。アプリのアイコンをクリックすると、IdP が署名済みのアサーション(証明)をアプリケーションに直接送ります。

```
  ユーザー         IdP (Okta)          アプリ (Wiki)
  |               |                   |
  | Okta ポータルに |                   |
  | ログイン済み    |                   |
  |               |                   |
  | "Wiki" クリック|                   |
  |-------------->|                   |
  |               |                   |
  |               | SAML アサーション   |
  |               | を Wiki に送信     |
  |               |------------------>|
  |               |                   |
  |               |    Wiki が有効な    |
  |               |    アサーションを   |
  |               |    確認            |
  |               |                   |
  |   Wiki にリダイレクト、ログイン済み  |
  |<----------------------------------|
```

### SP 起点の SSO（より一般的）

ユーザーがアプリケーション（Service Provider / SP）に直接アクセスします。アプリケーションがユーザーを認証のために IdP にリダイレクトします。ログイン後、IdP が本人確認の証拠とともにアプリケーションにリダイレクトで戻します。

```
  ユーザー         アプリ (Wiki)        IdP (Google)
  |               |                   |
  | Wiki にアクセス |                   |
  |-------------->|                   |
  |               |                   |
  |               | 未ログイン。        |
  |               | IdP にリダイレクト   |
  |               |                   |
  | Google ログインにリダイレクト        |
  |<--------------|                   |
  |---------------------------------->|
  |               |                   |
  | Google に既に  |                   |
  | ログイン済み?   |                   |
  | はい!（セッション存在）             |
  |               |                   |
  | 証拠付きでリダイレクトで戻す         |
  |<----------------------------------|
  |-------------->|                   |
  |               | 有効! ローカル     |
  |               | セッション作成     |
  |               |                   |
  | Wiki に        |                   |
  | アクセス可能    |                   |
  |<--------------|                   |
```

SSO の「魔法」は 2 番目のステップにあります: 「Google に既にログイン済み? はい!」 ユーザーは前のアプリにログインした際に IdP にセッションを持っているため、IdP は認証情報を再度求めません。新しい本人確認の証拠を発行してリダイレクトで戻すだけです。

### プロトコル

- **SAML 2.0** -- エンタープライズ標準。XML ベース。Okta、Azure AD、ADFS で使用。
- **OIDC** -- 現代の Web 標準。JSON/JWT ベース。Google、Auth0、Keycloak で使用。
- **ForwardAuth** -- リバースプロキシのパターン。Traefik や Nginx がリクエスト転送前に認証エンドポイントを呼び出す。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は Traefik との **ForwardAuth** パターンを使って、複数の下流アプリケーションで SSO を実現します。

**アーキテクチャ:**

```
  ブラウザ
  |
  | wiki.example.com/page へのリクエスト
  |
  v
  Traefik (リバースプロキシ)
  |
  | 転送前に volta の /auth/verify を呼び出す
  |
  v
  volta-auth-proxy
  |
  | セッション Cookie (__volta_session) を確認
  | セッション有効? JWT を発行、200 + ヘッダを返す
  | セッション無効? 401 を返す
  |
  v
  Traefik
  |
  | 200? リクエスト + X-Volta-* ヘッダを Wiki に転送
  | 401? ユーザーを volta のログインページにリダイレクト
  |
  v
  Wiki アプリがリクエストを ID ヘッダ付きで受信
```

**アプリ間での SSO の仕組み:**

1. ユーザーが `wiki.example.com` にアクセス。Traefik が volta に確認。セッションなし。ユーザーは volta のログインページにリダイレクト。
2. ユーザーが Google OIDC でログイン。volta がセッションを作成し `__volta_session` Cookie を設定。
3. ユーザーが wiki にリダイレクトで戻される。Traefik が volta に確認。セッション有効。Wiki が読み込まれる。
4. ユーザーが `admin.example.com` にアクセス。Traefik が volta に確認。**同じセッション Cookie が送信される**（全てのアプリが同じ Traefik インスタンスの背後にあり、Cookie のドメインがカバーしているため）。セッション有効。管理パネルが読み込まれる。

2 回目のログインは不要。これが SSO です。

```
  時間 --->

  Wiki 訪問   --> volta: セッションなし --> Google でログイン
                                          |
                                          v
                                     セッション作成
                                     Cookie: __volta_session=UUID
                                          |
  Wiki 読込  <-- volta: セッション有効 <---+
                                          |
  Admin 訪問 --> volta: セッション有効 <---+  (同じ Cookie!)
                                          |
  Admin 読込 <-- volta: セッション有効 <---+
                                          |
  Chat 訪問  --> volta: セッション有効 <---+  (まだ同じ Cookie!)
                                          |
  Chat 読込  <-- volta: セッション有効 <---+
```

**SSO 内のロールベースアクセス制御。** 全てのアプリが 1 つのログインを共有していても、volta はロールに基づいてアクセスを制限できます。`volta-config.yaml` ファイルでどのロールがどのアプリにアクセスできるかを定義します:

```yaml
apps:
  - id: app-wiki
    url: https://wiki.example.com
    allowed_roles: [MEMBER, ADMIN, OWNER]
  - id: app-admin
    url: https://admin.example.com
    allowed_roles: [ADMIN, OWNER]
```

MEMBER ロールのユーザーは wiki に SSO でアクセスできますが、admin はブロックされます。`/auth/verify` エンドポイントは 200 を返す前に、ユーザーのロールをアプリの `allowed_roles` と照合します。

**アプリに渡される ID 情報。** volta はヘッダを通じて下流のアプリケーションに ID データを渡します:
- `X-Volta-User-Id` -- ユーザーの UUID
- `X-Volta-Email` -- ユーザーのメールアドレス
- `X-Volta-Tenant-Id` -- 現在のテナント
- `X-Volta-Roles` -- ユーザーのロール
- `X-Volta-JWT` -- API コール用の短命 JWT

下流のアプリは独自のログインシステムを持つ必要がありません。毎回のリクエストで volta から ID を受け取ります。

## よくある間違い

**1. 各アプリに別々のログインを実装する。**
各アプリが独自のユーザーデータベースとログインフォームを持っていたら、SSO ではありません。「全てに同じパスワード」になっているだけで、もっと悪いです（1 つの漏洩で全アプリが侵害される）。

**2. セッション管理を一元化しない。**
SSO セッションが一元管理ではなくアプリごとに管理されると、1 つのアプリからログアウトしても他のアプリではログアウトされません。volta はセッションを 1 つのデータベースに一元化しています。セッションの失効は即座に全アプリに影響します。

**3. 全ての SSO ユーザーに同じ権限を与える。**
SSO は 1 回のログインであり、1 つの権限レベルではありません。常に SSO とロールベースのアクセス制御を組み合わせましょう。volta のアプリごとの `allowed_roles` 設定がこれを処理します。

**4. SSO のログアウトを忘れる。**
ユーザーが 1 つのアプリからログアウトしても他にまだログインしていたら、混乱を招き、セキュリティ上も問題になり得ます。volta のログアウトは中央のセッションを失効させ、次のリクエストで全アプリに影響します。

**5. SSO プロバイダからの ID 情報を検証しない。**
アプリがリバースプロキシから ID ヘッダを受け取るとき、リクエストが実際にプロキシを通ってきた場合にのみ信頼すべきです。アプリに直接アクセスできる場合（Traefik をバイパス）、攻撃者がヘッダを偽造できます。アプリがリバースプロキシの背後でのみアクセス可能であることを確認しましょう。

**6. SSO があれば MFA は不要だと思い込む。**
SSO はログインをより便利にします。素晴らしいことです。しかし、1 組の認証情報が侵害されると全てにアクセスされることも意味します。この影響範囲の拡大を補うために、SSO は常に多要素認証と組み合わせるべきです。
