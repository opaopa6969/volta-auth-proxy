# スタートアップ（サービス起動）

[English version](startup.md)

---

## これは何？

スタートアップとは、サービスが「停止中」から「リクエスト処理可能」になるまでの時間とプロセスです。Web アプリケーションを起動すると、設定の読み込み、データベース接続の初期化、ルートの設定、受信リクエストのリスニング開始が必要です。起動時間は、プロセス開始から最初のリクエストを処理できる瞬間までの時間です。

朝のレストラン開店のようなものです。シェフが到着し、オーブンをつけ、食材を準備し、予約を確認し、正面ドアを開ける。すべてが終わって初めてレストランは「開店」し、客を迎えられます。10分で開けるレストラン（フードトラック）もあれば、1時間かかるレストラン（高級レストラン）もあります。客が外で待っているときにこの差は重要です。

ソフトウェアでは、起動時間はミリ秒（軽量フレームワーク）から分（エンタープライズアプリケーションサーバー）まで幅があります。この差は開発体験、デプロイ速度、運用レジリエンスに大きな影響を与えます。

---

## なぜ重要？

起動時間は運用のほぼすべての面に影響します：

- **開発速度**：アプリケーションの起動に30秒かかるなら、コード変更のたびに30秒待ちます。1日の開発で合計すると数時間になります。
- **デプロイ速度**：ローリングデプロイでは、旧インスタンス停止前に新インスタンスが起動する必要があります。遅い起動 = 遅いデプロイまたはダウンタイム。
- **復旧時間**：サービスがクラッシュしたら再起動が必要です。200ms の再起動はユーザーに見えません。30秒の再起動は小規模な障害です。
- **スケーリング速度**：オートスケーリングは負荷増加時に新インスタンスを追加します。起動に30秒かかると、負荷スパイクに30秒遅れます。
- **コンテナオーケストレーション**：Kubernetes や Docker のヘルスチェックにはタイムアウトがあります。遅い起動はより長いタイムアウト設定を要し、準備完了を遅延させます。

## どう動くのか？

### 起動中に何が起きるか

```
  プロセス開始
       │
       ▼
  1. JVM / ランタイム初期化
     （Java：クラスローディング、JIT ウォームアップ）
       │
       ▼
  2. 設定の読み込み
     （YAML/環境変数の読み込み、設定の検証）
       │
       ▼
  3. データベース初期化
     （接続、マイグレーション実行、スキーマ検証）
       │
       ▼
  4. フレームワーク初期化
     （ルート、ミドルウェア、フィルターの設定）
       │
       ▼
  5. 依存関係の初期化
     （キャッシュ、メッセージキュー、外部サービスへの接続）
       │
       ▼
  6. ポートでリスニング開始
     （「リクエスト受付準備完了」）
       │
       ▼
  7. ヘルスチェック通過
     （「生きていて準備完了」）
```

### 起動時間の比較

| 技術 | 典型的な起動時間 | 理由 |
|------|----------------|------|
| **volta（Javalin）** | ~200ms | 最小限のフレームワーク、アノテーションスキャンなし、DI コンテナなし |
| **Spring Boot** | 3-8秒 | コンポーネントスキャン、自動設定、DI コンテナ初期化 |
| **Keycloak** | 20-40秒 | フル Java EE スタック、テーマコンパイル、SPI ロード |
| **Node.js（Express）** | 100-500ms | V8 の起動は速いが、npm 依存関係のロードは様々 |
| **Go（net/http）** | 10-50ms | コンパイル済みバイナリ、ランタイム初期化なし |
| **Rails** | 5-15秒 | Eager ローディング、アセットコンパイル、gem 初期化 |

### 起動を遅くするもの

| 要因 | 影響 | 例 |
|------|------|-----|
| **クラスパススキャン** | アノテーション検索で数千クラスをスキャン | Spring の @ComponentScan |
| **依存性注入** | オブジェクトグラフの構築と接続 | Spring/Guice コンテナ |
| **データベースマイグレーション** | 起動時のスキーマ変更実行 | Flyway/Liquibase |
| **外部接続** | リモートサービスの応答待ち | Redis、Kafka、外部 API |
| **JIT コンパイル** | JVM がホットパスをコンパイルする初回リクエストのレイテンシ | 全 JVM アプリ |
| **設定のパース** | 検証やデフォルト値を伴う複雑な設定 | XML ヘビーな設定（Keycloak） |

## volta-auth-proxy ではどう使われている？

volta-auth-proxy は約 **200ミリ秒** で起動します。これは偶然ではなく、意図的な設計の成果です。

### volta が高速起動を実現する方法

| 決定 | 起動への影響 |
|------|------------|
| **Spring Boot でなく Javalin** | コンポーネントスキャンなし、DI コンテナなし。ルートは明示的な Java コード。 |
| **PostgreSQL でなく SQLite** | ネットワーク接続不要。データベースはローカルファイル。 |
| **アノテーション処理なし** | @Controller、@Service、@Repository 等のクラスパススキャンなし。 |
| **最小限の依存関係** | Spring Boot の 100 以上に対して約10の依存関係。ロードが少ない。 |
| **明示的な配線** | オブジェクトは注入でなく `new` で作成。コンストラクタはマイクロ秒で実行。 |
| **マイグレーションフレームワークなし** | スキーマ作成はシンプルな SQL スクリプト。Flyway マイグレーションチェーンではない。 |

### volta の起動シーケンス

```
  t=0ms     JVM 起動、Main.class をロード
  t=20ms    YAML から設定を読み込み
  t=40ms    SQLite データベースを開く（ローカルファイル I/O）
  t=60ms    スキーマの検証 / 必要に応じて作成
  t=80ms    Javalin アプリ作成、ルート登録
  t=120ms   ミドルウェア設定（CSRF、セッション、認証）
  t=160ms   Javalin が Jetty をポート 8080 で起動
  t=200ms   "volta-auth-proxy started on port 8080"
            リクエスト処理準備完了。
```

### 200ms が volta のユースケースで重要な理由

| シナリオ | 200ms 起動 | 30秒起動 |
|---------|-----------|---------|
| **開発者の再起動** | 即座のフィードバック | 変更のたびにコーヒーブレイク |
| **Docker 再起動** | 1秒以内にコンテナ健全 | 30秒以上コンテナ不健全 |
| **ヘルスチェック** | 最初のチェックで通過 | 長い start_period が必要 |
| **クラッシュ復旧** | ユーザーは何も気づかない | ユーザーが30秒間エラーを見る |
| **CI/CD パイプライン** | 統合テストが即座に開始 | パイプラインが起動待ち |

### 開発体験の違い

```
  volta の開発サイクル：
  ┌──────────────────────────────────────────┐
  │  コード編集 → 保存 → 自動再起動（200ms） │
  │  → 結果確認 → 再び編集                   │
  │  合計サイクル：約1秒                      │
  └──────────────────────────────────────────┘

  Keycloak の開発サイクル：
  ┌──────────────────────────────────────────┐
  │  コード編集 → 保存 → リビルド（30秒）     │
  │  → 起動待ち（30秒） → 結果確認           │
  │  → 再び編集                               │
  │  合計サイクル：60秒以上                    │
  └──────────────────────────────────────────┘
```

## よくある間違いと攻撃

### 間違い1：開発中に起動時間を無視する

遅い起動は蓄積します。1日50回再起動し、毎回30秒かかれば、日に25分 -- 週2時間以上 -- をただ待つことに失います。

### 間違い2：起動時にやりすぎる

可能なすべての設定の読み込み、すべてのキャッシュのウォームアップ、すべてのサービスへの接続を起動時に行うと、プロセスが脆弱になります。どれか1つの依存関係が利用不可なら、アプリケーション全体が起動に失敗します。非重要な依存関係には遅延初期化を推奨します。

### 間違い3：起動時間を計測しない

計測しなければ、じわじわ長くなっていることに気づきません。起動時に経過時間を出力するログ行を追加しましょう。volta は `"started on port 8080"` をタイムスタンプ付きでログ出力するので、劣化が即座に見えます。

### 間違い4：起動時間と初回リクエストのレイテンシを混同する

アプリケーションは 200ms で「起動」しても、JIT コンパイル、遅延クラスローディング、コールドキャッシュにより最初のリクエストが2秒かかる場合があります。両方を計測しましょう。
