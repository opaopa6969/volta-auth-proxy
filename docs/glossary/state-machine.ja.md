# State Machine（状態マシン）

[English version](state-machine.md)

---

## これは何？

状態マシンとは、システムを定義された状態の集合とそれらの間の遷移ルールとして記述するモデルです。任意の時点で、システムは正確に1つの状態にあります。特定のイベント（トリガー）が発生し、特定の条件（ガード）が満たされた場合にのみ状態が変化します。現在の状態 + トリガー + ガード の組み合わせが次の状態を決定します。

ボードゲームのようなものです。あなたの駒はちょうど1つのマスにあります。自分の番（トリガー）のときだけ動け、サイコロの目が条件（ガード）を満たした場合のみ進めます。ランダムなマスにテレポートはできません。ルールに従います。状態マシンも同じです：スキップなし、曖昧さなし、未定義の動作なし。

volta-auth-proxyは認証フロー全体を8状態のマシンとしてモデル化しています。すべてのユーザーは常にこの8状態のいずれか1つにあり、すべての操作は定義された遷移です。

---

## なぜ重要なのか？

状態マシンがなければ、認証ロジックは複数ファイルにまたがるif/elseの絡まった蜘蛛の巣になります。エッジケースが見落とされ、ユーザーが不可能な状態に陥り、誰もテストしていない境界にセキュリティホールが現れます。

- **未定義の動作がない**：すべての状態に定義された出口がある。ユーザーが「詰む」ことがない
- **設計によるセキュリティ**：未認証ユーザーがログインフローを経ずに認証済み状態に到達できないことを証明できる
- **テスト容易性**：すべての遷移が列挙可能で、すべてテストできる
- **デバッグ**：問題が起きたとき、ユーザーがどの状態にいて、どの遷移が失敗したかが正確にわかる
- **ドキュメント**：状態マシン図がそのままドキュメントになる

## どう動くのか？

### 基本概念

```
  ┌─────────────────────────────────────────────────┐
  │  状態マシン = 状態 + 遷移 + ガード               │
  │                                                  │
  │  状態（State）：システムが今どこにあるか          │
  │  トリガー：    何のイベントが起きたか             │
  │  ガード：      真でなければならない条件            │
  │  アクション：  実行される副作用                    │
  │  遷移：        状態A ──(トリガー + ガード)──► 状態B│
  └─────────────────────────────────────────────────┘
```

### 簡単な例

```
  ┌──────────┐  コイン投入    ┌──────────┐  ボタン押下   ┌──────────┐
  │ ロック中  │ ──────────────►│  準備完了 │ ────────────►│  販売中  │
  └──────────┘                └──────────┘               └──────────┘
       ▲                           │                          │
       │        タイムアウト        │       商品排出           │
       └───────────────────────────┘          └───────────────┘
```

この自販機には3つの状態、4つの遷移、明確なルールがあります。コインを入れずに飲み物は出ません。

### 決定論的遷移

重要な性質：同じ状態 + トリガー + ガード値であれば、結果は常に同じ。ランダム性や「時々こう、時々ああ」はありません。

```
  状態: UNAUTHENTICATED
  トリガー: GET /login
  ガード: !request.accept_json
  ──────────────────────────────
  結果: 常に → AUTH_PENDING（Googleにリダイレクト）
        それ以外にはならない。
```

### グローバル遷移

一部の遷移は多くの状態から適用されます。すべての状態定義で繰り返す代わりに、グローバルとして一度だけ宣言されます：

```
  グローバル: logout
  対象: UNAUTHENTICATED、AUTH_PENDING以外のすべての状態
  結果: → UNAUTHENTICATED

  グローバル: session_timeout
  対象: UNAUTHENTICATED以外のすべての状態
  結果: → UNAUTHENTICATED
```

## volta-auth-proxyではどう使われているか？

### 8つの状態

```
  ┌─────────────────────┐
  │   UNAUTHENTICATED   │ ◄── セッションなし。ログインが必要。
  └─────────┬───────────┘
            │ GET /login
            ▼
  ┌─────────────────────┐
  │    AUTH_PENDING      │ ◄── OIDCコールバック待ち。
  └─────────┬───────────┘
            │ GET /callback（成功）
            ▼
       ┌────┴────┬──────────────┐
       ▼         ▼              ▼
  ┌─────────┐ ┌───────────┐ ┌──────────────┐
  │NO_TENANT│ │TENANT_SEL. │ │INVITE_CONSENT│
  └─────────┘ └─────┬─────┘ └──────┬───────┘
       │            │               │
       │            ▼               │
       │    ┌───────────────┐       │
       └───►│ AUTHENTICATED │◄──────┘
            └───────┬───────┘
                    │ tenant.suspended
                    ▼
            ┌─────────────────┐
            │TENANT_SUSPENDED │
            └─────────────────┘
```

### `dsl/auth-machine.yaml`からの状態定義

| 状態 | 説明 | フェーズ |
|------|------|---------|
| `UNAUTHENTICATED` | 有効なセッションなし | 1 |
| `AUTH_PENDING` | IdPにリダイレクト済み、コールバック待ち | 1 |
| `INVITE_CONSENT` | 招待同意画面を表示中 | 1 |
| `TENANT_SELECT` | 複数テナントに所属、選択が必要 | 1 |
| `NO_TENANT` | 認証済みだがテナントメンバーシップなし | 1 |
| `AUTHENTICATED` | 完全認証済み、テナント確定、JWT利用可能 | 1 |
| `TENANT_SUSPENDED` | 現在のテナントが一時停止中 | 1 |
| `MFA_REQUIRED` | 認証済みだがMFA未完了（Phase 3） | 3 |

### 遷移の例：ログインフロー

```yaml
# auth-machine.yamlから
UNAUTHENTICATED:
  transitions:
    login_browser:
      trigger: "GET /login"
      guard: "!request.accept_json"
      actions:
        - { type: side_effect, action: create_oidc_flow }
        - { type: http, action: redirect, target: google_authorize_url }
      next: AUTH_PENDING
```

これは「システムがUNAUTHENTICATEDの状態でブラウザ（JSONでない）から`GET /login`を受け取ったら、OIDCフローを作成し、Googleにリダイレクトし、AUTH_PENDINGに遷移する」と読めます。

### バグを防ぐ不変条件

状態マシンは[不変条件](invariant.ja.md)を宣言します。常に成り立つべきルールです：

```yaml
invariants:
  - id: no_deadlock
    rule: "非終端状態には到達可能な出力遷移が少なくとも1つ必要"
  - id: reachable_auth
    rule: "どの状態からもAUTHENTICATEDかUNAUTHENTICATEDへのパスが存在する必要がある"
  - id: logout_always_possible
    rule: "認証済みのどの状態からもlogoutに到達可能でなければならない"
```

### Javaの実装

`AuthService.java`が状態マシンを実装しています。`authenticate()`メソッドがセッションから現在の状態を解決し、各ルートハンドラが個々の遷移を実装します：

```java
// AuthService.java
public Optional<AuthPrincipal> authenticate(Context ctx) {
    // セッションCookieを読む → セッションを検索 → 現在の状態を判定
    // セッションなし: UNAUTHENTICATED
    // session.valid && tenant.active: AUTHENTICATED
    // 等
}
```

## よくある間違いと攻撃

### 間違い1：暗黙の状態

コードに`if (user != null && tenant != null && !suspended)`のような認証チェックがあれば、暗黙の状態マシンがあります。問題は、すべての状態を見渡せず、隙間がないことを証明できないこと。voltaはすべての状態を明示的に名前付きにします。

### 間違い2：遷移の欠落

TENANT_SELECTのユーザーがAUTHENTICATEDに到達する方法がない場合（例：「選択」ボタンが壊れている）、デッドロックです。`no_deadlock`不変条件がこれを防ぎます。

### 間違い3：保留状態のタイムアウトなし

AUTH_PENDINGには600秒のタイムアウトがあります。これがなければ、ログインを開始したがOIDCコールバックを完了しなかったユーザーが永久にAUTH_PENDINGに留まり、リソースを消費します。

### 攻撃：状態の混乱

攻撃者は状態をスキップしようとするかもしれません。例えば、UNAUTHENTICATEDの状態で`/auth/verify`（AUTHENTICATED用の遷移）を呼び出す。voltaは遷移のたびに現在の状態をチェックするため、即座に401が返ります。
