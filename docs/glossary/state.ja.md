# State パラメータ (OAuth / OIDC)

## これは何？

state パラメータとは、ユーザーをログインのために認証プロバイダ（Google など）に送る前に、アプリケーションが生成するランダムで推測不可能な文字列です。Google が認証を完了してユーザーをアプリケーションにリダイレクトで戻すとき、同じ state の値がリダイレクト URL に含まれます。アプリケーションは、返ってきた state が最初に送ったものと一致するか確認します。

クロークの預かり札のようなものです。コートを預けて番号付きの札をもらい、戻ってきたとき、同じ札を見せないとコートを返してもらえません。別の人が違う札で来ても、何も受け取れません。

OIDC（OpenID Connect）のログインフローでは、こうなります:

```
  あなたのアプリ             Google                 あなたのアプリ
  (ログイン前)              (認証プロバイダ)          (リダイレクト後)
  +-------------+          +---------+             +-------------+
  |             |          |         |             |             |
  | state=abc123|  リダイレクト|         |  リダイレクト | 確認:        |
  | を生成       | -------> | ログイン  | ----------> | state=abc123|
  | DB に保存    |          | 画面     | ?code=xyz  | が保存した値  |
  |             |          |         | &state=    | と一致する?   |
  +-------------+          +---------+ abc123     +-------------+
```

## なぜ重要？

state パラメータは、OAuth/OIDC のログインフローに対する CSRF（クロスサイトリクエストフォージェリ）という種類の攻撃を防ぎます。これがないと、攻撃者はあなたのブラウザを騙して、攻撃者のアカウントや盗んだ認可コードでログインフローを完了させることができてしまいます。

state パラメータがない場合の攻撃シナリオ:

```
  1. 攻撃者があなたのアプリで OIDC ログインを開始する
  2. 攻撃者が自分の Google アカウントでログインする
  3. Google がリダイレクト: /callback?code=攻撃者のコード
  4. 攻撃者はここで止まる -- リダイレクトには従わない
  5. 攻撃者がリンクを作成: https://your-app.com/callback?code=攻撃者のコード
  6. 攻撃者が被害者にそのリンクをクリックさせる
  7. アプリがコードを交換し、攻撃者の ID 情報を取得
  8. アプリが被害者のブラウザに攻撃者のアカウントのセッションを作成
  9. 被害者は攻撃者としてログインした状態になる
  10. 被害者がすること（アップロード、設定変更）は全て攻撃者のアカウントに反映される
```

これは「ログイン CSRF」攻撃と呼ばれます。攻撃者は被害者のアカウントを盗むのではなく、被害者を攻撃者のアカウントに入れ、被害者がそこで行うことを全て回収します。

state パラメータがあると、ステップ 7 で失敗します。被害者のブラウザはその state 値を生成していないからです。アプリがデータベースを確認し、一致する state がないことを発見して、リクエストを拒否します。

## どう動くの？

state パラメータは単純なチャレンジ・レスポンスのパターンで動作します:

**ステップ 1: 生成と保存。**
認証プロバイダにユーザーをリダイレクトする前に、アプリはランダムな文字列を生成し、後でユーザーのセッションからアクセスできる場所（データベース、サーバーサイドセッション、署名付き Cookie）に保存します。

**ステップ 2: 認可 URL に含める。**
state の値は認可 URL のクエリパラメータとして追加されます:
```
https://accounts.google.com/o/oauth2/v2/auth?
  response_type=code&
  client_id=YOUR_CLIENT_ID&
  redirect_uri=https://your-app.com/callback&
  scope=openid email profile&
  state=abc123xyz789
```

**ステップ 3: 認証プロバイダがそのまま返す。**
Google（または他の OIDC プロバイダ）は state の値を一切解釈しません。変更せずにそのままアプリへのリダイレクトに含めます:
```
https://your-app.com/callback?code=認可コード&state=abc123xyz789
```

**ステップ 4: 検証。**
アプリはコールバックを受け取り、state パラメータを取り出して、保存した値と照合します。一致すればフローを続行。不一致ならリクエストを拒否します。

**ステップ 5: 消費（一回限り使用）。**
検証後、保存された state は削除して再利用できないようにします。同じコールバック URL を二度使おうとするリプレイ攻撃を防ぎます。

全体の流れ:

```
  ブラウザ          アプリ（サーバー）          Google
  |                |                         |
  |  ログインクリック|                         |
  |--------------->|                         |
  |                |                         |
  |                | state=RND123 を生成      |
  |                | DB に保存                |
  |                | 有効期限 = 10分           |
  |                |                         |
  |  302 Google にリダイレクト + state=RND123  |
  |<---------------|                         |
  |                                          |
  |  ユーザーが Google でログイン               |
  |----------------------------------------->|
  |                                          |
  |  302 リダイレクト: /callback?code=XYZ     |
  |               &state=RND123              |
  |<-----------------------------------------|
  |                                          |
  |  GET /callback?code=XYZ&state=RND123    |
  |--------------->|                         |
  |                |                         |
  |                | DB で state=RND123 を検索 |
  |                | 見つかった? → 続行。       |
  |                | DB から削除（使用済み）。    |
  |                |                         |
  |                | コードをトークンに交換      |
  |                |------------------------>|
  |                |<------------------------|
  |                |                         |
  |  セッション設定  |                         |
  |<---------------|                         |
```

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は Google との OIDC ログインフローの一部として、state パラメータの生成と検証を行います。

**生成。** ユーザーが「Google でログイン」をクリックすると、volta の `OidcService.createAuthorizationUrl()` メソッドが `SecurityUtils.randomUrlSafe(32)` を使って 32 バイトの暗号学的にランダムな URL セーフ文字列を生成します。これは Java の `SecureRandom` を使っており、暗号学的に強力な乱数生成器です。

**保存。** state は `OidcFlowRecord` の一部として、他のフローデータとともにデータベースに保存されます:
- state の値そのもの（検索キーとして使用）
- nonce（id_token のリプレイ防止用 -- nonce の記事を参照）
- PKCE コード検証値（認可コードの保護用）
- `returnTo` URL（ログイン後にユーザーを送る先）
- 招待コード（招待経由で参加する場合）
- 有効期限のタイムスタンプ（作成から 10 分後）

**検証。** Google が `/callback` にリダイレクトで戻すと、volta の `OidcService.exchangeAndValidate()` メソッドが:
1. コールバック URL から `state` パラメータを取り出す。
2. `store.consumeOidcFlow(state)` を呼び出す。これはレコードの検索と削除を一度の操作で行う（消費 = 検索 + 削除）。
3. レコードが見つからなければ、「Invalid state」エラーを投げてログイン失敗。
4. レコードが見つかっても有効期限切れ（10 分以上経過）なら、「State expired」エラーを投げる。
5. state が有効かつ期限内の場合のみ、認可コードのトークン交換に進む。

**バンドル方式。** volta は state、nonce、PKCE 検証値を 1 つのデータベースレコードにまとめています。3 つの値は全て同じログイン試行に属し、一緒に消費されるべきなので、これはきれいな設計です。

## よくある間違い

**1. state パラメータを使わない。**
一部の OAuth チュートリアルは簡略化のために state パラメータを省略しています。これではログインフローが CSRF 攻撃に対して丸裸です。必ず使いましょう。

**2. 予測可能な値を使う。**
state は暗号学的にランダムで推測不可能でなければなりません。タイムスタンプ、連番、ユーザーのメールアドレスのハッシュを使うと、目的が果たせません。volta は `SecureRandom` の出力 32 バイトを使い、256 ビットのエントロピーを確保しています。

**3. state をブラウザだけに保存する（localStorage など）。**
攻撃者がページ上で JavaScript を実行できる場合（XSS 経由）、期待される state を読み取って有効なコールバック URL を作れてしまいます。volta のデータベースのようなサーバーサイドのストレージの方が安全です。

**4. state に有効期限を設定しない。**
永遠に有効な state は、永遠に開いたままの窓です。volta は state レコードを 10 分で期限切れにします。ユーザーがそれ以上かかったら、単にやり直します。

**5. 使用後に state を削除しない。**
検証成功後に state を消費（削除）しないと、コールバック URL を傍受した攻撃者がリプレイできてしまいます。volta の `consumeOidcFlow()` は検索時にアトミックにレコードを削除し、一回限りの使用を保証します。

**6. state と nonce を混同する。**
両方とも OIDC フローのランダムな値ですが、目的が異なります。state パラメータはリダイレクトフローを保護します（CSRF 防止）。nonce は id_token を保護します（リプレイ防止）。違いについては nonce の記事を参照してください。
