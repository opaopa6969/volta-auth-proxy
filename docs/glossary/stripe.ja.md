# Stripe

[English version](stripe.md)

---

## これは何？

Stripeは、開発者フレンドリーなAPIを通じてクレジットカードの受付、サブスクリプション管理、請求処理を可能にするオンライン決済処理プラットフォームです。2010年に設立され、SaaSアプリケーションの決済処理におけるデファクトスタンダードとなっています。スタートアップからオンラインで何かを購入したことがあれば、Stripeが決済を処理した可能性が高いです。

Stripeは、顧客とビジネスの間に座る銀行の窓口係のようなものです。顧客が窓口係にクレジットカードを渡し、窓口係が決済を処理し、少額の手数料を取り、残りをあなたの口座に入金します。窓口係はペーパーワーク -- 領収書、返金、不正チェック、税務報告も処理します。あなたはクレジットカードに直接触れる必要がありません。

Stripeの革新的な点はAPI設計です。Stripe以前は、オンライン決済の受付に複数ページのマーチャントアカウント申請書の記入、レガシー決済ゲートウェイとの連携、脆弱な統合コードの記述が必要でした。Stripeはこれを数行のコードとクリーンなREST APIに削減しました。

---

## なぜ重要なのか？

SaaS製品において、課金は認証・認可と密接に結びついています。顧客がプランをアップグレードすると、権限が変わります。サブスクリプションが期限切れになると、アクセスを取り消す必要があります。有料プランに新しいチームメンバーが追加されると、課金を更新する必要があります。

ここがStripeとID管理の接点です。volta-auth-proxyはアクセスルールを適用するためにテナントの課金状態を知る必要があります：

- このテナントは無料プランか有料プランか？
- シート制限を超えていないか？
- 支払い方法は有効か、それとも失敗したか？
- アップグレードしたばかりで、新機能をすぐにアンロックする必要があるか？

課金と認証を統合しないと、アクセスすべきでないユーザーがアクセスを持つ（期限切れのサブスクリプション）か、アクセスすべきユーザーがアクセスを持たない（アップグレードしたがまだ機能が有効になっていない）状態になります。

## どう動くのか？

### コア概念

| 概念 | 説明 |
|------|------|
| **Customer** | 支払いエンティティを表すStripeレコード（通常voltaテナントにマッピング） |
| **Subscription** | 定期支払いプラン（例：月額$49のProプラン） |
| **Product** | 販売対象（例：「volta Pro」「volta Enterprise」） |
| **Price** | 製品の特定の価格オプション（例：月額$49、年額$490） |
| **Invoice** | サブスクリプション期間に生成される請求書 |
| **Payment Intent** | 単一の支払い試行を表す |
| **Webhook** | 何かが起きた時にStripeがサーバーに送信するHTTPコールバック |

### Webhook駆動アーキテクチャ

Stripeは**Webhook** -- 設定したURLに送信されるHTTP POSTリクエストを通じて、変更をアプリケーションに通知します。イベント駆動アーキテクチャです：

```
  Stripe                          あなたのサーバー（volta）
  ======                          =======================

  顧客がプランをアップグレード
       │
       ├──► Webhook: customer.subscription.updated
       │         { subscription: { plan: "pro", status: "active" } }
       │
       │                          Webhookを受信
       │                          署名を検証
       │                          DBのテナントプランを更新
       │                          200 OKを返す
       │
  支払い失敗
       │
       ├──► Webhook: invoice.payment_failed
       │         { customer: "cus_xxx", attempt: 3 }
       │
       │                          Webhookを受信
       │                          テナントを「payment_failed」にマーク
       │                          アクセスを制限（猶予期間）
       │                          通知を送信
```

### 一般的なWebhookイベント

| イベント | 発火タイミング | voltaがすべきこと |
|---------|-------------|-----------------|
| `customer.subscription.created` | 新規サブスクリプション開始 | テナントプランの作成/アップグレード |
| `customer.subscription.updated` | プラン変更（アップ/ダウングレード） | テナントの機能/制限を更新 |
| `customer.subscription.deleted` | サブスクリプションキャンセル | 無料プランにダウングレード |
| `invoice.payment_failed` | 支払い試行の失敗 | 猶予期間を開始、通知 |
| `invoice.paid` | 支払い成功 | 支払い失敗フラグをクリア |
| `customer.deleted` | Stripeから顧客を削除 | テナント非アクティブ化の処理 |

### Webhook署名の検証

Stripeはすべてのwebhookに秘密鍵で署名します。これにより攻撃者が偽のwebhookをサーバーに送信するのを防ぎます：

```
Stripe-Signature: t=1614556800,
  v1=5257a869e7ecebeda32affa62cdca3fa51cad7e77a0e56ff536d0ce8e108d8bd
```

サーバーはイベントを処理する前にこの署名を検証する必要があります。検証なしでは、webhookURLを発見した誰もが「サブスクリプションがアップグレードされた」などのイベントを偽造し、プレミアムアクセスを無料で獲得できます。

## volta-auth-proxy ではどう使われている？

volta-auth-proxyは、課金状態を認証・認可と同期するためにStripe Webhookを取り込みます。これは決済処理ではありません -- voltaはクレジットカードを扱いません。各テナントが何に権限を持っているかを知ることが目的です。

### 統合アーキテクチャ

```
  Stripe Cloud
       │
       │  Webhooks（HTTPS POST）
       ▼
  ┌──────────────────┐
  │  volta-auth-proxy │
  │                   │
  │  /webhook/stripe  │  ← イベントを受信
  │       │           │
  │       ▼           │
  │  署名を検証       │
  │  テナントDBを更新  │
  │  機能を調整       │
  │  必要に応じて通知  │
  └──────────────────┘
```

### voltaがStripeデータで行うこと

1. **プランマッピング**：Stripeサブスクリプション製品がvoltaテナントプラン（free、pro、enterprise）にマッピング。各プランは機能フラグと制限を定義。
2. **シートカウント**：Stripeサブスクリプション数量がvoltaのテナントごとのユーザー制限にマッピング。
3. **猶予期間**：支払いが失敗しても、voltaはすぐにアクセスを取り消さない。設定可能な猶予期間に入り、テナント管理者に警告を表示。
4. **機能ゲーティング**：プレミアム機能（例：SAML SSO、監査ログ、カスタムドメイン）はテナントのStripeサブスクリプションプランによってゲーティング。

### なぜポーリングではなくWebhookか？

voltaがStripe APIのポーリングではなくWebhookを使用する理由：

- **リアルタイム**：変更が数秒以内に反映される（ポーリング間隔ではなく）
- **効率的**：発生していない変更をチェックする無駄なAPI呼び出しがない
- **信頼性**：Stripeは失敗したwebhook配信を最大72時間リトライ
- **Stripeの推奨**：Stripeはサブスクリプション状態管理にwebhookを明示的に推奨

## よくある間違いと攻撃

### 間違い1：Webhook署名を検証しない

`Stripe-Signature`ヘッダーを検証せずにwebhookを処理すると、誰でもwebhook URLに偽のイベントをPOSTできます。攻撃者が`customer.subscription.created`イベントを偽造し、無料でプレミアムプランを獲得できます。

### 間違い2：重複イベントを処理しない

Stripeは同じイベントを複数回送信する可能性があります（ネットワークリトライ）。webhookハンドラーは**べき等**でなければなりません -- 同じイベントを2回処理しても同じ結果になる必要があります。イベントIDを使用して重複を検出してください。

### 間違い3：Webhookハンドラーからエラーを返す

webhookエンドポイントが非2xxステータスを返すと、Stripeは配信をリトライします。ハンドラーが一貫して失敗すると、Stripeはwebhookエンドポイント全体を無効にします。エラーを適切に処理し、後で処理するためにイベントをキューに入れる必要がある場合でも200を返してください。

### 間違い4：マッピングなしでStripeデータを信頼する

StripeのID内部IDを認可ロジックで直接使用しないでください。Stripeカスタマーをvoltaテナントに、Stripe製品をvoltaプランにマッピングしてください。マッピングが壊れた場合、fail open（アクセスを許可）ではなくfail closed（アクセスを制限）にしてください。

### 攻撃：Webhook URLの発見

攻撃者がwebhook URLを発見すると、細工されたイベントの送信を試みることができます。署名検証でこれを防ぎますが、推測困難なパス（例：`/webhook/stripe/a7b3c9d2`）を使用し、StripeがwebhookソースIPを公開している場合はIP許可リストも検討してください。
