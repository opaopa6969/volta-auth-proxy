# SuperTokens

[English version](supertokens.md)

---

## これは何？

SuperTokensは、主にNode.jsで書かれた（Javaベースのコアを持つ）オープンソースの認証ソリューションです。メール/パスワード、パスワードレス、ソーシャルログイン、MFAなどのビルド済みログインフローを、SDKを通じてアプリケーションに統合します。セルフホストまたはマネージドクラウドサービスとして使用できます。

SuperTokensは、家に設置するプレハブキッチンのようなものです。レイアウト（メールログイン、Googleログイン、パスワードレス）を選ぶと、キッチン会社がキャビネット、カウンタートップ、家電を届け、自宅に設置します。キッチンを所有し修正できますが、ゼロから設計したわけではありません。これに対しvolta-auth-proxyは、生の木材を買ってキッチンを自分で建てるようなものです -- 作業は多いですが、すべてのジョイントとネジを理解しています。

SuperTokensは[Auth0](auth0.md)（マネージド）、[Keycloak](keycloak.md)（セルフホスト、重量級）、volta-auth-proxy（セルフホスト、軽量）と同じ競合空間にいます。主な差別化要因はSDK駆動の統合モデルです：別のIDサービスを実行する代わりに、ライブラリ呼び出しでアプリケーションにSuperTokensを埋め込みます。

## なぜ重要なのか？

volta-auth-proxyの文脈では、SuperTokensは**セルフホスト認証空間の競合製品**として重要です。開発者がセルフホスト認証ソリューションを評価する際、SuperTokensは最も頻繁に言及される代替手段の一つです。

### セルフホスト認証のランドスケープ

| ソリューション | 言語 | アーキテクチャ | 重さ |
|-------------|------|-------------|------|
| [Keycloak](keycloak.md) | Java | スタンドアロンサーバー | 重い（~512MB-2GB RAM） |
| [ZITADEL](zitadel.md) | Go | スタンドアロンサーバー | 中程度（~200-500MB RAM） |
| [Ory Stack](ory-stack.md) | Go | 複数マイクロサービス | 中〜重い |
| SuperTokens | Node.js + Javaコア | SDK + コアサービス | 中程度（~200-400MB） |
| volta-auth-proxy | Java | スタンドアロン + ForwardAuth | 軽い（~30-50MB） |

SuperTokensは開発者体験で注目されています -- React/Next.js SDKはログイン用のビルド済みUIコンポーネントを提供し、バックエンドSDKはセッション管理を自動処理します。これはvoltaのアプローチの正反対で、voltaでは認証コードを自分で書きます。

## どう動くのか？

### アーキテクチャ

```
  ┌──────────────────────────────────────────┐
  │            あなたのアプリケーション          │
  │                                           │
  │  ┌───────────────┐  ┌─────────────────┐  │
  │  │ フロントエンドSDK│  │ バックエンドSDK  │  │
  │  │  (React, Vue)  │  │  (Node, Python) │  │
  │  │                │  │                 │  │
  │  │  ビルド済み     │  │  認証ルート     │  │
  │  │  ログインフォーム│  │  セッション管理  │  │
  │  └───────────────┘  └────────┬────────┘  │
  │                              │            │
  └──────────────────────────────┼────────────┘
                                 │ HTTP
                                 ▼
                    ┌──────────────────────┐
                    │  SuperTokens Core    │
                    │  （Javaプロセス）      │
                    │                      │
                    │  ユーザーストレージ    │
                    │  セッション管理       │
                    │  レシピロジック       │
                    └──────────┬───────────┘
                               │
                               ▼
                    ┌──────────────────────┐
                    │  PostgreSQL / MySQL   │
                    └──────────────────────┘
```

### レシピ

SuperTokensは「レシピ」システムを使用します -- 各認証方法がレシピです：

| レシピ | 説明 |
|--------|------|
| **EmailPassword** | 従来のメール + パスワードログイン |
| **ThirdParty** | ソーシャルログイン（Google、Apple、GitHubなど） |
| **Passwordless** | メール/SMSによるマジックリンクまたはOTP |
| **Session** | ローテーティングリフレッシュトークンによるセッション管理 |
| **UserRoles** | ロールベースアクセス制御 |
| **MultiFactorAuth** | TOTPベースのMFA |

### SuperTokens vs volta-auth-proxy

| 観点 | SuperTokens | volta-auth-proxy |
|------|-------------|-----------------|
| 統合モデル | アプリに埋め込むSDK | リバースプロキシForwardAuth |
| 言語エコシステム | Node.js/Python/Go SDK | Java（Javalin） |
| フロントエンド | ビルド済みReactコンポーネント | jteテンプレート（サーバーレンダリング） |
| マルチテナント | サポート（クラウドの有料機能） | コア設計原則 |
| プロキシパターン | 非サポート | コアアーキテクチャ（[ForwardAuth](forwardauth.md)） |
| セッション管理 | ローテーティングリフレッシュトークン（SDKが処理） | サーバーサイドセッション（Postgres） |
| 認証コードの可視性 | SDK抽象化 | 完全なソースコード制御 |
| npm依存関係 | 多数（SDK + 推移的） | ゼロ（Java、npmなし） |

## volta-auth-proxy ではどう使われている？

volta-auth-proxyはSuperTokensを**使用しません**。根本的に異なる統合哲学を持つ競合製品です。

### voltaが異なるアプローチを選んだ理由

1. **SDK vs プロキシ**：SuperTokensはSDKでアプリに認証を埋め込む。voltaはアプリの前に[ForwardAuth](forwardauth.md)プロキシとして位置する。プロキシアプローチにより、アプリは認証について知る必要がない -- IDヘッダーを受け取るだけ。

2. **npmリスク**：SuperTokensのNode.js SDKは何百ものnpm依存関係をもたらす。voltaの哲学はnpmゼロ -- MavenによるJava、npmエコシステムからのサプライチェーンリスクなし。

3. **ブラックボックスの懸念**：SuperTokens SDKは認証ロジックを抽象化する。問題が起きた時、SDKレイヤーを通してデバッグする。voltaは自分が書いた（または読める）コード -- すべてのハンドラー、すべてのSQLクエリ、すべての検証ステップがIDEで見える。

4. **マルチテナンシーアプローチ**：SuperTokensはマルチテナンシーを後から機能として追加。voltaは初日からマルチテナンシー向けに設計 -- テナントの概念がすべてのテーブル、すべてのクエリ、すべてのハンドラーに浸透。

5. **ForwardAuthパターン**：SuperTokensはリバースプロキシForwardAuthパターンをサポートしない。単一の認証レイヤーの背後で複数のサービスを保護したい場合（典型的なマイクロサービスセットアップ）、voltaと[Traefik](traefik.md)がネイティブに対応。SuperTokensでは各サービスにSDKを統合する必要がある。

### voltaよりSuperTokensを選ぶべきケース

- Node.js/Reactアプリケーションを構築中でビルド済み認証UIが欲しい
- ソーシャルログイン + メール/パスワード + パスワードレスをまとめて欲しい
- リバースプロキシForwardAuthパターンが不要
- npm依存関係に抵抗がない
- SuperTokensマネージドクラウドオプションが欲しい

### SuperTokensよりvoltaを選ぶべきケース

- ForwardAuthプロキシパターンが必要（複数サービスの保護）
- Javaショップで、アプリケーションと同じ言語での認証が欲しい
- npm依存関係ゼロが欲しい
- 認証コードのすべての行を理解・制御したい
- マルチテナンシーがアドオンではなくコア要件
- 最小限のリソース使用量が欲しい

## よくある間違いと攻撃

### 間違い1：SuperTokens Coreを更新しない

SuperTokens Core（Javaプロセス）とSDKはバージョン互換性が必要です。新しいSDKを古いCoreで実行すると、サイレントな障害やセキュリティ問題を引き起こす可能性があります。同期を保ってください。

### 間違い2：Core APIを公開する

SuperTokens Coreはポート3567でHTTP APIを公開します。インターネットから到達可能な場合、攻撃者がユーザーを作成、セッションを変更、認証をバイパスできます。localhostまたは内部ネットワークのみにバインドしてください。

### 間違い3：保護をフロントエンドSDKだけに頼る

フロントエンドSDKはUIコンポーネントを提供しますが、セキュリティを強制できません。すべてのセキュリティチェックはバックエンドSDKで行う必要があります。決意のある攻撃者はフロントエンドの保護をバイパスできます。

### 間違い4：デフォルトUIをそのまま本番に出す

SuperTokensのビルド済みUIは機能的ですが汎用的です。変更なしで本番に出すとプロフェッショナルでなく見え、ブランドを期待するユーザーを混乱させる場合があります。UIをカスタマイズするか、SDKのヘッドレスモードで独自に構築してください。

### 攻撃：XSSによるセッショントークンの窃取

SuperTokensはセッショントークンをCookieに保存します。アプリケーションに[XSS](xss.md)脆弱性がある場合、攻撃者がこれらのトークンを窃取できます。SuperTokensはHttpOnly Cookieで軽減しますが、XSSは同じブラウザコンテキスト内でユーザーに代わってアクションを実行できます。
