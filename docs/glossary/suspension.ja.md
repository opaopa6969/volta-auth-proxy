# サスペンション（一時停止）

[English version](suspension.md)

---

## これは何？

サスペンション（一時停止）は、[テナント](tenant.md)またはユーザーアカウントを一時的に無効にし、関連するすべてのアクセスをブロックする行為です。削除（永久的）とは異なり、サスペンションは可逆です。アカウントとデータはそのまま保持されますが、サスペンションが解除されるまで誰もログインやサービス利用ができません。

図書館カードを凍結するようなものです。カードは破壊されません。貸出履歴、予約、読書リストはすべて残ります。しかし凍結が解除されるまで本を借りることはできません。voltaの停止されたテナントも同じです：すべてのデータは保持されますが、全メンバーがロックアウトされます。

サスペンションは、管理者アクション、課金失敗（[Stripe Webhookインジェスション](ingestion.md)経由）、セキュリティインシデント、または自動ポリシー適用によってトリガーされます。

---

## なぜ重要なのか？

サスペンションはマルチテナントSaaSの重要な運用ツールです：

```
  シナリオ1：課金失敗
  ┌──────────────────────────────────────┐
  │ Stripe：「Acmeの支払いが失敗」        │
  │   → voltaがAcmeテナントを停止         │
  │   → Acmeの50人全員がロックアウト       │
  │   → Acmeが支払い方法を更新            │
  │   → voltaがテナントを再有効化          │
  │   → 50人全員が再びログイン可能         │
  └──────────────────────────────────────┘

  シナリオ2：セキュリティインシデント
  ┌──────────────────────────────────────┐
  │ 不審なアクティビティを検出             │
  │   → 管理者がテナントを停止             │
  │   → すべてのセッションが即座に取り消し  │
  │   → 調査を実施                        │
  │   → 問題解決、テナント再有効化          │
  └──────────────────────────────────────┘

  シナリオ3：利用規約違反
  ┌──────────────────────────────────────┐
  │ テナントがToSに違反                    │
  │   → 管理者がテナントを停止             │
  │   → 法的保全のためデータ保持           │
  │   → テナントがサポートに連絡           │
  │   → 問題解決またはデータエクスポート    │
  └──────────────────────────────────────┘
```

サスペンションがないと、選択肢は「アクティブ」か「削除」だけです。一時的な状況に対する中間地帯がありません。

## どう動くのか？

### サスペンションの仕組み

```
  ┌─────────────────────────────────────────────────┐
  │              サスペンションプロセス                  │
  │                                                   │
  │  1. トリガー（管理者/課金/ポリシー）                 │
  │     │                                             │
  │  2. ▼ tenant.status = SUSPENDEDに設定              │
  │     │  tenant.suspended_at = NOW()                 │
  │     │  tenant.suspended_reason = "..."             │
  │     │                                             │
  │  3. ▼ すべてのアクティブセッションを取り消し          │
  │     │  DELETE FROM sessions                        │
  │     │  WHERE tenant_id = :tid                      │
  │     │                                             │
  │  4. ▼ すべてのCookieを無効化                       │
  │     │  （セッション消失 → Cookieは孤立）             │
  │     │                                             │
  │  5. ▼ Webhook Outboxに書き込み                     │
  │     │  イベント：tenant.suspended                   │
  │     │                                             │
  │  6. ▼ 以降のログイン試行はすべて → 403              │
  │     │  「組織が停止されています」                    │
  │     │                                             │
  │  すべて1つのデータベーストランザクション内            │
  └─────────────────────────────────────────────────┘
```

### サスペンション中にブロックされるもの

| アクション | ブロック？ | 理由 |
|---------|---------|------|
| ログイン（新規セッション） | **はい** | 新しいアクセスを許可しない |
| 既存セッションの使用 | **はい** | セッションは取り消し済み |
| JWTでのAPIコール | **はい** | voltaがテナント状態をチェック |
| テナントの[M2M](m2m.md)コール | **はい** | M2Mトークンはテナントにスコープ |
| [SCIM](scim.md)プロビジョニング | **はい** | 停止テナントにユーザー追加不可 |
| [招待](invitation-code.md)の承認 | **はい** | 停止テナントに参加不可 |
| 管理者による再有効化 | **いいえ** | サスペンション解除できる必要がある |
| データエクスポート | **いいえ** | プラットフォーム管理者がデータにアクセス可能 |
| [Webhook](webhook.md)配信 | **いいえ** | イベントは引き続き配信（停止イベント自体） |

### 再有効化

```
  ┌─────────────────────────────────────────────────┐
  │              再有効化プロセス                      │
  │                                                   │
  │  1. トリガー（管理者/課金復旧）                    │
  │     │                                             │
  │  2. ▼ tenant.status = ACTIVEに設定                │
  │     │  tenant.reactivated_at = NOW()               │
  │     │  suspended_reasonをクリア                     │
  │     │                                             │
  │  3. ▼ Webhook Outboxに書き込み                     │
  │     │  イベント：tenant.reactivated                │
  │     │                                             │
  │  4. ▼ ユーザーが再びログイン可能に                  │
  │     │  （新しいセッションが必要 - 古いものは         │
  │     │   サスペンション中に取り消し済み）              │
  └─────────────────────────────────────────────────┘
```

### テナント状態のステートマシン

```
  ┌──────────┐
  │  ACTIVE  │◄────────────────────────────┐
  │          │                             │
  └────┬─────┘                             │
       │                                   │
       │ suspend()                reactivate()
       │                                   │
       ▼                                   │
  ┌──────────┐                             │
  │SUSPENDED │─────────────────────────────┘
  │          │
  └────┬─────┘
       │
       │ delete()（永久、まれ）
       │
       ▼
  ┌──────────┐
  │ DELETED  │  （ソフトデリート、データは
  │          │   コンプライアンス期間保持）
  └──────────┘
```

## volta-auth-proxyではどう使われている？

### 管理者APIによるサスペンション

```
  POST /api/v1/tenants/{tid}/suspend
  Authorization: Bearer <platform-admin-token>
  {
    "reason": "利用規約違反"
  }

  レスポンス：
  {
    "tenant_id": "acme-uuid",
    "status": "SUSPENDED",
    "suspended_at": "2026-04-01T12:00:00Z",
    "reason": "利用規約違反",
    "sessions_revoked": 47
  }
```

### Stripe課金によるサスペンション

voltaがStripeから`customer.subscription.deleted`イベントを[取り込む](ingestion.md)と：

```
  Stripe                   volta-auth-proxy
  ======                   ================

  subscription.deleted ──► POST /webhooks/stripe
                            │
                            ├─ Stripe署名を検証
                            ├─ Stripeカスタマー → voltaテナントにマッピング
                            ├─ tenant.suspend(reason="billing")
                            │    ├─ 全セッション取り消し
                            │    ├─ Outbox書き込み：tenant.suspended
                            │    └─ 成功を返す
                            └─ Stripeに200を返す
```

### 認証フローでのサスペンション状態チェック

volta経由のすべてのリクエストでテナント状態をチェック：

```java
// ForwardAuthHandler内
AuthPrincipal principal = authenticate(request);
Tenant tenant = tenantService.get(principal.tenantId());

if (tenant.isSuspended()) {
    ctx.status(403).json(Map.of(
        "error", "tenant_suspended",
        "message", "組織が停止されています",
        "reason", tenant.suspendedReason()
    ));
    return;
}

// テナントはアクティブ → アップストリームに転送
```

### サスペンション + [ABAC](abac.md)統合

ABACポリシーがサスペンション状態を参照できます：

```json
{
  "condition": {
    "subject.tenant_status": {"eq": "ACTIVE"}
  },
  "effect": "ALLOW"
}
```

## よくある間違いと攻撃

### 間違い1：サスペンション時にセッションを取り消さない

テナント状態をSUSPENDEDに設定してもセッションをアクティブのままにすると、セッションが自然に期限切れになるまでユーザーはサービスを使い続けられます。常にセッションを即座に取り消してください。

### 間違い2：課金に猶予期間がない

最初の支払い失敗で即座に停止するのは攻撃的です。停止前に通知付きの猶予期間（例：3日）を検討してください。

### 間違い3：サスペンションではなくハードデリート

支払いを忘れただけのテナントのデータを削除するのは壊滅的です。サスペンションはアクセスを制限しながらデータを保持します。

### 間違い4：テナント管理者への通知がない

ユーザーはなぜロックアウトされているか、どう解決するかを知るべきです。エラーレスポンスに常にサスペンション理由を含め、テナント管理者に通知してください。

### 攻撃：課金操作による強制サスペンション

課金システムにアクセスした攻撃者がテナントのサブスクリプションをキャンセルしてサスペンションをトリガーする可能性。防御：課金変更に強力な認証が必要、予期しないサブスクリプションキャンセルがアラートをトリガー。

### 攻撃：キャッシュされたトークンによるサスペンション回避

サスペンション前に発行されまだ期限切れでない[JWT](jwt.md)があると、ユーザーがサスペンションチェックを迂回できる可能性。防御：voltaはトークン発行時だけでなくリクエストごとにテナント状態をチェック。
