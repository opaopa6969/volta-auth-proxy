# テナントライフサイクル

[English version](tenant-lifecycle.md)

---

## これは何？

テナントライフサイクルとは、テナント（ワークスペース/組織）が作成から削除までたどる段階のことです。入社し、休職し、最終的に退職する社員のように、テナントもアクティブ、停止、削除の状態を持ち、各状態変化は中にいるメンバーに影響を与えます。

---

## なぜ重要？

ライフサイクルを間違えると深刻な問題が起きます：
- **無制限に作成：** スパムテナント、リソース枯渇
- **通知なしに停止：** ユーザーが説明なくロックアウト
- **データ処理なしに削除：** 永久的なデータ損失、法的責任（GDPR はデータ保持期間を要求）

各遷移には、メンバー、セッション、データ、課金に何が起きるかの明確なルールが必要です。

---

## 簡単な例

```
[作成済み] ──── 通常利用 ────> [アクティブ]
                                    |
                          管理者が停止
                                    |
                                    v
                                [停止中]
                                    |
                        管理者が再開 / 管理者が削除
                          /              \
                         v                v
                    [アクティブ]        [削除済み]
```

### テナントが停止されると、メンバーはどうなる？

| 観点 | 想定される動作 |
|------|--------------|
| **ログイン** | ログインはできるがワークスペースにアクセスできない |
| **セッション** | 既存セッションは残るがアクセスは拒否 |
| **データ** | 保持されるが読み取り専用（または完全ブロック） |
| **課金** | 通常は一時停止またはフリープランに切り替え |
| **通知** | メンバーに説明が通知されるべき |

## volta-auth-proxy では

volta は3つのライフサイクル操作をサポートしています：

### 作成

テナントは自動的に（初回ログイン時の個人テナント）、または API 経由で手動で作成されます。すべてのテナントは `name`、`slug`、`is_active = true` で作成されます。

### 停止

オーナーは以下でテナントを停止できます：
```
POST /api/v1/admin/tenants/{tenantId}/suspend
```

テナントレコードの `is_active = false` を設定します。volta は `tenant.suspended` アウトボックスイベントもエンキューし、下流サービスが反応できるようにします：

```java
store.setTenantActive(tenantId, false);
store.enqueueOutboxEvent(tenantId, "tenant.suspended",
    "{\"tenant_id\":\"" + tenantId + "\"}");
```

停止されたテナントのメンバーがリソースにアクセスしようとすると、volta は `TENANT_SUSPENDED` を返し、ワークスペースの切り替えまたはサポートへの連絡を案内するエラーページを表示します。

### 再開

```
POST /api/v1/admin/tenants/{tenantId}/activate
```

`is_active = true` を設定。メンバーは即座にワークスペースに再びアクセスできます。

### 削除

volta は現在、テナントのハード削除を実装していません。これは意図的なものです。テナント削除にはデータ保持ポリシー、監査ログの保存、メンバー通知の慎重な対応が必要です。Phase 2 以降の機能です。
