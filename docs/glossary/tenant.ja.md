# Tenant / マルチテナンシー

[English version](tenant.md)

---

## 初心者の方はここから

ソフトウェアエンジニアリングが初めてで、「テナント」という言葉が賃貸契約のように聞こえるなら、それほど間違っていません！ソフトウェアにおけるテナントは、基本的にアプリケーションの「借り手」です。マンションの住人がそれぞれ自分のプライベートな空間を持ちながら同じ建物を共有するように、SaaSアプリケーションのテナントはそれぞれ自分のプライベートなワークスペースを持ちながら同じソフトウェアを共有します。

**最もシンプルな考え方：** あなたの会社がSlackを使っているなら、会社全体がSlackのシステムにおける1つの「テナント」です。Slackを使っている別の会社は別のテナントです。両方とも同じSlackソフトウェアを使っていますが、お互いのメッセージやチャンネルを見ることはできません。

volta-auth-proxyでは、テナントはワークスペース（通常は会社や組織）です。誰かがログインすると、voltaはその人がどのテナントに属しているかを特定し、そのテナントのデータだけを見られるようにします。たとえ話を使った初心者向けの詳しい説明は[multi-tenant.md](multi-tenant.md)をご覧ください。

この記事の残りの部分では、より詳細な技術的内容に入ります。

---

## これは何？

テナントは、SaaSアプリケーションでワークスペースを共有するユーザーのグループです。Slackを使っていれば、あなたの会社のSlackワークスペースがテナントです。各テナントは独自のデータ、メンバー、設定を持ち、他のテナントから完全に隔離されています。マルチテナンシーとは、1つのアプリケーションが同時に多くのテナントにサービスを提供することです。

マンションのようなものです。各部屋（テナント）は独自の鍵を持つプライベートな空間です。住人は建物のインフラ（エレベーター、配管、電気）を共有しますが、お互いの部屋には入れません。建物管理（volta-auth-proxy）が、誰がどの部屋の鍵を受け取るかを管理します。

---

## なぜ重要なのか？

SaaS製品を作るなら、マルチテナンシーが必要です。これがないと：

- ある顧客が別の顧客のデータを見える可能性がある
- 顧客ごとの請求、プラン、制限を管理する方法がない
- 組織ごとの設定（MFA要件など）を提供できない
- 顧客ごとに別々のデプロイメントが必要（高コスト、メンテナンスが大変）

マルチテナンシーはSaaSを可能にするアーキテクチャの基盤です。間違えるとデータ漏洩が起き、顧客Aが顧客Bのデータを見てしまいます。これはSaaS企業にとって最も深刻なインシデントの1つです。

---

## どう動くのか？

### データ分離戦略

テナントデータを分離する3つの主要なアプローチがあります：

```
  戦略1：行レベル分離（voltaが使用）
  ═══════════════════════════════════

  1つのデータベース、1つのスキーマ、全テナントが同じテーブルを共有。
  すべての行にtenant_idカラムがある。

  ┌─────────────────────────────────────────────┐
  │  itemsテーブル                                │
  │  ┌────────────┬───────────┬────────────────┐ │
  │  │ tenant_id  │ id        │ data           │ │
  │  ├────────────┼───────────┼────────────────┤ │
  │  │ acme-uuid  │ item-1    │ ACMEのデータ    │ │
  │  │ acme-uuid  │ item-2    │ ACMEのデータ    │ │
  │  │ globex-uuid│ item-3    │ Globexのデータ  │ │
  │  │ globex-uuid│ item-4    │ Globexのデータ  │ │
  │  └────────────┴───────────┴────────────────┘ │
  └─────────────────────────────────────────────┘

  すべてのクエリにWHERE tenant_id = ?が必須。
  忘れると、テナント間でデータが漏洩する。

  長所：シンプル、効率的、管理しやすい
  短所：1つのバグ = データ漏洩、物理的分離なし


  戦略2：スキーマレベル分離
  ═══════════════════════════

  1つのデータベース、テナントごとに別スキーマ。

  ┌──────────────────────────────────┐
  │  データベース: volta_db            │
  │                                   │
  │  スキーマ: acme                    │
  │  ┌──────────────────────────┐    │
  │  │ items: ACMEのデータのみ   │    │
  │  └──────────────────────────┘    │
  │                                   │
  │  スキーマ: globex                  │
  │  ┌──────────────────────────┐    │
  │  │ items: Globexのデータのみ │    │
  │  └──────────────────────────┘    │
  └──────────────────────────────────┘

  長所：より強い分離、データのエクスポート/削除が容易
  短所：スキーママイグレーションがN回実行、コネクション管理が複雑


  戦略3：データベースレベル分離
  ═══════════════════════════════

  テナントごとに別データベース。

  ┌─────────────────┐  ┌─────────────────┐
  │ DB: acme_db      │  │ DB: globex_db    │
  │ items: ACMEデータ │  │ items: Globex    │
  └─────────────────┘  └─────────────────┘

  長所：最強の分離、テナント単位のバックアップ/リストアが容易
  短所：高コスト、大規模での管理が困難、テナント横断クエリ不可能
```

voltaは**行レベル分離**を使用します。SaaSアプリケーションで最も一般的な選択肢で、最もシンプルでスケールしやすいです。トレードオフは、アプリケーションコードが常にクエリにtenant_idを含めなければならないことです。

### テナント解決（voltaがどのテナントかを判断する方法）

リクエストが来たとき、voltaはユーザーがアクセスしているテナントを特定する必要があります。4レベルの優先度を使います：

```
  優先度1：セッションCookie（最高優先度）
  ──────────────────────────────────────
  ユーザーにtenant_id付きのアクティブセッションがあれば、
  そのテナントを使用。一般的なケースに対応。

  優先度2：URLサブドメイン
  ──────────────────────
  URLがwiki.acme.example.comなら、tenant_domainsテーブル
  で"acme"を検索。

  優先度3：メールドメイン
  ─────────────────────
  ユーザーがtaro@acme.comでログインしたら、
  tenant_domainsテーブルで"acme.com"を検索。

  例外：「フリーメール」ドメインは除外：
  gmail.com、outlook.com、yahoo.com、yahoo.co.jp、
  hotmail.com、icloud.com、protonmail.com

  優先度4：手動選択 / 招待
  ──────────────────────
  上記のいずれも一致しない場合、ユーザーに表示：
  - 招待コード入力画面
  - またはテナント選択画面（複数に所属している場合）
```

### 複数テナントへの所属

voltaは1人のユーザーが異なるロールで複数のテナントに所属することを許可します：

```
  ユーザー: taro@example.com

  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
  │ テナント: ACME   │  │ テナント: Globex  │  │ テナント: Initech│
  │ ロール: OWNER    │  │ ロール: MEMBER    │  │ ロール: VIEWER   │
  └──────────────────┘  └──────────────────┘  └──────────────────┘

  太郎はテナントを切り替えられる：
  - GET /select-tenant（UI）
  - POST /auth/switch-tenant（API）
  - Volta.switchTenant("globex-uuid")（SDK）
```

## volta-auth-proxyではどう使われているか？

### テナントデータモデル

```
  tenantsテーブル：
  ┌─────────────────────────────────────────────┐
  │  id:       UUID                              │
  │  name:     "ACME Corp"                       │
  │  slug:     "acme"（URLフレンドリーな識別子）  │
  │  plan:     "pro"（将来の請求用）              │
  │  status:   "active" | "suspended"            │
  └─────────────────────────────────────────────┘

  tenant_domainsテーブル：
  ┌─────────────────────────────────────────────┐
  │  tenant_id:  acme-uuid                       │
  │  domain:     "acme.com"                      │
  │  verified:   true                            │
  └─────────────────────────────────────────────┘

  membershipsテーブル：
  ┌─────────────────────────────────────────────┐
  │  user_id:    taro-uuid                       │
  │  tenant_id:  acme-uuid                       │
  │  role:       "ADMIN"                         │
  │  active:     true                            │
  │  joined_at:  2026-03-31T09:00:00Z            │
  └─────────────────────────────────────────────┘
```

### テナント分離の強制

voltaは複数のレベルでテナント分離を強制します：

1. **ForwardAuth：** `X-Volta-Tenant-Id`ヘッダーはvoltaが設定し、クライアントではない。アプリはこのヘッダーを信頼する。

2. **Internal API：** アプリが`/api/v1/tenants/{tid}/members`を呼ぶと、voltaは認証済みユーザーのJWTの`volta_tid`がURLの`{tid}`と一致するか確認。テナント横断アクセスは構造的に防止。

3. **JWTクレーム：** JWTの`volta_tid`クレームがユーザーのアクセス先テナントをアプリに伝える。アプリはすべてのデータベースクエリでこれを使うべき。

4. **テナント停止：** テナントが停止されると、すべてのメンバーのアクセスがブロックされる。他のテナントへの所属は影響を受けない。

### テナントスコープの設定

`volta-config.yaml`で、各アプリがアクセス可能なロールを指定します：

```yaml
apps:
  - id: app-wiki
    url: https://wiki.example.com
    allowed_roles: [MEMBER, ADMIN, OWNER]

  - id: app-admin
    url: https://admin.example.com
    allowed_roles: [ADMIN, OWNER]  # 管理者とオーナーのみ
```

これはForwardAuthレベルで強制されます。VIEWERがapp-adminにアクセスしようとすると403 Forbiddenが返されます。

## よくある間違いと攻撃

### 間違い1：クエリでtenant_idを忘れる

マルチテナントアプリケーションで最も一般的で最も危険なバグ：

```sql
-- 悪い例：すべてのテナントのデータを返す
SELECT * FROM items WHERE id = ?;

-- 良い例：1つのテナントにスコープ
SELECT * FROM items WHERE id = ? AND tenant_id = ?;
```

`AND tenant_id = ?`の1つの欠落がデータ漏洩です。

### 間違い2：クライアント提供のtenant_idを信頼する

クライアントがリクエストパラメータでアクセスするテナントを選べるようにしないでください。テナントIDは認証済みセッションまたはJWTから取得すべきで、クライアントが設定できるURLパラメータやヘッダーからではありません。

### 間違い3：テナント停止を処理しない

テナントが停止された場合（未払い、不正使用など）、すべてのアクセスが即座に停止すべきです。voltaはすべてのForwardAuthリクエストでテナントの状態を確認します。

### 攻撃：テナント列挙

攻撃者が異なるテナントスラッグやIDを試して、どのテナントが存在するかを発見する。voltaは「テナントが見つからない」と「アクセス拒否」に同じエラーを返すことで軽減します。

### 攻撃：IDORによるテナント横断データアクセス

IDOR（Insecure Direct Object Reference）：アプリが連番ID（item-1、item-2...）を使う場合、テナントAの攻撃者がテナントBのアイテムIDを推測してアクセスする可能性がある。防御：クエリに常にtenant_idを含める、連番の代わりにUUIDを使用。
