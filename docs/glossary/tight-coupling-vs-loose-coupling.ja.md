# 密結合 vs 疎結合（Tight Coupling vs Loose Coupling）

[English version](tight-coupling-vs-loose-coupling.md)

---

## これは何？

結合度（カップリング）とは、ソフトウェアのある部分が別の部分にどれだけ依存しているかです。密結合は2つの部分が深く絡み合っていること -- 一方を変えると、もう一方も変えなければなりません。疎結合は2つの部分が薄く明確に定義された境界を通じてやりとりすること -- 一方を変えても、もう一方はほとんど気づきません。

音楽に例えるとわかりやすいです。マーチングバンドは密結合です：すべてのミュージシャンが同じ曲を、同じテンポで、同じキーで演奏しなければなりません。トランペットセクションが別の曲に切り替えたら、バンド全体が崩壊します。ジャズアンサンブルは疎結合です：ミュージシャンたちはコード進行について合意しますが、その構造の中でそれぞれ自由にアドリブします。ピアニストが何か違うことを試みても、ベーシストが適応します。

どちらが「良い」わけではありません。マーチングバンドは統一感のために密結合が必要です。ジャズアンサンブルは創造性のために疎結合が必要です。正しい選択は、何を達成しようとしているかによります。

---

## なぜ重要なのか？

ソフトウェアにおいて、結合度は最も重要な設計判断の1つです。結合が多すぎるとシステムは脆くなります -- すべての変更がどこにでも波及します。結合が少なすぎるとシステムは散らばります -- 何がどこにあるか誰にもわからず、ネットワーク境界を越えて物事をつなぐことに時間を費やします。

ほとんどのソフトウェアのアドバイスは「疎結合は常に良い」と言います。volta-auth-proxyは、少なくとも特定のことについては同意しません。

---

## 密結合が実は良いとき

voltaの設定は意図的に密結合です。すべての設定は2つの場所にあります：

```
.env                   → シークレット（データベースURL、OAuth認証情報）
volta-config.yaml      → 設定（アプリリスト、ロール）
```

5つの設定ファイル、3つの環境変数グループ、データベーステーブルに分散していません。2つのファイルに集約されています。これは設定ソースとアプリケーション間の密結合です。

なぜこれが良いのか？

```
疎結合の設定（Keycloakスタイル）：
  realm.json           → レルム設定
  standalone.xml       → サーバー設定
  環境変数             → オーバーライド
  管理コンソール       → ランタイム変更
  SPI設定              → 拡張機能設定

  質問：「セッションタイムアウトはどこで設定されている？」
  回答：「5箇所のどこかにあるかも。がんばって。」

密結合の設定（voltaスタイル）：
  .env                 → シークレット
  volta-config.yaml    → それ以外すべて

  質問：「セッションタイムアウトはどこで設定されている？」
  回答：「コードの中。設定ファイルはデプロイメントごとに
         異なるもののためにある。」
```

密結合は、見る場所が1つということ。デバッグする場所が1つ。理解する場所が1つ。設定においては、これは欠点ではなく利点です。

---

## 疎結合が良いとき

voltaは内部アーキテクチャには疎結合を使います -- 具体的にはJavaインターフェースを通じて：

```java
// インターフェース（疎結合の境界）
interface SessionStore {
    void createSession(...);
    Optional<SessionRecord> findSession(...);
    void revokeSession(...);
}

// Phase 1：Postgres実装
class PostgresSessionStore implements SessionStore { ... }

// Phase 2：Redis実装（認証ロジックを変更せずにドロップイン）
class RedisSessionStore implements SessionStore { ... }
```

認証ロジックはセッションがPostgresに保存されているかRedisに保存されているか知りませんし、気にしません。`SessionStore`インターフェースとだけ対話します。これは疎結合です -- そして拡張性に不可欠です。

---

## voltaの姿勢：「密結合、謝らない」

voltaは結合の選択について正直です：

| コンポーネント | 結合度 | 理由 |
|---------------|--------|------|
| 設定の集約 | 密 | 見る場所が1つ、デバッグする場所が1つ |
| Java + Javalin | 密 | 完全なスタックトレースの可読性 |
| OIDC + セッション + JWT | 密（1プロセス） | 認証ステップ間のネットワークホップなし |
| SessionStoreインターフェース | 疎 | 認証を書き直さずにPostgresからRedisに交換可能 |
| PolicyEvaluatorインターフェース | 疎 | Phase 4でJavaルールからjCasbinに交換可能 |
| アプリ連携（ForwardAuth） | 疎 | アプリは言語非依存、どのフレームワークでも動作 |

パターン：デバッグと理解を簡素化するところでは密結合、将来の拡張性を可能にするところでは疎結合。これは矛盾ではなく、意識的な設計戦略です。

---

## 「密結合、謝らない」の本当の意味

voltaが「密結合、謝らない」と言うとき、それは：

1. **マイクロサービスの代わりに単一Javaプロセスを選んだ。** OIDCハンドラー、セッションマネージャー、JWT発行者、ForwardAuthエンドポイントがすべて1つのコードベース、1つのプロセス、1つのデプロイメントに存在します。これは密結合です。JWTの発行をセッションマネージャーから独立してスケールすることはできません。voltaはこれについて謝りません。なぜなら、ターゲットのユースケースにおいて、完全なスタックトレース、簡単なデバッグ、サブミリ秒の内部呼び出しという利点が、独立したスケーリングよりも価値があるからです。

2. **分散した設定の代わりに集約された設定を選んだ。** すべてが2つのファイルに。これはオペレーターと設定フォーマット間の密結合です。voltaは謝りません。なぜなら代替案はconfig hellだからです。

3. **すべてを設定可能にする代わりに意見のあるデフォルトを選んだ。** RS256、8時間セッション、PKCE必須。これらはコードベースに密結合されています。voltaは謝りません。なぜならこれらは変更する必要のない正しい選択だからです。

---

## volta-auth-proxyでは

voltaは設定、デプロイメント、プロセスアーキテクチャ（1プロセス、2つの設定ファイル、意見のあるデフォルト）に密結合を、後のフェーズで代替実装が必要になる可能性のある内部インターフェース（`SessionStore`、`PolicyEvaluator`）に疎結合を意図的に使用しています。
