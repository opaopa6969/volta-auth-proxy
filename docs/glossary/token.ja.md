# トークン

[English version](token.md)

---

## これは何？

トークンとは、IDや権限、認証済みであることを表すデータのことです。リクエストのたびにユーザー名とパスワードを送る代わりに、一度認証してトークンを受け取ります。以降はトークンを提示するだけで自分が誰かを証明できます。

コンサートのリストバンドを思い浮かべてください。入口でチケット（認証情報）を見せると、スタッフがリストバンド（トークン）をくれます。会場内では誰もチケットを求めません -- リストバンドを見るだけです。VIPリストバンドならバックステージに入れます。一般入場リストバンドなら客席エリアです。リストバンドがあなたの権限を持ち運んでいるのです。

Webシステムでは、トークンにはさまざまな形があります。不透明な文字列（それ自体では意味のないランダムなID）、[JWT](jwt.md)のような構造化されたトークン（内部にデータを持つ）、[Cookie](cookie.md)に保存されたセッションIDなどです。共通するのは、最初の認証後に認証情報の代わりとなることです。

---

## なぜ重要なのか？

トークンがなければ、すべてのリクエストに完全な認証情報を含める必要があります。パスワードが常にネットワーク上を流れ、サーバーはクリックのたびにパスワードを再検証し、サードパーティへの限定的なアクセス付与もできません。トークンはこれらすべてを解決します：

- **セキュリティ**: 認証情報が露出するのはログイン時の1回だけ。トークンは有効期限を持ち、スコープを限定でき、無効化できる
- **パフォーマンス**: トークン（特にJWT）の検証は、ユーザーの再認証よりはるかに低コスト
- **委譲**: ユーザーのパスワードを共有せずに、限定的な権限のトークンをサードパーティに発行できる

トークンが欠けていたり壊れていたりすると、認証フロー全体が崩壊します。ユーザーはログイン状態を維持できません。APIは呼び出し元を検証できません。マイクロサービス同士が信頼し合えません。

## どう動くのか？

### トークンのライフサイクル

```
  ユーザー                認証サーバー               アプリサーバー
   │                          │                          │
   │  1. ログイン(メール+パス) │                          │
   │ ─────────────────────────>                          │
   │                          │                          │
   │  2. トークンをどうぞ      │                          │
   │ <─────────────────────────                          │
   │                          │                          │
   │  3. リクエスト + トークン  │                          │
   │ ──────────────────────────────────────────────────> │
   │                          │                          │
   │                          │  4. トークン検証          │
   │                          │ <─────────────────────── │
   │                          │  5. 有効！               │
   │                          │ ───────────────────────> │
   │                          │                          │
   │  6. データをどうぞ        │                          │
   │ <────────────────────────────────────────────────── │
```

### トークンの種類

| 種類 | 内容 | 例 | 無効化可能？ |
|------|------|-----|------------|
| **不透明トークン** | ランダム文字列、サーバー参照なしでは意味なし | `550e8400-e29b-41d4...` | はい（DBから削除） |
| **JWT** | 自己完結型、内部に[クレーム](claim.md)を含む | `eyJhbGciOiJSUzI1Ni...` | 直接は不可（期限切れを待つ） |
| **セッションID** | [Cookie](cookie.md)に保存された不透明トークン | `__volta_session=abc123` | はい（DBからセッション削除） |
| **リフレッシュトークン** | 新しいアクセストークン取得用の長寿命トークン | `dGhpcyBpcyBhIHJl...` | はい（DBで無効化） |

### 不透明 vs 自己完結型

```
  不透明トークン（セッションID）:
  ┌────────────────────────┐
  │  550e8400-e29b-41d4... │    ← ただのID。単体では意味なし。
  └────────────────────────┘
           │
           ▼  サーバーがDBで参照する必要あり
  ┌────────────────────────────┐
  │  user_id: alice            │
  │  tenant: acme              │
  │  roles: [admin]            │
  │  expires: 2026-04-01T17:00 │
  └────────────────────────────┘

  自己完結型トークン（JWT）:
  ┌──────────────────────────────────────────┐
  │  header.payload.signature                │
  │                                          │
  │  payload = {                             │
  │    "sub": "alice",                       │
  │    "volta_tid": "acme-uuid",             │
  │    "volta_roles": ["admin"],             │
  │    "exp": 1743530400                     │
  │  }                                       │
  │                                          │
  │  ← データがトークンの中にある              │
  │  ← クレーム読み取りにDB参照不要            │
  │  ← 署名が改ざんされていないことを証明      │
  └──────────────────────────────────────────┘
```

### トークンの有効期限

トークンは永遠に生きるべきではありません。盗まれた場合、有効期限が被害を限定します：

- **短寿命**（volta JWT: 5分）: 悪用の時間枠を最小化。[サイレントリフレッシュ](silent-refresh.md)が必要
- **中寿命**（voltaセッション: 8時間）: セキュリティと利便性のバランス。[スライディングウィンドウ](sliding-window-expiry.md)を使用
- **長寿命**（リフレッシュトークン: 日/週単位）: 便利だがリスク高。安全に保存し、無効化可能にする必要あり

## volta-auth-proxy ではどう使われている？

voltaは2種類のトークンを連携して使います：

### 1. セッショントークン（不透明、Cookie内）

ユーザーがログインすると、voltaはセッションを作成し、`__volta_session` [署名Cookie](signed-cookie.md)にセッションIDを格納します。これは不透明なトークンで、Cookie値は`sessions`テーブルの行にマッピングされるUUIDです。

### 2. JWTアクセストークン（自己完結型、ヘッダー内）

下流のアプリがユーザーを検証する必要があるとき、voltaは短寿命JWT（5分の有効期限）を[RS256](rs256.md)で署名して発行します。JWTには`volta_tid`（テナントID）、`volta_roles`、`volta_display`などの[クレーム](claim.md)が含まれます。

```java
// JwtService.java -- トークン発行
JWTClaimsSet claims = new JWTClaimsSet.Builder()
        .subject(principal.userId().toString())
        .expirationTime(Date.from(now.plusSeconds(config.jwtTtlSeconds())))
        .claim("volta_tid", principal.tenantId().toString())
        .claim("volta_roles", principal.roles())
        .build();
SignedJWT jwt = new SignedJWT(header, claims);
jwt.sign(new RSASSASigner(rsaKey.toPrivateKey()));
```

### なぜ2種類のトークン？

- **セッションCookie**はDB行を削除すれば即座に無効化できるが、サーバーへのラウンドトリップが必要
- **JWT**は無効化できないが、自己検証可能（DBアクセス不要）で短寿命（最大5分の被害）
- 両方を組み合わせることで、即座の無効化（セッション経由）とステートレスな検証（JWT経由）の両方を実現

## よくある間違いと攻撃

### 間違い1: 有効期限のないトークン

有効期限のないトークンは永久のマスターキーです。盗まれたら、攻撃者は永遠にアクセスできます。必ず有効期限を設定しましょう。voltaのJWTは5分、セッションは無操作8時間で期限切れになります。

### 間違い2: localStorageへのトークン保存

`localStorage`はページ上のどのJavaScriptからもアクセス可能です。XSS攻撃ですべてのトークンが盗まれます。voltaはセッショントークンをHttpOnly Cookie（JavaScriptから不可視）に保存し、JWTはブラウザに永続化しません。

### 間違い3: トークンのクレームを検証しない

`iss`（発行者）、`aud`（対象者）、`exp`（有効期限）を確認せずにトークンを受け入れると、他のシステムからのトークンや期限切れトークンを受け入れてしまいます。voltaの`JwtService.verify()`は3つすべてを検証します。

### 攻撃: トークンの窃取

攻撃者が有効なトークンを入手すると、ユーザーになりすませます。対策: 短い有効期限、HTTPS限定、HttpOnly Cookie、[トークン窃取](token-theft.md)検知。[セッションハイジャック](session-hijacking.md)も参照。

### 攻撃: トークンの偽造

適切な[検証](verification.md)がなければ、攻撃者は偽のトークンを作れます。JWTは[暗号署名](cryptographic-signature.md)でこれを防ぎます -- ただし、検証者が署名アルゴリズムを確認し、正しい[署名鍵](signing-key.md)を使う場合に限ります。
