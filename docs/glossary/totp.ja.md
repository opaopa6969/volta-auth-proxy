# TOTP（時間ベースワンタイムパスワード）

[English version](totp.md)

---

## これは何？

TOTPは、30秒ごとに変わるワンタイムパスワードを生成する方法です。Google AuthenticatorやAuthyを開いてカウントダウンする6桁のコードが見えたら、それがTOTPです。スマホとサーバーが秘密を共有し、現在の時刻を使って独立して同じコードを生成します。

同期した時計とコードブックのようなものです。あなたとサーバーが同じコードブックを持っています。30秒ごとに、両者が現在の時刻とコードブックを見て同じ6桁のコードを生み出します。会話を盗聴している人は次のコードを推測できません。コードブックを持っていないからです。

---

## なぜ重要なのか？

TOTPは「持っているもの」要素（スマホ）を提供し、以下の特徴があります：

- **オフライン対応：** インターネットや電波なしで動作（SMSと違って）
- **標準的：** どのTOTPアプリでも動作（Google Authenticator、Authy、1Passwordなど）
- **無料：** SMS送信料なし、ベンダーロックインなし
- **SIMスワッピング耐性：** SMSコードと違い、電話番号に紐づかない

TOTPは世界で最も広く展開されている第二要素です。

## どう動くのか？

### セットアップ（1回だけ）

```
  ステップ1：サーバーがランダムな秘密を生成（通常20バイト）

    secret = "JBSWY3DPEHPK3PXP"（base32エンコード）

  ステップ2：サーバーが以下を含むQRコードを表示：

    otpauth://totp/volta:taro@acme.com
      ?secret=JBSWY3DPEHPK3PXP
      &issuer=volta
      &algorithm=SHA1
      &digits=6
      &period=30

  ステップ3：ユーザーが認証アプリでQRコードをスキャン

    アプリが秘密を保存。サーバーも秘密を保存。
    両者が同じ秘密を持つ。

  ステップ4：ユーザーが確認コードを入力してセットアップを確認

    アプリがQRコードを正しくスキャンしたことの証明。
```

### コード生成（30秒ごと）

```
  入力：
    secret = "JBSWY3DPEHPK3PXP"（共有秘密）
    time   = 現在のUnixタイムスタンプ / 30（タイムステップカウンター）

  アルゴリズム：
    1. counter = floor(unixタイムスタンプ / 30)
       例: floor(1711899700 / 30) = 57063323

    2. hmac = HMAC-SHA1(secret, counterの8バイト表現)
       結果: 20バイトのHMAC出力

    3. offset = hmacの最後のニブル（0-15）
       そのオフセットから4バイトを抽出

    4. code = extract_31_bits(hmac[offset:offset+4]) mod 1000000
       結果: 6桁の数値

  例：
    時刻: 2026-03-31T09:00:00Z → counter = 57063323
    秘密: JBSWY3DPEHPK3PXP
    コード: 847293

    30秒後：
    時刻: 2026-03-31T09:00:30Z → counter = 57063324
    コード: 159482（完全に異なる）
```

### 検証

```
  ユーザーが入力: 847293
  サーバーが計算：
    - 現在のタイムステップのコード: 847293  ← 一致！
    - 前のタイムステップのコード:   391057  （時計のずれに±1を許容）
    - 次のタイムステップのコード:   159482  （時計のずれに±1を許容）

  入力されたコードが3つ（現在、前、次）のいずれかと一致すれば：
    → 有効！認証成功。

  一致しなければ：
    → 無効。再試行（レート制限あり）。
```

### なぜTOTPが安全か

```
  攻撃者に必要なもの：            それがある場所：
  ┌──────────────────────────┐   ┌──────────────────────────┐
  │ 共有秘密                  │   │ サーバー: DBで暗号化       │
  │                          │   │ スマホ: 認証アプリの       │
  │                          │   │        安全なストア内      │
  └──────────────────────────┘   └──────────────────────────┘

  攻撃者が見えるもの：
  ┌──────────────────────────┐
  │ 6桁のコード               │  ← 30-60秒間のみ有効
  │（肩越しに覗いた場合）      │  ← ここから秘密は導出できない
  │                          │  ← 次のコードは予測できない
  └──────────────────────────┘
```

攻撃者が1つのコードを見ても、再使用できず（期限切れ）、将来のコードも導出できません（HMACは一方向）。

## volta-auth-proxyはTOTPをどう使う予定か？

### 現在の状態（Phase 1）

TOTPはPhase 1では**実装されていません**。voltaは依存関係（pom.xml）に`googleauth`ライブラリを既に含んでいますが、TOTPの登録・検証フローはPhase 3で計画されています。

### Phase 3の計画

```
  TOTP登録フロー：
  ══════════════

  1. ユーザーが/settings/securityに移動
  2. 「TOTPを有効化」をクリック
  3. voltaがランダムな秘密を生成
  4. QRコード + 手動入力キーを表示
  5. ユーザーが認証アプリでスキャン
  6. ユーザーが確認コードを入力
  7. voltaがコードを秘密に対して検証
  8. 有効なら：暗号化した秘密をDBに保存
     ユーザーアカウントに関連付け
  9. リカバリコードを生成して表示
     （1回限りの使用、スマホ紛失時用）

  TOTP検証フロー（ログイン時）：
  ═══════════════════════════

  1. ユーザーがGoogle OIDCログインを完了
  2. voltaチェック：このユーザーはTOTPが有効か？
  3. はいの場合：TOTPチャレンジページを表示
  4. ユーザーがアプリから6桁のコードを入力
  5. voltaが保存された秘密に対して検証
  6. 有効なら：ログイン完了、amr=["google_oidc","totp"]を設定
  7. 無効なら：エラーを表示（レート制限あり）

  テナントでの強制：
  ══════════════════

  テナント管理者がポリシーを設定可能：
  - 「MFAオプション」（デフォルト）
  - 「ADMINとOWNERにMFA必須」
  - 「全メンバーにMFA必須」

  「必須」テナントに所属するMFA未設定ユーザーは、
  次回ログイン時にTOTPのセットアップを促されます。
```

### リカバリコード

ユーザーがTOTPを有効化すると、voltaは8-10個の使い捨てリカバリコードを生成します：

```
  リカバリコード（安全な場所に保管してください）：

  a4k2-m8p3-x7w1
  b9n5-j2q8-r6t4
  c1v7-h3s9-y5u2
  d6e8-f4g1-z3k7
  ...

  各コードは1回しか使えません。
  認証アプリにアクセスできなくなった場合に使ってください。
```

## よくある間違いと攻撃

### 間違い1：リカバリコードを提供しない

ユーザーがスマホを紛失してリカバリコードがなければ、永久にロックアウトされます。TOTP設定時に必ず使い捨てリカバリコードを提供してください。

### 間違い2：TOTP秘密を暗号化せずに保存する

データベース内のTOTP共有秘密は暗号化しなければなりません。データベースが漏洩した場合、暗号化されていない秘密があれば攻撃者がすべてのユーザーの有効なコードを生成できます。

### 間違い3：TOTP試行をレート制限しない

6桁のコードには1,000,000通りの可能性があります。レート制限なしでは、攻撃者は数分でブルートフォースできます。1分あたり5回の試行に制限してください。

### 間違い4：時計のずれを無制限に許容する

TOTPは時計のずれに対するウィンドウを許容します（通常±1タイムステップ = ±30秒）。多くのタイムステップを受け入れすぎると（例：±5 = 5分）、以前使ったコードがまだ有効になりえます。

### 攻撃：リアルタイムフィッシング

攻撃者が本物のログインページをプロキシし、TOTPコードをキャプチャして即座にリプレイ。TOTPはコードが特定のドメインに紐づいていないため、これに脆弱です。WebAuthn（パスキー）がこの問題を解決します。[webauthn.md](webauthn.md)参照。
