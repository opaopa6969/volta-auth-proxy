# トレードオフ（Trade-off）

[English version](tradeoff.md)

---

## これは何？

トレードオフとは、1つのことを改善すると必然的に別のことが悪化する状況です。すべてを手に入れることはできません。すべての選択にはコストがあります。エンジニアリングとは、どのコストを受け入れるかを選ぶ技術です。

住む場所を選ぶようなものです。郊外の大きな家はスペースをくれますが通勤時間がかかります。都心の小さなアパートは利便性をくれますがスペースが犠牲になります。安い価格で通勤ゼロの大きな家は手に入りません。何が最も重要かを選び、失うものを受け入れる必要があります。

---

## なぜ重要なのか？

ソフトウェアエンジニアリングはトレードオフでできています。速いか信頼性が高いか？シンプルか柔軟か？安いか機能豊富か？すべてを手に入れられると偽るチームは、結局遅くて複雑で高価で信頼性も低いシステムを作ります -- 何を犠牲にするか意識的に選ばなかったからです。

volta-auth-proxyは興味深い存在です。トレードオフについて正直だからです。すべてになろうとはしません。「これが得られるもの。これが諦めるもの。それに応じて選んでください」と言います。

## voltaの具体的なトレードオフ

### トレードオフ1: コントロール vs 利便性

```
  Auth0 / Clerk:                       volta-auth-proxy:
  ┌──────────────────────┐             ┌──────────────────────┐
  │ 高い利便性           │             │ 高いコントロール     │
  │                      │             │                      │
  │ ✓ ダッシュボード設定 │             │ ✓ 完全なソースコード │
  │ ✓ あらゆるSDK        │             │ ✓ あらゆる動作を変更 │
  │ ✓ マネージドホスティング│          │ ✓ データを自分で所有 │
  │ ✓ セキュリティチーム │             │ ✓ ベンダーロックインなし│
  │                      │             │                      │
  │ ✗ セルフホスト不可   │             │ ✗ 自分でホスト       │
  │ ✗ コード変更不可     │             │ ✗ 自分で設定         │
  │ ✗ 料金がスケール     │             │ ✗ 自分でセキュリティ │
  │ ✗ ベンダーロックイン │             │ ✗ 自分でメンテナンス │
  └──────────────────────┘             └──────────────────────┘
```

voltaは利便性をコントロールと交換します。ボタンをクリックして5分で認証をセットアップすることはできません。でもコードのすべての行を読め、あらゆる動作を変更でき、ベンダーの条件変更を心配する必要がありません。

### トレードオフ2: セキュリティ責任 vs 自由

これが最大のトレードオフです。Auth0を使うとき、彼らのセキュリティチームが脆弱性を監視し、CVEに対応し、エクスプロイトをパッチします。月額$2,400を支払い、その一部がセキュリティ専門知識を買っています。

voltaを使うとき、**あなたがセキュリティチームです。**

```
  Auth0:                               volta:
  ┌──────────────────────┐             ┌──────────────────────┐
  │ 認証ライブラリにCVE？ │             │ 認証ライブラリにCVE？ │
  │ Auth0がパッチする。   │             │ あなたがパッチする。  │
  │ あなたは何もしない。  │             │ 依存関係を更新する。  │
  │                      │             │ 再デプロイする。      │
  │                      │             │ 修正を検証する。      │
  └──────────────────────┘             └──────────────────────┘
```

これは小さなことではありません。セキュリティ責任は実際の仕事です。voltaは意識的にこのトレードオフを行います：月額$0、しかしセキュリティの負担はあなたのもの。

### トレードオフ3: 機能 vs シンプルさ

voltaは意図的にKeycloakより少ない機能を持ちます：

| 機能 | Keycloak | volta |
|------|----------|-------|
| SAML 2.0 IdP | はい | Phase 3（計画中） |
| LDAP/ADフェデレーション | はい | なし |
| カスタム認証フロー | はい（複雑） | なし（1つのフロー：OIDC + PKCE） |
| ユーザーセルフ登録 | はい | OIDC経由のみ（Googleが処理） |
| パスワード管理 | はい | なし（IdPに委譲） |
| アカウント管理ポータル | はい | なし（API経由） |

voltaはシンプルさを選びました。少ない機能はより少ないコード、より少ないバグ、より速い起動、より簡単な理解を意味します。でも明日LDAPフェデレーションが必要なら、今日のvoltaにはできません。

### トレードオフ4: 意見のある設計 vs 柔軟性

voltaはあなたの代わりに選択します：

| 決定事項 | voltaの選択 | 代替案 |
|---------|------------|--------|
| JWTアルゴリズム | RS256のみ | RS384、RS512、ES256のサポートも可能... |
| セッションタイムアウト | 8時間スライディングウインドウ | テナントごとに設定可能にもできる |
| 最大セッション数 | ユーザーあたり5 | 設定可能にもできる |
| JWT有効期限 | 5分 | 設定可能にもできる |
| OIDCフロー | Authorization Code + PKCE | Implicit、Hybridもサポート可能 |

各「可能」は、誰かが学び、ドキュメント化し、テストし、デバッグしなければならない設定です。voltaは言います：「RS256が正しい選択。8時間が正しいタイムアウト。私たちが決めたので、あなたが決める必要はありません。」

トレードオフ：voltaの意見があなたのユースケースに合わない場合、ソースコードを変更する必要があります。変更する設定はありません。

### トレードオフ5: シングルプロセス vs 分散

voltaは単一プロセスとして動作します。これはシンプルですが水平スケーリングに制限があります：

```
  volta（シングルプロセス）：
  ┌──────────────────┐
  │ volta-auth-proxy  │  1つのプロセスがすべてを処理。
  │ + Postgres        │  シンプル。スケーリングに限界。
  └──────────────────┘

  分散（例：Ory）：
  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │Proxy │ │Login │ │OAuth │ │Perms │  4つのプロセス。
  │      │ │      │ │      │ │      │  複雑。スケーラブル。
  └──────┘ └──────┘ └──────┘ └──────┘
```

数千（数百万ではなく）のユーザーを持つ大半のSaaS製品では、単一のPostgreSQLインスタンスの前の単一のvoltaインスタンスで十分すぎます。しかし1,000万DAUの次のSlackを構築しているなら、voltaのアーキテクチャには適応が必要です。

## トレードオフについての考え方

最悪のエンジニアリング判断は、チームがトレードオフを認めることを拒否したときに起こります：

- 「Auth0の利便性もセルフホスティングもゼロコストも欲しい」-- 2つを選んでください。
- 「Keycloakの機能もvoltaのシンプルさも欲しい」-- 不可能です。
- 「最大のセキュリティもゼロの運用努力も欲しい」-- Auth0に払うか、仕事をしてください。

最良のエンジニアリング判断は、チームが明確に述べるときに起こります：

- 「利便性より**コントロール**を重視する。ホスティングの負担を受け入れる。」
- 「機能より**シンプルさ**を重視する。制限を受け入れる。」
- 「速度より**理解**を重視する。学習の投資を受け入れる。」

voltaはこれらの特定の選択をするチーム向けです。あなたの優先順位が異なるなら、異なるツールが正しいツールです。
