# アップストリーム

[English version](upstream.md)

---

## これは何？

リバースプロキシの文脈では、「アップストリーム」はプロキシの背後にあり、転送されたリクエストを受信するサーバーを指します。volta-auth-proxyがリクエストを認証して転送するとき、あなたのアプリケーションサーバーが「アップストリーム」です。クライアントはアップストリームと直接通信しません。voltaが間に入り、リクエストを通す前に認証を処理します。

病院の受付のようなものです。あなた（クライアント）はまず受付（volta-auth-proxy）と話します。受付が身分証を確認し、予約を検証してから、医師（アップストリーム）に案内します。医師はIDチェックを心配せずに自分の仕事（アプリのビジネスロジック）に集中できます。受付は「ダウンストリーム」（クライアントに近い）で、医師は「アップストリーム」（クライアントから遠い、受付の背後）です。

注意：「アップストリーム」はクライアントから遠い方を意味するため、用語が紛らわしい場合があります。名前はデータが「上流に」流れることに由来します。リクエストがクライアントからオリジンサーバーに向かって流れます。

---

## なぜ重要なのか？

アップストリーム関係はvoltaのアーキテクチャにおけるセキュリティ境界を定義します：

```
  インターネット（非信頼）  volta-auth-proxy（信頼境界）    アップストリーム（信頼）
  ==================    ================================    ==================

  ┌──────────┐           ┌────────────────────────┐          ┌──────────────┐
  │ クライアント│──────────►│  volta-auth-proxy      │─────────►│  あなたのアプリ│
  │（ブラウザ）│           │                        │          │（アップストリーム）│
  │          │           │  1. セッション/JWT確認   │          │              │
  │          │           │  2. テナント検証         │          │  受信：       │
  │          │           │  3. 停止チェック         │          │  - 元の      │
  │          │           │  4. ABAC評価            │          │    リクエスト  │
  │          │           │  5. X-Volta-*           │          │  - X-Volta-* │
  │          │           │     ヘッダー追加         │          │    ヘッダー   │
  │          │           │  6. リクエスト転送       │          │              │
  │          │           │                        │          │  信頼：       │
  │          │           │  または：401/403で拒否   │          │  X-Volta-*   │
  └──────────┘           └────────────────────────┘          └──────────────┘
```

アップストリームアプリは`X-Volta-*` [ヘッダー](header.md)を信頼できます：

1. voltaがクライアントが提供した`X-Volta-*`ヘッダーを転送前にすべて除去する
2. voltaだけがこれらのヘッダーを設定できる
3. アップストリームはインターネットから直接アクセスできない

## どう動くのか？

### アップストリーム設定

voltaは`volta-config.yaml`でアップストリームサーバーのアドレスを設定します：

```yaml
# volta-config.yaml
upstream:
  url: "http://localhost:8081"    # あなたのアプリ
  timeout_ms: 30000               # 30秒タイムアウト
  health_check: "/health"         # オプションのヘルスエンドポイント
```

### リクエスト転送

```
  クライアントリクエスト：
    GET /api/v1/tenants/acme/members
    Cookie: volta_session=abc123
    X-Custom-Header: my-value

  volta処理：
    1. セッションCookieを検証 → ユーザー認証済み
    2. クライアントリクエストからX-Volta-*ヘッダーを除去
    3. 検証済みアイデンティティのX-Volta-*ヘッダーを追加
    4. アップストリームに転送

  アップストリームに転送：
    GET /api/v1/tenants/acme/members
    X-Custom-Header: my-value           ← 保持
    X-Volta-User-Id: user-uuid          ← voltaが追加
    X-Volta-Email: alice@acme.com       ← voltaが追加
    X-Volta-Roles: ADMIN                ← voltaが追加
    X-Volta-Tenant-Id: acme-uuid        ← voltaが追加
    X-Volta-M2M: false                  ← voltaが追加
```

### voltaがアップストリームに送信するヘッダー

| ヘッダー | 内容 | 例 |
|---------|------|-----|
| `X-Volta-User-Id` | 認証済みユーザーのUUID | `550e8400-...` |
| `X-Volta-Email` | ユーザーのメール | `alice@acme.com` |
| `X-Volta-Roles` | カンマ区切りの[ロール](role.md) | `ADMIN,MEMBER` |
| `X-Volta-Tenant-Id` | [テナント](tenant.md)UUID | `acme-uuid` |
| `X-Volta-M2M` | [M2M](m2m.md)リクエストか？ | `true` / `false` |
| `X-Volta-Client-Id` | M2Mクライアント名（M2Mの場合） | `billing-service` |
| `X-Volta-Request-Id` | トレーシング用の一意リクエストID | `req-uuid` |

### アップストリームレスポンスの処理

voltaはアップストリームのレスポンスをクライアントに変更せずに返します：

```
  アップストリームレスポンス：
    HTTP 200 OK
    Content-Type: application/json
    {"members": [...]}

  voltaが追加：
    Cache-Control: no-store      （セキュリティヘッダー）
    X-Volta-Request-Id: req-uuid （トレーシング）

  クライアントが受信：
    HTTP 200 OK
    Content-Type: application/json
    Cache-Control: no-store
    X-Volta-Request-Id: req-uuid
    {"members": [...]}
```

### 複数アップストリーム（パスベースルーティング）

voltaはリクエストパスに基づいて異なるアップストリームにルーティングできます：

```yaml
# volta-config.yaml
upstream:
  routes:
    - path: "/api/v1/*"
      url: "http://main-app:8081"
    - path: "/api/v2/*"
      url: "http://v2-app:8082"
    - path: "/admin/*"
      url: "http://admin-app:8083"
```

```
  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │ クライアント│────►│ volta-auth-  │──┬─►│ main-app     │
  │          │     │ proxy        │  │  │ :8081        │
  │ /api/v1/ │     │              │  │  └──────────────┘
  │ /api/v2/ │     │  パスプレフィ │  │  ┌──────────────┐
  │ /admin/  │     │  ックスで     │  ├─►│ v2-app       │
  │          │     │  ルーティング │  │  │ :8082        │
  └──────────┘     └──────────────┘  │  └──────────────┘
                                     │  ┌──────────────┐
                                     └─►│ admin-app    │
                                        │ :8083        │
                                        └──────────────┘
```

## volta-auth-proxyではどう使われている？

### ForwardAuthパターン

voltaはTraefikやNginxなどのリバースプロキシでよく使われるForwardAuthパターンを実装します：

```
  オプション1：voltaがスタンドアロンプロキシ（直接）
  ┌────────┐     ┌──────────────┐     ┌──────────┐
  │クライアント│────►│ volta-auth-  │────►│アップ     │
  │        │     │ proxy        │     │ストリーム │
  └────────┘     └──────────────┘     └──────────┘

  オプション2：ロードバランサーの背後（ForwardAuth）
  ┌────────┐     ┌─────────┐  認証?  ┌──────────────┐
  │クライアント│────►│ Traefik │────────►│ volta-auth-  │
  │        │     │ (LB)    │◄────────│ proxy        │
  │        │     │         │ 200 OK  │ （認証のみ）  │
  │        │     │         │         └──────────────┘
  │        │     │         │────────►┌──────────┐
  │        │     │         │         │アップ     │
  └────────┘     └─────────┘         │ストリーム │
                                     └──────────┘
```

### アップストリーム信頼モデル

アップストリームアプリはvoltaのヘッダーを信頼しなければならず、インターネットから直接アクセスできてはなりません：

```
  正しい：
  インターネット → volta → アップストリーム（プライベートネットワーク）
  アップストリームはX-Volta-*ヘッダーを信頼 ✓

  間違い：
  インターネット → アップストリーム（直接アクセス可能！）
  誰でもX-Volta-*ヘッダーを偽造可能 ✗
```

### ヘルスチェック

voltaはアップストリームの健全性を監視できます：

```
  volta → GET http://upstream:8081/health → 200 OK?
    YES → 通常通りリクエストを転送
    NO  → クライアントに503を返し、運用にアラート
```

### タイムアウト処理

アップストリームが遅い場合、voltaがタイムアウトを適用：

```
  クライアント → volta → アップストリーム（処理中...）
                        ↓
                30秒経過
                        ↓
  volta → 504 Gateway Timeout → クライアント
```

## よくある間違いと攻撃

### 間違い1：アップストリームをインターネットに直接公開する

クライアントがvoltaを経由せずにアップストリームに到達できると、`X-Volta-*`ヘッダーを偽造して任意のユーザーになりすませます。アップストリームはvoltaのネットワークからのみアクセスできるようにしてください。

### 間違い2：クライアントが提供したX-Volta-*ヘッダーを除去しない

voltaが元のリクエストから`X-Volta-*`ヘッダーを除去しないと、クライアントが`X-Volta-Roles: ADMIN`を設定してアップストリームが信頼してしまいます。voltaは独自のヘッダーを追加する前に常にこれらを除去します。

### 間違い3：アップストリームが異なる方法で独自の認証チェックを行う

アップストリームが異なるメカニズム（例：別のトークンの検証）で再認証すると、認証結果が矛盾する可能性があります。voltaのヘッダーだけを信頼してください。それがプロキシパターンの全目的です。

### 間違い4：タイムアウト設定がない

タイムアウトがないと、遅いアップストリームがvoltaに接続を無期限に保持させ、リソースを枯渇させます。常にアップストリームタイムアウトを設定してください。

### 攻撃：アップストリーム設定ミスによるSSRF

voltaのアップストリームURLがユーザー制御可能（例：ヘッダーから）な場合、攻撃者がリクエストを内部サービスにリダイレクトできます。防御：アップストリームURLはオペレーターが`volta-config.yaml`で設定し、リクエストデータから導出されることはありません。

### 攻撃：ヘッダーインジェクション

攻撃者がヘッダー値に改行を含むリクエストを送り、追加ヘッダーを注入する。防御：voltaが転送前にヘッダー値をサニタイズ。
