# ベンダーロックイン

[English version](vendor-lock-in.md)

---

## これは何？

ベンダーロックインとは、特定の会社の製品に依存しすぎて、代替品への切り替えが極めて困難、高額、または事実上不可能になることです。

昔の携帯電話の契約で考えてみましょう。2年契約を結んだとします。「無料」の端末をもらいました。でも気づきます：電話番号はそのキャリアに紐づいている、解約料がかかる、端末はSIMロックされている、家族割のためにみんな同じプランに入っている。別のキャリアに乗り換えるには、番号を失い、違約金を払い、新しい端末を買い、家族全員を説得しなければなりません。だから残る -- サービスが悪くても、料金が上がり続けても。

ソフトウェアのベンダーロックインもまったく同じです。簡単だから使い始めます。時間が経つにつれ、コード、データ、ワークフローがそのサービスと深く絡み合います。離れるのが月を追うごとに難しくなります。

---

## なぜ重要なのか？

ロックインが危険なのは、交渉力を失うからです。簡単に離れられないとベンダーが分かれば、こうなります：

1. **値上げ** -- 受け入れるしかない（さもなくば何か月もかけて移行する）
2. **条件変更** -- 新しい制限、新しい制約、嫌なら出ていけ
3. **機能廃止** -- 頼りにしている機能が消える
4. **障害発生** -- 向こうのサービスが落ちるとあなたのプロダクト全体が止まる。しかもどうしようもない

### Auth0やClerkがロックインする仕組み

| ロックインの仕組み | どう機能するか |
|-------------------|-------------|
| **独自SDK** | フロントエンドが`@auth0/auth0-spa-js`や`@clerk/nextjs`を使う。すべてのページ、すべてのコンポーネントに彼らのコードが織り込まれる。 |
| **ユーザーデータベース** | ユーザーレコードがすべてAuth0/Clerkのクラウドにある。エクスポートするとパスワードハッシュを失う（Auth0からbcryptハッシュはエクスポートできない）。 |
| **Webhookフォーマット** | 彼ら独自のイベントフォーマットに合わせて自動化を構築。新しいプロバイダはイベント名もペイロードも違う。 |
| **カスタムルール/アクション** | Auth0の「Actions」やClerkの「webhooks」は独自APIを使う。何も移行できない。 |
| **セッション管理** | 彼らのセッションCookie、彼らのトークンフォーマット、彼らのリフレッシュフロー。切り替え = 全ユーザーがログアウト。 |
| **ログインUI** | Auth0のUniversal Login、Clerkの`<SignIn />`コンポーネント。デザインが彼らのコンポーネント前提で構築されている。 |

結果：認証プロバイダの切り替えは通常2〜6か月のエンジニアリングプロジェクトになります。ほとんどのチームは切り替えません。ただ払い続けます。

## セルフホストでどう回避するか？

認証をセルフホストすれば（volta-auth-proxyのように）、すべてを自分が所有します：

```
  クラウドホスト型（Auth0/Clerk）：    セルフホスト型（volta）：
  ┌──────────────────────┐           ┌──────────────────────┐
  │  彼らのサーバー       │           │  あなたのサーバー     │
  │  彼らのデータベース    │           │  あなたのデータベース  │
  │  彼らのSDK           │           │  標準ヘッダー         │
  │  彼らのルールエンジン  │           │  あなたのコード       │
  │  彼らのログインUI     │           │  あなたのテンプレート  │
  │  彼らの料金体系       │           │  あなたの料金（$0）   │
  └──────────────────────┘           └──────────────────────┘
       すべてをレンタル。                  すべてを所有。
```

voltaの場合：

- **ユーザーデータ**は自分のPostgreSQLデータベースにある。標準SQLでクエリ、バックアップ、移行できる。
- **独自SDKなし** -- アプリは標準HTTPヘッダー（`X-Volta-User-Id`、`X-Volta-Tenant-Id`）を読む。voltaから離れるには、同じヘッダーをセットする新しい認証サービスを書けばよい。
- **ログインUI**は完全にコントロール可能な[jte](jte.ja.md)テンプレート。隠れた動作を持つ「テーマ対応」コンポーネントではない。
- **設定**は`.env`ファイルとシンプルな`volta-config.yaml`。他人のサーバーにしか存在しないクラウドダッシュボードではない。

## voltaのアプローチ

voltaは設計上、セルフホスト専用です。voltaのクラウドサービスもSaaSプランもありません。これは意図的です：

1. **データを所有する。** ユーザーレコード、セッション、監査ログ -- すべてあなたのPostgresに。
2. **コードを所有する。** voltaはオープンソース。明日プロジェクトが消えても、動く認証システムは手元に残る。
3. **デプロイを所有する。** AWS、GCP、$5のVPS、Raspberry Piで動かせる。誰も値上げできない。
4. **標準的な統合。** voltaは[ForwardAuth](forwardauth.ja.md)と標準HTTPヘッダーを使う。アプリはvolta専用SDKをインポートしない。結合度は最小限。

トレードオフは現実にあります：運用（稼働率、バックアップ、セキュリティパッチ）は自分の責任です。ただし、コントロールは自分にあります。

## ロックインリスクの評価方法

ツールやサービスを選ぶとき、自分に問いかけましょう：

1. **データをエクスポートできるか？**（パスワードハッシュやセッションも含めて全部？）
2. **自分で動かせるか？**（セルフホストオプションはあるか？）
3. **コードがそのSDKに依存しているか？**（標準プロトコルを使っているか？）
4. **料金が2倍になったらどうなるか？**（1か月以内に切り替えられるか？）
5. **サービスが終了したらどうなるか？**（すべて失うか？）

これらの大半に「いいえ」と答えるなら、ロックインされています。
