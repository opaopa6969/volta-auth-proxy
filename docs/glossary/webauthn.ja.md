# WebAuthn / FIDO2 / パスキー

[English version](webauthn.md)

---

## これは何？

WebAuthn（Web Authentication）は、パスワードの代わりに（または追加で）、セキュリティキー（YubiKeyなど）、デバイスの指紋リーダー、顔認証を使ってウェブサイトにログインできる標準規格です。「パスキー」と言えば、デバイス間で同期するWebAuthn認証情報（iCloudキーチェーン、Googleパスワードマネージャなど経由）のことです。

あなたの指紋でしか開かない鍵のようなものです。盗むパスワードもなく、フィッシングするコードもありません。あなたのデバイスがあなた本人であることを暗号的なチャレンジで証明し、しかもその鍵を登録した特定のウェブサイトに対してのみ行います。

---

## なぜ重要なのか？

WebAuthnはパスワードとTOTPの2つの最大の問題を解決します：

1. **フィッシング耐性。** 認証情報はウェブサイトのドメインに暗号的に紐づけられています。攻撃者が`g00gle.com`に偽のログインページを作っても、あなたのパスキーは`google.com`に登録されているのでそこでは動きません。TOTPをバイパスするリアルタイムフィッシング攻撃が不可能になります。

2. **共有秘密がない。** パスワードやTOTPと異なり、サーバーに盗まれうる秘密が保存されていません。サーバーが保存するのは公開鍵だけです。秘密鍵はデバイスの外に出ることがありません。

```
  パスワード: 共有秘密 → サーバーから盗まれうる
  TOTP:       共有秘密 → サーバーから盗まれうる
  WebAuthn:   サーバーには公開鍵のみ → 盗んでも役に立たない
```

## どう動くのか？

### 登録（1回だけのセットアップ）

```
  ユーザー                      ブラウザ                   volta-auth-proxy
  ========                      ========                   ================

  1. /settings/securityに移動
     「パスキーを登録」をクリック
                               ──────────────────────────►
                                                           2. チャレンジを生成
                                                              (ランダムバイト)
                                                              チャレンジを保存
                               ◄──────────────────────────
                                                           3. チャレンジ +
                                                              サーバー情報を送信

                               4. ブラウザが
                                  navigator.credentials.create()
                                  を呼び出す

  5. デバイスがプロンプト：
     「セキュリティキーにタッチ」
     または「Face IDを使用」
     または「指紋を使用」

  6. ユーザーが生体認証/タッチで認証

                               7. デバイスが新しい鍵ペアを生成：
                                  - 秘密鍵（デバイスに残る、絶対に出ない）
                                  - 公開鍵（サーバーに送られる）

                               8. デバイスが新しい秘密鍵で
                                  チャレンジに署名

                               9. デバイスが返すもの：
                                  - 公開鍵
                                  - 署名済みチャレンジ
                                  - 認証情報ID
                                  - アテステーションデータ

                               ──────────────────────────►
                                                           10. 署名を検証
                                                               公開鍵 + 認証情報ID
                                                               をDBに保存
                               ◄──────────────────────────
                                                           11. 「パスキー登録完了！」
```

### 認証（ログインのたび）

```
  ユーザー                      ブラウザ                   volta-auth-proxy
  ========                      ========                   ================

  1. /loginに移動
     「パスキーでログイン」をクリック
                               ──────────────────────────►
                                                           2. 新しいチャレンジを生成
                                                              ユーザーの認証情報ID
                                                              を検索
                               ◄──────────────────────────
                                                           3. チャレンジ +
                                                              許可された認証情報ID
                                                              を送信

                               4. ブラウザが
                                  navigator.credentials.get()
                                  を呼び出す

  5. デバイスがプロンプト：
     「セキュリティキーにタッチ」
     または「Face IDを使用」

  6. ユーザーが認証

                               7. デバイスがこのドメインに
                                  一致する秘密鍵を検索

                               8. デバイスが秘密鍵で
                                  チャレンジに署名

                               9. 署名済みチャレンジ
                                  + 認証情報IDを返す

                               ──────────────────────────►
                                                           10. 認証情報IDで
                                                               公開鍵を検索
                                                           11. 公開鍵で署名を検証
                                                           12. 有効なら：ログイン！
```

### なぜフィッシングが通用しないか

```
  本物のサイト: volta.example.com
  偽サイト: v0lta.example.com（攻撃者のフィッシングページ）

  パスキー登録時：
    credential.domain = "volta.example.com"

  ユーザーが偽サイトを訪問した場合：
    ブラウザ: 「デバイスさん、v0lta.example.comの認証情報ある？」
    デバイス: 「ないよ。volta.example.comのならあるけど。」
    結果: 認証失敗。ユーザーはフィッシングされない。

  TOTPの場合、ユーザーは偽サイトにコードを入力し、
  攻撃者がそれを本物のサイトにリレーする。
  WebAuthnでは、これは構造的に不可能。
```

### パスキー vs ハードウェアセキュリティキー

| 特徴 | ハードウェアキー（YubiKey） | パスキー（同期型） |
|------|-------------------------|------------------|
| 秘密鍵の場所 | 物理キーの中 | iCloud/Googleクラウド |
| デバイス紛失 | バックアップキーが必要 | 新しいデバイスに自動同期 |
| クロスデバイス | キーを差し込む必要 | すべての同期デバイスで利用可能 |
| フィッシング耐性 | はい | はい |
| 購入が必要 | はい（約25-50ドル） | いいえ（OSに内蔵） |

## volta-auth-proxyはWebAuthnをどう使う予定か？

### 現在の状態（Phase 1）

WebAuthnはPhase 1では**実装されていません**。voltaは認証にGoogle OIDCを使用しています。

### Phase 2の計画：第二要素としてのパスキー

```
  Phase 2 WebAuthnフロー：

  1. ユーザーがGoogle OIDCでログイン（一次認証）
  2. voltaチェック：このユーザーにパスキーが登録されているか？
  3. はいなら：パスキーにチャレンジ
  4. ユーザーがセキュリティキーにタッチまたは生体認証
  5. voltaが署名を検証
  6. ログイン完了、amr=["google_oidc", "webauthn"]

  登録：
  1. ユーザーが/settings/securityに移動
  2. 「パスキーを追加」をクリック
  3. ブラウザが生体認証/セキュリティキーを要求
  4. voltaが公開鍵 + 認証情報IDを保存
  5. ユーザーは複数のパスキーを登録可能（バックアップ）
```

### 将来：一次認証としてのパスキー（パスワードレス）

後のフェーズでは、voltaはパスキーのみのログインをサポートできます：

```
  1. ユーザーが「パスキーでログイン」をクリック
  2. ブラウザが生体認証/セキュリティキーを要求
  3. voltaが署名を検証
  4. ログイン完了（Google OIDCは不要）

  これにより認証のGoogleへの依存が除去されます。
```

### データベーススキーマ（計画）

```sql
CREATE TABLE webauthn_credentials (
    id              UUID PRIMARY KEY,
    user_id         UUID REFERENCES users(id),
    credential_id   BYTEA NOT NULL,  -- WebAuthn登録から
    public_key      BYTEA NOT NULL,  -- COSEエンコードされた公開鍵
    sign_count      BIGINT DEFAULT 0, -- リプレイ防止
    name            TEXT,             -- ユーザーフレンドリーな名前（「私のYubiKey」）
    created_at      TIMESTAMPTZ DEFAULT now(),
    last_used_at    TIMESTAMPTZ
);
```

## よくある間違いと攻撃

### 間違い1：署名カウンターをチェックしない

WebAuthnには使用するたびにインクリメントする署名カウンターがあります。サーバーが保存値より低いカウンターを見たら、認証情報がクローンされたことを意味します。常にカウンターをチェックして強制してください。

### 間違い2：ユーザーごとに複数の認証情報をサポートしない

ユーザーは複数のパスキーを登録できるべきです（例：YubiKeyとスマホ）。1つを紛失しても、もう1つでログインできます。

### 間違い3：フォールバックを提供しない

ユーザーの唯一のパスキーが紛失し、フォールバック（TOTP、リカバリコード）がなければ、永久にロックアウトされます。常に代替のリカバリ方法を提供してください。

### 攻撃：デバイスの窃取

誰かがあなたのスマホ（ロック解除済み）を盗んだり、デバイスのPINを知っていれば、あなたのパスキーを使えます。デバイスのロック画面と生体認証要件で軽減されます。

### 攻撃：認証情報のクローニング（ハードウェア）

理論的には、高度な攻撃者がセキュリティキーの秘密鍵をクローンできます。署名カウンターがこれを検知します。本物とクローンの両方のキーが使われると、カウンターが一致しません。実際には、この攻撃には物理的アクセスと専門機器が必要です。
