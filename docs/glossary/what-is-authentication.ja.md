# 認証とは、実際には何なのか？

[English version](what-is-authentication.md)

---

## よくある誤解

「認証（auth）が何か知ってるよ。ログインのことでしょ。」

違う。ログインは認証のほんの一部だ。見える部分にすぎない。認証は氷山であり、ログインは水面から出ている先端だ。水面下には、本人確認、認証情報の保存、セッション管理、トークン発行、信頼の確立、そして失効という広大なシステムが広がっている。ほとんどのエンジニアはその20%しか見ていない。

そして事態を悪化させるのは、誰もがウェブサイトにログインした経験があるため、誰もが認証を理解していると思い込んでいることだ。分散システムセキュリティにおけるダニング＝クルーガー効果だ。

---

## 認証が実際に意味すること

認証とは、あるエンティティが自分が主張する人物であることを証明する行為だ。それだけだ。「入れてあげる」ことではない。「権限を与える」ことでもない。ただ、アイデンティティの証明。

しかし「アイデンティティの証明」は、5秒以上考えると途方もなく複雑な問題であることがわかる。

### ほとんどの人が見ていないレイヤー

**レイヤー1：認証情報の交換。** ユーザーが何か（パスワード、OAuthトークン、生体認証）を提示して、自分が主張する人物であることを証明する。これが人々が「auth」と考える部分だ。

**レイヤー2：IDプロバイダーの信頼。** voltaがGoogle OIDCを使うとき、ユーザーのアイデンティティを直接検証しているのではない。Googleがアイデンティティを検証したことを信頼し、Googleからのメッセージが本物であることを検証している。これは信頼の連鎖だ。voltaはOIDC IDトークンの署名を検証し、`iss`（発行者）クレームを確認し、`nonce`を検証し（リプレイ攻撃防止）、`aud`（対象者）が一致することを確認する。これらのいずれかが失敗すれば、Googleの観点からはユーザーが「正常にログイン」していても、アイデンティティは拒否される。

**レイヤー3：セッションの確立。** ユーザーが一度アイデンティティを証明した。それからどうする？ リクエストのたびにもう一度証明する？ それでは使い物にならない。だからセッションを作成する -- 「このブラウザ、このCookieは、この検証済みユーザーのものだ」というサーバー側のレコードだ。セッションは認証ではない。以前に行われた認証の記憶だ。

**レイヤー4：トークンの発行。** voltaはセッション検証後にJWTを発行する。JWTは、下流のアプリがvoltaにコールバックせずに検証できる、携帯可能な認証の証明だ。しかしJWTも認証そのものではない。「voltaがこの時点でこのユーザーのアイデンティティを検証した」という署名付きの声明だ。

**レイヤー5：継続的な検証。** セッションは期限切れになる。トークンは期限切れになる。鍵はローテーションされる。ユーザーは無効化される。テナントは停止される。認証は一回限りのイベントではない。継続的なメンテナンスが必要な継続的な状態だ。voltaの8時間スライディングウィンドウ、5分のJWT有効期限、デバイスごとのセッション制限は、すべてこのレイヤーの一部だ。

**レイヤー6：失効。** 誰かの認証を取り消す能力。「このユーザーのセッションはもう有効ではない。」「このユーザーのすべてのセッションが終了した。」ここでステートレス認証の神話が崩壊する -- 追跡していないものを取り消すことはできない。

## 認証ではないもの

**認証は認可ではない。** 認証は「あなたは山田太郎です」と言う。認可は「山田太郎はプロジェクトを編集できるが、ユーザーを削除することはできない」と言う。voltaは両方を処理するが、それらは別々のシステムだ。ForwardAuthは認証（有効なセッションか？）と認可（このアプリに対して正しいロールか？）を別々のステップとしてチェックする。

**認証はセッション管理ではない。** セッションは、すべてのリクエストで再認証を避けるための便利機能だ。セッションなしの認証（毎回認証情報を検証）もありえるし、適切な認証なしのセッション（セッション固定攻撃：攻撃者がセッションを作成し、ユーザーにそれを使わせる）もありえる。voltaはまさにこれを防ぐために、ログインごとにセッションIDを再生成する。

**認証は認証情報の保存ではない。** voltaはパスワードを保存しない。認証情報の検証をOIDC経由でGoogleに委任している。これは意図的な設計判断だ：最も難しい部分（パスワードのハッシュ化、ブルートフォース保護、漏洩した認証情報の検出）はIDプロバイダーに任せ、それ以外のすべてに集中する。

**認証は暗号化ではない。** 「すべてを暗号化しています」は認証戦略ではない。暗号化は転送中および保存時のデータを保護する。認証はアイデンティティを検証する。補完的だが別の関心事だ。voltaは暗号化を使用する（鍵の保存にAES-256-GCM、JWT署名にRS256）が、これらは認証に奉仕するツールであり、認証そのものではない。

## voltaが実際にやっていること

完全な認証フロー、全6レイヤーはこうだ：

1. **認証情報の交換：** ユーザーが「Googleでサインイン」をクリック。voltaはPKCEチャレンジ、stateパラメータ、nonceとともにGoogleのOIDCエンドポイントにリダイレクトする。
2. **IDプロバイダーの信頼：** Googleがユーザーを認証し、認可コードとともにリダイレクトバック。voltaはコードをトークンに交換し、GoogleのJWKSに対してIDトークンの署名を検証し、iss/aud/nonce/expクレームを検証する。
3. **セッションの確立：** voltaはサーバー側セッションを作成し、署名付きセッションCookieを生成し、セッションをデータベースに保存する。セッションIDは新規（以前のセッションから再利用しない）。
4. **トークンの発行：** voltaは短命のJWT（5分）を発行し、ユーザーID、テナントID、ロール、表示名を含める。下流のアプリはvoltaのJWKSエンドポイントを使ってこのJWTを検証する。
5. **継続的な検証：** ForwardAuthがすべてのリクエストでセッションの有効性をチェック。JWTはvolta-sdk-jsによって期限切れ時にサイレントリフレッシュされる。セッションはアクティビティとともにスライドするが、最大有効期間後にハード期限切れになる。
6. **失効：** ユーザーは個別のセッションまたはすべてのセッションを無効化できる。管理者はロールを変更できる（次のJWTリフレッシュ時に有効）。テナントの停止は、セッションレイヤーですべてのメンバーアクセスを即座にブロックする。

これが認証だ。「ログイン」ではない。

## 知らなくても大丈夫

認証はログイン画面のことだと思っていたなら、仲間へようこそ。ほとんどのエンジニアはそこから始まる。問題は無知ではない。業界が「auth」をまるで一つのものであるかのように語ることだ。実際には、トレンチコートを着た6つのものなのに。

今、トレンチコートの下にあるものが見えた。見た目ほど怖くはない。各レイヤーには明確な目的があり、それらを個別に理解すれば、システム全体が意味を成す。

もう、あなたは知っている。
