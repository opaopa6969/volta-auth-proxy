# セキュリティとは、実際には何なのか？

[English version](what-is-security.md)

---

## よくある誤解

「セキュリティは重要です。」誰もがそう言う。すべてのREADME、すべてのピッチ資料、すべての求人情報に書いてある。そして、そう言う人のほとんどが、実際に何を意味しているのか説明できない。

不都合な真実がある。ほとんどの人はセキュリティを「機能」として扱っている。追加するもの。スプレッドシートのチェックボックス。「セキュリティはありますか？ はい。チェック。」そして次の機能に移る。

それはセキュリティではない。それは**セキュリティシアター**（見せかけのセキュリティ）だ。

---

## セキュリティが実際に意味すること

セキュリティは機能ではない。継続的な実践だ。製品よりも衛生習慣に近い。衛生習慣を人生に一度「追加」して完了とは宣言しない。毎日歯を磨く。食事前に手を洗う。やらない結果は静かに蓄積され、ある日一気に襲いかかるからやるのだ。

セキュリティも同じだ。「このシステムは安全か？」という問いの答えは常に「いいえ」だ。なぜならセキュリティはバイナリ（二値）の状態ではないからだ。本当の問いはこうだ：

- **何を守るのか？**（ユーザーデータ、セッショントークン、秘密鍵）
- **誰から？**（外部の攻撃者、内部の不正、偶発的な漏洩）
- **失敗したらどうなる？**（データ漏洩、規制上の罰金、信頼の崩壊）
- **今日、何をしているのか？**（去年ではない。今日。）

セキュリティとは、これらの問いに継続的に答え、その答えに基づいて行動する実践だ。

---

## セキュリティ vs. セキュリティシアター

セキュリティシアターとは、セキュリティのように見えるが、実際には何も意味のあるものを守っていない行為だ。例を挙げよう：

| セキュリティシアター | 実際のセキュリティ |
|---|---|
| 大文字＋数字＋記号を含むパスワードの強制 | ログイン試行をIPごとに10回/分にレート制限 |
| ログインページに鍵アイコンを表示 | すべてのOIDCフローでPKCE + state + nonceを使用 |
| 「暗号化を使用しています」（何を、どこで、どうやってかは言わない） | 秘密鍵をAES-256-GCMで保存時暗号化、JWT署名にRS256、HS256/noneアルゴリズムの拒否 |
| ランディングページに「SOC2準拠」と記載 | すべての認証イベントを実際にログに記録し、レビューする |

違いは具体性だ。セキュリティシアターは一般論で語る。実際のセキュリティは具体的に語る。

## voltaが実際にやっていること（雰囲気ではなく、具体的に）

volta-auth-proxyは「安全です」とは主張しない。具体的なことを行っていると主張する：

**本人確認：** Google OIDCにPKCE（S256）+ stateパラメータ（CSRF防止）+ nonce（リプレイ攻撃防止）を使用。1つのログインフローに3つの独立した保護。見た目が良いからではなく、それぞれが異なるクラスの攻撃を防ぐからだ。

**セッション管理：** 署名付きCookie。8時間のスライディングウィンドウ。ログインごとにセッションIDを再生成（セッション固定攻撃の防止）。ユーザーあたり最大5つの同時セッション（認証情報盗難の影響範囲を制限）。

**トークンセキュリティ：** RS256のみ。HS256と`none`アルゴリズムはホワイトリストで明示的に拒否。JWTの有効期限は5分。秘密鍵はAES-256-GCMで保存時暗号化。猶予期間付きの鍵ローテーションAPI。

**不正利用防止：** ログインエンドポイントはIPあたり10リクエスト/分のレート制限。APIエンドポイントはユーザーあたり200リクエスト/分。すべての認証エンドポイントのCache-Controlヘッダーを`no-store, private`に設定し、ブラウザの戻るボタンによるデータ漏洩を防止。

**監査証跡：** すべての認証イベントを記録 -- ログイン、ログアウト、ロール変更、招待の承諾、セッションの無効化。

これらはそれぞれ、具体的で検証可能な主張だ。コードを読んで確認できる。それがセキュリティとセキュリティシアターの違いだ。

## 機能ではなく、実践

voltaのようなプロジェクトにおけるセキュリティの日常的な実践はこうだ：

1. **依存関係の監視。** voltaはJJWT、Javalin、HikariCPなどに依存している。それぞれに潜在的な脆弱性がある。CVEを監視し、迅速にアップデートすることがセキュリティだ。`dependabot`のアラートを無視することはセキュリティではない。

2. **プロトコルの認識。** OAuth2とOIDCには既知の攻撃ベクトルがある。新しいものが発見される。PKCEに対する新しい攻撃が公開されたとき、誰かがvoltaが影響を受けるかどうかを評価する必要がある。その「誰か」は、運用者であるあなただ。

3. **インシデント対応の準備。** ユーザーがアカウントの侵害を報告したらどうする？ voltaはツールを提供する（セッション無効化、監査ログ、全デバイスログアウト）。しかし、ツールがあることと、それを使う計画があることは別だ。

4. **正直な評価。** voltaにはランタイムの異常検知がない。自動化されたペネトレーションテストもない。すべてのコミットをレビューする専任のセキュリティチームもない。これらはギャップだ。ギャップを知ることがセキュリティだ。ギャップがないふりをすることはセキュリティシアターだ。

## みんなが間違えること

最大のセキュリティの過ちは、技術的な失敗ではない。セキュリティは他の誰かの問題だという思い込みだ。「フレームワークがやってくれる。」「クラウドプロバイダーがやってくれる。」「認証ライブラリがやってくれる。」

誰もやってくれない。ツールは助けになる。ライブラリも助けになる。しかしセキュリティは製品ではなく責任だ。Auth0の代わりにvoltaのようなセルフホスト型認証ソリューションを選ぶとき、セキュリティを減らしているのではない。セキュリティを自分で所有することを選んでいるのだ。それは正当な選択だ -- ただし、何にサインアップしているのかを理解している場合に限る。

---

## 知らなくても大丈夫

セキュリティは追加してチェックを入れれば済む機能だと思っていたなら、あなただけではない。ほとんどのエンジニアがキャリアの初期にそう考える。そういう話し方をされるからだ。「アプリにセキュリティを追加して。」「アプリにはセキュリティが必要だ。」言葉そのものが誤解を招く。

今あなたは知った。セキュリティは持つものではない。するものだ。毎日。そして最初の一歩は、実際にやっていることと、やっているふりをしていることについて正直になることだ。

もう、あなたは知っている。
