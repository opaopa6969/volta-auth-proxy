# ゼロトラストとは、実際には何なのか？

[English version](what-is-zero-trust.md)

---

## よくある誤解

ゼロトラストはセキュリティにおいて最もバズワード化された用語だ。マーケティング部門に採用され、ランディングページに散りばめられ、VPNからエンドポイント検出ソフトウェア、アイデンティティプラットフォームまで、あらゆるものを売るために使われている。

10人にゼロトラストの意味を聞けば12の回答が返ってくる。そのほとんどは「セキュリティに関するもの」か「何も信頼しないということ」に要約される。どちらも正しいと同時に、役に立たない。

---

## ゼロトラストが実際に意味すること

ゼロトラストは一つの原則に基づくセキュリティモデルだ：**決して信頼せず、常に検証する。**

従来のネットワークセキュリティモデル（「城と堀」と呼ばれることがある）では、境界がある。境界の内側にあるものはすべて信頼される。外側にあるものは信頼されない。ファイアウォールが境界を守る。VPNが外部から境界の中にトンネルを掘る。一度中に入れば、どこにでも行ける。

ゼロトラストはこう言う：内側は存在しない。すべてのリクエストが、すべてのソースから、すべてのレイヤーで、認可されていることを証明しなければならない。社内ネットワークにいることで何かにアクセスできるわけではない。有効なVPN接続があっても何かにアクセスできるわけではない。各リソースが各リクエストを独立して検証する。

3つの原則：

1. **明示的に検証する。** 利用可能なすべてのデータに基づいて常に認証・認可する：アイデンティティ、場所、デバイスの健全性、サービスやワークロード、データの分類、異常。
2. **最小権限アクセスを使用する。** アクセスを必要最小限に制限する。ジャストインタイムかつジャストイナフのアクセス。
3. **侵害を前提とする。** 攻撃者がすでに内部にいるかのように設計する。影響範囲を最小化し、アクセスをセグメント化し、エンドツーエンドの暗号化を検証し、脅威を検出するために分析を使用する。

---

## ゼロトラストが意味しないこと

**「何も信頼しない」という意味ではない。** 信頼する。検証した後で信頼し、限られた時間、限られた範囲で信頼する。ゼロトラストはパラノイアではない。継続的な検証を伴う条件付きの信頼だ。

**「自分のコードを信頼するな」という意味ではない。** ゼロトラストはネットワークとアイデンティティのアーキテクチャについてであり、自分の`if`文を信頼するかどうかについてではない。自分のデータベースからのデータが有効であると仮定するのは依然として必要だ。ゼロトラストは境界に適用されるのであり、コードのすべての行にではない。

**「製品を購入する必要がある」という意味ではない。** ゼロトラスト周辺のベンダー状況は巨大で、大部分が寄生的だ。ゼロトラストはアーキテクチャと原則のセットだ。オープンソースツール、リバースプロキシ、慎重な設計で実装できる。年間6桁のコストがかかる「ゼロトラストプラットフォーム」は必要ない。

**「リスクゼロ」という意味ではない。** リスクを排除するセキュリティモデルはない。ゼロトラストは侵害の影響範囲を縮小し、横方向の移動を困難にする。侵害を不可能にするわけではない。

## voltaがゼロトラストを実装する方法（そう呼ばずに）

volta-auth-proxyのForwardAuthパターンは、プロジェクトがバズワードを使わなくても、実践におけるゼロトラストだ。

マッピングはこうだ：

### 決して信頼せず、常に検証する

下流アプリへのすべてのリクエストはTraefikを通過し、リクエストがアプリに到達する前にvoltaのForwardAuthエンドポイントに転送される。すべてのリクエスト。最初のだけではない。ログインだけではない。

```
ユーザーリクエスト → Traefik → volta ForwardAuth → (200 OK?) → 下流アプリ
                                                 → (401?)    → ログインにリダイレクト
```

下流アプリは認証されていないリクエストを決して見ない。自分で認証をチェックする必要がない。HTTPヘッダー（`X-Volta-User-Id`、`X-Volta-Tenant-Id`、`X-Volta-Roles`）経由で事前検証済みのアイデンティティを受け取る。しかし -- これがゼロトラストの部分だ -- これらのヘッダーはvoltaがこの特定のリクエストでセッションを検証したからこそ存在する。リクエストが検証をスキップする「信頼されたゾーン」はない。

### 最小権限アクセス

volta-config.yamlが、どのロールがどのアプリにアクセスできるかを定義する：

```yaml
apps:
  - name: admin-panel
    allowed_roles: [OWNER, ADMIN]
  - name: dashboard
    allowed_roles: [OWNER, ADMIN, MEMBER]
  - name: public-docs
    allowed_roles: [OWNER, ADMIN, MEMBER, VIEWER]
```

VIEWERは管理パネルにアクセスできない。管理パネルがロールをチェックするからではない（チェックする必要がない）。voltaのForwardAuthがリクエストが管理パネルに到達する前に拒否するからだ。アクセスはアプリケーションではなくゲートウェイで拒否される。

### 侵害を前提とする

voltaのアーキテクチャは、どのコンポーネントも侵害される可能性があることを前提としている：

- **JWTの有効期限は5分。** トークンが漏洩しても、悪用の窓は小さい。
- **セッション制限（ユーザーあたり最大5）** は、侵害された認証情報で無制限のセッションを生成できないことを意味する。
- **レート制限** は、ブルートフォース攻撃がすぐに壁にぶつかることを意味する。
- すべての認証イベントの**監査ログ** は、異常を検出できることを意味する。
- **テナント分離** は、一つのテナントでの侵害が別のテナントへのアクセスを与えないことを意味する。APIパスのテナントIDがJWTクレームと一致しなければならない -- ポリシーで防止されるのではなく、構造的に防止される。

### ネットワークは信頼されない

voltaのForwardAuthはリクエストがどこから来るかを気にしない。内部ネットワーク、外部ネットワーク、VPN、直接接続 -- すべてのリクエストが同じ検証を受ける。「10.0.0.0/8からのリクエストは信頼される」というIP許可リストはない。セッションCookieがアイデンティティだ。セッション検証が信頼境界だ。それ以外には何もない。

## 正直なギャップ

voltaはNIST 800-207の意味での完全なゼロトラスト実装ではない。完全なゼロトラストには、デバイスポスチャーの評価、継続的な行動分析、ネットワークレイヤーでのマイクロセグメンテーション、およびエンドポイント検出・応答（EDR）システムとの統合が含まれる。

voltaはアイデンティティとアクセス制御の部分を処理する。SaaSアプリケーションにとって最も重要な部分だが、完全な全体像ではない。規制産業のために完全なゼロトラスト準拠が必要な場合、voltaはパズルの一つのピースであり、パズル全体ではない。

---

## 知らなくても大丈夫

ゼロトラストが購入できる製品だと思っていたり、何かを信頼しないという漠然とした原則だと思っていたなら、あなたは多数派だ。この用語はあまりにも積極的にマーケティングされてきたため、その実際の意味がセールストークの下に埋もれてしまった。

核心的なアイデアはシンプルだ：すべてのリクエストを検証し、最小限のアクセスを付与し、最悪を想定する。voltaはForwardAuthを通じてこれを行っており、アーキテクチャ文書で「ゼロトラスト」というフレーズを一度も使っていない。原則を実装する最良の方法は、名前をつけることではなく、ただ実行することかもしれない。

もう、あなたは知っている。
