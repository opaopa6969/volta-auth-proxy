# ホワイトリスト

[English version](whitelist.md)

---

## これは何？

ホワイトリスト（許可リストとも呼ばれる）は、明示的に許可された項目のリストです。リストにあるものは許可され、それ以外はデフォルトで拒否されます。セキュリティにおいて、ホワイトリストは「デフォルト拒否」アプローチです。すべての悪いものをブロックしようとする（ブラックリスト）代わりに、既知の良いものだけを許可します。

結婚式のVIPゲストリストのようなものです。門番は街の中の入場させるべきでないすべての人を覚えようとしません。代わりに、入場を許可された人の短いリストを持っています。名前がリストになければ、どんなに着飾っていても入場できません。

volta-auth-proxyでは、暗号アルゴリズム、リダイレクトURL、Webhookドメイン、その他のセキュリティ上重要な設定を、既知の安全な値のみに制限するためにホワイトリストを使用しています。

---

## なぜ重要なのか？

ブラックリスト（既知の悪いものをブロック）は負け戦です：

```
  ブラックリストアプローチ（脆弱）：
  ┌─────────────────────────────────────────┐
  │  ブロック：HS256（弱いアルゴリズム）       │
  │  ブロック：none（アルゴリズムなし攻撃）    │
  │  ブロック：HS384（HMAC混同攻撃）          │
  │  ブロック：???（次の脆弱性？）            │
  │                                         │
  │  問題：すべての悪いものを事前に            │
  │  知る必要がある。1つ漏れ → 侵害。         │
  └─────────────────────────────────────────┘

  ホワイトリストアプローチ（堅牢）：
  ┌─────────────────────────────────────────┐
  │  許可：RS256（RS256のみ）                 │
  │                                         │
  │  それ以外はすべて自動的に拒否             │
  │  まだ発明されていない攻撃も含む。          │
  └─────────────────────────────────────────┘
```

ホワイトリストは未知のものを拒否するため、本質的に安全です。新しい攻撃ベクトルは更新なしにデフォルトでブロックされます。

## どう動くのか？

### 原則

```
  ┌───────────────────────────────────────────────┐
  │                                               │
  │    入力 ──► ホワイトリストにある？ ──► YES     │
  │                      │                   │    │
  │                      │                 許可    │
  │                      │                        │
  │                      ▼                        │
  │                     NO ──────────────► 拒否   │
  │                                               │
  └───────────────────────────────────────────────┘
```

### ホワイトリスト vs ブラックリストの比較

| 側面 | ホワイトリスト（許可リスト） | ブラックリスト（拒否リスト） |
|------|--------------------------|--------------------------|
| デフォルト | すべて拒否 | すべて許可 |
| リストに含む | 既知の良いもの | 既知の悪いもの |
| 未知の項目 | ブロック | 許可（危険！） |
| メンテナンス | 新しい良い項目を追加 | 新しい脅威に追従 |
| セキュリティ姿勢 | 強い | 弱い（受動的） |
| 使いやすさ | 新機能で更新が必要かも | 新機能追加が容易 |

### セキュリティでの一般的なホワイトリストパターン

```
  アルゴリズムホワイトリスト：
    allowed = ["RS256"]
    if (jwt.algorithm NOT IN allowed) → 拒否

  リダイレクトURLホワイトリスト：
    allowed_domains = ["app.example.com", "staging.example.com"]
    if (redirect_url.host NOT IN allowed_domains) → 拒否

  IPホワイトリスト：
    allowed_ips = ["10.0.0.0/8", "203.0.113.0/24"]
    if (client_ip NOT IN allowed_ips) → 拒否

  HTTPメソッドホワイトリスト：
    allowed_methods = ["GET", "POST"]
    if (request.method NOT IN allowed_methods) → 拒否
```

## volta-auth-proxyではどう使われている？

### アルゴリズムホワイトリスト（JWT）

voltaは[JWT](jwt.md)署名検証にRS256のみをホワイトリストしています：

```java
// JwtService.java
private static final Set<String> ALLOWED_ALGORITHMS =
    Set.of("RS256");

public AuthPrincipal verify(String token) {
    DecodedJWT decoded = JWT.decode(token);

    if (!ALLOWED_ALGORITHMS.contains(decoded.getAlgorithm())) {
        throw new SecurityException(
            "許可されていないアルゴリズム: " + decoded.getAlgorithm()
        );
    }
    // RS256検証を続行...
}
```

これにより、攻撃者がJWTの`alg`ヘッダーを`HS256`や`none`に変更する「アルゴリズム混同」攻撃を防止します：

```
  攻撃：アルゴリズム混同
  ┌─────────────────────────────────────────────────┐
  │  攻撃者がalg: "none"のJWTを作成                   │
  │  署名：（空）                                     │
  │                                                   │
  │  ホワイトリストなし：                               │
  │    voltaがalg=noneを見る → 検証をスキップ           │
  │    → 攻撃者が有効なトークンを持つ！                 │
  │                                                   │
  │  ホワイトリストあり：                               │
  │    voltaがalg=noneを見る → [RS256]にない            │
  │    → 即座に拒否                                    │
  └─────────────────────────────────────────────────┘
```

### リダイレクトURLホワイトリスト

voltaは[OAuth2](oauth2.md)リダイレクトURLを事前登録されたドメインに制限します：

```
  テナント設定：
    allowed_redirect_urls:
      - "https://app.acme.com/callback"
      - "https://staging.acme.com/callback"

  ログインリクエスト：
    redirect_uri=https://evil.com/steal-token

  voltaチェック：
    "https://evil.com/steal-token"は許可リストにない
    → 拒否（オープンリダイレクト攻撃を防止）
```

### Webhook URLホワイトリスト

テナントが[Webhook](webhook.md)エンドポイントを登録する際、voltaがURLを検証します：

```
  許可：
    ✓ https://api.acme.com/webhooks
    ✓ https://hooks.slack.com/services/...

  ブロック：
    ✗ http://169.254.169.254/metadata  （AWSメタデータ - SSRF）
    ✗ http://localhost:8080/admin       （内部サービス - SSRF）
    ✗ http://10.0.0.1/secret           （プライベートネットワーク - SSRF）
```

voltaがブロックするもの：
- プライベートIPレンジ（10.x、172.16-31.x、192.168.x）
- ループバックアドレス（127.x、localhost）
- リンクローカルアドレス（169.254.x）
- クラウドメタデータエンドポイント

### CORSオリジンホワイトリスト

voltaはCross-Origin Resource Sharingを許可されたオリジンに制限します：

```yaml
# volta-config.yaml
cors:
  allowed_origins:
    - "https://app.acme.com"
    - "https://staging.acme.com"
```

### アップストリーム転送の[ヘッダー](header.md)ホワイトリスト

voltaは特定のヘッダーのみを[アップストリーム](upstream.md)アプリケーションに転送し、ヘッダーインジェクションを防止します：

```
  転送ヘッダー（ホワイトリスト）：
    X-Volta-User-Id
    X-Volta-Email
    X-Volta-Roles
    X-Volta-Tenant-Id
    X-Volta-M2M

  クライアントからの他のX-Volta-*ヘッダー → 除去
  （クライアントがアイデンティティヘッダーを偽造するのを防止）
```

## よくある間違いと攻撃

### 間違い1：ホワイトリストの代わりにブラックリストを使う

ブラックリストはすべての攻撃を予測する必要があります。ホワイトリストはデフォルトで拒否します。セキュリティ上重要な判断には常にホワイトリストを優先してください。

### 間違い2：過度に広いホワイトリストエントリ

`app.example.com`の代わりに`*.example.com`をホワイトリストすると、攻撃者が制御するサブドメインも含むすべてのサブドメインが許可されます（例：サブドメインを作成できる場合の`evil.example.com`）。

### 間違い3：ホワイトリストを更新しない

新しい正当なサービスやドメインを追加する際、ホワイトリストの更新を忘れないでください。さもないと、正当なリクエストがブロックされます。

### 間違い4：クライアント側のみのホワイトリスト

ホワイトリストがブラウザ（JavaScript）でのみ適用されている場合、攻撃者はAPIを直接呼び出して回避できます。常にサーバー側でホワイトリストを適用してください。

### 攻撃：ホワイトリスト迂回によるオープンリダイレクト

攻撃者がホワイトリストされたドメイン上のリダイレクトするURLを見つける（例：`https://app.acme.com/redirect?url=https://evil.com`）。防御：ドメインだけでなく完全なURLパスを検証。リダイレクトチェーンを禁止。

### 攻撃：DNSリバインディング

攻撃者のドメインがホワイトリストチェック時にパブリックIPに解決し、実際のリクエスト時にプライベートIPに再解決する。防御：登録時だけでなくリクエスト時にドメインを解決してIPをチェック。
