# YAGNI（You Aren't Gonna Need It）

[English version](yagni.md)

---

## これは何？

YAGNI は "You Aren't Gonna Need It"（それ、いらないよ）の略です。ソフトウェア開発の原則で、「実際に必要になるまで機能を作るな」という意味です。いつか必要になるかもしれない、ではダメです。あったら便利かも、でもダメです。今、現実に要件があるときだけ作ります。

週末のキャンプ旅行の荷造りを想像してください。YAGNI はこう言います：テント、寝袋、食料を持っていけ。カヤック、ポータブル発電機、衛星アンテナを「念のため」持っていくな。キャンプが本格的になったら、そのとき取りに戻ればいい。でも、いつか必要になるかもしれないものを全部車に積んだら、出発すらできません。

YAGNI の反対は「投機的汎用化」です -- 想像上の将来の要件のために抽象化や機能を作ることです。これはソフトウェアプロジェクトが肥大化し、遅延し、保守不能になる最も一般的な原因の一つです。

---

## なぜ重要？

作るすべての機能には、初期開発時間をはるかに超えるコストがあります：

- **保守コスト**：すべてのコード行は負債です。テスト、ドキュメント作成、デバッグ、将来の変更との互換性維持が必要です。
- **複雑性コスト**：使われない機能はコードベースの理解を困難にします。新しい開発者は目的のないコードを学ばなければなりません。
- **機会コスト**：投機的な機能の構築に費やした時間は、ユーザーが今実際に必要としている機能に費やせない時間です。
- **決定麻痺**：過剰設計はより多くの決定を生みます。より多くの設定。より多くの壊れる可能性。

調査によると、計画された機能の大部分は使われません。それらを作ることは純粋な無駄です。

## どう動くのか？

### YAGNI の判断プロセス

```
  新機能のアイデアが到着
        │
        ▼
  具体的で現在の要件があるか？
        │
   ┌────┴────┐
   No        Yes
   │          │
   ▼          ▼
  やめる。   要件を満たす最もシンプルな
  後で使う   解決策か？
  ためにメモ     │
             ┌──┴──┐
             No    Yes
             │      │
             ▼      ▼
         まず     作る
         簡素化
```

### YAGNI が意味しないこと

YAGNI は以下を意味しません：

- **良いアーキテクチャを省略する。** クリーンコードと関心の分離は常に必要です。
- **セキュリティを無視する。** 認証、認可、入力検証は現在の要件であり、投機的なものではありません。
- **すべての計画を避ける。** 先を考えるべきです -- ただし先に作るべきではありません。
- **使い捨てコードを書く。** YAGNI のコードも構造は良くあるべきです。ただ小さいだけです。

### YAGNI vs. 拡張性

ニュアンスは：後で拡張しやすいコードを書く。ただし、今は実際に拡張しない。

```
  悪い例（投機的）：
    PluginManager + PluginLoader + PluginRegistry
    + AbstractPlugin + PluginConfig + PluginLifecycle
    ... 現在プラグインが0個のシステムに対して

  良い例（YAGNI）：
    必要なことを1つだけ行う単一クラス。
    プラグインが本当の要件になったら、そのときリファクタリング。
    リファクタリングは想像ではなく、実際のニーズに基づく。
```

## volta-auth-proxy ではどう使われている？

volta-auth-proxy は YAGNI をコア設計哲学として構築されています。これを**フェーズミニマル設計**と呼びます。現在のフェーズに必要なものだけを作り、それを上手く作るという考え方です。

### volta での具体例

| 機能 | YAGNI アプローチ | エンタープライズアプローチ |
|------|-----------------|------------------------|
| **認証プロバイダー** | Google OIDC のみ（Phase 1） | 抽象プロバイダーインターフェース + 5プロバイダー |
| **データベース** | SQLite（単一ファイル） | PostgreSQL クラスター + リードレプリカ |
| **セッションストア** | 同じ SQLite データベース | Redis クラスター + フェイルオーバー |
| **設定形式** | YAML ファイル、起動時に読み込み | 動的設定サービス + ホットリロード |
| **マルチテナンシー** | シングルプロセス、インメモリテナントマップ | テナントごとの個別データベース |
| **API バージョニング** | `/api/v1/` のみ | バージョンネゴシエーション + 廃止フレームワーク |

### フェーズミニマル哲学

volta は明示的にフェーズを定義します。各フェーズには明確なスコープがあります：

- **Phase 1**：Google OIDC プロバイダーのみ、SQLite、シングルノードデプロイ。インディーハッカーや社内ツールには十分。
- **Phase 2**：追加 OIDC プロバイダー、PostgreSQL オプション、基本的な HA。
- **Phase 3**：エンタープライズ機能（SOC2、SLA、SSO）。

Phase 2 の機能は Phase 1 のコードベースに存在しません。スタブもなく、フィーチャーフラグの裏に隠れてもおらず、抽象インターフェースで待機してもいません。単純にまだ存在しないのです。

### なぜこれが volta で機能するか

volta のターゲットユーザー -- インディーハッカーや初期スタートアップ -- はエンタープライズ機能を必要としません。それらを構築すると：

1. 実際に必要な機能の開発が遅れる
2. ターゲット層を怖がらせる設定の複雑さが増す
3. 不必要に攻撃面が増大する
4. 約200msの起動時間が不可能になる

## よくある間違いと攻撃

### 間違い1：YAGNI を怠惰と混同する

YAGNI はスコープ管理の規律あるアプローチです。機能に対して積極的に「ノー」と言う必要があります。怠惰は、実際に必要な作業 -- エラー処理、ログ記録、テストなど -- を省略することです。

### 間違い2：YAGNI を使ってセキュリティを省略する

「まだ CSRF 対策は要らない」は YAGNI ではなく、怠慢です。アプリケーションがユーザーセッションを扱うなら、CSRF 対策は現在の要件です。YAGNI は機能に適用されるもので、セキュリティの衛生管理には適用されません。

### 間違い3：早すぎる抽象化

認証プロバイダーが1つしかないのに `AbstractAuthProviderFactory` を作るのは、典型的な反 YAGNI の動きです。抽象化の推測は外れ、そして悪い抽象化は抽象化なしより変更が困難です。

### 間違い4：ゴールドプレーティング

出荷前に機能に「もう一つだけ」追加すること。機能は動くのに、キャッシュ、リトライロジック、バッチ処理、ダッシュボードを追加する -- 誰も頼んでいないのに。
