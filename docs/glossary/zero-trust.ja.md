# ゼロトラスト

[English version](zero-trust.md)

---

## これは何？

ゼロトラストは、「決して信頼せず、常に検証する」という一つの原則に基づくセキュリティモデルです。自分のネットワーク内部からのリクエストであっても、ユーザー、デバイス、ネットワークリクエストをデフォルトでは信頼しません。

病院の施錠された薬品棚で考えてみましょう。古い病院では、スクラブ（医療服）を着て薬局に入った人は誰でも薬を取れました -- 「建物の中にいるなら信頼できる」という前提です。ゼロトラストの病院では、すべての棚にバッジスキャンが必要で、すべての薬品リクエストが記録され、医師でさえ何かにアクセスするたびに身元を証明しなければなりません。建物の中にいることは信頼されることを意味しません。

従来のネットワークセキュリティは堀のある城のようなものです：壁（ファイアウォール）を越えれば、どこにでも行けます。ゼロトラストはこう言います：壁はない。すべてのドアは施錠されている。すべての人がすべてのドアでIDを見せなければならない。

---

## なぜ重要なのか？

古いモデル（「内部ネットワークを信頼する」）が失敗するのは：

1. **VPNでは不十分。** 攻撃者がVPNアクセスを得たら（盗まれた認証情報、侵害されたノートPC）、ネットワーク上のすべてにアクセスできる。
2. **内部者が脅威になりうる。** すべての脅威が外部から来るわけではない。不満を持つ従業員、侵害されたアカウント、ラテラルムーブメント攻撃は内部の信頼を悪用する。
3. **クラウドが境界を曖昧にする。** アプリがAWSに、データベースがGCPに、ユーザーがリモートにいるとき、「内部」はもう存在しない。ネットワーク境界は存在しない。
4. **リモートワークは恒久的。** 自宅、カフェ、空港からシステムにアクセスする従業員には、場所に関係なく同じセキュリティが必要。

ゼロトラストはネットワークが常に敵対的であると仮定します。すべてのリクエストが正当であることを証明しなければなりません。

## ゼロトラストの実践

### Cloudflare Zero Trust

Cloudflare Zero Trust（旧Cloudflare Access）はゼロトラスト原則の商用実装です。アプリケーションの前に認証レイヤーを置きます：

```
  ゼロトラストなし：
  ┌──────────┐     ┌──────────────┐
  │ Internet │────►│ あなたのアプリ │  ← ネットワーク上の誰でもアクセス可能
  └──────────┘     └──────────────┘

  Cloudflare Zero Trustあり：
  ┌──────────┐     ┌────────────────┐     ┌──────────────┐
  │ Internet │────►│ Cloudflare     │────►│ あなたのアプリ │
  └──────────┘     │（先に検証）     │     └──────────────┘
                   └────────────────┘
                   「あなたは誰？証明して。」
```

### voltaがゼロトラスト原則をどう実装しているか

volta-auth-proxyの[ForwardAuth](forwardauth.ja.md)パターンは、アプリケーション層でのゼロトラスト実装です：

```
  volta ForwardAuth = あなたのアプリ向けゼロトラスト：

  ┌──────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────┐
  │ ブラウザ  │────►│ Traefik      │────►│ volta        │     │あなたの  │
  └──────────┘     │（リバース     │     │（ID確認、    │     │ アプリ   │
                   │ プロキシ）    │     │ ロール確認、 │     └──────────┘
                   └──────────────┘     │ テナント確認）│          ▲
                          │             └──────────────┘          │
                          │                    │                  │
                          │              200 OK + ヘッダー         │
                          └──────────────────────────────────────►│
                                 （voltaがOKと言った場合のみ）
```

すべてのアプリへのすべてのリクエストがこのフローを通ります。例外なし。「信頼された」ネットワークなし。リバースプロキシの背後の内部サービスも保護されます。これがゼロトラストです：

| ゼロトラスト原則 | voltaの実装 |
|----------------|------------|
| **決して信頼せず、常に検証** | すべてのリクエストがForwardAuthを通る。バイパスなし。 |
| **IDの検証** | すべてのリクエストでセッションCookieをPostgreSQLと照合して検証。 |
| **認可の検証** | すべてのリクエストでユーザーのロールをアプリの`allowed_roles`と照合。 |
| **コンテキストの検証** | テナントメンバーシップを確認。停止されたテナントはブロック。 |
| **最小権限** | アプリはvoltaが提供するヘッダーだけを見る。ユーザーデータへの直接DB接続なし。 |
| **侵害を前提** | JWTは5分で期限切れ。セッションにはスライディング + 絶対タイムアウト。鍵ローテーション対応。 |

### Cloudflare Zero Trustとの比較

| 機能 | Cloudflare Zero Trust | volta ForwardAuth |
|------|----------------------|-------------------|
| ネットワーク層 | エッジ（CDNレベル） | アプリケーション層（リバースプロキシ） |
| IDソース | 複数のIdP | Google OIDC（+ 設定可能なIdP） |
| マルチテナンシー | なし（単一組織） | あり（コア機能） |
| セルフホスト | 不可（Cloudflareクラウド） | 可（専用） |
| コスト | 使用量ベース課金 | $0 |
| 最適な用途 | 内部ツール保護、VPN代替 | マルチテナントSaaSアプリ認証 |

関連するが異なる問題を解決します。Cloudflare Zero TrustはVPNを置き換え、内部ツールへのアクセスを保護します。voltaはテナント対応のIDでマルチテナントSaaSアプリケーションを保護します。

## シンプルな例

ゼロトラストなし：

```
  1. 従業員が会社のVPNに接続
  2. VPNが内部ネットワークへのアクセスを許可
  3. 従業員はあらゆる内部サービスにアクセス可能
  4. 攻撃者がVPN認証情報を盗んだら → すべてにアクセス
```

ゼロトラストあり（voltaスタイル）：

```
  1. ユーザーが wiki.example.com にアクセス
  2. TraefikがvoltaにUnicodeかけ：「このユーザーは許可されている？」
  3. voltaが確認：有効なセッション？アクティブなユーザー？正しいテナント？適切なロール？
  4. すべてのチェックに合格した場合のみ → リクエストが通る
  5. 攻撃者がセッションを盗んだら → そのセッションのロールが許可するアプリのみアクセス
     → セッションは最大8時間で期限切れ
     → 管理者が即座に無効化可能
```
