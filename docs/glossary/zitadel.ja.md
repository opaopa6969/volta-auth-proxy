# ZITADEL

[English version](zitadel.md)

---

## これは何？

ZITADELは、Go言語で書かれたオープンソースのID管理プラットフォームです。認証、認可、ユーザー管理をターンキーソリューションとして提供します -- [Auth0](auth0.md)や[Keycloak](keycloak.md)が提供するものと同様ですが、クラウドネイティブなマルチテナント環境向けにゼロから構築されたモダンなアーキテクチャを持っています。

ZITADELは大規模なマンション管理会社のようなものです。入居申請、鍵の配布、建物のアクセスルール、来客管理、戸別の請求まですべてを処理します。これらを自分で構築する必要はありません -- サービスを借りるだけです。管理会社にはWebポータルがあり、入居者が自分の設定を管理し、ルームメイトを招待し、ゲストアクセスを設定できます。

ZITADELは、組み込みのマルチテナント機能で特に注目されています。Keycloakが「レルム」を作成して手動で分離を管理する必要があるのに対し、ZITADELは初日から組織（テナント）を第一級の概念として設計されました。

---

## なぜ重要なのか？

volta-auth-proxyの文脈では、ZITADELは**競合製品**として重要です。両者は同じ問題を解決します：SaaS向けのマルチテナントID管理。ZITADELを理解することで、voltaがなぜ存在し、各アプローチがどのようなトレードオフを行っているかを理解できます。

ID管理のランドスケープにはいくつかの層があります：

| 層 | 例 | トレードオフ |
|----|-----|-----------|
| マネージドSaaS | [Auth0](auth0.md)、[Okta](okta.md) | 簡単だが高価、ベンダーロックイン、データがサーバー外へ |
| セルフホスト（重量級） | [Keycloak](keycloak.md)、ZITADEL | 完全な制御だが複雑、高リソース使用量 |
| セルフホスト（軽量級） | volta-auth-proxy | 完全な制御、最小限のリソース、ただし自分で構築する部分が多い |
| DIY | 手作りの認証 | 最大限の制御だが最大限のリスク |

ZITADELは「セルフホスト重量級」カテゴリに位置します。独自のデータベース、独自のUI、独自のAPI、すべてがどう動くべきかについての独自の意見を持つ完全なプラットフォームです。

## どう動くのか？

### アーキテクチャ

ZITADELはCQRS（コマンドクエリ責任分離）を伴うイベントソーシングアーキテクチャを使用します：

```
  ┌─────────────────────────────────────────┐
  │              ZITADEL                     │
  │                                          │
  │  ┌──────────┐    ┌──────────────────┐   │
  │  │  Console  │    │  Management API  │   │
  │  │  (Web UI) │    │  (gRPC / REST)   │   │
  │  └──────────┘    └──────────────────┘   │
  │         │                │               │
  │         ▼                ▼               │
  │  ┌──────────────────────────────────┐   │
  │  │        コアエンジン               │   │
  │  │  イベントソーシング + CQRS        │   │
  │  │  読み取りモデル用プロジェクション  │   │
  │  └──────────────────────────────────┘   │
  │         │                               │
  │         ▼                               │
  │  ┌──────────────┐                      │
  │  │  CockroachDB  │  （またはPostgreSQL）│
  │  │  （イベントストア）│                  │
  │  └──────────────┘                      │
  └─────────────────────────────────────────┘
```

### 主要機能

| 機能 | 説明 |
|------|------|
| **マルチテナント** | 分離されたユーザー、ロール、設定を持つ組み込みの組織 |
| **OIDC/OAuth2** | ディスカバリ、PKCEなどを備えた完全な[OIDC](oidc.md)プロバイダー |
| **SAML** | SAML 2.0 IDプロバイダー |
| **パスワードレス** | WebAuthn / FIDO2サポート |
| **MFA** | TOTP、WebAuthn、SMS、メール |
| **ブランディング** | 組織ごとのログインページカスタマイズ |
| **Actions** | カスタムロジック用のサーバーサイドスクリプト |
| **マシンユーザー** | サービス間認証用のAPIキー |
| **監査ログ** | すべてのID操作の組み込みイベントログ |

### ZITADEL vs Keycloak vs Auth0 vs volta

| 観点 | ZITADEL | [Keycloak](keycloak.md) | [Auth0](auth0.md) | volta-auth-proxy |
|------|---------|---------|-------|-----------------|
| 言語 | Go | Java | N/A（SaaS） | Java |
| デプロイ | バイナリ / Docker | Docker / K8s | クラウドのみ | Docker |
| マルチテナント | ネイティブ（組織） | レルム（手動） | Organizations（有料） | ネイティブ（tenantsテーブル） |
| DB | CockroachDB / Postgres | Postgres / MySQL等 | マネージド | Postgres |
| メモリ使用量 | ~200-500MB | ~512MB-2GB | N/A | ~30-50MB |
| 起動時間 | ~5-10秒 | ~30-60秒 | N/A | ~200ms |
| UIカスタマイズ | 組織ごとのブランディング | FreeMarkerテーマ | Universal Login | jteテンプレート（完全制御） |
| 哲学 | プラットフォーム（すべてを行う） | プラットフォーム | プラットフォーム | ライブラリ（自分で組み立てる） |

## volta-auth-proxy ではどう使われている？

volta-auth-proxyはZITADELを**使用しません**。同じ問題を異なる哲学で解決する競合製品です。

### ZITADELが存在するのにvoltaが存在する理由

ZITADELは優れたソフトウェアですが、voltaが意図的に拒否する哲学を体現しています：

1. **プラットフォーム vs ライブラリ**：ZITADELはデプロイして設定するプラットフォーム。voltaはスタックに組み込むライブラリ。voltaは認証コードのすべての行を理解すべきだと信じています。

2. **リソース使用量**：ZITADELはCockroachDB（またはPostgres）＋200-500MB RAMのGoバイナリが必要。voltaは単一のJVM（~30MB）とおそらくすでに持っているPostgresデータベースだけ。

3. **複雑さの予算**：ZITADELには数百の機能（SAML、SCIM、マシンユーザー、Actions、ブランディング、パスワードレス）。voltaはPhase 1-2の集中した機能セットを持ち、段階的に機能を追加。voltaのすべての機能はあなたが含めることを選んだもの。

4. **ブラックボックスリスク**：ZITADELのCQRSプロジェクション付きイベントソーシングアーキテクチャは洗練されていますが、問題が発生した時にデバッグが困難。voltaは直接的なSQLクエリとIDEでステップ実行可能なJavalinハンドラーを使用。

### voltaよりZITADELを選ぶべきケース

- SAML、SCIM、パスワードレス認証が今すぐ必要
- 構成可能なライブラリではなく、箱出しで完全なプラットフォームが欲しい
- ZITADELのデプロイを管理する専任のDevOpsチームがいる
- 認証の内部を理解・修正する必要がない
- Goとイベントソーシングに慣れている

### ZITADELよりvoltaを選ぶべきケース

- 認証コードのすべての行を理解したい
- 最小限のリソース使用量が必要（エッジ/VPSデプロイ）
- プラットフォームを採用するのではなく、認証を段階的に進化させたい
- Javaを好み、IDEで認証をデバッグしたい
- 「コードを書かず設定で」というプラットフォームで痛い目にあったことがある

## よくある間違いと攻撃

### 間違い1：運用の複雑さを過小評価する

CockroachDB付きのZITADELには運用の専門知識が必要です。CockroachDBクラスター管理、バックアップ戦略、モニタリングは簡単ではありません。DevOpsチームがいない場合、Postgresのみモードはシンプルですが、CockroachDBのマルチリージョンの利点を失います。

### 間違い2：ZITADELをブラックボックスとして扱う

ZITADELのイベントプロジェクションが失敗したり遅延したりした場合、デバッグにはイベントソーシングアーキテクチャの理解が必要です。ブラックボックスとして扱うチームは問題が発生した時に苦労します。

### 間違い3：Actionsに過度に依存する

ZITADEL Actions（サーバーサイドスクリプト）はメンテナンスの負担になり得ます。Actionsの複雑なビジネスロジックはテスト、バージョン管理、デバッグが困難です。そのロジックがアプリケーションに属するべきかどうか検討してください。

### 間違い4：移行パスを無視する

ZITADELの独自APIとデータモデルに構築した後、移行は困難です。アプリケーションがIDレイヤーを抽象化し、永続的にロックインされないようにしてください。

### 攻撃：組織分離のバイパス

マルチテナントIDシステムにおける最大のリスクは、テナント間のデータ漏洩です。ZITADELの組織分離がユースケースで正しく機能することを検証してください -- 組織Aのユーザーが組織Bのリソースにアクセスできないことをテストしましょう。
